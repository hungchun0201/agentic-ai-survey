The KV Connector & SimpleCPUOffload

Part 1 · Foundation

1. Why a Connector Exists

vLLM's V1 scheduler owns a single GPU block pool and treats it as the only truth about kv_cache. Real deployments need more: DRAM offload to survive bigger working sets, remote stores that share prefixes across replicas, and prefill/decode disaggregation that hands off KV from one machine to another. Rather than merging every backend into the scheduler, V1 exposes a narrow interface -- the KV Connector -- that lets any external KV source participate in the scheduling loop without the scheduler knowing how it works internally.

Key design choice: the scheduler never calls CPU offload or any specific backend directly. It calls self.connector.get_num_new_matched_tokens() and self.connector.update_state_after_alloc(). Whether that means pinned DRAM, RDMA to another GPU, S3, or Mooncake is entirely the connector's problem.

1.1 The Scheduler / Worker Split

Every connector has two roles. The scheduler-side role is invoked inside the scheduler process, runs synchronously during schedule(), and owns planning: answering "do you have this prefix?" and "here are the GPU blocks I reserved, please plan the transfer". The worker-side role lives in each model worker, executes in the forward-pass path, and owns actual I/O: launching async memcpys, driving network transports, polling completions.

SCHEDULER

Scheduler-side role

Co-located with the Python scheduler process. Pure planning. No tensors, no CUDA calls.

  • get_num_new_matched_tokens()
  • update_state_after_alloc()
  • build_connector_meta()
  • update_connector_output()
  • request_finished()
  • take_events()
WORKER

Worker-side role

Co-located with each model worker / GPU process. Owns the CUDA streams, network handles, pinned CPU tensors, and actual memcpys.

  • register_kv_caches()
  • bind_connector_metadata()
  • start_load_kv() / save_kv_layer()
  • get_finished()
  • handle_preemptions()
  • build_connector_worker_meta()

The factory (KVConnectorFactory.create_connector) is called twice -- once with role=SCHEDULER in the scheduler process, once with role=WORKER in every worker process. Both sides see the same connector class, but the constructor branches and only instantiates the half that is needed.

Part 1 · Interface

2. The KVConnectorBase_V1 Contract

All connectors subclass KVConnectorBase_V1 in vllm/distributed/kv_transfer/kv_connector/v1/base.py. The contract is small but load-bearing -- the scheduler's entire extensibility story hinges on these eight scheduler-side methods and seven worker-side methods.

2.1 Scheduler-side Methods

get_num_new_matched_tokens(req, n) → (int|None, bool)

Look up how many additional prefix tokens the connector already has for this request, beyond the n already matched locally on GPU. Returns None to request "skip me this step, ask again later". Second element: is_async -- if True, this request will wait for a remote load to finish before running.

update_state_after_alloc(req, blocks, n_ext)

The scheduler just reserved blocks on GPU and n_ext of the tokens were claimed by you. Now the connector records the pairing: "remote block X must land in GPU block Y."

build_connector_meta(scheduler_output) → KVConnectorMetadata

End-of-step hook. The connector bundles everything the worker needs to know this step -- blocks to load, blocks to store, preemption flush flags -- into a single metadata dataclass attached to scheduler_output.

update_connector_output(output)

Worker returned completion events for prior steps. The scheduler-side manager accepts "these stores are done", "these loads arrived", and does the bookkeeping -- promoting CPU blocks into the prefix-cache hash table, releasing reference counts, etc.

request_finished(req, block_ids) → (bool, params?)

Called exactly once when a request terminates. Returning True tells the scheduler "don't free these GPU blocks yet, I'm still using them for an outbound transfer." The blocks will be released later by the worker's get_finished().

take_events() → Iterable[KVCacheEvent]

Optional. Surfaces KV cache events (BlockStored / BlockRemoved) for the observability subsystem, letting external dashboards see what a non-GPU tier is doing.

2.2 Worker-side Methods

register_kv_caches(kv_caches)

Called once at startup with the per-layer GPU KV tensors. The worker inspects stride / layout, allocates a mirrored pinned CPU pool, pins it via cudaHostRegister, and creates low-priority CUDA streams.

bind_connector_metadata(meta)

At the start of every forward pass the worker receives the per-step KVConnectorMetadata that the scheduler produced. Pending load / store event indices are enqueued here.

start_load_kv() / wait_for_layer_load()

Hooks for connectors that want to overlap per-layer KV loads with compute (NIXL, Mooncake). Most DRAM offload backends make these no-ops and defer everything to get_finished().

save_kv_layer(layer_name, kv) / wait_for_save()

Symmetric hooks for store-side overlap. Again, DRAM offload can no-op these -- stores are launched in get_finished() on a dedicated stream so they run in parallel with compute automatically.

get_finished(req_ids) → (sent, recv)

The workhorse. Called after model execution. Returns which requests finished async sends / receives this step. SimpleCPUOffload uses this moment to launch copies and poll completions.

handle_preemptions(meta)

Before preempted blocks can be reused by another request, every in-flight transfer touching them must complete. This method is a barrier.

build_connector_worker_meta() → KVConnectorWorkerMetadata

Reverse direction: the worker reports to the scheduler which event indices completed this step. The scheduler-side manager aggregates across TP/PP ranks to decide "this store is fully done."

SupportsHMA (mixin)

Hybrid Memory Allocator opt-in. Connectors declare they can cope with per-kv-group block lists via request_finished_all_groups(). The factory enforces that HMA-enabled deployments only load compatible connectors.

6. SimpleCPUOffloadScheduler.__init__

The scheduler-side brain lives at vllm/v1/simple_kv_offload/manager.py:67-155. Its smartest move is that it reuses the existing vLLM BlockPool / coordinator machinery to manage the CPU tier rather than inventing a separate cache. The CPU side ends up with the same prefix-cache hash table, the same Merkle-chained block hashes, and the same LRU free queue -- just pointing at pinned-DRAM tensors instead of GPU tensors.

self.cpu_kv_cache_config

Built by _derive_cpu_config (line 156). Mirrors the GPU KVCacheConfig but scales num_blocks by the cpu_bytes / gpu_bytes ratio so the CPU pool has proportionally more blocks than GPU.

self.cpu_coordinator

A real KVCacheCoordinator obtained via get_kv_cache_coordinator(...). This is the same class the scheduler uses for its GPU tier, which means find_longest_cache_hit just works.

self.cpu_block_pool: BlockPool

Extracted from the coordinator. Owns the LRU free queue, the cached_block_hash_to_block map, and all block metadata for the CPU tier.

self._gpu_block_pool

Late-bound via bind_gpu_block_pool. Needed for touch() calls that prevent the scheduler from recycling GPU blocks while an async copy is still using them.

self._reqs_to_load / self._reqs_to_store

Per-request state tables. Load side remembers (GPU, CPU) block pairings for in-flight H2D transfers. Store side (eager only) tracks which blocks per request have already been scheduled for offload.

self._cursor: KVCacheBlock | None

Lazy mode only. Pointer into the GPU free_block_queue doubly-linked list. Each step walks forward from here, offloads any cached-but-not-yet-on-CPU blocks it finds, and advances.

self._load_event_counter / _store_event_counter

Monotonic integer job IDs. Every batch of async transfers gets a fresh ID; workers report completions by ID so the scheduler doesn't need to know about individual blocks.

self._store_event_pending_counts

For TP / PP. Each worker rank reports the same event_idx when it finishes its copy. The scheduler only commits the store (promotes CPU blocks into the prefix cache) when all world_size ranks have reported.

Part 2 · Load path

7. get_num_new_matched_tokens -- the DRAM Hit Lookup

This is the first method the scheduler calls. Location: manager.py:208-228. Input: the request and how many tokens the GPU already has cached. Output: the number of additional tokens available in DRAM plus a flag saying whether fetching them requires an async wait.

# vllm/v1/simple_kv_offload/manager.py  --  lines 208-228
def get_num_new_matched_tokens(
    self, request: "Request", num_computed_tokens: int
) -> tuple[int | None, bool]:
    """Return (num_new_tokens, is_async) from consecutive CPU cache hits."""
    skipped = num_computed_tokens // self.block_size
    remaining_hashes = request.block_hashes[skipped:]           # 1

    if not remaining_hashes:
        return 0, False
    # Must recompute at least the last token, matching
    # kv_cache_manager.get_computed_blocks()'s policy.
    max_hit_len = request.num_tokens - 1 - num_computed_tokens  # 2
    if max_hit_len <= 0:
        return 0, False
    _, hit_length = self.cpu_coordinator.find_longest_cache_hit(
        remaining_hashes, max_hit_len                           # 3
    )

    if hit_length > 0:
        return hit_length, True                             # 4
    return 0, False

1. Skip the blocks the GPU already has

skipped = num_computed_tokens // block_size. We start matching at the first block the GPU did not already serve.

2. Never match the final token

max_hit_len = num_tokens - 1 - num_computed_tokens. vLLM's prefix cache refuses to serve the request entirely from cache -- at least one token must go through the model to produce logits. This mirrors kv_cache_manager.get_computed_blocks.

3. Reuse find_longest_cache_hit

The scheduler's own prefix-cache matcher is called -- but against cpu_coordinator. Same algorithm, same block-hash format, different storage tier.

4. Return hit_length with is_async = True

Any hit forces the request into WAITING_FOR_REMOTE_KVS on the scheduler side -- the request gets its GPU blocks reserved but is not actually scheduled to run compute until get_finished() reports the H2D load is done.

Side-effect free: the base contract says this method may be called multiple times per request and must be idempotent. This implementation is -- it only reads from cpu_coordinator, never mutates anything.

8. update_state_after_alloc -- Pairing CPU Blocks with GPU Slots

Once the scheduler has reserved GPU blocks via kv_cache_manager.allocate_slots(), it calls back here. Location: manager.py:232-313. Goal: figure out exactly which CPU blocks hold the matched prefix, decide which GPU blocks they should land in, and pin both sides so nothing evicts them mid-transfer.

Inputs
request, blocks = newly-allocated GPU KVCacheBlocks across all groups, num_external_tokens = hit length reported in step 7.

A. Eager-mode registration line 245-250
In eager mode, register an empty StoreRequestState so future build_connector_meta calls can start appending newly-computed blocks for this request.

Why call find_longest_cache_hit twice? Step 7 only needed a count. Step 8 needs the actual KVCacheBlock objects so it can build (GPU, CPU) pairs. Re-querying is cheap -- it's a hash-table walk -- and avoids smuggling state between the two calls (which would break the idempotency guarantee of step 7).

9. build_connector_meta -- Packaging the Step's Transfers

Called once per schedule() tick. Location: manager.py:315-364. Produces a SimpleCPUOffloadMetadata that is shipped via IPC to every worker and tells them what to copy.

# vllm/v1/simple_kv_offload/metadata.py  --  lines 15-38
@dataclass
class SimpleCPUOffloadMetadata(KVConnectorMetadata):
    """
    Metadata passed from scheduler to worker for CPU offload operations.
    The worker receives flat block lists keyed by a monotonic event_idx.
    Job->req_id translation is handled by the scheduler-side manager
    (via inverse maps), so the worker never knows about request identities.
    """

    # Load event per step. -1 means no blocks to load this step.
    load_event: int = INVALID_JOB_ID
    load_gpu_blocks: list[int] = field(default_factory=list)
    load_cpu_blocks: list[int] = field(default_factory=list)
    load_event_to_reqs: dict[int, list[str]] = field(default_factory=dict)

    # Store event per step. -1 means no blocks to store this step.
    store_event: int = INVALID_JOB_ID
    store_gpu_blocks: list[int] = field(default_factory=list)
    store_cpu_blocks: list[int] = field(default_factory=list)

    # Whether any requests were preempted this step -- worker must flush.
    need_flush: bool = False

Each tick builds at most one load event and one store event; both carry flat block lists. The load_event_to_reqs dict is the only piece of request identity that reaches the worker -- everything else is block IDs, so the worker can treat stores as identity-free.

# vllm/v1/simple_kv_offload/manager.py  --  lines 335-353
# --- Loads ---
load_event = -1
load_gpu: list[int] = []
load_cpu: list[int] = []
load_req_ids: list[str] = []
for req_id, load_state in self._reqs_to_load.items():
    if load_state.load_event is not None:
        continue                      # already flushed in an earlier step
    assert load_state.transfer_meta is not None
    load_gpu.extend(load_state.transfer_meta.gpu_block_ids)
    load_cpu.extend(load_state.transfer_meta.cpu_block_ids)
    load_req_ids.append(req_id)
if load_req_ids:
    load_event = self._load_event_counter         # assign this batch an ID
    self._load_event_counter += 1
    for req_id in load_req_ids:
        self._reqs_to_load[req_id].load_event = load_event
    self._load_event_to_reqs[load_event] = load_req_ids
Batching discipline: all not-yet-launched loads across all requests are coalesced into one event this step. The worker then launches a single batched cuMemcpyBatchAsync covering every pair. This amortises the per-call overhead (~5 ms) across the whole batch.
Part 2 · Store path

10. Two Store Modes -- Lazy Cursor vs Eager Scan

The interesting half of the PR is how newly-computed blocks are decided to be offloaded to CPU. The connector supports two strategies, selected by lazy_offload in extra config. They have very different performance envelopes.

EAGER

_prepare_eager_store_specs

Walk every running request, for every kv_cache_group, scan its block_ids from num_stored_blocks[g] up to ready_blocks_g = confirmed_tokens // g_block_size. Offload everything newly computed, stopping only if CPU pool runs out.

Trade-off
  • + Maximum coverage -- any block that could be useful later is saved.
  • + Predictable latency -- the store happens soon after compute finishes.
  • − Wastes bandwidth on blocks that will never be reused (e.g. streaming workloads).
LAZY

_prepare_lazy_store_specs

Walk the GPU free_block_queue starting from self._cursor. Each cached block the cursor passes over gets offloaded if it isn't already on CPU. Stops when target_free blocks are covered or CPU pool is full.

Trade-off
  • + Only offloads blocks that are about to be evicted -- bandwidth spent on blocks that actually matter.
  • + Piggy-backs on the GPU pool's LRU ordering -- zero extra tracking state.
  • − May miss a block if the scheduler frees it faster than the cursor advances (worker launches after evict wins).

10.1 Lazy cursor -- the single-pass walk

Location: manager.py:375-441. The heart of lazy mode is a single pointer into the GPU free queue's doubly-linked list. Because vLLM's FreeKVCacheBlockQueue is in LRU order, walking forward from the head is walking from "most likely to be evicted soon" to "least likely."

# vllm/v1/simple_kv_offload/manager.py  --  lines 390-428
# Validate cursor: stale if the block it points to was removed from free queue.
if self._cursor is not None and self._cursor.ref_cnt > 0:
    self._cursor = None

# Determine start node.
if self._cursor is None:
    node = free_queue.fake_free_list_head.next_free_block
else:
    node = self._cursor.next_free_block

tail = free_queue.fake_free_list_tail
gpu_ids: list[int] = []
block_hashes: list[bytes] = []
covered = 0
last_visited = self._cursor

while (
    node is not None                         # not off the end
    and node is not tail                      # not the sentinel tail
    and covered < self._target_free            # budget not reached
    and len(gpu_ids) < num_cpu_free            # CPU pool has room
):
    last_visited = node
    bhash = node.block_hash

    if (
        bhash is not None                      # block has a prefix hash
        and not node.is_null
        and cpu_pool.cached_block_hash_to_block.get_one_block(bhash) is None
        #   ^ and it's not already cached on CPU
    ):
        gpu_ids.append(node.block_id)
        block_hashes.append(bhash)

    covered += 1
    node = node.next_free_block

self._cursor = last_visited                 # save for next step
Cursor correctness: when a request reclaims a cached block by incrementing its ref_cnt, that block is unlinked from the free queue. If the cursor happens to point at it, the next step sees ref_cnt > 0 and resets to the head -- no stale pointer chasing.

10.2 Eager per-request scan

Location: manager.py:443-579. Iterates yield_req_data(scheduler_output) to get every running request's newly-written blocks. The "confirmed_tokens" guard is the subtle part: we cap the scan at blocks whose KV data is actually computed, not speculatively written by async scheduling.

# vllm/v1/simple_kv_offload/manager.py  --  lines 497-545
# Confirmed tokens: KV data written and visible to all streams.
req = state.request
confirmed_tokens = req.num_computed_tokens - req.num_output_placeholders

for g in range(num_groups):
    already_stored_g = state.num_stored_blocks[g]
    group_gpu_ids = block_ids_by_group[g]

    # Cap to blocks with confirmed KV data.
    g_block_size = kv_cache_groups[g].kv_cache_spec.block_size
    ready_blocks_g = confirmed_tokens // g_block_size
    scannable = group_gpu_ids[already_stored_g:ready_blocks_g]

    for gpu_block_id in scannable:
        gpu_block = gpu_block_pool.blocks[gpu_block_id]
        if gpu_block.is_null:
            advanced_per_group[g] += 1
            continue

        bhash_with_group = gpu_block.block_hash
        if bhash_with_group is None:
            break                                   # not hashed yet, stop

        if (
            gpu_block_id in gpu_blocks_this_step
            or cpu_block_pool.cached_block_hash_to_block.get_one_block(
                bhash_with_group
            ) is not None
        ):
            advanced_per_group[g] += 1
            continue                                # dedup: already scheduled or cached

        if num_free <= 0:
            out_of_space = True                     # CPU pool full
            break
        num_free -= 1

        gpu_block_ids.append(gpu_block_id)
        block_hashes_to_store.append(bhash_with_group)
        advanced_per_group[g] += 1
The confirmed_tokens subtraction: vLLM's async scheduler can optimistically advance num_computed_tokens before the GPU has actually finished those tokens. num_output_placeholders counts those in-flight tokens. Subtracting gives us a lower bound on what is really computed -- so we never offload a block whose KV data is still a CUDA graph placeholder.
Part 2 · Completion accounting

11. update_connector_output -- TP-Aware Completion

Location: manager.py:581-648. This is where the scheduler folds worker completion reports back into its state. Loads are simple -- a load is done when any one worker reports it. Stores are tricky because the same event_idx must be completed by every TP/PP rank before the scheduler can safely promote CPU blocks into the prefix cache.

11.1 What _process_store_event actually does

# vllm/v1/simple_kv_offload/manager.py  --  lines 625-648
def _process_store_completion(
    self, gpu_block_ids: list[int], cpu_block_ids: list[int]
) -> None:
    """Cache CPU blocks per-group and release GPU refs.

    Block hashes were stamped on CPU blocks at allocation time. Here we
    just register them in the cache map so they become discoverable by
    the load path."""
    cpu_blocks = [self.cpu_block_pool.blocks[bid] for bid in cpu_block_ids]
    for cpu_block in cpu_blocks:
        bhash = cpu_block.block_hash
        assert bhash is not None
        self.cpu_block_pool.cached_block_hash_to_block.insert(
            bhash, cpu_block
        )
    # Release touch refs -- LRU resumes.
    self.cpu_block_pool.free_blocks(cpu_blocks)
    assert self._gpu_block_pool is not None
    self._gpu_block_pool.free_blocks(
        self._gpu_block_pool.blocks[bid] for bid in gpu_block_ids
    )
Hash stamping happens at store-prep time: when _prepare_lazy_store_specs or _prepare_eager_store_specs allocates the CPU blocks, it immediately writes the GPU block's hash onto the CPU block's _block_hash field (manager.py:434-435 and 552-553). The worker then runs the copy. When the scheduler hears "store complete," it just inserts the pre-stamped hash into the lookup table -- no hashing work on the hot path.
Part 2 · Worker side

12. Worker Counterpart -- Streams, Events, cuMemcpyBatchAsync

The worker lives at vllm/v1/simple_kv_offload/worker.py (305 lines). It turns the declarative metadata produced by the scheduler into actual CUDA work.

register_kv_caches() worker.py:66-182

Startup. Inspects per-layer tensors, deduplicates shared storage, flattens into [num_blocks, block_bytes] views. Allocates a mirrored pinned CPU pool via cudaHostRegister (bypassing PyTorch's caching allocator that rounds up to the next power of 2). Spins up two low-priority CUDA streams so KV I/O yields to compute.

bind_connector_metadata() worker.py:184-189

At the start of every forward pass. Stores the SimpleCPUOffloadMetadata and records the pending load / store event indices so get_finished() knows what to poll.

get_finished() worker.py:203-259

The heart of the worker. Called after model execution. (1) Launches the batched H2D / D2H copies on dedicated streams. (2) Polls event queries (non-blocking) to advance the load / store high-water marks. (3) Returns finished request IDs to the scheduler; queues completed store event indices for the reverse metadata.

handle_preemptions() worker.py:271-289

Barrier. If the scheduler told us need_flush=True (because some request was preempted and its blocks will be reused), synchronize every in-flight load and store event, then clear the tracking lists.

build_connector_worker_meta() worker.py:261-269

Reverse channel. Returns SimpleCPUOffloadWorkerMetadata(completed_store_events={event_idx: 1}) for every store this rank finished since the last call. Each rank reports 1; the scheduler's _store_event_pending_counts sums across ranks until it hits world_size.

12.1 Why launches are deferred to get_finished()

The base class offers start_load_kv() and save_kv_layer() for per-layer overlap. SimpleCPUOffload makes them all no-ops. Instead, everything happens in get_finished(), which runs once per step on the dedicated streams. Reasoning:

  • Batching wins over fine-grained overlap for DRAM. PCIe DMA's per-call overhead (~5 ms resolving the driver API, setting up descriptors) dominates small transfers. Batching 64 block copies into one cuMemcpyBatchAsync amortises this away.
  • Dedicated low-priority streams already overlap with compute. Since register_kv_caches created load_stream and store_stream at the lowest CUDA priority, the GPU's scheduler runs them whenever the compute queue has idle SMs. The Python side doesn't need to orchestrate per-layer overlap -- CUDA does it.
  • Correctness is easier. Launching after compute means the KV data is definitely written. No need to insert cross-stream synchronization, no subtle data races.

13.1 Design Principles PR #37160 Preserves

No scheduler changes beyond opt-in hooks

Seven lines in scheduler.py. Everything else in the PR is isolated to the connector package.

Reuse BlockPool, don't replicate

The CPU tier is just another BlockPool with its own LRU and prefix-cache map -- zero new cache code.

Flat block IDs across IPC

Worker sees only list[int] + event_idx. No request identities cross the boundary except for the load_event_to_reqs reverse map.

Event-count TP quorum, not broadcast

TP/PP completion is resolved by counting per-rank reports in _store_event_pending_counts. No explicit rank-0 leader, no broadcast collectives.

Pin memory manually, not via PyTorch

cudaHostRegister avoids PyTorch's pin_memory=True allocator that rounds every allocation up to the next power of 2.

Batch + defer beats fine-grained overlap

Per-layer hooks are no-ops. Everything launches in one batched call in get_finished() on a low-priority stream.

13.2 Known Limitations

  • reset_cache() raises NotImplementedError -- the connector cannot synchronously flush all in-flight transfers, so reset_prefix_cache() (used in RLHF weight updates) is unsupported.
  • DCP / PCP world sizes must be 1 (asserted at init). Multi-layer context-parallel is not yet wired.
  • num_stored_blocks can go stale if the CPU pool evicts a mid-request block -- the source comments this as a future fix (FIXME (yifan): handle CPU cache eviction...).
  • Eager mode may miss the final block of a request that finishes the same step its last block completes -- comments flag TODO: flush on finish.
Bottom line: PR #37160 is 1,354 lines of concentrated work. It delivers a production-ready DRAM offload tier by leaning on three things the scheduler already had -- BlockPool LRU, prefix-cache hashing, Merkle-chained block hashes -- and adding only two genuinely new concepts: the monotonic event counter for async accounting, and the cursor that walks the GPU free queue in lazy mode. Everything else is pure plumbing.
