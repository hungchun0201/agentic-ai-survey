FCFS vs LMCache vs Continuum вҖ” Full Experiment Report

Three-way KV cache strategy comparison across 4 GPUs, per-turn latency analysis, cache starvation measurement, and a deep-dive bottleneck ablation identifying wait_for_save as LMCache's true overhead

1. Experiment Context

1.1 Research Question

Multi-turn agent workloads (chain-of-thought, tool calling, ReAct-style loops) suffer a pathological behavior under standard vLLM: the KV cache for a job's context is computed during turn N, evicted during the tool-call gap between turn N and N+1, then recomputed from scratch on turn N+1. At 13K tokens/context this recompute alone dominates end-to-end latency. We study three KV cache management strategies for eliminating this recompute and quantify their tradeoffs under real load.

1.2 Three Strategies Under Test

FCFS (baseline)

Stock vLLM v1 scheduler. Block-level prefix cache is enabled, but evicted blocks are simply recomputed on the next turn. No explicit cross-turn KV preservation.

Failure mode: recompute of full context every turn under high concurrency

LMCache (v0.3.7)

Upstream production KV cache layer. Offloads evicted KV blocks to CPU DRAM via PCIe, reloads on hit. Integrates with vLLM through the v1 KV transfer connector interface (LMCacheConnectorV1).

Failure mode: wait_for_save blocks the step loop on every prefill вҖ” see Sections 5вҖ“13

Continuum

Research prototype (this work). Keeps KV blocks for active agent jobs pinned in GPU VRAM across tool-call gaps, bypassing the evict/recompute cycle entirely. No CPU transfer. Identifies jobs via job_id and releases pin on job_completed signal.

Failure mode: pinned blocks can starve new admissions (Turn 2 explosion) вҖ” see В§3

1.3 Workload Design

The benchmark simulates an 8-turn agent job: the client sends a prompt, the server generates a short response containing a tool call, the client sleeps for a fixed tool-execution time, then appends a (large) tool output back onto the context and starts the next turn. Each turn grows the prompt by ~2K tokens, so turn 8 sees a prompt of ~13K tokens. Jobs arrive on a Poisson process at rate JPS (jobs per second).

1.4 Why Random Prompts Are Load-Bearing

A retracted earlier result

An earlier version of this benchmark (dir 20260406_static-prompt_jps-sweep_d90s_tool0.5s/) sent the same tool output string to every job. vLLM's block-level prefix cache вҖ” enabled by default вҖ” then hashed identical blocks across jobs, giving ~99% prefix cache hit and effectively 0 real KV eviction. Under that regime, LMCache appeared to beat FCFS by 39вҖ“66% on A100/L40S. That result is an artifact and is retracted. Under random prompts (31% prefix cache hit, realistic KV pressure), LMCache is slower than FCFS on every tested large-VRAM GPU.

The fix (committed in ab_benchmark.py:70) uses per-job per-turn seeded randomness:

def _generate_random_tool_output(job_id: str, turn_idx: int,
                                 target_tokens: int) -> str:
    # Uses a seeded RNG per (job_id, turn_idx) for reproducibility.
    seed = hashlib.md5(f"{job_id}:{turn_idx}".encode()).hexdigest()
    rng = random.Random(seed)
    header = f"[{job_id}:turn{turn_idx}:{seed[:8]}] "
    ...  # random words from a ~80-word pool

Each (job, turn) combination gets a unique token sequence. vLLM's prefix cache can still match the first few tokens of system prompt but cannot share tool-output bodies across jobs. All results in Sections 2вҖ“13 use this randomized prompt.

Why larger VRAM shifts the saturation point, not the ordering

More VRAM = more concurrent jobs at the same JPS before KV eviction kicks in. Below the saturation point, all three policies perform similarly because prefix cache handles reuse. LMCache is consistently worse than FCFS on large-VRAM GPUs

Every JPS Г— GPU cell shows LMCache at or worse than FCFS. Worst case: H100 JPS=3 with +158%. The mechanism is identified in В§5вҖ“13: wait_for_save forces the step loop to block on a PCIe DMA every time a prefill request is in the batch, and under random prompts LMCache's CPU cache hit rate drops to 31% so the offload cost isn't being amortized by reuse.

At JPS=1 none of the policies help вҖ” FCFS wins

Below the saturation point there is no real KV pressure: a single job's blocks fit in VRAM, nothing gets evicted, prefix cache handles reuse for free. Adding LMCache adds wait_for_save overhead (+40% H200, +28% H100). Adding Continuum adds pin bookkeeping (+23% H200, +15% H100). Both are overhead without benefit at this load level.

3. Per-Turn Latency вҖ” Where the Time Goes

Average job completion time hides the mechanism. Breaking latency down by turn reveals three distinct failure modes вҖ” one per policy. All charts below use H200 (141 GB) controlled-sweep data.

3.1 Load regime: JPS=3 (below saturation)

At JPS=3 on H200 there's headroom. All three policies stay below 4s per turn. FCFS has the characteristic small bump from turn 2 onwards (each turn's recompute gets larger) but the effect is mild because KV pressure hasn't kicked in. LMCache is slightly better than FCFS in this regime вҖ” this is the only measured load level where LMCache actually helps on H200, and it's because the save pipeline is almost never triggered.

3.2 Load regime: JPS=6 (saturation threshold)

At JPS=6 the two pictures start to diverge. FCFS and LMCache climb monotonically because each turn's recompute cost grows with context, and under saturation the cost of that recompute is paid in full. Continuum flips the picture: turn 1 is slow (21s вҖ” new admission waits), turn 2 is slow (56s вҖ” first real prefill competes with other new jobs), but from turn 3 onwards the pinned KV means the job only decodes 20 new tokens. Turns 3вҖ“8 each take 5вҖ“14s regardless of context size.

3.3 Load regime: JPS=10 (high load)

The ceiling separates sharply. FCFS and LMCache hit ~88 s/turn at the peak (turn 7): at 13K-token context, that's essentially the prefill compute of the entire context every turn. The two lines are nearly identical вҖ” LMCache's DRAM offload provides no recompute savings in this regime, it's just paying extra wait_for_save on top. Meanwhile Continuum's burden is concentrated entirely in turn 2 (56s spike); turns 3вҖ“8 are in the 5вҖ“15s range.

3.4 Load regime: JPS=15 (fully saturated)

Continuum's Turn 2 explosion вҖ” and why FCFS/LMCache don't have one

At H200 JPS=15, Continuum's turn 2 alone takes 115.5 s while every subsequent turn takes only 5вҖ“15 s. FCFS and LMCache show no such spike вҖ” their per-turn cost climbs monotonically from T1вүҲ21s to T7вүҲ140s. This is not Continuum being "worse at T2" in some absolute sense; it is a structural difference in where each strategy pays its prefill cost.

FCFS and LMCache recompute the full context at every turn because the KV from the previous turn has been evicted during the 0.5s tool gap. Why the queue forms only at T2 вҖ” verified by scheduler trace

Continuum has an implicit admission to the pinned pool at T2. At T1 a new job only prefills 92 tokens and generates 20 вҖ” this occupies almost no VRAM, so the scheduler freely admits it. Between T1 and T2 the client appends the ~1,728-token tool output; now at T2 the job wants Continuum to pin ~1,800 tokens of KV for the remaining ~160 seconds of its lifetime. That is a commitment to hold a long-term VRAM slot. Under JPS=15 saturation, the pinned working set is near-full with jobs in T3вҖ“T8, and new T2 requests queue behind them waiting for slots to free up.

FCFS and LMCache don't have this gating because they don't have a pinned pool. Every request вҖ” whether it's a job's T1, T2, or T7 вҖ” is just a normal prefill request that joins the batch using standard vLLM fairness. Queueing still happens, but it's distributed uniformly across all turns based on batch-level contention, producing the monotonic T1вҶ’T8 climb we see in FCFS/LMCache. There is no single "admission gate" turn. The total wait time across the 8 turns is larger for FCFS (~710s) than Continuum's single T2 spike (~115s), but because it's spread out it looks less dramatic on the per-turn chart.

On H100 the same effect is sharper (T2 = 178s at JPS=15) because H100's 80 GB VRAM fits fewer concurrent pinned jobs than H200's 141 GB, so the pinned pool saturates earlier and the admission queue grows faster. The T2 spike is a pin-specific failure mode, not a generic "Continuum is slow at T2" claim. A potential mitigation is admission control that reserves VRAM slots for new arrivals in advance, or a shorter pin TTL вҖ” both are in future work (В§12).

Finding 2 вҖ” T2 TTFT grows with arrival order, and Continuum grows 3.7Г— steeper

The 661 jobs of each policy were binned by arrival order (first 50, 100вҖ“200, 300вҖ“400, last 50) and the mean T2 TTFT per bin was computed. For the first 50 jobs вҖ” arriving into an empty system вҖ” all three policies deliver T2 TTFT in ~1 s. As arrival order progresses, the queue builds. FCFS and LMCache rise linearly to ~70 s for the last 50 jobs. Continuum rises to 269 s вҖ” 3.7Г— steeper.

This is direct evidence that Continuum's T2 queue is a different queue from the FCFS/LMCache T2 queue. Under FCFS/LMCache, a T2 request competes with other batch-level prefill requests; under Continuum it additionally competes for a slot in the pinned working set, which is a scarcer resource. The first few jobs get slots for free. Late arrivals wait for existing pinned jobs to finish and release their slots, which takes the ~190s of Continuum's steady-state JCT вҖ” and the last 50 jobs' 269s T2 TTFT is approximately that.

Finding 3 вҖ” Continuum jobs are "in the club" after T2: arrival order no longer matters

This is the cleanest confirmation. At T7, FCFS's TTFT grows from 32s (first 50) to 151s (jobs 300-400), matching the arrival-order pattern seen at T2. FCFS's late-arriving jobs pay a penalty on every turn. Continuum's T7 TTFT, in contrast, stays at 0.5вҖ“1.4s regardless of arrival order вҖ” the first 50 jobs and the last 50 jobs have indistinguishable T7 performance. Once a Continuum job has crossed the T2 admission gate, it is decoupled from admission-queue pressure for the rest of its lifetime. The pin guarantees fast access to its pre-prefilled KV, and the incremental per-turn prefill work is tiny.

Note on sample sizes: FCFS T7 last-50 bin has too few completed jobs to report (many jobs in the tail don't reach T7 within the benchmark + post-benchmark window under FCFS's slow service rate). LMCache T7 has the same truncation. Continuum's T7 bin for the last 50 is fully populated because all 661 Continuum jobs finished all 8 turns. That truncation asymmetry understates FCFS/LMCache's per-turn pain at the tail вҖ” their true T7 numbers for late jobs would be worse than 151s.

What the trace data settles

(1) Continuum's T2 latency is dominated by queue wait, not compute вҖ” trace TTFT at T2 is 116s p50 while pure compute for a 1,748-token prefill is ~1.5s. (2) The queue is specific to T2 (0.4вҖ“0.5s at T3вҖ“T8). (3) The queue grows 3.7Г— steeper with arrival order than FCFS/LMCache's T2 queue, which rules out "Continuum just has ordinary queue wait at every turn". (4) The shape of Continuum's "fast after T2" behavior is exactly what a pin-based admission gate would produce: zero cost to cross the gate, then cheap service on the inside. The working hypothesis in the previous paragraph is now empirically confirmed, not a guess.

3.5 Three distinct failure modes

FCFS: monotonic recompute wall

Each turn's latency is proportional to context size because each turn recomputes the full KV. The line grows with turn number in a shape that tracks the prompt-length column in В§1.3.

LMCache: FCFS shape + wait_for_save tax

LMCache's per-turn line is the same shape as FCFS, slightly elevated. The DRAM offload fires but the 31% cache hit rate doesn't deliver enough reuse to pay for wait_for_save. Net: worse than FCFS at every turn вүҘ 2.

Continuum: front-loaded T1/T2 stall

All the cost is concentrated in turn 1 (admission wait) and turn 2 (first real prefill competes for compute). Turns 3вҖ“8 are flat near 5вҖ“15s because the pinned KV means decode-only work from then on. Total JCT = T1 + T2 + 6 Г— decode вүҲ T2 dominates.

5. Root Cause: Why LMCache Is Slower

5.2 Inside wait_for_save: cudaStreamSynchronize

cudaStreamSynchronize(stream) is a CUDA Runtime API call that blocks the calling CPU thread until every operation previously queued on the given stream has completed on the GPU. 6.2 Sync Ablation (4 Modes)

The initial hypothesis (В§6) was that torch.cuda.synchronize() in LMCache's KV-onload path blocks the CPU scheduler thread and starves the GPU. To falsify or confirm this, we need to replace the synchronization primitive itself with progressively looser variants and see whether wall-clock latency improves. If it does, the sync primitive was the bottleneck; if it doesn't, the bottleneck is somewhere else. The tool is our_experiments/scripts/patch_sync.py, which runtime-patches gpu_connector.py inside the LMCache installation in the conda env (so vLLM picks up the change on server restart).

The target line in original LMCache (gpu_connector.py) is at the end of batched_to_gpu, after firing the async CPUвҶ’GPU KV load on a dedicated load_stream:

# NOTE: wait until both compute and load streams are done
torch.cuda.synchronize()   # вҶҗ the line under test

Each mode replaces exactly this call with a different primitive (the rest of LMCache code is untouched). All four modes are tested head-to-head against the same FCFS/Continuum baselines under identical JPS and workload. Here is what each mode does and what it is supposed to test:

D default

What it does: No change вҖ” runs LMCache's original torch.cuda.synchronize(). This is a device-wide block: the CPU thread waits until every kernel on every CUDA stream on the current device has completed, not just the load stream.

What it tests: baseline вҖ” the "worst-case" sync strategy that blocks on everything.

S stream_only

What it does: replaces torch.cuda.synchronize() with explicit per-stream syncs on only the streams this operation cares about:

self.load_stream.synchronize()
self.compute_stream.synchronize()

What it tests: whether the "device-wide" scope of default sync is over-conservative. If stream-scoped syncs are meaningfully faster, then default was wasting time waiting for unrelated kernels on other streams.

E event_async

What it does: replaces the device sync with a CUDA event that records a specific point on the current stream, then waits for just that point:

self._sync_event = torch.cuda.Event()
self._sync_event.record()
self._sync_event.synchronize()

What it tests: events are the cheapest fine-grained sync primitive CUDA offers вҖ” only one condition variable to check, no scan over all device work. If the bottleneck is CPU overhead inside the sync primitive itself, event_async should show it.

Caveat: record() without a stream argument records on the current stream (typically the default compute stream), so this event may not actually capture load_stream activity. If that's true, event_async is effectively incorrect вҖ” which is part of what the experiment reveals.

N no_sync вҡ  UNSAFE

What it does: removes all sync calls in the onload path вҖ” both the torch.cuda.synchronize() and the subsequent self.load_stream.synchronize():

pass  # sync removed for ceiling measurement

What it tests: the ceiling вҖ” how fast could LMCache possibly run if every synchronization primitive in the onload path were free? This is not a correct implementation (the next kernel could read paged KV blocks before the DMA finishes), but as long as it doesn't crash and the outputs are approximately right, it gives an upper bound on "best case if sync cost were zero".

Logical progression of the four modes

The four modes are ordered from most conservative to least: default blocks on the whole device, stream_only blocks only on the two involved streams, event_async blocks only on a single event, and no_sync blocks on nothing. If the sync-hypothesis is correct, each step should give a progressively larger speedup; if it's wrong, the curve will be flat. The no_sync ceiling is especially load-bearing вҖ” if even the free-sync case doesn't beat default by more than a few percent, the bottleneck is demonstrably not in any sync primitive. Spoiler: Finding 1 in В§9 shows exactly this flat curve.

What is a scheduler "step"?

A step in vLLM is one full iteration of the engine loop: the scheduler picks a batch of requests, the GPU runs one model.forward() on that batch (which produces exactly one output token per request in the batch), the sampler picks the token, and the loop returns to the scheduler for the next iteration. A request with an 8,000-token prompt and 20 completion tokens will go through at least 20 steps: the first few steps chunk-prefill its prompt, the subsequent steps each generate one decode token. Every metric in the table below counts per-step averages across the 28-run benchmark (4 JPS values Г— 7 policies).

8.4 Why the blocking wait exists вҖ” upstream rationale

A natural question after reading В§5вҖ“В§8 is: why does vLLM force this blocking wait at every step boundary instead of letting the save continue asynchronously across steps? If the DMA is genuinely async, running it concurrently with the next forward pass would hide most of its cost. The fact that vLLM chose to block deserves an explanation. The vLLM and LMCache GitHub repos, RFCs, PRs, and the vLLM blog were searched. What was found вҖ” and not found вҖ” is reported here.

The one upstream sentence on the subject

The only authoritative upstream text explaining why wait_for_save blocks is a single-sentence docstring in the v1 connector base class:

"Block until all the save operations is done. This is called as the forward context exits to ensure that the async saving from save_kv_layer is complete before finishing the forward. This prevents overwrites of paged KV buffer before saving done."

Source: vllm/distributed/kv_transfer/kv_connector/v1/base.py вҖ” KVConnectorBase_V1.wait_for_save docstring

That sentence is the entire upstream design discussion. There is no GitHub issue debating the trade-off, no PR proposing a non-blocking alternative, no blog post elaborating on the constraint, and no design doc. The wait-for-save barrier is treated as a self-evident safety requirement rather than an explicit trade-off.

What the docstring actually means: the paged block reuse hazard

Unpacking "prevents overwrites of paged KV buffer" from first principles: vLLM's paged attention allocates KV cache in fixed-size blocks (2 MiB, 16 tokens on Llama 3.1 8B bf16). The scheduler's block allocator treats a block as free and available for reuse the moment the request holding it finishes or is evicted. In the normal vLLM design, blocks cycle rapidly through freeвҶ’allocatedвҶ’writtenвҶ’readвҶ’free. The next step can and routinely does assign a just-freed block to a new request's prefill. Without wait_for_save, the following race is possible:

Step N вҖ” request A

Prefill writes KV for request A into block #47. At step N's forward-context exit, save_kv_layer() fires an async GPUвҶ’CPU DMA reading from block #47, queued on store_stream.

вҶ“

(no wait_for_save)

vLLM's engine loop proceeds immediately to step N+1. The store_stream DMA is still running in the background, reading block #47 byte by byte.

вҶ“

Step N+1 вҖ” request B hijacks block #47

Scheduler's block allocator sees #47 as free (because request A's lifetime ended) and allocates it to request B's prefill. B's attention kernel writes new KV to block #47. At the same moment the DMA from step N is still reading вҖ” result: CPU-side cache receives a mix of A's pre-overwrite and B's post-overwrite bytes. The cache is corrupted silently.

The blocking wait_for_save forbids this by freezing the engine loop until every in-flight save has finished reading every dirty block. Only then can the block allocator's free pool be trusted for reuse. It is a correctness barrier for the block allocator, not a performance choice вҖ” there is no way to skip it without changing the semantics of the free pool.

9. Key Findings

Finding 1: The sync hypothesis is FALSIFIED

Removing torch.cuda.synchronize() entirely (no_sync mode) produced minimal improvement: JPS=10 only 2% better, JPS=3 actually 51% worse. All sync variants (stream, event, nosync) performed equal or worse than default.

Finding 2: wait_for_save is the dominant overhead

The distribution of wait_for_save() is bimodal (В§5.1): 91% of calls (decode-only) return in ~0.01 ms; the remaining 9% (prefill-step) average 115.7 ms (median 73 ms, p95 253 ms). Per В§5.2, 96.6% of that 115.7 ms is cudaStreamSynchronize вҖ” 24 sync calls per prefill step at 4.74 ms each. The actual GPU-side DMA accounts for only 8.9% of wallclock, running at 53.6 GB/s (В§5.3). The optimization target (В§5.4) is to coalesce 24 syncs to 1, yielding an estimated вҲ’95% reduction per prefill step.

Finding 3: Overhead is structural, not transfer-related

nocache (1GB CPU cache, almost no actual KV stored) is worse than default LMCache: JPS=3 +62% vs FCFS, JPS=10 +26%. Even when LMCache has nothing to transfer, its connector metadata management and per-step save/load hooks impose a significant tax.

Finding 4: stream_only makes things worse

Replacing torch.cuda.synchronize() with stream.synchronize() was uniformly worse across all JPS (+16% to +32%). The device-level sync may provide unintended ordering guarantees that prevent contention.

10. Future Directions

10.1 Optimize wait_for_save (highest impact)

The save path runs synchronously after every forward pass. Potential optimizations:

  • Lazy save вҖ” only save KV when the request is about to be evicted, not after every step
  • Async save pipeline вҖ” move save to a background thread/stream, don't block the main loop
  • Selective save вҖ” skip saving for short-lived requests unlikely to be reused
10.2 Reduce LMCache Framework Overhead

The nocache experiment shows structural overhead even without transfers. Investigate:

  • Eliminate per-step build_connector_meta cost when no loads/saves are needed
  • Skip save_kv_layer calls entirely when CPU cache is full/disabled
  • Profile Python-level overhead (dict lookups, tensor metadata, etc.)
10.3 VRAM-aware Tuning for Smaller GPUs

On L40S (48GB), LMCache's buffer allocations reduce batch capacity 3.3Г—. Potential fixes:

  • Dynamic buffer sizing based on available VRAM
  • Reduce gpu-memory-utilization for LMCache to leave headroom for KV cache slots
10.4 Clean L40S wait_for_save Baseline

The L40S nsys-bn run shared a node with the ctrl-sweep fill job (6428519), contaminating the wait_for_save latency measurements (though from_gpu DMA floor remained clean). A dedicated rerun on an isolated L40S node would provide a clean Gen4 wait_for_save baseline for comparison with the clean A100 Gen4 data.
