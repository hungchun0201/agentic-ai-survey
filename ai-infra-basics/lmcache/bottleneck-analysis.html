FCFS vs LMCache vs Continuum — Full Experiment Report

Three-way KV cache strategy comparison across 4 GPUs, per-turn latency analysis, cache starvation measurement, and a deep-dive bottleneck ablation identifying wait_for_save as LMCache's true overhead

Llama-3.1-8B-Instruct H200 / H100 / L40S Georgia Tech PACE wait_for_save: 224ms median / prefill step vllm-continuum

1. Experiment Context

1.1 Research Question

Multi-turn agent workloads (chain-of-thought, tool calling, ReAct-style loops) suffer a pathological behavior under standard vLLM: the KV cache for a job's context is computed during turn N, evicted during the tool-call gap between turn N and N+1, then recomputed from scratch on turn N+1. At 13K tokens/context this recompute alone dominates end-to-end latency. We study three KV cache management strategies for eliminating this recompute and quantify their tradeoffs under real load.

1.2 Three Strategies Under Test

FCFS (baseline)

Stock vLLM v1 scheduler. Block-level prefix cache is enabled, but evicted blocks are simply recomputed on the next turn. No explicit cross-turn KV preservation.

Failure mode: recompute of full context every turn under high concurrency

LMCache (v0.3.7)

Upstream production KV cache layer. Offloads evicted KV blocks to CPU DRAM via PCIe, reloads on hit. Integrates with vLLM through the v1 KV transfer connector interface (LMCacheConnectorV1).

Failure mode: wait_for_save blocks the step loop on every prefill — see Sections 5–13

Continuum

Research prototype (this work). Keeps KV blocks for active agent jobs pinned in GPU VRAM across tool-call gaps, bypassing the evict/recompute cycle entirely. No CPU transfer. Identifies jobs via job_id and releases pin on job_completed signal.

Failure mode: pinned blocks can starve new admissions (Turn 2 explosion) — see §3

1.3 Workload Design

The benchmark simulates an 8-turn agent job: the client sends a prompt, the server generates a short response containing a tool call, the client sleeps for a fixed tool-execution time, then appends a (large) tool output back onto the context and starts the next turn. Each turn grows the prompt by ~2K tokens, so turn 8 sees a prompt of ~13K tokens. Jobs arrive on a Poisson process at rate JPS (jobs per second).

1.4 Why Random Prompts Are Load-Bearing

A retracted earlier result

An earlier version of this benchmark (dir 20260406_static-prompt_jps-sweep_d90s_tool0.5s/) sent the same tool output string to every job. vLLM's block-level prefix cache — enabled by default — then hashed identical blocks across jobs, giving ~99% prefix cache hit and effectively 0 real KV eviction. Under that regime, LMCache appeared to beat FCFS by 39–66% on A100/L40S. That result is an artifact and is retracted. Under random prompts (31% prefix cache hit, realistic KV pressure), LMCache is slower than FCFS on every tested large-VRAM GPU.

The fix (committed in ab_benchmark.py:70) uses per-job per-turn seeded randomness:

def _generate_random_tool_output(job_id: str, turn_idx: int,
                                 target_tokens: int) -> str:
    # Uses a seeded RNG per (job_id, turn_idx) for reproducibility.
    seed = hashlib.md5(f"{job_id}:{turn_idx}".encode()).hexdigest()
    rng = random.Random(seed)
    header = f"[{job_id}:turn{turn_idx}:{seed[:8]}] "
    ...  # random words from a ~80-word pool

Each (job, turn) combination gets a unique token sequence. vLLM's prefix cache can still match the first few tokens of system prompt but cannot share tool-output bodies across jobs. All results in Sections 2–13 use this randomized prompt.

Key Takeaways

Continuum dominates at medium-to-high load (JPS ≥ 3)

On H200: −43% at JPS=3, −70% at JPS=6, −74% at JPS=10, −73% at JPS=15. On H100: −10% at JPS=3, −50% at JPS=6, −63% at JPS=10, −63% at JPS=15. On A100: −75% at JPS=3, −78% at JPS=6, −82% at JPS=10, −85% at JPS=15 (0 timeouts vs 525 FCFS / 615 LMCache). On L40S: −65% at JPS=3, −71% at JPS=6, −73% at JPS=10, −79% at JPS=15. The benefit grows with load and is amplified by Gen4 PCIe (A100/L40S), where LMCache's wait_for_save DMA floor is 2× higher, making the software-overhead fraction (§9.2.1 Finding 4) even more punishing.

LMCache is consistently worse than FCFS on large-VRAM GPUs

Every JPS × GPU cell shows LMCache at or worse than FCFS. Worst case: H100 JPS=3 with +158%. The mechanism is identified in §5–13: wait_for_save forces the step loop to block on a PCIe DMA every time a prefill request is in the batch, and under random prompts LMCache's CPU cache hit rate drops to 31% so the offload cost isn't being amortized by reuse.

At JPS=1 none of the policies help — FCFS wins

Below the saturation point there is no real KV pressure: a single job's blocks fit in VRAM, nothing gets evicted, prefix cache handles reuse for free. Adding LMCache adds wait_for_save overhead (+40% H200, +28% H100). Adding Continuum adds pin bookkeeping (+23% H200, +15% H100). Both are overhead without benefit at this load level.

3. Per-Turn Latency — Where the Time Goes

Average job completion time hides the mechanism. Breaking latency down by turn reveals three distinct failure modes — one per policy. All charts below use H200 (141 GB) controlled-sweep data.

3.1 Load regime: JPS=3 (below saturation)

At JPS=3 on H200 there's headroom. All three policies stay below 4s per turn. FCFS has the characteristic small bump from turn 2 onwards (each turn's recompute gets larger) but the effect is mild because KV pressure hasn't kicked in. LMCache is slightly better than FCFS in this regime — this is the only measured load level where LMCache actually helps on H200, and it's because the save pipeline is almost never triggered.

3.2 Load regime: JPS=6 (saturation threshold)

At JPS=6 the two pictures start to diverge. FCFS and LMCache climb monotonically because each turn's recompute cost grows with context, and under saturation the cost of that recompute is paid in full. Continuum flips the picture: turn 1 is slow (21s — new admission waits), turn 2 is slow (56s — first real prefill competes with other new jobs), but from turn 3 onwards the pinned KV means the job only decodes 20 new tokens. Turns 3–8 each take 5–14s regardless of context size.

3.3 Load regime: JPS=10 (high load)

The ceiling separates sharply. FCFS and LMCache hit ~88 s/turn at the peak (turn 7): at 13K-token context, that's essentially the prefill compute of the entire context every turn. The two lines are nearly identical — LMCache's DRAM offload is not actually saving any recompute in this regime, it's just paying extra wait_for_save on top. Meanwhile Continuum's burden is concentrated entirely in turn 2 (56s spike); turns 3–8 are in the 5–15s range.

3.4 Load regime: JPS=15 (fully saturated)

Continuum's Turn 2 explosion — and why FCFS/LMCache don't have one

At H200 JPS=15, Continuum's turn 2 alone takes 115.5 s while every subsequent turn takes only 5–15 s. FCFS and LMCache show no such spike — their per-turn cost climbs monotonically from T1≈21s to T7≈140s. This is not Continuum being "worse at T2" in some absolute sense; it is a structural difference in where each strategy pays its prefill cost.

Per-turn prefill work: three different shapes

FCFS and LMCache recompute the full context at every turn because the KV from the previous turn has been evicted during the 0.5s tool gap. Continuum pins the KV and only prefills the incremental tokens added since the last turn. The table below compares the prefill token count that each strategy actually feeds to the model:

At JPS=3 Continuum T2 is only 1.5s — pure compute with no queueing. So the 1,748-token prefill itself costs ~1.5s on H200. The 115.5s at JPS=15 means the other ~114 seconds are queue wait, not computation. Something in Continuum's scheduling is serializing T2 requests under high load.

Why the queue forms only at T2 — verified by scheduler trace

Continuum has an implicit admission to the pinned pool at T2. At T1 a new job only prefills 92 tokens and generates 20 — this occupies almost no VRAM, so the scheduler freely admits it. Between T1 and T2 the client appends the ~1,728-token tool output; now at T2 the job wants Continuum to pin ~1,800 tokens of KV for the remaining ~160 seconds of its lifetime. That is a commitment to hold a long-term VRAM slot. Under JPS=15 saturation, the pinned working set is near-full with jobs in T3–T8, and new T2 requests queue behind them waiting for slots to free up.

FCFS and LMCache don't have this gating because they don't have a pinned pool. Every request — whether it's a job's T1, T2, or T7 — is just a normal prefill request that joins the batch using standard vLLM fairness. Queueing still happens, but it's distributed uniformly across all turns based on batch-level contention, producing the monotonic T1→T8 climb we see in FCFS/LMCache. There is no single "admission gate" turn. The total wait time across the 8 turns is larger for FCFS (~710s) than Continuum's single T2 spike (~115s), but because it's spread out it looks less dramatic on the per-turn chart.

On H100 the same effect is sharper (T2 = 178s at JPS=15) because H100's 80 GB VRAM fits fewer concurrent pinned jobs than H200's 141 GB, so the pinned pool saturates earlier and the admission queue grows faster. The T2 spike is a pin-specific failure mode, not a generic "Continuum is slow at T2" claim. A potential mitigation is admission control that reserves VRAM slots for new arrivals in advance, or a shorter pin TTL — both are in future work (§12).

Finding 2 — T2 TTFT grows with arrival order, and Continuum grows 3.7× steeper

I binned the 661 jobs of each policy by arrival order (first 50, 100–200, 300–400, last 50) and computed the mean T2 TTFT per bin. For the first 50 jobs — arriving into an empty system — all three policies deliver T2 TTFT in ~1 s. As arrival order progresses, the queue builds. FCFS and LMCache rise linearly to ~70 s for the last 50 jobs. Continuum rises to 269 s — 3.7× steeper.

This is direct evidence that Continuum's T2 queue is a different queue from the FCFS/LMCache T2 queue. Under FCFS/LMCache, a T2 request competes with other batch-level prefill requests; under Continuum it additionally competes for a slot in the pinned working set, which is a scarcer resource. The first few jobs get slots for free. Late arrivals wait for existing pinned jobs to finish and release their slots, which takes the ~190s of Continuum's steady-state JCT — and the last 50 jobs' 269s T2 TTFT is approximately that.

Finding 3 — Continuum jobs are "in the club" after T2: arrival order no longer matters

This is the cleanest confirmation. At T7, FCFS's TTFT grows from 32s (first 50) to 151s (jobs 300-400), matching the arrival-order pattern seen at T2. FCFS's late-arriving jobs pay a penalty on every turn. Continuum's T7 TTFT, in contrast, stays at 0.5–1.4s regardless of arrival order — the first 50 jobs and the last 50 jobs have indistinguishable T7 performance. Once a Continuum job has crossed the T2 admission gate, it is decoupled from admission-queue pressure for the rest of its lifetime. The pin guarantees fast access to its pre-prefilled KV, and the incremental per-turn prefill work is tiny.

Note on sample sizes: FCFS T7 last-50 bin has too few completed jobs to report (many jobs in the tail don't reach T7 within the benchmark + post-benchmark window under FCFS's slow service rate). LMCache T7 has the same truncation. Continuum's T7 bin for the last 50 is fully populated because all 661 Continuum jobs finished all 8 turns. That truncation asymmetry understates FCFS/LMCache's per-turn pain at the tail — their true T7 numbers for late jobs would be worse than 151s.

What the trace data settles

(1) Continuum's T2 latency is dominated by queue wait, not compute — trace TTFT at T2 is 116s p50 while pure compute for a 1,748-token prefill is ~1.5s. (2) The queue is specific to T2 (0.4–0.5s at T3–T8). (3) The queue grows 3.7× steeper with arrival order than FCFS/LMCache's T2 queue, which rules out "Continuum just has ordinary queue wait at every turn". (4) The shape of Continuum's "fast after T2" behavior is exactly what a pin-based admission gate would produce: zero cost to cross the gate, then cheap service on the inside. The working hypothesis in the previous paragraph is now empirically confirmed, not a guess.

3.5 Three distinct failure modes

FCFS: monotonic recompute wall

Each turn's latency is proportional to context size because each turn recomputes the full KV. The line grows with turn number in a shape that tracks the prompt-length column in §1.3.

LMCache: FCFS shape + wait_for_save tax

LMCache's per-turn line is the same shape as FCFS, slightly elevated. The DRAM offload fires but the 31% cache hit rate doesn't deliver enough reuse to pay for wait_for_save. Net: worse than FCFS at every turn ≥ 2.

Continuum: front-loaded T1/T2 stall

All the cost is concentrated in turn 1 (admission wait) and turn 2 (first real prefill competes for compute). Turns 3–8 are flat near 5–15s because the pinned KV means decode-only work from then on. Total JCT = T1 + T2 + 6 × decode ≈ T2 dominates.

The Continuum fairness tradeoff is real and unavoidable

Continuum's starvation ratio is consistently worse than FCFS/LMCache (4.7–6.2× vs 2.6–3.7×) across both GPUs and both load levels. This is a direct consequence of the pin mechanism: once an early job gets in and its KV is pinned, it consumes blocks that new jobs cannot touch. Late-arriving jobs face a more crowded KV budget than early-arriving ones, so their turn-2 prefill wait grows. The 5.3× ratio at H200 JPS=15 means the 80–100% quintile waits 5.3× longer than the 0–20% quintile — but even the Q5 late jobs under Continuum (320s) are still faster than the Q1 earliest jobs under FCFS or LMCache (292s–387s). Continuum trades worse fairness for much better absolute performance.

Why FCFS/LMCache ratios are lower

FCFS and LMCache distribute the pain more evenly: every job eats full recompute every turn, so there's no "in-club" advantage. Early and late jobs face roughly the same per-turn cost, just compounded by different amounts of queue wait. That produces a gentler 3× ratio. It's more fair, but in the worst sense of the word — everybody is equally slow.

7.2 Sync Ablation (4 Modes)

The initial hypothesis (§6) was that torch.cuda.synchronize() in LMCache's KV-onload path blocks the CPU scheduler thread and starves the GPU. To falsify or confirm this, we need to replace the synchronization primitive itself with progressively looser variants and see whether wall-clock latency improves. If it does, the sync primitive was the bottleneck; if it doesn't, the bottleneck is somewhere else. The tool is our_experiments/scripts/patch_sync.py, which runtime-patches gpu_connector.py inside the LMCache installation in the conda env (so vLLM picks up the change on server restart).

The target line in original LMCache (gpu_connector.py) is at the end of batched_to_gpu, after firing the async CPU→GPU KV load on a dedicated load_stream:

# NOTE: wait until both compute and load streams are done
torch.cuda.synchronize()   # ← the line under test

Each mode replaces exactly this call with a different primitive (the rest of LMCache code is untouched). All four modes are tested head-to-head against the same FCFS/Continuum baselines under identical JPS and workload. Here is what each mode does and what it is supposed to test:

D default

What it does: No change — runs LMCache's original torch.cuda.synchronize(). This is a device-wide block: the CPU thread waits until every kernel on every CUDA stream on the current device has completed, not just the load stream.

What it tests: baseline — the "worst-case" sync strategy that blocks on everything.

S stream_only

What it does: replaces torch.cuda.synchronize() with explicit per-stream syncs on only the streams this operation cares about:

self.load_stream.synchronize()
self.compute_stream.synchronize()

What it tests: whether the "device-wide" scope of default sync is over-conservative. If stream-scoped syncs are meaningfully faster, then default was wasting time waiting for unrelated kernels on other streams.

E event_async

What it does: replaces the device sync with a CUDA event that records a specific point on the current stream, then waits for just that point:

self._sync_event = torch.cuda.Event()
self._sync_event.record()
self._sync_event.synchronize()

What it tests: events are the cheapest fine-grained sync primitive CUDA offers — only one condition variable to check, no scan over all device work. If the bottleneck is CPU overhead inside the sync primitive itself, event_async should show it.

Caveat: record() without a stream argument records on the current stream (typically the default compute stream), so this event may not actually capture load_stream activity. If that's true, event_async is effectively incorrect — which is part of what the experiment reveals.

N no_sync ⚠ UNSAFE

What it does: removes all sync calls in the onload path — both the torch.cuda.synchronize() and the subsequent self.load_stream.synchronize():

pass  # sync removed for ceiling measurement

What it tests: the ceiling — how fast could LMCache possibly run if every synchronization primitive in the onload path were free? This is not a correct implementation (the next kernel could read paged KV blocks before the DMA finishes), but as long as it doesn't crash and the outputs are approximately right, it gives an upper bound on "best case if sync cost were zero".

Logical progression of the four modes

The four modes are ordered from most conservative to least: default blocks on the whole device, stream_only blocks only on the two involved streams, event_async blocks only on a single event, and no_sync blocks on nothing. If the sync-hypothesis is correct, each step should give a progressively larger speedup; if it's wrong, the curve will be flat. The no_sync ceiling is especially load-bearing — if even the free-sync case doesn't beat default by more than a few percent, the bottleneck is demonstrably not in any sync primitive. Spoiler: Finding 1 in §10 shows exactly this flat curve.

8.2 Scheduler Step Analysis (H200)

What is a scheduler "step"?

A step in vLLM is one full iteration of the engine loop: the scheduler picks a batch of requests, the GPU runs one model.forward() on that batch (which produces exactly one output token per request in the batch), the sampler picks the token, and the loop returns to the scheduler for the next iteration. A request with an 8,000-token prompt and 20 completion tokens will go through at least 20 steps: the first few steps chunk-prefill its prompt, the subsequent steps each generate one decode token. Every metric in the table below counts per-step averages across the 28-run benchmark (4 JPS values × 7 policies).

The load path's cumulative wall-clock cost is actually larger than the save path (1,894 ms vs 325 ms) even though it moves 11× less data. Answering the original concern

So to the original question — "is wait_for_save really just two CUDA kernels?" — the answer is: no, it's far more. A single entry into wait_for_save triggers ~6 Python-layer function calls, one custom CUDA kernel launch, dozens of cudaMemcpyAsync issuings (one per chunk × batched request × direction), 1 cudaStreamSynchronize that may block for anywhere from 0 μs (decode-only step, nothing to save) to ~224 ms (prefill step with a full 13K-token ~1.6 GB payload), and a storage_manager hash-table insertion per chunk. The "1 LMCache-specific kernel" count is an accurate statement about the GPU side but dramatically understates the complexity. The hot path is dominated by memcpy issue + stream synchronize + per-chunk Python bookkeeping, not by kernel execution — which is exactly why the sync-ablation experiments in §8.1 (replacing torch.cuda.synchronize with event-based waits, per-stream sync, or no sync at all) made essentially no difference: they were optimizing the wrong part of the stack. §9.2 further decomposes the 224 ms into ~30 ms of pure PCIe DMA and ~194 ms of framework overhead.

8.4 L40S Results (48GB VRAM) — where the mechanism differs

L40S shows a much more severe LMCache penalty than H200 — average batch size drops from 26.7 (FCFS) to 8.0 (LMCache default), a 3.3× reduction. On H200 the same comparison is 16.6 vs 17.6, basically unchanged. The obvious-sounding explanation — "L40S has less VRAM so LMCache's buffers eat into KV budget" — is wrong. The actual mechanism: step-rate throttling from wait_for_save + chunked prefill

Two configuration facts combine to produce the L40S penalty:

  1. L40S's max_num_batched_tokens = 2048, auto-tuned by vLLM based on available VRAM. H200's is 8,192 — 4× larger. This means each step on L40S can process at most 2048 tokens total (prefill chunks + decode tokens combined). A 13K-token prompt under chunked prefill therefore needs at least 7 prefill chunks (13,193 / 2,048 ≈ 6.4) to finish its prefill — 7 separate steps just for one request's prefill.
  2. LMCache adds wait_for_save = 187 ms per prefill step on L40S (vs 133 ms on H200). Every prefill chunk on L40S is followed by a 187 ms stall while the scheduler waits for the GPU→CPU offload DMA. FCFS has no such stall.

Multiply these together. Under LMCache on L40S, one 13K-token prefill takes 7 chunks × 187 ms ≈ 1.3 s of pure wait_for_save idle, on top of the actual prefill compute. Under FCFS on L40S, the same prompt takes 7 chunks × 0 ms = 0 extra idle. The engine-loop cycle time is dominated by the 187 ms stall, so L40S LMCache can run roughly 5 steps/sec (1000 ms / 187 ms), while L40S FCFS runs ~20+ steps/sec. Fewer steps per second means fewer opportunities for the scheduler to promote waiting requests into the running state — and that is exactly what the telemetry shows: running count stuck at 6–16 for LMCache, but 23–48 for FCFS.

Why H200 escapes this penalty

On H200 the math flips in LMCache's favor: max_num_batched_tokens = 8,192 means a 13K-token prompt finishes in only 2 chunks (13,193 / 8,192 ≈ 1.6). Only 2 prefill chunks → only 2 wait_for_save stalls per job → ~266 ms total stall vs L40S's ~1.3 s. And H200's base compute is faster, so the 133 ms stall is a smaller fraction of each step's wall-clock time. Result: step rate barely drops, running count barely drops, and average batch size stays near FCFS (17.6 vs 16.6). The L40S penalty is not about VRAM pressure — it is about max_num_batched_tokens being too small for the prompt length, forcing many prefill chunks and therefore many wait_for_save stalls.

Testable prediction

This mechanism makes a falsifiable prediction: if you manually set --max-num-batched-tokens=8192 on L40S (overriding the auto-tuner), the LMCache batch-size penalty should shrink dramatically, because each 13K-token prompt then finishes in 2 chunks like H200. This is cheap to verify and would be a clean single-variable experiment. Pending re-run.

  1. cudaStreamSynchronize alone is 96.6% of prefill-step wait_for_save wallclock. Mechanistic explanation: why 23.5 syncs per step?

A 13K-token request produces ~51 KV chunks × 128 KiB/chunk in LMCache's default layout. For each prefill step, the LMCacheEngine.store path queues one cudaMemcpyAsync per chunk and then calls cudaStreamSynchronize periodically in the save loop to reclaim source tensor buffers before the next chunk can be written. At JPS=1 this averages ~19 memcpy launches plus ~24 sync calls per prefill step. Each sync pays a fixed ~3-4 ms floor (kernel driver wake-up, Python-C FFI round-trip, GIL re-acquire) even when the actual DMA behind it has already completed on the GPU. Stacked across 24 calls per step, that fixed cost is the dominant wallclock: ~80 ms out of ~115 ms is sync overhead that would simply not exist if the implementation batched all chunks into a single stream and synchronized once per step.

This is a sharper, more directly-optimizable finding than the 'software tax' framing from §9.2. The old analysis said "the remaining ~194 ms is some combination of Python bookkeeping, stream serialization, and CUDA runtime sync." The new CUDA Runtime data says: it's almost entirely one API, cudaStreamSynchronize, called too often. A concrete optimization would be:

  • Coalesce all chunk-level cudaMemcpyAsync calls for a given prefill step into a single stream, queue them back-to-back, and issue exactly one cudaStreamSynchronize at the end of the step (instead of ~24). Conservative estimate: drops per-prefill-step wfs from ~115 ms to ~10-15 ms (actual DMA + one sync call + launch overhead). That's ~100 ms saved per prefill step.
  • Extrapolated to a 45s benchmark at jps=1: 159 prefill steps × 100 ms saved = ~16 s recovered out of 18.4 s total wait_for_save time. Extrapolated to jps=10 (where prefill fraction is ~88% not ~9%): the savings would be proportionally larger.
  • Even without coalescing, a cheaper fix: use cudaEventRecord + cudaEventSynchronize with a single event per step, rather than cudaStreamSynchronize which synchronizes the entire stream (including unrelated kernels). Events are lighter-weight.

Caveat: the 3-4 ms per-sync-call fixed cost is itself an empirical observation on this particular PACE H200 node and may vary with kernel version, driver version, and load. The qualitative story (sync-call count is the dominant cost, not DMA) should hold universally; the exact per-sync cost may not. The prefill-only / decode-only split and the 96.6% attribution are direct measurements and are not sensitive to the per-sync assumption.

9.5 Why the blocking wait exists — upstream rationale

A natural question after reading §9.1–§9.4 is: why does vLLM force this blocking wait at every step boundary instead of letting the save continue asynchronously across steps? If the ~224ms DMA is genuinely async, running it concurrently with the next forward pass would hide most of its cost. The fact that vLLM chose to block deserves an explanation. I searched the vLLM and LMCache GitHub repos, RFCs, PRs, and the vLLM blog. What I found — and did not find — is reported here.

The one upstream sentence on the subject

The only authoritative upstream text explaining why wait_for_save blocks is a single-sentence docstring in the v1 connector base class:

"Block until all the save operations is done. This is called as the forward context exits to ensure that the async saving from save_kv_layer is complete before finishing the forward. This prevents overwrites of paged KV buffer before saving done."

Source: vllm/distributed/kv_transfer/kv_connector/v1/base.py — KVConnectorBase_V1.wait_for_save docstring

That sentence is the entire upstream design discussion. There is no GitHub issue debating the trade-off, no PR proposing a non-blocking alternative, no blog post elaborating on the constraint, and no design doc. The wait-for-save barrier is treated as a self-evident safety requirement rather than an explicit trade-off.

What the docstring actually means: the paged block reuse hazard

Unpacking "prevents overwrites of paged KV buffer" from first principles: vLLM's paged attention allocates KV cache in fixed-size blocks (2 MiB, 16 tokens on Llama 3.1 8B bf16). The scheduler's block allocator treats a block as free and available for reuse the moment the request holding it finishes or is evicted. In the normal vLLM design, blocks cycle rapidly through free→allocated→written→read→free. The next step can and routinely does assign a just-freed block to a new request's prefill. Without wait_for_save, the following race is possible:

Step N — request A

Prefill writes KV for request A into block #47. At step N's forward-context exit, save_kv_layer() fires an async GPU→CPU DMA reading from block #47, queued on store_stream.

↓

(no wait_for_save)

vLLM's engine loop proceeds immediately to step N+1. The store_stream DMA is still running in the background, reading block #47 byte by byte.

↓

Step N+1 — request B hijacks block #47

Scheduler's block allocator sees #47 as free (because request A's lifetime ended) and allocates it to request B's prefill. B's attention kernel writes new KV to block #47. At the same moment the DMA from step N is still reading — result: CPU-side cache receives a mix of A's pre-overwrite and B's post-overwrite bytes. The cache is corrupted silently.

The blocking wait_for_save forbids this by freezing the engine loop until every in-flight save has finished reading every dirty block. Only then can the block allocator's free pool be trusted for reuse. It is a correctness barrier for the block allocator, not a performance choice — there is no way to skip it without changing the semantics of the free pool.

10. Key Findings

Finding 1: The sync hypothesis is FALSIFIED

Removing torch.cuda.synchronize() entirely (no_sync mode) produced minimal improvement: JPS=10 only 2% better, JPS=3 actually 51% worse. All sync variants (stream, event, nosync) performed equal or worse than default.

Finding 2: wait_for_save is the dominant overhead

The distribution of wait_for_save() is bimodal, not a smooth average. On decode-only steps (43.6% of all steps on H200) it returns in <0.1ms because there is nothing to save. On steps containing at least one prefill (52.7%) it blocks for a median of ~224ms. The payload is ~1.6 GB per 13K-token request (GQA: 8 KV heads × 128 head_dim × 32 layers × 2 (K+V) × 2 bytes × 13,000 tokens), which on PCIe Gen5 runs in ~30 ms of pure DMA — the other ~194 ms is framework overhead (§9.2). The widely-quoted 133.7ms mean on H200 (and 187.4ms on L40S) is the average across all steps; it is diluted by the 43.6% zero-cost decode steps. The actual per-prefill-step cost that blocks the entire batch is ~224ms, of which ~13% is hardware-bound and ~87% is optimizable framework overhead.

Finding 3: Overhead is structural, not transfer-related

nocache (1GB CPU cache, almost no actual KV stored) is worse than default LMCache: JPS=3 +62% vs FCFS, JPS=10 +26%. Even when LMCache has nothing to transfer, its connector metadata management and per-step save/load hooks impose a significant tax.

Finding 4: stream_only makes things worse

Replacing torch.cuda.synchronize() with stream.synchronize() was uniformly worse across all JPS (+16% to +32%). The device-level sync may provide unintended ordering guarantees that prevent contention.

12. Future Directions

12.1 Optimize wait_for_save (highest impact)

The save path runs synchronously after every forward pass. Potential optimizations:

  • Lazy save — only save KV when the request is about to be evicted, not after every step
  • Async save pipeline — move save to a background thread/stream, don't block the main loop
  • Selective save — skip saving for short-lived requests unlikely to be reused
12.2 Reduce LMCache Framework Overhead

The nocache experiment shows structural overhead even without transfers. Investigate:

  • Eliminate per-step build_connector_meta cost when no loads/saves are needed
  • Skip save_kv_layer calls entirely when CPU cache is full/disabled
  • Profile Python-level overhead (dict lookups, tensor metadata, etc.)
12.3 VRAM-aware Tuning for Smaller GPUs

On L40S (48GB), LMCache's buffer allocations reduce batch capacity 3.3×. Potential fixes:

  • Dynamic buffer sizing based on available VRAM
  • Reduce gpu-memory-utilization for LMCache to leave headroom for KV cache slots
12.4 Clean L40S wait_for_save Baseline

The L40S nsys-bn run shared a node with the ctrl-sweep fill job (6428519), contaminating the wait_for_save latency measurements (though from_gpu DMA floor remained clean). A dedicated rerun on an isolated L40S node would provide a clean Gen4 wait_for_save baseline for comparison with the clean A100 Gen4 data.
