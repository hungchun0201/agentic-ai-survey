LMCache Bottleneck Analysis

Why LMCache is slower than FCFS on large-VRAM GPUs — a systematic ablation study falsifying the sync hypothesis and identifying wait_for_save as the true bottleneck

Llama-3.1-8B-Instruct H200 / H100 / L40S Georgia Tech PACE wait_for_save: 130-190ms/step vllm-continuum

1 Motivation

Prior controlled-sweep experiments established two key findings about multi-turn agent serving with the vllm-continuum system:

Continuum Works

Reduces multi-turn agent latency by 50–74% vs FCFS at medium-to-high concurrency, by pinning KV cache in VRAM across turns.

LMCache is Slower

CPU offload/restore of KV is consistently slower than FCFS on large-VRAM GPUs (H100/H200), despite its design goal of saving recompute by restoring KV from CPU memory.

The Puzzle

LMCache uses 65% less GPU compute and 45% fewer kernel launches than FCFS — the GPU is doing less work, yet wall-clock time is longer. This demanded a root-cause analysis.

2 Initial Hypothesis

H0: CPU-side scheduling bottleneck from torch.cuda.synchronize()

Hypothesized Mechanism

  1. torch.cuda.synchronize() in LMCache's onload path (CPU→GPU KV transfer) blocks the CPU scheduler thread once per layer (32 calls per request)
  2. While the CPU is blocked, no new work can be submitted to the GPU
  3. This creates a pipeline bubble: GPU sits idle → smaller effective batch sizes → lower throughput
↓

Evidence FOR H0

  • GPU utilization: 25.9% vs 47.5%
  • Fewer kernel launches (94K vs 172K)
  • Median kernels/step: 40 vs 99

Evidence AGAINST H0

  • cudaDeviceSync averaged only 0.041ms (no-ops)
  • FCFS spent MORE time in sync (11.08s vs 5.05s)
  • Large gaps (>10ms) did NOT correlate with sync events

VRAM-dependent Behavior

On H200 (141GB), batch sizes are comparable (16.6 vs 17.6). On L40S (48GB), LMCache reduces batch size by 3.3×. LMCache's buffer allocations compete with model weights and KV cache slots for memory, creating a different bottleneck on smaller GPUs.

5 Key Findings

Finding 1: The sync hypothesis is FALSIFIED

Removing torch.cuda.synchronize() entirely (no_sync mode) produced minimal improvement: JPS=10 only 2% better, JPS=3 actually 51% worse. All sync variants (stream, event, nosync) performed equal or worse than default.

Finding 2: wait_for_save is the dominant overhead

Every LMCache step spends 127–150ms in wait_for_save(), the GPU→CPU offload sync after each forward pass. On H200: 133.7ms mean. On L40S: 187.4ms mean. Over thousands of steps, this accumulates to minutes of wall-clock overhead.

Finding 3: Overhead is structural, not transfer-related

nocache (1GB CPU cache, almost no actual KV stored) is worse than default LMCache: JPS=3 +62% vs FCFS, JPS=10 +26%. Even when LMCache has nothing to transfer, its connector metadata management and per-step save/load hooks impose a significant tax.

Finding 4: stream_only makes things worse

Replacing torch.cuda.synchronize() with stream.synchronize() was uniformly worse across all JPS (+16% to +32%). The device-level sync may provide unintended ordering guarantees that prevent contention.

6 Revised Bottleneck Model

Original Hypothesis (FALSIFIED)

❌ torch.cuda.synchronize() in onload path

Blocks CPU scheduler → GPU idle → small batches → slow

↓ FALSIFIED

Actual Bottleneck (CONFIRMED)

① wait_for_save() — 130–190ms/step

GPU→CPU offload sync runs after every forward pass. The largest single contributor. Runs even when nocache — the save pipeline is unconditional.

↓

② Connector Metadata Overhead

Per-step bookkeeping: build_connector_meta, get_num_new_matched_tokens, update_state_after_alloc

↓

③ Memory Management

LMCache's CPU buffer allocation and reference tracking. On smaller GPUs (L40S), buffer allocations compete with KV cache slots → 3.3× batch reduction.

7 Future Directions

7.1 Optimize wait_for_save (highest impact)

The save path runs synchronously after every forward pass. Potential optimizations:

  • Lazy save — only save KV when the request is about to be evicted, not after every step
  • Async save pipeline — move save to a background thread/stream, don't block the main loop
  • Selective save — skip saving for short-lived requests unlikely to be reused
7.2 Reduce LMCache Framework Overhead

The nocache experiment shows structural overhead even without transfers. Investigate:

  • Eliminate per-step build_connector_meta cost when no loads/saves are needed
  • Skip save_kv_layer calls entirely when CPU cache is full/disabled
  • Profile Python-level overhead (dict lookups, tensor metadata, etc.)
7.3 VRAM-aware Tuning for Smaller GPUs

On L40S (48GB), LMCache's buffer allocations reduce batch capacity 3.3×. Potential fixes:

  • Dynamic buffer sizing based on available VRAM
  • Reduce gpu-memory-utilization for LMCache to leave headroom for KV cache slots
7.4 Complete H100 and A100 Experiments

H100 is in progress (10/28 complete). A100 is pending. These will confirm whether the H200 findings generalize across GPU generations.
