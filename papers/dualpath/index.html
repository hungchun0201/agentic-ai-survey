Section 1

Why Agentic Workloads Shift the Bottleneck

As LLMs evolve from single-turn chatbots into agentic systems with long-running, multi-turn interactions, the dominant performance bottleneck shifts from compute to storage I/O bandwidth.

The Agentic Paradigm

Unlike traditional chatbots, agentic LLMs interact with external environments through tools, web browsers, and code interpreters over dozens or even hundreds of turns. Each individual turn appends only a short piece of new context (mean append length ~429 tokens), but the accumulated context grows to extreme lengths (mean context length ~32.7K tokens). This creates a workload that is fundamentally I/O-bound rather than compute-bound.

Massive Context Reuse

With typically ≥95% of tokens in each round being reused from prior context, the vast majority of the KV-Cache can be loaded from storage rather than recomputed. The short-append, multi-turn pattern leads to very high KV-Cache hit rates, meaning only newly appended tokens need prefill computation. This makes KV-Cache loading speed -- not GPU compute -- the dominant performance factor.

External Storage Necessity

Agent trajectories can reach up to one million tokens. DRAM and HBM-based KV-Cache stores (like Mooncake) can only hold a small proportion of these massive caches, necessitating the use of larger, cheaper external SSD-based KV-Cache storage such as DeepSeek's 3FS distributed filesystem.

Hardware Trends Worsen the Gap

From NVIDIA Ampere to Blackwell, GPU compute has scaled 28.8x, PCIe bandwidth only 2.0x, and GPU memory only 2.4x. The I/O-to-compute ratio decreases by 14.4x, making the storage bandwidth bottleneck increasingly severe across hardware generations.

DualPath transforms storage I/O from a single-bottleneck resource into a globally pooled, schedulable capacity by aggregating SNICs from all engines.

Inference Engines

Each engine manages one GPU, categorized into Prefill Engines (PEs) for prompt processing and Decode Engines (DEs) for token generation. PEs adopt layerwise prefill (from LayerKV/PrefillOnly), where each layer loads only its own KV-Cache, allocates and frees per layer, boosting effective batch size by a factor equal to the number of layers.

Traffic Manager

A per-engine component that conducts host-device memory copies (H2D/D2H), KV-Cache transfers between PEs and DEs, and storage reads/writes via the SNIC. Adopts a CNIC-centric traffic management approach to isolate KV-Cache traffic from latency-sensitive model inference collective communications.

Request Scheduler

A central scheduler that receives client requests and distributes them across engines. It is responsible for dynamically distributing data traffic between the two paths based on real-time load metrics including storage NIC queue lengths, GPU compute load, and token counts.

Why This Works Without Congestion The compute network (8x 400Gbps CNICs per node) has much larger aggregate bandwidth than the storage network (1x 400Gbps SNIC per node). Compute network traffic from model execution (AllToAll, ReduceScatter, AllGather) occurs in intermittent, sub-millisecond bursts, leaving substantial idle bandwidth. DualPath routes KV-Cache transfers as low-priority traffic on these idle windows via Virtual Lane (VL) isolation in InfiniBand, ensuring zero interference with latency-critical collective communications.

All data traffic in and out of a GPU is routed through the CNIC, which acts as the central QoS scheduler. Weighted Round Robin (WRR) at VL arbiters reserves ~99% of bandwidth for high-priority inference traffic, with KV-Cache transfers using only spare bandwidth windows.

Why Not GPUDirect Storage?

GPUDirect Storage directly reads KV-Cache to GPU HBM, but it cannot isolate KV-Cache traffic from latency-sensitive collective communications. Both contend for PCIe bandwidth without QoS controls. Similarly, CUDA copy engine (host DRAM to GPU HBM) also interferes with model execution. DualPath's CNIC-centric approach is currently the only practical method to ensure KV-Cache I/O does not degrade model execution performance.

CNIC vs. CUDA Copy Performance

Surprisingly, CNIC-assisted H2D/D2H outperforms the CUDA copy engine when handling many small data chunks (as in layerwise prefill). A single cudaMemcpyAsync incurs 5-7 microseconds latency overhead, while submitting an RDMA Write request via CNIC takes only ~1 microsecond, and can be further amortized using doorbell batching.

Section 8

Challenge to the DistServe/Splitwise Framing

DualPath fundamentally challenges the compute-centric optimization focus that has dominated the PD-disaggregation era, revealing that agentic workloads demand a radically different systems perspective.

DistServe/Splitwise Era Assumptions

  • Prefill is compute-intensive: GPUs are the bottleneck during prompt processing
  • Decode is memory-bound: HBM bandwidth limits token generation
  • Optimization focus: better GPU utilization through workload isolation
  • KV-Cache transfer between PE and DE is a secondary concern
  • Network is assumed adequate for KV-Cache transfers
  • Context length is moderate, KV-Cache hit rates are low

DualPath's Agentic Reality

  • Prefill becomes I/O-bound: storage NIC is the bottleneck, not GPU compute
  • Decode engines have idle storage bandwidth that can be harvested
  • Optimization focus: network bandwidth balancing across engine types
  • KV-Cache loading from storage is the dominant performance factor
  • Storage NIC bandwidth is the critical scarce resource
  • Context lengths are extreme (up to 1M tokens), hit rates ≥95%
💡
The Paradigm Shift DistServe (OSDI 2024) and Splitwise (ISCA 2025) optimized for a world where GPUs were the bottleneck. DualPath reveals that under agentic workloads, the bottleneck has moved to the storage network. The paper provides the first systematic evidence that storage NIC bandwidth, not GPU compute or HBM capacity, is the binding constraint in production agentic inference. This shift demands rethinking PD-disaggregation from a network-centric perspective rather than a compute-centric one.

Evidence: P/D Ratio Experiment

The paper's P/D ratio experiment provides the clearest evidence that storage bandwidth, not compute, is the binding constraint:

  • Basic 1P1D and Basic 2P1D perform comparably -- doubling prefill compute does not help because the bottleneck is not computation.
  • DualPath 1P1D and DualPath 2P1D also perform comparably -- each pair has equivalent available storage bandwidth (since DualPath uses all nodes' SNICs).
  • This confirms that total available storage bandwidth, not the P/D allocation of compute, determines throughput.
  • DualPath achieves an average speedup of 1.64x (up to 2.46x) across all P/D configurations tested on DS 27B.
Section 9

Significance for Agentic Infrastructure

DualPath is the most directly relevant disaggregation paper for agentic LLM inference, offering practical, deployable solutions for the emerging I/O-bound regime that production systems face today.

Production Validation

Evaluated on DeepSeek's actual production traces from agentic RL training workloads, not synthetic benchmarks. The traces come from real coding agent tasks with 157 average turns and contexts up to 64K tokens, making results directly applicable to production deployments.

Production Traces RL Training Coding Agents

Minimal Implementation Overhead

DualPath requires only approximately 5,000 lines of code changes on top of an existing modern inference framework. It uses standard infrastructure components: InfiniBand VL isolation, RDMA, and a centralized scheduler with sub-10-core CPU overhead. No specialized hardware modifications are needed.

~5K LoC Standard HW

DeepSeek-AI Collaboration

Joint work with DeepSeek-AI, using their 3FS distributed filesystem and internal inference framework (FlashMLA, DeepGEMM, DeepEP). Results are validated against SGLang with Mooncake as a baseline, providing credible comparison with state-of-the-art open-source systems.

3FS FlashMLA DeepGEMM

Applicability to RL Training

The agentic inference workload is also prevalent in reinforcement learning training, where the rollout phase generates multi-step agent trajectories. During rollout, substantial data occupies DRAM for reward models and optimizer states, further constraining available DRAM for KV-Cache and reinforcing the need for external storage.

Future Directions and Limitations

Open Research Questions

  • Split reading from both PE and DE sides simultaneously for a single request
  • More adaptive P/D ratio configuration using online simulators
  • Lower TTFT percentiles under large-scale deployment
  • Reducing TPOT overhead from DE buffer management in small models
  • Combining DualPath with distributed DRAM caches (marginal gains observed)

Network Technology Extensions

  • RoCE support via DSCP-based Traffic Class and Priority Flow Control
  • UnifiedBus (emerging 2026 interconnect)
  • Ultra Ethernet v1.0.2 QoS mechanisms
  • Any fabric supporting heterogeneous traffic classification
