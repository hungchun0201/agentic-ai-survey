★ Overview

FlashInfer is a foundational attention engine designed to accelerate attention computation in large language model (LLM) inference serving. Unlike previous approaches that optimize individual attention variants in isolation, FlashInfer provides a unified framework that simultaneously addresses three fundamental challenges: heterogeneous KV-cache storage patterns, diverse attention mechanism variants, and dynamic input workload distributions. The system has been integrated into major production LLM serving frameworks including SGLang, vLLM, MLC-Engine, and TensorRT-LLM.

Unified data structure using Block Sparse Row (BSR) format with arbitrary block sizes (B_r, B_c). Unifies PageAttention, RadixAttention, sparse masks, and tree attention under a single abstraction.

CUDA/CUTLASS attention templates with a customizable variant specification interface. Supports QueryTransform, KeyTransform, LogitsTransform, LogitsMask, and OutputTransform functors via JIT compilation.

Dynamic runtime scheduler distributing work evenly across GPU SMs. Inspired by Stream-K, uses a priority-queue-based algorithm that maintains CUDAGraph compatibility through deterministic execution.

Composable Formats

Multiple block-sparse matrices with different block sizes enable shared-prefix decomposition, optimizing memory reuse for scenarios like prefix caching and parallel generation.

Key Implementation Modules

include/flashinfer/attention/

23+ header files containing the core CUDA kernel templates. Framework-agnostic -- uses raw pointers only. Supports Turing (SM75) through Blackwell (SM100+). Includes FA2-style kernels for Turing/Ampere/Ada and FA3-style kernels for Hopper.

flashinfer/jit/core.py

Defines JitSpec, the core compilation metadata abstraction. Manages two-level caching: Python @functools.cache for loaded modules and file-level ~/.cache/flashinfer/ for compiled .so files. Automatic invalidation via SHA256 hash.

flashinfer/jit/attention/modules.py

Module generators producing JitSpec for each attention variant. Computes unique URI from parameters (dtype, head_dim, encoding mode, etc.), renders Jinja templates, copies source files, and returns a buildable spec.

include/flashinfer/attention/scheduler.cuh

C++ implementation of the load-balanced scheduling algorithm. Uses binary search to compute optimal KV chunk sizes and partition pages across CTAs, maintaining CUDAGraph compatibility with fixed grid sizes.

include/flashinfer/attention/state.cuh

Implements the attention state tuple (o, m, d) with composable merge operations. The state_t struct provides merge() and normalize() operations used for online softmax and attention composition across KV chunks.

include/flashinfer/attention/variant_helper.cuh

Macro framework for defining attention variant functors. Provides REGISTER_LOGITS_TRANSFORM, REGISTER_LOGITS_MASK, REGISTER_QUERY_TRANSFORM, REGISTER_KEY_TRANSFORM, and REGISTER_OUTPUT_TRANSFORM -- all compiled into the kernel at JIT time.

Critical Design Rule: include/ headers must never include Torch headers, maintaining strict framework separation. The csrc/ layer bridges frameworks via TVM-FFI (Apache TVM's Foreign Function Interface), enabling bindings for PyTorch, C++, and other frameworks through a cross-language unified ABI.

▦ Block-Sparse KV-Cache Format

FlashInfer introduces a unified block-sparse representation that treats all KV-cache management strategies -- PageAttention, RadixAttention, tree attention, and sparse masks -- as special cases of a Block Compressed Sparse Row (BSR) matrix. This abstraction supports arbitrary block sizes (B_r, B_c) where B_r is the query tile size and B_c is the page size defined by the KV-cache management algorithm.

Composable Formats for Memory Efficiency

FlashInfer uses multiple block-sparse matrices with different block sizes to decompose the KV-cache. Shared prefixes use a larger B_r (e.g., 3), allowing multiple queries to share KV-cache data in high-bandwidth shared memory. Unique suffixes use B_r=1, where each thread block accesses KV-cache independently through global memory.

Data Loading: Global to Shared Memory

FlashInfer uses asynchronous LDGSTS copy instructions with 128-byte width to maximize memory bandwidth when loading sparse KV tiles. For Hopper GPUs with contiguous KV-cache, Tensor Memory Accelerator (TMA) is used for further acceleration. The last dimension of KV-cache (head dimension d, commonly 128 or 256) remains contiguous, maintaining coalesced access.

⚡ JIT-Compiled Customizable Templates

FlashInfer provides a customizable attention variant interface inspired by FlexAttention, enabling users to define custom attention behaviors through CUDA code strings that are JIT-compiled into optimized kernels. The variant specification includes six functor types that map to different parts of the FlashAttention kernel body.

JIT Compilation Flow

The variant specification is a Python-side CUDA code string that gets inserted into the kernel template. FlashInfer's JIT system operates in three layers:

struct FlashSigmoid : AttentionVariantBase {
  static constexpr bool use_softmax = false;
  using Params_ = typename Params;
  using KernelTraits_ = typename KernelTraits;
  float scale;

  template <typename Params>
  __device__ __host__ FlashSigmoid(const Params& params, ...) {
    scale = params.scale;
  }

  REGISTER_LOGITS_TRANSFORM(params, logits, ..., {
    return 1. / (1. + exp(-logits_score * scale + bias));
  })
};

Microkernel Tile Size Selection

FlashInfer implements the FlashAttention-2 algorithm across multiple tile sizes to adapt to varying operational intensities. The heuristic selects tiles based on hardware and workload:

Query Tile Sizes KV Tile Sizes Selection Strategy
{1, 16, 32, 64, 128} {32, 64, 128} FA2 kernels (SM75-SM89)
Multiples of 64 Multiples of 64 FA3 kernels (SM90 Hopper, WGMMA-aligned)
Selection Heuristic: (1) Determine average query length per batch, choose minimal query tile size meeting or exceeding it. (2) Formulate register and shared memory constraints as functions of KV tile size, maximizing SM occupancy. (3) For GQA, query length is fused with head group dimension.

Algorithm Details

The scheduling algorithm (Algorithm 1 in the paper) computes a balanced work distribution in two phases:

Phase 1: Chunk Size Computation

Compute maximum KV chunk size L_kv by dividing total work equally across available CTAs. Uses binary search to find the minimum chunk size satisfying grid size constraints. Implemented in PrefillBinarySearchKVChunkSize() and PartitionPagedKVCacheBinarySearchMinNumPagePerBatch() in scheduler.cuh.

Phase 2: Priority Queue Assignment

Sort work items by descending KV length. Use a min-heap priority queue to assign each chunk to the CTA with the least accumulated cost. Produces (1) CTA work queues and (2) index mapping from partial to final outputs. Plan information is cached GPU-side.

// From scheduler.cuh: Binary search for optimal KV chunk size
inline auto PrefillBinarySearchKVChunkSize(
    const bool enable_cuda_graph,
    const uint32_t max_batch_size_if_split,
    const std::vector<int64_t>& packed_qo_len_arr,
    const std::vector<int64_t>& kv_len_arr,
    const uint32_t qo_chunk_size) {
  int64_t low = min_kv_chunk_size, high = max_kv_len;
  while (low < high) {
    const int64_t mid = (low + high) / 2;
    int64_t new_batch_size = 0;
    for (uint32_t i = 0; i < batch_size; ++i)
      new_batch_size += ceil_div(packed_qo_len_arr[i], qo_chunk_size)
                      * ceil_div(kv_len_arr[i], mid);
    if (new_batch_size > max_batch_size_if_split) low = mid + 1;
    else high = mid;
  }
  return std::make_tuple(enable_cuda_graph || low < max_kv_len, low);
}
CUDAGraph Compatibility: Both attention and contraction stages use persistent kernels with fixed grid sizes. The plan() function runs on CPU (not captured by CUDAGraph) while run() executes on GPU and can be captured. Plan information is asynchronously copied to a fixed GPU workspace buffer, allowing the same CUDAGraph to be replayed across different generation steps.

🔗 Framework Integration

FlashInfer provides a programming interface following the plan-run pattern inspired by the Inspector-Executor model. This design separates CPU-side scheduling from GPU-side execution, enabling CUDAGraph capture and seamless integration with existing LLM serving frameworks.

Programming Interface (Plan-Run Pattern)

# Create workspace buffer and sequence length tracker
workspace = torch.empty(..., device="cuda")
seqlen_info.init()

# Compile: create CUDAGraphs for each attention configuration
graphs = []
for task_info in task_infos:
    attn = AttentionWrapper(attn_spec, task_info, workspace)
    g = torch.cuda.CUDAGraph()
    attn.plan(seqlen_info)     # CPU: compute scheduling plan
    with torch.cuda.graph(g):
        attn.run(...)           # GPU: captured in CUDAGraph
    graphs.append(g)

# Runtime: select best graph and replay
g = select_graph(graphs)
while not finished:
    seqlen_info.update()       # Update sequence lengths
    attn.plan(seqlen_info)     # Re-plan on CPU (not captured)
    g.replay()                 # Execute attention on GPU

Architectural Strengths

Clean Separation

Framework-agnostic kernels (include/), TVM-FFI bindings (csrc/), and Python API (flashinfer/) maintain strict boundaries. Torch headers never enter kernel templates.

Developer Experience

JIT-by-default means editing .cuh files automatically triggers recompilation. No reinstallation needed. Two-level caching (memory + disk) minimizes compilation overhead.

Extensibility

The attention variant template system allows adding new attention mechanisms (FlashSigmoid, ALiBi, attention sinks) with minimal code -- just CUDA strings plugged into the existing kernel skeleton.
