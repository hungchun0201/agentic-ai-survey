Pie: A Programmable Serving System for Emerging LLM Applications

Decompose the monolithic token generation loop into fine-grained handlers and delegate control to user-provided programs called "inferlets" -- enabling custom KV cache strategies, decoding logic, and I/O within the serving system

Authors: In Gim, Zhiyao Ma, Seung-seob Lee, Lin Zhong
Affiliation: Yale University Venue: SOSP 2025, Seoul, Korea Implementation: 13,650 LoC (Rust + Python)

Overview

Current LLM serving systems (vLLM, SGLang, TGI) are built around a monolithic prefill-decode loop that enforces global policies for KV cache management, uses a fixed "predict-then-sample" generation process, and isolates inference from external operations. This design is fundamentally inflexible for emerging LLM applications like agentic workflows, custom reasoning strategies, and multi-modal interactions.

Pie decomposes this monolithic loop into independent, fine-grained service handlers (embed, forward, sample, etc.) and delegates end-to-end control to user-provided programs called inferlets. Inferlets run in WebAssembly (Wasm) sandboxes and orchestrate generation through a rich 42-API interface.

Three Requirements for Next-Gen LLM Serving

R1: Application-Specific KV Cache Control

Workflows like Tree-of-Thought, map-reduce summaries, and multi-step generations require explicit, fine-grained control over which KV cache blocks to retain, discard, reuse, or duplicate. Global heuristics (LRU, prefix caching) cannot accommodate application-specific logic.

R2: Customizable Generation Processes

Emerging decoding methods (parallel decoding, MCTS-based generation, grammar-constrained decoding, safety grounding) require the ability to modify the token prediction and sampling loop on a per-request or per-step basis.

R3: Integrated Computation and I/O

Agentic workflows must tightly couple token generation with external API calls, code execution, and symbolic solvers -- without incurring round-trip latency or losing the KV cache state between interactions.

The Root Cause of Inflexibility

All existing serving systems effectively operate with a single, fixed inferlet -- an autoregressive loop. The tight coupling of application control logic with the core execution engine makes it impossible to customize behavior without modifying system internals. Pie separates these concerns by elevating programs to first-class citizens of the serving infrastructure.

The Inferlet Programming Model

What is an Inferlet?

An inferlet is a user-provided program that orchestrates the complete generation workflow. Each inferlet runs in its own Wasm sandbox, issues API calls to Pie's handlers, and has explicit control over resources, generation logic, and I/O. Developers can write inferlets in any language that compiles to Wasm (C++, Rust, Python).

Figure: Traditional systems use a monolithic loop; Pie delegates control to per-application inferlets that issue fine-grained API calls.

A Complete Inferlet Example: Autoregressive Generation

// Autoregressive text generation in Pie (just 27 lines!)
prom = tokenize("Hello, ")
tok_limit = len(prom) + 10

// Resource allocation
prom_emb = alloc_emb(len(prom))
gen_emb  = alloc_emb(1)
kv       = alloc_kvpage(tok_limit)

// Prefill
pos = list(range(len(prom)))
embed_txt(prom, pos, prom_emb)
forward([], prom_emb, kv[:len(prom)], gen_emb)

// Decode loop
for i in range(len(prom), tok_limit):
    dist = await get_next_dist(gen_emb)
    gen  = dist.max_index()
    print(detokenize(gen))
    embed_txt(gen, [i], gen_emb)
    forward(kv[:i], gen_emb, kv[i:i+1], gen_emb)

// Resource cleanup
dealloc_emb(prom_emb)
dealloc_emb(gen_emb)
dealloc_kvpage(kv)
Or Even Simpler with the Support Library

The support library (728 lines of Rust) provides high-level abstractions. The same autoregressive loop can be written in just 3 lines:
ctx = Context(model)
ctx.fill("Hello, ")
ctx.generate_until(max_tokens=10)

Three-Layer System Architecture

Figure: Pie's three-layer architecture. The application layer runs inferlets in Wasm sandboxes, the control layer batches and schedules API calls, and the inference layer executes GPU operations.

Batch Scheduling

Pie's batch scheduler groups compatible API calls using two techniques:

  • Vertical batching: Consecutive commands of the same type within a single queue (e.g., multiple writes to the same KvPage)
  • Horizontal batching: Commands of the same type across different inferlet queues (e.g., multiple forward calls from different inferlets)

An adaptive work-conserving scheduling policy achieves up to 17x higher throughput compared to no batching (Eager mode).

Command Queues

Each inferlet can create multiple command queues to express parallelism and dependencies. API calls within a queue execute in order; calls across queues with different priorities can be reordered. This enables Pie to exploit both intra-inferlet and inter-inferlet parallelism.

Key Advantage: Composable Optimizations

Because each inferlet controls its own KV cache and generation process, optimizations stack. For an agentic workflow, a Pie user can: (1) retain KV cache of frequently used API docs using export_kvpage, (2) concurrently call external APIs upon detecting callable signatures, and (3) drop one-time API specs from context using mask_kvpage. These three optimizations together yield 3.5x throughput over vLLM baselines.

Detailed Agentic Workflow Analysis

Pie's performance advantage on agentic workflows comes from three stacking optimizations:

Optimization 1: Cache Retention

Retain KV cache of frequently used API documentation using export_kvpage. This avoids re-prefilling common context that is shared across agent turns, saving 40-60% of prefill computation.

Optimization 2: Concurrent API Calls

Detect callable function signatures in generated tokens and immediately initiate external API calls concurrently, rather than waiting for the full generation to complete. Uses Pie's async I/O (http_get) within the inferlet.

Optimization 3: Context Pruning

Drop KV cache of API specifications used only once from the context using mask_kvpage. This frees GPU memory for larger batch sizes and reduces attention computation on irrelevant past context.

As shown in the evaluation, each optimization stacks upon the previous ones, culminating in a 3.5x throughput increase over vLLM. The per-application nature of inferlets means these optimizations are tailored to the specific workflow rather than being generic system heuristics.

Discussion and Implications

Security Considerations

Delegating control to user inferlets expands the attack surface compared to closed-loop serving. Potential risks include side-channel leakage, resource-exhaustion (DoS), and misuse of token distribution access. Pie mitigates these through Wasm sandboxing and FCFS resource contention policies, but production deployment would benefit from capability-based I/O, per-inferlet rate limits, and logit obfuscation (top-K caps).

Opportunity Cost of Fine-Grained APIs

Pie's decomposition introduces an opportunity cost compared to monolithic systems. The primary source is the lack of pipelined input embedding with sampling (0.23-1.39ms per token). For 8B models, this overhead is just 2.39% of the per-token latency, but for latency-sensitive 1B models it reaches 11.41%. The authors suggest that specialized fused APIs (e.g., forward_with_sampling) could recover this cost for applications that do not need fine-grained control.

Broader Impact: Programmability as a First-Class Concern

Pie represents a paradigm shift from "serving systems optimize for the system" to "serving systems empower the application." Just as programmable network switches (P4) and programmable GPUs (CUDA) transformed their domains, programmable LLM serving may enable an entirely new class of inference optimizations discovered by application developers rather than system engineers. The 42-API surface area is a starting point -- future traits (e.g., IntrospectiveForward returning per-layer attention statistics) could enable even more sophisticated application-level optimizations.

Implementation and Performance Deep Dive

Pie System Implementation

Pie is implemented with 13,650 lines of code measured by cloc:

  • 11,640 lines in Rust: Core system including the control layer, event dispatcher, batch scheduler, command queue management, resource allocator, and the Wasm runtime integration
  • 2,010 lines in Python: GPU-side API handler implementations using PyTorch and FlashInfer kernel library
  • 728 lines support library (Rust): High-level abstractions, procedural macros, async runtime, sampling utilities, and Context API

Native C++/CUDA Inference Layer

In addition to the Python-based inference layer, Pie provides a native C++/CUDA implementation that does not depend on PyTorch. This implementation achieves 10-30% lower end-to-end latency and more efficient GPU memory utilization because it pre-allocates and reuses GPU buffers directly. The native layer is provided as part of the open-source release.

Trait Extensibility

Pie organizes APIs into Traits (analogous to Rust traits or Java interfaces). This provides a clean extension mechanism:

// Current trait hierarchy
Allocate                    // Resource management
  Forward(Allocate)         // Model forward pass
    InputText(Allocate, Forward)  // Text embedding
    InputImage(Allocate, Forward) // Image embedding
  OutputText(Allocate)      // Token sampling
  Tokenize(InputText)      // Tokenization

// Adding new capabilities (e.g., audio input)
// 1. Define the trait: InputAudio(Allocate, Forward)
// 2. Model implements embed_audio() handler
// 3. Existing inferlets continue to work unchanged

This trait system means that adding support for new modalities (audio, video) or new capabilities (e.g., IntrospectiveForward returning per-layer attention statistics) only requires defining new traits and having models implement them. Existing inferlets are unaffected.

Batch Scheduling Strategy Details

Pie's adaptive batch scheduler uses a work-conserving policy with two states:

  • GPU busy: Queue incoming API calls. As soon as the GPU becomes idle, the inference layer notifies the control layer via IPC to trigger batch formation.
  • GPU idle: Immediately form and dispatch a batch from pending calls.

This simple strategy outperforms three alternatives:

StrategyRequests/s (128 inferlets)
Eager (no batching)5.61
K-only (fixed size threshold)30.09
T-only (fixed timeout)78.11
Adaptive (work-conserving)84.85

The adaptive strategy achieves 17x throughput over naive eager execution and outperforms fixed-rate strategies by 8-40% because it avoids both underutilizing the GPU (flushing too early) and inflating latency (waiting too long).

Inter-Process Communication Architecture

The control layer and inference layer communicate via ZeroMQ IPC. The control layer runs in the main Python process alongside the Wasm runtime, while the inference layer runs in a separate process on the GPU host. This separation ensures that Wasm execution and Python-based API call deserialization do not block GPU computation. The IPC boundary crossing adds minor, constant latency (~0.006ms for control layer, ~0.001ms for application-control layer boundary).

Artifacts and Reproducibility

Pie received all three SOSP artifact evaluation badges: Artifacts Available, Artifacts Functional, and Results Reproduced. The evaluation uses a GCP VM G2 instance (g2-standard-32) with an NVIDIA L4 GPU (24GB), running Llama 3 models (1B, 3B, 8B). The complete evaluation suite, including all inferlet implementations and benchmark scripts, is available in the open-source repository.

Deliberate Prompting Strategies as Inferlets

Pie implements four deliberate prompting strategies as inferlets, each approximately 80-100 lines of code. These demonstrate Pie's ability to express complex generation patterns:

Tree-of-Thought (ToT)

198 lines. Generates multiple candidate reasoning paths, evaluates them, and selects the best. Uses copy_kvpage to branch KV cache at decision points and export_kvpage to share evaluation context across branches. Achieves up to 28% latency reduction and 34% throughput improvement over baseline Python implementations.

Recursion-of-Thought (RoT)

106 lines. Recursively branches up to depth 5 with maximum 2^5=32 branches. Pie's advantage is especially pronounced here because SGLang's RadixAttention cannot effectively apply to the dynamic graph structure. KV cache reuse via export_kvpage / import_kvpage eliminates redundant prefill computation across recursive calls.

Graph-of-Thought (GoT)

87 lines. Models reasoning as a directed graph where nodes represent thought transformations. Document summarization is used as the evaluation task. Not supported by either vLLM or SGLang natively -- Pie is the only system that can express this pattern as a first-class program.

Skeleton-of-Thought (SkoT)

82 lines. Generates an outline first, then fills in details in parallel. Uses multiple command queues for parallel section generation, with each queue writing to its own KV cache pages. Achieves efficient utilization through Pie's vertical batching of parallel forward calls.

Attention-Level Techniques as Inferlets

Pie's mask_kvpage API and the mask argument in forward enable implementing various attention variants without any kernel changes:

  • Attention Sink (StreamingLLM): 60 lines. Retains initial tokens ("sink" tokens) and recent window tokens, masking out intermediate positions. Achieves 1.5x lower latency and 30x higher throughput compared to the original StreamingLLM implementation.
  • Windowed Attention: 60 lines. Restricts attention to a sliding window of recent tokens.
  • Hierarchical Attention: 42 lines. Implements multi-resolution attention patterns.

These implementations are remarkable not only for their brevity but for the fact that they match or exceed specialized implementations. The Attention Sink comparison is influenced by differences in underlying GPU kernel libraries, but it demonstrates Pie's ability to effectively express and implement sophisticated attention-level techniques.

Resource Contention and Fairness

When KvPage demand exceeds GPU memory capacity, Pie currently applies a First Come First Serve (FCFS) policy, terminating the most recently started inferlet to free resources. While simple and effective, this policy could be improved with:

  • Per-inferlet quotas: Limit maximum KV pages per inferlet to prevent resource monopolization
  • SLO-aware admission: Prioritize inferlets based on latency targets or business priority
  • Preemption: Swap KvPages to CPU memory rather than terminating inferlets
  • Overprovisioning: Allow KvPage swapping between CPU and GPU transparently

These enhancements are identified as promising future work that would strengthen Pie's viability for multi-tenant production deployments.

Summary of Contributions

Pie makes three fundamental contributions to LLM serving: (1) identifying the mismatch between monolithic serving architectures and emerging application requirements (custom KV cache control, flexible decoding, I/O integration), (2) designing the inferlet programming model with 42 APIs organized into extensible traits that provide Turing-complete control over the generation process, and (3) demonstrating competitive performance on standard tasks (3-12% overhead) while enabling substantial improvements on advanced applications (1.3-3.5x throughput on agentic workflows). Pie represents a paradigm shift toward programmable LLM serving where applications, not systems, drive optimization decisions.

Model Support and GPU Backend

Pie currently supports the Llama family of models (Llama 3: 1B, 3B, 8B). The inference layer uses:

  • FlashInfer GPU kernel library: Provides the primary attention computation backend. Both Pie, vLLM, and SGLang use the same FlashInfer backend in evaluations to ensure fair architectural comparison.
  • PyTorch-based handlers: The Python API handlers use PyTorch for GPU operations. A native C++/CUDA implementation is also provided that avoids PyTorch overhead.
  • BF16 precision: Weights, KV cache, and activations all use BF16 format.

The evaluation uses a GCP VM G2 instance (g2-standard-32) with an NVIDIA L4 GPU (24GB) to host both Pie and baselines. End-to-end latency is measured from a remote Python client on the campus network, ensuring realistic network conditions are accounted for.

Lineage of Programmable Systems

Pie builds on a rich lineage of programmable systems across computing domains that expose fine-grained interfaces to separate application-level control from system internals:

  • Exokernel (1995): OS design separating resource management from protection, allowing applications to manage hardware directly
  • Click Modular Router (1999): Composable packet processing elements for network routers
  • SPIN (1995): Extensibility, safety, and performance in kernel extensions
  • eBPF (2024): Extended Berkeley Packet Filter for programmable kernel behavior
  • OpenFlow/SDN (2008): Programmable network switches via flow tables
  • Arrakis (2014): OS architecture separating control and data plane for application-level resource management

Pie applies this core principle -- separating application-level control from system internals -- to the domain of LLM serving, representing a natural evolution as LLM applications become increasingly diverse and sophisticated.

Speculative Decoding Implementation

Pie implements speculative decoding as a 255-line inferlet that creates two command queues -- one for the draft model and one for the verification model. The inferlet orchestrates the draft-verify-accept cycle using Pie's standard APIs, achieving performance comparable to vLLM's deeply-integrated speculative decoding implementation. Notably, SGLang does not support speculative decoding, making this comparison exclusive to Pie vs vLLM.

The Forward API in Detail

The forward API is the most important API in Pie's interface. It takes input Embeds and/or input KvPages, performs the model forward pass, and produces output Embeds and/or output KvPages:

// Forward API signature
forward(
    q: Queue,          // Command queue for batching
    ikv: list[KvPage],  // Input KV pages (attention context)
    iemb: Embed,        // Input embeddings (current tokens)
    okv: list[KvPage],  // Output KV pages (write generated cache)
    oemb: Embed         // Output embedding (next-token prediction)
)

// Key behaviors:
// - Omitting ikv means no prior KV context (new sequence)
// - Omitting okv means no KV cache is written (read-only pass)
// - forward operates on explicit sequence positions inferred from KvPage layout
// - An optional boolean attention mask can be provided for custom masking
// - forward_with_adapter supports LoRA adapters for fine-tuned models

This explicit resource-based API contrasts sharply with vLLM's implicit request-based API where the system manages all resources internally. The explicitness is what enables the full range of inferlet-level optimizations.

Concurrency and Inferlet Lifecycle

Pie is designed to handle hundreds of concurrent inferlets efficiently. The Inferlet Lifecycle Manager (ILM) in the application layer manages creation, destruction, and communication:

  • Creation: Client sends a Wasm binary to Pie via RPC. The ILM JIT-compiles it (or uses cached compilation) and launches it in a new Wasm instance. Cold start: ~50ms; warm start (cached binary): ~35ms.
  • Execution: The inferlet runs in its own single-threaded event loop, issuing async API calls. Concurrency within an inferlet is achieved through non-blocking I/O (http_get, receive) and multiple command queues.
  • Communication: Users communicate with running inferlets via send/receive APIs. Inferlets can also communicate with each other via broadcast/subscribe for multi-agent scenarios like Swarm.
  • Termination: Explicit cleanup (dealloc_emb, dealloc_kvpage) or automatic resource reclamation when the inferlet exits. Under resource contention, FCFS policy terminates the most recently started inferlet.

Evaluation Baselines and Fairness

To ensure fair comparison, all baseline systems in the evaluation use the same FlashInfer GPU kernel backend for attention computation. The baselines are:

  • vLLM v0.6.0: State-of-the-art serving engine with PagedAttention, prefix caching, and speculative decoding support
  • SGLang v0.4.4: Serving engine with RadixAttention for prefix tree caching and fork/join generation primitives
  • LMQL v0.7.3: Query language for LLMs with constrained decoding support (used for structured generation comparisons)
  • StreamingLLM: Specialized system for attention sink technique (used for attention-level technique comparisons)

For baseline implementations of agentic workflows, the authors replicate the same high-level application logic using Python scripts interacting with vLLM/SGLang's API, applying best-effort optimizations (prefix caching, parallel client requests) to make the comparison as favorable as possible for the baselines.

The evaluation comprehensively covers standard text completion (validating minimal overhead), deliberate prompting strategies (ToT, RoT, GoT, SkoT), attention-level techniques (attention sink, windowed attention, hierarchical attention), existing serving features (prefix caching, speculative decoding, beam search, EBNF grammar), and agentic workflows (ReACT, CodeACT, Swarm). This breadth of evaluation validates Pie as both expressive and efficient across the full spectrum of LLM applications.
