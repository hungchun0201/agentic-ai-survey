Jenga: Effective Memory Management for Serving LLMs with Heterogeneous Embeddings

A two-level memory allocator that tackles fragmentation and enables customizable caching for heterogeneous LLM architectures (MLA, sliding window, Mamba, VLMs)

SOSP 2025 arXiv:2503.18245 Memory Management KV Cache

Chen Zhang, Kuntai Du, Shu Liu, Woosuk Kwon, Xiangxi Mo, Yufeng Wang, Xiaoxuan Liu, Kaichao You, Zhuohan Li, Mingsheng Long, Jidong Zhai, Joseph Gonzalez, Ion Stoica

Tsinghua University · University of Chicago · UC Berkeley

Overview

Figure 1 (from paper): Traditional LLMs (left) vs. Latest LLMs (right). LLMs are becoming more and more heterogeneous and produce KV caches with different sizes and dependencies, which demands a new GPU memory manager design.

Modern LLM architectures have evolved far beyond homogeneous transformer stacks. Models now combine self-attention, sliding window attention, Mamba state-space layers, cross-attention (for VLMs), and speculative decoding -- each producing KV caches of different sizes and with different token-dependency patterns. The original PagedAttention design assumed fixed-size embeddings and full-prefix dependencies, and these assumptions are now violated.

Jenga introduces a two-level memory management system that addresses these challenges through: (1) an LCM allocator for compatible page sizes, (2) customized per-layer-type allocators and evictors, and (3) request-aware allocation to minimize fragmentation.

Impact: Merged into vLLM Main Branch

Jenga is not just a research prototype -- it has been merged into the main branch of vLLM, the dominant open-source LLM inference engine. It is compatible with all 90+ models supported by vLLM v0.6.4 and is transparent to users of the inference engine. This represents direct production-level impact.

The Heterogeneity Problem

Figure 2 (from paper): Contrasting traditional LLMs (top left) and latest LLMs. LLMs are becoming more and more heterogeneous: the KV cache sizes may differ, the KV cache dependencies are different, and the LLM architecture can also diverge.

Four Sources of Heterogeneity in Modern LLMs

Sparse Attention

Modern models mix full self-attention with sliding window attention (Gemma-2, Mistral). Window layers only need recent tokens, wasting memory on old KV cache entries that will never be read again.

State Space Models

Mamba layers (Jamba) produce large fixed-size state tensors per token, while interleaved self-attention layers produce small per-token KV caches. These two patterns require fundamentally different memory management.

Multi-Modal Models

Vision Language Models (Llama 3.2 Vision, InternVL) have cross-attention layers for image tokens and self-attention for text tokens, producing KV caches of completely different sizes.

Multiple Concurrent Models

Speculative decoding runs a small draft model and a large verification model simultaneously, each with differently-sized KV caches that must be managed together.

Key Design Decisions

LCM Page Size

Uses Least Common Multiple of all embedding sizes as the compatible large page size. For Llama 3.2 Vision with 256-byte image KV and 384-byte text KV: LCM(256, 384) = 768 bytes. This avoids internal fragmentation without requiring custom GPU kernels.

Page-Layer Partition

Unlike PagedAttention's layer-page partition (pages per layer), Jenga uses page-layer partition: memory is first divided into pages, then each page is partitioned by layer. This enables inter-type memory exchange and makes layers addressable within a single page.

Request-Aware Allocation

Small pages within a large page are associated with the same request. This ensures large pages can be returned when a request completes, avoiding the interleaved fragmentation that plagues naive allocators.

Allocation Algorithm

Jenga implements a priority-based allocation strategy for each layer type:

  1. Allocate a request-associated unused small page. Prefer pages already tied to the current request.
  2. Request a new large page from the LCM allocator if no suitable small page exists, partition it, and associate all small pages with this request.
  3. Evict a large page using LRU if no empty large pages exist. The LRU timestamp is the latest access time among all small pages in the large page.
  4. Allocate any unused small page from other requests' large pages as a last resort.
  5. Evict a small page individually only as a final fallback.
Why Request-Aware Allocation Matters

Without request-awareness, allocations for requests A and B become interleaved within large pages (ABAB pattern). When request A completes and its small pages are freed, the large page cannot be returned because B still occupies slots in it. Jenga's request-aware approach (AAAA, BBBB pattern) ensures clean deallocation when requests finish.

Customizable Prefix Caching

Figure 4 (from paper): Balanced and aligned cache eviction policy can improve hit rate. Different layer types have different caching semantics, requiring customized eviction strategies.

The Challenge: Layer-Specific Eviction and Hit Rules

Different layer types have fundamentally different caching semantics:

  • Self-attention: A cache hit requires all prefix tokens to remain unevicted
  • Sliding window: Only the last window_size tokens need to be cached for a hit
  • Mamba: Only cache states every 512 tokens (due to large state size)
  • Vision cross-attention: Evict entire images at once to avoid recomputation

Three API Primitives

class LayerSupportsPrefixCache:
    def update_last_access(self, r: Request, time: int):
        """Update access timestamps (customized per layer type)"""
        ...
    def set_prefix_length(self, r: Request):
        """Set prefix length for aligned eviction priorities"""
        ...
    def get_possible_prefix(self, is_hit: List[bool]):
        """Return valid prefixes based on cached token status"""
        ...

# Example: Sliding Window Layer customization
class SlidingWindowLayer(LayerSupportsPrefixCache):
    def update_last_access(self, r, time):
        # Only update tokens within the sliding window
        for i in range(r.len - sliding + 1, r.len + 1):
            self.evictor.update_last_access(r.page[i], time)

    def get_possible_prefix(self, is_hit):
        # A prefix is valid as long as the window portion is cached
        return {p | for all x in [0, sliding], is_hit[p-x] is True}

Implementation Details

Built on vLLM v0.6.4

Jenga is implemented with ~4000 lines of Python on top of vLLM. It requires no changes to PagedAttention CUDA kernels. The key changes to inference engine workers are minimal:

  1. Allocate a single KV cache tensor and assign different offsets (KV_cache_start_ptr) per layer
  2. Prepare attention metadata (page_id, page_size) per layer type rather than globally
# Jenga works with existing PagedAttention kernels unchanged
# Each layer type gets its own view of memory:
attention = paged_attention(
    query, key_cache, value_cache,
    kv_cache_start_ptr=layer_offset,  # Per-layer offset
    page_size=small_page_size,        # Per-type page size
    pageid_exec=page_ids_for_type     # Per-type page table
)

Case Studies and Advanced Features

Vision Embedding Cache for VLMs

Jenga treats vision embedding cache as another layer type with a specific hidden size. This enables two powerful optimizations:

  • Allocate-on-demand KV cache: Jenga frees vision embeddings immediately after each chunked prefill step, preventing simultaneous growth of both vision embeddings and KV cache. Peak memory usage stays bounded.
  • Cache reuse: When the vision embedding of a token is used before generating its KV cache, Jenga can cache the vision embedding for reuse, completely eliminating redundant vision encoder runs on repeated images.

Mamba Layer Caching

Mamba layers produce significantly larger per-token state tensors than attention layers. Caching all Mamba states would be prohibitively expensive. Jenga's customizable caching system addresses this by:

  • Only caching Mamba state every 512 tokens (configurable granularity)
  • Implementing get_possible_prefix to only hit prefixes with length that is a multiple of 512
  • Only updating the last cached token's access time via update_last_access

Dynamic KV Cache Allocation

Unlike vLLM which statically allocates the same amount of KV cache to all layers, Jenga dynamically adjusts allocation between sliding window and self-attention layers based on the workload. In dynamic workload traces, the rate of self-attention KV cache ranges from 27.8% to 54.5% of total allocated KV cache memory, demonstrating how Jenga adapts to changing demands in real time.

Compatibility and Transparency

Jenga is compatible with all 90+ models in vLLM v0.6.4. For standard self-attention-only models (e.g., Llama 3.1), Jenga's automatic memory management reaches the optimal case with zero overhead. For heterogeneous models, Jenga achieves an averaged 1.58x throughput improvement without any manual memory management configuration -- it parses the model structure automatically.

Technical Deep Dive: Memory Layout and Kernel Compatibility

Page-Layer Partition vs Layer-Page Partition

Standard PagedAttention uses a layer-page partition: memory is first partitioned by layer, then each layer's allocation is split into pages. This is simple because executing one layer only needs that layer's memory. However, it prevents exchanging pages between different layer types because the memory regions are disjoint.

Jenga uses a page-layer partition: memory is first divided into pages (large LCM pages), then each page is internally partitioned by layer. This seemingly simple change has profound implications:

  • Pages from different layer types coexist within the same large page, enabling inter-type memory exchange
  • The KV cache for different layers shares a common address space, unified by the LCM allocator
  • Attention kernels still see contiguous per-layer memory through kv_cache_start_ptr offsets and page_size_exec parameters

Kernel Compatibility

Despite the new memory layout, Jenga maintains full compatibility with existing PagedAttention CUDA kernels (FlashAttention, FlashInfer). Each layer type's memory view can still be represented by:

# Standard PagedAttention kernel interface
paged_attention(
    query=query_tensor,
    key_cache=key_cache_tensor,
    value_cache=value_cache_tensor,
    kv_cache_start_ptr=layer_specific_offset,   # Jenga assigns per-layer
    page_size=small_page_size_for_this_type,     # Jenga: per-type page size
    pageid_exec=page_ids_for_this_layer_type     # Jenga: per-type page table
)

This means Jenga achieves its memory efficiency gains through a pure software abstraction without requiring any GPU kernel modifications -- a crucial property for production deployment and maintainability.

API Design for Custom Caching Policies

Jenga's three-method API (update_last_access, set_prefix_length, get_possible_prefix) was carefully designed to provide maximum flexibility while maintaining cross-layer consistency. The prefix-subset evictor coordinates eviction across all layer types by taking the intersection of valid prefixes from each layer's get_possible_prefix call. This ensures that tokens are only marked as "cached" if they are present in the KV cache of every layer type that needs them.

Integration with Existing Serving Ecosystem

Jenga's design philosophy is minimal invasion, maximum impact. The major changes to the inference engine workers are limited to two areas:

  1. KV cache tensor allocation: Allocate a single unified tensor and compute per-layer offsets, replacing the standard per-layer allocation
  2. Attention metadata preparation: Prepare page_id and page_size parameters per layer type rather than using global values

These changes amount to approximately 4,000 lines of Python code -- a manageable engineering effort that has been validated through merger into the vLLM main branch. Importantly, Jenga introduces zero overhead for homogeneous models (standard transformer stacks), making it a safe default memory manager.

Design Alternatives: Why LCM over GCD or MAX

The choice of LCM (Least Common Multiple) as the compatibility layer page size was not obvious. Here is the detailed comparison of three strategies and why LCM emerged as the best trade-off:

GCD Page (Greatest Common Divisor)

Pros: Zero internal fragmentation since pages exactly divide all embedding sizes.
Cons: Requires custom GPU kernels for many different page sizes. The GCD is often very small (e.g., GCD(256,384)=128), leading to excessive page table entries and reduced burst memory efficiency. Multiple custom attention kernels significantly increase engineering overhead.

MAX Page (Maximum Size)

Pros: No new GPU kernels needed. Simple implementation.
Cons: Severe internal fragmentation for layers with smaller embedding sizes. For Jamba 52B, mamba state requires 1,344 tokens per page to avoid fragmentation, exceeding typical request lengths (~1,085 tokens). Coarser granularity hurts both memory allocation and prefix cache hits.

LCM Page (Least Common Multiple) -- Jenga's Choice

Pros: No new GPU kernels needed. Compatible with all existing attention kernels. Minimal internal fragmentation (addressed by request-aware allocation). In practice, the largest LCM across all vLLM v0.6.4 models is only 84x the smallest page size (from Jamba), and no performance degradation is observed.
Cons: Could theoretically produce very large LCM values, but this has not been an issue in practice.

Future Directions

  • Hamba support: NVIDIA's Hamba architecture lacks essential GPU kernels in current open-source frameworks. Once these become available, Jenga can manage Hamba's heterogeneous cache natively.
  • Multi-GPU serving: Extending Jenga's two-level allocation across multiple GPUs for tensor-parallel or pipeline-parallel inference.
  • Integration with KV compression: Novel KV cache designs (MLA, MiniCache, quantized caches) can benefit from Jenga's heterogeneous management, as different layers may use different compression ratios.
  • Full multi-model serving: The custom_kv_cache API enables registering multiple models with compatible page sizes, allowing KV cache pages to be exchanged between models served on the same GPU.

On the smaller L4 GPU (24GB), memory efficiency becomes even more critical. Jenga achieves:

The Jamba-1.5 model (52B parameters) is skipped on L4 as it cannot fit in a single 24GB GPU. This highlights that Jenga's memory savings directly translate to the ability to serve larger models and longer sequences on resource-constrained hardware.

Summary of Contributions

Jenga makes three key contributions: (1) identifying the growing heterogeneity in LLM architectures as a new memory management challenge, (2) proposing a two-level LCM-based allocation scheme with customizable per-layer caching policies, and (3) demonstrating production-level impact through merger into vLLM. The result is a general-purpose memory manager that handles the full spectrum of modern LLM architectures -- from standard transformers to hybrid Mamba models to multi-modal VLMs -- without requiring GPU kernel changes or manual configuration.

Benchmark Methodology and Datasets

Jenga's evaluation follows the standard vLLM benchmarking methodology with carefully selected datasets for each model type:

  • MMLU-pro: Used for text-only models (Jamba-1.5, character.ai, PyramidKV, Llama 3.1). MMLU-pro's maximum length of 3,076 tokens provides a representative short-to-medium context workload.
  • MMMU-pro: Used for multi-modality models (Llama 3.2 Vision). Contains ~6,193 image tokens and 43 text tokens per request on average, making memory waste from heterogeneous allocation immediately apparent.
  • arXiv-QA: Used for sliding window models (Gemma-2, Ministral). Long-context question answering over arXiv articles with average request length of 92,408 tokens, stressing the sliding window eviction policy.

All experiments are performed end-to-end by changing only the memory management system (vLLM vs Jenga), keeping all other system components identical. This ensures that throughput differences are attributable solely to memory management efficiency.
