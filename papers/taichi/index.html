TaiChi: Prefill-Decode Aggregation or Disaggregation? Unifying Both for Goodput-Optimized LLM Serving

A hybrid aggregation-disaggregation serving system that uses differentiated-capability GPU instances and latency shifting to achieve superior goodput under any combination of TTFT and TPOT SLOs.

Chao Wang, Pengfei Zuo, Zhangyu Chen, Yunkai Liang, Zhou Yu, Ming-Chang Yang — CUHK, Huawei Cloud, Sun Yat-sen University

Up to 77% Goodput Improvement 13.2x TTFT Reduction 1.69x TPOT Reduction Latency Shifting

Overview

TaiChi resolves a fundamental debate in LLM serving: whether prefill-decode (PD) aggregation or PD disaggregation is the superior approach. The answer, TaiChi demonstrates, is neither -- or rather, both. By unifying the two paradigms with differentiated GPU instances and a novel latency-shifting scheduling strategy, TaiChi achieves optimal goodput under any combination of TTFT and TPOT service level objectives.

Core Insight: Latency Shifting

The key innovation is latency shifting: strategically degrading the latency of requests that already satisfy their SLO constraints, thereby reallocating GPU resources to requests at risk of SLO violation. This is achieved through differentiated P-heavy and D-heavy instances, flowing decode scheduling, and length-aware prefill scheduling.

Root Causes of Failure

PD Aggregation Bottleneck

TPOT violation from prefill-decode interference. All instances co-locate prefill and decode tasks. Linear operations from chunked prefill directly increase decode latency with a strong R2=0.99 correlation. Each additional prefill token per output token adds 0.2ms to TPOT.

PD Disaggregation Bottleneck

TTFT violation from request queuing. Dedicated prefill instances have lower processing capacity since decode instances cannot assist. Queue times dominate TTFT, especially as PD ratios change from 4:4 to 7:1. Queuing time contributes most of p90 TTFT.

Observations Leading to TaiChi

Opportunity 1: Latency Headroom Exists

In PD aggregation, 75% of requests achieve TTFT below 60% of the SLO. In PD disaggregation, 100% of requests achieve TPOT below 60% of the SLO. This surplus latency can be "shifted" to help struggling requests.

Opportunity 2: Latency is Shiftable via Scheduling

Requests prioritized for GPU resources experience better latency, while deprioritized ones experience worse latency. By selectively degrading well-performing requests, we free resources for those at risk of SLO violation.

Extreme Configurations

By adjusting sliders, TaiChi can emulate pure PD aggregation (SD = SP, all same chunk size) or pure PD disaggregation (SD = 0, no prefill on decode instances). Under balanced SLOs, TaiChi finds hybrid configurations that outperform both extremes.

Configuration Selection

The optimal configuration for a given workload and SLO is determined via offline search, following the approach from DistServe. This search completes in minutes, which is far shorter than typical hourly workload shifts. Reconfiguration is triggered only on significant workload changes, using a search-and-reconfigure strategy.

Implementation

TaiChi is implemented on the open-source vLLM framework. Chunked prefill is extended to support per-instance chunk sizes. KV cache transfer between P-heavy and D-heavy instances uses NCCL, extending vLLM's one-to-one transmission to many-to-many. The transfer is decoupled from the critical path using fused CUDA operators for storing received KV cache into vLLM's paged memory, minimizing CPU overhead.

Hybrid-Mode Inference

TaiChi's hybrid mode uniquely combines two complementary scheduling principles: aggregated batch handling for high resource utilization and disaggregated request handling for fine-grained latency control.

Aggregated Batch Handling

Both P-heavy and D-heavy instances process mixed batches containing prefill and decode tasks. This boosts total prefill throughput by enabling all instances to contribute to prefill processing.

Disaggregated Request Handling

A single request's prefill and decode phases can be executed on different instances. Prefill on a P-heavy instance for fast TTFT; decode on a D-heavy instance for low TPOT. KV cache transferred via NCCL.

Flowing Decode Scheduling

To enable request-level TPOT degradation, TaiChi introduces flowing decode scheduling that dynamically migrates decode requests between D-heavy and P-heavy instances.

Flowing Decode: Three-Stage TPOT Control
Stage 1
Low-Interference Decode Init
All decode requests start on D-heavy instances. Prevents premature TPOT violations for short-output requests (output length unknown a priori).
Stage 2
Longest-First Degradation Flowing
When D-heavy HBM reaches watermark M (95%), offload the longest-output decode requests to P-heavy instances. These have the most TPOT budget remaining and can absorb degradation.
Stage 3
TPOT-Aware Decode Backflow
Monitor TPOT of migrated requests on P-heavy instances. When current TPOT approaches SLO (exceeds alpha * SLO, alpha=0.96), flow request back to D-heavy instance. Output length resets.

Decode Scheduling Algorithm

def decode_schedule(S, tpot_slo, memory_usage, memory_watermark, instance_type):
    if instance_type == "P-heavy":
        # Backflow: requests approaching TPOT SLO
        O = {r for r in S if r.current_tpot > tpot_slo * 0.96}
        return O  # Send back to D-heavy

    elif instance_type == "D-heavy":
        D = []
        m_release = 0
        while memory_usage - m_release > memory_watermark:
            # Select request with longest current output
            r_star = argmax(r.current_output_len for r in S - D)
            D.append(r_star)
            m_release += r_star.memory
        return D  # Send to P-heavy for degradation

Flowing Decode: Design Rationale

Why Longest-First Selection?

No Output Length Prediction Needed

Short-output requests are more vulnerable to TPOT degradation because output length acts as the denominator in interference intensity calculations. Since output lengths cannot be predicted reliably (60-81% accuracy on some datasets), TaiChi avoids prediction entirely. Instead, it selects requests with the currently longest output for degradation -- these have already benefited from many low-interference decode iterations and have the largest TPOT budget remaining to absorb high-interference P-heavy execution.

Backflow as a Configuration Signal

Frequent backflow from P-heavy to D-heavy instances is not merely a recovery mechanism -- it is a diagnostic signal indicating improper architectural configuration. Frequent backflow suggests:

Oversized P-heavy Chunks

If SP is too large, prefill-decode interference on P-heavy instances overwhelms migrated decode requests quickly, triggering rapid backflow.

Insufficient D-heavy Count

Too few D-heavy instances means decode capacity is limited, forcing more requests to P-heavy, which then need frequent backflow.

Workload Mismatch

The current slider configuration may not match the workload characteristics. A new offline search should be triggered to re-tune RPD, SP, and SD.

Memory Watermark Threshold

The predefined memory watermark M (set to 95% HBM utilization in experiments) ensures that at least one new decode request can always be accepted. This prevents complete memory exhaustion on D-heavy instances while providing a clear trigger point for degradation flowing. When memory usage exceeds M, the scheduler repeatedly selects and migrates the longest-output request until usage drops below M.

Length-Aware Prefill Scheduling

To enable request-level TTFT degradation, TaiChi's proxy employs a length-aware scheduling strategy that routes short prefill requests to slower D-heavy instances when their estimated TTFT remains within SLO bounds.

Scheduling Decision

def prefill_schedule(request, instances, ttft_slo):
    feasible = []
    for i in instances:
        Q = sum(estimate_exec(r.len, i.chunk, i.batch) for r in i.queue)
        E = estimate_exec(request.len, i.chunk, i.batch)
        T = (request.transfer_size / link_bw) if i.type == "P-heavy" else 0

        if Q + E + T < ttft_slo:
            feasible.append(i)

    if feasible:
        # Pick instance with fewest queued prefill tokens
        # Favors D-heavy (fewer queue tokens = degradable)
        return argmin(sum(r.len for r in i.queue) for i in feasible)
    else:
        return None  # No feasible instance, assign randomly

Both queuing and execution times are estimated using an execution time predictor (e.g., Vidur), which completes estimation within negligible tens of microseconds. The transfer time is typically negligible under high-speed NVLINK (600 GB/s) or InfiniBand (800 Gbps) interconnects.

Edge Case: No Feasible Instance

When no instance can process the request within the TTFT SLO (often due to sudden prefill workload surges), the request is randomly assigned. Prior work proposes early rejection strategies for such cases. TaiChi uses random assignment in experiments for fair comparison with PD aggregation, which generally has sufficient prefill capacity.

Overhead Analysis

Negligible System Overhead

KV cache transfer: 0.20% of total request time. Prefill scheduling: 0.01%. Decode scheduling: 0.89%. The transfer overhead is negligible thanks to high-speed NVLINK interconnects (600 GB/s), while scheduling decisions complete in tens of microseconds using Vidur's execution time predictor.

Key Design Insights

Chunk Size is Key

Chunk size directly controls the TTFT-TPOT tradeoff. Chunks below 1024 constrain TPOT but cause prohibitively high TTFT. TaiChi uses different chunk sizes on different instances to have the best of both worlds.

Longest-First Works

Selecting the longest-output decode request for degradation is ideal because (1) these have the most TPOT budget remaining and (2) output length uncertainty is reduced for long-running requests. No prediction model needed.

Backflow Signals Issues

Frequent backflow from P-heavy to D-heavy instances indicates improper configuration (oversized chunks or insufficient D-heavy instances). It serves as a signal to re-tune sliders for current workload characteristics.

Key Takeaways

End the Debate

Neither PD aggregation nor PD disaggregation is universally optimal. TaiChi unifies both via differentiated-capability instances (P-heavy and D-heavy), achieving optimal goodput under any TTFT/TPOT combination.

Latency Shifting

The core innovation: strategically degrade latency of requests meeting SLOs to reallocate GPU resources to those at risk. Flowing decode scheduling handles TPOT; length-aware prefill handles TTFT.

Up to 77% Goodput Gain

Under balanced SLO constraints where both PD aggregation and disaggregation struggle (16% and 50% attainment respectively), TaiChi achieves 91.2% SLO attainment through incremental application of its three techniques.
