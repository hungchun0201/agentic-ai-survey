Overview

The State Restoration Bottleneck in Stateful LLM Serving

Modern LLM applications are increasingly stateful: multi-round conversations, RAG pipelines, and code assistants all depend on historical context. In these settings, the KV cache from prior interactions must be restored when a conversation resumes. Analysis of ShareGPT4 and L-Eval traces shows that a single A100-40GB GPU can keep only 7-20 multi-round conversations or 1-3 long contexts in GPU memory simultaneously.

When KV caches are evicted (which happens frequently), they must be restored. The two existing approaches are both expensive:

  • Token recomputation: Re-executes the full prefill phase from original tokens. This requires full attention + FFN computation, making it 20-26x slower than ideal TTFT.
  • KV offloading: Stores KV cache on host storage and transfers it back via PCIe. The KV cache is large (10^5x larger than tokens), making transfers 6.5-13x slower than ideal.

Both approaches occupy extreme ends of the compute-IO spectrum: recomputation uses only GPU compute, KV offloading uses only IO bandwidth. Neither leverages both resources simultaneously.

HCache: A Third Approach

HCache introduces a fundamentally new state representation for LLM serving: caching intermediate activations (hidden states) instead of KV tensors or original tokens. Hidden states are the inputs to each transformer layer, from which KV pairs can be reconstructed via a single GEMM projection. This approach is:

  • 2x smaller than KV cache (one tensor vs K+V pair per layer)
  • 6x faster to recompute KV from hidden states than from tokens (skips attention and FFN)
  • Pipelineable: IO transmission and computation can overlap concurrently
  • Lossless: Zero accuracy degradation since K = W_k * H and V = W_v * H are exact

Background: Stateful LLM Applications

Multi-Round Conversations

In ChatGPT-style interactions (ShareGPT4 trace analysis), the average input per round is 66.8 tokens and the average output is 358.8 tokens. While individual rounds are short, history accumulates: the CDF of conversation history shows that 50% of conversations exceed 2.5K tokens. Each round that resumes a conversation requires restoring the full KV cache of all prior rounds.

Long-Context Applications

RAG applications and document Q&A (L-Eval traces) involve context lengths averaging 16,340 tokens, with individual subtasks like Paper Assistant extending to 10,603 tokens. These contexts may be reused hours apart. The KV cache for a 7B model at 16K tokens is ~2GB, too large for persistent GPU residency but frequently needed.

Why Hidden States Are Better

The key insight is that the hidden state H_i at each layer is the minimal sufficient representation from which KV pairs can be exactly reconstructed. Compared to the two existing approaches:

  • vs KV Offloading: Hidden states are exactly half the size of KV cache (one tensor vs K+V), halving IO transmission time. Additionally, the GEMM to recompute KV from H overlaps with IO.
  • vs Token Recomputation: Recomputing KV from hidden states requires only the QKV projection (a single GEMM per layer), skipping the expensive attention (O(N^2*D)) and FFN (16*N*D^2) modules entirely. The theoretical speedup is at least 6x.

Theoretical Cost Analysis

IO Transmission Cost

IO_hidden = (N_seq * D_hidden) / BW
IO_KV = (2 * N_seq * D_hidden) / BW

Hidden states are exactly 2x smaller than KV cache. For Llama2-7B: D_hidden = 4096, each token's hidden state is 8KB (BF16), while KV cache per token is 16KB. The IO savings apply to every layer.

Computation Cost

C_hidden = (4 * N_seq * D_hidden^2) / FLOPS
C_recompute = (24 * N_seq * D_hidden^2 + N_seq^2 * D_hidden) / FLOPS

The relative speedup of HCache vs recomputation: (24*N*D^2 + N^2*D) / (4*N*D^2) = 6 + N/(4*D). For sequences longer than 4K tokens with D=4096, HCache is >7x faster computationally.

Pipelined End-to-End Restoration Time

HCache overlaps IO transmission with computation: while hidden states for layer L_{n+1} are being transmitted from storage, the KV cache for layer L_n is being computed from its already-loaded hidden states. The end-to-end restoration time is therefore:

T_hidden = max(IO_hidden, C_hidden)

This is strictly better than both KV offloading (T_kv = IO_KV, no computation overlap) and recomputation (T_rec = C_attn + C_ffn, no IO utilization). HCache uses both IO and compute simultaneously.

Bubble-Free Restoration Scheduler

The Pipeline Bubble Problem

HCache's pipelined restoration (IO for layer N+1 overlapping with compute for layer N) does not always produce perfectly balanced stages. If computation is much faster than IO (or vice versa), pipeline bubbles waste resources. The bubble-free scheduler solves this by partitioning model layers into two groups:

  • L_H layers: Restored using HCache (hidden states loaded + KV recomputed)
  • L_O layers: Restored using a complementary method (KV offloading or token recomputation)

The partition is chosen to eliminate bubbles: the slower resource on HCache layers is balanced by using the complementary method on L_O layers.

State Partition Algorithm

Given hardware profiling results (IO_H for hidden state transmission, IO_KV for KV cache transmission, C_Token for token recompute, C_H for hidden-to-KV compute), the optimal partition is:

If C_H > IO_H:   L_H = ceil(N_Layer * IO_KV / (IO_KV + C_H - IO_H))
If C_H ≤ IO_H:   L_H = ceil(N_Layer * C_Token / (C_Token + IO_H - C_H))
L_O = N_Layer - L_H

This ensures computation and transmission finish almost simultaneously, eliminating pipeline bubbles. The paper uses layer-wise partition over token-wise partition because cuBLAS GEMM kernels are well-optimized for standard matrix sizes, and token-wise partitions create irregular matrices that degrade cuBLAS performance.

Storage Management

Chunk-Based Storage Format

Hidden states for each layer are split into fixed-size 64-token chunks. Chunks from the same layer are colocated in contiguous blocks, maximizing sequential read bandwidth. Chunks are distributed round-robin across multiple SSDs to aggregate read bandwidth (e.g., 4x PM9A3 SSDs = ~28 GB/s aggregate read).

This format is optimized for restoration (token-before-layer order), not generation (layer-before-token order), which is why a two-stage saving strategy is needed.

Two-Stage State Saving

Saving hidden states during inference happens off the critical path:

  1. Stage 1 (GPU): A single cudaMemcpy snapshots the entire batch's hidden states from GPU to host memory. This frees the GPU buffer for the next layer immediately.
  2. Stage 2 (CPU): A background daemon running on 8 CPU threads reorganizes the hidden states into chunk format and flushes them to SSDs. Small writes are accumulated into large sequential writes that SSDs favor.

No stalling occurs during state saving. For Llama2-13B, prefilling 1,024 tokens at one layer generates ~10MB of hidden states in 3ms, well below PCIe bandwidth constraints.

GPUDirect Storage for Restoration

For restoration, HCache uses SPDK and GDRCopy to realize GPUDirect Storage: NVMe read commands target the GPU's BAR (Base Address Register) space directly, achieving P2P copies from SSD to GPU memory without host memory bounce buffers. This eliminates redundant memory copies and minimizes latency.

Multi-GPU support: With tensor parallelism, each GPU reads disjoint shards (split by token) of hidden states concurrently, aggregating read bandwidth across all GPUs. GPUs then perform an all-gather via NVLink to reconstruct the full hidden state for their partition of layers.

Implementation

System Details

HCache is implemented in 5,731 lines of CUDA, C++, and Python, built on DeepSpeed-MII v0.2.0 with SplitFuse scheduling. Key implementation choices:

  • cuBLAS: Used for projecting hidden states to KV cache (H * W_k, H * W_v). Batch sizes are rounded to cuBLAS-optimized sizes.
  • Custom CUDA kernel: Applies RoPE positional embeddings to recomputed keys and copies results to KV cache.
  • Pipelined streams: Two dedicated CUDA streams -- one for upstream state transmission, one for downstream state saving. cudaEvent coordinates ordering between streams.
  • Request scheduling: Extends continuous batching with a restoration phase. Evicted requests first undergo bubble-free restoration, then proceed to standard prefill + decode.

Sensitivity Analysis

Varying GPU Platforms

HCache was tested across A100, A30, RTX 4090, L20, and H800 GPUs with DRAM as storage backend:

  • vs KV Offload: 1.33-1.81x speedup across all GPUs
  • vs Recomputation: 5.04-9.05x speedup

Platforms with low compute power (A30) are least friendly to HCache since the GEMM recomputation takes longer. Even so, HCache still achieves 1.33x over KV offload on A30.

Varying Storage Bandwidth

With 1-4 PM9A3 SSDs (6.9 GB/s each), HCache delivers 1.7-2.6x over KV offload and 2.3-6.1x over recomputation. With fewer disks, IO can become the bottleneck, but the bubble-free scheduler automatically allocates more layers to token recomputation or KV offload to compensate.

Varying Sequence Length

From 1K to 16K tokens on A100 with 4 SSDs:

  • Token recomputation degrades by 28% (7B model) due to quadratic attention scaling
  • KV offload scales linearly (proportional to N_seq)
  • HCache also scales linearly (both hidden state size and GEMM cost are O(N_seq))

The speedup of HCache over recomputation therefore increases with sequence length, making it especially valuable for long-context applications.

Significance and Limitations

Novel State Representation

HCache introduces a third approach to LLM state management beyond the existing binary choice of "store KV" or "recompute from tokens." By caching hidden states, it simultaneously reduces both computation (6x fewer FLOPs) and IO bandwidth (2x less data), exploiting a previously unused sweet spot in the compute-IO design space.

Broad Compatibility

Works with any MHA-based LLM without model modifications. Supports over 20,000 HuggingFace models including Llama2, Phi2, Gemma, and Qwen1.5. The approach is completely lossless with zero accuracy degradation -- a critical advantage over lossy compression methods.

Production-Ready Systems Design

5,731 lines of CUDA/C++/Python built on DeepSpeed-MII. Features include GPUDirect Storage via SPDK+GDRCopy, cuBLAS-optimized batch GEMM, pipelined state transmission with cudaEvent synchronization, and continuous batching integration with SplitFuse scheduling.

Complementary to Other Techniques

HCache is orthogonal to GPU-resident KV cache reuse (PromptCache, SGLang), KV compression (quantization, token dropping), and MQA/GQA architectures. These techniques can be combined with HCache for additional speedups on the cache-miss path.

Limitations: (1) HCache's advantage diminishes on platforms where IO speed far exceeds compute (e.g., H800 with 64 GB/s DRAM bandwidth), though it remains at least 1.15x faster than KV offloading even in high-skewness workloads. (2) Saving hidden states during the generation phase incurs ~4% TBT overhead due to cudaMemcpy. (3) For MQA/GQA architectures where the KV cache is already compressed (fewer KV heads), the 2x size advantage of hidden states over KV is reduced. (4) The current implementation targets single-node multi-GPU setups; distributed cross-node scenarios require WAN-aware optimizations.

Ablation Studies

Bubble-Free Scheduler Impact

Three variants were compared under three hardware settings (IO-sufficient A30+7B+4SSD, compute-sufficient A100+7B+1SSD, balanced A100+13B+4SSD):

  • HCache-O: Uses hidden states only, no complementary methods. Incurs pipeline bubbles on resource-skewed platforms.
  • Naive Hybrid: Uses bubble-free scheduler to mix token recomputation and KV offload without hidden states. Outperforms individual methods by 1.28-1.42x.
  • HCache (full): Combines hidden states with the bubble-free scheduler for complementary methods. Consistently outperforms Naive Hybrid by 1.35-1.64x in IO-sufficient settings and 1.45-2.66x overall vs KV offload.

The bubble-free scheduler is critical on resource-skewed platforms. Without it, HCache-O is 13% slower than KV offload in IO-sufficient settings because recomputation bubbles waste IO resources.

State Partition Methods

Comparing layer-wise vs token-wise partition for the 13B model with 1024-token history on A100+1SSD:

  • Token-wise (naive): 12% slower than layer-wise due to irregular cuBLAS matrix sizes
  • Token-wise + round-up: 7% slower -- rounding to nearest optimized cuBLAS size helps but unbalanced compute/transmission persists
  • Layer-wise: Optimal. All tokens recomputed together at each layer, forming standard matrix sizes well-optimized by cuBLAS

Layer-wise partition produces a schedule using hidden states for 31 layers and KV offload for 1 layer (7B model), or 36 hidden + 4 KV (13B model).

Two-Stage State Saving Evaluation

Comparing HCache's two-stage saving (cudaMemcpy to host + background daemon to SSD) with DirectIO (saving hidden states directly to SSD from GPU):

  • At small batch sizes (≤5), DirectIO and HCache have similar TBT since IO time is less than decode time
  • At batch size 16, DirectIO's TBT is 34% higher for the 7B model because one batch's hidden states cannot be flushed to SSD before the next layer's decode begins
  • HCache's two-stage approach avoids this bottleneck: cudaMemcpy to host DRAM is fast and fixed-cost, while the background daemon handles the slower SSD writes off the critical path

No stalling occurred during hidden state snapshots in any experiment. For Llama2-13B, prefilling 1,024 tokens generates ~10MB of hidden states per layer in ~3ms, well below PCIe bandwidth constraints.

Related Work and Context

Stateful LLM Optimizations

GPU KV Reuse: PromptCache, SGLang, and ChunkAttention cache and reuse KV on GPU for shared prefixes across requests. These optimize the GPU cache hit path and are complementary to HCache, which optimizes the cache miss path (restoration from storage).

Tiered KV Storage: AttentionStore, RAGCache, Pensieve, and APIServe maintain KV caches across GPU, DRAM, and SSD tiers. HCache differs by storing hidden states instead of KV cache, which is fundamentally smaller and enables faster restoration through GEMM-only recomputation.

Offline Inference: FlexGen offloads both KV cache and output activations for batched inference. HCache exploits hidden states specifically for online serving with restoration latency as the primary goal.

LLM State Compression

Quantization: GPTQ, SmoothQuant, KIVI, and Atom compress KV caches from 16-bit to 4-bit or lower. These methods reduce hidden state size as well, and can be combined with HCache for even smaller storage footprint.

Token Dropping: H2O, SnapKV, and Gist Tokens drop less-important tokens from KV cache. These reduce the number of tokens that need to be stored and can be integrated into HCache to reduce both transmission and computation overhead.

MQA/GQA: These architecture modifications reduce the KV cache by using fewer key-value heads. HCache can currently support all MHA-based LLMs using the standard attention mechanism. Adapting to MQA/GQA models where KV is already compressed would change the hidden-to-KV size ratio but the core approach remains valid.

GPU KV Reusing with HCache

In production systems, GPU KV cache reuse (keeping frequently accessed KV caches on GPU) and HCache (restoring evicted KV caches from storage) are complementary. Using the L-Eval dataset with Zipfian skewness workloads:

  • With uniform arrivals (alpha=1.0), cache hit ratio is ~15% and HCache is 1.67x faster than KV offloading
  • At skewness alpha=2.0, cache hit ratio rises to 94%, reducing restoration frequency. HCache still outperforms KV offloading by 1.15x and recomputation by 1.98x

Even with high cache hit rates, HCache provides value for the remaining cache misses, which tend to involve the longest and most expensive contexts.
