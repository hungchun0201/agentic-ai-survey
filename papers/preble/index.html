Preble: Efficient Distributed Prompt Scheduling for LLM Serving

The first distributed LLM serving platform that co-optimizes KV state reuse and computation load balancing across GPU clusters through a novel Exploitation + Exploration (E2) scheduling algorithm.

Vikranth Srivatsa*, Zijian He*, Reyna Abhyankar, Dongming Li, Yiying Zhang

University of California, San Diego

arXiv:2407.00023 GitHub Repository 1.5-14.5x Avg Latency 2-10x P99 Latency

Overview

Modern LLM workloads are characterized by two trends: prompts are significantly longer than generated outputs (37x to 2494x), and 85-97% of prompt tokens are shared across requests. Existing prefix caching solutions (SGLang, vLLM) operate within a single GPU, but production LLM serving systems are distributed by nature. Preble bridges this gap by extending prefix caching to the cluster level with a hierarchical scheduling architecture and the E2 algorithm.

Key Contributions

Workload Study

First comprehensive study of five real LLM workloads (tool use, embodied agent, programming, video QA, long document QA) plus the Azure LLM trace, revealing that prompts are 37x-2494x longer than outputs and that 85-97% of tokens are shared.

E2 Scheduling Algorithm

A new distributed request scheduling algorithm based on Exploitation + Exploration. E2 co-optimizes KV cache reuse (exploitation) and computation load balancing (exploration) using a prompt-aware load metric that accounts for historical load, eviction cost, and new request cost.

Preble Serving System

The first distributed LLM serving platform targeting long and shared prompts. Features a two-level scheduling hierarchy (global request-level + per-GPU iteration-level), post-assignment load rebalancing, auto-scaling, and prefill-decoding balancing.

Comprehensive Evaluation

Evaluation on two LLMs (Mistral 7B, Llama-3 70B), five workloads, and two GPU clusters (4x A6000, 8x H100) demonstrating significant improvements over SGLang and vLLM.

Repository Analysis

Preble is implemented as a standalone layer on top of SGLang with modifications for prefix-aware distributed scheduling. The repository is a fork of SGLang, adding a preble/ directory that contains the core cluster-level scheduling logic, while the python/sglang/ tree holds the modified local runtime with chunked prefill and radix cache extensions.

Core Module Dependencies

The data flow between modules follows a clear hierarchy:

  1. Server (server.py) receives HTTP requests and pushes them into an async queue
  2. DataParallelRequestRouter selects a routing policy (round-robin, LOR, or custom E2)
  3. GlobalSchedulerWithTime implements E2: inserts tokens into the global LPRadixCache, computes exploitation vs exploration, calculates prompt-aware load via SlidingWindowHistogram
  4. ModelDetails / EndpointRuntimeInterface dispatches the request to the selected GPU's SGLang/vLLM endpoint
  5. Local RadixCache + Scheduler performs iteration-level scheduling with priority-based wait queue ordering

System Architecture

Preble uses a two-level hierarchical scheduling architecture. A centralized global scheduler performs coarse-grained, request-level scheduling across GPUs, while per-GPU local schedulers perform fine-grained, iteration-level scheduling. This design provides a centralized view for E2 decision-making while allowing each GPU to adapt quickly to local conditions.

Global Scheduler Benefits

  • Centralized global view of cluster load and prompt caching state
  • Coarse-grained request-level scheduling scales to hundreds of GPUs
  • Lock-free radix tree access (read-only except GPU assignment + ref count)
  • Background threads for rebalancing and eviction bookkeeping

Local Scheduler Benefits

  • Fine-grained iteration-level scheduling adapts to real-time GPU conditions
  • Maintains per-GPU radix tree and priority-based wait queue
  • Chunked prefill for long prompts (budget = 512 tokens per chunk)
  • Reports eviction events asynchronously to global scheduler

E2 Scheduling Algorithm

E2 stands for Exploitation + Exploration. The core insight is that greedily maximizing prefix cache hits can lead to load imbalance, while purely balancing load ignores cache reuse opportunities. E2 dynamically chooses between exploiting existing cached prefixes and exploring new GPUs based on a principled cost comparison.

EXPLOIT

EXPLORE

Cost = Li + Mi + Pi

Prompt-Aware Load Metric (Exploration Mode)

When E2 explores, it selects the GPU with the lowest total cost, defined as:

Total Cost = Li + Mi + Pi

Component Definition Implementation Detail
Li - Historical Load Overall GPU computation load in the history window H (default 3 minutes). Sum of estimated prefill + decode time for all requests in H. Tracked in SlidingWindowHistogram. Prefill estimated via regression on non-matching tokens. Decode estimated via observed average TPOT.
Mi - Eviction Cost Recomputation cost of evicting KV cache entries to make room for the new request. Weighted by hit rate and sharing count of evicted nodes. Computed by virtual_evict_for_routing() which simulates LRU eviction without actually evicting.
Pi - New Request Cost Prefill time for the non-cached (missed) tokens of the new request Rk on GPUi. Estimated via profiled regression function prefill_time(num_tokens, context_length).
i
Key insight: All three cost components are calculated as GPU computation time using profiled regression functions rather than raw token counts. This unifies prefill cost, decode cost, and eviction cost into a single comparable metric. Shared prompt tokens are treated as zero cost (since their KV is already cached), naturally accounting for prefix sharing potential.

Implementation of the E2 Decision

def runtime_selector(self, text, request_id, input_ids, sampling_params, ...):
    # 1. Insert into global radix tree and find matching prefix
    leaf_node = self.cache.insert(tuple(input_ids), split_nodes=split_nodes)
    important_node = self.get_important_node(leaf_node)

    # 2. EXPLOITATION: matched prefix > remaining tokens
    if leaf_node.num_tokens < leaf_node.context_so_far:
        # Get GPUs that cache the parent prefix
        gpu_selected = self.get_parent_gpu_allocation(leaf_node)
        if len(gpu_selected) > 1:
            # Among caching GPUs, pick lowest load
            runtime_idx = self.calculate_min_load_cost(leaf_node, gpu_selected)
        else:
            runtime_idx = list(gpu_selected)[0]

    # 3. EXPLORATION: remaining tokens dominate, search all GPUs
    else:
        runtime_idx = self.calculate_min_load_cost(
            leaf_node, selected_gpus=range(self.num_gpus))

    # 4. Update GPU allocations and histogram
    self.update_gpu_allocation_for_parent(leaf_node, {runtime_idx})
    self.cache.update_allocated_size(leaf_node, runtime_idx)
    self.histogram.update(datetime.now(), important_node, leaf_node, runtime_idx, ...)

    # 5. Post-assignment: eviction + rebalancing
    if self.enable_eviction:
        self.handle_eviction(runtime_idx)
    if self.enable_rebalancing:
        self.handle_important_node_stealing(runtime_idx)
    return runtime_idx

Post-Assignment Load Adjustment

Load Rebalancing (Node Stealing)

When the most loaded GPU exceeds 1.5x the lightest GPU's load, the scheduler migrates prefix subtrees from heavy to light GPUs. It uses a min-heap of node costs and greedily steals the cheapest nodes until balance is restored.

// Trigger condition:
if larger_cost > 1.5 * smaller_cost:
    // Steal smallest-cost nodes first
    while larger - cost > smaller + cost:
        migrate(node, heavy_gpu, light_gpu)

Autoscaling (Prefix Replication)

When a single prefix's load exceeds what one GPU can handle (detected by TTFT doubling between the first and second half of the monitoring window), the scheduler replicates the prefix to an additional GPU, splitting future requests between them.

// TTFTWindowedOverloadedDetector
if avg_second_half >= 2 * avg_first_half:
    // Node is overloaded: replicate
    gpu_allocations[node].add(light_gpu)
    // Future requests split across GPUs

Prefill-Decode Balancing

Preble leverages the insight that a fully cached request acts as a decode-only unit, while an uncached request acts as a prefill-heavy unit. By mixing requests with different sharing levels, the global scheduler naturally balances the prefill-to-decode ratio on each GPU. If a GPU is heavily loaded with decode-phase requests, the scheduler directs new exploration requests to it (since they incur prefill computation that uses idle compute capacity).

Implementation Deep Dive

Global Prefix Radix Tree

The LPRadixCache extends a standard radix tree with distributed metadata. Each TreeNode tracks per-GPU reference counters, cached/evicted GPU sets, and context-length information necessary for the E2 cost calculation. 

class TreeNode:
    def __init__(self, num_nodes):
        self.id = uuid4()
        self.children = defaultdict(TreeNode)    # radix children
        self.parent = None
        self.value = None                       # token IDs in this node
        self.ref_counter = [0] * num_nodes     # per-GPU active ref count
        self.evicted_gpus = set()              # GPUs that evicted this node
        self.cached_gpus = set()               # GPUs currently caching this node
        self.decode_length = deque()           # observed decode lengths
        self.context_length = 0               # total tokens from root to here
        self.depth = 0                          # depth in the tree

    @property
    def context_so_far(self):              # tokens from ancestors (matched prefix)
        return self.context_length - self.num_tokens

Sliding Window Histogram

The SlidingWindowHistogram maintains per-node, per-GPU cost tracking over a configurable window (default 3 minutes). It computes the prompt-aware load metric used in E2's exploration mode.

Cost Computation

def get_node_cost(self, node, gpu, tpot):
    # Prefill cost: miss_rate * request_count * profiled_time / num_sharing_GPUs
    prefill_cost = self.prev_mis_rates[node] * \
                   self.node_to_count[node] * \
                   prefill_time(node.num_tokens, node.context_length) / \
                   len(self.gpu_allocations.get(node))

    # Decode cost: active_requests * output_length * time_per_output_token
    active_requests = node.ref_counter[gpu]
    decode_cost = active_requests * output_len * tpot

    return prefill_cost + decode_cost

Virtual Eviction for Routing

A distinctive feature of Preble is virtual eviction: when computing the exploration cost, the scheduler simulates LRU eviction on each GPU without actually modifying the cache state. This estimates the recomputation penalty of evicting existing cached prefixes. 

def virtual_evict_for_routing(self, leaf_node, runtime_selected):
    # Heuristic: eviction size = new leaf's tokens (GPU memory assumed full)
    num_to_evict = leaf_node.num_tokens
    evicted_nodes = self.cache.virtual_lru_eviction(num_to_evict, runtime_selected)
    eviction_cost = 0
    for victim in evicted_nodes:
        # Cost = miss_rate * ref_count * prefill_time (for important/large nodes only)
        eviction_cost += self.histogram.get_eviction_prefill_cost(
            victim, runtime_selected, self.is_large_node(victim))
    return eviction_cost

Important Node Concept

Preble distinguishes between "important" (large) nodes and small nodes in the radix tree. A node is considered "important" when its own token count exceeds the context from its ancestors: 

def is_large_node(self, node):
    return node.num_tokens > node.context_so_far
    # Equivalently: this node's tokens > all ancestor tokens combined

Important nodes represent the "key portion" of a prompt -- the segment that dominates computation. Rebalancing and autoscaling decisions operate at the granularity of important nodes, and eviction cost is only computed for important nodes (small nodes have zero eviction cost).

Server Entry Point

The preble/server/server.py exposes a FastAPI server that serializes request scheduling through an async queue. A single background coroutine processes the queue, calling request_router.select_runtime() for each request to ensure thread-safe access to the global radix tree. 

# server.py - Simplified flow
async def process_runtime_selection():
    while True:
        obj, request_id = await runtime_request_queue.get()
        runtime_id = request_router.select_runtime(
            text=obj.text,
            input_ids=obj.input_ids,
            sampling_params=obj.sampling_params.dict(),
            ...
        )
        # Signal the waiting request handler with the selected GPU
        runtime_events[request_id][0].set()

Paper vs Implementation

A close comparison between the ICLR 2025 paper and the open-source implementation reveals both faithful reproductions and practical divergences. The core E2 algorithm is well-implemented, while some paper features are present in alternative forms or partially implemented.

Paper Feature Implementation Status Details
E2 Exploitation/Exploration decision Implemented Faithfully implemented in runtime_selector(). Uses leaf_node.num_tokens < leaf_node.context_so_far as the decision boundary.
Global prefix radix tree Implemented LPRadixCache with split, insert, match, per-GPU ref counters, and allocated size tracking.
Prompt-aware load metric (Li + Mi + Pi) Implemented calculate_min_load_cost() combines histogram-based allocation cost with virtual eviction cost.
3-minute sliding window (H) Implemented SlidingWindowHistogram(window_duration=timedelta(minutes=3)). Entries expire automatically.
Virtual eviction for routing Implemented virtual_lru_eviction() simulates eviction without modifying cache state. Assumes GPU memory is always full.
Post-assignment load rebalancing Implemented handle_important_node_stealing() with threshold HIGH_LOAD_THRESHOLD = 1.5 and recursive multi-GPU balancing.
Autoscaling (prefix replication) Implemented TTFTWindowedOverloadedDetector checks if TTFT doubled in the second half of the window. Triggers only when a single dominant node exists.
Prefill-decode balancing Partial Mentioned in the paper as directing exploration requests to decode-heavy GPUs. The implementation includes decode cost tracking in the histogram but does not have explicit prefill/decode ratio checks in the global scheduler. The local scheduler uses chunked prefill.
Priority-based wait queue (P groups) Partial The local scheduler in the modified SGLang supports fcfs-mpq (multi-priority queue) scheduling heuristic but the P-group proportional selection described in the paper is not fully exposed.
Lock-free radix tree access Partial The paper claims lock-free read access with atomic GPU assignment and ref-count updates. The implementation uses a threading.Lock() for the entire runtime_selector() call. The perf variant (GlobalSchedulerWithTimePerf) removes the lock.
Profiled regression functions Implemented Hardware-specific profiling via LP_mistral_7b_A6000_sglang_extend_flashinfer and Llama-3 70B H100 variants. Imported from model_equations.
Iterative eviction feedback Implemented Local RadixCache tracks EvictionData per iteration, reported asynchronously to global scheduler via ZMQ. Toggleable via enable_iterative_eviction.
Multi-backend support (SGLang + vLLM) Implemented Abstracted via EndpointRuntimeInterface with concrete classes for SGLang, vLLM, SSH, URL, and Simulation backends.
Hardcoded model / GPU assumptions Limitation The global scheduler hardcodes max_tokens_gpu = [198516] and initializes TPOT at 0.15s. Prefill regression imports are model-specific. Changing models requires code edits.

Notable Implementation Observations

Two Scheduler Variants

The repository contains two versions of the global scheduler: GlobalSchedulerWithTime (with threading lock) and GlobalSchedulerWithTimePerf (performance-optimized, lock-free). The perf variant also includes an inline TTFTWindowedOverloadedDetector and uses incremental cost tracking (per_node_per_gpu_allocation_cost) instead of recomputing full allocation costs.

Exploration Policy Breadth

The CustomPolicyType enum reveals an extensive experimental history with 20+ scheduling variants tested during development: from basic memory schedulers to histogram-based approaches, LP-based schedulers, oracle policies, and the final GlobalScheduler variants. This suggests significant iteration before arriving at E2.

Hit Ratio Feedback Loop

The ModelDetails class includes a background loop that polls each GPU's /windowed_prefix_hit_ratio endpoint every 0.5 seconds. When the highest hit ratio exceeds 0.7, the system uses it as a hint for the E2 decision (runtime_id_with_highest_hit_rate). This runtime feedback mechanism is not discussed in the paper.

Simulation Infrastructure

A full ServerRuntimeSimulator and Simulation class enable offline evaluation without GPUs. Combined with eventsim.py and sim.py, this provides a complete simulation environment for algorithm development and testing -- a significant engineering investment not mentioned in the paper.

!
Deployment consideration: The implementation has hardcoded references to mistralai/Mistral-7B-v0.1 as the default tokenizer and model-specific profiling functions. Deploying with a different model requires updating the prefill regression imports in global_scheduler_with_time.py and potentially re-profiling. The max_tokens_gpu capacity is also hardcoded at 198,516 tokens.

Summary

Preble makes a strong contribution by identifying and solving the cluster-level prefix caching problem for distributed LLM serving. The E2 algorithm's exploitation/exploration framework is well-motivated by the workload study and elegantly balances cache reuse against load distribution. The prompt-aware load metric, which unifies historical load, eviction cost, and new request cost into GPU computation time, is a principled approach that avoids the pitfalls of simpler heuristics.

The implementation faithfully captures the paper's core contributions, with the global radix tree, E2 decision logic, sliding window histogram, rebalancing, and autoscaling all present. The addition of a simulation infrastructure, multiple backend support, and a FastAPI-based server makes the system practical for both research and deployment. Areas for improvement include reducing model-specific hardcoding, improving the lock granularity for better scalability, and more fully implementing the priority-based wait queue scheduling described in the paper.
