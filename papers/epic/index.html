01Overview: Position-Independent Caching

Modern LLM serving relies on Context Caching (CC) to avoid redundant prefill computation. Existing prefix-based caching (vLLM, SGLang, Gemini) reuses KV vectors only when requests share an exact prefix match. This breaks down in RAG, multi-document QA, and few-shot learning where identical document chunks appear but are preceded by different prefixes.


Position-Independent Caching (PIC) enables modular reuse of cached KV vectors regardless of what precedes them. EPIC formalizes PIC via a two-step framework inspired by classical software compilation: first compile chunks independently, then link them together with selective recomputation to restore accuracy. The core algorithm, LegoLink, recomputes only the first k tokens of each chunk boundary, achieving up to 8x TTFT reduction and 7x throughput improvement.

Key Insight

Just as position-independent code can be loaded at any memory address, position-independent KV caches can be concatenated in any order -- enabling modular, order-agnostic reuse of precomputed context. The "linking" step resolves cross-chunk attention discrepancies analogous to symbol resolution in a linker.

Explicit Caching Paradigm

Unlike implicit caching systems (vLLM prefix caching, SGLang RadixAttention) that manage cache internally via hash tables or radix trees, EPIC adopts an explicit caching paradigm with user-facing cache IDs. This is analogous to Google Gemini's and Kimi's context caching APIs, giving users fine-grained control -- particularly beneficial in RAG scenarios where document ordering varies between invocations.

03The LegoLink Algorithm

Why "LegoLink"?

Cached KV chunks are assembled like LEGO bricks -- independently compiled, then linked together. The key innovation: only recompute the first k tokens of each chunk (except the first chunk), exploiting the static attention sink phenomenon where initial tokens disproportionately absorb attention scores.

LegoLink Recomputation Steps (per layer)

1
Compute the embedding matrix E of the selected k' tokens (k from each non-first chunk, shape (k', d)).
2
Compute new Q, K, V matrices for these k' tokens: Q = EW_Q, K = EW_K, V = EW_V.
3
Expand K and V by incorporating cached KV vectors of the N - k' unselected tokens, forming K_exp and V_exp (both shape (N, d)).
4
Compute attention: A = softmax(Q * K_exp^T * MASK) where MASK ensures k' tokens only attend to tokens before them. Output: O = A * V_exp * W_O.

Complexity Advantage

Fully RecomputeO(N2)None

Since k is fixed (typically 2-16) and does not grow with N, LegoLink achieves linear complexity in the link step, while FR and CacheBlend remain quadratic.

LegoLink-0: Zero Linking Overhead

A variant that eliminates all recomputation in the link step:

C
Compile step: Prepend 4 dummy tokens (e.g., BOS tokens) to each chunk, then discard their KV vectors after compilation. This eliminates "attention sinks" before linking.
L
Link step: Skip recomputation entirely -- zero runtime overhead. The attention sink phenomenon disappears from the middle of the sequence.

Implementation Note

The README states: "This prototype focuses solely on functionality rather than performance." The compile step collects KV vectors via hack_kv (cloned tensors from the attention layer), which is a demonstration approach. A production system would use the vLLM paged KV cache directly. The authors indicate a full production system (similar to LMCache) is under active development.

05Core Concepts

The Attention Sink Phenomenon

When chunks are compiled independently (each starting from position 0), their initial tokens disproportionately absorb attention scores -- a well-documented phenomenon called the "attention sink." After concatenation, these artificial sinks from non-first chunks prevent the model from attending to genuinely relevant context.

Static Sparsity (LegoLink)

LegoLink exploits static attention sparsity: the observation that the initial tokens of each independently-compiled chunk always serve as attention sinks. This is predictable before runtime.

  • Token selection is predetermined from chunk boundaries
  • No runtime attention analysis needed
  • Recompute the first k tokens of each non-first chunk
  • Complexity: O(kN) where k is typically 2-16

Dynamic Sparsity (CacheBlend)

CacheBlend uses dynamic attention sparsity: it recomputes all KV vectors in layer 1, compares old vs. new attention maps, then selects the top r% of divergent tokens for subsequent layers.

  • Token selection requires runtime comparison
  • Full recomputation at layer 1 adds overhead (16-64% of TTFT)
  • Recompute top-r% across all tokens by attention divergence
  • Complexity: O(rN2), remains quadratic

Selective Recomputation in Detail

The partial attention mask is constructed in _make_partial_bias_gqa / _make_partial_bias_mha. It creates a causal mask from the full sequence length, then selects only the rows corresponding to the recomputed positions:

vllm/attention/backends/xformers.py:872-905
def _make_partial_bias_gqa(cache_fuse_metadata, device, num_kv_heads, num_queries_per_kv): seq_len = cache_fuse_metadata['org_seq_len'] padded_len = (seq_len + 7) // 8 * 8 dtype = cache_fuse_metadata['kv_cache_dtype'] imp_indices = cache_fuse_metadata['imp_indices'] # Create full upper-triangular causal mask attn_mask = torch.triu(torch.ones(padded_len, padded_len, dtype=dtype, device=device), diagonal=1) attn_mask = (attn_mask * torch.finfo(dtype).min) .view(1, 1, 1, padded_len, padded_len) # Select only the rows for recomputed positions attn_mask = attn_mask[:, :, :, imp_indices] attn_mask = attn_mask.expand(1, num_kv_heads, num_queries_per_kv, -1, -1) return attn_mask[:, :, :, :, :seq_len]

This ensures that only the k' recomputed tokens serve as queries while attending to all N key-value vectors, implementing the equation: A = softmax(Q * KexpT * MASK).

07Agentic AI Relevance

Why EPIC Matters for Agents

In agentic AI systems, LLMs are invoked repeatedly with overlapping but reordered context. Traditional prefix-based caching fails because the ordering of documents varies between invocations. EPIC's position-independent caching directly addresses this.

RAG Pipelines

Retrieved documents differ per query but are individually cacheable. EPIC enables caching each document's KV vectors once, then linking them in any retrieval order -- eliminating redundant computation across queries targeting the same knowledge base.

Multi-Document QA

Agents analyzing multiple documents (legal contracts, medical records, code files) can compile each document independently. Different analysis tasks reorder or subset these documents without recomputing any document's KV cache.

Few-Shot Prompting

In-context learning examples remain immutable across invocations. EPIC caches each example independently, allowing the system prompt and example ordering to change without invalidating cached representations.

Agent-Specific Advantages

ScenarioPrefix CachingEPIC (PIC)
Same docs, different order Cache miss -- full recompute Cache hit -- only link-step needed
Docs A,B,C then A,C,D Only "A" prefix reused A and C fully reused, only D compiled

Multi-turn agent dialogue Only common prefix reused All immutable context chunks reused
Concurrent users, same docs Each user prefix may differ Shared doc cache, per-user link
Architectural Pattern for Agents

EPIC suggests a compile-once, link-many pattern for agentic systems: maintain a pre-compiled KV cache library for all knowledge documents, tool descriptions, and system instructions. Each agent invocation becomes a lightweight link step, assembling the required context and paying only O(kN) for recomputation. This is especially valuable when agent tool-calling patterns cause different document subsets to appear in different orders.

What the Repo Proves

  • The LegoLink algorithm is correct -- static selection of initial tokens is sufficient to restore accuracy
  • The attention sink phenomenon is real and consistently exploitable at chunk boundaries
  • The partial attention mask implementation (selecting Q-rows by imp_indices) is functionally validated
  • Both MHA and GQA attention configurations are supported
  • The LegoLink-0 variant with dummy prefix stripping achieves zero linking overhead
  • All 6 benchmark datasets with proper metrics are reproducible

Gaps vs. Paper Claims

  • Demo prototype only -- explicitly stated in README as "focusing solely on functionality rather than performance"
  • No production cache management -- KV vectors stored as cloned tensors, not in paged KV cache
  • No serving API -- the OpenAI-compatible endpoints described in Section 3 are not implemented
  • Batch size limited to 1 -- partial bias functions contain "FIXME: Now only focus on bsz=1" comments

  • Production system "similar to LMCache" mentioned but not yet open-sourced
Overall Assessment

The repository faithfully implements the core algorithmic contributions of the paper (LegoLink, LegoLink-0, CacheBlend comparison) with sufficient fidelity to validate the published results. The system-level contributions (EPIC serving system, explicit caching APIs, scheduler integration) remain at the design/description level. This is consistent with the authors' transparent disclosure that the repo is a "demonstration system" with a full production system under development. The 2K lines of modifications over vLLM 0.7.0 cleanly demonstrate the PIC concept while the comprehensive benchmark suite enables end-to-end reproducibility.

Key Implementation Patterns

Metadata-Driven Control Flow

The entire PIC behavior is controlled by a single dictionary (cache_fuse_metadata) threaded through the model. The kvlink field acts as a tri-state switch: None (off), [positions...] (compute bias), [] (bias already computed). This pattern enables runtime switching between Naive, FR, CacheBlend, and LegoLink without model changes.

Two-Pass Inference

The evaluation runs two LLM invocations per test case: (1) a compile pass with collect=True that captures raw KV vectors via hack_kv, and (2) a link pass with kvlink=positions that performs selective recomputation. This cleanly separates the PIC phases while reusing the same vLLM engine.
