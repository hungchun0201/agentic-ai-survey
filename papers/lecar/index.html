LeCaR

Driving Cache Replacement with ML-based Regret Minimization — dynamically blending LRU and LFU using online learning

HotStorage'18 (USENIX) Regret Minimization LRU + LFU Blend Foundation for CACHEUS

Paper Overview

LeCaR (Learning Cache Replacement) is a pioneering cache replacement algorithm that uses regret-based online learning to dynamically weight two fundamental eviction experts: LRU (recency) and LFU (frequency). Instead of committing to one policy, LeCaR maintains a probability distribution over both and learns from mistakes using discounted regret signals.

Core Innovation
Regret Learning

When an evicted page is re-requested, the policy that evicted it receives a negative reward proportional to the time elapsed

The Core Idea

Traditional cache policies make a permanent bet: LRU assumes recency predicts future access, LFU assumes frequency does. But workloads are non-stationary -- sometimes recency matters, sometimes frequency matters, sometimes neither. LeCaR avoids this rigid commitment by maintaining a learned probability distribution over both policies.

⚡
Key Insight: Learning from Mistakes When a page evicted by LRU is re-requested and found in LRU's ghost history, LRU receives a negative reward -- the weight shifts toward LFU. The reward magnitude is discounted by time: recent mistakes are penalized more heavily than distant ones.

Algorithm Walkthrough

  1. Request arrives for page P. If P is in the cache, it is a HIT: update LRU position (move to MRU end) and increment frequency in LFU heap. No eviction needed.
  2. Cache MISS -- check ghost histories. If P is found in LRU history, LRU made a mistake: compute regret as -(discount_rate ^ elapsed_time) and apply to LRU's weight. Delete from history, carry over frequency. Vice versa for LFU history.
  3. Eviction selection. If cache is full, select victim by sampling from {LRU, LFU} using weights W. If LRU is selected, evict the least recently used page. If LFU is selected, evict the least frequently used page. If both point to the same page, evict it without assigning to either policy's history.
  4. Insert new page. Add the new page to both the LRU deque (at MRU end) and the LFU heap (with frequency 1 or carried-over frequency from history).
  5. Record eviction. Add the evicted page to the corresponding policy's ghost history with an eviction timestamp. If history is full, evict the oldest ghost entry.

The Regret Mechanism

Weight Update Rule (Multiplicative Weights)
Wi = Wi * exp(learning_rate * rewardi)
Then normalize: W = W / sum(W)

Discount Rate

The discount rate controls how much we penalize old mistakes versus recent ones. It is set to:

discount_rate = 0.005(1/C)
where C is the cache size

This ensures the discount is relative to the cache size: a re-request after C time steps receives a discount of 0.005, while a re-request after 1 step receives almost full penalty. The choice of base (0.005) means that after a full cache cycle, the regret has decayed to 0.5% of its original value.

Learning Rate

LeCaR uses a fixed learning rate of 0.45. This is deliberately aggressive -- the algorithm needs to adapt quickly to workload phase changes. The multiplicative weights update ensures that:

  • Weights are always valid probabilities (sum to 1)
  • Neither weight drops below 0.01 (clamped)
  • Neither weight exceeds 0.99 (clamped)
  • The response to regret is exponential, not linear

Regret Computation

def miss(self, oblock):
    freq = 1
    if oblock in self.lru_hist:
        # LRU made a mistake: it evicted this page and it came back
        entry = self.lru_hist[oblock]
        freq = entry.freq + 1           # carry over frequency
        del self.lru_hist[oblock]

        # Compute discounted regret: recent mistakes hurt more
        reward_lru = -(self.discount_rate ** (self.time - entry.evicted_time))
        self.adjustWeights(reward_lru, 0)  # penalize LRU only

    elif oblock in self.lfu_hist:
        # LFU made a mistake
        entry = self.lfu_hist[oblock]
        freq = entry.freq + 1
        del self.lfu_hist[oblock]

        reward_lfu = -(self.discount_rate ** (self.time - entry.evicted_time))
        self.adjustWeights(0, reward_lfu)  # penalize LFU only

    # Evict if cache is full, then insert
    if len(self.lru) == self.cache_size:
        evicted, policy = self.evict()
    self.addToCache(oblock, freq)
def adjustWeights(self, rewardLRU, rewardLFU):
    # Multiplicative weights update
    reward = np.array([rewardLRU, rewardLFU], dtype=np.float32)
    self.W = self.W * np.exp(self.learning_rate * reward)
    self.W = self.W / np.sum(self.W)  # normalize to probability distribution

    # Clamp to prevent degenerate weights
    if self.W[0] >= 0.99:
        self.W = np.array([0.99, 0.01], dtype=np.float32)
    elif self.W[1] >= 0.99:
        self.W = np.array([0.01, 0.99], dtype=np.float32)

Data Structures

DequeDict

A doubly-linked list with a hash table overlay. Provides O(1) for all operations: lookup, insertion, deletion, and move-to-back (MRU promotion). The LRU victim is always at the head of the deque.

# O(1) operations
deque[key] = value   # insert at tail (MRU)
deque[key] = value   # update: remove + re-add
first = deque.first() # LRU item (head)
del deque[key]       # O(1) removal

HeapDict

A min-heap with a hash table for O(1) lookup by key. The LFU victim is always at the root (index 0). When frequency changes, the entry is re-heapified in O(log n). Both deque and heap store pointers to the same LeCaR_Entry objects.

# O(log n) operations
heap[key] = entry   # insert + heapify-up
heap[key] = entry   # update + re-heapify
min_entry = heap.min()  # LFU item (root)
del heap[key]       # swap with last + heapify

Complete Implementation

Eviction Decision

def getChoice(self):
    # Stochastic selection based on learned weights
    return 0 if np.random.rand() < self.W[0] else 1

def evict(self):
    lru = self.getLRU(self.lru)    # LRU candidate: head of deque
    lfu = self.getHeapMin()        # LFU candidate: root of heap

    evicted = lru
    policy = self.getChoice()      # sample from W

    # Special case: if both point to the same object, evict it
    # without assigning to any history (policy = -1)
    if lru is lfu:
        evicted, policy = lru, -1
    elif policy == 0:
        evicted = lru
    else:
        evicted = lfu

    # Remove from both data structures
    del self.lru[evicted.oblock]
    del self.lfu[evicted.oblock]
    evicted.evicted_time = self.time
    self.addToHistory(evicted, policy)
    return evicted.oblock, policy

Legacy and Evolution to CACHEUS

LeCaR's design revealed two key limitations that motivated its successor, CACHEUS:

Fixed Learning Rate

LeCaR uses lr=0.45 for all workloads. But the optimal rate depends on workload volatility -- stable workloads need lower rates, phase-changing workloads need higher rates. CACHEUS introduces an adaptive learning rate via stochastic hill climbing.

Fixed Discount Rate

The discount rate 0.005^(1/C) is tuned for general workloads. CACHEUS replaces this with adaptive mechanisms that respond to observed hit rate changes.

Limited Expert Pool

LRU and LFU are fundamental but not comprehensive. CACHEUS introduces SR-LRU (scan-resistant LRU) and CR-LFU (churn-resistant LFU) as enhanced experts that handle adversarial access patterns.

Mathematical Foundation

Multiplicative Weights Framework

LeCaR is grounded in the Multiplicative Weights Update (MWU) method, a powerful framework from online learning theory. The key guarantee: over any sequence of T decisions with N experts, the algorithm's total regret is bounded by O(sqrt(T * ln(N))).

Regret Bound for N=2 Experts
Regret(T) ≤ sqrt(T * ln(2) / 2)
This sublinear regret means the average per-step regret vanishes as T grows.

Why Exponential Updates?

The exponential update W *= exp(lr * reward) has three properties that make it ideal for the caching context:

  • Multiplicative: Weights change proportionally to their current value, preventing any expert from being permanently suppressed
  • Bounded: After normalization, weights remain in [0,1] and sum to 1 -- a valid probability distribution
  • Responsive: Large negative rewards cause rapid weight reduction, enabling fast adaptation to workload changes

Discount Rate Derivation

The discount rate 0.005^(1/C) ensures that the penalty for a mistake decays with time elapsed since eviction. The choice of base 0.005 means:

Time Since EvictionPenalty Factor
1 step0.005^(1/C) -- near 1.0
C/2 steps0.005^(0.5) -- 0.071
C steps0.005 -- negligible
2C steps0.000025 -- effectively zero

Stochastic vs Deterministic Eviction

LeCaR uses stochastic expert selection: it samples from the weight distribution rather than always choosing the higher-weight expert. This is deliberate and important:

⚡
Exploration-Exploitation Tradeoff Deterministic selection (always choose the higher-weight expert) would prevent the algorithm from discovering that the other expert might now be better. Stochastic selection ensures the minority expert is still tried occasionally, enabling faster adaptation when the workload shifts. The weight clamping at [0.01, 0.99] guarantees at least 1% exploration.

Why LeCaR Dominates at Small Cache Sizes

At small cache sizes (relative to working set), the choice between LRU and LFU becomes critical. A wrong choice wastes a large fraction of the cache on suboptimal objects. LeCaR's continuous adaptation ensures it always tracks the better policy, while ARC's linear parameter adjustment is too slow to react.

Scan Workloads

Sequential access patterns that pollute LRU with one-time-use pages. LeCaR shifts weight toward LFU, protecting frequently-used pages from sequential scans.

Popularity Shifts

When popular objects change (new trending content), LFU retains stale popularity counts. LeCaR shifts toward LRU to flush outdated popular objects.

Mixed Workloads

Real workloads typically combine multiple access patterns. LeCaR maintains a balanced weight distribution, effectively creating a dynamic hybrid policy.

Entry Object Structure

class LeCaR_Entry:
    def __init__(self, oblock, freq=1, time=0):
        self.oblock = oblock        # the page/block identifier
        self.freq = freq            # access frequency counter
        self.time = time            # last access timestamp
        self.evicted_time = None   # when evicted (for regret)

    # Used by HeapDict for min-heap ordering
    def __lt__(self, other):
        if self.freq == other.freq:
            return self.oblock < other.oblock  # tie-break by ID
        return self.freq < other.freq          # lower freq = smaller
💡
Shared Object References A crucial implementation detail: the same LeCaR_Entry object is stored in both the DequeDict (for LRU order) and the HeapDict (for LFU order). When the entry is updated (e.g., frequency incremented), both data structures see the change immediately. The if lru is lfu check in the eviction code uses Python's identity check (is) to detect when both experts nominate the same victim.

Design Rationale and Tradeoffs

Why Two Experts Are Sufficient

LeCaR deliberately restricts itself to just two experts: LRU and LFU. This is not a limitation but a conscious design choice grounded in both theory and practice.

Theoretical Coverage

LRU captures temporal locality (recent access predicts future access), while LFU captures frequency locality (popular items remain popular). Together they span the two fundamental dimensions of cache-worthiness that all classical algorithms attempt to balance.

Regret Bound Scaling

The regret bound for multiplicative weights is O(sqrt(T * ln(N))). With N=2, ln(2) = 0.693. Adding more experts increases ln(N) and requires exponentially more data to achieve the same per-expert regret.

History Efficiency

Each expert requires its own ghost history. With 2 experts, the metadata overhead is 2 * C entries. More experts would proportionally increase memory usage without guaranteed performance improvement.

Why Not Online RL (Like DQN)?

LeCaR was published in 2018, well after the DQN revolution. The authors deliberately chose regret minimization over deep RL for several reasons:

PropertyRegret Minimization (LeCaR)Deep RL (DQN)
Training dataNone needed (online)Requires offline logs or simulation
Convergence guaranteesO(sqrt(T)) regret boundNo formal guarantees

Key Parameters and Their Effects

Learning Rate (eta = 0.45)

Controls how aggressively the algorithm responds to mistakes. Higher values mean faster adaptation but more oscillation. The fixed value of 0.45 was empirically chosen to balance responsiveness and stability across FIU workloads.

# After LRU mistake (reward = -0.8):
#   eta=0.1: W -> [0.48, 0.52]  (slow)
#   eta=0.45: W -> [0.39, 0.61] (moderate)
#   eta=1.0: W -> [0.27, 0.73]  (aggressive)

History Size (= C)

Equal to cache size. Smaller histories miss regret signals for distant evictions. Larger histories waste memory tracking ancient decisions. C provides a natural balance: it covers one full cache cycle of evictions.
