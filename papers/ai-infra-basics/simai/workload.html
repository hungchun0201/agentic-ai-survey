Section 1

AICB Overview

AICB (AI Communication Benchmark) is the workload generation and GPU profiling tool in SimAI. It serves as the bridge between real hardware measurements and simulation — without AICB, SimAI would have no way to know how long actual GPU kernels take to execute. For inference workloads specifically, AICB performs three critical functions:

  1. Defines model architectures (DeepSeek, Qwen3-MoE, Qwen3-Next) as "mocked models" — Python classes that mirror real model structure without storing weights
  2. Profiles actual GPU kernel execution times using AIOB (AI Operation Benchmark) — measures real CUDA timing on Hopper/Blackwell GPUs
  3. Generates workload description files consumed by astra-sim for network simulation — text files listing every compute and communication operation

The mocked model approach is particularly elegant: rather than requiring full model weights (hundreds of GB), AICB only needs the architectural parameters — hidden size, number of heads, number of experts, etc. — to compute the exact tensor dimensions for every operation. This makes workload generation fast, portable, and deterministic.

AICB Inference Workload Pipeline
Key Insight: AIOB profiling requires SM90+ GPUs (Hopper/Blackwell) for DeepGEMM and FlashMLA kernels, but once profiled, the resulting compute cache can be used for CPU-only simulation on any machine. This means you profile once on expensive hardware, then simulate endlessly on cheap hardware.
Section 2

Inference Workload Generation Pipeline

The workload generation pipeline transforms a model architecture definition into a concrete sequence of compute and communication operations. This is the core of what AICB produces for the simulator. Each transformer layer is decomposed into individual operations, each with measured compute time and associated collective communication.

2.1 Work_Item Data Structure

Every single operation in the workload is represented as a Work_Item dataclass. This is the fundamental unit of work in AICB's output — each one captures a single layer operation with its compute time and communication requirements for forward pass, backward pass, and data-parallel dimensions.

aicb/workload_generator/work_item.py AICB 
import dataclasses

@dataclasses.dataclass
class Work_Item:
    # Name of this operation (e.g., "attention_layer", "moe_route", "norm")
    name: str = "none"

    # Forward pass compute time in microseconds (measured by AIOB)
    forward_compute_time: int = 0

    # Forward communication type: ALLREDUCE, ALLGATHER, ALLTOALL, etc.
    forward_comm: str = "NONE"

    # Forward communication size in bytes
    forward_comm_size: str = "0"

    # Backward pass compute time (0 for inference-only)
    backward_compute_time: int = 0

    # Backward communication type
    backward_comm: str = "NONE"

    # Backward communication size in bytes
    backward_comm_size: int = 0

    # Data-parallel compute time
    dp_compute_time: int = 0

    # Data-parallel communication type
    dp_comm: str = "NONE"

    # Data-parallel communication size in bytes
    dp_comm_size: int = 0

    # Processing time (default 100 microseconds)
    process_time: int = 100

For inference workloads, only the forward pass fields (forward_compute_time, forward_comm, forward_comm_size) are populated. The backward and data-parallel fields remain at their defaults since inference does not involve gradient computation or weight synchronization.

2.2 SimAIWorkload Class (Core Generator)

The SimAIWorkload class is the central workload generator. It iterates through the model's layer structure and, for each layer type, computes the communication sizes and associates them with the profiled compute times. The key method get_comm_size() calculates all communication volumes based on whether the current phase is prefill or decode.

aicb/workload_generator/simai_workload.py AICB 
class SimAIWorkload:
    """Core workload generator for SimAI inference simulation."""

    def get_comm_size(self):
        # For DECODE phase: m = batch_size (typically small, e.g., 1-64)
        # For PREFILL phase: m = seq_length (larger, e.g., 4096)
        m = self.args.batch_size if self.is_decode else self.args.seq_length

        # Tensor-parallel ALLREDUCE size: 2 * m * hidden_size
        # Factor of 2 accounts for both up-proj and down-proj needing reduction
        tp_comm_size = 2 * m * self.args.hidden_size

        # Expert-parallel ALLTOALL (combine): same shape as TP but scaled by topk/tp
        ep_combine_size = tp_comm_size * self.topk // self.tp

        # Expert-parallel ALLTOALL (dispatch): includes FP8 overhead factor
        # DeepSeek and Qwen3 use FP8 for dispatch to reduce communication
        ep_dispatch_size = ep_combine_size * FP8_FACTOR

        return (tp_comm_size, ep_dispatch_size, ep_combine_size)

    def workload_generate_aiob(self):
        """Generate workload with AIOB-profiled compute times."""
        work_items = []

        for layer_type in self._get_model_details():
            if layer_type == "attention":
                # Attention layer: ALLREDUCE in tensor-parallel group
                # Compute time from AIOB profiling of QKV + Flash + O_proj
                item = Work_Item(
                    name="attention_layer",
                    forward_compute_time=self.aiob_cache["attention"],
                    forward_comm="ALLREDUCE",
                    forward_comm_size=str(tp_comm_size),
                )

            elif layer_type == "moe_route":
                # MoE routing: ALLTOALL in expert-parallel group for dispatch
                # Tokens are routed to their assigned experts across GPUs
                item = Work_Item(
                    name="moe_route",
                    forward_compute_time=self.aiob_cache["route"],
                    forward_comm="ALLTOALL",
                    forward_comm_size=str(ep_dispatch_size),
                )

            elif layer_type == "moe_expert":
                # MoE expert compute: ALLTOALL for combining expert results
                # Expert outputs are gathered back to original GPU assignments
                item = Work_Item(
                    name="moe_expert",
                    forward_compute_time=self.aiob_cache["expert"],
                    forward_comm="ALLTOALL",
                    forward_comm_size=str(ep_combine_size),
                )

            work_items.append(item)

        return work_items

The communication size calculation is crucial for accurate simulation. Notice that the dispatch uses FP8_FACTOR — this reflects DeepSeek's optimization where dispatch tokens are quantized to FP8 before sending, reducing communication volume by roughly half. The combine phase sends results in the original precision.

2.3 Prefill vs Decode Phase Differences

The distinction between prefill and decode phases is one of the most important aspects of inference workload modeling. These two phases have fundamentally different computational characteristics, and AICB models them separately to ensure simulation accuracy.

Aspect Prefill Decode
m (input dimension) seq_length (e.g., 4096) batch_size (e.g., 1)
Attention kernel flash_mla_sparse_fwd() flash_mla_with_kvcache()
GEMM kernel bench_contiguous() bench_masked()
KV cache Build incrementally Use existing
Token layout Contiguous sequence Scattered batch elements
Communication Larger (m * hidden_size) Smaller (batch * hidden_size)
Compute intensity Compute-bound (large matrices) Memory-bound (small matrices)
Bottleneck GPU FLOPs Memory bandwidth + network

This distinction explains why PD disaggregation is valuable: prefill and decode have such different resource profiles that running them on the same GPU wastes capacity. AICB captures these differences precisely through its dual-mode profiling, using different GEMM kernels (bench_contiguous vs bench_masked) and different FlashMLA variants for each phase.

Important: DeepGEMM's FP8 kernels reduce compute by ~50% and communication by FP8_FACTOR — this fundamentally changes workload characteristics. Without modeling FP8 accurately, simulation results would significantly overestimate both compute time and network utilization.
Section 3

DeepSeek Model Architecture (MockedDeepSeek.py)

The mocked DeepSeek model is AICB's most detailed model definition. It captures the full DeepSeek-V2/V3 architecture including Multi-head Latent Attention (MLA) and Mixture of Experts (MoE). The mocked model does not contain any trained weights — it only defines the computation graph structure and tensor dimensions, which is all that's needed for workload generation.

3.1 DeepSeekAttention (Multi-head Latent Attention)

MLA is DeepSeek's key innovation for reducing KV cache size. Instead of caching the full K and V tensors, MLA compresses them into a lower-dimensional latent space. The mocked attention module captures all the GEMM operations involved in this compression and decompression pipeline.

aicb/models/MockedDeepSeek.py AICB 
class DeepSeekAttention:
    """Mocked Multi-head Latent Attention for DeepSeek-V2/V3.
    
    Key dimensions:
      d_q_c  = 1536   (compressed query dimension)
      d_kv_c = 512    (compressed KV dimension)
      d_r    = 64     (rotary embedding dimension)
      heads  = 128    (number of attention heads)
      d_q    = 128    (per-head query dimension)
    """

    def __init__(self, args):
        self.hidden_size = args.hidden_size  # 7168 for DeepSeek-V2
        self.d_q_c = 1536                     # Compressed query dim
        self.d_kv_c = 512                      # Compressed KV dim
        self.d_r = 64                           # Rotary dim
        self.num_heads = 128
        self.d_q = 128                          # Per-head query dim

    def forward(self, m):
        # Step 1: QKV compression — (m, 7168) -> (m, 2112)
        # Output: 2112 = d_q_c(1536) + d_kv_c(512) + d_r(64)
        qkv_compressed = fp8_gemm_nt(m, self.hidden_size, 2112)

        # Step 2: Q uncompression — (m, 1536) -> (m, heads * d_q + d_r)
        # Expands compressed Q back to full head dimension
        q_full = fp8_gemm_nt(m, self.d_q_c, self.num_heads * self.d_q + self.d_r)

        # Step 3: Attention computation (FlashMLA)
        # Decode: flash_mla_with_kvcache() — reads from existing cache
        # Prefill: flash_mla_sparse_fwd() — builds cache incrementally
        attn_output = flash_mla(q_full, kv_cache, is_decode=self.is_decode)

        # Step 4: Output projection — (m, heads * d_q) -> (m, hidden_size)
        output = fp8_gemm_nt(m, self.num_heads * self.d_q, self.hidden_size)

        return output  # -> ALLREDUCE across TP group

The compression ratio is significant: standard multi-head attention would cache heads * d_q = 128 * 128 = 16384 dimensions per token for both K and V, while MLA caches only d_kv_c + d_r = 512 + 64 = 576 dimensions. This is a 28x reduction in KV cache size per token, which is critical for serving large batches.

3.2 DeepSeekMOE (Mixture of Experts)

The MoE layer is where expert parallelism introduces its characteristic all-to-all communication pattern. Each token is routed to its top-k experts (typically 6-8), which may reside on different GPUs. The dispatch operation sends tokens to experts via ALLTOALL, and the combine operation gathers results back.

aicb/models/MockedDeepSeek.py AICB 
class DeepSeekMOE:
    """Mixture of Experts layer for DeepSeek.
    
    FP8_FACTOR accounts for mixed-precision communication:
    - FP8 data: 1 byte per element
    - FP32 scaling factors: 4 bytes per 128-element block
    - Average bytes per element: (1 + 4/128) / 2 ≈ 0.516
    """

    FP8_FACTOR = (1 + 4/128) / 2  # ≈ 0.516 — FP8 + FP32 scale overhead

    def __init__(self, args):
        self.num_experts = args.num_experts       # e.g., 256
        self.topk = args.topk                     # e.g., 8
        self.expert_intermediate = args.expert_intermediate_size
        self.ep = args.ep_size                    # expert parallelism degree

    def dispatch(self, m):
        """Send tokens to assigned experts via ALLTOALL.
        Uses FP8 quantization to reduce communication volume."""
        dispatch_size = 2 * m * self.hidden_size * self.topk // self.tp
        dispatch_size_fp8 = int(dispatch_size * self.FP8_FACTOR)
        # all_to_all(dispatch_size_fp8) across EP group
        return dispatch_size_fp8

    def combine(self, m):
        """Gather expert results back via ALLTOALL.
        Results sent in original precision (no FP8)."""
        combine_size = 2 * m * self.hidden_size * self.topk // self.tp
        # all_to_all(combine_size) across EP group
        return combine_size

    def forward(self, m):
        """MoE forward pass: route -> dispatch -> expert_compute -> combine"""
        # Step 1: Routing — determine which experts process each token
        expert_assignments = self.router(m)

        # Step 2: Dispatch — send tokens to assigned experts (FP8)
        self.dispatch(m)  # -> ALLTOALL across EP group

        # Step 3: Expert compute — each expert processes its assigned tokens
        # Up-proj: (m_per_expert, hidden_size) -> (m_per_expert, intermediate_size)
        # Down-proj: (m_per_expert, intermediate_size) -> (m_per_expert, hidden_size)
        expert_output = self.expert_compute(m)

        # Step 4: Combine — gather results back to original GPUs
        self.combine(m)   # -> ALLTOALL across EP group

        return expert_output

3.3 FP8 Optimization with DeepGEMM

DeepGEMM provides specialized FP8 GEMM kernels that are fundamental to how DeepSeek and Qwen3 models operate efficiently. The key insight is that FP8 requires scale factors — per-token for matrix A and per-block for matrix B — which add a small overhead to the raw FP8 data. The FP8_FACTOR constant accounts for this overhead precisely.

Why FP8 Matters for Simulation

FP8 affects both compute and communication in ways that must be accurately modeled:

  • Compute: FP8 GEMMs are roughly 2x faster than FP16 on Hopper tensor cores
  • Communication: Dispatch uses FP8 (0.516x of FP16 size), combine uses full precision
  • Asymmetry: The dispatch/combine asymmetry means network traffic is not symmetric in EP

Qwen3-MoE and Qwen3-Next Models

Beyond DeepSeek, AICB supports two Qwen3 model variants. These share the MoE architecture but differ in their attention mechanisms. Qwen3-Next introduces a novel hybrid attention pattern that alternates between standard attention and a gated delta net, creating different compute and communication profiles per layer.

4.1 Qwen3-MoE

Qwen3-MoE Transformer Layer

aicb/models/MockedQwen3Moe.py AICB 
class Qwen3MoeTransformerLayer:
    """Single transformer layer for Qwen3-MoE architecture.
    
    Structure: attention_norm -> Qwen3MoeAttention -> moe_norm -> Qwen3MoeBlock
    
    Key differences from DeepSeek:
    - Uses standard GQA (Grouped Query Attention) instead of MLA
    - Different expert routing topology
    - Separate norm before MoE block
    """

    def __init__(self, args, layer_idx):
        self.attention_norm = RMSNorm(args.hidden_size)
        self.attention = Qwen3MoeAttention(args)
        self.moe_norm = RMSNorm(args.hidden_size)
        self.moe_block = Qwen3MoeBlock(args)  # Route + Expert

    def forward(self, hidden_states, m):
        # Pre-norm -> Attention -> Residual
        normed = self.attention_norm(hidden_states)
        attn_out = self.attention(normed, m)    # -> ALLREDUCE (TP)
        hidden_states = hidden_states + attn_out

        # Pre-norm -> MoE -> Residual
        normed = self.moe_norm(hidden_states)
        moe_out = self.moe_block(normed, m)     # -> ALLTOALL (EP)
        hidden_states = hidden_states + moe_out

        return hidden_states

4.2 Qwen3-Next (Hybrid Attention)

Qwen3-Next introduces a hybrid attention pattern where some layers use full softmax attention (Qwen3NextAttention) while others use a novel gated delta net (Qwen3NextGatedDeltaNet). The full_attention_interval parameter controls which layers use which attention type. This creates a non-uniform compute profile across layers.

Qwen3-Next Transformer Layer

aicb/models/MockedQwen3Next.py AICB 
class Qwen3NextTransformerLayer:
    """Single transformer layer for Qwen3-Next hybrid attention architecture.
    
    Uses full_attention_interval to determine attention type:
    - If layer_idx % full_attention_interval == 0: Qwen3NextAttention (standard)
    - Otherwise: Qwen3NextGatedDeltaNet (novel linear attention)
    """

    def __init__(self, args, layer_idx):
        self.layer_idx = layer_idx
        self.full_attention_interval = args.full_attention_interval

        # Choose attention type based on layer index
        if layer_idx % self.full_attention_interval == 0:
            # Full softmax attention — standard but expensive
            self.attention = Qwen3NextAttention(args)
            self.attn_type = "full"
        else:
            # Gated Delta Net — novel linear attention, cheaper
            self.attention = Qwen3NextGatedDeltaNet(args)
            self.attn_type = "gated_delta"

        # MoE block with expert parallelism
        self.moe_block = Qwen3NextMoeBlock(args)

    def forward(self, hidden_states, m):
        # Attention (type depends on layer)
        attn_out = self.attention(hidden_states, m)  # -> ALLREDUCE (TP)
        hidden_states = hidden_states + attn_out

        # MoE with expert routing
        moe_out = self.moe_block(hidden_states, m)   # -> ALLTOALL (EP)
        hidden_states = hidden_states + moe_out

        return hidden_states

Qwen3NextAttention (Full)

Standard grouped-query attention with softmax. Used at regular intervals (e.g., every 4th layer). Provides full quadratic attention for critical layers that need maximal representational power.

  • QKV projection + rotary embedding
  • Flash attention kernel
  • O projection + ALLREDUCE

Qwen3NextGatedDeltaNet

Novel linear attention variant used in most layers. Significantly cheaper than full attention — O(n) instead of O(n^2). Uses delta updates to maintain a compressed state representation.

  • Gate + delta computation
  • Linear state update
  • Output projection + ALLREDUCE

Section 5

AIOB GPU Profiling (AiobDeepSeek.py)

AIOB (AI Operation Benchmark) is the profiling subsystem that measures actual GPU kernel execution times. It runs the real CUDA kernels — DeepGEMM FP8 GEMMs, FlashMLA attention, RMSNorm, etc. — on actual GPU hardware and records precise timing using CUDA events. The resulting timing data forms the "compute cache" that drives simulation accuracy.

Profiling Pipeline

1

Embedding

Token embedding lookup with optional ALLREDUCE for vocabulary parallelism. Typically a small operation — embedding table lookup is memory-bound, not compute-bound.

2

RMSNorm

Layer normalization with CUDA timing. RMSNorm is simpler than LayerNorm (no mean subtraction), making it faster. Profiled separately because its latency depends on hidden size and batch dimensions.

3

Attention Sub-components

The attention layer is profiled as four separate sub-operations, each with its own CUDA timing:

  • _qkv_compression() — (m, 7168, 2112) FP8 GEMM that compresses QKV
  • _q_uncompression() — Decompress Q from latent to full head dimension
  • _attention() — FlashMLA kernel (decode: with KV cache, prefill: sparse)
  • _O_proj() — Output projection FP8 GEMM back to hidden size
4

MLP / MoE

For dense layers: up-projection and down-projection FP8 GEMMs. For MoE layers: routing computation, grouped GEMM (bench_masked or bench_contiguous depending on phase), and expert combine. Each sub-operation is independently timed.

CUDA Timing Measurement

Precise timing is critical — AIOB uses CUDA events rather than Python time.time() because CPU-side timing includes kernel launch overhead and is non-deterministic. CUDA events measure the actual GPU execution time in microseconds with hardware-level precision.

aicb/profiler/aiob_utils.py AICB 
def cuda_timing_decorator(func):
    """Decorator that wraps a GPU kernel call with CUDA event timing.
    
    Uses torch.cuda.Event for precise GPU-side timing measurement.
    Returns the execution time in microseconds.
    """
    def wrapper(*args, **kwargs):
        # Create CUDA events with timing enabled
        start_event = torch.cuda.Event(enable_timing=True)
        end_event = torch.cuda.Event(enable_timing=True)

        # Warmup run (discard timing — first run is always slower)
        func(*args, **kwargs)
        torch.cuda.synchronize()

        # Timed run
        start_event.record()
        result = func(*args, **kwargs)
        end_event.record()

        # Wait for GPU to finish and compute elapsed time
        torch.cuda.synchronize()
        elapsed_us = start_event.elapsed_time(end_event) * 1000  # ms -> us

        return result, elapsed_us

    return wrapper

Section 6

Workload File Format

The final output of AICB's workload generation is a text file consumed by astra-sim. This file describes every compute and communication operation in the model's forward pass. The format is structured with a header line containing parallelism configuration, followed by a count of operations, then one line per operation.

workload_output/deepseek_v3_decode.txt ASTRA-SIM 
# Header: model type, parallelism groups, and sizes
HYBRID_TRANSFORMER_FWD_IN_BCKWD  model_parallel_NPU_group: 2  ep: 8  pp: 1  vsp: 0

# Total number of operations in this workload
32

# Format: name | backward_time | forward_time | comm_type | comm_size | ...
# Each line represents one Work_Item serialized to text

norm | -1 | 0 | BROADCAST | 16384 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
attention_layer | -1 | 539986 | ALLREDUCE | 16777216 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
moe_route | -1 | 42310 | ALLTOALL | 8650752 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
moe_expert | -1 | 486330 | ALLTOALL | 16777216 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
mlp_layer | -1 | 486330 | ALLREDUCE | 16777216 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
norm | -1 | 0 | BROADCAST | 16384 | -1 | 0 | NONE | 0 | -1 | 0 | NONE | 0 | 100
... (repeated for each transformer layer)

Each line in the workload file maps directly to a Work_Item from the generator. The pipe-delimited format contains the following fields in order:

Field Description Example
name Operation name attention_layer
backward_compute Backward compute time (-1 = not used) -1
forward_compute Forward compute time (microseconds) 539986
forward_comm Communication collective type ALLREDUCE
forward_comm_size Communication size in bytes 16777216
process_time Scheduling overhead (microseconds) 100

The header line is critical because it tells astra-sim how to set up the communication topology. model_parallel_NPU_group: 2 means tensor parallelism across 2 GPUs, ep: 8 means expert parallelism across 8 GPUs, and pp: 1 means no pipeline parallelism. Astra-sim uses these to create the corresponding communication groups and simulate collective operations with the correct number of participants.

Section 7

MoE Routing Strategies

How tokens are distributed across experts has a major impact on both load balance and communication patterns. AICB supports multiple routing strategies to simulate different real-world scenarios. The routing strategy determines how many tokens each expert receives, which directly affects per-expert GEMM dimensions and, consequently, compute time.

aicb/workload_generator/moe_routing.py AICB 
from enum import Enum

class Strategy(str, Enum):
    """MoE routing strategies for simulation."""
    RoundRobin = "RoundRobin"       # Deterministic, perfectly balanced
    UniformRandom = "UniformRandom" # Stochastic, models real-world variability


def get_ep_expected_m_per_group(m, num_groups, topk, ep, strategy):
    """Calculate expected number of tokens per expert group.
    
    Args:
        m: Total number of tokens (batch_size for decode, seq_length for prefill)
        num_groups: Number of expert groups (= num_experts / experts_per_group)
        topk: Number of experts each token is routed to
        ep: Expert parallelism degree
        strategy: Routing strategy to use
    
    Returns:
        Expected m (tokens) per expert group on this GPU
    """
    if strategy == Strategy.RoundRobin:
        # Perfectly balanced: each group gets exactly m * topk / num_groups
        # This is the best-case scenario — no load imbalance
        return max(round(m * topk / num_groups), 1)

    elif strategy == Strategy.UniformRandom:
        # Simulates stochastic load via multinomial distribution
        # Models the variability seen in real inference workloads
        # where popular experts receive more tokens than others
        total_assignments = m * topk
        probs = torch.ones(num_groups) / num_groups  # uniform prior
        distribution = torch.distributions.Multinomial(
            total_count=total_assignments,
            probs=probs
        )
        sample = distribution.sample()

        # Return the expected (average) allocation per group
        # In reality, some groups get more, some get less
        expected_per_group = sample.mean().item()
        return max(round(expected_per_group), 1)
Critical Insight: MoE routing strategy significantly impacts expert load balance — UniformRandom models real-world variability. In practice, some experts become "hot spots" that receive more tokens than the average, creating compute imbalance. SimAI must model this accurately because the slowest expert determines the overall layer latency (the ALLTOALL combine cannot complete until all experts finish).

RoundRobin Strategy

Distributes tokens evenly across all expert groups in a deterministic round-robin fashion. This gives the ideal load balance — every expert processes exactly the same number of tokens. Useful as a baseline for measuring the impact of load imbalance.

m_per_group = m * topk / num_groups

UniformRandom Strategy

Simulates stochastic routing using multinomial distribution with uniform prior probabilities. While the expected allocation equals the balanced case, individual runs show variance. This models real-world behavior where certain experts are more popular for certain input distributions.

Multinomial(m * topk, uniform_probs)

Integration with vidur-alibabacloud

The compute times generated by AICB flow into the vidur-alibabacloud scheduler simulator. This integration path allows Vidur to use real GPU profiling data (from AICB/AIOB) instead of its default sklearn-based regression models for execution time prediction. This is critical for MoE models where the default Vidur approach (linear regression on dense models) would be inaccurate.

vidur-alibabacloud/execution_time_predictor/sklearn_execution_time_predictor.py VIDUR AICB 
class SklearnExecutionTimePredictor:
    """Execution time predictor that can use AICB profiling data.
    
    When AICB data is available, bypasses sklearn regression and uses
    direct GPU-profiled timing instead.
    """

    def _predict_from_models_by_aicb(self):
        """Use AICB profiling data for execution time prediction.
        
        This method is called when AICB compute cache is available.
        It replaces the default sklearn regression approach with
        direct lookup from GPU-profiled timing data.
        """
        predictions = {}

        # Get compute model predictions from AICB cache
        predictions = self._predict_for_compute_models_by_aicb()

        # Get attention layer predictions from AICB cache
        predictions.update(
            self._predict_for_attention_layer_models_by_aicb()
        )

        return predictions

    def _predict_for_compute_models_by_aicb(self):
        """Map AICB profiling keys to Vidur model component names.
        
        AICB key -> Vidur component name mapping:
          atten_qkv        -> attn_pre_proj
          atten_o_proj      -> attn_post_proj
          mlp_up            -> mlp_up_proj
          mlp_down          -> mlp_down_proj
          moe_up            -> moe_up_proj
          moe_down          -> moe_down_proj
          atten_q_uncomp    -> attn_q_uncompress
        """
        model_names = [
            "attn_pre_proj",
            "attn_post_proj",
            "mlp_up_proj",
            "mlp_down_proj",
            "moe_up_proj",
            "moe_down_proj",
            "attn_q_uncompress",
        ]

        predictions = {}
        for name in model_names:
            aicb_key = self._vidur_to_aicb_key(name)
            predictions[name] = self.aicb_cache[aicb_key]["time_gpu"]

        return predictions

    def _predict_for_attention_layer_models_by_aicb(self):
        """Get FlashMLA attention timing from AICB.
        
        Attention is special because decode uses flash_mla_with_kvcache
        and prefill uses flash_mla_sparse_fwd — different kernels with
        very different performance characteristics.
        """
        return {
            "attn_prefill": self.aicb_cache["atten_flash_prefill"]["time_gpu"],
            "attn_decode": self.aicb_cache["atten_flash_decode"]["time_gpu"],
        }

This integration creates a powerful two-layer simulation: Vidur handles the scheduling logic (when to batch requests, how to manage KV cache memory, when to preempt), while AICB provides the ground-truth compute times that determine how long each scheduled batch takes. The network communication is then simulated by astra-sim using the workload file.

This coupling is a potential maintenance concern as new model architectures are added.

Detailed Attention Profiling Flow

Understanding how AIOB profiles the attention layer in detail is important because attention is typically the most time-consuming operation in inference. The profiling breaks down attention into multiple sub-operations, each measured independently, to provide fine-grained timing data.

aicb/profiler/AiobDeepSeek.py AICB 
class AiobDeepSeek:
    """AIOB profiler for DeepSeek model attention operations."""

    def _qkv_compression(self, m):
        """Profile QKV compression GEMM: (m, 7168) -> (m, 2112).
        Uses DeepGEMM FP8 kernel for maximum throughput.
        Output 2112 = d_q_c(1536) + d_kv_c(512) + d_r(64)."""
        a_fp8, a_scales, b_fp8, b_scales = deepgemm.construct(
            m, self.hidden_size, 2112
        )
        _, time_us = cuda_timing(
            deepgemm.gemm_nt, a_fp8, b_fp8, a_scales, b_scales
        )
        return {"atten_qkv": {"time_gpu": time_us}}

    def _q_uncompression(self, m):
        """Profile Q uncompression: (m, 1536) -> (m, num_heads*d_q + d_r).
        Expands compressed queries back to full attention dimension."""
        out_dim = self.num_heads * self.d_q + self.d_r
        a_fp8, a_scales, b_fp8, b_scales = deepgemm.construct(
            m, self.d_q_c, out_dim
        )
        _, time_us = cuda_timing(
            deepgemm.gemm_nt, a_fp8, b_fp8, a_scales, b_scales
        )
        return {"atten_q_uncomp": {"time_gpu": time_us}}

    def _attention(self, m, seq_len):
        """Profile FlashMLA attention kernel.
        
        Decode mode: flash_mla_with_kvcache()
          - Reads from pre-populated KV cache
          - m is small (batch_size), seq_len affects cache length
        
        Prefill mode: flash_mla_sparse_fwd()
          - Builds KV cache incrementally
          - m is large (seq_length), causal mask applied
        """
        if self.is_decode:
            _, time_us = cuda_timing(
                flash_mla_with_kvcache,
                q=self.q_buffer, kv_cache=self.kv_cache,
                batch_size=m, cache_seqlens=seq_len
            )
        else:
            _, time_us = cuda_timing(
                flash_mla_sparse_fwd,
                q=self.q_buffer, k=self.k_buffer, v=self.v_buffer,
                seq_len=m
            )
        return {"atten_flash": {"time_gpu": time_us}}

    def _O_proj(self, m):
        """Profile output projection: (m, num_heads*d_q) -> (m, hidden_size).
        Uses DeepGEMM FP8 kernel."""
        a_fp8, a_scales, b_fp8, b_scales = deepgemm.construct(
            m, self.num_heads * self.d_q, self.hidden_size
        )
        _, time_us = cuda_timing(
            deepgemm.gemm_nt, a_fp8, b_fp8, a_scales, b_scales
        )
        return {"atten_o_proj": {"time_gpu": time_us}}

The total attention time reported in the workload file is the sum of all four sub-operations. By profiling each separately, AICB can also feed per-component timing to Vidur, enabling more accurate scheduling decisions — for example, Vidur can estimate whether a particular batch would be compute-bound (dominated by Q uncompression) or memory-bound (dominated by FlashMLA cache reads).

Summary

Communication Patterns in Inference Workloads

The workload file captures three distinct communication patterns that arise from the parallelism strategy. Understanding these patterns is essential for interpreting simulation results and for tuning the network topology in SimAI.

ALLREDUCE

Used by: Tensor parallelism (attention + dense MLP layers). All GPUs in the TP group exchange and sum partial results. Communication volume: 2 * m * hidden_size bytes. Occurs after every attention and dense MLP layer.

ALLTOALL

Used by: Expert parallelism (MoE dispatch + combine). Each GPU sends tokens to experts on other GPUs. Dispatch uses FP8 (FP8_FACTOR), combine uses full precision. Creates asymmetric bidirectional traffic.

BROADCAST

Used by: Norm layers and embeddings. Small communication volume (hidden_size bytes). Minimal network impact but included for completeness in timing.

The interplay between these communication patterns is what makes inference workload simulation challenging. In a typical DeepSeek-V3 forward pass with 61 layers, the workload generates approximately 120+ collective operations — each needing accurate timing from both the compute side (AICB) and the network side (astra-sim via SimCCL/ns-3).

Takeaway: AICB's workload generation is the foundation that makes SimAI's full-stack inference simulation possible. By accurately profiling GPU kernels, modeling FP8 optimizations, and capturing the MoE routing patterns, AICB ensures that the simulation's compute and communication characteristics faithfully reflect real-world inference behavior on modern MoE architectures like DeepSeek-V3 and Qwen3.
