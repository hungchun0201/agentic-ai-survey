CacheBlend: Fast Large Language Model Serving for RAG with Cached Knowledge Fusion

1 Overview

Large language models in RAG (Retrieval-Augmented Generation) applications often incorporate multiple text chunks as context. These chunks are reused across different queries, creating an opportunity to pre-compute and cache their KV (Key-Value) states. However, existing KV cache reuse techniques face a fundamental limitation: they either only work for prefix text (prefix caching) or ignore cross-attention between chunks (full KV reuse), degrading generation quality.

CacheBlend solves this by selectively recomputing only 5-18% of tokens at chunk boundaries—the tokens with the highest KV deviation from ground truth. This restores cross-attention accuracy while preserving the speed benefits of KV cache reuse.

Why This Matters for Agentic AI

AI agents assembling context from multiple retrieved documents face exactly the multi-chunk KV cache problem CacheBlend addresses. When an agent calls a RAG pipeline to gather evidence from diverse sources, each retrieved chunk's KV cache can be pre-computed independently and then fused efficiently at query time using CacheBlend's selective recomputation. This directly reduces the latency bottleneck identified in the "Cost of Dynamic Reasoning" paper (HPCA'26), where tool execution and LLM inference alternate sequentially.

Why Does Full KV Reuse Fail?

When KV caches are pre-computed independently for each chunk, the cross-attention between tokens in different chunks is completely missed. Cross-attention captures how tokens in one chunk relate to tokens in preceding chunks—information that can be vital for correct answers.

Illustrative Failure Case

Consider the query: "How many goals did Messi score more than Cristiano Ronaldo at FIFA World Cups?" prepended with two chunks—one about Messi's career stats, one about Ronaldo's. Full KV recompute gives the correct answer by computing attention across both chunks. Full KV reuse, which pre-computes each chunk's KV separately, misses the cross-chunk attention needed to compare the two players, producing a wrong answer.

3 Selective KV Recompute Mechanism

Core Idea

CacheBlend performs selective KV recompute: on each transformer layer, it recomputes the KV of only a small subset of tokens (those with the highest KV deviation from ground truth) while reusing the pre-computed KV for all other tokens. This restores the critical cross-attention at chunk boundaries with minimal compute overhead.

The compute overhead is proportional to the number of selected tokens. If we recompute r% of tokens per layer, the total compute is r% of a full prefill—typically only 5-18%.

Two Key Insights

Insight 1: On layer i, recomputing the KV of token j with a higher KV deviation reduces the attention deviation by a greater amount. Selecting the top-deviation tokens gives the best quality for a given compute budget.

Insight 2: Tokens with the highest KV deviations on one layer are likely to have the highest deviations on the next layer (Spearman's rank correlation > 0.8 between neighboring layers). This enables efficient layer-to-layer token selection without needing ground-truth KV.

This gradual filtering works because of attention sparsity: in transformer models, high attention typically occurs between a small number of tokens. About 10-15% of tokens have significantly higher KV deviation than others, and these tend to be consistent across layers (Spearman rank correlation > 0.8).

Why Does Updating Only 5-18% Suffice?

Attention Sparsity

In attention matrices, high attention values are concentrated among a small fraction of token pairs. Most tokens have very low cross-attention with other chunks' tokens, meaning their KV deviation between pre-computed and full-recomputed states is negligible. Only the tokens at chunk boundaries and semantically related positions across chunks have significant cross-attention that needs correction.

Three Key Components

Loading Controller

Solves two design problems: (1) choosing the recompute ratio so that selective recompute does not add extra TTFT, and (2) selecting the cheapest storage device (GPU, RAM, SSD) such that KV loading delay does not exceed recompute delay. Uses two delay estimators:

  • T_recompute(r%, LLM, L) = r% * Prefill(LLM, L)
  • T_load(LLM, L, device) = PerTokenKVSize(LLM) * L / Throughput(device)

KV Cache Fusor

Merges pre-computed KV caches via selective recompute, layer by layer. Critically, it pipelines the KV loading of layer L+1 with the selective recompute of layer L. If loading is faster than recompute, the extra delay for selective recompute is completely hidden.

KV Cache Store

Maps LLM input chunks to pre-computed KV caches using text hashing (same scheme as vLLM). Supports tiered storage: GPU memory, CPU RAM, NVME SSD, and slower disks. Evicts least-recently-used caches when storage is full.

def fetch_kv(text: str, layer_id: int) -> KVCache:
    """Fetch pre-computed KV cache from store by text hash."""
    hash_key = compute_hash(text)
    if hash_key in kv_store:
        return kv_store.load(hash_key, layer_id)  # torch.load() or .cuda()
    return None

def prefill_layer(input_dict: dict, kv_cache: KVCache) -> dict:
    """Selective KV recompute on one layer."""
    if input_dict["check_flag"]:
        # Find HKVD tokens by comparing new KV with precomputed
        hkvd_indices = find_hkvd_tokens(
            new_kv=input_dict["input_tensor"],
            pre_kv=kv_cache,
            ratio=recompute_ratio
        )
    else:
        hkvd_indices = input_dict["HKVD_indices"]
    # Partial prefill: only recompute HKVD tokens' KV
    return partial_prefill(input_dict, kv_cache, hkvd_indices)

Recompute Ratio vs. Quality Tradeoff

Across all datasets on Yi-34B, CacheBlend's quality loss is at most 0.002 F1-score or 0.002 Rouge-L compared to full KV recompute, using a 5-18% selective recomputation ratio. This translates to a 4.1-6.6x TTFT reduction versus full KV recompute and 3.4-6.1x versus prefix caching.

Throughput Under Load

Under varying request arrival rates (simulating production serving), CacheBlend achieves lower TTFT with higher throughput than all baselines at equivalent quality. The advantage grows more pronounced with larger batch sizes, because the prefill phase becomes increasingly dominant as batches grow.

Sensitivity Analysis

Performance improvement holds across chunk lengths. The compute time reduction ratio stays consistent regardless of individual chunk size.

CacheBlend's advantage grows with batch size. As decoding time increases slower than prefill time with larger batches, prefill dominates overall latency.

7 Relevance to RAG & Agentic Systems

RAG Pipeline Integration

CacheBlend directly targets the RAG use case where a user query is augmented with multiple retrieved document chunks. In typical RAG deployments:

  • Each chunk is 128-512 tokens (standard Langchain/LlamaIndex chunking)
  • 6-12 chunks are retrieved per query
  • The same chunks are reused across many different queries
  • Prefix caching can only save the prefill of the first chunk
CacheBlend vs. Alternative RAG Methods

Compared to MapReduce (process chunks independently, summarize, then merge): CacheBlend achieves 2-5x lower TTFT with higher F1 scores, because CacheBlend preserves cross-chunk attention while MapReduce loses inter-chunk context.

Compared to MapRerank (generate answers per-chunk, select best): While MapRerank has slightly lower TTFT, its quality is substantially worse because it processes chunks independently, ignoring inter-chunk dependencies.

For an agent making 5-10 retrieval-augmented reasoning calls per task, CacheBlend can reduce the total retrieval-augmented inference latency by 50-70%, directly addressing the sequential LLM-tool bottleneck identified in the Cost of Dynamic Reasoning paper.

Connection to Prefix Caching in Agent Serving

The HPCA'26 paper on dynamic reasoning costs showed that prefix caching provides a 5.62x throughput improvement for agent serving. CacheBlend extends this benefit beyond mere prefix matching: it enables reuse of any cached text chunk regardless of position, multiplying the opportunities for cache hits in agentic contexts where document orderings vary between queries.

8 Key Takeaways

CacheBlend in One Sentence

By selectively recomputing only 5-18% of tokens (those with the highest KV deviation at chunk boundaries), CacheBlend achieves the speed of full KV reuse with the quality of full KV recompute—reducing TTFT by 2.2-3.3x and increasing throughput by 2.8-5x for multi-chunk RAG inputs.

Speed Without Quality Loss

CacheBlend reduces TTFT by 2.2-3.3x compared to full KV recompute with quality drops of at most 0.02 F1-score or 0.03 Rouge-L. This makes it practical for production RAG deployments.

Attention Sparsity Insight

Only 10-15% of tokens have significant cross-attention with other chunks. This sparsity is the key enabling observation: most tokens' KV values are barely affected by the absence of cross-attention.

Cross-Layer Correlation

HKVD tokens on one layer predict HKVD tokens on the next (rank correlation >0.8). Gradual filtering exploits this to avoid expensive ground-truth computation for token selection.

Pipelining Hides Overhead

By overlapping KV loading with selective recompute, CacheBlend adds zero extra TTFT delay when storage bandwidth is sufficient. This enables storing KV caches on cheaper devices (RAM, SSD).

Non-Prefix KV Reuse

Unlike prefix caching (which only helps the first chunk), CacheBlend reuses KV for all chunks regardless of position. This is critical for RAG where the prefix is often just the system prompt.

Agentic AI Multiplier

In agent workflows with repeated RAG retrieval steps, CacheBlend's per-call speedup compounds, potentially reducing total agent latency by 50-70% for retrieval-heavy tasks.
