Overview

Dynamic Orchestration for Disaggregated LLM Serving

Disaggregated LLM serving separates the compute-bound prefill phase from the memory-bound decode phase into separate GPU pools. While this eliminates phase interference, existing systems (DistServe, Splitwise) face three fundamental limitations under realistic dynamic workloads:

  1. Static resource allocation cannot adapt to workload variability, leaving 20-40% of GPU resources idle at low request rates
  2. Inherent load imbalance between compute-bound prefill (>95% compute, ~35% memory) and memory-bound decode (~35% compute, high memory), causing one pool to bottleneck while the other idles
  3. Prefix cache-aware routing creates positive-feedback load skew, as high-hit-rate instances attract disproportionately more requests

BanaServe's Three Innovations

BanaServe decouples resource allocation from state management through three mechanisms:

  • Fine-grained resource migration: Layer-level weight migration (coarse) and attention-level KV cache migration (fine) enable rapid rebalancing between prefill and decode pools
  • Global KV Cache Store: A shared KV cache accessible to all prefill and decode instances, eliminating cache locality constraints on routing
  • Load-aware scheduling: Routes requests purely based on current load, not cache placement, with the Global KV Store handling cache sharing transparently

Problems with Current Disaggregated Serving

Problem 1: Static Allocation Waste

Empirical measurements on LLaMA-13B with both HFT and vLLM show 20-40% of GPU resources unused at RPS ≤ 10. The fixed prefill-to-decode ratio cannot be optimal across all traffic patterns. During bursty arrivals, one pool is overwhelmed while the other sits idle. Scaling operations require service restarts and model reloading -- too slow for dynamic workloads.

Problem 2: Prefix Cache Load Skew

Cache-aware routers preferentially send requests to instances with cached prefixes, creating a positive feedback loop. In a 3-instance cluster, Instance 1 (100% cache hits) runs at full capacity while Instance 2 (40% utilization) holds redundant caches and Instance 3 (60%) recomputes uncached prefixes. This skew persists even under moderate load because of the coupling between cache placement and routing decisions.

Problem 3: Prefill-Decode Asymmetry

Measurements using DistServe with LLaMA-13B on the Alpaca dataset reveal fundamental resource asymmetry:

Communication is dominated by one-way KV Cache transfers from prefill to decode, with other channels underutilized. BanaServe addresses this by making resources dynamically migratable between pools.

Mechanism 1: Layer-Level Weight Migration

Coarse-Grained Rebalancing

When workload imbalance is pronounced, BanaServe migrates contiguous sets of transformer layers between prefill and decode GPUs. This shifts both layer weights and associated KV Cache, performing a dynamic model parallelism reconfiguration at runtime.

The total payload for migrating a layer is:

S_total = S_w (weight size) + S_kv (KV cache size)

The migration latency depends on interconnect bandwidth:

T_layer = S_total / B_net + T_sync

Since S_w is typically much larger than S_kv, the latency is dominated by weight transfer. For llama-3.1-8B with GQA (h_kv=8), one layer's weights are ~250MB, transferable in ~0.5ms over NVLink at 400 GB/s, or ~15ms over PCIe Gen4.

Execution Correctness

For a migrated layer f_l with weights W_l and KV Cache KV_l, the computation is identical on the new GPU:

y_l = f_l(x_l; W_l, KV_l)

where x_l is the input activation from the preceding layer. By transferring (W_l, KV_l) together, auto-regressive decoding continues seamlessly after migration. The migration is pipelined with ongoing inference: while GPU-1 executes layer N, layers N+k can be transferred in the background.

Mechanism 2: Attention-Level KV Migration

Fine-Grained Load Adjustment

When finer load adjustment is needed (without the cost of moving full layer weights), BanaServe partitions the KV Cache along the attention head dimension. For a layer with H attention heads, the KV states are split into two disjoint subsets:

  • KV_1 (K^(1), V^(1)): Retained on the hot GPU for local attention computation
  • KV_2 (K^(2), V^(2)): Offloaded to a cold GPU for remote attention computation

No model weights are transferred -- only the KV activations for selected heads. The pipeline works as follows:

Local and remote attention run in parallel. Only partial softmax denominators (l^(1)) and output O^(2) are exchanged.
Parallel computation correctness: Since K^(1),V^(1) and K^(2),V^(2) are from disjoint head partitions, S^(1) = Q * K^(1)^T and S^(2) = Q * K^(2)^T can be computed entirely in parallel on separate devices. Only the partial softmax denominator l^(1) needs to be communicated to the cold GPU for normalization. The final output O = O^(1) + O^(2). Transfer size S_kv is much smaller than layer state S_l, so T_attn is much less than T_layer, making this method lightweight and latency-friendly.

Global KV Cache Store

Eliminating Cache Locality Constraints

The primary bottleneck of prefix cache-aware routers is that cached prefixes are local to each GPU. This forces routing decisions to consider both cache placement and load, creating the skew problem described earlier. BanaServe solves this with a Global KV Cache Store backed by CPU/SSD storage:

  • All prefix KV caches and newly generated incremental KV caches are stored in the shared store
  • Any prefill instance can fetch cached prefixes from the store and perform only incremental prefill for new tokens
  • Decode instances load the full KV cache from the store for token generation
  • The router can focus purely on load balancing, without considering cache placement

Three-Stage Layer-Wise Pipeline

Loading KV cache from the store could introduce latency. BanaServe hides this with a three-stage pipeline that leverages the layer-wise execution property of transformers:

T_KV = (S_kv * L * r) / B   (per-layer KV transfer time)
T_F,layer = (T_F * r) / N   (per-layer forward computation time)

For a llama-3.1-8B model with d_model=4096, h_kv=8 (GQA), BF16 precision, N=32 layers, L=1000 tokens, r=0.5 cache hit rate, B=200 Gbps bandwidth:

T_F,layer = 4.22 ms   vs   T_KV = 0.082 ms

Since T_KV is much less than T_F,layer, the KV cache transfer is fully overlapped with computation, making the store access transparent to the serving workflow.

Dynamic Migration and Load-Aware Scheduling

Algorithm 1: Adaptive Module Migration

Runs in a periodic control cycle, monitoring normalized utilization U_d = C_d/C_max + M_d/M_max for each GPU (range 0 to 2.0):

  1. Measure load: Compute combined compute + memory utilization for each device
  2. Classify: Identify overloaded (load > min + delta) and underloaded (load < max - delta) devices
  3. Migrate: While imbalance exists, invoke MigrateLayer or MigrateKVHeads between overloaded and underloaded GPUs
  4. Update: Revise module allocation tracking after each migration

Hysteresis thresholds (delta_up, delta_down) prevent oscillation. The complexity per cycle is O(|D| + N_m), where |D| is the number of GPUs and N_m the modules on the most loaded device.

Algorithm 2: Load-Aware Request Scheduling

Unlike cache-aware routers, BanaServe's scheduler routes requests purely based on current load metrics:

  1. Measure: For each prefill instance, compute utilization U and queue length
  2. Sort: Rank instances by (load, queue_length) ascending
  3. Dispatch: For each incoming request, assign to least-loaded instance (if below threshold delta_L) or shortest queue (fallback)

Complexity: O(|P| log |P| + |Q|). This is practical for real-time scheduling since |P| is small (tens of GPUs).

# Algorithm 1: Adaptive Module Migration (simplified) def adaptive_migration(devices, cfg, delta): load = {d: MeasureUtilization(d) for d in devices} # compute + memory overloaded = {d for d in devices if load[d] - min(load.values()) > delta} underloaded = {d for d in devices if max(load.values()) - load[d] > delta} while overloaded and underloaded: d_o = max(overloaded, key=lambda d: load[d]) d_u = min(underloaded, key=lambda d: load[d]) gap = load[d_o] - load[d_u] if gap >= delta and SupportsLayerMigration(d_o): MigrateLayer(d_o, d_u) # coarse: weights + KV elif gap >= delta and SupportsAttnMigration(d_o): MigrateKVHeads(d_o, d_u) # fine: KV heads only # Cost-benefit check: Benefit/Cost >= rho load[d_o] -= estimated_relief load[d_u] += estimated_relief update_sets(overloaded, underloaded, load, delta)

Performance Modeling

Optimization Objective

BanaServe formulates performance optimization as a multi-objective problem:

max_M   alpha * U_avg(M) - beta * T_avg_latency(M) + gamma * Theta(M)

where M is the migration plan, U_avg is average utilization, T_avg_latency is average TTFT+TPOT latency, and Theta is throughput. The weighting coefficients alpha, beta, gamma balance these objectives.

Latency Model

T_TTFT = T_p + T_x + T_q

Where T_p is prefill compute, T_x = T_load + T_fetch is KV cache transfer (from storage + remote GPUs), and T_q is queuing delay.

T_TPOT = T_d + T_c + T_m

Where T_d is decode compute, T_c is cache access latency, and T_m is memory bandwidth stall time.

Throughput Model

Theta = (N * L_out) / (T_TTFT + L_out * T_TPOT)

Given N concurrent requests and output length L_out. The scheduler periodically selects the migration plan M* that maximizes the weighted objective, subject to migration cost constraints T_mig(M) ≤ T_budget and latency SLOs.

Migration cost per k modules: Cost_mig = k * (T_x_lat + T_sync + T_mem_realloc).

Significance and Limitations

Beyond Static Disaggregation

BanaServe transforms disaggregated serving from a static deployment decision into a dynamic runtime optimization. The prefill-to-decode ratio adapts continuously based on real-time workload metrics, eliminating the need for offline capacity planning or service restarts.

Decoupling State from Scheduling

The Global KV Cache Store is a key architectural innovation: by separating KV cache placement from request routing, the system achieves both high cache hit rates (through the shared store) and balanced load (through pure load-based scheduling). This resolves the fundamental tension in cache-aware routing.

Multi-Granularity Migration

The two-level migration hierarchy (layer-level for coarse rebalancing, attention-level for fine adjustment) provides a flexible toolkit. Layer migration handles large imbalances quickly; attention migration fine-tunes without the overhead of weight transfer. The orchestrator selects the appropriate level based on the magnitude of imbalance and the cost-benefit ratio.

Analytical Modeling

BanaServe provides a comprehensive analytical framework for PD disaggregation performance, including utilization models, latency decomposition, throughput formulas, and migration cost equations. This model-driven approach enables principled optimization rather than ad-hoc heuristics.

Limitations: (1) The scheduling and migration policies lack comprehensive optimization for heterogeneous hardware (mixed-precision accelerators, specialized communication fabrics), potentially leaving performance on the table in mixed clusters. (2) Workload management is primarily reactive, relying on instantaneous load metrics without predictive modeling from historical patterns, which may be less effective under rapid workload fluctuations. (3) The design is restricted to single-region clusters and has not been extended to distributed, multi-region deployments with WAN-aware KV cache sharing. (4) Evaluation is limited to 13B-parameter models; validation on larger models (70B+) with tensor parallelism across many GPUs remains future work.

Related Work and Context

Monolithic LLM Serving Systems

vLLM (Kwon et al. 2023): Introduces PagedAttention for block-level KV cache management, enabling efficient memory fragmentation handling and continuous batching. However, vLLM co-locates prefill and decode on the same GPU, creating contention between compute-bound and memory-bound phases.

SGLang (Zheng et al. 2024): Optimizes prompt processing through prefix caching and cache-aware scheduling. Routes requests to instances with cached prefixes, which improves hit rates but creates the load skew problem that BanaServe addresses.

DeepSpeed-FastGen (Holmes et al. 2024): Builds on DeepSpeed-Inference with enhanced kernel fusion and adaptive batching. Like vLLM, it struggles with prefill-decode interference under strict latency SLOs.

TensorRT-LLM (NVIDIA 2024): Graph-level compilation with fused attention kernels and quantization-aware execution for NVIDIA GPUs. NVIDIA Dynamo extends this to multi-node setups with heterogeneous resource scheduling.

Disaggregated Serving Systems

DistServe (Zhong et al. 2024): Pioneer of GPU-level disaggregation, dynamically allocating separate GPU devices for prefill and decode. Introduces parallelism adjustments per phase but uses static resource allocation.

Splitwise (Patel et al. 2024): Advances disaggregation by partitioning processing phases across dedicated machines, enabling hardware-specific optimization per phase.

LoongServe (Wu et al. 2024): Refines disaggregation through sequence parallelism and flash decoding for long-context scenarios.

Mooncake (Chen et al. 2024) and MemServe (Hu et al. 2024): Explore distributed memory management with KV cache pools and elastic memory pools across instances.

KVDirect (Wang et al. 2024): Uses RDMA for direct KV cache transfer between machines. BanaServe's Global KV Cache Store shares a similar philosophy but adds layer-wise pipeline overlap.

Load Balancing in LLM Serving

Several approaches address load balancing in disaggregated settings:

  • Llumnix (Zhang et al. 2024): Dynamic load balancing through continuous monitoring and request redistribution between nodes
  • SpotServe (Li et al. 2024): Leverages spot instances with adaptive resource allocation for cost-effective serving
  • UELLM (Liu et al. 2024): Adaptive load balancing strategy for handling demand spikes
  • Sarathi-Serve (Agrawal et al. 2024): Taming throughput-latency tradeoff with balanced prefill-decode scheduling

These approaches typically assume homogeneous workloads and do not explicitly address the compute-memory asymmetry between prefill and decode phases. BanaServe's key differentiator is decoupling state management from scheduling through the Global KV Cache Store, enabling pure load-based routing.

Two 13B-parameter decoder-only models were evaluated to ensure cross-architecture validity:

Deployed on A100-80GB GPUs with NVLink interconnect for layer migration. Each experiment repeated 5 times with different random seeds, reporting mean values with 95% confidence intervals.

Benchmarks and Load Testing

Alpaca (short-context): 52,000 instruction-following examples with 5-40 token inputs. Tests scheduling, TTFT optimization, and high-frequency context switching.

LongBench (long-context): 21 diverse tasks across 6 categories with 2K-85K token contexts. Tests KV cache management, memory optimization, and long-sequence throughput.

Load testing with Poisson arrival process at 1-20 RPS. Each configuration warmed up for 60 seconds before 5-minute measurement windows. Maximum output length capped at 512 tokens for consistency.

Metrics

  • Throughput (tokens/s): Raw processing capacity reflecting GPU utilization and batch efficiency
  • Total Time (s): End-to-end request latency including all overheads (model loading, KV management, generation)
  • Average Latency (ms): Decomposed into TTFT and inter-token latency for user-experience analysis
  • SLO Attainment (%): Percentage of requests meeting target latency bounds

BanaServe achieves consistent advantages across all metrics, with the largest gains at high RPS (10-20) where conventional systems experience contention-driven degradation while BanaServe's dynamic rebalancing sustains near-optimal utilization.

Future Directions

Predictive Workload Management

The current reactive migration approach responds to observed load imbalance. Incorporating predictive models based on historical request patterns could enable proactive migration, reducing tail latency during traffic spikes. Time-series forecasting of request rates and context lengths would allow pre-positioning of layer weights before imbalance materializes.

Heterogeneous Hardware Support

Current policies assume homogeneous GPU clusters. Extending to mixed clusters (A100 + H100 + inference accelerators) with heterogeneous compute and memory capacities would require hardware-aware migration cost models and asymmetric load thresholds. The analytical framework already supports this extension through device-specific C_max and M_max parameters.

Multi-Region Deployment

The Global KV Cache Store currently operates within a single datacenter. Extending to WAN-aware KV cache sharing across geo-distributed sites would enable edge-to-cloud LLM serving with conversation continuity. Key challenges include latency-aware cache placement, selective cache replication, and consistency under network partitions.

Larger Model Validation

Scaling to 70B+ models with tensor parallelism across 8+ GPUs would test the migration overhead at scale. Layer migration for tensor-parallel models requires coordinated weight transfer across all GPUs in the TP group, which adds synchronization complexity. The attention-level migration is more naturally suited to TP models since attention heads can be independently partitioned.
