NIMBLE: From Skew to Symmetry

Node-Interconnect Multi-Path Balancing with Execution-time Planning for Modern GPU Clusters

arXiv:2604.00317 Preprint 2026 H100-SXM4 · NDR400 All-to-Allv Kernel-Based RDMA MoE / EP Serving

Jinghan Yao, Kaushik Kandadi, Bharath Ramesh, Hari Subramoni, Dhabaleswar K. Panda

Department of Computer Science and Engineering · The Ohio State University, Columbus, OH, USA

1. Problem: Skew Wastes the Fabric

Modern HPC/AI clusters pack enormous aggregate bandwidth per node — e.g., 8× NVLink intra-node and 4× InfiniBand NDR400 or RoCE rails inter-node. Yet real workloads (MoE token routing, DLRM embedding all-to-all, GNN, HPC stencils with boundary hotspots) routinely leave most links idle while a few saturate. This is traffic skew: not random imbalance, but structural, dynamic concentration of flows on a small subset of links.

Figure 1 (from paper): In skewed workload, heavy-load links dominate the overall latency. NIMBLE evaluates load imbalance and automatically redirects message transfer through less busy paths, reducing the maximum latency across all links.

1.1 Why Existing Libraries Fail

NCCL, RCCL, and MPI (including MPI-over-UCX) rely on static routing at init time: either fastest-path (peer-to-peer via the shortest NVLink/PCIe hop) or rail-matched (symmetric NIC pairs to minimize switch hops). Both pick one path per (src, dst) pair at initialization, so when actual payloads are skewed, hot paths queue up while cold paths sit idle.

1.2 Why Runtime Orchestration is the Only Answer

  • Traffic patterns are data-dependent (MoE expert selection, embedding lookups, GNN sparsity) — unknown at init time.
  • Offline profiling + synthesis (e.g., TACCL) takes minutes-to-hours on mixed-integer linear solvers, but the skew is transient.
  • Routing decisions must complete in milliseconds — comparable to all-to-all latency — or the overhead cancels the gain.

Contribution 1: Online MCF Balancing

Formulate link balancing as a multi-commodity flow (MCF) problem, solved with a capacity-normalized Multiplicative Weights Update (MWU) approximation in milliseconds.

Contribution 2: CUDA Kernel-based RDMA

Pipelines intermediate GPUs and rail-matched NICs in a single fused kernel so multi-hop forwarding is nonblocking, ordering-preserving, and deterministic.

Contribution 3: Transparent Integration

Drop-in behind AllToAllv / Send / Recv in existing NCCL/MPI apps — no framework or model code changes.

NIMBLE is a runtime-resident library that sits underneath NCCL-compatible point-to-point and collective calls. For every outgoing message, it consults the current link-utilization snapshot, decides whether to route the message directly or split it across a mix of direct, intra-node 2-hop, and inter-node NIC-forwarded paths, and then launches a single GPU kernel that drives all of the underlying transfers at once.

Key insight: Since the aggregate link budget of a modern node is 8× the per-link bandwidth, even a mildly skewed job leaves plenty of cold links to steal. Treating them as routable via 2-hop forwarding turns "one fat flow" into "many thin parallel flows" — a symmetry that both NCCL and MPI throw away.

3. Architecture

NIMBLE is positioned between the application/framework layer and the hardware interconnects. It intercepts communication calls from PyTorch MoE, DLRM, CFD solvers, graph engines, and stencil codes via the standard NCCL API (Send, Recv, AllToAllv) and orchestrates dynamic link-aware data transfers.

3.1 Why Three Components, Not Two

A naive rewrite might just replace NCCL's routing table. NIMBLE adds the Monitoring Module because the planner has no closed-form estimate of NVLink or NIC contention — the orchestration engine must see real-time utilization. And it adds the Kernel Scatter & Buffer Pipeline because simply issuing multiple ncclSend calls in a group would still serialize on host-side synchronization; the kernel pipeline fuses the scatter into one launch, so multi-hop forwarding never touches the CPU after launch.

4. Hop-Adaptive MCF Formulation

NIMBLE treats communication routing as a classic multi-commodity flow problem: each (source GPU → destination GPU) traffic demand is a commodity; the edges are the physical interconnect links; and the objective is to minimize the maximum link load $Z$ (the max-load minimization MCF form).

Figure 4 (from paper): Given 4-GPU 4-NIC computing nodes, intra-node link balancing utilizes all available links between source and destination GPUs; inter-node link balancing leverages rail-matched paths among nodes as well as intra-node links on both nodes.

4.1 Integer Program (Intuition Only)

The variables are: $K$ = set of messages (traffic demands), $E$ = set of directed links, $V$ = set of GPUs, $x_{k,e}$ = units of flow for message $k$ on edge $e$, $Z$ = maximum link load across $E$.

(1) $$\min \; Z$$
(2) $$\sum_{e \in \text{out}(v)} x_{k,e} - \sum_{e \in \text{in}(v)} x_{k,e} = \begin{cases} d_k & \text{if } v = s_k \\ -d_k & \text{if } v = t_k \\ 0 & \text{otherwise} \end{cases} \quad \forall\, k \in K,\; v \in V$$
(3) $$\sum_{k \in K} x_{k,e} \leq Z \quad \forall\, e \in E$$
(4)–(5) $$x_{k,e} \in \mathbb{Z}_{\geq 0} \quad \forall\, k \in K,\; e \in E \qquad Z \in \mathbb{Z}_{\geq 0}$$

General MCF with integer unknowns is NP-hard and (for realistic clusters with hundreds of GPUs and thousands of messages) takes seconds-to-hours with MILP solvers — far longer than the communication latency it is trying to optimize. That is why NIMBLE does not solve this program directly.

4.2 Candidate Path Set (Hop-Adaptive)

Instead of considering every simple path, NIMBLE restricts the search to a hop-adaptive set built from hardware topology:

  • Intra-node direct path: one-hop NVLink or NVSwitch between source and destination GPUs.
  • Intra-node 2-hop path: forward through a third GPU sibling that has idle NVLinks on both sides (src→fwd→dst).
  • Inter-node direct path: rail-matched NIC pair that sits on the sender and receiver nodes.
  • Inter-node NIC-included path: rail-matched NIC plus an additional sibling GPU on the same node that owns a currently idle NIC.

This decomposition keeps the candidate set small ($|K| \times 8$ paths in the worst case for a typical 8-GPU / 4-NIC node) while preserving multi-path diversity. Each candidate path is scored by a custom cost function $c_e = F(L_e)$ that grows sharply as the link load $L_e$ rises, discouraging reuse of hot links.

5. MWU Link-Load Balancing Algorithm

To solve the hop-adaptive MCF within practical time constraints, NIMBLE employs an iterative approximation algorithm inspired by the Multiplicative Weights Update (MWU) framework — notably the Garg-Könemann algorithm for minimum-congestion MCF. Communication link costs are dynamically updated based on current load, and routing decisions are remade in successive iterations.

5.1 Pseudocode (adapted from paper Algorithm 1)

function MWU_LinkBalance(GPUs N, NICs Nic, Links E,
                         Pairs K, workloads d, fraction λ, chunk ε):
    # Initialize edge load L_e and cost c_e = F(0)
    for e in E:
        L[e] = 0;   c[e] = F(L[e])

    r[s,d]   = d[s,d]   for all (s,d) in K
    D_tot    = Σ d[s,d];   R_tot = D_tot

    Paths[s,d]  = []     # empty list of chosen (path, fraction)
    Flows[s,d]  = []

    while R_tot > 0:
        for each pair (s,t) in K with r[s,t] > 0:
            best  = ∞;   P_best = None

            # ---- 1. Direct path ----
            c_dir = c[ (s,d) ]
            if c_dir < best:
                best = c_dir;  P_best = [ (s,d) ]

            # ---- 2. Intra-node 2-hop paths via sibling GPU i ----
            for i in N \ {s, d}:
                e1 = (s, i);  e2 = (i, d)
                cost = max(c[e1], c[e2])                 # bottleneck
                if cost < best:
                    best = cost;  P_best = [e1, e2]

            # ---- 3. Inter-node NIC-included 3-hop paths ----
            for i in N:
                e1 = (s, i_s);  e2 = (i_s, i_d);  e3 = (i_d, d)
                cost = max(c[e1], c[e2], c[e3])
                if cost < best:
                    best = cost;  P_best = [e1, e2, e3]

            # ---- Chunked residual routing ----
            f_route = 0
            if r[s,d] < ε:
                f_route = r[s,d]                         # residual
            else:
                f_route = ⌊ r[s,d] · λ ⌋_ε               # multiple of ε

            if f_route > 0:
                Paths[s,d].append(P_best)
                Flows[s,d].append(f_route)

                # Update load and cost on every edge in chosen path
                for e in P_best:
                    L[e] += f_route;   c[e] = F(L[e])

                r[s,d] -= f_route;   R_tot -= f_route

    return Paths, Flows

5.2 Why MWU Terminates in Milliseconds

  • Residual-chunked routing: each iteration routes only a fraction $\lambda$ of the remaining demand, so costs update smoothly and no single path is overloaded in one step.
  • Custom cost function: $F(L_e)$ grows sharply (not exponentially as in textbook Garg-Könemann, which is too aggressive for short-lived bursts) to steer flows away from saturation while keeping numerical stability.
  • Small candidate set: only $O(|K|)$ direct + $O(|K| \cdot |N|)$ 2-hop + $O(|K| \cdot |\text{NICs}|)$ 3-hop paths are examined per iteration.
  • Capacity normalization: costs are scaled by each link's physical peak so a fast NVLink edge and a slower IB NIC edge are compared on equal footing.

Combined, these choices let the planner converge in sub-millisecond time on an 8-GPU node — comparable to the cost of issuing the underlying ncclSend/Recv calls.

Paper measurement: For sizes 16/32/64/128/256 MB, NIMBLE's orchestration algorithm time ranges from 0.0221 ms to 0.0480 ms intra-node, and 0.0374 ms to 0.6390 ms inter-node (Table I). At no point does it exceed 1 ms, compared to 0.1–2 ms for the underlying NCCL communication itself.

6. CUDA Kernel-Based RDMA Pipelining

Even a perfect routing plan is useless if the transfers are serialized on the CPU. NIMBLE solves this with a fused CUDA kernel that performs the scatter, forward, and gather stages in one launch, spanning both intra-node NVLink traffic and inter-node RDMA Verbs.

Figure 5 (from paper): On every forwarding GPU, a small peer-to-peer buffer is allocated and RDMA memory regions are registered if a NIC is involved. These intermediate buffers serve as the buffer pool in the pipeline, much smaller than actual data buffers on sender/receiver sides.

6.1 Source → Forwarding → Destination Pipeline

Source GPU — Kernel Scatter

  • Thread blocks partition the payload by chunk-id according to the MWU plan
  • Writes directly into peer GPU RDMA-mapped P2P buffers (CUDA IPC handles)
  • Decouples dispatch from completion via per-thread send counters
↓

Forwarding GPU — Non-blocking Pass-through

  • Intermediate buffers are pre-allocated and recycled on the intra-node pipeline
  • Forward path can cross over to inter-node RDMA send on the destination-side sibling NIC
  • Ordering is preserved via per-channel sequence counters — determinism matches baseline NCCL bit-for-bit
↓

Destination GPU — Kernel Gather

  • Per-channel recv counter polls incoming buffers against the scatter plan
  • Reassembles chunks by chunk-id back into a single contiguous receive buffer
  • Signals completion to the application layer through NCCL's standard event API

6.2 Peer-Exclusive Kernel Pairing

In NIMBLE's orchestration, a given GPU may simultaneously (1) send part of its own payload, (2) forward a chunk from a sibling, and (3) receive yet another chunk. To run all three in parallel, every GPU performs a one-time peer-exclusive kernel pairing: at init time, all send/recv queue pairs are wired up to the same pre-declared P2P buffer pool that persists across the whole application, and different channel groups are assigned to disjoint thread blocks. No host-side synchronization is needed between source-role and sink-role work on the same GPU.

6.3 Integration Path: Link Against libnccl

NIMBLE ships as a modified libnccl.so that exposes two new primitives (ncclSendP / ncclRecvP) alongside the standard ncclSend / ncclRecv / AllToAllv API. Applications compiled against vanilla NCCL headers continue to work; applications opting into NIMBLE simply swap in the replacement library via LD_LIBRARY_PATH.

Two things stand out in this call site:

  1. ncclSendP takes the full send_sizes vector (world_size × world_size entries), not just the one entry ncclSend would take. NIMBLE uses the global vector to plan multi-path routing across all (src,dst) pairs in the same call.
  2. The call site is still wrapped in a standard ncclGroupStart/End, so NIMBLE is transparent to any NCCL-aware framework.

8.3 MoE End-to-End Latency

Figure 8 (from paper): MoE end-to-end latency breakdown for NCCL vs. NIMBLE. For each token size, paired stacks show NCCL (left) and NIMBLE (right) with dispatch, compute, and combine phases; the purple trace overlays the end-to-end speedup.

For Mixture-of-Experts inference on two nodes × eight-GPU expert parallelism (4 experts per GPU, FP16, FFN dim 4096×4 expansion), NIMBLE is measured against NCCL at global token counts {2K, 4K, 8K, 16K, 32K, 64K} and hotspot ratios {0.4, 0.5, 0.6, 0.7, 0.8, 0.9}:

  • Average end-to-end speedup climbs from 1.13× at hotspot 0.4 to 1.26× at hotspot 0.9.
  • Peak speedup 1.35× at 16K tokens under 90% hotspot — dispatch/combine dominate there and NIMBLE's symmetry yields the largest reduction.
  • Compute phase (green bars in Figure 8) is identical across NCCL and NIMBLE — all gains come from slimmer dispatch and combine blocks.
Practical guidance: The paper recommends enabling NIMBLE when (i) global tokens ≥ 16K AND (ii) hotspot ratio ≥ 0.7, where gains often reach ~1.3×. For smaller or milder jobs (2K tokens, hotspot 0.5), prefer batching multiple steps before switching to amortize the orchestration overhead.

Strengths & Limitations

Strengths

  • Transparent drop-in — no framework or model code changes
  • Determinism preserved: bit-identical vs vanilla NCCL
  • Runtime overhead is sub-millisecond even for 256 MB messages
  • Captures genuine multi-path capacity (up to 3.8× intra, 3.5× inter)
  • Microbenchmark is self-contained and easy to reproduce

Limitations

  • Single-NVSwitch nodes (e.g., DGX A100/H100 with 1 NVSwitch): 8 GPUs share a central NVSwitch, so intra-node forwarding becomes tricky — every multi-hop path still has to route through the same switch
  • Library itself is closed-source — auditing the MWU implementation requires trusting the paper
  • Paper does not discuss interaction with CC layers (DCQCN, HPCC) — NIMBLE relies on them for fairness across tenants
  • Small / mildly skewed workloads (≤2K tokens, ratio ≤0.5) show no meaningful gain
  • Cross-job isolation relies on the underlying fabric, not NIMBLE itself

NIMBLE's distinctive contribution is compressing path planning into a sub-millisecond runtime cost. TACCL and MSCCL already solve much the same MCF problem, but their solutions are frozen at synthesis time and cannot adapt to MoE expert skew that changes every step. The closest runtime competitors — CC-layer mechanisms like DCQCN and HPCC — operate at the queue level and cannot re-path messages the way NIMBLE does.

Open Questions

  1. Can NIMBLE co-exist with TACCL/MSCCL compile-time schedules by using them as defaults and only re-pathing on detected skew?
  2. How does NIMBLE behave under multi-tenant contention where another job owns half the NICs?
  3. Single-NVSwitch topologies limit intra-node forwarding — the authors explicitly flag this as a limitation; what topology changes would unblock it?
  4. No source code for the MWU planner means independent researchers cannot audit fairness, convergence, or edge cases — will the authors open-source it?
