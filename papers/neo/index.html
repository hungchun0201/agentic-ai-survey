NEO: Saving GPU Memory Crisis with CPU Offloading for Online LLM Inference

MLSys 2025

Xuanlin Jiang, Yang Zhou, Shiyi Cao, Ion Stoica, Minlan Yu

An online LLM inference system that offloads attention compute and KV cache to the host CPU, increasing GPU batch size and throughput by up to 7.5x on memory-constrained GPUs.

arXiv 2411.01142 Paper PDF GitHub Repo

Overview

Modern LLM inference engines rely on request batching to improve throughput on expensive GPUs. However, GPU memory is severely limited: models like LLaMA-3.1-70B consume dozens of GB just for weights, and KV caches grow linearly with sequence length. This GPU memory crisis leaves significant GPU compute resources wasted.

NEO is the first CPU offloading system for online LLM inference that achieves performance gains over GPU-only approaches on the same hardware. It offloads decoding attention computation and KV cache states from the GPU to the local host CPU, effectively increasing the GPU batch size and thus inference throughput, without compromising accuracy or latency.

Motivation: The GPU Memory Bottleneck

GPU Memory is Scarce

A T4 GPU has only 16 GB VRAM. After loading LLaMA-2-7B weights (~14 GB), barely any memory remains for KV cache, limiting the batch size to a handful of requests.

Batch Size Limits Throughput

Throughput increases almost linearly with batch size on modern GPUs (A10, V100, H100). Small batch sizes leave massive GPU compute resources unused.

CPU Memory is Abundant

Host CPUs come "free" with GPU instances: an AWS g5.4xlarge has 64 GB of CPU memory alongside a single A10G GPU. NEO exploits this asymmetry.

Why CPU Offloading Works for Attention

The decoding attention operation is memory-bandwidth-bound on both GPU and CPU due to low arithmetic intensity. Crucially, the memory bandwidth gap between GPUs and CPUs is much smaller than their compute gap:

Since decoding attention is memory-bandwidth-bound, the actual performance gap between GPU and CPU for this specific operation is only ~3x (600 vs. 200 GB/s), not 100x as the raw FLOPS would suggest. This makes CPU offloading viable for attention.

System Architecture

NEO Architecture Overview

NEO consists of a request scheduler on the CPU control plane that maintains three queues (prefilling waitqueue, GPU decoding runqueue, CPU decoding runqueue), a GPU worker executing linear operations and prefill attention, and a CPU worker running the PACPU (Paged-Attention-for-CPU) library. The scheduler makes iteration-level adaptive decisions on whether to run incoming requests on GPU or CPU. NEO features two key techniques:

  1. Asymmetric GPU-CPU pipelining to fully leverage both GPU and CPU compute within each iteration
  2. Load-aware scheduling to handle dynamically-changing workloads across iterations

Asymmetric Pipelining

NEO's core innovation is asymmetric pipelining, which divides work between GPU and CPU in a carefully unbalanced way that maximizes throughput. This design emerged from analyzing two strawman approaches:

Strawman 1: Simple Offloading

Offloads all decoding attention and KV cache to CPU. Problem: CPU is always idle during GPU linear operations, wasting compute.

Strawman 2: Symmetric Pipelining

Splits decoding batch into two equal halves, overlapping linear (GPU) and attention (CPU). Problems: GPU memory underutilization, unbalanced overlap, impractical equal splitting.

NEO: Asymmetric Pipelining

Forms two deliberately unequal sub-batches. Batch-0 is large (prefill + GPU decode), batch-1 is attention-heavy (CPU decode). Complementary structure minimizes idle periods.

Asymmetric Pipeline Execution Timeline
Why Asymmetric?

Batch-0 consolidates all prefilling requests and GPU decoding requests -- it has a long linear stage with little CPU attention. Batch-1 holds mostly CPU decoding requests -- it has a short linear stage but long CPU attention. The two batches are complementary: when GPU processes batch-0's linear ops, CPU computes batch-1's attention, and vice versa. This "long-short alternation" minimizes pipeline bubbles.

Layer-Wise Swapping

NEO leverages layer-wise swapping: since KV values of newly prefilled requests are computed layer by layer, PCIe transmission of KV cache to CPU can begin immediately after each layer rather than waiting for the entire iteration. This overlaps communication with computation.

Load-Aware Scheduling

Real-world inference workloads are dynamically changing with variable input/output lengths. NEO's scheduler follows four key principles:

Greedy

Compare GPU-only schedule vs. two-batch asymmetric pipelining; pick whichever has higher estimated throughput (T/x ratio).

Balancing

Minimize pipeline bubbles by ensuring estimated CPU busy time and GPU busy time are as close as possible: T_l0 >= T_ca1 and T_l1 + T_ga0 >= T_ca0.

Hiding CPU

Ensure the CPU is never busy while the GPU is idle. The CPU attention should always overlap with GPU linear operations.

Maximizing GPU

Pack as many requests as possible into the GPU (from waitqueue and via swap-in), to maximize GPU memory utilization.

# Simplified scheduling logic from swiftllm/server/scheduler.py
class Scheduler:
    def get_next_batch(self) -> tuple[list[SubBatch], ...]:
        # If always_use_gpu mode, use traditional scheduling
        if self.engine_config.always_use_gpu:
            return self._get_next_batch_old()
        # Otherwise, use NEO's asymmetric pipelining scheduler
        return self._get_next_batch_new()

    def _decide_mode_and_gen_batch(self, gpu_pref, cpu_pref, budget):
        batches = [SubBatch(self.predictor) for _ in range(2)]
        gpu_only_batch = SubBatch(self.predictor)

        # Step 1: Put all prefill + GPU decode into batch-0
        for req in gpu_pref: batches[0].add_pref(req, is_gpu=True)
        for req in cpu_pref: batches[0].add_pref(req, is_gpu=False)
        for req in self.gpu_decoding_q: batches[0].add_gdec(req)

        # Step 3: Split CPU decoding requests across batches
        for req in self.cpu_decoding_q:
            batches[next_batch_idx].add_cdec(req)
            remains = self._get_remains(batches)
            if min(remains) < 0: continue  # skip if overloaded
            next_batch_idx = remains[1] > remains[0]  # pick less loaded

        # Step 5: Compare pipelined vs sequential throughput
        sequential_rate = len(gpu_only_batch) / sequential_time
        pipelined_rate = sum(len(b) for b in batches) / pipelined_time
        return batches if pipelined_rate > sequential_rate else [gpu_only_batch]

Implementation Details

NEO is built on SwiftLLM, a lightweight LLM inference engine (~4K lines Python + 1.5K lines C++) that provides vLLM-equivalent performance on a single GPU. Key implementation highlights:

PACPU: CPU Attention Kernels

A custom C++ torch extension library called Paged-Attention-for-CPU. It calls multi-threaded C++ code that internally uses ISPC (Intel SPMD Program Compiler) for vectorized SIMD instructions (AVX2, AVX512, ARM Neon).

Paged KV Cache

PACPU uses the same paged memory layout as vLLM's PagedAttention on GPU, with block-granularity allocation to avoid fragmentation on both CPU and GPU memory.

Flash-Decoding on CPU

For each request, computation is partitioned into tasks along the request dimension. Each thread handles equal numbers of blocks, with outputs aggregated via a parallel reduction.

CUDA C++ Kernels

Replaced Triton-JIT kernels with hand-written CUDA C++ to reduce kernel launch overhead. This is critical since Python's GIL prevents parallel CPU and GPU kernel launches.

Block Manager: Dual-Device Memory Control

# From swiftllm/server/block_manager.py
class BlockManager:
    def __init__(self, engine_config, model_config):
        self.gpu_block_manager = DeviceBlockManager("cuda", engine_config)
        self.cpu_block_manager = DeviceBlockManager("cpu", engine_config)

    def _initiate_swap(self, reqs, is_swap_out, use_itm=False):
        """Swap KV cache blocks between GPU and CPU."""
        src_mgr = self.gpu_block_manager if is_swap_out else self.cpu_block_manager
        dst_mgr = self.cpu_block_manager if is_swap_out else self.gpu_block_manager
        src_pids = src_mgr.free(reqs)       # Free source blocks
        dst_vids, dst_pids = dst_mgr.alloc(reqs) # Allocate destination
        return src_pids, dst_vids, dst_pids

# From swiftllm/perfpredictor.py -- Used for scheduling decisions
class TablePerfPredictor(PerfPredictor):
    """Uses offline profiling + linear interpolation to estimate:
       - get_linr_T(S): GPU linear layer time for iteration width S
       - get_pref_T(S): GPU prefill attention time
       - get_gdec_T(N): GPU decoding attention time for N tokens
       - get_cdec_T(S,N): CPU decoding attention time (bilinear interp)
    """
    def get_cdec_T(self, S: int, N: int) -> float:
        # Bilinear interpolation across batch size (S) and token count (N)
        s_idx = self.cdec_S_lb_idx[S]
        ts1 = self._interp_1d(N, self.cdec_N_list_agg, self.cdec_T_lists[s_idx])
        ts0 = self._interp_1d(N, self.cdec_N_list_agg, self.cdec_T_lists[s_idx-1])
        return self._interp(S, s0, s1, ts0, ts1)

Comparison with FastDecode+

FastDecode+ (which offloads all decoding attention to CPU) suffers from inflexibility with dynamic workloads. When there are many CPU decode requests but few prefilling requests, FastDecode+ cannot launch GPU batches, while NEO's greedy scheduler can always fall back to GPU-only mode. As output lengths grow, FastDecode+ drops to less than 60% of baseline, while NEO remains strictly above baseline.

Relevance to Agentic AI Systems

NEO is directly relevant to the infrastructure requirements of agentic AI. Autonomous agents powered by LLMs require continuous, low-latency inference with large context windows. As agents maintain long conversation histories, tool-use logs, and reasoning traces, KV cache pressure grows rapidly, making GPU memory the primary bottleneck.

Multi-Agent Sessions

NEO's CPU offloading enables significantly more concurrent agent sessions on the same GPU hardware. By storing KV caches in CPU memory (up to 120 GB on g5.16xlarge), the effective batch size increases dramatically, directly translating to more agents per GPU.

Cost Efficiency

Agent workloads are often memory-bound rather than compute-bound. NEO uses the CPU that already ships with GPU instances (at no extra cost) to absorb the memory-bandwidth-bound attention computation, delivering free throughput gains.

Latency Preservation

Unlike quantization-based approaches, NEO preserves full model accuracy and maintains comparable per-token latency to GPU-only inference. This is critical for reasoning-intensive agent tasks where accuracy cannot be traded for throughput.

Complementary with MLA

NEO's CPU offloading approach is complementary to architectural innovations like Multi-Head Latent Attention (MLA) used in DeepSeek-V2/V3/R1. While MLA reduces the KV cache size at the model level, NEO provides a system-level solution that works with any existing model architecture. Combining both approaches could yield multiplicative benefits: smaller KV caches (MLA) distributed across more memory (NEO).
