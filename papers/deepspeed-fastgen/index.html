DeepSpeed-FastGen: Dynamic SplitFuse

High-throughput text generation for LLMs via a novel prompt and generation composition strategy. Up to 2.3x higher effective throughput, 2x lower latency, and 3.7x lower P95 tail latency compared to vLLM.

Authors: Connor Holmes, Masahiro Tanaka, Michael Wyatt, et al.
Affiliation: Microsoft DeepSpeed
Repo: github.com/microsoft/DeepSpeed

Overview

DeepSpeed-FastGen introduces Dynamic SplitFuse, a novel token composition strategy for prompt processing and generation that fundamentally changes how LLM serving systems schedule forward passes. Unlike prior systems that treat prompt processing and token generation as separate, competing phases, Dynamic SplitFuse decomposes long prompts and fuses short prompts with generation tokens to maintain consistent, optimally-sized forward passes.

Core Insight

The composition of sequences in a forward pass (batch size) has a negligible impact on performance compared to the raw number of tokens. An effective scheduler can therefore be built around a single signal: the total number of tokens in each forward pass.

⚒

Split: Long Prompt Decomposition

Long prompts are split into smaller chunks scheduled across multiple forward passes, with only the final pass performing generation. This prevents preemption of ongoing generation.

★

Fuse: Short Prompt Composition

Short prompts are composed to fill a target token budget. Even short prompts may be decomposed to ensure precise budget alignment and well-sized forward passes.

⚡

Consistent Forward Passes

Every forward pass operates in the throughput-saturating region, where the GPU achieves peak TFLOPS, ensuring maximum hardware utilization at all times.

📈

No Preemption

Unlike vLLM (which preempts generation for prompt processing), SplitFuse processes prompts and generation concurrently, eliminating tail latency spikes.

Dynamic SplitFuse Mechanism

Three Key Performance Insights

Insight 1: Token Count Dominates

Token latency is predominantly determined by the number of forward tokens, not batch size. Batch sizes of 1, 2, 4, and 8 produce nearly identical latency curves when plotted against forward tokens.

Insight 2: Throughput Saturation

LLMs have two operating regions with a steep transition. Below the saturation point, throughput scales with tokens (memory-bound). Above it, throughput is near-constant (compute-bound). SplitFuse keeps all passes in the saturated region.

Insight 3: Equal Partitioning

For a concave latency-to-throughput function f(x), the partitioning that maximizes throughput for P tokens over F forward passes is equal division: P/F tokens per pass.

Chunked Prefill Connection to Sarathi

A similar approach was independently proposed in Sarathi (Agrawal et al., arXiv:2308.16369), which splits prompts into smaller chunks to combine more token generation with prompt processing. SplitFuse extends this by also fusing short prompts upward to fill the token budget, ensuring every forward pass is optimally sized.

Design Tradeoff: Chunked Prefill vs. Disaggregation

Chunked prefill (as in SplitFuse) mixes phases on one GPU, yielding better data locality and no KV cache transfer overhead. In contrast, PD disaggregation separates prefill and decode onto different GPUs, providing better isolation but incurring KV transfer costs. This fundamental tradeoff between co-location and separation defines the design space for modern LLM serving systems.

Effective Throughput (SLA-Based Metric)

The SLA framework considers both first-token latency (tokens_in_prompt/512 seconds) and generation latency EMA (2, 4, or 6 tokens/sec). Only requests meeting both SLAs count as successful. DeepSpeed-FastGen achieves up to 2.3x higher effective throughput than vLLM under this metric.

Usage Examples

Non-Persistent Pipeline

from mii import pipeline

# Create a quick inference pipeline (non-persistent)
pipe = pipeline("mistralai/Mistral-7B-v0.1")
output = pipe(
    ["Hello, my name is", "DeepSpeed is"],
    max_new_tokens=128
)
print(output)

Persistent Server Deployment

import mii

# Start a persistent GRPC server with load balancing
mii.serve("mistralai/Mistral-7B-v0.1")

# Connect from a client (can be multiple concurrent clients)
client = mii.client("mistralai/Mistral-7B-v0.1")
output = client.generate("DeepSpeed is", max_new_tokens=128)
print(output)

# Terminate when done
client.terminate_server()

Benchmarking Methodology

Performance was measured using 512 total requests with 1-32 concurrent clients. Prompt and generation lengths follow normal distributions (e.g., 1200/2600 prompt tokens, 60/128 generation tokens) with 30% variance to simulate realistic workloads. Both throughput-latency curves and effective throughput (SLA-based) metrics are reported.

Scalability

DeepSpeed-FastGen offers replica-level load balancing with near-perfect linear scaling. Using 4 A100 GPUs per replica on 8 nodes (16 replicas total) for Llama 2 70B, throughput scales from 1.46 qps (1 replica) to 23.7 qps (16 replicas), demonstrating a 16.2x speedup.
