Murakkab

Resource-Efficient Agentic Workflow Orchestration in Cloud Platforms

Declarative Profile-Guided MILP Optimizer arXiv:2508.18298

Gohar Irfan Chaudhry, Esha Choukse, Haoran Qiu, Íñigo Goiri, Rodrigo Fonseca, Adam Belay, Ricardo Bianchini

MIT CSAIL & Microsoft Azure Research

Overview

Murakkab is a resource-efficient serving system for multi-tenant agentic workflows, built on two principles: declarative workflow specification that decouples application logic from execution configuration, and an adaptive, SLO-aware runtime that dynamically reconfigures workflows to optimize quality, latency, energy, and cost.

2.8x GPU Reduction

Murakkab reduces GPU usage by up to 2.8x compared to state-of-the-art static baselines like LangGraph, through profile-guided optimization and cross-workflow multiplexing.

3.7x Less Energy

Energy consumption drops by up to 3.7x by intelligently selecting hardware configurations (H100 vs A100), model sizes, and parallelism strategies based on profiled trade-offs.

4.3x Cost Savings

Total cost reduction of up to 4.3x while maintaining quality and latency SLOs, achieved through dynamic workflow reconfiguration and resource multiplexing across tenants.

Motivation: The Fragmented Stack

Today's agentic workflows are fragmented across agent frameworks (LangChain, LlamaIndex), agent providers (OpenAI, Databricks), and cloud platforms (AWS, Azure, GCP). Each layer pursues different goals with minimal coordination, leading to three fundamental inefficiencies.

Figure 1 (from paper): Today workflow developers use frameworks to call agents from different providers hosted on multiple cloud platforms. This fragmentation results in inefficiencies.
Figure 2a (from paper): Video Q/A workflow: multi-modal with tools handling different modalities before feeding into an LLM for answering the query.
Figure 2b (from paper): Code generation workflow: text-only with an LLM-Debate structure to write, test and execute code.

1. Tight Coupling

Hard-coded parameters (models, hardware choices) in imperative workflow definitions prevent automated optimization. Any change requires manual updates and redeployment across the entire stack.

2. Disjoint Orchestration

Frameworks (build workflows) and resource managers (deploy and serve) operate in silos, causing suboptimal scheduling. Cloud platforms have no visibility into workflow internals.

3. Difficult Trade-offs

Configuration complexity grows combinatorially: O(#WorkflowKnobs x #AgentKnobs x #HardwareKnobs). Even modest workflows produce explosive Pareto frontiers across accuracy, latency, energy, and cost.

Insight 1: Opacity Problem

Cloud platforms lack visibility into workflow internals (tasks, requirements), preventing end-to-end optimization. Meanwhile, developers lack control or insight into system-level resource behavior. This two-sided opacity is the root cause of resource waste.

# Tightly coupled: logic + config

    fn=SceneDetector(),
    key=AWS_KEY,
    resources={"CPUs": 32}

question_answer = LLM(
    name="Llama-3.2",
    key=DATABRICKS_KEY,

    resources={"GPUs": 8,
               "Type": "H100"},
)
DECLARATIVE (Murakkab) 
# Decoupled: logic only
scene_detect  = "Given videos, identify scenes."
frame_extract = "Extract frames from scenes."
stt           = "Convert audio to text."
q_a           = "Answer the query given context."

def workflow(query, videos):
    scenes     = scene_detect(videos)
    frames     = frame_extract(scenes)
    transcript = stt(scenes)
    answer     = q_a(query, [frames, transcript])
    return answer

Executor Library

Murakkab maps unknown tasks to a finite set of known executors from its library. Each executor exposes three attributes: a textual description, an interface specification, and configurable parameters (knobs).

Executor Type Examples Configurable Knobs
LLM Gemma-3-27B, Llava-OneVision-7B, DeepSeek-Qwen-32B, Phi-4 Model, batch size, tensor parallelism, GPU type
Structured Composition LLM-Debate, Self-Reflection Number of debaters/rounds, model per agent
Tool OpenCV, SceneDetector, OmDet, Whisper Number of frames, CPU cores, precision
ML Model Word2Vec, CLIP, sentiment classifiers Hardware accelerator, batch size

Two-Layer Profiling

Workflow Profiles

Capture task accuracy and executor-level load under various combinations of workflow-level knobs (enable STT or not, number of frames) and executor-level knobs (model, frames extracted). Quality assessed using open-source benchmarks (VideoMME, HumanEval, Math).

Model Profiles

Capture performance across software configurations (tensor parallelism, prefill/decode separation) and hardware setups (GPU type, clock frequency). Each profile reports: (1) latency (TTFT and TPOT), (2) energy consumption, and (3) cost per configuration.

Profile-Guided Optimizer

The optimizer transforms the logical workflow from the orchestrator into an executable workflow by selecting: (1) workflow parameters, (2) models/tools per executor, and (3) hardware and parallelism strategies.

SLO Tier System

Each request specifies a quality, latency, and cost SLO tier. The four tiers correspond to percentile ranges of achievable quality and latency:

Tier Quality Percentile Latency Percentile Optimization Strategy
Best 95th percentile accuracy Tightest latency Largest model, most resources
Good 80th percentile Moderate latency Balanced model/resource mix
Fair 50th percentile Relaxed latency Smaller models, higher batching
Basic Minimum viable Most relaxed Lightest resources, max multiplexing

Runtime Execution

At runtime, Murakkab receives incoming requests from end-users with a payload containing the workflow identifier, input query/data, and SLOs. It looks up the registry to obtain the corresponding executable workflow and submits it for execution.

Dynamic Workflow Requests

End-users can invoke a particular workflow or send a natural language query without specifying a workflow. Murakkab dynamically composes workflows from existing building blocks, mapping sub-tasks to executors.

Auto-Scaler

Monitors per-model instance load over short windows (seconds to minutes) and rapidly scales out when needed. Thresholds are based on the performance-throughput characteristic in executor profiles, prioritizing SLO compliance over short-term allocation optimality.

Runtime Optimization

The optimizer runs in the background after every optimization epoch (60 minutes), adapting to the most up-to-date load and resource availability. Uses EWMA (alpha=0.5) to predict workload demand.

Key Evaluation Finding

Murakkab automatically leverages H100 GPUs as they become available, reducing A100 allocation from 1292 to 0 while cutting energy consumption from 24.7 MWh to 11.0 MWh. The system adapts its resource allocation without code rewrites or redeployments.

MILP Formulation

Murakkab formulates resource allocation as a temporal optimization problem for serving multiple workflows with heterogeneous SLOs across diverse model profiles.

Decision Variables

// Core decision variables
n_m in Z+       // Number of instances of model profile m
x_peak  in R+    // Peak load allocation from (workflow, SLO, config) to model m
x_avg   in R+    // Average load allocation from (workflow, SLO, config) to model m

Constraint Summary

Constraint Purpose Key Equation
Demand Satisfaction (Peak) Ensure peak demand is met with buffer Sum(x_peak) >= lambda_peak * alpha (alpha=1.15)
Demand Satisfaction (Avg) Similar bounds for average demand Sum(x_avg) >= lambda_avg * alpha
Capacity with Multiplexing Account for statistical multiplexing mu_m * Sum(x_peak * t_c) <= n_m * theta_m
Quality SLO Filtering Filter configs below quality threshold x_peak = 0 if a_c < tau_quality
Latency SLO Filtering Filter configs exceeding latency limit x_peak = 0 if TTFT + t_c*TPOT > tau_latency
Resource Budget GPU count per type within limits Sum(n_m * g_m) <= B_g for all GPU types g

Objective Functions

// Three optimization objectives

// 1. Minimize Energy
minimize  Sum( n_m * e_m * g_m )   // e_m = energy per model instance

// 2. Minimize Cost
minimize  Sum( n_m * g_m * c_g(m) ) // c_g = cost per GPU type

// 3. Maximize Accuracy Under Budget
maximize  Sum( x_avg * a_c ) / Sum( lambda_avg )
          - epsilon * Cost_total   // epsilon = 0.001

Key Takeaways

1. Declarative Decoupling Works

Separating workflow logic from execution configuration enables automated optimization that no amount of manual tuning can match. Developers focus on what to compute; the system decides how.

2. Profiles Are Lightweight

Profiling is performed once per configuration and reused across workflows. Decoupling workflow and model profiles enables independent evolution, rapid model onboarding, and efficient hardware adaptation.

3. Multiplexing Amplifies Savings

Joint optimization across workflow-SLO combinations maximizes colocation and sharing of model instances. Multiplexing alone adds 21% GPU reduction, 20% energy reduction, and 17% cost reduction.

4. 60-Minute Epoch Sweet Spot

The optimization frequency analysis reveals a clear three-zone structure. A 60-minute interval offers high utilization, low cost, and manageable shortage risk, balancing responsiveness and efficiency.
