System Architecture

llm-d integrates vLLM model servers with the Kubernetes Inference Gateway (IGW), Envoy proxy, and intelligent scheduling.

llm-d Kubernetes-Native Architecture
⚙
Kubernetes-native by design. llm-d does not reinvent the load balancer -- it extends Envoy proxy with LLM-specific scoring algorithms via the Kubernetes Inference Gateway (IGW) extension point. The control plane is the Kubernetes API. Everything composes with standard K8s primitives: HPA, LeaderWorkerSets, Helm, Prometheus.

Two Routing Modes

Approximate (Default)

Observes only the request traffic stream to predict prefix cache locality. No changes to vLLM required. Good baseline that improves over round-robin.

Precise (KV-Events)

Uses vLLM KV-Events (published over ZMQ) to build a real-time global cache index. Eliminates guesswork -- the scheduler knows exactly which blocks are cached on which pod. Supports centralized ZMQ endpoint or pod-discovery mode for HA.

Disaggregated Prefill / Decode

Scheduler-directed P/D disaggregation with NIXL point-to-point KV transfer, orchestrated by a sidecar alongside decode pods.

⚙

Scheduler-Directed RPC

The inference scheduler decides which prefill and decode instances receive each request. This allows latency and throughput to be traded off dynamically based on real-time load and SLA targets.

  • Independent scaling of prefill and decode pools (xPyD)
  • Runtime reconfiguration without restart
  • Heterogeneous hardware across pools
🔁

PD Sidecar

A lightweight sidecar runs alongside each decode instance. It coordinates the KV transfer transaction: instructs vLLM to provide point-to-point KV cache transfer over NIXL.

  • Supports IB/RoCE RDMA, TPU ICI, and DCN
  • UCCL-based transport resilience (v0.5)
  • Non-blocking: GPU continues forward passes during transfer
📈

Hardware Breadth

P/D disaggregation tested across NVIDIA (A100, H100, H200, B200), Intel XPU, Google TPU, and AMD MI250+. The NIXL abstraction enables a single code path for all transports.

  • vLLM KVConnector interface for pluggable backends
  • Wide expert parallelism (EP) for MoE models
  • DeepSeek-R1 reference architecture available

Tiered Prefix Cache Hierarchy

Offload KV cache from HBM to CPU RAM, local SSD, and shared remote storage for dramatically expanded cache capacity.

Tier 1: GPU HBM

Native vLLM prefix cache in high-bandwidth memory. Fastest access but most constrained. Always the first tier checked.

Tier 2: CPU RAM

Async offload to host pinned memory. Typically 4-8x more capacity than HBM. CPU-to-GPU transfer is faster than recomputation for most cases. Minimal operational overhead.

Tier 3: Local SSD / NVMe

For workloads that can tolerate slightly higher latency. Uses the llm-d filesystem backend or GDS for direct GPU access. Consider shared storage if capacity is still insufficient.

Tier 4: Shared Remote Storage

CephFS, GCP Lustre, IBM Storage Scale, S3. Massive capacity independent of deployment size. Multiple replicas share the same cache. New nodes get warm caches instantly on scale-up.

Why Tiered Caching Matters

💡
HBM is always the bottleneck. State-of-the-art inference engines cache prefix KV tensors in GPU HBM, but for large models, HBM is already saturated by model weights and active decode sequences. CPU offloading alone can provide 4-8x more cache capacity with negligible overhead for async offloading.

Cache-Tier-Aware Scheduling (WIP)

The inference scheduler is being extended to understand which cache tier holds the prefix for a given request. A hit in HBM is preferred over CPU RAM, which is preferred over SSD. This allows the scheduler to factor in cache location when scoring pods.

P2P Cache Sharing

Instances can form a peer-to-peer network to share caches across HBM and CPU memory without requiring additional storage infrastructure. This enables cache sharing without the operational overhead of external storage systems.

Enable CPU OffloadingBash
# In vLLM pod environment variables
VLLM_CPU_OFFLOAD_ENABLED=true
VLLM_CPU_OFFLOAD_GB=64  # Pinned CPU memory for KV cache

# Or via llm-d filesystem backend
VLLM_KV_CONNECTOR="llmd-fs"
VLLM_KV_BACKEND_PATH="/mnt/shared-storage"
Workload Variant Autoscaling

Traffic-, hardware-, and SLO-aware autoscaler that optimizes the mix of prefill, decode, and latency-tolerant instances.

Variant Autoscaler (WVA)

The Workload Variant Autoscaler measures the capacity of each model server variant (hardware, software, topology), derives a load function accounting for different request shapes and QoS, and assesses recent traffic mix to calculate the optimal instance configuration.

Measure

Capacity Profiling

Measures throughput and latency characteristics of each variant (e.g., B200 prefill vs. H200 decode).

Model

Load Function

Derives a load function that takes into account request shapes (ISL/OSL), QoS targets, and hardware-specific performance curves.

Optimize

Instance Mix

Calculates optimal mix of prefill, decode, and latency-tolerant instances. Can reassign P/D roles dynamically.

Execute

HPA + IGW

Uses standard Kubernetes HPA with IGW metrics for SLO-level efficiency scaling.

📈

Reproducible Benchmark Workflows

Standardized benchmark suites with automated tooling for reproducible performance measurement. Validated ~3.1k tok/s per B200 decode GPU with wide expert parallelism.

🗃

Hierarchical KV Offloading

Full tiered cache stack: HBM, CPU RAM, local SSD, shared remote storage. vLLM KVConnector abstraction with pluggable backends: LMCache, Mooncake, KVBM, llm-d FS backend.

🔧

Cache-Aware LoRA Routing

The inference scheduler now routes by LoRA adapter identity combined with prefix cache overlap. Multi-adapter serving achieves per-adapter cache locality.

🛡

Active-Active HA

Multiple inference scheduler replicas running in active-active mode. With pod-discovery mode, each replica maintains a global KV-Events view independently for resilience.

🔌

UCCL Transport Resilience

UCCL-based transport layer for P/D KV transfers provides resilience against network failures and supports graceful degradation.

⚡

Scale-to-Zero Autoscaling

Model server instances can scale down to zero replicas when idle and cold-start when traffic arrives. Ideal for development, testing, and low-traffic model variants.

Quick Start

Deploy llm-d on your Kubernetes cluster using Helm charts.

Step-by-Step DeploymentBash
# 1. Set up namespace
export NAMESPACE=llm-d
kubectl create namespace ${NAMESPACE}

# 2. Create HuggingFace token secret
kubectl create secret generic llm-d-hf-token \
  --from-literal=HF_TOKEN="hf_your_token_here" \
  -n ${NAMESPACE}

# 3. Deploy inference scheduling stack
cd guides/inference-scheduling
helmfile apply -n ${NAMESPACE}

# 4. Apply HTTPRoute
kubectl apply -f httproute.yaml -n ${NAMESPACE}

# 5. Verify deployment
helm list -n ${NAMESPACE}
kubectl get pods -n ${NAMESPACE}

# 6. Send test request
curl -s http://gateway-host/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3-32B",
       "messages":[{"role":"user","content":"Hello!"}]}'
