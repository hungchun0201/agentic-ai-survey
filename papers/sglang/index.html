SGLang — RadixAttention

Efficient Execution of Structured Language Model Programs through Automatic KV Cache Reuse and a Structured Generation Frontend

Lianmin Zheng, Liangsheng Yin, Zhiqiang Xie, Chuyue Sun, Jeff Huang, Cody Hao Yu, Shiyi Cao, Christos Kozyrakis, Ion Stoica, Joseph E. Gonzalez, Clark Barrett, Ying Sheng

Stanford University · UC Berkeley · Shanghai Jiao Tong University · Texas A&M University

NeurIPS 2024 arXiv:2312.07104 sgl-project/sglang Up to 6.4x Throughput

1 Overview

SGLang is a system for programming and executing complex LLM programs that require multiple generation calls, control flow, and structured I/O. It consists of a domain-specific frontend language and an optimized runtime.

Core Thesis: Modern LLM applications (agents, RAG, tree-of-thought, multi-turn chat) involve multiple, dependent generation calls that share significant prompt prefixes. Existing inference engines discard KV caches after each request, wasting computation. SGLang systematically exploits this multi-call structure for automatic KV cache reuse via a radix tree data structure, achieving up to 6.4x higher throughput.

The Problem

  • LLM programs require multiple generation calls with shared prefixes
  • Current engines discard KV caches after each request finishes
  • Programming multi-call workflows is tedious and error-prone
  • Constrained decoding (JSON, regex) decodes only one token at a time

The Solution: SGLang

  • Frontend: Python-embedded DSL with primitives for generation, selection, forking, and joining
  • Runtime: RadixAttention for automatic KV cache reuse across calls
  • Compressed FSM: Decode multiple constrained tokens in a single forward pass
  • API Speculative Execution: Reduce API calls for black-box models

2 Radix Tree Architecture

RadixAttention uses a radix tree (prefix tree / trie) to manage KV cache tensors across all requests. Unlike typical inference engines that discard caches after each request, SGLang retains them in a tree for automatic reuse.

Each edge carries a label denoting a substring or a sequence of tokens. Nodes store references to KV cache pages.
Left branch: multi-turn chat sessions share the system prompt prefix (node A). Right branch: few-shot queries share the examples prefix (node F).

Radix Tree Properties

  • Edges can be labeled with sequences of tokens (not just single elements)
  • Each node maps to KV cache tensors stored in non-contiguous, paged layout
  • Page size is equivalent to one token
  • LRU eviction removes least recently used leaf nodes first
  • Reference counters prevent eviction of nodes used by running requests

Memory Management

  • No fixed-size memory pool preallocated for cache
  • Cache and running requests share the same memory pool dynamically
  • Under memory pressure, cached tokens are evicted to favor larger batch sizes
  • Tree operations are maintained on CPU with negligible overhead (0.3%)
  • Compatible with paged attention and continuous batching

3 RadixAttention Mechanism

RadixAttention enables automatic and systematic KV cache reuse during runtime. Unlike existing systems that discard the KV cache after each generation, SGLang retains caches in a radix tree for efficient matching, insertion, and eviction.

Theorem 3.1: For a batch of requests, an optimal cache hit rate can be achieved by visiting the radix tree of the requests in depth-first search order, with a cache size ≥ the maximum request length. The longest-shared-prefix-first order is equivalent to a depth-first search order.

5 Cache-Aware Scheduling

The order of request execution significantly impacts cache hit rate. SGLang uses a cache-aware scheduling algorithm that prioritizes requests with longer shared prefixes to minimize cache thrashing.

Cache-Aware Scheduling: Longest-Shared-Prefix-First
Waiting Queue (sorted by matched prefix length)
Priority 1
Cached: System + Few-shot + Turn 1-3
New: Turn 4
Priority 2
Cached: System + Few-shot
New: Query Q4
Priority 3
Cached: System
New: Different context
Priority 4
New: Entirely new request (no prefix match)
Cached prefix (reused, no prefill)
New suffix (requires computation)

Batch Processing Mode

Requests are sorted by matched prefix length and prioritized with longest-shared-prefix-first order. This is provably optimal for maximizing cache hit rate when cache size ≥ max request length (Theorem 3.1). The ordering is equivalent to a depth-first search traversal of the full radix tree.

Online / Continuous Batching Mode

In the online setting, the pure DFS order is disrupted by asynchronous arrivals. SGLang approximates the DFS behavior on the augmented radix tree. The cache-aware scheduling achieves 96% of the optimal hit rate on average across benchmarks. Integration with fair scheduling methods is left as future work.

Cache Hit Rate Definition: hit_rate = number_of_cached_prompt_tokens / total_number_of_prompt_tokens. Across the evaluated benchmarks, cache hit rates range from 50% to 99%. Higher hit rate leads to larger batch sizes (less memory per request), higher throughput, and lower first-token latency.

6 Structured Frontend Language

SGLang is a domain-specific language embedded in Python. It provides composable primitives for generation, selection, parallelism control, and constrained output, drastically simplifying multi-call LLM program development.

Language Primitives

Primitive Syntax Description
gen gen("name", regex=...) Call the model to generate; store result in variable. Supports regex constraint and stop token.
select select("name", choices=[...]) Choose the highest-probability option from a list of candidates.
extend / += s += "text" Append a string to the prompt state.
[var] s["variable_name"] Fetch the result of a previous generation.
fork s.fork(n) Create n parallel copies of the prompt state (triggers Frontend Hint to runtime).
join join(forks) Rejoin parallel forks of the prompt state.
image / video image(path) Include image or video inputs for multi-modal models.

Example: Multi-Dimensional Essay Judge

A complete SGLang program implementing the branch-solve-merge prompting technique for multi-dimensional evaluation. Primitives provided by SGLang are highlighted.

SGLang Program
dimensions = ["Clarity", "Originality", "Evidence"] @function def multi_dimensional_judge(s, path, essay): # Set up chat template with multi-modal input s += system("Evaluate an essay about an image.") s += user(image(path) + "Essay:" + essay) s += assistant("Sure!") # Check relevance using select primitive s += user("Is the essay related to the image?") s += assistant(select("related", choices=["yes", "no"])) if s["related"] == "no": return # Fork into parallel branches for each dimension forks = s.fork(len(dimensions)) for f, dim in zip(forks, dimensions): f += user("Evaluate based on: " + dim) f += assistant("Judgment:" + gen("judgment", stop="END")) # Merge judgments and generate final output judgment = "\n".join(f["judgment"] for f in forks) s += user("Provide the judgment, summary, and a letter grade") s += assistant(judgment + " In summary," + gen("summary", stop=".") + " The grade of it is " + gen("grade")) # Constrained JSON output via regex schema = r'{"summary": "[\w\d\s]+\.", "grade": "[ABCD][+-]?"}' s += user("Return in the JSON format.") s += assistant(gen("output", regex=schema))
2.1x Code Reduction: An equivalent program using a standard OpenAI API-like interface would require 2.1x as many lines of code due to manual string manipulation and parallelism control. SGLang programs are also more readable because the prompt construction is explicit and sequential.

Interpreter Mode (Default)

The prompt state is managed as an asynchronous stream. Primitives like extend, gen, and select are submitted non-blocking to the stream. Similar to launching CUDA kernels asynchronously. Each prompt is managed by a stream executor in a background thread, enabling intra-program parallelism. Fetching results blocks until ready.

Compiler Mode

SGLang programs can be compiled as computational graphs and executed with a graph executor. This enables tracing and additional static optimizations. The compiler mode can further optimize execution by analyzing the program structure ahead of time. The paper focuses on interpreter mode results.

8 Limitations & Future Directions

While SGLang delivers significant improvements, several limitations remain that present opportunities for future research.

LRU Eviction Is Suboptimal for Agentic Workflows

The default LRU eviction policy operates without knowledge of future access patterns. In agentic workflows where prefix reuse is predictable (e.g., an agent that will clearly revisit certain tool templates), LRU may evict soon-to-be-needed caches. A frequency-aware or predictive eviction policy could yield better results for structured agent programs.

Starvation in Greedy Scheduling

The cache-aware scheduling algorithm uses a greedy longest-shared-prefix-first policy. While this maximizes throughput, it can starve requests with no prefix match if the queue is consistently filled with related requests. Integration with fair scheduling methods is left as future work.

Single-Level Memory Hierarchy

RadixAttention currently operates only on GPU memory. Extending the radix tree to span multiple levels of the memory hierarchy (GPU VRAM, CPU DRAM, SSD) could enable much larger effective cache sizes and handle workloads with longer conversation histories.

No Fuzzy Semantic Matching

The radix tree requires exact prefix matches at the token level. This misses opportunities for KV cache reuse when prompts are semantically similar but differ slightly in wording. Enabling fuzzy semantic matching within RadixAttention could unlock additional reuse.

Limited Static Optimization

The SGLang compiler could perform more advanced static optimizations such as automatic scheduling, memory planning, and program transformation. The current paper focuses on interpreter mode; compiler optimizations remain underexplored.

Additional Output Modalities

SGLang currently supports text, image, and video inputs but could be extended to support additional output modalities and more complex multi-modal generation patterns as models evolve.

Key Limitation for Agentic AI: The LRU eviction policy treats all cached prefixes as equally likely to be reused, evicting the least recently used leaf. In structured agent workflows with predictable future activation patterns (e.g., a planning agent that systematically revisits tool definitions), this policy is suboptimal. An eviction policy that incorporates workflow-level knowledge of upcoming reuse could significantly improve cache hit rates for agent-heavy deployments.
