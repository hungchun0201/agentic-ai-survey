KVFlow

Efficient Prefix Caching for Accelerating LLM-Based Multi-Agent Workflows

NeurIPS 2025 arXiv: 2507.07400 First Workflow-Aware KV Cache Management

UC San Diego · Amazon Web Services

1 Overview

Why workflow-awareness matters for multi-agent KV cache management

The Core Insight

LLM-based agentic workflows repeatedly invoke specialized agents -- each defined by a large, fixed prompt -- in structured execution patterns. Existing LLM serving systems (vLLM, SGLang) employ prefix caching to reuse KV tensors for these fixed prompts, but rely on Least Recently Used (LRU) eviction when GPU memory fills. LRU is fundamentally mismatched with agentic workflows: it evicts caches based on past access time, completely ignoring the known future execution order encoded in the workflow structure. KVFlow is the first system to bridge this gap.

Two Key Innovations

  • Workflow-Aware Eviction: Abstracts agent execution as an Agent Step Graph, assigns steps-to-execution values to each agent, and propagates these as eviction priorities into the KV cache tree -- replacing blind LRU with informed, future-aware cache management.
  • Fully Overlapped KV Prefetching: Proactively loads upcoming agents' KV caches from CPU to GPU in background threads, combined with status-aware scheduling to ensure zero cache miss stalls during generation.

2 The LRU Problem in Agentic Workflows

Why standard cache eviction fails for multi-agent systems

LRU Eviction (Existing Systems)

  • Evicts based on last access time only
  • Cannot anticipate future agent invocations
  • Discards caches right before they are needed
  • Frequent cache misses trigger costly recomputation
  • CPU-to-GPU reactive loading stalls generation
  • Dynamic suffixes retained unnecessarily

KVFlow Workflow-Aware Eviction

  • Evicts based on predicted future usage
  • Knows upcoming execution order from Step Graph
  • Retains caches needed by soon-to-execute agents
  • Dramatically reduces cache miss rate
  • Proactive prefetching overlaps CPU-GPU transfer
  • Varying suffixes always evicted first

4 Agent Step Graph

A flexible abstraction capturing execution dependencies among agents

What is the Agent Step Graph?

The Agent Step Graph is a directed graph where each node represents an agent invocation and each edge encodes a dependency relation. Unlike traditional control-flow graphs (CFGs) or DAGs, the Agent Step Graph supports diverse workflow structures through step aggregation functions attached to each node, enabling unified representation of synchronization barriers, conditional branches, and sequential pipelines.

Steps-to-Execution (STE): Each node's STE value estimates the minimum number of agent invocations before it will be activated. Lower STE means sooner execution. The currently active agent has STE = 0. These values guide all eviction and prefetching decisions.
Dependency Type Aggregation Function Semantics Example
Synchronization Barrier max(predecessors) + 1 Agent waits for all predecessors to complete Expresser needs results from both Executor 1 and 2
Conditional Branch min(predecessors) + 1 Agent runs after any one predecessor completes Expresser needs result from either Executor 1 or 2
Sequential Pipeline predecessor + 1 Agent runs immediately after its single predecessor Executor follows Planner in a chain

5 Workflow-Aware Priority Eviction

Replacing LRU with fine-grained KV node-level eviction priorities

From Agent Priorities to KV Node Priorities

A naive approach would assign eviction priorities at the agent level. However, in tree-structured prefix caches, multiple agents can share common prefix segments -- e.g., a shared system prompt. KVFlow therefore propagates priorities down to individual KV cache nodes, ensuring shared prefixes are retained as long as any dependent agent needs them.

Multi-workflow conflicts: When KV nodes are shared across concurrent workflows, KVFlow assigns the minimum (most conservative) priority among all referencing workflows, ensuring no active workflow suffers premature eviction.

Priority Assignment Rules

  • Varying suffixes (dynamic prompt parts) always receive priority +infinity -- evicted first since they change across invocations and are unlikely to be reused.
  • Fixed prompt nodes receive the STE value of their owning agent, assigned to the last KV node of the fixed prompt.
  • Shared parent nodes inherit the minimum priority among all children -- retained as long as any downstream agent is active.
  • Propagation direction: Priorities flow upward from leaf nodes to the root of the radix tree.

KV Node Status Tracking

Each cache node maintains a status variable to coordinate prefetching, scheduling, and eviction:

In GPU Memory -- ready for computation
Backup in CPU -- available for prefetch
Loading -- CPU-to-GPU transfer in progress
Offloading -- GPU-to-CPU transfer (protected from eviction)
i
PCIe full-duplex: Model forward (GPU compute) and KV loading (CPU-to-GPU transfer) utilize different hardware resources. PCIe supports full-duplex communication, allowing bidirectional transfers without contention. This is why the overlap is achievable without performance degradation.
Pseudocode
// KVFlow prefetching + status-aware scheduling logic
def on_agent_start(current_agent, step_graph):
    // Identify next agents from the Step Graph
    next_agents = step_graph.get_successors(current_agent)

    for agent in next_agents:
        kv_nodes = cache_tree.get_fixed_prefix_nodes(agent)
        if kv_nodes.status == "backup_in_cpu":
            // Proactively prefetch from CPU to GPU in background
            background_thread.start_load(kv_nodes)
            kv_nodes.status = "loading"

def schedule_next_request(ready_queue):
    // Status-aware: skip agents whose cache is still loading
    for request in ready_queue:
        kv_nodes = cache_tree.get_nodes(request.agent)
        if any(node.status == "loading" for node in kv_nodes):
            continue  // skip -- not ready yet
        return request  // dispatch this agent

def on_load_complete(kv_nodes):
    kv_nodes.status = "in_gpu_memory"
    scheduler.notify_ready(kv_nodes.agent)
Figure 4. Illustration of overlapped KV prefetching. Compared to reactive loading, KVFlow combines proactive prefetching that loads upcoming agents in advance with status-aware scheduling, minimizing the CPU-GPU transfer overhead.

7 Implementation Details

Built on SGLang v0.4.4 with modifications to both frontend and backend

Frontend Modifications

  • Each sgl.function corresponds to one agent
  • Just-in-time substitution embeds workflow metadata into HTTP requests
  • Metadata includes: agent identity, STE values for all agents, client ID
  • Primitive interface to mark fixed vs. dynamic prompt boundary
  • Heuristic fallback: tracks cache hit history to auto-detect fixed prefix

Backend Modifications

  • Extended radix tree to support per-node eviction priorities
  • Priority update logic triggered on each incoming request
  • Background thread pool for asynchronous CPU-to-GPU prefetching
  • Status variable per cache node (4 states)
  • Scheduler modified to check node status before dispatching

Client Tracking for Concurrent Workflows

Multiple agentic workflows may run concurrently on the same backend. Different workflows can define agents with the same name (e.g., both have a "Planner"). KVFlow assigns a unique client ID to each application instance, attached to every HTTP request, allowing the backend to disambiguate agent identities and maintain separate Step Graph contexts.

Scaling Behavior

  • Fixed prefix length: KVFlow's advantage grows with longer fixed prompts. At 8192 tokens, average speedup is 1.48x vs. only 1.28x at 4096 tokens. Longer prefixes mean higher cache miss cost, making proactive management more valuable.
  • Output token length: Relative gains diminish as output tokens increase (e.g., from 1024 output), because LLM decoding latency dominates total runtime and is orthogonal to cache management.
  • Concurrency: Higher fixed prompt length combined with higher concurrency provides the largest benefits, as cache pressure increases and uninformed eviction becomes costlier.

9 Evaluation: Real-World Workflows

MetaGPT-Style Workflows

MetaGPT structures agent collaboration around software engineering roles (Product Manager, Architect, Engineer, QA). Agents have fixed prompts containing detailed role descriptions, behavioral instructions, and few-shot examples -- often exceeding 3000 tokens per agent.

  • Sequential pipeline of specialized agents
  • Large fixed prompts with small dynamic suffixes
  • Cyclic execution for iterative refinement

PEER-Style Workflows

PEER (Plan, Execute, Express, Review) organizes four agents in a cyclic pipeline. Each agent has a role and an instruction, with prompts generated by an LLM to introduce variability while maintaining partially overlapping prefixes.

  • 4 agents per workflow instance
  • Financial QA dataset as input
  • Prompts range from dozens to several hundred tokens
  • Partially redundant prefixes across agents

10 Significance and Contributions

First system to exploit workflow structure for serving-level cache optimization

Paradigm Shift: From Backward-Looking to Forward-Looking Cache Management

Prior to KVFlow, all LLM serving systems (vLLM, SGLang, TensorRT-LLM) treated cache eviction as a purely reactive, backward-looking problem: evict what was used least recently. KVFlow demonstrates that for agentic workloads, the execution graph provides a forward-looking oracle that fundamentally changes the optimization landscape. This is the first system to bridge the gap between application-layer workflow semantics and system-level cache management.

Agent Step Graph

A novel abstraction that captures diverse workflow structures -- synchronization barriers, conditional branches, sequential chains -- through configurable step aggregation functions. More expressive than CFGs or DAGs.

Workflow-Aware Eviction

Fine-grained KV node-level priority assignment that propagates STE values through the cache tree. Naturally handles shared prefixes across agents and concurrent workflows through conservative min-priority merging.

Overlapped Prefetching

Fully overlapped KV prefetching combining proactive loading with status-aware scheduling. Eliminates all cache miss stalls by leveraging PCIe full-duplex capability and intelligent request reordering.

Orthogonal optimizations: KVFlow is complementary to speculative decoding, KV cache sparsity (attention sinks), and early exit strategies. These optimize the decoding phase, while KVFlow optimizes the prefill and cache management phase. They can be combined for further gains.

Implications for Multi-Agent Systems

  • Framework-agnostic: While prototyped on SGLang, the approach applies to any LLM serving system with prefix caching. The core idea -- exploiting workflow structure for cache decisions -- transfers to vLLM, TensorRT-LLM, and custom backends.
  • Scaling with agent count: As multi-agent workflows grow to 10+ agents with long fixed prompts, KV cache pressure intensifies. KVFlow's benefits scale proportionally with workflow complexity.
  • Real-world deployments: Production systems (MetaGPT, PEER, AutoGen) can integrate KVFlow with minimal changes -- just embedding Step Graph metadata in HTTP requests.
  • Semantic preservation: KVFlow guarantees bit-identical outputs since it only modifies cache management, not model computation. This makes adoption risk-free.

Limitations and Scope

  • Requires workflow knowledge: KVFlow needs the Agent Step Graph to be provided or inferred. Fully ad-hoc agent interactions without known structure cannot benefit from the eviction optimization.
  • Memory fragmentation: KVFlow does not resolve the fragmented memory layout of KV storage in SGLang. While it achieves better PCIe overlap, addressing fragmentation could yield further gains.
  • Diminishing returns at high output lengths: When output generation dominates total latency (e.g., 1024+ output tokens), the relative benefit of cache optimization decreases.
  • Concurrency ceiling: If concurrency is so high that all GPU memory is consumed by active requests, no prefix caches can be maintained, placing the scenario beyond KVFlow's optimization scope.
