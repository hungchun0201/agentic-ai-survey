Speculative Actions: A Lossless Framework for Faster Agentic Systems

Generalizing speculation to the entire agentic environment — LLM calls, tool APIs, MCP servers, and human responses — with lossless, as-if-sequential semantics and up to 55% prediction accuracy driving 20% end-to-end latency reduction.

Ye*, Ahuja*, Liargkovas*, Lu*, Kaffes, Peng · Columbia University

Overview

LLM-driven agents are shifting from single-shot predictions to long-running processes in rich environments: operating systems, game engines, e-commerce platforms, and multi-hop research workflows. Their execution is dominated by sequential API calls — each tool invocation, LLM query, MCP server request, or even human response must complete before the next step begins. This serialization creates a critical bottleneck that scales linearly with the number of environment interactions.

Speculative Actions breaks this sequential dependency by introducing a fast Speculator that predicts likely next actions and pre-launches them in parallel while a slower, authoritative Actor deliberates. When the Actor's decision matches a speculation, the pre-computed result is committed instantly. When it does not, the speculative work is safely discarded. The result is an as-if-sequential, lossless interface with parallel, opportunistic internals.

What Makes This Different

G

Generalizes Beyond Tool Calls

Unlike speculative tool calling (which targets individual tool invocations), this framework treats ALL environment interactions as speculative candidates: LLM calls, external APIs, MCP servers, and even human responses.

L

Lossless Semantics

The final output is identical to strictly sequential execution. Semantic guards, safety envelopes (only idempotent/reversible speculative actions), and repair paths ensure correctness without compromise.

M

MDP Formalization

The agentic system is modeled as an MDP (s_t, a_t). Every action is an API call that can be speculatively pre-launched. This formalization unifies LLM calls, tool calls, and human interactions under one framework.

A

Actor-Speculator Architecture

Actors are authoritative but slow (powerful LLMs with high reasoning effort). Speculators are cheap and fast (smaller models, simpler prompts, domain heuristics). They run concurrently with no interference.

Core Question Answered

"Must an agent interact with its environment in a strictly sequential manner?" — No. The paper proves that with non-zero prediction probability p > 0, speculative actions always reduce expected runtime, with asymptotic speedup approaching 1 - p/(1+p) · α/(α+β) where α and β are the speculator and actor rates.

Framework

Speculative Actions: Actor-Speculator Architecture

Unified Action Abstraction

The framework treats every environment interaction as an API call (h_t, q_t) that returns a response a_t. This abstraction subsumes three categories:

LLM Calls

Each invocation of an LLM within the agent (the main reasoning call, sub-agent queries, prompt chains) is an action whose response can be speculatively predicted.

Tool / MCP Server Calls

Terminal access, web search, deep research APIs, weather APIs, browser-use MCPs — each treated as an API call with speculative pre-launch capability.

Human-as-API Calls

Human responses (user inputs, approval workflows) can be abstracted as API calls and speculatively predicted, often with the longest latencies of all action types.

Safety Requirements

  • Semantic guards: Actors confirm equivalence of state transitions before committing speculative results.
  • Safety envelopes: Only idempotent, reversible, or sandboxed speculative side effects are permitted.
  • Repair paths: When mispredictions occur, rollback or compensating actions restore the correct state.

Core Algorithm

// Algorithm 1: Speculative Actions with k-way parallel next calls
Require: State s_0, horizon T, transition f, policy π, predictor g_hat, cache C

for t = 0 to T-1 do
    // 1. Policy decides the API call
    (h_t, q_t) ← π(s_t)

    if (h_t, q_t) ∈ C then             // Cache HIT from previous speculation
        a_bar_t ← C[(h_t, q_t)]
        a_t ← await(a_bar_t)          // await if not already returned
        s_{t+1} ← f(s_t, a_t)
        continue                        // skip to next step!
    end if

    // 2. Actor + Speculator run in PARALLEL
    a_bar_t ← h_t(q_t)                  // Actor: issue real request (async)
    {a_hat^(i)} ← await(g_hat(s_t))    // Speculator: k guesses

    for i = 1 to k do
        s_hat^(i) ← f(s_t, a_hat^(i))   // simulate next state per guess
        (h_hat, q_hat) ← π(s_hat^(i))  // derive next API call
        a_bar_next ← h_hat(q_hat)        // PRE-LAUNCH (non-blocking)
        C[(h_hat, q_hat)] ← a_bar_next   // cache the pending action
    end for

    // 3. Wait for Actor's true response
    a_t ← await(a_bar_t)
    s_{t+1} ← f(s_t, a_t)
end for
Lossless Guarantee

The final trajectory (s_0, a_0, s_1, a_1, ..., s_T) is identical to the trajectory produced by purely sequential execution. Speculation only affects timing, never correctness. On cache hit, the system skips ahead; on miss, it proceeds normally with zero overhead (speculative work is discarded).

Environment 1: Chess

Chess Speculation: Prediction Accuracy and Time Saved

Implementation built on TextArena. The Actor uses GPT-5 with high reasoning effort; the Speculator uses GPT-5 with low reasoning effort and a speed-optimized prompt. While the current player P deliberates, player Q's Speculator predicts P's move and pre-computes Q's response for up to k candidates. With 3 predictions across 5 runs of 30 steps: 54.7% accuracy, 19.5% time saved.

Theoretical Analysis

Proposition 1: Expected Speedup

Under Assumptions 1 (non-zero prediction accuracy p) and 2 (concurrent reversible pre-launch), with Speculator latency Exp(α) and Actor latency Exp(β) where β < α:

E[T_s] / E[T_seq] &xrightarrow;{T → ∞} 1 - p/(1+p) · α/(α+β)

Key implications:

  • Upper bound: Maximum latency reduction of 50% (p=1, α=∞)
  • Always beneficial: For any p > 0, expected runtime is strictly less than sequential
  • Expected hits: S_n = p/(1+p) · n + p^2/(1+p)^2 · (1-(-p)^n) by round n
  • Diminishing returns with k: Increasing speculations follows bounded marginal improvement

Extensions

Multi-Step Speculation

Predict not just the next step but up to s steps ahead, yielding a tree-search structure with deeper rollouts and potentially higher total hit rates.

Adaptive Speculation

Estimate confidence per guess (via prompting or uncertainty quantification). Expand most promising branches in a beam-search-like manner rather than uniform k guesses.

Cost-Latency Tradeoff

More speculative calls improve accuracy but raise token/API costs. For self-hosted LLMs, cost is mitigated through batching; for commercial APIs, it trades dollars for speed.

Relevance to Agentic AI Systems

Systems-Level Design Principle

Speculative Actions reveals a fundamental design principle for modern agentic platforms: opportunistic parallelism in environment interactions. Rather than waiting for each API call to return before planning the next one, agents can speculatively pre-launch likely next actions, transforming idle waiting time into productive computation.

MCP Server Integration

Directly compatible with MCP servers: each server call is an API action that can be speculatively pre-launched. Aligns with the emerging MCP perspective on agentic systems (Anthropic, 2024).

Human-in-the-Loop Agents

Human responses are the slowest API calls. Speculating user intent (e.g., predicting a user asking about returns will need order details) can pre-fetch information before the user finishes typing.

Multi-Agent Orchestration

In orchestrator-worker architectures, the orchestrator can speculate which sub-agents to invoke next, pre-sending context and queries to likely candidates while waiting for the current worker.

Training Loop Acceleration

Agent training requires hundreds of environment interactions per episode. Speculative actions can accelerate training loops by 10-20%, compounding into significant speedups over thousands of episodes.

Future Vision

Speculative Actions points toward a future where speculation becomes a first-class mechanism in agentic system design — through multi-step speculation, hierarchical actor-speculator architectures, uncertainty-aware adaptive branching, and tighter integration with RL-based environment simulators. Together, these directions move toward general-purpose, real-time agentic systems where idle waiting is eliminated and efficient decision-making is a first-class primitive.

Safety and Correctness Patterns

Speculation introduces the risk of executing actions that the Actor would not have chosen. The framework addresses this through three layered safety mechanisms:

S

Semantic Guards

Before committing speculative results, actors confirm that the state transitions match expectations. In chess, this means verifying the predicted move matches the actual move. In e-commerce, confirming the predicted API call matches the ground truth.

E

Safety Envelopes

Only idempotent (read-only), reversible (undoable), or sandboxed (containerized) speculative side effects are permitted. Web searches, database reads, and sandboxed code execution are safe. Purchases, file deletions, and API writes require additional guards.

R

Repair Paths

When mispredictions occur with non-trivial side effects, rollback (chess: undo move), snapshot restoration (OS: restore prior config), or roll-forward repair (e-commerce: refund/replace) restores correct state.

When NOT to Speculate

Speculation should be restricted or disabled for: (1) actions with irreversible side effects (database deletions, financial transactions, file modifications), (2) actions with high monetary cost (expensive API calls where wrong guesses waste budget), and (3) actions in adversarial environments where speculative probes could leak information or trigger unintended behaviors. The framework's modularity allows per-action-type speculation policies.

Implementation Patterns

Speculator Design Strategies

Same-Model, Different Effort

Use the same model as the Actor but with lower reasoning effort (fewer thinking tokens). Used in chess: GPT-5 with speed-optimized prompt. Maximizes accuracy while minimizing latency.

Smaller Model

Use a smaller, cheaper model (gpt-5-nano for Actor's gpt-5). Faster but lower accuracy. Good when latency budget is tight and marginal accuracy improvement has diminishing returns.

Multi-Model Ensemble

Run multiple speculators in parallel (different models, reasoning budgets). Aggregate their predictions into a candidate set. Achieves highest accuracy (38.5% in e-commerce) at the cost of more API calls.

Domain Heuristics

For predictable environments, hard-coded rules (e.g., "if user mentions return, pre-fetch get_order_details") can be faster and cheaper than any LLM-based speculator.

History Compression for Multi-Agent OS Tuning

The OS tuning environment uses distinct prompt structures for the two agents: 

// Actor receives compressed history + all Speculator actions
// since its last deliberation
["Compressed history for calls 1-10"]
RESULT for call #11 [SPECULATOR]: min_granularity_ns=500000 -> p95=36.24ms
RESULT for call #12 [SPECULATOR]: min_granularity_ns=400000 -> p95=35.18ms
...
RESULT for call #19 [SPECULATOR]: min_granularity_ns=200000 -> p95=30.26ms

// Speculator receives compressed Actor history + its own recent verbose replies
["Same compressed history from last Actor cycle"]
["Full verbose Speculator responses for recent iterations"]
Latest: min_granularity_ns=200000 -> p95=30.26ms

This dual-context mechanism allows the Actor to analyze long-term trends from a compact summary while the Speculator retains immediate, detailed context for its rapid, reactive decisions.

Connections to Related Work

Speculative Decoding in LLMs

Speculative decoding (Leviathan et al., 2023) uses a small draft model to propose tokens verified by a larger model. Speculative Actions generalizes this from token-level to action-level, from inference to the entire agent-environment loop.

Speculative Execution in CPUs

Branch prediction in CPUs (Tomasulo, 1967) speculatively executes instructions before branch conditions are resolved. Speculative Actions applies the same predict-verify pattern to API calls, with rollback on misprediction.

Thread-Level Speculation

TLS (Estebanez et al., 2016) parallelizes sequential program fragments. Speculative Actions extends this to agentic programs where "threads" are API calls to external services, LLMs, and humans.

Speculative Tool Calling (LLNL)

The LLNL work is a special case focused on inference engine integration. Speculative Actions subsumes it by treating tool calls as one of many action types in a general MDP framework.
