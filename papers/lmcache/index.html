LMCache: An Efficient KV Cache Layer for Enterprise-Scale LLM Inference

Cross-Query Prefix Reuse & Prefill-Decode Disaggregation at Datacenter Scale

LMCache treats KV cache as a first-class data structure -- extracting it from GPU memory, storing it across a multi-tier hierarchy (GPU HBM, CPU DRAM, local disk, remote storage), and sharing it across queries and inference engines for up to 15x throughput improvement.

arXiv: 2510.09665 Year: 2025 Org: Tensormesh & UChicago License: Apache 2.0 Engines: vLLM + SGLang

Overview

LMCache is the first and most widely adopted production-ready KV caching layer for enterprise-scale LLM inference. It sits between LLM inference engines (vLLM, SGLang) and heterogeneous storage/network devices, providing a standardized, high-performance substrate for KV cache movement and management. LMCache enables two critical use cases that traditional GPU-only KV caching cannot support:

Cross-Query Prefix Reuse (Context Caching)

Persists KV cache segments from one query in lower-tier storage (CPU DRAM, disk, Redis) and reuses them for subsequent queries sharing a common prefix. Critical for document analysis, multi-turn QA, and RAG workloads where the same context appears across multiple queries.

Prefill-Decode (PD) Disaggregation

Enables splitting inference into a prefill phase and a decode phase on separate GPUs/nodes. LMCache efficiently transfers KV caches from prefill GPUs to decode GPUs via NVLink, RDMA, or TCP, ensuring the latency-sensitive decode phase is not interrupted by throughput-oriented prefilling.

First-Class KV Cache Management APIs

Exposes APIs for locating, pinning, evicting, moving, and compressing KV caches. These first-class management interfaces allow higher-level components (routers, schedulers, ML ops) to make KV-cache-aware decisions for optimal utilization.

Real-World Adoption

LMCache is integrated or referenced by vLLM, SGLang, NVIDIA Dynamo, llm-d (RedHat), AIBrix, KServe, and the vLLM Production Stack. Enterprise adopters include GMI Cloud (4x boost), Google Cloud, CoreWeave, and infrastructure partners Redis, Weka, and PliOps. Real-world usage data shows KV cache size has grown to exceed GPU memory capacity by 5.57x, with reuse per token increasing over time.

Multi-Tier Storage Architecture

LMCache organizes storage in a hierarchical tier system. KV caches flow between tiers based on access patterns, with configurable chunk sizes (default 256 tokens) that batch small GPU pages into larger transfer units to saturate bandwidth.

Modular KV Connector Design

The KV Connector interface (7 functions in Table 2 of the paper) decouples LMCache from the inference engine. It hooks into two places: the scheduler (for prefix-match token counting and scheduling decisions) and the model runner (for layer-wise KV cache loading/storing with compute overlap). This design has been adopted by NVIDIA Dynamo, llm-d, AIBrix, and the vLLM Production Stack.

Core Features Deep Dive

Pin API for Agent Sessions

A financial company working with LMCache requested the ability to explicitly pin frequently accessed financial documents in the caching system. The Pin API provides explicit retention control -- pinned KV caches are protected from eviction, enabling deterministic cache behavior for agent sessions. The controller exposes pin(tokens, instance, device) and unpin(tokens, instance, device) as external APIs.

pin(tokens, instance, device)

Pins the KV cache for the specified tokens at a given instance and storage device location. The PinMonitor tracks all pinned objects with registration timestamps. Configurable pin_timeout_sec (default 300s) prevents memory leaks by forcibly unpinning stale entries.

unpin(tokens, instance, device)

Explicitly releases a pinned KV cache, returning it to the normal LRU eviction pool. Applications call this when an agent session concludes or a document is no longer needed. The PinMonitor's background thread runs every pin_check_interval_sec (default 30s).

Batched Data Movement

Modern inference engines use paged memory with small pages (16-64 KB). Transferring these individually wastes bandwidth. LMCache batches multiple pages into configurable chunks (default 256 tokens) for bulk transfer.

LMCache achieves 400 Gbps CPU loading bandwidth

Compared to vLLM's native CPU offloading at 88 Gbps, LMCache achieves 4.5x higher effective bandwidth by transferring data chunk-by-chunk rather than page-by-page. Each chunk is gathered from scattered GPU pages into a contiguous streaming buffer via a custom CUDA kernel, then DMA-transferred collectively.

Minimum Data Copy & Zero-Copy Operations

Reference Counting

When KV cache is written to multiple destinations simultaneously (e.g., CPU DRAM + remote disk), LMCache increments a reference counter on the shared data rather than creating copies. Data is released when the count reaches zero. Analogous to PCB counters in classic OS design.

Dynamic Offloading

Uses a three-pointer system (start, current, end) to manage progressive GPU-to-CPU page duplication. Only a subset of free pages are duplicated at any time, balancing memory overhead against allocation stall probability. Supports 4 states from initialization through steady-state.

CacheBlend: Non-Prefix KV Reuse

CacheBlend (EuroSys'25 Best Paper) extends LMCache beyond prefix-only reuse. It enables reusing KV caches from arbitrary token sub-sequences at any position, not just shared prefixes. This is critical for RAG workloads where knowledge chunks appear at different positions across queries.

Selective Recomputation

CacheBlend does not blindly reuse all matched KV caches. It applies configurable blend_recompute_ratios and blend_thresholds per layer to decide which tokens need partial recomputation to maintain accuracy. The blend_check_layers config selects which layers to evaluate for quality.

Integrated into LMCache v1

CacheBlend is fully integrated: enable_blending: true in config activates the SegmentTokenDatabase and BlendServerV2 multi-process server. Uses blend_special_str (default " # # ") as segment separator. When enabled, save_unfull_chunk is forced to true for completeness.

Overall Assessment

The implementation not only matches all paper claims but significantly exceeds them in several dimensions. The codebase includes features not covered in the paper: storage plugin system for custom backends, lazy memory allocator with configurable expansion, NUMA-aware placement (NUMADetector / NUMAMapping), health monitoring, Prometheus/OpenTelemetry observability, Intel XPU and Habana HPU support, Bloom filter-based chunk statistics, and a full internal API server. The codebase is clearly production-grade and actively evolving (76KB cache_engine.py, 86KB memory_management.py, 77KB gpu_connectors.py).

Real-World Deployment Insights

Remote storage loads faster than prefill

Traditionally assumed slower, but with modern high-throughput storage (Amazon S3 Express at ~1 GBps), loading KV cache from remote storage can be faster than recomputing. Company C achieved 22-32% lower TTFT using their own remote object store with LMCache.

Context truncation halves hit rate

Many enterprises truncate long contexts to fit model windows. Using real traces from Company F, prefix cache hit rates dropped from 85% to 45% when truncating to keep only the latest tokens. This validates that dynamically removing context tokens destroys prefix KV cache reuse.

Unexpected 50% hit rate in production

Company G found a 50% prefix cache hit rate in production -- much higher than expected. Modern applications exhibit "dynamically reusable contexts" (conversation histories, coding assistants, RAG pipelines) that greatly increase reuse beyond fixed system prompts.

Flexibility trumps raw performance

The paper notes that "the flexibility to evolve faster and integration smoothness is more important than the language performance itself." Industry deprioritized customizing attention algorithms in favor of performance, stability, and compatibility -- driving Python + careful optimization over Rust rewrites.
