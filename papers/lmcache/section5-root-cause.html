5. Root Cause: Why LMCache Is Slower

Section §2 established that LMCache adds +2.73 s per job relative to FCFS under the benchmark conditions (H200, JPS=1, Exp C). This section traces that overhead to a single, quantifiable mechanism: excessive CUDA stream synchronization inside wait_for_save, which blocks the vLLM scheduler main loop on every prefill step.

5.1 The Cost: wait_for_save Blocks the Scheduler

vLLM's KV Connector API requires the scheduler to call wait_for_save before dispatching each step. NVTX profiling of 51 completed jobs (H200, JPS=1, Exp C) captured 1,769 invocations. The distribution is strongly bimodal: a small minority of prefill-step calls account for almost all wallclock time.

Call Type Count % of Calls Mean Duration Median p95 Source
Prefill-step calls 159 9% 115.71 ms 73.00 ms 253 ms H200, JPS=1, Exp C
Decode-only calls 1,610 91% 0.01 ms 0.01 ms 0.02 ms H200, JPS=1, Exp C
Total 1,769 100% Prefill-step calls are 11,571× slower on average
Key insight: Decode-only calls complete in ~0.01 ms — essentially zero cost. All measurable overhead is concentrated in the 9% of calls that coincide with a prefill step. The question is why prefill-step calls take ~115 ms.

5.2 Inside wait_for_save: cudaStreamSynchronize

cudaStreamSynchronize(stream) is a CUDA Runtime API call that blocks the calling CPU thread until every operation previously queued on the given stream has completed on the GPU. Internally the call performs a user-to-kernel transition (via ioctl to the NVIDIA kernel driver), checks stream completion status, sleeps if the GPU is still working (using futex or driver poll), and resumes on GPU interrupt. Even when the GPU finishes its DMA in 0.67 ms, the round-trip through the kernel — transition + interrupt latency + wakeup — adds ~3–4 ms of fixed overhead per call on PACE nodes.

CUDA Runtime attribution was measured over the 159 prefill-step wait_for_save windows using Nsight Systems (Exp C). The table below shows how the wallclock time is distributed across CUDA Runtime API calls within those windows.

Mechanism summary: LMCache issues one cudaMemcpyAsync per KV chunk and then immediately calls cudaStreamSynchronize to confirm completion before proceeding to the next chunk. With ~19 chunks per prefill step, this produces ~24 round-trip kernel-mode waits (some steps have additional overhead from metadata syncs). Each wait incurs ~4.74 ms of fixed overhead regardless of how fast the GPU DMA completes — causing the CPU thread to be idle ~85% of the time inside each sync call.

Upgrading PCIe bandwidth (Gen4 → Gen5) halves the DMA time but provides no relief against the per-call kernel overhead, which remains ~4.74 ms regardless.

5.4 Optimization Path

The fix is mechanical: replace the per-chunk synchronization pattern (issue async copy → synchronize → issue async copy → synchronize → …) with a coalesced pattern (issue all async copies → synchronize once). This reduces sync count from ~24 to 1 per prefill step while keeping the GPU busy continuously.

Current (per-chunk sync) ~24 115.71 ms — measured, H200, JPS=1, Exp C
Coalesced (1 sync per step) 1 5.41 ms −110.3 ms (−95%) 1 × 4.74 ms + 0.674 ms DMA

Why coalescing is safe

All KV chunks for a single prefill step target independent GPU memory regions. Issuing all cudaMemcpyAsync calls before any synchronization does not introduce data races; CUDA guarantees in-order execution within a stream. A single barrier at the end of the loop is semantically equivalent to N per-chunk barriers.

Scope of the fix

The relevant loop is located in LMCache's GPU-to-CPU offload path (triggered by wait_for_save during prefill). The change is local to approximately 5–10 lines of Python; no architectural change to the vLLM KV Connector API is required. The optimization applies equally to all four GPU families tested (H200, H100, A100, L40S).
