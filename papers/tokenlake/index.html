TokenLake: Unified Segment-level Prefix Cache Pool

Aggregates GPU memories into a unified segment-level prefix cache pool for fine-grained elastic long-context LLM serving. Up to 2.6x throughput and 2.0x hit rate improvement via declarative cache interfaces and heavy-hitter-aware load balancing.

Authors: Bingyang Wu, Zili Zhang, Yinmin Zhong, et al.
Affiliations: Peking University, StepFun
Built on: FlashInfer, SGLang, ZMQ, Ray

Overview

Prefix caching is crucial for accelerating multi-turn interactions and requests with shared prefixes. However, existing prefix caching systems are tightly coupled with request scheduling, leading to load imbalance, data redundancy, and memory fragmentation across instances. TokenLake proposes a declarative prefix cache interface that decouples cache management from scheduling, enabling a unified GPU memory pool.

Key Insight: Segment-level Granularity

Although locality is important for reducing communication volume, it is not strictly necessary for high performance. As long as the segment size exceeds a certain threshold (approximately 568 tokens in practice), the distributed query across remote instances can achieve better performance than local computation alone, because the aggregated memory bandwidth of multiple GPUs can be exploited for prefix attention.

P2P Asynchronous Architecture

TokenLake runs two independent engines per instance via NVIDIA Multi-Process Service (MPS): a Query Engine for computing prefix attention on remote segments, and a Transfer Engine for layer-wise KV cache migration. Both use CUDA IPC for zero-copy data sharing with the compute engine, and NCCL for inter-instance communication. Communication is overlapped with computation to minimize overhead.

Segment-based Load Balancing

Segment Size Tradeoff

The minimum segment size C for GPU pooling to be beneficial is determined by the condition that computation savings from distributed prefix attention exceed communication overhead. In the experimental setup with A100 GPUs, C ~ 568 tokens, enabling fine-grained segment-level pooling.

Heavy-Hitter-Aware Algorithm

Segments are classified as heavy hitters (frequently accessed) or normal segments. Normal segments are hash-distributed uniformly. Heavy hitters are selectively replicated to overloaded instances (up to O(N log N) replicas for N instances). This is based on the established result that absorbing traffic from the top O(N log N) heavy hitters guarantees balanced load with high probability.

Normal Segments

Hash-based distribution spreads them uniformly across all instances. A hash function maps each segment to an instance, achieving effective load balancing without migrations.

Heavy Hitters

Identified via BFS from the prefix tree root. Selectively replicated to the least-loaded instances. Uses the power-of-two-choices (PoT) policy for replica selection.

Eviction Policy

Global LRU policy based on last access time. For replicated segments, the query is directed to the less loaded of two randomly sampled replicas (balls-into-bins variant).

Dispatching Optimization

Bipartite Matching-Based Algorithm

Even with balanced load, the communication volume of query and put operations depends on dispatching. TokenLake converts the dispatching problem into a bipartite matching problem to minimize total communication. A bipartite graph G = (U, V, E) is constructed where U = batches with transfer plans, V = instances, and edge weights represent communication cost.

Stateless Elastic Scheduling

TokenLake enables goodput-optimized stateless scheduling by combining PD disaggregation, chunked prefill, and elastic sequence parallelism. The scheduler formulates a dynamic programming problem to minimize overall latency subject to SLO constraints on Time Between Tokens (TBT). The complexity is O(M_p^2 x N^2) where M_p is the number of prefill requests and N is the number of instances.

Effectiveness of Pooling and Elasticity

Ablation studies show that prefix cache pooling (TokenLake-1P3D and TokenLake-3P1D) reduces latency by 26.62-86.05x and 2.18-39.48x compared to SGLang-MoonCake variants. The full system with elastic scheduling (TokenLake-Elastic) further reduces latency by up to 5.22x and 2.82x versus fixed PD configurations.

Design Principles and Technical Details

Segment Size Analysis

The computation latency to process a segment of size C is bounded by either compute or memory bandwidth: Tcomp(C) = max(4Cd/F, 4Cd/Bmem) = kcomp · C, where F is FLOPS, Bmem is memory bandwidth, and d is hidden dimension. Communication time is Tcomm(C) = 2αnet + 4d/Bnet. Pooling is beneficial when Tcomp(C) exceeds Tcomm(C), yielding the threshold: C ≥ 2αnet/kcomp + 4d/(kcomp · Bnet).

Practical Threshold

With A100 80GB GPUs (NVLink 400 GB/s, 312 TFLOPS FP16), the threshold is approximately 568 tokens. At this segment size in the decoding phase with 1000 prefix tokens, the communication overhead is only 4.66 microseconds per layer -- negligible in practice.

Communication Volume Optimization

For query tensors, the communication volume per request is 2dlNp, where d is hidden dimension, l is sequence length, and Np is the number of instances hosting relevant segments. This is comparable to communication in Expert Parallelism (2dlNe where Ne is the number of experts), making TokenLake's overhead a well-accepted trade-off in practice.

Computation Overlap Strategy

TokenLake leverages the fully P2P asynchronous architecture to overlap all communication with computation. Specifically:

Layer-wise KV Put Overlap

After generating KV cache for layer i, the transfer engine begins sending it while the compute engine processes layer i+1. This hides the transfer latency entirely behind computation.

Query Scatter/Gather Overlap

Scattering query tensors to remote instances and gathering results overlaps with local self-attention computation, which is quadratic in sequence length (much larger than the linear communication).

Multi-Engine Isolation

The compute, query, and transfer engines run as independent CUDA MPS processes, triggered at different times to minimize network contention. Different destined instances also reduce conflicts.

Goodput-Optimized Scheduling Formulation

# Dynamic Programming for Optimal Scheduling
# f(i, k) = minimum overall latency for first i requests on k instances

def optimal_schedule(requests, n_instances, slo_constraint):
    # Sort by context length for similar batching
    requests = sorted(requests, key=lambda r: r.context_len)
    M = len(requests)
    N = n_instances

    # DP table: f[i][k] = min latency for first i reqs using k instances
    f = [[float('inf')] * (N + 1) for _ in range(M + 1)]
    f[0][0] = 0

    for i in range(1, M + 1):
        for k in range(1, N + 1):
            for j in range(0, i):
                for l in range(0, k):
                    batch_latency = estimate_T(j+1, i, k-l, cache_load=L)
                    if meets_slo(batch_latency, slo_constraint):
                        f[i][k] = min(f[i][k], f[j][l] + batch_latency)

    return min(f[M][k] for k in range(N + 1))
