Cold-RL

Learning Cache Eviction with Offline Reinforcement Learning for NGINX — the first production-grade RL eviction algorithm operating under microsecond SLOs

Paper Overview

Cold-RL replaces NGINX's default Least-Recently-Used (LRU) eviction with a learned policy that uses a Dueling Deep Q-Network. The system is architected for production: offline training on access logs, bounded inference via an ONNX sidecar process, and instantaneous fallback to native LRU when the 500 microsecond timeout fires.

Core Insight Eviction is fundamentally a prediction problem disguised as recency bookkeeping. By focusing on the K coldest LRU-tail objects (typically 8-32), Cold-RL achieves O(K) complexity while capturing the vast majority of eviction candidates.

System Architecture

Cold-RL embodies a philosophy of separation: training is separated from inference, inference from serving, and learning from failing. Three components cooperate to produce a safe, deployable system.

NGINX Module (C)

Hooks ngx_http_file_cache_forced_expire to intercept eviction before native LRU. Extracts features from K-tail candidates, serializes them into a 192-byte IPC request (for K=8), and communicates with the sidecar via a Unix domain socket. Every operation is designed for failure: null checks, timeout handlers, invalid response detection.

Inference Sidecar (C++/ONNX)

A standalone process that loads the trained ONNX model once at startup. Uses ONNX Runtime with single-threaded inference, graph-level optimization, and sequential execution mode for minimal latency. The entire inference path touches no syscalls, allocates no memory, and holds no locks.

The Dueling DQN Architecture

The architecture decomposes the Q-function into value and advantage streams. This is crucial because under high cache pressure, most eviction candidates are similarly poor -- the dueling architecture provides stability when A(s,a) values are close to zero.

Model Implementation (PyTorch)

class DuelingDQN(nn.Module):
    """Dueling DQN for cache eviction. ~10K params, fits in L2 cache."""
    def __init__(self, config: ModelConfig):
        super().__init__()
        self.input_dim = config.input_dim * config.max_k  # Flattened: K * 6
        # Shared feature extraction
        self.feature_extractor = nn.Sequential(
            nn.Linear(self.input_dim, 256), nn.LayerNorm(256), nn.ReLU(),
            nn.Linear(256, 256), nn.LayerNorm(256), nn.ReLU()
        )
        # Value stream: V(s) -> scalar
        self.value_stream = nn.Sequential(
            nn.Linear(256, 128), nn.ReLU(), nn.Linear(128, 1)
        )
        # Advantage stream: A(s,a) -> K values
        self.advantage_stream = nn.Sequential(
            nn.Linear(256, 128), nn.ReLU(), nn.Linear(128, config.max_k)
        )

    def forward(self, x, k_mask=None):
        features = self.feature_extractor(x)
        value = self.value_stream(features)          # [B, 1]
        advantage = self.advantage_stream(features)  # [B, K]
        if k_mask is not None:
            advantage = advantage.masked_fill(~k_mask, -1e8)
        # Dueling combination
        q_values = value + (advantage - advantage.mean(dim=1, keepdim=True))
        return q_values
⚡
Why ~10K Parameters? The entire model fits in L2 cache. Smaller models are faster, more robust, and less prone to overfitting. The architecture totals approximately 10K parameters -- this is not a limitation but a deliberate design choice for microsecond inference.

Reward Signal

The reward is deceptively simple: a retained object receives +1 if it is accessed again before its TTL expires, -penalty otherwise. Evicted objects receive the negative of what they would have earned.

# Reward structure
if action == KEEP:
    reward = +hit_reward * future_hits
             - miss_penalty * if no_future_hits
             - size_penalty * (size_mb * 0.05)
elif action == EVICT:
    reward = -hit_reward * future_hits
             + 0.1 * hit_reward * if no_future_hits

Prioritized Experience Replay

The replay buffer uses a Sum Tree data structure for O(log n) proportional sampling. Transitions with high TD-error are sampled more frequently, with importance sampling weights to correct the bias.

class SumTree:
    """Binary tree where each leaf stores a priority.
    Internal nodes store subtree sums. O(log n) sampling."""
    def __init__(self, capacity):
        self.tree = np.zeros(2 * capacity - 1)  # internal + leaves
        self.data = [None] * capacity

    def get_leaf(self, s):  # s in [0, total_priority]
        parent_idx = 0
        while True:
            left = 2 * parent_idx + 1
            if left >= len(self.tree): break
            if s <= self.tree[left]:
                parent_idx = left
            else:
                s -= self.tree[left]; parent_idx = left + 1
        return parent_idx, self.tree[parent_idx], self.data[...]

ONNX Inference Sidecar

The sidecar is a C++ process that loads the ONNX model, listens on a Unix domain socket, and runs inference for each eviction request. The design is optimized for the absolute minimum latency.

// ONNX Runtime configuration for minimum latency
Ort::SessionOptions session_options;
session_options.SetIntraOpNumThreads(1);        // single-threaded
session_options.SetGraphOptimizationLevel(ORT_ENABLE_ALL);
session_options.SetExecutionMode(ORT_SEQUENTIAL);

// Inference: convert Q-values to eviction bitmask
bool DRLCacheModel::predict(const float* features, uint16_t k, uint32_t& mask) {
    // Pad input to MAX_K * 6, run ONNX session
    auto output_tensors = session_->Run(...);
    const float* q_values = output_tensors[0].GetTensorData<float>();
    // Positive Q-value = evict this candidate
    mask = 0;
    for (uint16_t i = 0; i < k; ++i)
        if (q_values[i] > 0.0f) mask |= (1U << i);
}

Hot Model Swap

The sidecar supports atomic model replacement at runtime. A new ONNX file is loaded into a separate Ort::Session, then swapped in using std::unique_ptr::swap under a mutex. Zero downtime, zero stale predictions.

Production Safety Mechanisms

Five layers of defense ensure Cold-RL never degrades service quality below what native LRU would provide.

1. Hard 500 us Timeout

If the sidecar does not respond within 500 microseconds, the NGINX module immediately falls back to native LRU eviction. No retries, no waiting.

2. Shadow Mode

Before production deployment, run the model in shadow mode where it logs what it would have evicted without actually applying the decision. Compare against LRU for validation.

3. Circuit Breaker

After N consecutive inference failures, the circuit breaker opens and all evictions route directly to LRU until the sidecar recovers.

4. Gradual Rollout

Percentage-based rollout: start with 1% of evictions using the RL policy, gradually increase while monitoring hit ratio and latency metrics.

5. Kill Switch

A single-flag configuration variable instantly disables the RL module and reverts all evictions to native LRU. No restart required.

Key Design Decisions

  • Offline training eliminates distribution shift risks inherent in online RL
  • Dueling architecture provides stability when advantage values are similar
  • K-tail sampling achieves O(K) complexity vs O(n) for full cache scan
  • Cosine LR schedule with warmup for stable convergence
  • Soft target updates (tau=0.005) for smoother Q-value targets
  • Prioritized replay with importance sampling correction

Deployment and Operational Considerations

Model Export Pipeline

The trained PyTorch model is exported to ONNX format with INT8 quantization for deployment. The export process includes validation to ensure numerical equivalence between the PyTorch and ONNX models within a tolerance of 1e-4.

def export_to_onnx(self, output_path):
    self.model.eval()
    dummy_input = torch.randn(1, self.config.model.max_k * self.config.model.input_dim)

    torch.onnx.export(
        self.model, dummy_input, output_path,
        export_params=True,
        opset_version=11,
        do_constant_folding=True,
        input_names=['input'],
        output_names=['output'],
        dynamic_axes={'input': {0: 'batch_size'},
                      'output': {0: 'batch_size'}}
    )
    # Validate: max difference must be < 1e-4
    self._validate_onnx_model(output_path, dummy_input)

Data Pipeline Architecture

The training pipeline transforms raw NGINX access logs into training trajectories through a multi-stage process. Each stage is independently cacheable and reproducible.

Log Parsing

Parses NGINX combined log format with cache status fields. Extracts cache_status (HIT/MISS/BYPASS/EXPIRED), cache_key, size_bytes, ttl_seconds, and upstream_time. Supports parallel chunk processing for large log files.

Cache Simulation

High-fidelity simulator that replays log events, maintaining an accurate cache state. Uses a warmup ratio of 10% to allow the simulated cache to reach steady state before generating training data.

Trajectory Generation

When eviction occurs during simulation, the K coldest LRU-tail candidates are extracted with their 6-feature vectors. Ground-truth actions come from LRU behavior; rewards are computed by looking at future accesses within TTL windows.

Dataset Splitting

Temporal splitting into 70/15/15 train/val/test sets. Temporal (not random) to prevent data leakage from future cache states into training data.

Broader Impact and Future Directions

Applicability Beyond Caching

The Cold-RL architecture -- offline training, bounded inference, instant fallback -- generalizes to any system requiring microsecond decisions with long-term consequences. Potential applications include:

  • Packet scheduling: Which packet to drop under congestion
  • Memory allocation: Which page to swap to disk
  • Database buffer management: Which block to evict from buffer pool
  • CDN routing: Which origin server to query
  • Load balancing: Which backend to route requests to
  • Auto-scaling triggers: When to provision new resources

Known Limitations

⚠
Cold Start Requires 24-48 hours of logs before the first model can be trained. During this period, native LRU operates alone.
⚠
Workload Drift Sudden workload shifts (new content going viral) need retraining cycles. The model cannot adapt in real-time.
⚠
K-tail Scope Decisions are limited to K candidates from the LRU tail. Optimal victims elsewhere in the cache are not considered.
