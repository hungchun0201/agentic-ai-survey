Agent.xpu

Efficient Scheduling of Agentic LLM Workloads on Heterogeneous SoC

arXiv:2506.24045 January 2026 Peking University & University of Hong Kong On-Device NPU+iGPU Hetero-SoC

Overview

Agent.xpu is the first LLM engine designed to orchestrate concurrent reactive and proactive agentic LLM flows on commodity heterogeneous SoCs. It introduces three key techniques: a heterogeneous execution graph (HEG), flow-aware NPU-iGPU coordination with stage elasticity, and fine-grained preemption with slack-aware piggybacking.

4.6x Lower Reactive Latency

Agent.xpu reduces reactive query latency by 91-97% compared to OpenVINO (iGPU) serving baseline, with even larger gains over other baselines, enabled by kernel-level preemption at sub-100ms granularity.

1.2-4.9x Proactive Throughput

For proactive-only workloads, Agent.xpu delivers 1.2-2.4x throughput over OpenVINO (iGPU) and 1.4-4.9x over serial NPU-iGPU inference, through flow-aware NPU-iGPU co-scheduling.

26.8% Less Energy

Reduces energy consumption by 26.8% and iGPU utilization by 32.5% compared to iGPU-only serving and NPU-iGPU static inference baselines by offloading prefill to NPU.

Three Fundamental Gaps in Current Ecosystems

Flexibility-Efficiency Trade-off

NPUs deliver high performance for static-shaped input via AOT compilation but are ill-suited for variable sequence lengths and dynamic batch sizes inherent to agentic LLM workloads. iGPUs are more flexible but suffer from lower energy efficiency and graphics interference.

Shared-Memory Contention

Concurrent NPU and iGPU kernel executions intensify contention for the limited DDR bandwidth. Memory-bound GEMV kernels are significantly more sensitive to bandwidth saturation than compute-bound GEMM, with up to 37.3% execution time increase during simultaneous co-execution.

Absence of Flow-Aware Abstractions

Most on-device LLM engines lack mechanisms such as in-flight batching, prioritized scheduling, and coordinated NPU-iGPU control, which are essential for launching concurrent reactive and proactive flows while meeting their distinct latency or throughput requirements.

Agent.xpu's design is grounded in comprehensive profiling of operator-level and task-level characteristics on the Intel Core Ultra processor, revealing opportunities distinct from cloud-centric LLM serving assumptions.

Key Profiling Finding: Stage-Divergent Batching

Prefill latency scales nearly linearly with batch size, saturating both NPU and iGPU without performance gain. Decode latency rises gradually with larger batches or longer prompts, since each task's MHA operates on its own KV cache with unchanged arithmetic intensity. This motivates disaggregating prefill and decode across NPU and iGPU.

Key Online Components

Request Manager

Interfaces with the agent frontend. Maintains a global request table recording UUID, lifecycle, KV cache allocation, prompt tokens, inference progress (phase, layer, current kernel, generated tokens). Enables task checkpointing and resuming without recomputation.

Dual Task Queues

Maintains separate real-time (reactive) and best-effort (proactive) queues for both prefill and decode. Enables efficient scheduling with immediate preemption support for reactive requests.

XPU Coordinator

Orchestrates concurrent execution across NPU, iGPU, and CPU. Operates on NPU/iGPU FIFO queues, binding elastic kernels to accelerators based on HEG annotations, task priority, and current load.

Double Activation Buffer

Maintains a reactive-proactive double buffer with single-layer activation buffers reused recurrently. Enables copy-free context switching for preemption -- shared memory eliminates KV cache swapping.

Five Coordination Mechanisms

1. Prefill Pipeline

Reactive requests partition token-wise ops across NPU and iGPU with elastic tensor parallelism. Proactive requests run mainly on NPU with residual fragments on iGPU. Prefill is compute-bound and runs singly per request.

2. Decode Pipeline

Continuous batching decoupled from prefill, scheduling at iteration granularity. Reactive and proactive requests are co-batched under adaptive strategies. All decode kernels run on iGPU to accommodate dynamic sequence lengths.

3. Pipeline Coordination

Prefill-first arbitration policy: iGPU kernels from prefill always take precedence over decode, since decode is memory-bound and generally longer. This avoids long stalls that would inflate TTFT.

4. Elastic Tensor Parallelism

Layer-wise NPU-iGPU partitioning at runtime. The coordinator solves for n_npu chunks such that NPU execution time matches iGPU's, minimizing the makespan. When decode is busy, all chunks go to NPU.

5. On-the-Fly NPU Warm-Up

CPU starts compiling NPU kernels immediately when a request is queued. If compilation completes before prefill begins, the request uses pure-NPU prefill, eliminating iGPU interference with decode.

Fine-Grained Preemption

Agent.xpu exploits the unified memory architecture and its double activation buffer design to achieve lightweight, copy-free context switching for reactive requests, with sub-100ms preemption latency.

1. Copy-Free Context Switching
Inference context (KV cache, progress metadata, generated tokens) lives in shared memory. The double activation buffer separates reactive and proactive activations, enabling instant preemption and restoration without data copying. No KV cache swapping needed.
2. Kernel-Level Prefill Preemption
Wait-then-execute preemption at kernel boundaries. Each kernel registers a lightweight completion callback that probes the reactive queue. When non-empty, it signals the prefill event loop to immediately dequeue the reactive job, preempting the proactive task.
3. Iteration-Level Decode Preemption
Reactive requests wait until the current decode iteration completes, then immediately join the next batch. Proactive jobs may be evicted under adaptive batching policies. Preemption is bounded by TPOT (typically less than 100ms for 8B models on SoC).
4. Slack-Aware Piggybacking
Proactive work opportunistically fills pipeline slack. Compute slack: proactive prefill overlaps decode when no reactive prefill is pending. Bandwidth slack: proactive requests are co-batched with reactive ones under elaborated rules bounding additional latency.

Reactive-First Adaptive Decode Batching

// Algorithm 2: Reactive-First Adaptive Decode Batching
Input:  B_dec (current batch), Q_react, Q_proact, B_cap (threshold=3)

// Step 1: Unconditionally admit ALL reactive requests
R_react = {r in B_dec | r.type == REACTIVE} union PopAll(Q_react)
B_dec = R_react            // reactive-first: always included

// Step 2: Sort proactive by sequence length (descending)
R_proact = {r in B_dec | r.type == PROACTIVE} union PopAll(Q_proact)
SortDescending(R_proact, key=S_len(r))

// Step 3: Piggyback proactive up to capacity
for r in R_proact:
    if |B_dec| < B_cap or R_react == empty:
        B_dec = B_dec union {r}
    else:
        PushBackToQueue(r, Q_proact)  // Evict to prevent TPOT inflation

Overhead Analysis

iGPU Utilization

Agent.xpu reduces overall iGPU utilization by 32.5% compared to Serial (NPU-iGPU) and 37.1% compared to OpenVINO (iGPU). Pure-iGPU prefill strikes 100% utilization; Agent.xpu's NPU offloading reduces iGPU occupation, freeing resources for graphics and other tasks.

Energy Efficiency

NPU power remains stable around 10W for chunked prompt length. iGPU power grows from 25W (decode) to 31W (prefill). Agent.xpu achieves 26.8% lower energy consumption than OpenVINO by offloading prefill to NPU and employing efficient batching. Per-token: 0.30 J/token vs 0.41 J/token (OpenVINO).

Analytical Kernel Modeling

Agent.xpu uses a roofline-inspired analytical model to predict kernel latency and bandwidth utilization as a function of sequence length or batch size, enabling adaptive kernel dispatching at runtime. 

// Roofline-based latency prediction for hetero-SoC accelerator kernels

// Given: W (FLOPs), Q (data bytes), I = W/Q (arithmetic intensity)
//        P_pk (peak compute), B_pk (peak bandwidth)

// Classical roofline performance bound
P(I) = min(P_pk, B_pk * I)

// Calibrated with two anchor measurements:
//   u_m = memory-bound utilization (measured)
//   u_c = compute-bound utilization (measured)
B_eff = u_m * B_pk                     // effective bandwidth ceiling
P_eff = min(P_pk, u_c * B_pk * I_c)   // effective compute ceiling

// Latency prediction (piecewise)
T(I) = {
    Q / (u_m * B_pk)   if I <= I_star   // memory-bound
    W / P_eff          if I >= I_star   // compute-bound
}

// Bandwidth utilization prediction
u(I) = min(u_m, u_c * I_c / I)
Why This Matters for Scheduling

The closed-form latency and bandwidth predictions require only two measured utilizations (u_m, u_c) and the problem size. This enables the scheduler to estimate TTFT for individual tasks and TPOT for batched decode, guiding adaptive kernel dispatching decisions between NPU and iGPU based on request priority, accelerator load, and bandwidth pressure.

Key Takeaways

1. Stage Decoupling is Essential

Disaggregating prefill (NPU) from decode (iGPU) is the single most impactful design choice. It mitigates DDR bandwidth contention, enables independent scheduling, and preserves both reactive responsiveness and proactive throughput simultaneously.

2. Shared Memory Enables Cheap Preemption

The unified memory architecture of hetero-SoCs, combined with a double activation buffer, enables copy-free context switching. This makes sub-100ms preemption practical without the overhead of KV cache swapping required in cloud GPU settings.

3. Reactive-First with Piggybacking

Prioritizing reactive requests unconditionally while opportunistically piggybacking proactive work into pipeline slack achieves the best of both worlds: user-facing responsiveness with background throughput. Starvation prevention via aging ensures proactive tasks complete.

4. Profiling-Driven Elastic Kernels

The HEG abstraction with profiling-guided annotations enables late binding of operators to accelerators. This adaptive approach handles the dynamic, unpredictable nature of agentic workloads far better than static accelerator assignment.
