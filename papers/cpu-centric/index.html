Q1: 延遲在哪裡？

Agentic AI 為何需要 CPU 視角？

傳統 LLM serving 主要關注 GPU 推理效能。但 Agentic AI 系統在 LLM 推理之外，還有大量 CPU-centric 的工作：

  • Web 搜尋 / URL 抓取 — I/O-bound，受網路延遲影響
  • 文件檢索（FAISS / embedding） — CPU-bound，受 SIMD / 記憶體頻寬影響
  • 程式碼執行（bash / pytest） — CPU-bound，受子程序排程影響
  • 數學計算（AST eval / Wolfram） — CPU-bound + I/O-bound（外部 API）
  • 框架開銷（LangGraph / Haystack） — CPU-bound，Python 編排開銷
關鍵發現：在許多 agentic workloads 中，非 LLM 部分（工具執行 + 框架開銷）佔總延遲的 30%–70%， 因此僅優化 GPU 推理是不夠的。

3. LangChain — Web Search Orchestrator

LangGraph 4-Stage Pipeline Batch Processing

LangChain 工作負載使用 LangGraph 實作四階段 Web 搜尋管線： 先搜尋網頁、抓取內容、提取摘要、最後交給 LLM 生成答案。

3.1 四階段管線

1
Web Search
Google Custom Search API 搜尋 10 個 URL
2
URL Fetch
並行下載網頁內容，每 query 取 2 頁
3
Summarize
LexRank 提取式摘要，並行處理
4
LLM Answer
vLLM batch inference 生成最終答案

3.2 LangGraph 狀態圖

# langchain/orchestrator.py
from langgraph.graph import StateGraph

# 定義 4-node 的 DAG
builder = StateGraph(GraphState)
builder.add_node('web_search', web_search)     # Google API 搜尋
builder.add_node('fetch_url', fetch_url)       # URL 並行抓取
builder.add_node('summarize', summarize)       # LexRank 摘要
builder.add_node('final_answer', final_answer) # LLM 答案生成

# 線性連接：web_search → fetch_url → summarize → final_answer
builder.add_edge(START, 'web_search')
builder.add_edge('web_search', 'fetch_url')
builder.add_edge('fetch_url', 'summarize')
builder.add_edge('summarize', 'final_answer')
builder.add_edge('final_answer', END)

compiled_graph = builder.compile()

4. Haystack — Large-Scale RAG System

FAISS 10M+ Documents ONNX Runtime Disk-Based

Haystack 工作負載實作大規模 RAG（Retrieval-Augmented Generation）， 處理超過 1000 萬篇文件的精確向量搜尋，使用基於磁碟的存儲方案以突破 RAM 限制。

1
Embedding
ONNX / Torch Sentence Transformer
2
FAISS Search
IndexFlatIP 精確最近鄰搜尋
3
Doc Retrieve
SQLite + JSONL Shard 文件檢索
4
LLM Generate
Haystack PromptBuilder + vLLM

4.2 核心類別

類別檔案職責
STEmbedder retrieval.py Sentence Transformer 封裝，支援 ONNX / Torch / OpenVINO
ExactFaiss faisss_optimized_disk.py FAISS IndexFlatIP + IDMap2 精確搜尋

ShardCache faisss_optimized_disk.py LRU cache，管理已開啟的 shard file handles
LargeScaleRetriever retrieval.py 主流程編排：embedding → search → doc fetch
HaystackRAGGenerator retrieval.py Haystack 元件：PromptBuilder + OpenAIGenerator

4.4 關鍵優化技術

Memory-Mapped I/O

使用 mmap 讀取 JSONL shard，避免將整個檔案載入記憶體。每次只映射需要的 byte range。

ONNX Runtime 推理

Embedding model 使用 ONNX Runtime 在 CPU 上推理，比 PyTorch 快 2-3x。支援 OpenMP 並行。

LRU Shard Cache

ShardCache 管理已開啟的 shard file handles，避免重複 open/close。預設保持 24 個 handle。

並行文件檢索

ThreadPoolExecutor 並行從多個 shard 讀取文件，doc_workers 控制並發數。

4.5 CPU 效能調優

# 依 CPU 核心數調優參數

# 64+ 核心（伺服器等級）
python haystack/retrieval.py \
    --omp-threads 64 --ort-intra 8 --ort-inter 1 --doc-workers 8

# 16-32 核心（工作站）
python haystack/retrieval.py \
    --omp-threads 16 --ort-intra 4 --ort-inter 1 --doc-workers 4

# 4-8 核心（筆電）
python haystack/retrieval.py \
    --omp-threads 4 --ort-intra 2 --ort-inter 1 --doc-workers 2
參數說明： omp-threads 控制 FAISS 的 OpenMP 並行度； ort-intra 控制 ONNX Runtime 的 intra-op 並行（單一算子內部）； ort-inter 控制 inter-op 並行（算子之間）； doc-workers 控制文件檢索的 thread 數。

4.6 索引建置（Indexing）

# 1. 下載 C4 語料庫（~305GB English variant）
python haystack/download.py

# 2. 建立 FAISS 索引 + Document Store（需 4-6 天）
python haystack/indexing.py \
    --data-root /path/to/c4 \
    --store-dir ./rag_flat_store \
    --shard-size 100000
注意：完整索引建置需要 305GB 磁碟空間（C4 語料庫）+ 64GB+ RAM + 4-6 天時間。 如果只是做 benchmark 測試，可以使用已建好的索引。

            except Exception:
                if attempt == 2: raise

Toolformer 工作負載重現 Toolformer 論文的概念：LLM 學會在生成文字的過程中 插入工具呼叫（Calculator），然後將工具結果嵌回文本繼續生成。

6.2 兩種 Calculator 實作

WolframCalculator（外部 API）

延遲：100ms–2s（受網路影響）

CalculatorTool（本地 AST）

6.4 AST 安全評估

class CalculatorTool:
    def evaluate(self, expression: str) -> float:
        # 清理表達式
        expr = expression.strip()
        expr = expr.replace("^", "**")      # ^ → **
        expr = expr.replace("$", "")         # 移除貨幣符號

        # AST 安全解析（只允許數學運算，禁止任意程式碼）
        tree = ast.parse(expr, mode='eval')
        # 驗證所有節點都是允許的運算...
        return eval(compile(tree, '<calc>', 'eval'))

10.3 能耗計算

# energy/calculate.py
def calculate_energy(power_samples, interval_s=0.1):
    """
    Energy (Joules) = Σ Power(Watts) × Δt(seconds)
    """
    total_energy = sum(p * interval_s for p in power_samples)
    return total_energy  # 單位：焦耳
RAPL 限制：pyRAPL 只支援 Intel CPU。AMD 系統需要使用其他工具（如 perf 或 likwid）。 另外，RAPL 計數器需要 root 權限或適當的 Linux capability。

14.2 並行策略選擇

工作類型建議策略原因
I/O-bound（Web Search、URL Fetch） ThreadPoolExecutor GIL 不影響 I/O 等待，threading overhead 低
CPU-bound（LexRank 摘要） ProcessPoolExecutor 繞過 GIL，真正的 CPU 並行
GPU-bound（LLM 推理） vLLM batch API vLLM 內部管理 GPU batch scheduling
I/O + CPU（文件檢索） ThreadPoolExecutor + mmap 磁碟 I/O 是瓶頸，mmap 利用 OS cache

14.3 錯誤處理策略

Graceful Degradation

URL 抓取失敗時跳過該 URL 繼續處理其他結果；不會因單一失敗而中止整個 batch。

Timeout Protection

Mini-SWE-Agent 的 bash 命令有 30 秒 timeout；VLLMModel 有 3 次重試邏輯。

Incremental Saving

長時間 benchmark 中每完成一題就存檔，避免因中途失敗而丟失所有結果。
