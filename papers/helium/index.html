Helium: Efficient LLM Serving for Agentic Workflows

arXiv:2603.16104

A Data Systems Perspective -- Modeling agentic workflows as database query plans with LLMs as first-class operators. Template-aware radix-tree scheduling for proactive caching and cache-aware optimization.

Noppanat Wadlom, Junyi Shen, Yao Lu

1 Overview

Agentic workflows are composed of sequences of interdependent LLM calls that have become a dominant workload in modern AI systems. These workflows exhibit extensive redundancy from overlapping prompts and intermediate results due to speculative and parallel exploration. Existing LLM serving systems like vLLM focus on optimizing individual inference calls, overlooking cross-call dependencies and leading to significant inefficiencies.

Core Insight: An agentic workflow resembles a batch analytics pipeline where LLM calls function as operators. By modeling workflows as query plans and applying classical database query-optimization principles, Helium achieves holistic end-to-end optimization across workflows.
Workflow-Aware Serving

Models agentic workflows as query plans with LLM invocations as first-class operators, enabling proactive caching and global scheduling.

Proactive KV Caching

Pre-warms KV caches for static prompt prefixes identified during compilation. Maintains a global prompt cache to bypass redundant operators.

Cache-Aware Scheduling

Cost-based scheduling that leverages a templated radix tree to capture prompt structure and maximize prefix cache reuse across batches.

Up to 1.56x Speedup

Substantial speedups over state-of-the-art systems across diverse workload patterns while preserving exact semantics.

2 Background & Motivation

Three key disparities differentiate LLM workflows from traditional SQL pipelines, making existing per-call optimization strategies insufficient:

Three Disparities: Traditional SQL vs. Agentic Workflows

Representative Workflow Patterns

W1: Map-Reduce

Multiple expert agents process inputs concurrently, followed by a summarizer that aggregates outputs.

W2: Multi-Agent Debate

Multiple agents debate over a given context and question before arriving at a final answer through rounds.

W3: Reflection

An expert drafts an answer, followed by multiple critic agents that provide feedback to refine the output.

W4: Iterative Refinement

Documents are divided into chunks; a summarizer iteratively refines the summary over each chunk.

W5: Parallel Chains

Expert agents with distinct roles extract insights from separate input chunks, aggregated by a writer agent.

3 System Architecture

Helium adopts a classic multi-stage query processing architecture, dividing execution into parsing, optimization, and processing phases. Each agentic workflow is represented using a procedural language that specifies individual LLM or agent calls and their dependencies.

Helium System Architecture

4 Query Optimizer Design

The query optimizer identifies sharing opportunities at multiple granularities, from entire sub-workflows to prompt prefixes, and constructs the logical plan for execution. The optimization process comprises two stages.

Stage 1: Initial Plan Pruning

Operator Pruning

Backward traversal from output nodes removes operators that do not contribute to the final output. Efficiently removes speculative branches that are not rendered.

Common Subgraph Elimination (CSE)

Identifies and merges structurally identical subgraphs sharing the same inputs. Subgraphs are hashed based on topology and input node identifiers to detect duplicates efficiently.

Stage 2: Cache-Aware Logical Plan Optimization

Helium leverages a global prompt cache to bypass redundant computation at the operator level. The optimizer performs a recursive, bottom-up traversal of the DAG:

Cache Substitution: For each operator, a signature is computed from its type and materialized inputs. On a cache hit, the LLM operator is replaced with a lightweight CacheFetch operator that stores a pointer to the cached result. This converts a computational dependency into a simple data retrieval. 
// Pseudo-code: Cache-Aware Plan Optimization
for each operator in bottom_up_traversal(DAG):
    signature = compute_signature(op.type, op.materialized_inputs)
    if prompt_cache.contains(signature):
        replace(operator, CacheFetch(cached_result_ptr))
    else:
        prompt_cache.insert(signature, operator)

5 Query Processor Design

Templated Radix Tree (TRT)

The TRT is a novel data structure that represents the prefix structure of both static prompt components and dynamic parts derived from other operators' outputs. It captures dependencies among its leaf nodes.

Workflow DAG to Templated Radix Tree

Proactive Cache Management

During the first execution of a workflow, the query processor precomputes and stores the KV cache for static token sequences in GPU memory. In subsequent batches, the LLM engines can directly use these precomputed KV tensors, avoiding redundant prefill computations.

Scheduling Problem Formulation

The scheduling task is modeled as a multi-worker scheduling problem to find an assignment of calls to workers and an execution order that optimizes the makespan. The cost model uses token usage and precedence delays:

// Token usage decomposition
u(i,j) = alpha_i * len_out(l_j) * u_p(i,j) + u_d(i,j)

// Prefill usage: new tokens not shared with previous call
u_p(i,j) = sum(omega(v)) for v in LCApath(l_{j-1}, l_j)

// Objective: Minimize total token step (makespan)
T(sigma) = max_{i,j} c(i,j)
subject to precedence delays and dependency constraints

6 Cache-Aware Scheduling Algorithm

The scheduling problem is NP-hard (reducible from parallel machine makespan minimization). Helium proposes a cost-based, cache-aware greedy algorithm inspired by SGLang's DFS scheduling but operating at the operator-level TRT.

function Schedule(workflow_dag):
    partitioned_dag = PartitionWorkflowDAG(workflow_dag)
    tree = BuildSchedulingTree(partitioned_dag)
    while not Recurse(tree, tree.root, false) do
        Recurse(tree, tree.root, true)  // Force progress if stuck
    return tree.GetFinalSchedule()

function Recurse(tree, node, force):
    if node.is_leaf:
        if tree.CanSchedule(node, force): tree.ScheduleNode(node)
    else:
        for each child in SelectChildren(node, force):
            if Recurse(tree, child, force): node.RemoveChild(child)
        tree.UpdateState(node.children)
        force = false  // Only force first child
    return node.IsEmpty()
Key Insight: The nested sequence structure is crucial for maximizing cache reuse. Workers process LLM calls query-by-query within an inner sequence (to reuse per-query prefix) and operator-by-operator across inner sequences (to reuse static prefix). Time complexity is O(|V_int| * c_max^3 + |E'| * d_max).

7 Agentic Workflows DSL

Helium implements a Python-based DSL that constructs a symbolic DAG of primitive operators under a lazy dataflow model. Operator calls record nodes and edges instead of executing.

from helium import graphs, helium, ops

# Define the agentic workflow
q = ops.placeholder("q")
answer = ops.llm([ops.Msg("user", q)])
revise_prompt = ops.fmt("Revise answer...", q, answer)
fin_answer = ops.llm([ops.Msg("user", revise_prompt)])

# Build and compile the DAG
graph = graphs.build([fin_answer]).compile(q=["How many inches is 1 meter?"])

# Execute the DAG with Helium runtime
print(helium.invoke(graph))

Multi-GPU Scaling

With two H100 NVL GPUs, Helium assigns disjoint operator partitions to each worker, reducing per-worker queue depth. The query optimizer emits a single optimized DAG; the query processor's cost-based assignment distributes leaf operators across workers while respecting data dependencies. Inter-GPU KV cache sharing is not implemented in the current prototype — cached tensors reside on the GPU that computed them, so the scheduler prefers to keep same-prefix calls on the same worker.

12 Related Work

Helium draws on, and departs from, several lines of prior work spanning LLM serving infrastructure, agentic orchestration, and classical database systems.

LLM Serving Infrastructure

vLLM & Continuous Batching

vLLM introduced PagedAttention for memory-efficient KV cache management and continuous batching for high GPU utilization. Helium builds directly on vLLM v0.16.0 as its execution backend but adds a workflow-aware optimization layer above it, treating vLLM as a stateful LLM operator rather than the entire serving stack.

SGLang & RadixAttention

SGLang's RadixAttention uses a radix tree over request-level KV cache blocks to maximize prefix reuse. Helium's TRT generalizes this idea to the operator-template level: rather than per-request trees, it tracks structural prompt templates across a batch, enabling proactive pre-warming rather than reactive caching. At 16 branches, Helium's TRT uses 552 KiB vs SGLang's 14.8 MiB.

Parrot & Semantic Variables

Parrot introduces Semantic Variables to expose LLM I/O dependencies across requests and enables dataflow-based scheduling at the request level. It operates as a proxy layer over unmodified LLM backends. Helium differs by modeling full workflow DAGs, applying compiler-style plan pruning and CSE, and integrating a cost-based greedy scheduler that reasons about multi-worker makespan.

KVFlow

KVFlow uses Agent Step Graphs to model per-agent KV cache lifecycles and performs hierarchical prefetching to hide cache miss latency. It is the closest prior work to Helium but focuses on single-agent step sequences rather than multi-operator DAGs. Helium subsumes KVFlow's proactive prefetching via its Query Processor while adding global cross-workflow sharing through the prompt cache and CSE.

Agentic Orchestration Frameworks

LangGraph

LangGraph provides a graph-based orchestration framework with a Runnable interface and async batch execution. It achieves 1.28x overhead vs Helium on average because it launches concurrent requests without topology-aware ordering, leaving prefix cache reuse to chance rather than design.

AgentScope

AgentScope adopts an actor-based distributed model with decentralized agent servers. The actor-per-agent overhead introduces significant coordination cost (2.10x latency vs Helium), particularly for fine-grained workflows where message-passing latency dominates over LLM inference time.

Database Query Optimization Connections

Helium's design is explicitly inspired by classical database query processing. The correspondence is intentional: the Query Optimizer maps to the logical planner in a database engine, the TRT mirrors the role of access path selection, and the cost-based scheduling algorithm parallels the Selinger optimizer's cost estimation. Key differences include the non-deterministic output of LLM operators (making statistics harder to estimate) and the GPU memory constraint that ties caching to execution ordering in a way SQL KV store queries typically are not.

13 Mechanism Deep Dive

TRT Construction Algorithm

Building the Templated Radix Tree from a compiled workflow DAG proceeds in three passes. The first pass performs a topological sort of DAG nodes to determine static vs. dynamic token segments for each LLM operator's prompt. The second pass identifies shared static prefixes across operators using a trie-merge procedure. The third pass inserts placeholder nodes for dynamic segments and records inter-leaf dependency edges derived from DAG data edges.

// Pass 1: Classify token segments per operator
for each op in topo_sort(DAG):
    segments = []
    for each part in op.prompt_template:
        if part.is_literal:
            segments.append(StaticNode(part.tokens))
        else:
            segments.append(PlaceholderNode(part.source_op))

// Pass 2: Merge shared static prefixes into TRT
trt = RadixTree()
for each op in operators:
    trt.insert(op.static_prefix_tokens, leaf=op)

// Pass 3: Record placeholder+dependency structure
for each leaf_op in trt.leaves:
    for each dep in leaf_op.data_dependencies:
        trt.add_dependency_edge(dep.source_leaf, leaf_op)

Cost Model Details

The scheduling cost model estimates the token-step cost of executing call j on worker i after call j-1. The model decomposes total usage into prefill cost (tokens not already in KV cache) and decode cost (output length), weighted by a throughput ratio alpha that accounts for GPU parallelism efficiency.

Prefill Cost u_p(i,j)

Computed as the sum of token weights along the LCA path between the previous call l_{j-1} and the current call l_j in the TRT. Tokens already cached contribute zero prefill cost; diverging path tokens must be prefilled.

Decode Cost u_d(i,j)

Modeled as a fixed multiple of the expected output token count. Since LLM outputs are not known in advance, Helium uses profiled average output lengths per operator type, updated with exponential moving averages across batches.

Precedence Delay d(i,j)

When call j depends on the output of call k on a different worker, d(i,j) captures the data transfer and synchronization time. For KV-tensor sharing on the same GPU, this delay is minimal; across GPUs it includes NVLink transfer overhead.

Common Subgraph Elimination (CSE) in Practice

CSE is most impactful in speculative workflows where multiple branches explore the same sub-computation before the final branch is selected. In the Multi-Agent Debate pattern (W2), all agents share an identical context-reading sub-DAG rooted at the shared document. Without CSE, this sub-DAG is independently executed per agent. With CSE, Helium merges these into a single shared operator whose output is fanned out to all debate agents.

CSE Hash Collision Guarantee: The subgraph hash is computed over (operator type, model parameters, quantized prompt template, topological depth) using SHA-256. Structural identity guarantees semantic equivalence because operator types are deterministic functions of their inputs. Hash collisions trigger a fallback full structural comparison before merging.

14 Limitations & Future Work

Helium represents a significant step toward database-style optimization of LLM workloads, but several limitations remain.

Current Limitations

Static Workflow Assumption

Helium requires the workflow DAG to be fully known at compile time. Dynamic agentic patterns where an LLM decides at runtime which next agent to invoke (tool-calling loops, ReAct-style agents) cannot be represented as a static DAG and fall outside Helium's optimization scope. Handling such workflows requires online plan adaptation.

Single-Cluster Scope

The current prototype targets a single server with up to two GPUs. Distributed multi-node deployments require a cross-node KV cache coherence protocol and a distributed version of the TRT, both of which are open engineering challenges. Cross-GPU KV cache sharing is not yet implemented, meaning same-prefix calls must be co-located on one GPU.

Output Length Estimation

The cost model relies on profiled average output lengths per operator type. For highly variable outputs (creative generation, code synthesis), these estimates can be off by 2–5x, degrading scheduling quality. Speculative decoding length predictors or learned cost models could improve accuracy.

DSL Expressiveness

The current Python DSL supports DAG-structured workflows but does not natively express iterative loops with early termination (e.g., retry until answer quality threshold is met). Users must manually unroll such loops into fixed-depth DAGs, which may over- or under-allocate compute.

Future Directions

Online Plan Adaptation

Extend Helium to support partial-dynamic DAGs where sub-graphs can be spawned at runtime based on LLM decisions. A speculation-then-commit mechanism could generate candidate branches optimistically and prune them on confirmation, similar to speculative execution in out-of-order processors.

Learned Cost Models

Replace the hand-tuned alpha parameter and average output length estimates with a lightweight regression model trained on historical execution traces. This would improve scheduling accuracy for variable-output operators and adapt to changing model behavior across fine-tuning iterations.

Distributed KV Cache Coherence

Design a coherence protocol for TRT nodes across GPU nodes, analogous to distributed shared memory protocols. Nodes communicate cache validity bits over RDMA, allowing cross-node prefix reuse without full KV tensor transfer for short shared prefixes.

Heterogeneous Model Routing

Extend the worker model beyond homogeneous LLM instances to support routing across models of different sizes. Lightweight operators (format, classification) could be dispatched to smaller models while reasoning-heavy operators retain large models, enabling cost-aware model selection within the same DAG.

Persistent Cross-Batch Prompt Cache

The current prompt cache is maintained in-process and flushed between unrelated workflow executions. Persisting it across sessions (to disk or a shared KV store) would benefit workloads where the same workflow template is invoked repeatedly by different users, amortizing the static prefix prefill cost across thousands of requests.

Open Question: The database analogy suggests that as agentic workloads mature, a full query optimizer with statistics collection, histogram-based cardinality estimation for LLM output distributions, and adaptive re-optimization during execution may be warranted. Whether the overhead of such machinery is justified by gains depends on workload repetition rate and output predictability — both active research questions in the LLM serving community.
