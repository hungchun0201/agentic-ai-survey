Helium: Efficient LLM Serving for Agentic Workflows

A Data Systems Perspective -- Modeling agentic workflows as database query plans with LLMs as first-class operators. Template-aware radix-tree scheduling for proactive caching and cache-aware optimization.

Noppanat Wadlom, Junyi Shen, Yao Lu National University of Singapore vLLM v0.16.0 · Qwen3 Models

1 Overview

Agentic workflows are composed of sequences of interdependent LLM calls that have become a dominant workload in modern AI systems. These workflows exhibit extensive redundancy from overlapping prompts and intermediate results due to speculative and parallel exploration. Existing LLM serving systems like vLLM focus on optimizing individual inference calls, overlooking cross-call dependencies and leading to significant inefficiencies.

Core Insight: An agentic workflow resembles a batch analytics pipeline where LLM calls function as operators. By modeling workflows as query plans and applying classical database query-optimization principles, Helium achieves holistic end-to-end optimization across workflows.
Workflow-Aware Serving

Models agentic workflows as query plans with LLM invocations as first-class operators, enabling proactive caching and global scheduling.

Proactive KV Caching

Pre-warms KV caches for static prompt prefixes identified during compilation. Maintains a global prompt cache to bypass redundant operators.

Cache-Aware Scheduling

Cost-based scheduling that leverages a templated radix tree to capture prompt structure and maximize prefix cache reuse across batches.

Up to 1.56x Speedup

Substantial speedups over state-of-the-art systems across diverse workload patterns while preserving exact semantics.

2 Background & Motivation

Three key disparities differentiate LLM workflows from traditional SQL pipelines, making existing per-call optimization strategies insufficient:

Three Disparities: Traditional SQL vs. Agentic Workflows

Representative Workflow Patterns

W1: Map-Reduce

Multiple expert agents process inputs concurrently, followed by a summarizer that aggregates outputs.

W2: Multi-Agent Debate

Multiple agents debate over a given context and question before arriving at a final answer through rounds.

W3: Reflection

An expert drafts an answer, followed by multiple critic agents that provide feedback to refine the output.

W4: Iterative Refinement

Documents are divided into chunks; a summarizer iteratively refines the summary over each chunk.

W5: Parallel Chains

Expert agents with distinct roles extract insights from separate input chunks, aggregated by a writer agent.

3 System Architecture

Helium adopts a classic multi-stage query processing architecture, dividing execution into parsing, optimization, and processing phases. Each agentic workflow is represented using a procedural language that specifies individual LLM or agent calls and their dependencies.

4 Query Optimizer Design

The query optimizer identifies sharing opportunities at multiple granularities, from entire sub-workflows to prompt prefixes, and constructs the logical plan for execution. The optimization process comprises two stages.

Stage 1: Initial Plan Pruning

Operator Pruning

Backward traversal from output nodes removes operators that do not contribute to the final output. Efficiently removes speculative branches that are not rendered.

Common Subgraph Elimination (CSE)

Identifies and merges structurally identical subgraphs sharing the same inputs. Subgraphs are hashed based on topology and input node identifiers to detect duplicates efficiently.

Stage 2: Cache-Aware Logical Plan Optimization

Helium leverages a global prompt cache to bypass redundant computation at the operator level. The optimizer performs a recursive, bottom-up traversal of the DAG:

Cache Substitution: For each operator, a signature is computed from its type and materialized inputs. On a cache hit, the LLM operator is replaced with a lightweight CacheFetch operator that stores a pointer to the cached result. This converts a computational dependency into a simple data retrieval. 
// Pseudo-code: Cache-Aware Plan Optimization
for each operator in bottom_up_traversal(DAG):
    signature = compute_signature(op.type, op.materialized_inputs)
    if prompt_cache.contains(signature):
        replace(operator, CacheFetch(cached_result_ptr))
    else:
        prompt_cache.insert(signature, operator)

5 Query Processor Design

Templated Radix Tree (TRT)

The TRT is a novel data structure that represents the prefix structure of both static prompt components and dynamic parts derived from other operators' outputs. It captures dependencies among its leaf nodes.

Workflow DAG to Templated Radix Tree

Proactive Cache Management

During the first execution of a workflow, the query processor precomputes and stores the KV cache for static token sequences in GPU memory. In subsequent batches, the LLM engines can directly use these precomputed KV tensors, avoiding redundant prefill computations.

Scheduling Problem Formulation

The scheduling task is modeled as a multi-worker scheduling problem to find an assignment of calls to workers and an execution order that optimizes the makespan. The cost model uses token usage and precedence delays:

// Token usage decomposition
u(i,j) = alpha_i * len_out(l_j) * u_p(i,j) + u_d(i,j)

// Prefill usage: new tokens not shared with previous call
u_p(i,j) = sum(omega(v)) for v in LCApath(l_{j-1}, l_j)

// Objective: Minimize total token step (makespan)
T(sigma) = max_{i,j} c(i,j)
subject to precedence delays and dependency constraints

6 Cache-Aware Scheduling Algorithm

The scheduling problem is NP-hard (reducible from parallel machine makespan minimization). Helium proposes a cost-based, cache-aware greedy algorithm inspired by SGLang's DFS scheduling but operating at the operator-level TRT.

function Schedule(workflow_dag):
    partitioned_dag = PartitionWorkflowDAG(workflow_dag)
    tree = BuildSchedulingTree(partitioned_dag)
    while not Recurse(tree, tree.root, false) do
        Recurse(tree, tree.root, true)  // Force progress if stuck
    return tree.GetFinalSchedule()

function Recurse(tree, node, force):
    if node.is_leaf:
        if tree.CanSchedule(node, force): tree.ScheduleNode(node)
    else:
        for each child in SelectChildren(node, force):
            if Recurse(tree, child, force): node.RemoveChild(child)
        tree.UpdateState(node.children)
        force = false  // Only force first child
    return node.IsEmpty()
Key Insight: The nested sequence structure is crucial for maximizing cache reuse. Workers process LLM calls query-by-query within an inner sequence (to reuse per-query prefix) and operator-by-operator across inner sequences (to reuse static prefix). Time complexity is O(|V_int| * c_max^3 + |E'| * d_max).

7 Agentic Workflows DSL

Helium implements a Python-based DSL that constructs a symbolic DAG of primitive operators under a lazy dataflow model. Operator calls record nodes and edges instead of executing.

from helium import graphs, helium, ops

# Define the agentic workflow
q = ops.placeholder("q")
answer = ops.llm([ops.Msg("user", q)])
revise_prompt = ops.fmt("Revise answer...", q, answer)
fin_answer = ops.llm([ops.Msg("user", revise_prompt)])

# Build and compile the DAG
graph = graphs.build([fin_answer]).compile(q=["How many inches is 1 meter?"])

# Execute the DAG with Helium runtime
print(helium.invoke(graph))

8 Evaluation Setup

Models & Testbed

Qwen3-8B and Qwen3-14B on an AMD EPYC 9554 64-Core processor with two 94GB NVIDIA H100 NVL GPUs. Built on vLLM v0.16.0 with automatic prefix caching and chunked prefill enabled.

vLLM

Naive sequential execution per query on unmodified vLLM backend.

Graph-based orchestration framework with Runnable interface for async batch execution.

AgentScope

Actor-based distributed execution with decentralized agent servers.

Parrot

LLM service using Semantic Variables and dataflow analysis across requests.

KVFlow

KV cache management using Agent Step Graphs with hierarchical prefetching.

Cache-aware scheduling is the second-largest contributor (17.66%), emphasizing the value of intelligent task ordering for cache reuse.

Scheduling Optimality

Benchmarked against MILP optimal solutions, Helium achieves an average optimality gap of only 0.9 +/- 1.4% with a maximum gap of 3.6%, significantly outperforming all baselines (QueryWise: 72.4%, OpWise: 17.6%, Random: 16.3%, LSPF: 14.5%).

Overhead Analysis

Planning Overhead

Combined planning overhead (QO + CAS + TRT) remains below 230ms even at 16 branches, negligible relative to LLM inference cost.

Memory Footprint

At 16 branches, Helium consumes only 552 KiB vs SGLang's 14.8 MiB -- the TRT scales with workflow structure, not request count.

Batch Efficiency

1.18x more requests per batch and 1.41x higher peak effective tokens compared to LangGraph, improving GPU utilization.
