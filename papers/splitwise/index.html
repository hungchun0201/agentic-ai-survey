Section 1

Overview — Hardware Profiling of LLM Inference Phases

Splitwise is built on a fundamental observation: generative LLM inference consists of two computationally distinct phases with radically different hardware utilization profiles. By characterizing these phases on production traces from Microsoft Azure, the authors reveal that the conventional practice of running both phases on the same GPU is inherently wasteful.

The Core Insight

Every generative LLM request passes through two phases: prompt computation (prefill) processes all input tokens in parallel to generate the first output token, while token generation (decode) produces subsequent tokens one at a time, auto-regressively. These phases have fundamentally different computational requirements — prefill is compute-bound and saturates GPU FLOPs, while decode is memory-bandwidth-bound and leaves compute resources largely idle.

Production Trace Analysis

The paper uses production traces from two Azure LLM inference services (coding and conversation) captured on November 11, 2023. Key statistics: coding service has median prompt size of 1500 tokens and median output of 13 tokens; conversation service has median prompt of 1020 tokens with median output of 129 tokens. With mixed continuous batching, machines spend 60-70% of time with fewer than 20 active tokens batched.

Architecture: Heterogeneous Phase-Split Deployment

Splitwise separates the prompt and token phases onto different machine pools, connected via high-speed InfiniBand. A hierarchical scheduler routes requests across prompt, token, and mixed pools with KV-cache transfer optimization.

Splitwise System Architecture

Splitwise Design Variants

Variant Prompt Machine Token Machine Prompt-Token BW Relative Cost Relative Power
Splitwise-AA DGX-A100 DGX-A100 1× 1× 1×
Splitwise-HH DGX-H100 DGX-H100 2× 2.35× 1.75×
Splitwise-HA DGX-H100 DGX-A100 1× Mixed Mixed
Splitwise-HHcap DGX-H100 DGX-H100 (50% power) 2× 2.35× 1.23×

Serialized KV Transfer

Naive approach: wait for the entire prompt phase to complete, then transfer the full KV-cache to the token machine. Transfer time is proportional to prompt size and inversely proportional to interconnect bandwidth. For large prompts, this adds up to 3% E2E overhead.

Layer-wise Async Transfer

Optimized approach: as each layer finishes computing in the prompt phase, the corresponding KV-cache is immediately sent via a one-sided RDMA put using MSCCL++. This overlaps transfer with computation, reducing visible overhead to ~0.8% of E2E and constant ~5-8ms non-overlapped time.

Core: Discrete Event Simulator

simulator.py Python
class Simulator: """A discrete event simulator that schedules and runs Events.""" def __init__(self, end_time): self.time = 0 self.end_time = end_time self.events = [] # min-heap priority queue self.deleted_events = [] # lazy deletion list def schedule(self, delay, action): # Run immediately if delay is 0, else push to heap if delay == 0: action() return None event = Event(self.time + delay, action) heapq.heappush(self.events, event) def run(self): while self.events and self.time < self.end_time: event = heapq.heappop(self.events) if event in self.deleted_events: continue # lazy cancellation self.time = event.time event.action() # advance simulation clock

The simulator uses a global singleton pattern. TraceSimulator extends this to load request arrivals from CSV traces and wire them through Router → Application → Scheduler → Executor → Instance.

🔍
Implementation detail: The SplitwiseInstance batch selection uses a 5-phase priority scheme: (1) tasks exceeding max preemption limit, (2) pending prompt tasks, (3) blocked (previously preempted) token tasks, (4) tasks from the old batch, then (5) any remaining pending tasks. This prevents token starvation while maintaining prompt priority.

def schedule(self, request): # 1. Try to find prompt instance from prompt_pool, then mixed_pool prompt_instance = self.find_best_prompt_instance( [self.prompt_instances, self.mixed_instances]) # 2. Try to find token instance from token_pool, then mixed_pool token_instance = self.find_best_token_instance( [self.token_instances, self.mixed_instances]) # 3. If no prompt instance found → steal from token pool → mixed if prompt_instance is None: prompt_instance = min(self.token_instances, ...) self.mixed_instances.append(prompt_instance) # 4. If still no pair found → devolve to baseline mixed batching if prompt_instance == token_instance: prompt_task.chain = [token_task] # run-to-completion else: self.add_kv_cache_transfer(...) # insert DAG edge for KV flow

When a mixed-pool instance becomes idle, notify_free_instance() transitions it back to its original pool (prompt or token), enabling dynamic rebalancing based on load.

Section 5

Key Findings: Seven Insights from Hardware Profiling

The paper derives seven actionable insights from extensive profiling on A100 and H100 GPUs with production traces, forming the theoretical foundation for Splitwise's design.

Insight I

Diverse Workload Distributions

Different inference services have widely different prompt and token size distributions. Coding: large prompts (1500 median), few tokens (13 median). Conversation: moderate prompts (1020), many tokens (129).

Insight II

Poor Batching Utilization

Mixed continuous batching spends most of its time with very few active tokens batched. 60-70% of the time, machines run with 20 or fewer tokens. Coding experiences even worse batching.

Insight III

Token Phase Dominates E2E

For most requests, the majority of E2E time is spent in the token generation phase. A prompt phase with 1500 input tokens takes the same time as generating only 6 output tokens.

Insight IV

Asymmetric Batching Behavior

Prompt phase throughput drops after 2048 batch tokens. Token phase throughput keeps scaling with batch size until memory exhaustion. Splitting enables optimal batching for each phase.

Insight V (Critical)

Compute vs Memory Bound

Batching during prompt phase is compute-bound, whereas the token phase is memory capacity and bandwidth bound. This is the fundamental insight that enables phase splitting.

Insight VI

Distinct Power Profiles

Prompt phase utilizes the GPU power budget efficiently; power draw increases with batch size. Token phase does not — power capping by 50% has almost no impact on token generation latency.

Insight VII

Hardware Flexibility for Decode

Token generation can run on less compute-capable (cheaper, older) hardware with better Perf/W and Perf/$ efficiency. A100 shows similar or better cost-energy profile compared to H100 for decode.

✓
Design implication: These seven insights converge to a single conclusion — running both phases on the same homogeneous GPU wastes resources. Prefill needs compute (H100), decode needs memory bandwidth (A100 suffices), and their power profiles are opposites. Splitting them enables right-sizing hardware for each phase.

Section 7

Mixed-Pool Concept for Agentic Workloads

Splitwise's architecture is particularly relevant for agentic AI workloads, where multi-step reasoning chains create variable-length prefill and decode phases with unpredictable load patterns.

Why Splitwise Matters for Agents

  • Multi-step Reasoning: Agentic workflows involve chains of LLM calls where each step's output feeds the next. Phase splitting prevents the compute-heavy initial processing from blocking ongoing token generation of previous steps.
  • Tool-Calling Patterns: Agents frequently call tools, requiring rapid prefill of tool results (large prompts) followed by brief decode phases. Splitwise's prompt pool handles these compute-heavy bursts without disrupting ongoing generation.
  • Dynamic Load Balancing: The MixedPoolScheduler dynamically moves machines between prompt and token pools, adapting to bursty agentic workloads where prefill demand spikes during tool orchestration.
  • Cost Efficiency: Agentic systems are cost-sensitive due to high token usage across multiple LLM calls. Using cheaper A100s for decode phases reduces per-agent-step cost while maintaining latency SLOs.

Workload Robustness

The paper demonstrates that Splitwise is robust to workload changes:

  • Validated Running a conversation trace on a coding-optimized cluster: all Splitwise designs perform better than any baseline, with only 7% throughput reduction for heterogeneous variants.
  • Validated Switching from BLOOM-176B to Llama2-70B on the same cluster: Splitwise designs consistently outperform baselines at higher load, maintaining best latency.
  • Extensible All modern transformer-based generative LLMs (2B to 176B+ parameters), including MoE architectures, exhibit the same two-phase behavior that Splitwise exploits.
🔬
Agentic opportunity: The mixed pool acts as a natural buffer for the bursty, unpredictable traffic patterns of agentic orchestration, preventing both prompt starvation (waiting for compute) and token starvation (via preemption limits).

Section 8

Paper vs Implementation: Fidelity Analysis

The SplitwiseSim repository is the official artifact accompanying the ISCA'24 paper. Here we analyze how faithfully the simulator implements the paper's design and identify both strengths and gaps.

Implementation Fidelity Matrix

Paper Concept Status Implementation Details
Prompt/Token Phase Splitting Full Match Separate PromptTask/TokenTask classes with distinct memory, compute, and iteration handling. Request DAG cleanly models prompt→token dependency.
KV-Cache Transfer Full Match KVScheduler.add_kv_cache_transfer() inserts a Flow node into the request DAG. DummyLink with configurable bandwidth models transfer delay. Size estimated from model architecture.
Layer-wise Overlap Approximated OverlapKVJSQScheduler simulates 90% overlap by using 10× the actual bandwidth. Effective but not a true layer-by-layer overlap simulation.
3-Pool Architecture Full Match MixedPoolScheduler maintains prompt_instances, token_instances, and mixed_instances lists with dynamic pool transitions and notify_free_instance() callbacks.
JSQ Scheduling (CLS) Full Match Multiple JSQ variants: by request count (KVJSQScheduler) and by pending tokens (KVTokenJSQScheduler). O(n) search per scheduling decision.
Mixed Continuous Batching (MLS) Full Match ORCAInstance implements iteration-level scheduling with prompt priority and contiguous iteration optimization. SplitwiseInstance adds preemption support.
Token Preemption Full Match SplitwiseInstance.preempt_task() with max_preemptions limit (default: 4). Priority queue ordered by (preemption_count, arrival_time) prevents starvation.
Performance Model Full Match DatabasePerformanceModel with profiling data from DGX-A100/H100. Linear interpolation via scipy interp1d. Separate prompt and token predictors keyed by (model, hardware, TP).
Power Modeling Placeholder PowerModel exists but is marked TODO/unused. ConstantPowerModel provides per-task power values but cluster.py's power tracking is not connected to simulation logic.
Real Hardware Validation Separate KV-cache transfer implementation is done on vLLM (not in this repo). Available as a separate PR: github.com/vllm-project/vllm/pull/2809.
Dynamic Reprovisioning Partial Pool transitions are implemented (mixed pool overflow). But full machine re-purposing (changing model weights) is not simulated — treated as negligible overhead.
Production Traces Full Match generate_trace.py downloads Azure LLM traces and generates simulator-compatible CSVs with Poisson arrivals and production prompt/token distributions.

Strengths of Implementation

  • ✓ Clean object-oriented architecture with clear separation of concerns
  • ✓ Request DAG abstraction cleanly models prompt→KV-transfer→token pipeline
  • ✓ Extensive configurability via Hydra YAML — 15+ cluster configs, 10 schedulers
  • ✓ Profiling-based performance model with <3% MAPE validated against hardware
  • ✓ Validated end-to-end with 50K+ iterations on production traces
  • ✓ Parallel simulation support via Ray for design space exploration
  • ✓ Debug mode with per-instance iteration logs for Gantt chart visualization

Limitations & Gaps

  • ⚠ Power model is stub only — power-optimized results rely on analytical calculation, not simulation
  • ⚠ KV-cache overlap approximated as 10× bandwidth rather than true layer-by-layer modeling
  • ⚠ Network model is simplified — DummyLink ignores congestion and contention effects
  • ⚠ Arbiter and Router are no-ops — multi-application routing not implemented
  • ⚠ Global singleton pattern for simulator, performance_model, power_model limits testability
  • ⚠ JSQ scheduling uses O(n) search — acknowledged as inefficient in code comments
  • ⚠ No prompt chunking support — prompts are always fully processed in one iteration
📝
Artifact note from paper: "Artifact functionality was only tested for the traces and SplitwiseSim due to limited hardware availability." The real hardware validation (vLLM-based KV transfer over InfiniBand) was run on 2 DGX-A100 + 2 DGX-H100 machines on Microsoft Azure. The simulator is the primary evaluation vehicle for the cluster-scale design space exploration.

Overall Assessment

SplitwiseSim is a high-quality research artifact that faithfully implements the core concepts of the Splitwise paper. The three-pool architecture with dynamic rebalancing, the hierarchical CLS/MLS scheduling, KV-cache transfer via request DAG modification, and the profiling-based performance model are all implemented with strong alignment to the paper's design. The primary gaps are in power modeling (placeholder only) and network fidelity (simplified link model). The paper's headline claims — 1.4× throughput at 20% lower cost and 2.35× throughput at iso-budget — are supported by the simulator's design space exploration methodology, which is well-implemented and reproducible.
