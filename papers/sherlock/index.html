Sherlock: Reliable and Efficient Agentic Workflow Execution

arXiv:2511.00330

Selective Verification with Speculative Execution -- Combining counterfactual fault analysis with learned verifier selection and speculative execution for reliable agentic serving. 48.7% execution time reduction.

Yeonju Ro, Haoran Qiu, Inigo Goiri, Rodrigo Fonseca, Ricardo Bianchini, Aditya Akella, Zhangyang Wang, Mattan Erez, Esha Choukse

1 Overview

LLM inference is inherently error-prone and may produce incorrect results due to hallucinations, reasoning flaws, or inadequate context. In agentic workflows, mistakes at early nodes propagate downstream, amplifying as they go and contaminating the final output. Verification is essential but costly -- it can increase latency by up to 28.9x and cost by 53.2x.

Core Insight: Sherlock addresses three key questions: (1) which nodes are most error-prone and deserve verification, (2) how to select the most appropriate verifier for each node, and (3) how to use verification with minimal latency impact through speculative execution and selective rollback.

2 Motivation & Background

Agentic Workflows

A complex task is decomposed into subtasks forming a workflow graph. Each node represents an LLM call optionally augmented with tool invocations. Edges represent the output (history context) passed from upstream to child nodes. Workflows can be statically defined or dynamically generated by LLM planners.

Figure 1 (from paper): Example agentic workflow: an LLM planner decomposes a complex task into subtasks W1-W4 executed as a DAG, where each node may invoke tools and pass outputs downstream.

The Verification Problem

Verifiers validate LLM outputs through mechanisms like self-reflection, debate, or LLM-as-a-Judge. However, each verification incurs additional LLM calls, significantly increasing latency along the critical path.

6 Cost-Optimal Verifier Selection

Verifier Selection as Preference Learning

Given a node to verify, the next step is selecting which verifier to attach. Sherlock formulates this as a preference learning problem. For verifiers V and task prompt tau:

// Utility function: accuracy gain minus lambda * cost
U(v_i, tau) = P(v_i, tau) - lambda * C(v_i, tau)

// GRPO objective: maximize log-likelihood weighted by advantage
J_GRPO(theta) = E_{tau~D} [ sum_i A(v_i, tau) * log f_theta(v_i | tau) ]

// Advantage: normalize preferences within the group
A(v_i, tau) = U(v_i, tau) - (1/N) * sum_j U(v_j, tau)
Lambda as a System Knob: The lambda parameter represents the "willingness to pay" -- the marginal cost the system is willing to incur for improved performance. Users can tune lambda to control the accuracy-cost tradeoff, serving as a system-level knob.

Model Architecture

Each prompt is encoded into a feature vector using a pretrained distilbert-base encoder. The policy model applies linear layers over these encoded features to produce logits over candidate verifiers, normalized into a probability distribution defining the selection policy.

7 Speculative Execution

Since verifier-revised outputs often remain semantically consistent with the original (Insight 3), Sherlock introduces speculative execution that overlaps verification of a node with its downstream computation.

Selective Rollback

When verification revises an output, Sherlock determines whether speculative results can still be retained using task-specific similarity metrics (ROUGE-L for instruction/tool tasks; full rollback for code/math). If the similarity between the revised and original output exceeds a threshold, rollback is unnecessary.

Speculation Depth Bound

// Set of nodes eligible for speculation at node i
N_spec = { j | sum_{k=i}^{j} lat_exec^(k) < lat_vrf^(i) }

// Expected speculative cost
C_spec^i = (1 - m_i) * sum_{j in N_spec} (C_exec^j + C_vrf^j)

// Budget constraint
C_spec(d) <= B   // Tunable speculation budget

Finite State Machine

Each node transitions through states: waiting -> running -> verifying/completed. From verifying: success leads to completed; fail leads to failed; from failed: rerun leads back to completed.

End-to-End Workflow Execution Flow

Sherlock operates in two phases. During domain onboarding, it profiles the workflow topology and trains the verifier selector offline. During the online phase, it executes workflows with attached verifiers and speculative downstream execution.

GRPO Training: How the Verifier Selector Learns

The selector is trained with Group Relative Policy Optimization (GRPO). For each task prompt, all candidate verifiers are scored to form a group. The advantage for each verifier within the group is its relative utility: how much better or worse it is than the group mean. The policy parameters are updated to increase the log-likelihood of high-advantage verifier choices.

Step 1: Score All Verifiers

For a given task prompt τ, compute utility U(vᵢ, τ) = P(vᵢ, τ) − λ·C(vᵢ, τ) for every candidate verifier vᵢ in the pool, where P is accuracy gain and C is normalized cost.

Step 2: Compute Group Advantage

Normalize utilities within the group: A(vᵢ, τ) = U(vᵢ, τ) − (1/N)·Σⱼ U(vⱼ, τ). This zero-mean advantage reduces variance and stabilizes training without a separate value network.

Step 3: Policy Gradient Update

Update θ by maximizing J_GRPO(θ) = E[Σᵢ A(vᵢ,τ)·log fθ(vᵢ|τ)]. High-advantage verifiers get positive gradient signal; low-advantage ones are suppressed, teaching the selector the accuracy-cost Pareto frontier.

Speculative Execution: Rollback Decision Logic

When a verifier revises node output, Sherlock does not blindly roll back all speculative work. It applies a task-aware similarity check. Only if the semantic distance between the original and revised output exceeds the fitted threshold does it invalidate and re-execute downstream nodes.

Rollback Decision Tree for Revised Verification Output
Verifier revises output at node i
Task type: Instruction / Tool
Compute ROUGE-L(original, revised). If score ≥ threshold → retain speculative results (commit). If score < threshold → rollback and re-execute affected descendants.
Task type: Code / Math
Any semantic change in code or numeric answer triggers full rollback of all speculative descendants. Exact correctness is non-negotiable in these domains.
Key stat: Match rates (original = revised) exceed 0.6–0.8 across benchmarks, meaning rollback is avoided in the majority of cases and speculative savings are fully realized.

11 System Implementation

Hardware and Software Stack

Sherlock is deployed on a single server with 8× NVIDIA A100 GPUs (80 GB each). The base executor LLM (Llama-3.1-8B-Instruct) occupies 2 GPUs; verifier models are co-located on the remaining GPUs. Workflows are generated by the Flow LLM planner and managed by the Sherlock runtime scheduler.

Lambda Knob: Controlling Accuracy-Cost Tradeoff at Serving Time

The λ hyperparameter in the GRPO utility function acts as a system-level dial that operators can adjust without retraining. A low λ favors accuracy regardless of cost; a high λ aggressively minimizes cost. This makes Sherlock deployable across a spectrum of SLA tiers, from interactive real-time agents to batch processing pipelines.

λ ≈ 0 (Accuracy-First)

Selector ignores cost; prefers high-accuracy verifiers (Adv-Refine, Debate). Appropriate for high-stakes workflows where correctness is critical: medical reasoning, legal document analysis.

λ = 0.5 (Balanced)

The default operating point in the paper. Approaches the Pareto frontier, delivering 26% cost reduction versus the oracle verifier while maintaining comparable accuracy.

λ → large (Cost-First)

Selector heavily penalizes expensive verifiers; gravitates toward Self-Refine and LLM-as-Judge. Appropriate for throughput-sensitive batch pipelines where minor accuracy drops are acceptable.

Speculation Budget Management

Sherlock bounds the depth of speculation to avoid runaway resource consumption. For each verifier at node i, only downstream nodes whose cumulative execution time is less than the verifier's own latency are eligible for speculative execution. An additional budget B caps total speculative cost.

Tunable knobs summary: (1) k — verifier budget (number of nodes to verify per workflow); (2) λ — accuracy-cost tradeoff for verifier selection; (3) B — speculation cost budget; (4) similarity threshold τ_sim — controls how aggressively speculative results are reused.

12 Limitations & Future Work

Sherlock is a significant step forward in reliable agentic serving, but several open challenges remain. Understanding these constraints clarifies where future research should focus.

Current Constraints

Offline Onboarding Cost

Fault injection and GRPO selector training require 15K+ execution traces per domain. This is a one-time cost, but makes Sherlock ill-suited for novel workflows that have not been profiled. Every new domain family requires a fresh onboarding pipeline run.

Static Workflow Assumption

Vulnerability scoring and verifier placement operate on a fixed DAG topology. Dynamically generated workflows (where the LLM planner constructs the graph at runtime) bypass the onboarding phase entirely and fall back to heuristics. Adapting to fully dynamic graphs remains an open problem.

Single-Server Evaluation

All experiments are conducted on a single 8-GPU server. The behavior of speculative execution under distributed multi-node deployments — where network latency adds noise to the verifier-execution overlap — has not been studied.

Verifier Pool is Fixed

The four verifiers (Self-Refine, LLM-as-Judge, Adv-Refine, Debate) are pre-defined. The framework does not currently support online addition of new verifiers or automatic discovery of task-specific verification strategies. Extending the pool requires re-running the GRPO training phase.

No Multi-Turn / Agentic Loop Support

Sherlock targets DAG-structured workflows with finite execution paths. Cyclic agentic loops (where the LLM iteratively refines until a termination condition) introduce unbounded speculation depth, which the current budget formulation cannot handle.

Rollback Granularity

Selective rollback currently operates at the node level. In practice, a node may have produced partial tool call results (database writes, API side effects) that cannot be cheaply undone. True transactional rollback with compensation actions is out of scope.

Promising Future Directions

Online / Few-Shot Selector Adaptation

Warm-starting the GRPO selector on a handful of traces from a new domain using meta-learning or in-context RL could eliminate the 15K-trace onboarding requirement for novel workflows, bringing Sherlock's benefits to zero-shot agents.

Dynamic Graph Support

Extending vulnerability estimation to dynamically shaped DAGs via graph neural networks that predict node-level fault propagation at scheduling time — without requiring full offline profiling — would unlock Sherlock for planner-driven agentic systems.

Side-Effect Aware Rollback

Integrating Sherlock with a transactional execution layer (e.g., saga pattern for distributed systems) would let agents safely speculate across tool calls that produce external side effects, dramatically expanding the scope of speculative execution.

Distributed Multi-Node Serving

Future work should study Sherlock under disaggregated inference deployments where verifier and executor LLMs reside on separate nodes connected via high-speed interconnects. Network-latency-aware speculation depth bounds could maintain the 48.7% latency improvement at datacenter scale.

Self-Improving Verifier Discovery

Rather than a fixed pool of verifiers, a generative approach could synthesize new verification prompts or fine-tuned judge models tailored to emerging failure modes detected in production. This would transform Sherlock from a static to a continuously self-improving reliability layer.

Broader Agentic AI Implications

Sherlock's topology-driven fault analysis generalizes beyond LLM serving: any complex DAG-structured pipeline with probabilistic node failures (robotic planning, data pipelines, scientific workflows) can benefit from counterfactual vulnerability scoring and cost-aware verifier assignment.

Key Takeaway
Sherlock demonstrates that reliable agentic serving does not require exhaustive verification of every node. By combining topology-aware vulnerability analysis, learned cost-optimal verifier selection, and speculative execution with selective rollback, it achieves state-of-the-art accuracy with 26% lower verification cost and 48.7% lower execution latency. The principled three-part decomposition — placement, selection, and runtime — provides a template for building reliable serving infrastructure for future large-scale agentic systems.
