Sherlock: Reliable and Efficient Agentic Workflow Execution

Selective Verification with Speculative Execution -- Combining counterfactual fault analysis with learned verifier selection and speculative execution for reliable agentic serving. 48.7% execution time reduction.

Yeonju Ro, Haoran Qiu, Inigo Goiri, et al. UT Austin · Microsoft Azure Research 8x A100 GPUs · Llama-3.1-8B

1 Overview

LLM inference is inherently error-prone and may produce incorrect results due to hallucinations, reasoning flaws, or inadequate context. In agentic workflows, mistakes at early nodes propagate downstream, amplifying as they go and contaminating the final output. Verification is essential but costly -- it can increase latency by up to 28.9x and cost by 53.2x.

Core Insight: Sherlock addresses three key questions: (1) which nodes are most error-prone and deserve verification, (2) how to select the most appropriate verifier for each node, and (3) how to use verification with minimal latency impact through speculative execution and selective rollback.

2 Motivation & Background

Agentic Workflows

A complex task is decomposed into subtasks forming a workflow graph. Each node represents an LLM call optionally augmented with tool invocations. Edges represent the output (history context) passed from upstream to child nodes. Workflows can be statically defined or dynamically generated by LLM planners.

The Verification Problem

Verifiers validate LLM outputs through mechanisms like self-reflection, debate, or LLM-as-a-Judge. However, each verification incurs additional LLM calls, significantly increasing latency along the critical path.

State-of-the-Art LLM Verifiers and Their Overhead
Insight 1
Verifiers bring significant cost and latency overhead, underscoring the need for principled verifier placement strategies in an agentic workflow.
Insight 2
Verifiers have distinct accuracy improvements and cost behaviors, which are highly task dependent. A single global verifier configuration is inefficient.
Insight 3
Revised output after verification may not necessarily change from its original output -- motivating speculative execution where downstream tasks proceed in parallel with verification.

3 Verifier Characteristics

Task-Dependent Accuracy-Cost Tradeoff

The relationship between accuracy and cost across verifiers is non-monotonic: higher cost does not necessarily yield better accuracy. In some cases, excessive or misapplied verification can even reduce accuracy, as redundant checks introduce inconsistencies or conflicting judgments.

Verification Redundancy

Most verifiers retain a large portion of original outputs, with match rates commonly above 0.6-0.8 for Instruction, Code, and Math tasks. This confirms that outputs are frequently unchanged after verification, justifying speculative execution.

5 Identifying Error-Prone Nodes

Fault Model for Agents

Each node in a workflow is abstracted as four elements: (1) context from upstream, (2) objective/instruction, (3) agent executor (LLM call), and (4) the node's output. Three failure modes are defined:

Topological Error Distribution

Systematic fault injection across 100+ graphs and 15K+ execution traces reveals three key observations:

Position-wise Sensitivity

Terminal nodes are most vulnerable, followed by initial nodes. Intermediate nodes contribute least to end-to-end failure propagation because downstream nodes can often correct intermediate errors.

Fan-in Degree

Strong positive correlation with vulnerability -- nodes aggregating multiple inputs are more likely to amplify or propagate upstream errors.

Fan-out Degree

Little to no correlation with error magnitude -- branching alone does not amplify vulnerability.

6 Cost-Optimal Verifier Selection

Verifier Selection as Preference Learning

Given a node to verify, the next step is selecting which verifier to attach. Sherlock formulates this as a preference learning problem. For verifiers V and task prompt tau:

// Utility function: accuracy gain minus lambda * cost
U(v_i, tau) = P(v_i, tau) - lambda * C(v_i, tau)

// GRPO objective: maximize log-likelihood weighted by advantage
J_GRPO(theta) = E_{tau~D} [ sum_i A(v_i, tau) * log f_theta(v_i | tau) ]

// Advantage: normalize preferences within the group
A(v_i, tau) = U(v_i, tau) - (1/N) * sum_j U(v_j, tau)
Lambda as a System Knob: The lambda parameter represents the "willingness to pay" -- the marginal cost the system is willing to incur for improved performance. Users can tune lambda to control the accuracy-cost tradeoff, serving as a system-level knob.

Model Architecture

Each prompt is encoded into a feature vector using a pretrained distilbert-base encoder. The policy model applies linear layers over these encoded features to produce logits over candidate verifiers, normalized into a probability distribution defining the selection policy.

7 Speculative Execution

Since verifier-revised outputs often remain semantically consistent with the original (Insight 3), Sherlock introduces speculative execution that overlaps verification of a node with its downstream computation.

Speculative Execution Timeline: Execute Ahead, Rollback if Needed

Selective Rollback

When verification revises an output, Sherlock determines whether speculative results can still be retained using task-specific similarity metrics (ROUGE-L for instruction/tool tasks; full rollback for code/math). If the similarity between the revised and original output exceeds a threshold, rollback is unnecessary.

Speculation Depth Bound

// Set of nodes eligible for speculation at node i
N_spec = { j | sum_{k=i}^{j} lat_exec^(k) < lat_vrf^(i) }

// Expected speculative cost
C_spec^i = (1 - m_i) * sum_{j in N_spec} (C_exec^j + C_vrf^j)

// Budget constraint
C_spec(d) <= B   // Tunable speculation budget

Finite State Machine

Each node transitions through states: waiting -> running -> verifying/completed. From verifying: success leads to completed; fail leads to failed; from failed: rerun leads back to completed.

End-to-End Comparison

Compared to the tabular approach, Sherlock consistently achieves higher accuracy at lower cost. Against AFlow (Monte Carlo Tree Search), Sherlock delivers higher accuracy and lower cost across all tasks due to its dynamic verifier selector that adapts flexibly to task characteristics.

Verifier Placement Effectiveness

Sherlock's topology-aware placement consistently achieves higher final accuracy with less budget compared to random and evenly spaced placements, demonstrating that it allocates verification resources more effectively by leveraging observed error distributions.

Accuracy-Cost Pareto Frontier

Sherlock lies on the Pareto frontier and approaches the oracle (best achievable accuracy at lowest cost). Compared to static verifier assignments, Sherlock achieves higher accuracy for the same cost, approaching the accuracy of the oracle verifier while reducing cost by 26%.
