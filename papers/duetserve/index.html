1 Overview

DuetServe introduces a unified LLM serving framework that operates in aggregated mode by default and dynamically activates SM-level GPU spatial multiplexing only when contention threatens latency SLOs. This avoids the resource waste of full disaggregation while preserving its isolation benefits.

2 Motivation: Three Observations

Obs 1: Token Budget Can Violate TBT

Fully consuming the token budget (8192 on H100) can cause per-iteration latency to exceed 180ms even when the roofline model suggests full utilization. A single decode request added to a prefill-only batch causes TBT violation.

Obs 2: Attention Dominates at Long Context

Token-budget schedulers assume linear layers dominate latency. But attention cost grows quadratically with context length, accounting for 25%+ of forward latency at 8K tokens, breaking the token-budget assumption.

Obs 3: PD Disaggregation Underutilizes GPUs

Disaggregated systems satisfy TBT but achieve less than half the total throughput of aggregated systems due to resource imbalance between dedicated P and D workers.

The Opportunity

Prefill is compute-bound (saturates SMs, underutilizes HBM bandwidth). Decode is memory-bound (high HBM traffic, underutilizes SMs). This complementary behavior reveals an opportunity for resource co-execution where both phases share SM and HBM resources efficiently.

3 Attention-Aware Roofline Model

DuetServe uses an analytical model to predict iteration latency based on operator-level compute and memory characteristics. Three operator categories:

// Attention FLOPs and Bytes per request
F_attn(q, c) = 4 * h_q * q * (q + c) * d_h + 2 * h_q * q * (q + c)
B_attn(q, c) = 2 * h_q * q * d_h * s + 2 * h_kv * (q + c) * d_h * s

// Total attention latency (roofline)
t_attn = SUM_r max(F_attn(q_r, c_r) / Pi_SM, B_attn(q_r, c_r) / B_HBM)

// This captures prefill (q>1, c=0), chunked prefill (q>1, c>0),
// and decode (q=1, c>0) uniformly

4 GPU Partitioning Optimization

When the roofline model predicts a TBT violation, DuetServe determines the optimal SM partition (S_d for decode, S_p for prefill) to maximize throughput under the TBT constraint:

// Optimization problem
maximize   (k * T_decode + T_prefill) /
           max(k * t_d(S_d), t_p(S_p))

subject to t_d(S_d) <= tau_TBT     // TBT SLO constraint
           S_d + S_p = S           // Total SM count

// Where:
// t_d(S_d) = roofline latency of decode on S_d SMs
// t_p(S_p) = roofline latency of prefill on S_p SMs
// k = number of decode steps per prefill batch

The scheduler enumerates feasible S_d values (typically single digits), discards those violating TBT, and selects the configuration maximizing throughput. This enumeration takes <1ms.

5 Interruption-Free Execution Engine

DuetServe uses two dedicated CUDA streams bound to different SM subsets via libsmctrl. Key innovations:

CUDA Graph Capture for Decode

Decode kernels are recorded into a reusable CUDA Graph and replayed with negligible launch latency. Prefill cannot be captured due to dynamic tensor shapes from variable-length prompts.

Look-Ahead Decode Execution

Pre-allocates k KV cache slots and prepares metadata for k future decode steps. Launches k pre-recorded CUDA Graphs consecutively without intermediate CPU synchronization, eliminating CPU-GPU stalls.

Adaptive Mode Switching

Seamlessly transitions between temporal sharing (no violation predicted) and spatial sharing (violation predicted) every iteration. No reconfiguration overhead since libsmctrl operates at the kernel launch level.

7 Ablation: Static vs. Dynamic SM Partitioning

Static SM partitions (e.g., Sd22-Sp44, Sd33-Sp33, Sd44-Sp22) show throughput varying noticeably across workloads, confirming that static partitioning leads to persistent imbalance. DuetServe's adaptive scheduling dynamically reallocates SMs at runtime, maintaining balanced utilization under diverse traffic.

Roofline Model Accuracy

For compute-bound prefill, the model closely tracks measured latency with nearly linear reduction up to ~40 TPCs, after which throughput saturates. For memory-bound decode, latency quickly plateaus after a small number of TPCs are active. The model slightly overpredicts decode latency but still captures the scaling trend.

8 Key Takeaways

Best of Both Worlds

DuetServe unifies the benefits of aggregation (high throughput, efficient resource use) with disaggregation (latency isolation) by activating spatial multiplexing only when needed.

Attention-Aware Scheduling

The roofline model's attention-level granularity catches TBT violations that token-budget schedulers miss, especially for long-context workloads where attention cost dominates.

Zero Overhead Isolation

Using libsmctrl for SM masking provides fine-grained, per-kernel SM partitioning without MIG/MPS reconfiguration overhead or multiple GPU contexts.

Production Viability

~3,000 lines of Python, built from scratch with PagedAttention, continuous batching, CUDA Graph capture, and FlashAttention-3. Scheduling overhead <1ms.

// DuetServe dual-stream execution
stream_prefill = cuda.Stream()  // Bound to S_p SMs
stream_decode  = cuda.Stream()  // Bound to S_d SMs

// When spatial sharing is activated:
libsmctrl.set_sm_mask(stream_prefill, prefill_sm_mask)

// Decode kernels: CUDA Graph captured at initialization
// Replayed with near-zero launch latency
decode_graph.replay(stream_decode)

// Prefill kernels: launched individually (dynamic shapes)
for kernel in prefill_kernels:
    kernel.launch(stream_prefill)

// CPU dispatches both streams, GPU executes concurrently
// No CPU-GPU synchronization between streams!

Look-Ahead Decode Execution Detail

In conventional decoding, each iteration requires CPU-side operations: sample tokens, filter completed requests, update KV cache mappings, prepare metadata. DuetServe eliminates these interruptions by:

  1. Pre-allocating k KV cache slots per request (sufficient for k decode steps)
  2. Pre-computing metadata for all k steps before GPU dispatch
  3. Launching k CUDA Graphs consecutively in the decode stream
  4. Deferring CPU-side operations until all k steps complete

This reduces CPU-GPU synchronization from once per step to once per k steps, significantly reducing decode pipeline bubbles.

Adaptive Scaling Across GPU Count

DuetServe's spatial multiplexing scales effectively under multi-GPU execution because SM partitioning operates independently per GPU. Each GPU's scheduler determines its own optimal (S_p, S_d) split based on local workload characteristics.

11 Comparison with Related Approaches

vs. Chunked Prefill (Sarathi-Serve)

Chunked prefill uses a token budget to limit prefill per iteration. DuetServe goes further: when the budget is insufficient, it physically isolates prefill and decode on separate SM subsets, preventing any interference rather than just limiting it.

vs. PD Disaggregation (DistServe, Dynamo)

Full disaggregation wastes resources through duplicated models and idle GPUs. DuetServe achieves the same isolation benefit within a single GPU by activating spatial multiplexing only when needed.

vs. MIG/MPS (NVIDIA)

MIG provides static, coarse-grained GPU partitioning with high reconfiguration cost. MPS offers coarser spatial sharing. DuetServe's libsmctrl approach provides per-kernel SM control with zero reconfiguration overhead.

vs. Drift / Semi-PD / Bullet

These systems explore static or feedback-based SM allocation. DuetServe uniquely combines an attention-aware roofline model for proactive violation detection with a throughput-maximizing partition optimizer, achieving better adaptability to workload changes.

Latency Breakdown

Profiled CPU and GPU activities show DuetServe's adaptive behavior across two consecutive iterations:

  1. Iteration 1 (Spatial): Scheduler assigns 48 TPCs to prefill stream and 18 TPCs to decode stream. 5 decode steps execute concurrently with prefill. Both streams achieve high SM utilization with stable overlap.
  2. Iteration 2 (Temporal): No TBT violation predicted. Execution switches back to standard PD mixed mode on the main stream. No partitioning overhead.

The seamless transition between modes -- every single iteration -- demonstrates DuetServe's zero-overhead adaptability.

12 Roofline Modeling: Deep Dive

Communication Operator Modeling

For tensor-parallel LLM serving with N GPUs, DuetServe models AllReduce latency as a ring protocol:

// AllReduce latency model
t_allreduce = 2(N-1) * alpha                   // Fixed startup
             + 2(N-1) * B_lin_o / (N * B_NVLink) // Data transfer
             + N(N-1) * B_lin_o / (N * Pi_SM)   // Local reduction

// Where:
// alpha = 3 us for H100 (startup latency)
// B_lin_o = linear operator output tensor size
// B_NVLink = aggregate NVLink bandwidth
// Pi_SM = compute throughput of active SMs

Overall Latency Estimation

DuetServe estimates total Transformer block latency by summing all sub-operations:

t_total = L * t_block + t_cls

where L is the number of layers, t_block sums token-level, sequence-level, and communication operator latencies, and t_cls is the final classifier (linear with d_i = d, d_o = vocab_size).

Model Validation Results

The roofline model closely tracks measured latency for compute-bound prefill across both Qwen3-8B (TP=1) and Qwen3-14B (TP=2). For prefill workloads (8x1024), nearly linear latency reduction is observed up to ~40 TPCs, after which throughput saturates as memory bandwidth becomes the bottleneck.

For memory-bound decode, latency quickly plateaus after a small number of TPCs are active (since decode is constrained by HBM bandwidth, not compute). The model slightly overpredicts decode latency but correctly captures the scaling trend, which is sufficient for the binary decision of whether to activate spatial multiplexing.

Partition Optimization Complexity

The scheduler enumerates all feasible (S_p, S_d) splits where S_p + S_d = S (total SM count). For H100 with 132 SMs, this is at most ~66 candidates. After filtering for TBT constraint satisfaction (t_d(S_d) ≤ tau_TBT), it evaluates throughput for remaining candidates. The entire optimization completes in <1 ms on the CPU, negligible relative to GPU iteration time.

13 Hardware Resource Characterization

SM vs. HBM Bandwidth Scaling

DuetServe's design exploits a key asymmetry in GPU hardware:

HBM Bandwidth

Scales super-linearly with active SM count. With just 20% of SMs active, approximately 60% of peak bandwidth is already achieved. This means even a small decode SM partition can access most of the memory bandwidth needed for memory-bound decode operations.

Compute FLOPs

Scales roughly linearly with active SM count (minor quantization effects from TPC granularity). Compute-bound prefill benefits directly from more SMs, making it natural to allocate the majority of SMs to prefill during spatial sharing.

This asymmetry is fundamental to why SM partitioning works: decode (memory-bound) achieves near-full memory bandwidth with few SMs, while prefill (compute-bound) efficiently uses the remaining SMs.

Three production traces capture distinct LLM serving patterns:

DuetServe's adaptive scheduling handles all three workload profiles effectively, transitioning between temporal and spatial sharing as the workload mix changes within each trace.

14 Conclusion and Future Directions

DuetServe demonstrates that the choice between aggregated and disaggregated LLM serving is a false dichotomy. By predicting contention with an attention-aware roofline model and activating fine-grained SM-level partitioning only when needed, DuetServe achieves:

Best-of-Both-Worlds Performance

High throughput of aggregation (shared resources, no duplication) combined with low latency of disaggregation (phase isolation when contention arises).

Production-Grade Simplicity

Single-GPU deployment, no cross-GPU KV transfer, no model duplication. The scheduling overhead (<1ms) is negligible relative to model computation time.

Future Extensions

Integration with multi-turn KV caching (Pensieve-style), cross-GPU spatial multiplexing for tensor-parallel models, and application to mixture-of-experts architectures where expert imbalance creates similar phase-interference patterns.
