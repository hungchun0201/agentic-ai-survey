Core Insight

Despite self-attention being memory-bound, end-to-end LLM serving is compute-bound when considering all operations together. Yet existing systems execute heterogeneous operations (compute, memory, network) sequentially, wasting GPU resources. NanoFlow splits inputs into nano-batches and runs these operations in parallel on different GPU resources within a single device.

What is a Nano-Batch?

A nano-batch is a fine-grained split of the input batch. Each nano-operation processes a portion of the batch independently, allowing compute-bound operations (GEMMs) to overlap with memory-bound (decode attention) and network-bound (AllReduce/AllGather) operations simultaneously.

Why Intra-Device Parallelism?

Prior work focuses on inter-device parallelism (TP, PP). NanoFlow exploits intra-device parallelism: overlapping heterogeneous kernels on the same GPU. The key challenge is kernel interference when concurrent kernels share GPU resources like execution units and caches.

2. The Problem: Sequential Execution Wastes Resources

2.1 Operation Classification

Compute-Bound (Dense)

KQV generation, O projection, Up/Gate/Down projections. Matrix-matrix multiplications with shared weights. Batch size amortizes weight loading cost.

Memory-Bound (Attention)

Decode attention loading per-request KV-cache. Prefill attention is compute-bound; decode attention is memory-bound since each request generates one token.

Network-Bound

AllGather (AG) and AllReduce (AR) for tensor parallelism synchronization. Requires high-bandwidth interconnect (NVLink).

4. Auto-Search: Automated Pipeline Construction

4.1 Two-Stage MILP Optimization

Auto-Search Two-Stage Pipeline Construction
Stage I: Structure Search
1. Profile interference-free kernels
2. Determine nano-batch count, sizes, ordering
3. Solve MILP ignoring kernel interference
4. Minimize total execution time
➤
Stage II: Interference Refinement
1. Fix structure from Stage I
2. Profile pairwise kernel interference
3. Allocate R (resource utilization) per nano-op
4. Map R to P using interference table

4.2 Kernel Interference Modeling

When two kernels run concurrently, they compete for GPU resources (execution units, caches, memory bandwidth). NanoFlow models this via a resource mapping table that maps resource utilization R to normalized performance P:

OperationR=0R=0.1R=0.2R=0.4R=0.8R=0.9R=1.0
GEMM (compute)00.10.20.40.80.91.0
GEMV (memory)00.20.30.50.850.951.0
Network (AG/AR)00.30.50.70.91.01.0
Key trade-off: Allocating R=0.2 to GEMM (getting 20% compute) frees resources for GEMV to achieve P=0.3 (30% memory performance). This non-linear exchange rate means overlapping is beneficial when the total work done exceeds sequential execution.

4.3 MILP Constraints

// Auto-search MILP formulation (simplified)
minimize   total_pipeline_time

subject to:
  // Resource constraint: concurrent R values sum to ≤ 1.0
  forall t: sum(R[op] for op in active_at(t)) ≤ 1.0

  // Batch size: chosen from multiples of 128 up to B_dense
  forall op: batch_size[op] in {128, 256, ..., B_dense}

  // Dependencies: nano-ops inherit parent op dependencies
  forall (a, b) in deps: end(a) ≤ start(b)

  // Overlap: only ops using DIFFERENT resources may overlap
  forall (a, b) co-scheduled: resource_type(a) ≠ resource_type(b)

  // Execution time: D_best / P(R) where P comes from Table 3
  forall op: time[op] = D_best[op] / P(R[op])

6. NanoFlow Runtime

6.1 Asynchronous Request Scheduling

NanoFlow forms the batch for iteration i+1 while iteration i is still executing on the GPU. At iteration i+1, it detects EOS tokens from iteration i-1 (not i). This introduces at most one extra decode token per request (<1% overhead for average decode lengths >100 tokens).

7.1 Nano-Batch Splitting (core/nanobatchSplit.py)

def split_nanobatch(op_list, op_nano_info_map, extra_links):
    """Split operations into nano-batches with redistribution."""
    nano_op_list = []
    for op in op_list:
        if op.name not in op_nano_info_map:
            nano_op_list.append(op)     # pass through unchanged
            continue

        nano_op_info_list = op_nano_info_map[op.name]
        op.isNanoSplit = True

        # Create Redist nodes for input/output redistribution
        for info in nano_op_info_list:
            copied_op = op.copy_nano(info.batch_idx)
            copied_op.setBatchSize(info.batch_size)
            # Wire redistribution edges
            for j, (key, value) in enumerate(copied_op.inputs.items()):
                redists_in[j].outputs[f"output_{info.batch_idx}"] >> value
            nano_op_list.append(copied_op)

    return nano_op_list, additional_virtual_ops

7.2 Executor with DAG Ordering (core/executor.py)

class Executor:
    def plan_layer_ordering(self):
        """Build DAG, topologically sort, plan CUDA events."""
        G = nx.DiGraph()
        for op in self.operations_layers_list:
            G.add_node(op.name, op=op)

        # Add dependency edges (same-layer, prev-layer, next-layer)
        for op in self.operations_layers_list:
            for dep, dep_on_prev_layer in op.prerequisites:
                if dep_on_prev_layer == 0:   # same layer
                    G.add_edge(dep.name, op.name)
                elif dep_on_prev_layer == 1: # previous layer
                    G.add_edge(prev_name, op.name)

        self.ordered_operations = list(nx.topological_sort(G))

    def execute(self, output, main_stream, plan_cuda_graph):
        with torch.cuda.graph(self.cuda_graph, stream=main_stream):
            for op_name in self.ordered_operations:
                op = self.ordered_graph.nodes[op_name]['op']
                op.wait_cuda_event()   # sync dependencies
                op.run()              # launch kernel

8.3 Ablation Study

Nano-batching Only (no overlap)

Splitting alone reduces performance by 13.2% due to smaller batch sizes and reduced GEMM efficiency. The benefit comes entirely from overlapping.

Overlapping Network + Compute

1.07x speedup from overlapping network-bound and compute-bound kernels alone.

Overlapping All Three

1.17x speedup from overlapping all three resource types (compute + memory + network), which more than compensates for nano-batching overhead.

KV-Cache Offloading

3.0% pipeline degradation but 3.02x compute savings for LMSYS-Chat multi-round workloads by avoiding KV recomputation.

8.4 Latency Under SLO Constraints

200ms normalized latency SLO: NanoFlow sustains 1.64x higher request rate than TensorRT-LLM within the SLO on the LMSYS-Chat-1M dataset. The 99th-percentile latency is only 1.07x of average at near-maximum throughput due to consistent batch sizes.

9. Impact and Significance

For LLM Serving Infrastructure

NanoFlow demonstrates that existing systems leave 50-78% of GPU compute unused. By closing this gap, the same hardware can serve 1.91x more requests -- directly translating to proportional cost savings in data center operations.

For Agentic AI Systems

Multi-agent architectures generate bursty, heterogeneous LLM workloads. NanoFlow's higher throughput density means fewer GPUs per agent cluster, and its SLO-awareness enables more predictable agent response times.

Complementary to Other Work

NanoFlow uses Sarathi-Serve's chunked prefill policy for request scheduling and is compatible with PagedAttention (vLLM), quantization, and other serving optimizations. It operates at a different granularity -- intra-device rather than inter-device.

Key Limitation

Kernel interference modeling is hardware-specific (profiled on A100). New GPU architectures would require re-profiling. The auto-search takes ~10 minutes per model/hardware combination, though this is a one-time cost.

OSDI'25 Best Paper: NanoFlow was awarded Best Paper at OSDI 2025 for revealing that LLM serving is compute-bound (not memory-bound as commonly assumed) and demonstrating that intra-device parallelism can close the gap to optimal throughput by a factor of 1.91x over state-of-the-art systems.

10. Theoretical Cost Model Deep Dive

10.1 Three Resource Perspectives

Memory Latency

T_mem = MemSize / MemBW
// All weights loaded each iter
// A100: 80GB / 2000 GB/s
//     = 40ms per iteration

Compute Latency

T_compute = 2 * B_Dense * P_Model
            / Compute
// Dominates when B_Dense
// is large (GQA helps)

Network Latency

T_net = 4 * N_GPU * B_Dense
        * D_model * S_type * L
        / NetBW
// AG+AR activations

10.2 The T_R Ratio (Memory-to-Compute)

The ratio T_R = T_mem / T_compute determines whether a workload is memory-bound (T_R > 1) or compute-bound (T_R < 1). For modern LLMs with Grouped Query Attention:

T_R = (Compute * MemSize) / (MemBW * P_model) * 1 / (2 * B_dense)

// LLaMA-2 70B on 8xA100, B_dense = 2048:
// T_R = (280e12 * 80) / (2000e9 * 70e9) * 1 / (2 * 2048)
// T_R approx 0.07 -- strongly compute-bound!

// Even LLaMA-3 8B on 1xA100:
// T_R approx 0.23 -- still compute-bound
