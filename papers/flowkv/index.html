FlowKV: Low-Latency KV Transfer

A disaggregated inference framework that reduces KV cache transfer latency by 96% (from 0.944s to 0.053s) via memory alignment optimizations and load-aware scheduling for balanced PD resource utilization.

Authors: Weiqing Li, Guochao Jiang, Xiangyong Ding, et al.
Affiliation: Alibaba Cloud Computing
Status: Preprint, Under Review

Overview

Disaggregated inference separates the compute-bound prefill (P) and memory-bound decode (D) stages onto different GPU instances, allowing each to be independently optimized. However, the KV cache must be transferred between P and D nodes, and this transfer can account for nearly a quarter of the entire request latency when using standard NCCL-based approaches with PagedAttention's fragmented block layout.

The KV Transfer Bottleneck

For a request with 13k input tokens and 100 output tokens on LongBench, KV cache transfer time between prefill and decode nodes accounts for approximately 25% of total inference latency. FlowKV eliminates this bottleneck through KV cache structure optimization and memory-aligned allocation.

Three Main Contributions

KV Cache Structure Optimization

Reshapes KV cache tensors from (L,2,B,H) to (B,L,2,H), combining separate per-layer tensors into continuous memory. Reduces NCCL API calls per block by a factor of L x 2.

Segment-Based Memory Allocation

Uses OS-inspired segment management to allocate new requests within contiguous memory segments. Bidirectional segment alignment merges transfers into minimal NCCL calls.

Load-Aware Scheduling

Global controller monitors all P/D node load status in real-time. Adapts scheduling strategy across normal, imbalanced, and extreme load conditions with elastic scaling.

KV Cache Transfer Optimization

Problem: Fragmented Block Transfers

Standard PagedAttention manages KV cache in fixed-size blocks at the block level. When transferring between nodes, each block requires separate NCCL communications. For a request with many blocks, this means thousands of small NCCL kernel invocations, each with its own launch overhead and competition for GPU SM resources with compute kernels like GEMM.

Three-Step Optimization

Step 1: KV Cache Shape Transformation

Reshape from (L, 2, B, H) to (B, L, 2, H) where L = layers, B = blocks, H = hidden dim. Combines separate K and V tensors across all layers into a single continuous tensor per block, reducing NCCL calls by L x 2.

Step 2: Segment-Based Allocation

Allocates new requests within contiguous memory segments using segment-based minimum heaps. Maintains continuous KV cache blocks, merging adjacent free segments during deallocation.

Step 3: Bidirectional Segment Alignment

Before triggering NCCL, compares block ID lists of sender and receiver to find contiguous matches. Merges aligned blocks into single transfers, reducing O(n) operations to O(1).

Hybrid Scheduler Architecture

Each node runs a hybrid scheduler containing both a prefill scheduler and a decode scheduler. Each has its own running, waiting, swapped, and pending queues, sharing a block manager. By default, prefill has priority; all nodes focus on prefill requests when available. Under computational imbalance, the global controller instructs idle nodes to switch inference roles.

Load-Aware Scheduler

Load Scenarios

Normal Load

Both P and D nodes have balanced workloads. The global controller selects the optimal P node minimizing TTFT (considering cache prefix hits) and the closest D node to minimize transfer latency.

Imbalanced Load

One type of node is GPU compute-bound while the other is idle. The global scheduler instructs idle nodes' hybrid schedulers to switch inference roles for several scheduling cycles, alleviating resource bubbles.

Extreme Load

Both P and D nodes are overloaded. The controller assesses load scores and triggers elastic scaling, dynamically increasing or decreasing node counts followed by cluster reconfiguration.

Node Status Indicators

The scheduler monitors multiple metrics per node using a sliding-window approach to smooth transient fluctuations. Load scores are computed via weighted combination of these indicators.

// Load-Aware Scheduling Algorithm (Pseudocode)
while system is running:
    for each node i:
        S_i = collect_status(L_r, L_w, L_sw, L_se, T_b, KV_u, G_u, MB_u)
        C_i_p = weighted_sum(S_i, weights_prefill)   // prefill load score
        C_i_d = weighted_sum(S_i, weights_decode)    // decode load score

    C_p = mean(C_i_p for all i)   // aggregate prefill load
    C_d = mean(C_i_d for all i)   // aggregate decode load

    if |C_p| <= epsilon_low and |C_d| <= epsilon_low:
        // Normal load: optimize for TTFT + transfer latency
        schedule_normal(request, P_nodes, D_nodes)
    elif |C_p| <= epsilon_high and |C_d| <= epsilon_high:
        // Imbalanced: switch idle nodes' roles
        switch_roles(idle_nodes)
    else:
        // Extreme: elastic scaling
        scale_cluster(increase_or_decrease)

Heterogeneous GPU Deployment

A key advantage of disaggregated inference is deploying different GPU types for P and D nodes. FlowKV supports heterogeneous configurations by aligning GPU characteristics with task requirements: compute-heavy GPUs for prefill, memory-bandwidth-heavy GPUs for decode.

Heterogeneous Results (L20 + H20 Servers)

The 4P4D (P-L20/D-H20) configuration outperforms 4P4D (P-H20/D-L20) with 34.67% faster E2E on gov_report, 40.1% on multi_news, and 8.8% on qmsum. Compared to vLLM PD-colocated, the optimal config achieves 48.9% faster E2E on gov_report with concurrent TPOT improvements of 44.57%, 24.2%, and 15%.
