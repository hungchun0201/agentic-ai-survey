CacheGen: KV Cache Compression and Streaming for Fast Large Language Model Serving

A custom codec that exploits distributional properties of KV cache tensors to achieve 3.5–4.3x compression, enabling network-efficient streaming of long-context LLM state with 3.2–3.7x lower time-to-first-token.

Yuhan Liu, Hanchen Li, Yihua Cheng, Siddhant Ray, Yuyang Huang, Kuntai Du,
Junchen Jiang — University of Chicago

1 Overview

The Problem: KV Cache Transfer is the Bottleneck

Modern LLM deployments reuse KV caches to avoid redundant prefill computation for repeated or shared contexts. However, KV caches are enormous — a single 4K-token context on a 7B-parameter model produces roughly 1.6 GB of KV state. Transferring this across the network (from a cache store to a GPU worker) or even loading it from local storage dominates the time-to-first-token (TTFT).

CacheGen addresses this by designing a custom compression codec specifically for KV cache tensors, achieving high compression ratios that generic codecs (gzip, zstd) cannot match, because it exploits the particular statistical structure of attention key and value tensors.

Encoding Path

  1. Split & Reshape: The KV tensor of shape [layers, 2, tokens, heads, head_dim] is split into K and V, each reshaped to [layers, tokens, channels] where channels = heads * head_dim.
  2. Layer-Aware Quantization: Each layer group is quantized with a different number of bins (e.g., 32 for early layers, 16 for later layers) via symmetric min-max quantization.
  3. CDF Computation: Per-channel histograms are computed across tokens and converted into cumulative distribution functions for the arithmetic coder.
  4. Arithmetic Encoding: A custom CUDA kernel (torchac_cuda) encodes each channel's token sequence into a compact bitstream using the per-channel CDF.

Decoding Path

  1. Deserialize: The byte payload is unpickled to recover the CDF, data chunks, and max tensors for dequantization.
  2. Arithmetic Decoding: The CUDA kernel decode_fast_prefsum decodes each channel's bitstream back into quantized integer tokens.
  3. Dequantize: The quantized integers are scaled back to floating point using the stored per-token max values: x = (q - C) / C * max.
  4. Reshape & Stack: K and V are stacked back into the original tensor format, cast to the model's precision (fp16 or bf16).

3 Custom Codec Design

Why Not Standard Compression?

Generic compressors like gzip or zstd treat KV cache bytes as unstructured data and achieve only ~1.1–1.3x compression. CacheGen exploits two key observations about KV cache tensors:

  • Channel-wise distributional regularity: Within a given channel (attention head dimension), values across token positions follow a characteristic distribution. This means a per-channel CDF yields tight arithmetic coding.
  • Layer-dependent sensitivity: Early transformer layers are more sensitive to quantization error than later layers. CacheGen assigns more quantization bins to early layers and fewer to later ones.

Layer-Aware Quantization Scheme

Different layers are quantized with different precision levels. For Longchat-7B (32 layers):

Key Tensors:

Layers 0–9: 32 bins
Layers 10–19: 16 bins
Layers 20–31: 16 bins

Value Tensors:

0–1: 32 bins
Layers 2–31: 16 bins

For the 70B model (80 layers), early layers (0–19) use 32 bins for keys; layers 20–39 use 32 bins; and layers 40–79 use 16 bins.

Quantization: Symmetric Min-Max

Each token row is independently scaled:

def torch_quant_vectorized(bins: Tensor, input_groups: Tensor):
    """bins: [nlayers], input_groups: [nlayers, ntokens, nchannels]"""
    MAX = (bins // 2 - 1)[:, None, None]           # [nlayers, 1, 1]
    max1 = torch.amax(torch.abs(input_groups), dim=-1, keepdim=True)
    factor = MAX / max1                                # per-token scale
    xq = torch.round(input_groups * factor + MAX).to(torch.int8)
    return xq, max1

The max1 tensor (one float per token per layer) is stored as side information for dequantization. The shift by +MAX maps values to [0, 2*MAX] for non-negative integer symbols suitable for arithmetic coding.

Arithmetic Coding with Per-Channel CDFs

After quantization, a CDF is computed per channel across all tokens. This CDF captures how frequently each quantization bin appears in that channel, enabling arithmetic coding to compress near the entropy bound. 

// CUDA kernel (torchac_cuda) exposes:
torchac_cuda.calculate_cdf(quantized_tensor, max_bins)     // -> CDF on GPU
torchac_cuda.encode_fast_new(cdf, encode_input, buffer, lengths)
torchac_cuda.decode_fast_prefsum(cdf, bytes_tensor, prefix_sums, output)

All operations run entirely on GPU. The encode_fast_new kernel processes all layers and channels in parallel, writing compressed bytes into a pre-allocated output buffer. The collect_bytes function then compacts the variable-length outputs into a single contiguous byte tensor.

✓
Sub-chunk Processing: Each chunk is further divided into sub-chunks of CACHEGEN_GPU_MAX_TOKENS_PER_CHUNK = 256 tokens. This bounds GPU memory usage for the output buffer and enables streaming encode/decode of arbitrarily long sequences.

CacheGenEngine: High-Level Chunk Interface

class CacheGenEngine:
    def chunk_kv(self, kv, doc_id, chunk_size=1024, encoded_dir=None):
        """Split KV into chunks, encode each independently."""
        for i in range(nchunks):
            chunk = kv[:, :, :, i*chunk_size:(i+1)*chunk_size]
            serializer = CacheGenSerializer(config, meta)
            encoded = serializer.to_bytes(chunk)
            torch.save(torch.frombuffer(encoded, dtype=torch.uint8), path)

    def decode_kv(self, encoded_dir, doc_id, chunk_size):
        """Decode each chunk and concatenate."""
        for i in range(self.nchunks):
            chunk = deserializer.from_bytes(raw_bytes)
            decoded_kv.append(chunk)
        return torch.cat(decoded_kv, dim=-2)

☆
Where does the compression come from? Quantizing from fp16 to ~4–5 bit integers provides a base ~3x reduction. Arithmetic coding then compresses the quantized symbols below their raw bit width, as the non-uniform per-channel distribution provides additional entropy reduction. Combined, this yields 3.5–4.3x total compression.

TTFT Breakdown

Time-to-first-token consists of: network transfer + decode time. CacheGen reduces both:

  • Transfer: 3.5–4.3x smaller payload directly reduces transfer time
  • Decode: GPU-accelerated arithmetic decoding adds only ~20–50ms overhead, far less than the network time saved
  • Net effect: 3.2–3.7x TTFT improvement at realistic network bandwidths (1–10 Gbps)

6 Adaptive Streaming

Bandwidth-Adaptive Quality Selection

CacheGen introduces an adaptive streaming mechanism inspired by video adaptive bitrate (ABR) streaming. Each chunk of the KV cache can be encoded at multiple quality levels (quantization levels 1–3). At runtime, the streamer selects the highest quality level that fits within the per-chunk time budget based on current network bandwidth.

Config Selection Algorithm

def config_selection(all_bws, chunk_delay, args, length, doc_id):
    num_chunks = round(length / args.chunk_size)
    ttft = 0
    configs = []
    for chunk_id in range(num_chunks):
        bw = all_bws[chunk_id]
        for quant_level in [3, 2, 1]:    # Try highest quality first
            bytestream = load(f"{doc_id}_{chunk_id}_{quant_level}.pkl")
            transfer_time = len(bytestream) / 1e9 * 8 / bw
            if transfer_time < args.slo / num_chunks:
                ttft += transfer_time
                configs.append(quant_level)
                break
        else:
            ttft += chunk_delay    # Fallback to local prefill
            configs.append(0)
    return ttft, configs

The algorithm greedily tries the highest quality (most bins, larger payload) and degrades only if the transfer time for a chunk would exceed SLO / num_chunks. Level 0 means the chunk is computed locally via prefill rather than transferred.

⚠
Mistral-7B Multi-Level Encoding: The implementation supports three quantization levels for Mistral-7B, each with different bin configurations. Level 1 uses 12–16 bins (aggressive compression), Level 2 uses 16–32 bins (balanced), and Level 3 uses 32 bins everywhere (highest quality). These are pre-encoded and stored on disk, then selected per-chunk at serving time.

7 Relevance to Agentic AI Systems

Why CacheGen Matters for Agents

Agentic AI systems involve multi-step reasoning where an LLM agent accumulates context across tool calls, observations, and reasoning chains. This accumulated context translates to a growing KV cache that must be preserved across inference steps, potentially across machines.

Distributed Agent Scenarios

  • Load balancing: Agent requests can be routed to any available GPU. CacheGen enables fast migration of accumulated KV state between workers.
  • Disaggregated prefill/decode: Prefill and decode stages can run on different machines. CacheGen compresses the KV cache produced by the prefill node for fast transfer to the decode node.
  • Multi-agent collaboration: Agents sharing a common context document can store the compressed KV cache centrally (e.g., Redis) and retrieve it on demand.
  • Checkpointing: Long-running agent sessions can checkpoint their KV cache to disk in compressed form for later resumption.

Integration with LMCache

CacheGen is integrated into the LMCache framework, which provides a chunk-based cache engine with pluggable backends (local, Redis, hybrid). The integration enables transparent KV cache compression:

  • remote_serde: "cachegen" in config activates CacheGen compression for all network transfers
  • The pipelined backend (LMCPipelinedRemoteBackend) overlaps network fetch and decompression across chunks
  • Chunk hashing enables deduplication: shared context prefixes are stored once and served to multiple agents
ⓘ
Pipelined Network + Decode: The LMCPipelinedRemoteBackend runs separate threads for network fetching and CacheGen deserialization, allowing chunk N+1 to be downloaded while chunk N is being decoded. This further reduces TTFT for multi-chunk agent contexts.

Strengths of the Implementation

  • End-to-end GPU pipeline: CDF computation, arithmetic coding, and quantization all execute on GPU via custom CUDA kernels, avoiding CPU-GPU transfer overhead.
  • Clean serialization interface: The Serializer / Deserializer abstraction allows easy swapping between CacheGen and plain torch serialization.
  • Production-grade integration: Embedded in LMCache with Redis backends, chunk hashing, pipelined retrieval, and debug wrappers.
  • Multi-model support: Pre-configured for Longchat-7B, Mistral-7B (with 3 quality levels), and LongAlpaca-70B.

Gaps and Limitations

  • Model coverage: from_model_name() only supports a fixed set of models. Adding new architectures requires manual calibration of layer group boundaries and bin counts.
  • No automated calibration: The paper discusses finding optimal bin configurations, but the code hardcodes them. No automated search or profiling tool is included.
  • Pickle-based serialization: The to_bytes() method uses Python pickle, which has known security and portability issues. A format like protobuf or flatbuffers would be more robust for production use.
  • Hardcoded sub-chunk size: CACHEGEN_GPU_MAX_TOKENS_PER_CHUNK = 256 is a fixed constant; the paper discusses this as a tunable parameter but the code does not expose it.
  • No streaming decode demo: While the pipelined backend exists, there is no end-to-end script demonstrating real-time adaptive streaming over a network link.

9 Technical Summary

Core Contribution

CacheGen demonstrates that KV cache tensors are not arbitrary floating-point data — they have exploitable structure. By building per-channel CDFs from quantized attention states and applying GPU-accelerated arithmetic coding, the system achieves compression ratios that are 3–4x better than generic compressors while maintaining near-lossless quality.

The adaptive streaming mechanism adds a practical networking dimension: by pre-encoding chunks at multiple quality levels and selecting at serving time based on available bandwidth, CacheGen ensures the time-to-first-token SLO is met even under variable network conditions — a design that directly mirrors video ABR streaming but applied to LLM attention state.
