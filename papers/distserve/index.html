DistServe: Disaggregating Prefill and Decoding for Goodput-optimized LLM Serving

Yinmin Zhong, Shengyu Liu, Junda Chen, Jianbo Hu, Yibo Zhu, Xuanzhe Liu, Xin Jin, Hao Zhang

OSDI '24 arXiv:2401.09670 Foundation Paper

The paper that formalized prefill-decode disaggregation. Virtually every production LLM serving stack adopted this pattern by 2025.

1 Paper Overview

The Problem: Prefill-Decode Interference

LLM inference consists of two computationally distinct phases. The prefill phase processes the entire input prompt in a single forward pass -- it is compute-bound, latency-sensitive, and produces all key-value (KV) cache entries. The decoding phase generates tokens one at a time autoregressively -- it is memory-bandwidth-bound and throughput-sensitive. Existing systems colocate both phases on the same GPUs, creating strong interference: long prefill batches stall decoding tokens, spiking per-token latency; large decode batches delay incoming prefill requests.

The Key Insight: Disaggregation

DistServe proposes to disaggregate prefill and decoding onto separate GPU pools, connected by KV-cache migration over high-bandwidth interconnects. This eliminates interference and, critically, enables independent resource allocation and parallelism strategies for each phase. Prefill workers can use aggressive tensor parallelism to minimize TTFT (time to first token), while decode workers can maximize batch size for throughput.

Key Contributions

Goodput Formalization

First paper to define goodput -- the maximum request rate a system can sustain while meeting both TTFT and TPOT SLO attainment targets -- as the primary optimization metric for LLM serving.

Prefill-Decode Disaggregation

Formalizes the disaggregation approach with separate GPU pools, a bridge queue for handoff, and efficient CUDA IPC-based KV-cache migration between pools.

Placement Algorithm

Co-optimizes the placement of prefill and decode instances, choosing per-phase parallelism (TP, PP) and replica counts to maximize goodput under a fixed GPU budget.

SwiftTransformer Backend

High-performance C++ inference library supporting FlashAttention, PagedAttention, Continuous Batching, and model/pipeline parallelism as the execution data-path.

Impact Note: DistServe's disaggregation paradigm became the de facto architecture for production LLM serving. By 2025, systems such as Mooncake (Moonshot AI), Splitwise (Microsoft), and TetriInfer all adopted prefill-decode separation. The concept is now integrated into vLLM's disaggregated prefill mode and numerous proprietary stacks.

Parallelism Configuration

DistServe introduces DisaggParallelConfig which holds two independent ParallelConfig instances -- one for prefill, one for decode. Each can independently set tensor parallelism (TP) and pipeline parallelism (PP), enabling configurations such as:

Prefill: TP=4, PP=1

Maximizes single-request prefill latency by spreading attention heads across 4 GPUs. Low batch size is acceptable since prefill is compute-bound.

Decode: TP=1, PP=1

Uses a single GPU per decode replica to maximize the number of replicas and total batch capacity. Memory-bandwidth bound decode benefits from large batches, not more parallelism.

Key Implementation: LLMEngine Orchestrator

The central LLMEngine class in engine.py implements the disaggregation pattern. It creates two independent stage engines and connects them through an asyncio bridge queue.

class LLMEngine:
    def __init__(self, model_config, disagg_parallel_config, ...):
        self.bridge_queue = asyncio.Queue()  # Bridge between stages

        # Two independent stage engines with separate GPU pools
        self.context_engine = ContextStageLLMEngine(
            self.bridge_queue,
            disagg_parallel_config.context,  # e.g. TP=4, PP=1
            ...
        )
        self.decoding_engine = DecodingStageLLMEngine(
            self.bridge_queue,
            disagg_parallel_config.decoding,  # e.g. TP=1, PP=1
            ...
        )

    async def initialize(self):
        # Initialize both engines in parallel
        await asyncio.gather(
            self.context_engine.initialize(),
            self.decoding_engine.initialize()
        )
        # Register KV cache memory handles for GPU-to-GPU migration
        await self.decoding_engine.register_kvcache_mem_handles(
            self.context_engine.parallel_config,
            self.context_engine.kv_cache_mem_handles
        )

Key Implementation: KV-Cache Migration

The MigratingRequest carries block indices and the source parallel config. Decode workers use CUDA IPC memory handles to perform GPU-to-GPU block transfer without staging through the host.

# In DecodingStageLLMEngine._migrate_blocks()
async def _migrate_blocks(self, migrating_req):
    # 1. Allocate blocks on decode-side GPU
    self.block_manager.allocate_blocks(migrating_req.req)
    target_blocks = self.block_manager.get_block_table(migrating_req.req.request_id)

    # 2. Issue GPU-to-GPU transfer via CUDA IPC
    await asyncio.wait(self._remote_call_all_workers_async(
        "migrate_blocks",
        migrating_req.block_indexes,          # source (prefill GPU)
        migrating_req.context_parallel_config, # source TP/PP config
        target_blocks                          # target (decode GPU)
    ))

    # 3. Free blocks on prefill side
    self.clear_migrated_blocks_callback(migrating_req)

Key Implementation: Placement Group Strategy

The engine places prefill and decode workers on the same physical node using Ray placement groups with STRICT_PACK strategy. This ensures co-located layers share PCIe/NVLink bandwidth for efficient KV transfer.

# In LLMEngine._init_placement_groups()
# Forces same layer of context & decoding to be on same node
# This simplifies KV cache migration (P2P over NVLink)
for i in range(num_placement_groups):
    placement_group = ray.util.placement_group(
        [{"GPU": 1}] * workers_per_placement_group,
        strategy="STRICT_PACK",   # All on same node
    )

SimDistServe: Discrete-Event Simulator

The repository includes simdistserve/, a SimPy-based discrete-event simulator used to search for optimal configurations. It models prefill and decode workers, estimates execution times from profiling data, and performs bisection search to find the configuration that maximizes goodput.

result = simulate_bisect_search(

4 Core Mechanisms

Goodput Optimization

DistServe formalizes goodput as the maximum request rate the system can serve while meeting both SLO targets simultaneously:

Goodput = max rate R such that P(TTFT < Tprefill) ≥ p1 AND P(TPOT < Tdecode) ≥ p2

where Tprefill, Tdecode are latency targets and p1, p2 are attainment percentiles (e.g. P90)

Scheduling Design

Context Stage: FCFS

The ContextStageFCFSScheduler uses first-come-first-serve with three limits: max batch size, max tokens per batch, and available GPU blocks (including blocks reserved for unaccepted requests still awaiting migration).

Decode Stage: FCFS + Swap

The DecodingStageFCFSScheduler manages three queues -- unaccepted, waiting, and active. It supports swap-in/out to CPU when GPU blocks are exhausted, using async CUDA streams with event-based synchronization.

Block Manager and Swap Mechanism

The BlockManager tracks block locations (GPU vs CPU) and implements a two-list scheme (free_blocks_list + swapping_blocks_list) to handle async swap operations. CUDA events ensure blocks are not reused before swap operations complete.

Block Swap Synchronization Protocol
Swap-Out Issued GPU blocks to CPU
➔
swapping_gpu_blocks In-flight, CUDA event tracked
➔
Event.synchronize() When allocation needs blocks
➔
free_gpu_blocks Safe to reuse

Backpressure and Flow Control

DistServe implements flow control between stages. When the decode pool is overwhelmed, the context engine's scheduler accounts for unaccepted_queue size and num_on_fly_request_block to throttle prefill scheduling, preventing KV cache exhaustion. The decode scheduler uses a waiting_block_prop_threshold (default 5%) to decide when to accept migrating requests from the bridge queue.

Key Findings

Throughput under Tight SLOs

On OPT-13B with ShareGPT traces, DistServe achieves 7.4x higher request throughput compared to vLLM while meeting both TTFT (200ms P90) and TPOT (100ms P90) targets. The disaggregation eliminates queuing delay caused by long prefills.

SLO Tightening

At the same throughput level, DistServe can meet 12.6x tighter SLO targets. For example, where vLLM requires 2000ms TTFT to meet P90, DistServe achieves P90 at 160ms.

Large Model Scaling

On OPT-175B (the largest model tested), DistServe achieves 4.48x goodput improvement by independently selecting TP=8 for prefill and TP=4 for decode, optimizing each phase's bottleneck.

Migration Overhead

KV-cache migration latency is bounded: approximately 2-5ms per request on NVLink, representing less than 5% of total end-to-end latency. The overhead is negligible compared to prefill compute.

6 Limitations for Agentic Workloads

Critical Limitation: DistServe was designed for single-turn stateless request serving. Several architectural decisions make it poorly suited for the multi-turn, tool-calling, and iterative workloads that characterize agentic AI systems.

Limitation Analysis

1. No Multi-Turn KV Cache Reuse

Every request undergoes a full prefill-migrate-decode lifecycle. When an agent makes a follow-up tool call that shares most of the conversation prefix, DistServe re-prefills the entire context from scratch and migrates the full KV cache again. There is no mechanism to retain KV state across turns.

Impact: Agentic workloads with 10+ turn conversations suffer O(n) redundant prefill compute and KV migration per turn. Systems like SGLang's RadixAttention and Mooncake's KVCache Pool solve this by prefix-aware caching.

2. FCFS Scheduling -- No Priority Support

Both schedulers use strict FCFS ordering. The Request class has a priority field, but no scheduler uses it. Agentic workloads require priority-based scheduling -- tool-call completions that unblock an agent chain should preempt new requests.

Impact: An agent waiting for a function-call result may be stuck behind a queue of unrelated batch requests, increasing end-to-end task latency by orders of magnitude.

3. No Prefix/Prompt Caching

DistServe's block manager treats every request's KV cache as independent. There is no shared prefix tree, no deduplication of common system prompts, and no RadixAttention-style prefix matching. Agentic frameworks typically share long system prompts across all calls.

Impact: A 4K-token system prompt repeated across 100 agent calls wastes 100x the prefill compute and GPU memory that prefix caching would eliminate.

4. Unidirectional KV Migration

KV cache flows only from prefill GPUs to decode GPUs. There is no mechanism to move KV state back for continued prefill (e.g., appending tool-call results to an existing context) or to share KV state between decode replicas.

Impact: Speculative decoding, prompt continuation, and multi-agent coordination patterns all require bidirectional or shared KV cache access.

5. No Request Abortion

The abort_request() method in single_stage_engine.py logs a warning and returns without action. Agentic systems frequently need to cancel in-flight generations when a tool call provides a better answer or a timeout occurs.

Impact: Resources remain locked to cancelled requests until they naturally complete, wasting GPU compute and memory blocks.

6. Static GPU Pool Sizing

The prefill/decode GPU ratio is fixed at launch time via DisaggParallelConfig. Agentic workloads are bursty -- tool calls may generate a spike of short prefills followed by extended decode. Dynamic rebalancing of GPU pools is not supported.

7 How to Reproduce

Prerequisites

  • Minimum 2 NVIDIA GPUs (A100 recommended, V100 supported)
  • CUDA 11.8+ with NVLink for optimal KV migration performance
  • Conda for environment management
  • At least 80GB GPU memory per GPU for OPT-13B; 8 GPUs for OPT-175B

Step 1: Build and Install

# Clone the repository
git clone https://github.com/LLMServe/DistServe.git && cd DistServe

# Create conda environment
conda env create -f environment.yml && conda activate distserve

# Build the SwiftTransformer C++ backend
git clone https://github.com/LLMServe/SwiftTransformer.git
cd SwiftTransformer && git submodule update --init --recursive
cmake -B build && cmake --build build -j$(nproc)
cd ..

# Install DistServe in development mode
pip install -e .

Step 2: Run Offline Inference (Smoke Test)

# Requires at least 2 GPUs -- one prefill, one decode
python examples/offline.py --model meta-llama/Llama-2-7b-hf

The offline example configures TP=1/PP=1 for both stages and generates 64 tokens per prompt with temperature=0.8 sampling.

Step 3: Run the Simulator (No GPUs Required)

# Set the dataset environment variable
export DATASET=/path/to/datasets

# Find optimal configuration for OPT-13B on 8 GPUs
python simdistserve/simulate.py \
    --backend distserve \
    --model-type opt_13b \
    --num-node 1 \
    --ngpu-per-node 8 \
    --workload sharegpt \
    --prefill-target 200 \
    --decode-target 100 \
    --N 300

Step 4: Full Evaluation (Paper Reproduction)

# Build the evaluation Docker image
cd evaluation

# Clone baseline systems
git clone https://github.com/interestingLSY/vllm.git --branch distserve-baseline-vllm
git clone https://github.com/interestingLSY/DeepSpeed-MII.git --branch distserve-baseline

# Build Docker
docker build -f Dockerfile.evaluation -t distserve-evaluation:latest ../../

# Run the benchmark suite
# See evaluation/docs/README-AE.md for full instructions

Step 5: Launch Online API Server

# Start the DistServe API server (OpenAI-compatible)
python -m distserve.api_server.distserve_api_server \
    --model meta-llama/Llama-2-13b-hf \
    --context-tensor-parallel-size 2 \
    --context-pipeline-parallel-size 1 \
    --decoding-tensor-parallel-size 1 \
    --decoding-pipeline-parallel-size 1 \
    --block-size 16 \
    --gpu-memory-utilization 0.9

# Client example
python examples/online.py
