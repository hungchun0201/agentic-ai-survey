Overview

The Problem: Synchronous Disaggregation Bottleneck

Current disaggregated LLM serving systems (DistServe, Splitwise) separate prefill and decode into different GPU pools. However, the KV cache transfer between pools is synchronous: decode cannot begin until the entire KV cache from prefill has been received. For long prompts and large models, this KV transfer time becomes a dominant bottleneck—the decode GPU sits idle, wasting resources and inflating both TTFT and end-to-end latency.

WindServe: Asynchronous Stream-Based Execution

WindServe introduces a stream-based disaggregation paradigm that breaks the synchronous KV transfer barrier. Instead of waiting for the complete KV cache, decode begins as soon as the first layer's KV data arrives. Prefill and decode execute concurrently: while prefill generates KV for layer L+k, decode processes layer L using already-received KV data. The stream abstraction pipelines KV transfer with both prefill computation and decode computation across the network.

KV Transfer: The Hidden Cost

To appreciate why stream-based KV transfer matters, it is useful to quantify the KV cache size and resulting transfer time for realistic deployments.

KV Cache Size Formula

For a transformer with N layers, H attention heads, D head dimension, and a prompt of S tokens:

# KV cache size (bytes) KV_size = 2 * N * H * D * S * dtype_bytes # Example: Llama-3 70B, 128K tokens, FP16 N = 80 # transformer layers H = 8 # KV heads (GQA) D = 128 # head dimension S = 131072 # 128K tokens dtype_bytes = 2 # FP16 KV_size = 2 * 80 * 8 * 128 * 131072 * 2 # ≈ 42 GB # Transfer time over 100 Gbps InfiniBand: transfer_time = 42e9 * 8 / 100e9 # ≈ 3.36 seconds (!)

Even with 400 Gbps HDR InfiniBand, transferring 42 GB of KV takes ~840ms for a 70B model at 128K context. Stream-based transfer converts this stall into a near-zero wait by overlapping transfer with decode.

Per-Layer KV Size

For Llama-3 70B at 128K context, each layer's KV is ~525 MB, transferring in ~10ms over 400 Gbps. WindServe waits 10ms (one layer) vs 840ms (all layers) — an 84x ratio that drives the TTFT improvement.

GQA / MQA Impact

GQA reduces KV size 8x, shrinking transfer time proportionally. Even with GQA at 128K context, KV is several GB — large enough for stream-based transfer to yield significant TTFT gains.

CUDA Stream Architecture on the Decode Node

Three Concurrent CUDA Streams

The decode node uses three CUDA streams executing concurrently, coordinated by CUDA events:

DMA / Transfer Stream

Issues RDMA receive operations to pull incoming KV blocks into staging. Each completed receive triggers a CUDA event for the corresponding layer.

Compute Stream

Runs decode attention kernels. Each kernel is gated by a stream-wait on the layer's CUDA event, executing only after that layer's KV has fully arrived.

Memory Management Stream

Copies received KV from staging to the permanent KV page pool. Runs concurrently with compute, keeping staging buffer available for new incoming blocks.

CUDA Events as Gates

DMA stream records an event after each layer's KV arrives. Compute stream issues a stream-wait on that event. This is zero-CPU-overhead synchronization — the GPU stalls the compute stream internally until the event fires.

Key Mechanisms

Stream-Based KV Transfer

KV cache is transferred layer-by-layer as a stream rather than as a monolithic block. Each layer's KV is sent immediately after prefill computes it. The decode GPU begins processing layer 1 as soon as its KV arrives, overlapping with the ongoing transfer of subsequent layers. This hides the majority of transfer latency.

Async Concurrent Execution

Three operations run concurrently: (1) prefill computing layer L+k, (2) network transferring layer L+j KV, (3) decode processing layer L. CUDA streams and events coordinate data dependencies without blocking. The effective latency is dominated by the slowest of the three operations, not their sum.

Adaptive Stream Scheduling

The system dynamically adjusts the streaming granularity (single layer vs. multi-layer chunks) based on network bandwidth, GPU compute speed, and model architecture. For bandwidth-limited scenarios, larger chunks reduce header overhead; for compute-limited cases, finer granularity enables better overlap.

Pipeline Bubble Elimination

In standard disaggregated serving, the decode GPU is completely idle during KV transfer (a "bubble"). WindServe eliminates this bubble by starting decode computation as soon as possible. For a model with N layers, the bubble is reduced from N layers of transfer time to approximately max(transfer_1_layer, decode_1_layer).

// WindServe async pipeline pseudocode // Prefill GPU: for layer in range(N): kv = prefill_layer(layer, input_tokens) async_send(kv, decode_gpu, stream=transfer_stream) # non-blocking // Decode GPU (concurrent): for layer in range(N): wait_event(kv_ready[layer]) # wait for this layer's KV only output = decode_layer(layer, kv_cache[layer]) # Next layer's KV may already be arriving while we compute # The GPU compute stream is gated per-layer; no CPU spin-wait needed # TTFT = T_prefill + T_transfer_layer[0] + T_decode_first_step # vs traditional: T_prefill + T_transfer_all_layers + T_decode_first_step

For a chatbot serving 128K-context requests: if baseline TTFT is 3 seconds, WindServe reduces it to ~700ms — crossing the perceptual threshold between "responsive" and "lagging".

What 1.5x TPOT P99 Means in Practice

Significance

Paradigm Shift in Disaggregation

WindServe transforms disaggregated serving from a two-phase process (prefill-then-transfer-then-decode) into a pipelined stream. This fundamentally changes the latency model by overlapping all three operations.

ISCA Publication

Accepted at ISCA 2025, a premier computer architecture venue. This signals that LLM serving is becoming a first-class concern in architecture research, bridging systems and hardware design.

TTFT-Critical Applications

The 4.28x TTFT improvement is transformative for interactive applications where time-to-first-token directly impacts user experience. Chatbots, code completion, and real-time translation all benefit from faster initial response.

Complementary to Other Optimizations

Stream-based execution is orthogonal to KV cache compression, model parallelism, and speculative decoding. WindServe's architectural approach can be combined with these techniques for compound benefits.

Future Work & Open Questions

Multi-Node Tensor Parallelism

When both prefill and decode are sharded across multiple GPUs using tensor parallelism, stream-based KV transfer must be coordinated across all ranks simultaneously. Future work may explore hierarchical streaming where each rank streams its shard independently, then a final all-gather reconstructs the complete KV for the decode ranks.

Speculative Decoding Integration

Combining WindServe's stream-based pipeline with speculative decoding is a promising direction. The draft model could begin generating token candidates using partially received KV, with verification using the full KV once transfer is complete.

Heterogeneous Hardware Clusters

Datacenter clusters often mix GPU generations. The streaming granularity and scheduling policy may need to adapt to mismatched compute-to-bandwidth ratios, ensuring the slower node does not become a persistent bottleneck within the stream pipeline.

KV Compression + Streaming

Applying quantization or sparse attention to reduce KV size before streaming could further reduce transfer time per block, increasing the degree of overlap. This compounds WindServe's gains with compression techniques.

Why TPOT also improves

The 1.5x TPOT P99 improvement arises because faster TTFT means fewer requests are concurrently in-flight at the decode node, reducing queueing pressure. Shorter queues translate to lower tail latency for per-token steps.

System Design Details

Realizing stream-based KV transfer requires changes to the memory management, scheduling, and networking layers of a standard LLM serving stack.

Block-Level Streaming Protocol

KV Block Granularity

A KV block in WindServe corresponds to one transformer layer's key and value tensors for all tokens. After prefill computes layer L, the resulting KV tensors are immediately enqueued for DMA transfer.

Ping-Pong Buffering

The decode node maintains a double-buffered staging area: one buffer receives incoming KV via DMA while the other holds the block being read by the decode kernel. Buffers swap roles when DMA completes.

Async Scheduling on the Decode Node

Layer-Gated Execution

The decode scheduler maintains a per-layer readiness flag set by a CUDA event that fires when DMA completes. The decode kernel is enqueued onto a compute stream but will not execute until the event is signaled.

// Decode node: per-layer event-gated scheduling for layer in range(N_layers): # DMA thread sets this event when block[layer] transfer completes event = cuda_event[layer] # Compute stream waits for the event before reading KV compute_stream.wait_event(event) # Kernel is pre-enqueued; actual execution deferred until event fires compute_stream.enqueue(decode_kernel, layer, kv_buffer[layer]) # Meanwhile, DMA stream continues bringing in layer+1, layer+2, ...
Network Transport Layer

RDMA-Based Block Transfer

WindServe uses RDMA to move KV blocks from prefill GPU HBM to decode staging buffer without CPU involvement. RDMA write operations are issued per layer, with completion notifications triggering the CUDA event on decode GPU.

Streaming Granularity Tuning

The system can stream multiple layers at a time as a chunk rather than single layers. Coarser chunks reduce RDMA initiation overhead; optimal chunk size depends on RDMA setup latency vs per-layer KV size.

Memory Management

KV Page Allocation Across Nodes

WindServe changes KV allocation: prefill node writes into a transmit ring buffer and streams each layer; decode node permanently places received blocks into its own KV page pool.

Phase Prefill Node Memory Decode Node Memory
During prefill Writes KV to TX ring buffer Pre-allocates RX staging (ping-pong)
During streaming Frees TX slots as RDMA completes Moves blocks from staging to KV pool
After transfer TX ring fully drained and reusable Full KV resident in decode KV pool
Implementation note: The ping-pong buffering and per-layer CUDA events add small overhead per request. WindServe's gains are most significant for long-context requests where KV transfer time dominates.

Throughput vs Latency Trade-off

Latency-Optimal Operating Point

WindServe's primary benefit is latency reduction. At very high request rates where compute is saturated, the streaming benefit becomes marginal since queueing dominates.

Long-Context Scaling

The relative benefit of WindServe grows with prompt length. For very long prompts, KV transfer becomes the dominant TTFT component and WindServe's gain is most pronounced.

vs. DistServe

DistServe established the disaggregation paradigm. WindServe accepts this model but replaces the synchronous transfer protocol with a streaming one, eliminating DistServe's key latency bottleneck.

vs. Chunked Prefill

Chunked prefill interleaves prefill chunks with decode on the same GPU, trading prefill throughput. WindServe takes the opposite approach: disaggregate to separate GPUs and use streaming to avoid the transfer stall.

vs. Prefill-Decode Colocation

Colocated serving avoids KV transfer but suffers from memory fragmentation and batch interference. WindServe preserves disaggregation's throughput benefits while closing the latency gap.

vs. Sarathi-Serve

Sarathi-Serve improves TPOT via chunked prefill but does not address TTFT for long-context requests. WindServe achieves 4.28x TTFT improvement where Sarathi-Serve yields more modest gains.

Key Takeaways

The single insight that drives everything

KV blocks are usable as soon as their layer is computed. There is no inter-layer dependency from the decode node's perspective — only intra-layer (all tokens' KV for one layer must arrive together). WindServe exploits this by issuing per-layer RDMA writes and per-layer decode kernel dispatches.

1. Transfer Stall is Eliminated, Not Hidden

Prior systems hid KV transfer by batching requests. WindServe eliminates the stall at the single-request level — effective even at low concurrency where batching-based hiding is ineffective.

2. ISCA Venue Signals Hardware-Level Impact

Publication at ISCA 2025 signals architecture-level contributions involving RDMA, CUDA streams, HBM bandwidth, and cross-chip data paths — not just software scheduling. Likely includes hardware characterization experiments.

3. The Bottleneck Shifts with Context Length

WindServe's 4.28x improvement is measured in the long-context regime where KV transfer dominates TTFT. For short prompts, prefill compute dominates and WindServe's gain is smaller.

4. Practical Deployment Considerations

Deploying WindServe requires RDMA-capable networking (InfiniBand or RoCE), GPUDirect support, and runtime modifications for per-layer event tracking.

One-line summary

WindServe replaces the synchronous "wait for full KV transfer" barrier with a per-layer streaming protocol, allowing decode to begin on the first KV block and run concurrently with ongoing transfer and prefill — yielding 4.28x TTFT and 1.5x TPOT P99 at ISCA 2025.

The core insight is that KV blocks are independently usable as soon as their layer is computed. By exploiting this at the hardware level — via RDMA per-layer writes and CUDA event gates — WindServe turns a sequential three-step process into a continuously pipelined stream.
