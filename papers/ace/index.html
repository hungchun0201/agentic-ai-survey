ACE — Agentic Context Engineering

Evolving Contexts for Self-Improving Language Models

ICLR 2026 arXiv:2510.04618 Context Engineering Agent Memory Self-Improving LLM

Qizheng Zhang1, Changran Hu1, Shubhangi Upasani3, Boyuan Ma3, Fenglu Hong3, Vamsidhar Kamanuru3, Jay Rainton3, Chen Wu3, Mengmeng Ji3, Hanchen Li3, Urmish Thakker3, James Zou1, Kunle Olukotun1

1Stanford University   2UC Berkeley   3SambaNova Systems

1. Overview & Problem

Prompt optimization and context adaptation aim to improve LLM performance by engineering better contexts (system prompts, agent memory, playbooks). However, prior approaches suffer from two critical failure modes:

Brevity Bias

Optimizers like GEPA prioritize concise, generalized instructions over comprehensive domain-specific heuristics. This drops critical tool-use guidelines and failure-mode patterns that matter in practice.

Context Collapse

When an LLM fully rewrites accumulated context at each step, catastrophic information loss can occur. The paper documents a case where context collapsed from 18,282 tokens (66.7% accuracy) to just 122 tokens (57.1% accuracy) in a single step.

Core Contributions

  1. ACE Framework — a three-role agentic architecture (Generator, Reflector, Curator) that treats contexts as evolving playbooks, accumulating and refining strategies through structured, incremental updates.
  2. Brevity Bias & Context Collapse Prevention — deterministic, non-LLM merging of delta updates prevents the information loss that plagues end-to-end rewriting approaches.
  3. Offline + Online Adaptation — optimizes both system prompts (offline) and agent memory (online/test-time), with optional offline warmup before online adaptation.
  4. Supervision-free — adapts effectively without labeled data by leveraging natural execution feedback (e.g., code execution success/failure).
Key Insight: ACE achieves +10.6% on agent benchmarks and +8.6% on finance tasks while reducing adaptation latency by 86.9% compared to baselines — by treating contexts as structured, incrementally-updated playbooks rather than monolithic prompts.

2. Architecture

ACE uses a three-role agentic architecture. Each role is implemented as a separate LLM call with specialized prompts. The central data structure is a playbook — a structured collection of bullet entries with metadata.

1. Generator

  • Receives task + current playbook as context
  • Produces reasoning trajectory (solution attempt)
  • Highlights which bullets were helpful or misleading
↓

2. Reflector

  • Critiques Generator traces to extract concrete lessons
  • Iterative refinement across multiple rounds (up to 5)
  • Works with or without ground-truth labels
↓

3. Curator

  • Synthesizes lessons into compact delta entries (candidate bullets)
  • Deterministic, non-LLM merge into existing playbook
  • Prevents context collapse — no monolithic rewriting
↓

Updated Playbook

  • Structured bullets with IDs and helpful/harmful counters
  • Grows incrementally — preserves all accumulated knowledge
  • Token budget management via dedup + pruning

2.1 Offline vs Online Modes

Offline Adaptation

System prompt optimization on a training split. Multi-epoch refinement (up to 5 epochs). Produces a best playbook selected by validation performance. Evaluated with pass@1 accuracy.

Online Adaptation

Test-time agent memory that adapts during inference. No separate training split needed. Optional "offline warmup" for initialization. Playbook updated every N samples (default: 15).

4. Core Mechanisms

4.1 Structured Bullet Format

Each bullet in the playbook is a structured entry with a unique ID, counters tracking its usefulness, and the actual strategy content:

[str-00001] helpful=5 harmful=0 :: Always verify data types before processing
[api-00042] helpful=3 harmful=2 :: Use batch API calls when possible to reduce latency
[err-00007] helpful=8 harmful=0 :: Check for null returns from get_user() before accessing .name
Design Choice: The ID system [section_slug-NNNNN] enables the Reflector to reference specific bullets when reporting which ones were helpful or harmful during task solving. Counters provide a lightweight signal for pruning low-value entries.

4.2 Delta Update Process

Unlike monolithic rewriting approaches (GEPA, Dynamic Cheatsheet), ACE uses a deterministic delta-merge strategy that prevents context collapse:

Step 1: Generator Feedback

Generator solves a task and highlights which bullets were helpful (✓) or harmful (✗). This explicit attribution enables the Reflector to target specific areas.

↓

Step 2: Iterative Reflection

Reflector analyzes the trace across up to 5 rounds, extracting increasingly nuanced lessons. Each round produces insights that feed into the next.

↓

Step 3: Curator Delta Synthesis

Curator produces compact delta entries — new bullets or modifications to existing ones. The LLM is only used for generating deltas, not for merging.

↓

Step 4: Deterministic Merge

New IDs are appended to the playbook. Existing bullets are updated in-place (counter increments). No LLM involved — eliminates rewriting variance entirely.

4.3 Grow-and-Refine Mechanism

ACE balances steady context expansion with redundancy control through two complementary mechanisms:

De-duplication

Compares bullets via semantic embeddings using BulletpointAnalyzer. Configurable similarity threshold (default: 0.9). Similar bullets are merged, preserving the higher-counter version.

Pruning

Can be done proactively (after each delta) or lazily (only when the playbook exceeds token budget). Bullets with high harmful-to-helpful ratio are pruned first. Default token budget: 80,000 tokens.

4.4 Curator Implementation

The Curator module in ace/core/curator.py is the critical component that separates ACE from prior work. It handles delta generation via LLM and deterministic merging via rule-based logic:

class Curator:
    def curate(self, reflector_output, current_playbook):
        # Step 1: LLM generates delta entries
        delta = self.llm.generate(
            prompt=self.curator_prompt,
            context={
                "reflector_lessons": reflector_output,
                "current_playbook": current_playbook
            }
        )

        # Step 2: Parse delta into structured bullets
        new_bullets = parse_delta(delta)

        # Step 3: Deterministic merge (NO LLM involved)
        for bullet in new_bullets:
            if bullet.id in current_playbook:
                # Update counters in-place
                current_playbook[bullet.id].helpful += bullet.helpful
                current_playbook[bullet.id].harmful += bullet.harmful
            else:
                # Append new bullet
                current_playbook.append(bullet)

        # Step 4: Optional dedup + pruning
        if self.dedup_enabled:
            current_playbook = self.analyzer.deduplicate(current_playbook)
        if token_count(current_playbook) > self.budget:
            current_playbook = prune_lowest(current_playbook)

        return current_playbook

4.5 Reflector Multi-round Refinement

The Reflector module in ace/core/reflector.py is the most impactful single component (per ablation results). It supports iterative refinement across up to max_num_rounds=3 rounds for incorrect answers:

class Reflector:
    def reflect(self, generator_trace, is_correct, playbook):
        lessons = []
        for round_idx in range(self.max_num_rounds):
            # Each round critiques the trace + prior lessons
            new_lessons = self.llm.generate(
                prompt=self.reflector_prompt,
                context={
                    "trace": generator_trace,
                    "prior_lessons": lessons,
                    "is_correct": is_correct,
                    "playbook": playbook
                }
            )
            lessons.extend(new_lessons)

            # Early stop if correct or no new insights
            if is_correct or not new_lessons:
                break
        return lessons
Supervision-free Mode: When ground-truth labels are unavailable, ACE leverages natural execution feedback. For agent tasks (e.g., AppWorld), code execution success/failure signals serve as implicit labels. This still achieves +14.8% over baseline — only 2.2% below the supervised version.

5.3 Finance Benchmarks

ACE demonstrates consistent gains on domain-specific tasks beyond agent benchmarks:

  • Offline: ACE surpasses ICL, MIPROv2, and GEPA by an average of +10.9%
  • Online: ACE exceeds Dynamic Cheatsheet by an average of +6.2%
  • Overall: Average improvement of +8.6% across FiNER and Formula benchmarks

5.4 Ablation Study

Ablation experiments isolate the contribution of each ACE component:

Reflector + Multi-round

Most impactful component. Removing the Reflector and multi-epoch significantly degrades performance. The iterative refinement is the key differentiator.

Offline Warmup

Initializing the playbook through offline adaptation before online adaptation provides additional gains.

Cross-LLM Generalization

Tested with GPT-OSS-120B, GPT-5.1, and Llama-3.3-70B-Instruct. Consistent gains observed across all models with no algorithm or prompt changes.

6.1 Strengths & Limitations

Strengths

  • Clean modular architecture — each role is a separate, testable class
  • Well-documented extension guide (EXTENDING_ACE.md, 15.8KB)
  • Multiple API provider support (SambaNova, Together, OpenAI, CommonStack)
  • Deterministic merge eliminates LLM rewriting variance
  • Comprehensive prompt templates in ace/prompts/

Limitations

  • Default model is DeepSeek-V3.1 for all roles — large compute requirement
  • AppWorld submodule requires separate setup and proprietary API
  • Playbook token budget (80K) may not fit smaller context window models
  • No built-in visualization of playbook evolution over time

7. How to Reproduce

7.1 Setup

# Clone the repo
git clone https://github.com/ace-agent/ace.git
cd ace

# Install dependencies
pip install -e .

# Set up API keys (choose provider)
export SAMBANOVA_API_KEY="your-key"
# or
export OPENAI_API_KEY="your-key"

7.2 Run Offline Adaptation

# Example: Finance benchmark (FiNER)
python eval/finance/run.py \
    --mode offline \
    --eval_steps 100 \
    --curator_frequency 1 \
    --playbook_token_budget 80000 \
    --test_workers 20

7.3 Run Online Adaptation

# Online with optional offline warmup
python eval/finance/run.py \
    --mode online \
    --warmup_playbook path/to/offline_playbook.json \
    --max_num_rounds 3
Hardware Requirements: The default configuration uses DeepSeek-V3.1-671B via API. No local GPU required for the ACE framework itself — all LLM calls are API-based. However, the AppWorld benchmark requires running a local AppWorld server.

7.4 Key Configuration

Parameter Default Description
max_num_rounds 3 Max reflection rounds for incorrect answers
