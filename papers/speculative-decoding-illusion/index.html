Speculative Decoding: Performance or Illusion?

First systematic study of speculative decoding on a production-grade inference engine (vLLM), revealing that verification dominates execution and large gaps remain to theoretical upper bounds

arXiv:2601.11580 arXiv 2026 Speculative Decoding vLLM Benchmarking

Xiaoxuan Liu, Jiaxiang Yu, Jongseok Park, Ion Stoica, Alvin Cheung

UC Berkeley

1. Overview & Key Findings

Speculative decoding (SD) accelerates LLM inference by drafting multiple candidate tokens in parallel and verifying them against the target model in a single forward pass. Despite widespread adoption, no prior study has systematically evaluated SD variants on a production-grade engine under realistic conditions. This paper fills that gap by benchmarking five SD methods on vLLM v0.10.1.1 across three model families, six datasets, and batch sizes 1–512.

1. Verification Dominates

Target model verification accounts for 42–95% of total SD execution time. Drafting overhead (especially for training-free n-gram) is negligible — the bottleneck is always the verify step.

2. Acceptance Length Varies Wildly

Acceptance length fluctuates substantially across token positions and datasets — from 1.28 median (GSM8K n-gram) to 6.57 median (InstructCoder n-gram). Fixed proposal lengths leave significant headroom.

3. Gap to Theoretical Max

An oracle that knows the exact acceptance length at each position would achieve up to 2.75× speedup vs. the observed 2.1× for n-gram on InstructCoder (batch=1). Adaptive method combination could push speedup further by up to 2.2× beyond fixed strategies.

Key Insight: The absolute drafting time is irrelevant; only its ratio relative to the target model forward pass matters. This means training-free methods (n-gram) can match or outperform learned drafters when prompt-output overlap is high.

2. Background: SD Variants

The paper evaluates five speculative decoding methods spanning the full spectrum of draft generation approaches — from training-free lookup to co-trained multi-token heads.

n-gram (Training-Free)

  • Matches recent output tokens against the prompt to find n-gram overlaps
  • Zero memory overhead, zero training cost
  • Excels on code-editing tasks with high prompt-output repetition
↓

EAGLE

  • Fine-tuned auxiliary head on top of target model's hidden states
  • 3.1–4.9% static memory overhead

EAGLE-3

  • Improved draft-model-free variant with better acceptance rates
  • Best performer on smaller models (8B)
↓

Draft-Model

  • Separate smaller model (e.g. Llama3.2-1B drafts for Llama3-70B)
  • Excels on larger target models (70B+)
  • Up to 1.77× per-token KV cache overhead

MTP (Multi-Token Prediction)

  • Co-trained prediction heads within the target model
  • Tested on GLM-4.5-Air-106B (native MTP support)

3. End-to-End Performance

The core finding: SD speedup degrades consistently as batch size increases, because the verification step becomes increasingly memory-bandwidth-bound and amortizes less per draft token. No SD variant sustains speedup beyond 1.3× at batch size 128+.

Figure 1a (from paper): End-to-end throughput and speedup for Llama3.1-8B on GSM8K. EAGLE-3 leads at batch size 1 (1.73×) but degrades to 1.21× at batch 128. n-gram offers negligible speedup on this math-oriented dataset.
Figure 1d: InstructCoder (code editing): n-gram dominates due to high prompt-output overlap, achieving ~2.1× at batch 1.
Figure 1h: Llama3-70B on InstructCoder: draft-model method becomes competitive with larger targets, but n-gram still leads.

Memory Formulas

(1) $$\text{Static Memory} = \frac{(P_{\text{target}} + P_{\text{draft}}) \times 2}{2^{30}} \;\text{GiB}$$
(2) $$\text{KV per token} = \frac{L_h \times 2 \times n_{kv} \times d_{\text{head}} \times 2}{2^{10}} \;\text{KiB}$$

where $L_h$ = number of hidden layers, $n_{kv}$ = number of KV heads, and $d_{\text{head}}$ = head dimension.

5. Acceptance Behavior Analysis

A central contribution: the paper measures acceptance length not just as a per-request average, but at each token position within a generation. This reveals significant positional and cross-request variability.

6. Case Study: n-gram on InstructCoder

The paper investigates why n-gram achieves disproportionately high speedup on code-editing tasks. The key metric is BLEU-4 between prompt and output — a proxy for n-gram overlap.

Figure 7 (from paper): The paper shows a code editing example from InstructCoder where the model reuses class structure, variable names, and boilerplate from the prompt — exactly the kind of repetition n-gram exploits. The output preserves most of the prompt content and applies localized edits, yielding long n-gram matches.
Figure 8a: BLEU-4 score vs. n-gram speedup. When BLEU-4 > 0.6, n-gram outperforms EAGLE by up to 53% (k=3 proposals).
Figure 8e: With k=5 proposals, the gap widens: up to 100% higher speedup than EAGLE at high BLEU-4. Training-free beats trained when repetition is present.
Takeaway: For agentic workloads that involve code generation, tool-call response parsing, or structured output with repetitive schemas, n-gram speculative decoding could be a zero-cost acceleration option.

7. Theoretical Speedup Upper Bound

7.1 Speedup Formula

The expected speedup of speculative decoding under fixed proposal length $k$ and constant acceptance rate $\alpha$:

(3) $$E(\text{speedup}) = \frac{1 - \alpha^{k+1}}{(1 - \alpha)(kc + 1)}$$

where $\alpha$ = token acceptance rate, $k$ = proposed token length, and $c$ = execution-time ratio of drafting to target model forward pass.

7.2 Oracle Analysis

The paper compares fixed proposal length against an oracle that knows the exact acceptance length at each position (proposing exactly as many tokens as will be accepted).

Figure 9a: Oracle (knows exact acceptance) vs. fixed proposal length on InstructCoder. Oracle achieves ~2.75× vs. fixed n-gram's ~2.1× at batch 1.
Figure 10a: Per-position difference between EAGLE-3 and n-gram acceptance lengths. Alternating regions where each method excels reveal complementary strengths.

7.3 Adaptive Combination

The paper's Oracle_Combine selects the best SD method at each token position. Results show up to 2.2× further speedup over fixed strategies on InstructCoder, confirming that position-aware adaptive selection is a promising research direction.

Figure 11 (from paper): Oracle_Combine speedup (selecting best method per position) vs. individual methods on InstructCoder. The combined approach substantially outperforms any single fixed strategy.

9. How to Reproduce

Hardware: Full vLLM profiling requires NVIDIA H100 (80 GB). The trace-driven simulator runs on any machine with Python 3.10+.

9.1 Simulator (Trace-Driven)

# Clone and setup
git clone https://github.com/SpecDecode-Bench/simulator.git
cd simulator
pip install -r requirements.txt

# Run full pipeline (all proposers × all datasets)
./simulate_and_plot.sh

# Or run specific configuration
python main.py --proposer eagle3 --datasets gsm8k instructcoder \
               --batch-sizes 1 8 16 32 64 128

# Generate plots
python plot.py --proposers eagle3 ngram combined \
               --datasets gsm8k instructcoder shareGPT cnn

9.2 Full vLLM Profiling

# Clone profiling-instrumented vLLM
git clone https://github.com/SpecDecode-Bench/vllm.git
cd vllm
pip install -e .

# Run with SD enabled (example: EAGLE-3 on Llama3.1-8B)
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.1-8B-Instruct \
    --speculative-model-type eagle3 \
    --num-speculative-tokens 3
