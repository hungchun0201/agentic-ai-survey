Overview

What is SOLA?

SOLA is a state-aware SLO (Service Level Objective) scheduling system for large language model serving. It addresses the challenge of meeting diverse latency and throughput SLOs in production LLM inference environments by incorporating request state information into scheduling decisions.

Unlike traditional schedulers that treat all requests uniformly, SOLA tracks the execution state of each request (e.g., prefill vs. decode phase, tokens generated so far, remaining budget until SLO deadline) and uses this information to make more intelligent scheduling, preemption, and prioritization decisions.

Key Insight In LLM serving, requests transition through distinct states (queued, prefilling, decoding) with dramatically different compute profiles. A scheduler that is aware of these states and each request's SLO progress can significantly improve overall SLO attainment rates.

Problem Statement

SLO Challenges in LLM Serving

Production LLM serving systems face several SLO-related challenges that SOLA aims to resolve:

  • Heterogeneous SLOs: Different applications require different latency targets — a chatbot may tolerate 200ms TTFT, while code completion needs 50ms.
  • Phase-dependent resource usage: Prefill is compute-bound; decode is memory-bandwidth-bound. Treating them identically wastes resources.
  • State blindness: Existing schedulers (FCFS, shortest-job-first) do not account for how much SLO budget a request has already consumed.
  • Preemption cost: Naively preempting requests wastes the compute already invested in their KV cache, degrading overall throughput.

Approach: State-Aware Scheduling

Core Mechanism

SOLA introduces state-aware scheduling by maintaining a runtime model of each request's execution state and SLO slack. The scheduler continuously evaluates:

  1. Request State: Whether a request is in the queue, prefill phase, or decode phase, along with its progress within each phase.
  2. SLO Slack: The remaining time budget before the SLO would be violated, computed as SLO_deadline - elapsed_time - estimated_remaining_time.
  3. Resource Availability: Current GPU compute utilization, memory pressure, and batch composition to determine optimal scheduling actions.

# Pseudocode: SOLA state-aware priority computation def compute_priority(request, current_time): # Track request state state = request.get_state() # QUEUED | PREFILL | DECODE # Compute SLO slack (time remaining before violation) elapsed = current_time - request.arrival_time estimated_remaining = estimate_completion(request, state) slack = request.slo_deadline - elapsed - estimated_remaining # State-aware priority: tighter slack = higher priority if slack < 0: return Priority.CRITICAL # Already at risk elif state == State.PREFILL: return ttft_aware_priority(slack, request.slo_type) elif state == State.DECODE: return decode_rate_priority(slack, request.tokens_left) else: return queue_priority(slack, request.slo_type)

State-Aware Preemption SOLA's preemption logic considers the cost of evicting a request (e.g., lost KV cache compute) against the benefit of admitting a higher-priority request. This avoids the wasted-work problem that plagues naive priority schedulers.

Advantages over State-Unaware Schedulers

  • Higher SLO attainment rates by prioritizing requests at risk of missing their deadlines
  • Reduced wasted compute through cost-aware preemption that considers KV cache investment
  • Better GPU utilization by composing batches with complementary resource profiles
  • Support for heterogeneous SLOs within a single serving instance
