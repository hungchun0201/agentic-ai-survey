IC-Cache: Efficient LLM Serving via In-Context Caching

Semantic-level caching that repurposes historical request-response pairs as in-context examples, enabling small LLMs to imitate and even exceed large model capabilities through live capability augmentation.

Yifan Yu, Yu Gan, Nikhil Sarda, Lillian Tsai, Jiaming Shen, Yanqi Zhou, Arvind Krishnamurthy, Fan Lai, Henry M. Levy, David E. Culler — Google & University of Illinois & University of Washington

1.4-5.9x Throughput 28-71% Latency Reduction 70%+ Semantic Overlap arXiv:2501.12689

Overview

IC-Cache is an in-context caching system for LLM serving that unlocks new sweet spots in the quality-latency-cost tradeoff. Rather than simply returning cached responses (which suffer from low exact-match rates and quality degradation), IC-Cache repurposes historical request-response pairs from larger models as in-context examples prepended to new requests. This enables live capability augmentation of small LLMs, allowing them to produce responses matching or exceeding their larger counterparts at a fraction of the cost.

Key Insight

Over 70% of daily LLM requests have a semantically similar historical counterpart (cosine similarity > 0.8). Rather than directly reusing cached responses (which hurts quality), IC-Cache prepends similar historical request-response pairs as in-context demonstrations, grounded in in-context learning theory, allowing small models to learn from high-quality examples on-the-fly.

Why Not Naive Semantic Caching?

Problem with Existing Semantic Caching

Returning the most similar cached response drops win rate from 50% to only 18%. Even high semantic similarity does not guarantee equivalent queries, and reusing responses risks off-topic replies. IC-Cache avoids this pitfall by using cached pairs as demonstrations rather than direct answers.

Serving Load Dynamics

Analysis of Microsoft's Azure LLM serving traces shows peak loads reaching up to 25x off-peak levels, with spikes occurring within minutes. Meeting SLOs under such bursty conditions is extremely challenging, making cost-effective offloading to smaller models crucial.

1. Example Selector

Retrieves the most helpful historical request-response pairs from the cache using a two-stage process: (1) fast relevance filtering via dense embeddings, then (2) fine-grained helpfulness prediction using a lightweight proxy model (<0.2% model size of Qwen2.5-7B).

2. Request Router

Routes requests to the most suitable LLM using a contextual multi-armed bandit (MAB). Considers request complexity, selected examples, and current system load. Incorporates load-aware biasing via exponential moving average (EMA) tracking.

3. Example Manager

Curates the example cache over time: cost-aware example replay refines quality offline, knapsack-based eviction optimizes cache utility under memory constraints, and adaptive caching responds to evolving query distributions.

Two-Stage Example Selection

A key technical contribution: semantic relevance alone has a weak correlation (Pearson r = 0.04-0.22) with actual helpfulness. IC-Cache's two-stage mechanism addresses this fundamental limitation.

Stage 1&2 Synergy

Two-stage selection dramatically improves response quality compared to relevance-only retrieval. On Alpaca, the average score improves from -0.51 (Stage 1 only) to -0.10 (Stage 1+2), and on Open Orca from -0.22 to -0.29. The proxy model accounts for model-specific capabilities, example quality, and diversity beyond mere textual similarity.

Clustering for Scalability

To keep Stage 1 efficient, IC-Cache clusters cached examples offline into K groups using K-Means. The optimal K is derived by minimizing total matching cost:

// Minimize: K comparisons (centroids) + N/K comparisons (within cluster)
// Solution:
K = arg minK(K + N/K) = sqrt(N)

// For N = 1,000,000 examples:
K = 1000 clusters
// Total comparisons per request: 1000 + 1000 = 2000
// vs. brute-force: 1,000,000 comparisons

Request Router: Bandit-Based Model Selection

IC-Cache frames the routing decision as a contextual multi-armed bandit (MAB) problem, where each "arm" corresponds to a candidate model (e.g., small LLM with examples, or large LLM without). This lightweight approach adapts to dynamic serving loads without expensive retraining.

Key Routing Features

Thompson Sampling

Maintains Beta distributions per model representing quality beliefs. Explores by sampling from these distributions, exploits by selecting highest sampled value. Convergence guaranteed: P(failure) ≤ (N-1)T-C after T rounds.

Load-Aware Biasing

Tracks serving load via EMA. When load exceeds threshold, applies tanh(load-threshold) bias to favor smaller models, providing smooth saturating response. Only active during actual overload conditions.

Selective Feedback

Collects user feedback only on uncertain decisions (confidence std < 0.1). Tags these with "Which response do you prefer?" queries, minimizing feedback collection overhead while maximizing informativeness.

Router Algorithm

def ServeRequests(request, sys_load):
    # Stage 1: Select high-utility examples from cache
    examples = RetrieveExamples(request)

    # Stage 2: Route to optimal model via bandit
    model = RouteRequest(request, examples, models, sys_load)

    # Stage 3: Generate response (prepend examples if offloading)
    response = GenerateResponse(model, request, examples)

    # Stage 4: Update cache with new pair
    ManageExamples(request, response)

    return response

def RetrieveExamples(request):
    # Stage 1: Lightweight relevance selection
    request_emb = ExtractEmbedding(request)
    relevant = GetSimilarExamples(request_emb, examples)

    # Stage 2: Helpfulness prediction
    for ex in relevant:
        ex.helpfulness = PredHelpfulness(request, ex)

    # Optimize combination accounting for diversity
    selected = RetrieveComb(relevant, helpfulness)
    return selected

Example Manager: Cache Optimization

The Example Manager curates and optimizes the cache through three mechanisms: cost-aware replay, knapsack-based eviction, and adaptive sizing.

Cost-Aware Example Replay

Not all cached examples are equally valuable. The Example Manager selectively replays examples offline to improve their quality, prioritizing those with the highest potential gain:

# Potential gain of refining example e:
G(e) = (1 - normalized_response_quality) * normalized_model_cost

# Higher gain means:
# - Response quality has room for improvement
# - The example enables offloading to cheaper models
# - Replay cost is justified by future savings

# Replay continues until:
# G(e) falls below cut-off (diminishing returns)
# or example has been replayed 5+ times (outlier filter)

Minimal Memory, Maximum Impact

IC-Cache achieves near-saturated performance with a remarkably small cache. On translation tasks, just 12,056 examples (under 20MB) match the performance of the full 600K example pool. The long-tailed access distribution means a small fraction of examples covers the vast majority of request patterns.

Integration & Implementation

IC-Cache is implemented as a lightweight ~3,000 LOC prototype that integrates with existing LLM serving frameworks with minimal code changes.

Minimal Integration Code

from IC_cache import IC_cacheClient

def generate_response(requests, generation_config):
    # Create client session to IC-Cache service
    client = IC_cacheClient(config=generation_config)

    # Get inference results with generation config
    response = client.generate(requests)

    # Register request-response pair to the cache
    client.update_cache(requests, response)
    client.stop()

Supported Frameworks

vLLM

High-performance LLM serving with PagedAttention. IC-Cache integrates as a middleware layer intercepting requests before they reach the vLLM backend.

HuggingFace Runtime

Serverless inference API supporting diverse model families. IC-Cache wraps API calls with example selection and routing logic.

LangChain

Framework for context-aware reasoning applications. IC-Cache integrates as a chain component for automatic example augmentation.

Backend Architecture

Distributed Design

The Example Retriever, Request Router, and Example Manager run as separate processes communicating via gRPC, enabling independent horizontal scaling. The Retriever uses GPU-accelerated FAISS for similarity search. The Router is implemented in Jax with a contextual bandit (0.5M parameters) and a prioritized replay buffer. Each component periodically checkpoints state for fault tolerance.

IC-Cache Complements Existing Systems

Combining IC-Cache with RAG and SFT

IC-Cache is complementary to both RAG and supervised fine-tuning. On MS MACRO, Gemma-2-2B + RAG + IC-Cache achieves a win rate of 62.40%, significantly outperforming RAG-only (52.63%) or IC-Cache-only (56.35%). Similarly, combining IC-Cache with SFT on Natural Questions improves out-of-domain (Alpaca) win rate from 32.33% to 46.69%.

Privacy Considerations

Access Control

APIs to control which queries enter the cache. Data sharing can be restricted to designated user domains (e.g., within organization).

Data Sanitization

Locally sanitize inputs before caching using tools like spaCy for PII removal. Examples are processed on client endpoints.

Differential Privacy

Supports DP-synthesized example pools for strict privacy. Only marginal quality decrease (62.0% vs 57.3% on MS MACRO) with DP synthetic examples.

Key Takeaways

Paradigm Shift

IC-Cache transforms semantic caching from "reuse responses" to "reuse knowledge." By treating cached pairs as in-context learning demonstrations, it sidesteps the quality degradation inherent in naive caching approaches.

Practical System

~3,000 LOC, integrates with vLLM/HuggingFace/LangChain in a few lines. Negligible overhead (<1% latency). Tested on millions of real-world requests across 8 datasets and multiple model families.

Small Models Can Win

With well-selected in-context examples, Gemma-2-2B can surpass Gemma-2-27B on MS MACRO (achieving >50% win rate), demonstrating that capability augmentation can close the gap between model tiers.
