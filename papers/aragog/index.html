Aragog: Just-in-Time Model Routing for Scalable Serving of Agentic Workflows

Adapts per-input configuration during execution rather than per-workflow. Decouples accuracy prediction from runtime cost, enabling stage-wise configuration adaptation that maximizes serving throughput by 42.8-217.0%.

Yinwei Dai, Zhuofu Chen, Anand Iyer, Ravi Netravali Princeton University · Georgia Institute of Technology 8x H100 80GB · SGLang Backend

1 Overview

Agentic workflows have emerged as a powerful paradigm for solving complex, multi-stage tasks, but serving them at scale is computationally expensive. LLM inference accounts for nearly 80% of end-to-end request time. Configuration selection -- assigning specific LLMs to each agent -- can reduce costs, but existing approaches bind configuration decisions before request execution, making them ill-suited for workflows' heterogeneous and lengthy nature.

Core Insight: The accuracy of a configuration for a given input is fixed, while its runtime cost (throughput, latency, utilization) rapidly fluctuates with system dynamics. By recognizing this fundamental asymmetry, Aragog decouples accuracy (static but expensive to compute) from cost (dynamic but cheap to assess), making frequent runtime reconfiguration practical.

Stage-Wise Adaptation

Unlike per-workflow or per-input approaches, Aragog reconfigures at each workflow stage, exploiting runtime dynamics that change as requests progress through the graph.

Configuration Predictor

Efficiently identifies all accurate configurations once per input using binary routers, rather than selecting a single best configuration upfront.

Runtime Scheduler

Beam search over FIFO-ordered requests jointly optimizes resource utilization and flexibility preservation across concurrent requests.

Error Tolerance

Workflows exhibit 25-70% correction rates from intermediate errors, dramatically expanding the set of viable configurations per input.

2 Background & Motivation

Limitations of Existing Optimization

Per-Workflow Optimization

Systems like Cognify and LLMSelector select one fixed configuration for all requests. Eliminates runtime overhead but sacrifices per-input adaptability entirely.

Per-Input Optimization

Existing LLM routing approaches select a single configuration at the start. Early binding becomes suboptimal as system states change rapidly during lengthy multi-stage execution.

Two Key Problems

C1: Prohibitive routing overhead. Configuration spaces grow exponentially: M^N configurations for N agents and M models. Routing overhead scales poorly and frequent rerouting can offset the benefit of flexible adaptation.
C2: Joint runtime reconfiguration under concurrency. Each request's configuration assignment constrains options for other concurrent requests. Independent scheduling fails to capitalize on configuration flexibility.

Configuration Flexibility of Workflows

Workflows exhibit remarkable error tolerance: downstream agents can often recover from intermediate errors via refinement agents. This self-correcting behavior dramatically expands the viable configuration space. Experiments show that incorporating dynamic runtime cost metrics and stage-wise adaptation can further improve serving capacity by 25-30%.

Static Approaches Achieve Suboptimal Serving Capacity

3 Design Overview

Aragog Architecture: Configuration Predictor + Runtime Scheduler

4 Configuration Predictor

Key Insight: Near-Monotonicity

The accuracy of a workflow exhibits near-monotonic behavior under model upgrades: replacing any agent's model with a more capable one maintains or improves end-to-end accuracy in the vast majority of cases (only 8 out of 288 upgrade cases showed violations, with accuracy drops capped at 1.5%).

Binary Router Strategy

Aragog decomposes the exponential configuration space into many simple binary classification problems, one per configuration pair. This achieves 94-96% precision and 71-77% recall, far outperforming a single large router (31-40% precision, 18-21% recall).

// Configuration prediction with pruning
for each config_chain in monotonic_chains(config_space):
    // Binary search to find accuracy boundary
    boundary = binary_search(chain, router_predictions)
    // All configs above boundary are accurate
    accurate_set.add(chain[boundary:])

// Verify remaining candidates with their binary routers
final_set = { c for c in accurate_set if router(c).predict() == accurate }

Efficient Router Inference

Binary routers share a common embedding model but have router-specific classifiers. Aragog fuses these classifiers layer by layer via custom Triton kernels, maximizing parallelism and reducing average per-input routing latency from 90.9-196.2 ms to 42.6-44.9 ms.

5 Runtime Scheduler

Joint Optimization via Beam Search

The scheduler maintains a FIFO queue of requests, each tagged with their viable configurations and current workflow stage. Beam search (B=4 by default) iteratively extends partial assignments:

Beam Search Scheduling: Jointly Optimize Across Concurrent Requests

Two Ranking Criteria

Utilization Improvement (U)

Total forecasted throughput improvement from the assignment, estimated using offline-profiled throughput-load curves for each model engine.

Flexibility Preservation (F)

Average percentage of configuration options preserved across requests for future stages. Serves as a tiebreaker to balance immediate utilization with downstream adaptability.

Supporting Complex Workflows

Aragog uses topological sorting and a two-level priority scheme: (1) inter-request: FIFO ordering prioritizes earlier requests; (2) intra-request: agents are prioritized by depth in descending order, scheduling critical-path agents first.

6 Implementation

Implemented as a 6.8K-line orchestration layer in Python and Triton, sitting between agentic workflow frontends (DSPy) and LLM serving engines (SGLang), without modifying either.

Frontend

Converts any DSPy program into Aragog's internal workflow representation, enabling runtime model reconfiguration and async stage-wise execution.

Router Inference

Dynamic batching for shared embedding model. Custom Triton kernels fuse classifiers layer-by-layer with CUDA graphs for reduced kernel launch overhead.

Runtime Scheduler

Async scheduler with FIFO queues, engine state tracking, configuration caches, and exponential moving averages of queue metrics. Beam search uses bit vectors for efficient state encoding.

Serving Backend

SGLang serving engine. Supports any backend with OpenAI-compatible APIs (vLLM, TensorRT-LLM, HuggingFace TGI).

Accuracy Preservation

Aragog maintains accuracy within 2% of the most expensive configuration across all five workloads and both model families, demonstrating that the configuration predictor maintains high precision without sacrificing quality.

Scaling with Complexity: Aragog's throughput improvements increase from 42.8-50.1% for self-refine workflows to 54.0-76.3% for more complex workflows (text-to-sql, task decomposition, voting). This is because longer workflows expose more stages for runtime reconfiguration, amplifying the benefits of stage-wise adaptation.

Latency Distribution

At peak load, Aragog reduces P25 latency by 23.6-60.1%, median by 32.5-71.1%, and P95 by 46.2-76.2% over per-input optimization. Against per-workflow optimization: P25 by 58.4-82.8%, median by 60.0-86.1%, and P95 by 63.2-89.0%.

9 Microbenchmarks

Routing Overhead

Average routing latency accounts for at most 3.5% of total latency at low request rates and less than 1% at high rates (when queueing delays dominate). Routing does not impose any additional delay on end-to-end latencies.
