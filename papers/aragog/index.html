Aragog: Just-in-Time Model Routing for Scalable Serving of Agentic Workflows

Adapts per-input configuration during execution rather than per-workflow. Decouples accuracy prediction from runtime cost, enabling stage-wise configuration adaptation that maximizes serving throughput by 42.8-217.0%.

arXiv 2025 arXiv:2511.20975 Just-in-Time Routing Agentic Workflows

Yinwei Dai, Zhuofu Chen, Anand Iyer, Ravi Netravali

Princeton University · Georgia Institute of Technology

1 Overview

Agentic workflows have emerged as a powerful paradigm for solving complex, multi-stage tasks, but serving them at scale is computationally expensive. LLM inference accounts for nearly 80% of end-to-end request time. Configuration selection -- assigning specific LLMs to each agent -- can reduce costs, but existing approaches bind configuration decisions before request execution, making them ill-suited for workflows' heterogeneous and lengthy nature.

Core Insight: The accuracy of a configuration for a given input is fixed, while its runtime cost (throughput, latency, utilization) rapidly fluctuates with system dynamics. By recognizing this fundamental asymmetry, Aragog decouples accuracy (static but expensive to compute) from cost (dynamic but cheap to assess), making frequent runtime reconfiguration practical.

2 Background & Motivation

Agentic workflows assemble multiple LLM-driven agents into a directed graph — coding refiners, critique-refine loops, voting ensembles, task decomposers — and each stage in that graph can independently choose which model to call. The configuration space therefore explodes combinatorially, and existing systems collapse it the wrong way: they pick one configuration before a request runs, then live with whatever load and latency the cluster happens to be in. This section motivates Aragog by characterizing two structural opportunities prior work leaves on the table: workflows tolerate intermediate errors better than expected, and many configurations are accuracy-equivalent on any given input.

Figure 1 (from paper): Examples of common agentic workflows that vary in terms of both structure and application domains.

Limitations of Existing Optimization

Per-Workflow Optimization

Systems like Cognify and LLMSelector select one fixed configuration for all requests. Eliminates runtime overhead but sacrifices per-input adaptability entirely.

Per-Input Optimization

Existing LLM routing approaches select a single configuration at the start. Early binding becomes suboptimal as system states change rapidly during lengthy multi-stage execution.

Two Key Problems

C1: Prohibitive routing overhead. Configuration spaces grow exponentially: M^N configurations for N agents and M models. Routing overhead scales poorly and frequent rerouting can offset the benefit of flexible adaptation.
C2: Joint runtime reconfiguration under concurrency. Each request's configuration assignment constrains options for other concurrent requests. Independent scheduling fails to capitalize on configuration flexibility.

Configuration Flexibility of Workflows

Workflows exhibit remarkable error tolerance: downstream agents can often recover from intermediate errors via refinement agents. This self-correcting behavior dramatically expands the viable configuration space. Experiments show that incorporating dynamic runtime cost metrics and stage-wise adaptation can further improve serving capacity by 25-30%.

3 Design Overview

Aragog couples two components that together turn configuration selection from a one-shot decision into a continuous game. An offline Configuration Predictor takes each input and returns the set of all configurations whose end-to-end accuracy matches the most-expensive configuration — not a single best choice. An online Runtime Scheduler then picks one element of that set at every workflow stage, using live engine load to maximize joint utilization across concurrent requests.

4 Configuration Predictor

The predictor's job is to answer one question per input: which configurations are safe to use? The naive answer — train one big classifier over the entire MN configuration space — fails at both ends, with poor accuracy and prohibitive latency. Aragog instead exploits two structural properties: workflow accuracy is near-monotonic in model capability, and the answer can be assembled from many cheap binary classifiers rather than one expensive multi-class router.

Key Insight: Near-Monotonicity

The accuracy of a workflow exhibits near-monotonic behavior under model upgrades: replacing any agent's model with a more capable one maintains or improves end-to-end accuracy in the vast majority of cases (only 8 out of 288 upgrade cases showed violations, with accuracy drops capped at 1.5%).

Figure 7 (from paper): Workflow accuracy is largely monotonic under model upgrades (rare < 1.5% violations), validating the chain-based pruning used by the predictor.

Binary Router Strategy

Aragog decomposes the exponential configuration space into many simple binary classification problems, one per configuration pair. This achieves 94-96% precision and 71-77% recall, far outperforming a single large router (31-40% precision, 18-21% recall).

Figure 6 (from paper): Routing accuracy drops as each router's configuration space grows — binary routers give the best precision/recall per parameter budget. 
// Configuration prediction with pruning
for each config_chain in monotonic_chains(config_space):
    // Binary search to find accuracy boundary
    boundary = binary_search(chain, router_predictions)
    // All configs above boundary are accurate
    accurate_set.add(chain[boundary:])

// Verify remaining candidates with their binary routers
final_set = { c for c in accurate_set if router(c).predict() == accurate }

Efficient Router Inference

Binary routers share a common embedding model but have router-specific classifiers. Aragog fuses these classifiers layer by layer via custom Triton kernels, maximizing parallelism and reducing average per-input routing latency from 90.9-196.2 ms to 42.6-44.9 ms.

5 Runtime Scheduler

Once the predictor has handed each request a set of accuracy-equivalent configurations, the scheduler must pick — at every workflow stage, for every concurrent request — which member of that set to dispatch to which engine. The decisions are coupled: assigning request A to model B occupies B and forces request C onto a slower engine. The scheduler therefore searches jointly over concurrent requests rather than greedily per-request, balancing immediate utilization against future flexibility.

Joint Optimization via Beam Search

The scheduler maintains a FIFO queue of requests, each tagged with their viable configurations and current workflow stage. Beam search (B=4 by default) iteratively extends partial assignments:

Figure 9 (from paper): Beam search iteratively extends partial assignments ranked by utilization and flexibility, keeping the top-B candidates per step.
Beam Search Scheduling: Jointly Optimize Across Concurrent Requests

Two Ranking Criteria

Utilization Improvement (U)

Total forecasted throughput improvement from the assignment, estimated using offline-profiled throughput-load curves for each model engine.

Flexibility Preservation (F)

Average percentage of configuration options preserved across requests for future stages. Serves as a tiebreaker to balance immediate utilization with downstream adaptability.

Figure 10 (from paper): Beam search reaches near-brute-force utilization and flexibility at greedy-level runtime cost.

Supporting Complex Workflows

Aragog uses topological sorting and a two-level priority scheme: (1) inter-request: FIFO ordering prioritizes earlier requests; (2) intra-request: agents are prioritized by depth in descending order, scheduling critical-path agents first.

6 Implementation

Implemented as a 6.8K-line orchestration layer in Python and Triton, sitting between agentic workflow frontends (DSPy) and LLM serving engines (SGLang), without modifying either.

Frontend

Converts any DSPy program into Aragog's internal workflow representation, enabling runtime model reconfiguration and async stage-wise execution.

Router Inference

Dynamic batching for shared embedding model. Custom Triton kernels fuse classifiers layer-by-layer with CUDA graphs for reduced kernel launch overhead.

Runtime Scheduler

Async scheduler with FIFO queues, engine state tracking, configuration caches, and exponential moving averages of queue metrics. Beam search uses bit vectors for efficient state encoding.

Serving Backend

SGLang serving engine. Supports any backend with OpenAI-compatible APIs (vLLM, TensorRT-LLM, HuggingFace TGI).

7 Evaluation

Aragog is evaluated against per-input and per-workflow optimization baselines on five workflow-dataset combinations spanning two model families. 8 Key Results

Three findings carry the evaluation. First, Aragog improves end-to-end serving capacity by 42.8–76.3% over per-input baselines and 78.1–217.0% over per-workflow baselines. Second, accuracy stays within 2% of the most-expensive configuration across all five workloads. Third, the gains scale with workflow complexity — longer pipelines amplify the benefit of stage-wise reconfiguration.

Aragog consistently achieves the highest throughput and lowest average latency under varying request rates across both Qwen and Llama model families.

Latency Distribution

At peak load, Aragog reduces P25 latency by 23.6-60.1%, median by 32.5-71.1%, and P95 by 46.2-76.2% over per-input optimization. Against per-workflow optimization: P25 by 58.4-82.8%, median by 60.0-86.1%, and P95 by 63.2-89.0%.

10 Related Work

Aragog occupies a previously empty point in the agentic-serving design space: it is the first system to perform stage-wise configuration adaptation during workflow execution, rather than locking in a model assignment before the first stage runs.

Configuration Optimization for Workflows

Per-Workflow (Cognify, LLMSelector)

Search a single best configuration offline and apply it to every request. Eliminates routing overhead but ignores per-input variability — Aragog beats this baseline by 78.1–217.0% on throughput.

Per-Input (RouteLLM, MasRouter, DAAO)

Train a router that selects one configuration per request before execution. Adapts to input difficulty but commits the choice before knowing how loaded each model engine will be at later stages — Aragog beats this by 50.0–76.3%.

Aragog (per-stage)

Predicts a set of accuracy-equivalent configurations once per input, then selects from that set at every stage based on live engine state. Decouples accuracy (static, expensive) from cost (dynamic, cheap).

Stack Position

Aragog deliberately avoids modifying either layer it sits between. The 6.8K-line orchestration shim consumes any DSPy program upstream and any OpenAI-compatible serving engine downstream (SGLang, vLLM, TensorRT-LLM, HF TGI). This lets Aragog stack on top of inference-engine-level optimizations like prefix caching, PD disaggregation, or speculative decoding without conflict.

11 Takeaways

Four ideas survive past the system itself and are worth carrying into future agentic-serving work. They are summarized below as conceptual lessons rather than numerical results, and each one corresponds to an architectural choice that could be reused independently of the rest of Aragog.

Asymmetry is the Hook

A configuration's accuracy on a given input is a property of the input — fixed, predictable offline. Its cost on the system is a property of the moment — fluctuating with queue depth, batching state, and concurrent workload. Treating these as a single decision (as all prior work did) wastes the asymmetry; treating them separately unlocks 2–3× headroom.

Predict a Set, Not a Choice

The binary-router-per-pair design plus monotonic-chain pruning is the enabling primitive: it makes "return all configurations as accurate as the best one" cheap enough (42–45 ms per input) to defer model choice to runtime. The shape of the prediction matters as much as the model behind it.

Beam Search Beats Greedy by a Lot

At B=1 (greedy) the gain over per-input is only 16–19%. At B=4 it jumps to 50–60%. Above B=4 the curve flattens. The non-trivial finding is that joint scheduling across concurrent requests — not single-request optimization — is what extracts the gain.

Error Tolerance Buys Flexibility

The reason multiple configurations achieve the same accuracy is that downstream agents (refiners, voters, aggregators) absorb upstream mistakes — workflows show 25–70% intermediate-error correction rates. This is a workflow-design property the serving system can exploit. If your agentic graph has no recovery path, Aragog has nothing to schedule across.

Implications for Agentic Infrastructure

A new layer in the stack. Aragog makes the case that configuration scheduling deserves to be its own layer between the workflow framework (DSPy) and the inference engine (SGLang/vLLM). It is not part of the model, not part of the engine, and not part of the agent program — it is the dispatch logic that holds them together. Expect this layer to absorb more responsibilities as agentic workloads mature.

Open Questions

  • Beyond near-monotonicity: the 8/288 violation rate is small but not zero. Workflows with capability tradeoffs (e.g., a small model that's actually better at structured output than a larger one) violate the assumption — does the binary-router-plus-chain pruning still cover them?
  • Routing under streaming: the evaluation uses request-rate-driven Poisson arrivals; real agentic traffic is bursty and bound to user sessions. How does the FIFO + beam-search scheduler behave when requests share KV-cache prefixes?
  • Cost as a first-class objective: the paper optimizes for throughput/latency under fixed hardware. Extending to cost-aware routing across rented endpoints (where models have $/1K-token prices) would require composing offline-profiled throughput curves with public pricing.
  • SLO-aware scheduling: the current scheduler treats all requests as equally urgent. Production deployments need per-request deadlines and the ability to spend extra latency on lower-priority requests to preserve options for premium ones.
