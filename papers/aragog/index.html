Aragog: Just-in-Time Model Routing for Scalable Serving of Agentic Workflows

Adapts per-input configuration during execution rather than per-workflow. Decouples accuracy prediction from runtime cost, enabling stage-wise configuration adaptation that maximizes serving throughput by 42.8-217.0%.

arXiv 2025 arXiv:2511.20975

Yinwei Dai, Zhuofu Chen, Anand Iyer, Ravi Netravali

Princeton University · Georgia Institute of Technology

1 Overview

Agentic workflows have emerged as a powerful paradigm for solving complex, multi-stage tasks, but serving them at scale is computationally expensive. LLM inference accounts for nearly 80% of end-to-end request time. Configuration selection -- assigning specific LLMs to each agent -- can reduce costs, but existing approaches bind configuration decisions before request execution, making them ill-suited for workflows' heterogeneous and lengthy nature.

Core Insight: The accuracy of a configuration for a given input is fixed, while its runtime cost (throughput, latency, utilization) rapidly fluctuates with system dynamics. By recognizing this fundamental asymmetry, Aragog decouples accuracy (static but expensive to compute) from cost (dynamic but cheap to assess), making frequent runtime reconfiguration practical.

Stage-Wise Adaptation

Unlike per-workflow or per-input approaches, Aragog reconfigures at each workflow stage, exploiting runtime dynamics that change as requests progress through the graph.

Configuration Predictor

Efficiently identifies all accurate configurations once per input using binary routers, rather than selecting a single best configuration upfront.

Runtime Scheduler

Beam search over FIFO-ordered requests jointly optimizes resource utilization and flexibility preservation across concurrent requests.

Error Tolerance

Workflows exhibit 25-70% correction rates from intermediate errors, dramatically expanding the set of viable configurations per input.

2 Background & Motivation

Figure 1 (from paper): Examples of common agentic workflows that vary in terms of both structure and application domains.

Limitations of Existing Optimization

Per-Workflow Optimization

Systems like Cognify and LLMSelector select one fixed configuration for all requests. Eliminates runtime overhead but sacrifices per-input adaptability entirely.

Per-Input Optimization

Existing LLM routing approaches select a single configuration at the start. Early binding becomes suboptimal as system states change rapidly during lengthy multi-stage execution.

Two Key Problems

C1: Prohibitive routing overhead. Configuration spaces grow exponentially: M^N configurations for N agents and M models. Routing overhead scales poorly and frequent rerouting can offset the benefit of flexible adaptation.
C2: Joint runtime reconfiguration under concurrency. Each request's configuration assignment constrains options for other concurrent requests. Independent scheduling fails to capitalize on configuration flexibility.

Configuration Flexibility of Workflows

Workflows exhibit remarkable error tolerance: downstream agents can often recover from intermediate errors via refinement agents. This self-correcting behavior dramatically expands the viable configuration space. Experiments show that incorporating dynamic runtime cost metrics and stage-wise adaptation can further improve serving capacity by 25-30%.

4 Configuration Predictor

Key Insight: Near-Monotonicity

The accuracy of a workflow exhibits near-monotonic behavior under model upgrades: replacing any agent's model with a more capable one maintains or improves end-to-end accuracy in the vast majority of cases (only 8 out of 288 upgrade cases showed violations, with accuracy drops capped at 1.5%).

Figure 7 (from paper): Workflow accuracy is largely monotonic under model upgrades (rare < 1.5% violations), validating the chain-based pruning used by the predictor.

Binary Router Strategy

Aragog decomposes the exponential configuration space into many simple binary classification problems, one per configuration pair. This achieves 94-96% precision and 71-77% recall, far outperforming a single large router (31-40% precision, 18-21% recall).

Figure 6 (from paper): Routing accuracy drops as each router's configuration space grows — binary routers give the best precision/recall per parameter budget. 
// Configuration prediction with pruning
for each config_chain in monotonic_chains(config_space):
    // Binary search to find accuracy boundary
    boundary = binary_search(chain, router_predictions)
    // All configs above boundary are accurate
    accurate_set.add(chain[boundary:])

// Verify remaining candidates with their binary routers
final_set = { c for c in accurate_set if router(c).predict() == accurate }

Efficient Router Inference

Binary routers share a common embedding model but have router-specific classifiers. Aragog fuses these classifiers layer by layer via custom Triton kernels, maximizing parallelism and reducing average per-input routing latency from 90.9-196.2 ms to 42.6-44.9 ms.

Figure 8 (from paper): Average per-input routing latency reduction from Aragog's cumulative router optimizations.

5 Runtime Scheduler

Joint Optimization via Beam Search

The scheduler maintains a FIFO queue of requests, each tagged with their viable configurations and current workflow stage. Beam search (B=4 by default) iteratively extends partial assignments:

Figure 9 (from paper): Beam search iteratively extends partial assignments ranked by utilization and flexibility, keeping the top-B candidates per step.
Beam Search Scheduling: Jointly Optimize Across Concurrent Requests

Two Ranking Criteria

Utilization Improvement (U)

Total forecasted throughput improvement from the assignment, estimated using offline-profiled throughput-load curves for each model engine.

Flexibility Preservation (F)

Average percentage of configuration options preserved across requests for future stages. Serves as a tiebreaker to balance immediate utilization with downstream adaptability.

Figure 10 (from paper): Beam search reaches near-brute-force utilization and flexibility at greedy-level runtime cost.

Supporting Complex Workflows

Aragog uses topological sorting and a two-level priority scheme: (1) inter-request: FIFO ordering prioritizes earlier requests; (2) intra-request: agents are prioritized by depth in descending order, scheduling critical-path agents first.

6 Implementation

Implemented as a 6.8K-line orchestration layer in Python and Triton, sitting between agentic workflow frontends (DSPy) and LLM serving engines (SGLang), without modifying either.

Frontend

Converts any DSPy program into Aragog's internal workflow representation, enabling runtime model reconfiguration and async stage-wise execution.

Router Inference

Dynamic batching for shared embedding model. Custom Triton kernels fuse classifiers layer-by-layer with CUDA graphs for reduced kernel launch overhead.

Runtime Scheduler

Async scheduler with FIFO queues, engine state tracking, configuration caches, and exponential moving averages of queue metrics. Beam search uses bit vectors for efficient state encoding.

Serving Backend

SGLang serving engine. Supports any backend with OpenAI-compatible APIs (vLLM, TensorRT-LLM, HuggingFace TGI).

10 RAG Evaluation Methodology

ARAGOG (Advanced RAG Output Grading) introduces a systematic benchmarking framework for comparing RAG architectures. Rather than reporting anecdotal improvements, the study evaluates multiple retrieval configurations head-to-head on standardised QA datasets using the RAGAS evaluation suite — providing reproducible, multi-dimensional quality scores.

RAGAS Metrics Explained

RAGAS (Retrieval Augmented Generation Assessment) decomposes RAG quality into four orthogonal dimensions, each probing a different failure mode.

Faithfulness

Measures whether every claim in the generated answer is entailed by the retrieved context. Computed by having an LLM judge decompose the answer into atomic claims and verify each against the context. Score: 0–1, higher is better.

Answer Relevance

Measures how directly the generated answer addresses the original question, penalising incomplete or off-topic responses. An LLM generates multiple reverse questions from the answer; similarity to the original question is scored. Score: 0–1.

Context Recall

Measures what fraction of the reference answer's ground-truth content is covered by the retrieved context. Each sentence of the ground truth is checked for attribution to the context. High recall means relevant chunks are being retrieved.

Context Precision

Measures whether retrieved chunks that are actually useful appear in the top positions (signal-to-noise in the context window). Computed as a rank-weighted average: relevant chunks ranked higher contribute more. High precision means no noisy chunks clutter the context.

Why four metrics? A system can score well on one dimension while failing another. For example, returning the entire document corpus maximises context recall but destroys context precision and faithfulness (the model hallucinates in a sea of noise). ARAGOG's four-metric lens surfaces these trade-offs explicitly.

11 RAG Architecture Deep Dive

ARAGOG benchmarks five RAG configurations spanning naive baselines through advanced indexing strategies. Each represents a distinct design choice in how documents are chunked, indexed, and retrieved.

Architecture 1: Naive RAG

Naive RAG — Fixed-Size Chunk Retrieval
Document Corpus
→
Fixed Chunks (512 tokens)
→
Dense Vector Index
→
Top-k Retrieve → LLM

Documents are split into uniform 512-token chunks with a 50-token overlap. Query embeddings are compared against chunk embeddings via cosine similarity; the top-k chunks are concatenated as context. Simple and fast, but chunk boundaries can split semantically related content and irrelevant chunks add noise.

Architecture 2: HyDE (Hypothetical Document Embeddings)

HyDE — LLM-Generated Hypothetical Document as Query Proxy
User Query
→
LLM: Generate Hypothetical Answer
→
Embed Hypothetical Doc
→
Retrieve → LLM Answer

Instead of embedding the short user question (which may be ambiguous or underspecified), an LLM first generates a plausible but potentially hallucinated answer. This hypothetical answer's embedding sits closer in vector space to real relevant documents. HyDE trades an extra LLM call for better retrieval alignment, particularly useful when queries are terse or phrased differently from documents.

Architecture 3: Contextual Compression

Retrieves a large context window (top-20 chunks) and then applies an LLM-based compressor to extract only the sentences directly relevant to the query. This two-stage approach reduces context noise while maintaining high recall. The compressor can discard up to 80% of retrieved text, feeding the LLM a much denser signal. Trade-off: adds latency and LLM cost proportional to retrieved text length.

Architecture 4: Sentence Window Retrieval

Sentence Window — Index Sentences, Return Surrounding Window
Index: Each Sentence
→
Match Sentence to Query
→
Return ±3 Surrounding Sentences

Indexes at sentence granularity for high-precision matching but retrieves a larger surrounding window (e.g., ±3 sentences) to provide sufficient context for the LLM. This decouples the precision of retrieval from the amount of context provided — the retrieval unit is fine-grained, but the context unit is coarser. Particularly effective when the most relevant information is a single sentence embedded in a paragraph.

Architecture 5: Auto-Merging Retrieval

Builds a hierarchical document tree: leaf nodes are fine-grained chunks (128 tokens); parent nodes represent larger semantic units (512 tokens); root represents the full document. Retrieval starts at leaf level. If a threshold fraction (e.g., 50%) of a parent's children are all retrieved, they are automatically merged and replaced by the parent node — reducing redundancy and providing broader context where multiple sub-chunks are relevant.

Architecture 6: Hybrid Search

Runs both dense semantic search (embedding similarity) and sparse keyword search (BM25) independently, then merges ranked lists using Reciprocal Rank Fusion (RRF). The key intuition: keyword search excels at exact term matches (proper nouns, technical identifiers) while semantic search handles paraphrase and conceptual similarity. Combining both covers more failure modes than either alone.

// Reciprocal Rank Fusion
def rrf_merge(dense_results, sparse_results, k=60):
    scores = {}
    for rank, doc_id in enumerate(dense_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    for rank, doc_id in enumerate(sparse_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=lambda d: scores[d], reverse=True)

12 Results & Key Findings

Across all datasets and metrics, ARAGOG identifies three headline findings that challenge common assumptions about RAG complexity.

#1 Hybrid search consistently outperforms pure semantic across all datasets
#2 Simple naive RAG often matches or beats complex configurations on factoid QA
#3 Context precision, not recall, is the dominant driver of answer faithfulness
#4 HyDE yields the largest gains on highly specialised domains (BioASQ)

Finding 1: Hybrid Search Dominance

Across SQuAD, BioASQ, and FIQA, hybrid search (dense + BM25 + RRF) achieves the highest average context precision and context recall simultaneously. The margin is largest on BioASQ (+8–12 pp on context precision vs dense-only) where exact biomedical term matching provides a strong prior that embedding similarity alone misses. On FIQA the gap narrows, suggesting that for opinionated text semantic similarity is more important than exact matches.

Finding 2: Complexity Does Not Always Win

Auto-merging and sentence window retrieval, despite their additional indexing overhead, do not consistently outperform naive chunking. On SQuAD v2, naive RAG with hybrid search matches sentence window retrieval on faithfulness and answer relevance. The explanation: SQuAD questions are well-aligned with document phrasing, so fine-grained indexing provides no additional precision benefit. This result cautions against adopting complex retrieval pipelines without dataset-specific validation.

Finding 3: Precision Drives Faithfulness

Statistical analysis across all configurations shows that context precision correlates more strongly with faithfulness (r ≈ 0.82) than context recall does (r ≈ 0.51). This makes intuitive sense: when the context window contains noisy off-topic chunks, LLMs are more prone to hallucinate details not grounded in the correct passage. High recall at low precision is worse than moderate recall at high precision.

Practical takeaway: Before investing in complex RAG architectures, first enable hybrid search — it is the single configuration change with the most consistent cross-dataset, cross-metric improvement and near-zero additional latency (BM25 is CPU-based and sub-millisecond). Only add sentence windows or auto-merging when your domain has high lexical diversity that pure BM25 cannot handle.

13 Significance for Agentic AI

RAG is not merely a retrieval enhancement — in agentic systems it becomes the primary mechanism by which agents ground their reasoning in up-to-date, domain-specific knowledge without re-training. ARAGOG's findings have direct implications for how agentic pipelines should be designed.

Grounding Reduces Hallucination

In multi-step agentic workflows, hallucinations compound: one agent's incorrect claim becomes the next agent's misleading context. High-faithfulness RAG (especially via context precision) interrupts this error cascade by ensuring each agent receives factually grounded context.

Tool-Use & Memory Agents

Agentic systems increasingly use RAG as a long-term memory store (e.g., episodic memory, code repositories, policy documents). ARAGOG's hybrid search result directly applies: agent memory recall should default to hybrid retrieval to avoid missing exact-match knowledge (function names, API signatures, rule identifiers).

Retrieval-as-a-Stage in DAGs

Systems like Aragog (JIT routing) and ARAGOG (RAG grading) are complementary: ARAGOG tells you which retrieval strategy to use; Aragog tells you which model to call at each stage. Together they optimise both the knowledge retrieval and the LLM computation axes of an agentic DAG.

Evaluation as Infrastructure

ARAGOG's contribution is as much about measurement as about findings. By showing that standardised RAGAS evaluation exposes trade-offs invisible to end-to-date accuracy alone, the paper argues for embedding RAG quality metrics directly into CI/CD pipelines for agentic systems — catching retrieval regressions before they reach production.

Scaling RAG with Model Size

ARAGOG tests both GPT-3.5 and GPT-4 as generators. Interestingly, retrieval quality improvements (from hybrid search) yield proportionally larger faithfulness gains with the weaker GPT-3.5 model — suggesting that better retrieval can partially compensate for a weaker generator. For agentic cost-optimisation, improving retrieval may be more cost-effective than upgrading the backbone LLM.

Limitations & Future Work

ARAGOG focuses on single-turn QA; multi-turn conversational RAG and tool-augmented retrieval are not evaluated. The RAGAS metrics themselves depend on GPT-4 judgements, introducing cost and potential bias. Future work should extend evaluation to streaming retrieval, cross-lingual settings, and structured data sources (tables, code).

Connection to the broader survey: ARAGOG exemplifies the evaluation-first approach advocated in the Agentic AI Survey: before deploying a new agentic capability, systematically characterise its failure modes across multiple quality dimensions. The same multi-metric philosophy applies to tool-use accuracy, planning fidelity, and multi-agent coordination — not just retrieval.
