JITServe

SLO-aware LLM Serving with Imprecise Request Information — just-in-time bandwidth allocation for diverse SLO classes (latency-sensitive, deadline-sensitive, compound)

arXiv:2504.20068 v3 · Dec 2025 SLO-aware Scheduling Service Goodput vLLM-based

1. Overview & Core Insight

Modern LLM serving stacks (vLLM, Sarathi-Serve, DistServe, Autellix) optimize one aggregate metric — usually average latency or throughput — under the assumption that all requests share a single SLO. JITServe's measurement of millions of real production requests plus 550+ user surveys finds the opposite: requests bring three fundamentally different SLO classes, and a real workload mixes all three at once.

6.2 Pseudocode

# Algorithm 1 — GMAX (one scheduling tick)
def gmax_schedule(active_requests, capacity_C, batch_size_B):
    # 1. compute per-request demand
    for r in active_requests:
        r.b_min      = (r.L_up_hat - r.L_done) / r.delta_t
        r.priority   = r.b_min                      # tighter requests first
        r.margin_gp  = marginal_goodput(r)        # dgoodput / dbandwidth

    # 2. reserve b_min for every reachable request (SLO compliance)
    reachable = [r for r in active_requests
                 if r.delta_t > 0 and r.b_min <= capacity_C]
    reachable.sort(key=lambda r: -r.priority)
    chosen, used = [], 0
    for r in reachable:
        if used + r.b_min <= capacity_C and len(chosen) < batch_size_B:
            chosen.append(r); used += r.b_min

    # 3. distribute SURPLUS by marginal goodput
    surplus = capacity_C - used
    while surplus > 0:
        candidates = [r for r in chosen if r.can_take_more()]
        if not candidates: break
        winner = max(candidates, key=lambda r: r.margin_gp)
        share  = min(surplus, winner.cap_increment())
        winner.bandwidth += share
        surplus          -= share

    # 4. compose a batch of similar input lengths
    return group_by_input_length(chosen)

9. Implementation

9.1 Where the code lives

…

# conceptual integration with vLLM
class JITServeScheduler(vllm.scheduler.Scheduler):
    def _get_next_batch(self):
        active = self.running + self.waiting
        for r in active:
            self.analyzer.refine(r)               # QRF + graph match
        return self.gmax.schedule(active,
                                  capacity_C=self.free_cap(),
                                  batch_size_B=self.max_batch)

class RequestAnalyzer:
    qrf:           QRFModel                    # sklearn-quantile, ~10MB
    pattern_db:    GraphIndex                  # LSH over recent workflows
    def refine(self, r):
        r.L_up_hat = self.qrf.predict_quantile(r.features, q=0.95)
        if r.is_compound:
            r.next_stages = self.pattern_db.match(r.partial_graph)

10.1 Setup

Workloads (4 categories)

  • Chatbot — LMSys, WildChat traces (latency-sensitive)
  • Deep Research — multi-step web/document agents (compound)
  • Agentic CodeGen — MetaGPT, GAIA-style task graphs (compound)
  • Math Reasoning — long-chain CoT (deadline-sensitive)

Baselines

  • vLLM FCFS (default upstream)
  • Sarathi-Serve (chunked-prefill)
  • SJF / SRTF with predicted-length ranking
  • LAS / Autellix PLAS (program-level service)
  • EDF (earliest deadline first)
  • Oracle (ground-truth length scheduler — upper bound)

Metrics

  • Service goodput — primary; tokens that satisfy per-request SLO
  • Throughput — sanity that goodput gain is not paid in raw throughput
  • TTFT, TBT — for latency class
  • End-to-end latency — for deadline / compound
