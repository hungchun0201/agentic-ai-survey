Continuum: Efficient and Robust Multi-Turn LLM Agent Scheduling with KV Cache Time-to-Live

A KV cache-aware scheduling framework that uses predicted tool-call durations to decide whether to retain or evict cached state during agent pauses

Paper Overview

The Problem

Multi-turn LLM agents (e.g., coding assistants running SWE-Bench tasks) alternate between generating text and executing tool calls (shell commands, API calls). During tool execution, the agent's KV cache sits idle on GPU memory. In high-concurrency settings, this creates a fundamental tension:

  • Keep the KV cache pinned: the next turn avoids expensive prefill recomputation, but GPU memory is wasted while the tool runs.
  • Evict the KV cache: GPU memory is freed for other requests, but the returning request must re-prefill all its context.

Standard FCFS scheduling in vLLM is unaware of this multi-turn structure. The paper reports that 58.2% of end-to-end latency in baseline vLLM comes from scheduling bubbles -- idle time caused by requests waiting in queue while KV cache is locked by paused agents.

Key Contributions

  1. KV Cache TTL (Time-to-Live): A mechanism that assigns an expiration timer to each pinned KV cache. If the tool call returns before the TTL expires, the cache is reused; otherwise it is evicted to free memory.
  2. Tool-call duration prediction: Uses historical inter-request intervals (per-command averages) to estimate how long each tool call will take, enabling informed TTL decisions.
  3. Program-level FCFS scheduling: A ContinuumRequestQueue that groups requests by job_id and applies job-level first-come-first-served ordering, prioritizing pinned requests whose cache is still warm.
  4. Cost-benefit eviction logic: Compares predicted tool-call duration against a threshold to decide whether retaining the cache (saving prefill) is worth the memory occupation cost.

Evaluation Summary

The system is evaluated on SWE-Bench Verified (500 instances) and BFCL traces using Llama-3.1 8B and 70B models on H100 GPUs. Key claim: Continuum achieves a 2.16x improvement in average job completion time over baseline vLLM FCFS scheduling for 500 concurrent SWE-Bench jobs with Llama-3.1-70B on 4 H100s.

Core Mechanisms

4.1 TTL Computation and Pin/Unpin Logic

How TTL Works in the Code

When a request finishes generating, the scheduler calls _free_blocks(). If the scheduling policy is Continuum and the request is not on its last step, the estimator is consulted to determine a pin duration. If the predicted tool-call time is short (below the threshold), the KV cache is pinned for a fixed TTL window.

# scheduler.py -- _free_blocks() [lines 1349-1370]
def _free_blocks(self, request: Request):
    assert request.is_finished()
    # First, unpin any existing pin for this job_id
    for req, end_time in self.pinned_requests:
        if req.job_id == request.job_id:
            self.unpin_request(req, end_time)

    # Continuum TTL decision
    if self.policy == SchedulingPolicy.CONTINUUM and not request.is_last_step:
        length_of_pin = self.tool_call_estimator.set_up_pin(request)
        if length_of_pin > 0.01:  # Floating point guard
            self.pin_request(request, length_of_pin)
            del self.requests[request.request_id]
            return

    self.kv_cache_manager.free(request)
    del self.requests[request.request_id]

The set_up_pin() method in ToolCallEstimator implements the cost-benefit decision:

# estimate_with_func.py [lines 150-159]
FIXED_THRESHOLD_CONTINUUM = 2.0  # seconds

def set_up_pin(self, request: Request) -> float:
    if request.this_func_call is None:
        return 0

    this_func_call_exec_time = self.get_func_call_exec_time(
        request.this_func_call) or 0.0

    if this_func_call_exec_time > FIXED_THRESHOLD_CONTINUUM:
        return 0          # Tool too slow -- don't pin, evict cache

    return FIXED_THRESHOLD_CONTINUUM  # Pin for the threshold duration

4.2 TTL Expiry and Regular Unpinning

At the start of every scheduling step, unpin_requests_regular() checks the wall-clock time against each pinned request's expiry. If the TTL has elapsed and the job has not yet returned to the waiting queue, the cache is freed.

# scheduler.py [lines 248-256]
def unpin_requests_regular(self) -> None:
    waiting_job_ids = [req.job_id for req in self.waiting]

    for request, end_time in self.pinned_requests:
        # Unpin if: (1) job not back in waiting AND (2) TTL expired
        if request.job_id not in waiting_job_ids and time.time() >= end_time:
            self.unpin_request(request, end_time)

4.3 Tool-Call Duration Prediction

How Prediction Works

The ToolCallEstimator maintains a dictionary mapping command names (e.g., "ls", "git", "python") to their observed execution times. When a request returns from a tool call, the estimator updates the running average for that command.

# estimate_with_func.py [lines 136-147]
def update_func_call_exec_time(self, job_id: str) -> None:
    last_departure_time = self.job_to_history[job_id][-1]["departure_time"]
    func = self.job_to_history[job_id][-1]["func_call"]
    exec_time = time.time() - last_departure_time

    if func not in self.record_func_call_to_exec_time:
        self.record_func_call_to_exec_time[func] = [exec_time]
    else:
        self.record_func_call_to_exec_time[func].append(exec_time)
    # Running average across all observed executions of this command
    self.func_call_to_exec_time[func] = (
        sum(self.record_func_call_to_exec_time[func]) /
        len(self.record_func_call_to_exec_time[func])
    )

The ToolCallParser extracts the tool name from LLM output using regex on markdown bash code blocks -- the same format used by mini-swe-agent:

# estimate_with_func.py [lines 73-92]
class ToolCallParser:
    def parse(self, text: str) -> Optional[str]:
        # Same regex as mini-swe-agent
        actions = re.findall(r"```bash\s*\n(.*?)\n```", text, re.DOTALL)
        if len(actions) == 1:
            bash_action = actions[0].strip()
            words = bash_action.split()
            if words:
                return words[0]  # e.g., "ls", "git", "find"
        return None

4.4 Program-Level FCFS Scheduling

ContinuumRequestQueue Design

Unlike standard FCFS (which orders by individual request arrival time), the Continuum queue orders by job-level first entry time. Each job_id (representing an agent session) is assigned the timestamp of its first request, and all subsequent turns from that job inherit that priority. Additionally, jobs whose KV cache is currently pinned are given scheduling priority.

# request_queue.py -- ContinuumRequestQueue.peek_request() [lines 241-273]
def peek_request(self, pinned_requests, kv_cache_manager, connector):
    if not self:
        raise IndexError("peek from an empty queue")

    pinned_request_job_id_set = {req.job_id for req, _ in pinned_requests}

    # Phase 1: Prioritize requests with pinned KV cache (warm cache)
    earliest_request = None
    earliest_entry_time = float('inf')
    for request in self:
        if request.job_id in pinned_request_job_id_set:
            job_entry_time = self.job_id_first_entry_time.get(
                request.job_id, request.arrival_time)
            if job_entry_time < earliest_entry_time:
                earliest_entry_time = job_entry_time
                earliest_request = request
    if earliest_request is not None:
        return earliest_request    # Use warm cache -- avoid re-prefill

    # Phase 2: Fall back to job-level FCFS
    for request in self:
        job_entry_time = self.job_id_first_entry_time.get(
            request.job_id, request.arrival_time)
        if job_entry_time < earliest_entry_time:
            earliest_entry_time = job_entry_time
            earliest_request = request
    return earliest_request

4.5 Continuum-Aware Preemption

When the scheduler cannot allocate KV cache blocks for a new token, it must preempt a running request. The Continuum policy avoids preempting requests that are on their last step (about to finish), preferring to evict the most recently arrived job:

# scheduler.py [lines 195-234]
def pop_running_request_based_on_last_step(self, request):
    if len(self.running) <= 1:
        # If only one running req, try to reclaim from pinned_requests
        # (evict the pin with the latest end_time)
        latest_pin_end_request = None
        latest_pin_end_time = -float('inf')
        for req, end_time in self.pinned_requests:
            if end_time > latest_pin_end_time:
                latest_pin_end_time = end_time
                latest_pin_end_request = req
        if latest_pin_end_request is not None:
            self.pinned_requests.remove((latest_pin_end_request, latest_pin_end_time))
            return latest_pin_end_request, True  # is_unpin=True

    # Prefer to evict non-last-step requests with latest entry time
    latest_request = None
    latest_entry_time = -float('inf')
    for req in self.running:
        job_entry_time = self.running_job_id_first_entry_time.get(req.job_id)
        if job_entry_time > latest_entry_time and not req.is_last_step:
            latest_entry_time = job_entry_time
            latest_request = req

Paper-Level Claims (from arXiv:2511.02230)

Key Claims in the Paper

  • 58.2% of end-to-end delay in baseline vLLM comes from scheduling bubbles during tool-call pauses.
  • Continuum eliminates scheduling bubbles by retaining KV cache during predicted-short tool calls.
  • Program-level FCFS ensures fairness across multi-turn agent sessions rather than individual requests.
  • The approach is complementary to CPU offloading (tested with LMCache connector).
  • Evaluated on SWE-Bench Verified and BFCL traces with Llama-3.1 8B and 70B.

CPU offloading integration YES LMCache connector support via --kv-transfer-config. Scheduler handles connector for external KV cache tokens.
Eliminates 58.2% scheduling bubbles PLAUSIBLE The 2.16x average speedup on 500 jobs is consistent with this claim. The Continuum_Recorder logs all timestamps needed to verify this, but the repo does not include a script that calculates bubble percentage directly.

Key Gap: Simplified TTL Model. The README explicitly states: "This repository contains a modified version of vLLM with Continuum-style scheduling support (without the estimation in the paper)." The sophisticated cost-benefit optimization described in the paper (which models prefill token cost, memory pressure, and request queue depth) is replaced by a simple fixed-threshold check. This is a significant simplification -- the core algorithmic contribution of the paper is only partially implemented.
Despite the simplification, the approach still achieves strong results. The fixed 2.0-second threshold appears to work well for SWE-Bench traces where most tool calls (shell commands) are fast. The per-command averaging is present and functional, suggesting the estimation infrastructure is ready for a more sophisticated model.

Implementation Maturity Assessment

Strengths

  • Clean integration into vLLM V1 scheduler architecture
  • All metadata plumbed end-to-end from client to scheduler
  • Full experiment pipeline (agent, server, analysis)
  • Real benchmark results for 500 SWE-Bench jobs
  • Comprehensive event recording for analysis
  • CPU offloading support via LMCache

Limitations

  • Fixed 2.0s TTL threshold (paper's cost-benefit model not implemented)
  • Multiple TODO comments indicating work-in-progress
  • Linear scan over pinned_requests (O(n) per scheduling step)
  • ToolCallParser only handles bash code blocks (SWE-Bench specific)
  • No automated test suite for Continuum-specific logic
  • Some debug print statements left in protocol.py
