1. 問題背景

傳統 LLM serving（如 chatbot）是一問一答：request 進來 → 生成 tokens → 結束 → 釋放 KV cache。 但 Agentic workloads（如 SWE-bench coding agent）是多輪的： 

// 一個 Agent job 的生命週期
[LLM 推理] → 輸出 bash 指令 → [Agent 執行 tool call] → 結果回傳 → [LLM 推理] → ... (重複 N 輪)

在 Agent 執行 tool call 的期間（可能幾毫秒到幾十秒），LLM 沒在用這個 job 的 KV cache， 但如果釋放了，下一輪就要重新 prefill（浪費 GPU 時間）。如果不釋放，又佔住寶貴的 VRAM。

核心問題

Tool call 執行期間，要不要保留 KV cache？保留多久？

  • 保留（pin）：下一輪 re-enter 時零 prefill，但佔 VRAM
  • 釋放（free）：VRAM 給其他 request，但下一輪要重新 prefill（代價高）

Continuum 的做法：根據 tool call 的歷史執行時間，動態決定 pin 的時間長度（TTL）。

2. Continuum 策略概覽

Continuum 在 vLLM 的 v1 scheduler 中新增了 SchedulingPolicy.CONTINUUM， 核心邏輯分為三個部分：

A. Tool Call 偵測

LLM 生成完 tokens 後，解析輸出中的 bash code block，萃取出 tool name（如 ls、git、pytest）。

ToolCallParser.parse()

B. 執行時間估計（CDF）

維護每個 tool name 的歷史執行時間，計算移動平均。與固定閾值（2.0s）比較，決定要不要 pin。

ToolCallEstimator.set_up_pin()

C. KV Cache Pin / 排程

被 pin 的 request 佔住 VRAM，優先被排入下一輪。TTL 到期後自動 unpin + free。

Scheduler.pin_request()
ContinuumRequestQueue.peek_request()

4. Tool Call 解析

檔案：estimate_with_func.py:64-92

4.1 解析流程

當 LLM 輸出 tokens 結束後，ToolCallEstimator.request_finished() 被呼叫。 它做三件事：

  1. Detokenize：tokenizer.decode(request.output_token_ids) → 純文字
  2. Parse：ToolCallParser.parse(output_text) → 用正則萃取 tool name
  3. Record：存入 job_to_history[job_id]

4.2 ToolCallParser 實作

class ToolCallParser:
    def parse(self, text: str) -> Optional[str]:
        # 用 mini-swe-agent 相同的 regex 解析 bash code block
        actions = re.findall(r"```bash\s*\n(.*?)\n```", text, re.DOTALL)

        if len(actions) == 1:
            bash_action = actions[0].strip()
            words = bash_action.split()
            if words:
                return words[0]  # 回傳第一個 word = tool name

        return None

範例

LLM 輸出解析結果
```bash\ngit status\n```"git"
```bash\nfind . -name "*.py" | head -20\n```"find"
```bash\npytest tests/ -x -v\n```"pytest"
(沒有 bash block)None
限制：只萃取 tool name（第一個 word），不看完整指令和引數。 同一個 tool name（如 grep）的不同用法（快速的 grep pattern file vs 慢速的 grep -rn pattern /） 會被視為同一種工具，共用同一個平均執行時間。

4.3 完整的 request_finished() 流程

# estimate_with_func.py:177-205
def request_finished(self, request: Request) -> None:
    this_func_call = None

    if self.tokenizer is not None and len(request.output_token_ids) > 0:
        # 1. Detokenize
        output_text = self.tokenizer.decode(
            request.output_token_ids, skip_special_tokens=True
        )
        # 2. Parse tool call
        this_func_call = self.parser.parse(output_text)

    # 3. 存入 request 和 history
    request.this_func_call = this_func_call
    self.job_to_history[request.job_id].append({
        "departure_time": time.time(),
        "func_call": request.this_func_call
    })

5. CDF 時間估計

檔案：estimate_with_func.py:94-159

5.1 資料結構

class ToolCallEstimator:
    def __init__(self, ...):
        # tool name → 平均執行時間（秒）
        self.func_call_to_exec_time: dict[str, float] = {}

        # tool name → 所有歷史執行時間樣本
        self.record_func_call_to_exec_time: dict[str, list[float]] = {}

        # job_id → 事件序列 [{arrival_time, departure_time, func_call}, ...]
        self.job_to_history: dict[str, list[dict]] = {}

5.2 執行時間更新

當一個 job 的 tool call 執行完、重新進入 scheduler 時：

# estimate_with_func.py:136-147
def update_func_call_exec_time(self, job_id: str) -> None:
    # 從 history 中取得上一次的 departure_time 和 func_call
    last_departure_time = self.job_to_history[job_id][-1]["departure_time"]
    func = self.job_to_history[job_id][-1]["func_call"]

    # exec_time = 現在時間 - 上次 departure
    exec_time = time.time() - last_departure_time

    # 累積到歷史樣本
    self.record_func_call_to_exec_time[func].append(exec_time)

    # 更新移動平均
    self.func_call_to_exec_time[func] = (
        sum(self.record_func_call_to_exec_time[func])
        / len(self.record_func_call_to_exec_time[func])
    )

exec_time 計算時序圖


  ── LLM 推理 ──┐                         ┌── LLM 推理 ──
                 │                         │
           departure_time             arrival_time (重新進入)
                 │                         │
                 └──── tool call 執行 ─────┘
                 │←─── exec_time = now - departure_time ───→│

5.3 Pin 決策

固定閾值：FIXED_THRESHOLD_CONTINUUM = 2.0 秒

# estimate_with_func.py:150-159
def set_up_pin(self, request: Request) -> float:
    if request.this_func_call is None:
        return 0  # 沒偵測到 tool call → 不 pin

    this_func_call_exec_time = self.get_func_call_exec_time(request.this_func_call) or 0.0

    if this_func_call_exec_time > FIXED_THRESHOLD_CONTINUUM:
        return 0     # 預估執行時間 > 2s → 不 pin（太慢，不值得佔 VRAM）

    return FIXED_THRESHOLD_CONTINUUM  # 預估 ≤ 2s → pin 2 秒

Pin 決策邏輯表

條件回傳效果
this_func_call is None 0 沒偵測到 tool call → 立即 free KV cache
歷史平均 exec_time > 2.0s 0 Tool 太慢 → 立即 free KV cache
歷史平均 exec_time ≤ 2.0s 2.0 Tool 夠快 → pin KV cache 2 秒
第一次見到此 tool（無歷史） 2.0 預設 exec_time = 0 → pin 2 秒（樂觀策略）
論文原意：Continuum 論文使用 CDF（累積分佈函數）來決定 TTL。 但 vllm-continuum 的實作簡化為：平均值 vs 固定閾值 2.0 秒。 Pin 的時間也是固定的 2.0 秒，而非根據 CDF 動態計算。

6. Pin / TTL 機制

檔案：scheduler.py:237-262

6.1 Pin

# scheduler.py:237-239
def pin_request(self, request: Request, length_of_pin: float) -> None:
    self.continuum_recorder.request_pinned(request)           # 記錄事件
    self.pinned_requests.append((request, time.time() + length_of_pin))  # 存入 list

pinned_requests 是 list[Tuple[Request, float]]， 每個 tuple 是 (request, end_time)。
Pin 之後，request 的 KV cache blocks 不會被 free，佔住 VRAM 直到 TTL 到期。

6.2 TTL 自動到期 (Unpin)

# scheduler.py:248-256
def unpin_requests_regular(self) -> None:
    # 在每次排程步驟開始時呼叫
    waiting_job_ids = [req.job_id for req in self.waiting]

    for request, end_time in self.pinned_requests:
        if request.job_id not in waiting_job_ids and time.time() >= end_time:
            self.unpin_request(request, end_time)

Unpin 的兩個條件（必須同時滿足）：

  1. request.job_id not in waiting_job_ids — job 還沒回來（tool 還在執行）
  2. time.time() >= end_time — TTL 已到期

如果 job 已經回到 waiting queue，就不 unpin（因為馬上要排程用它的 cache）。

6.3 Unpin 實作

# scheduler.py:241-244
def unpin_request(self, request: Request, end_time: float) -> None:
    self.pinned_requests.remove((request, end_time))   # 從 list 移除
    self.continuum_recorder.request_unpinned(request)   # 記錄事件
    self.kv_cache_manager.free(request)                 # 真正釋放 KV blocks

6.4 _free_blocks() — Pin 的觸發點

當 request 完成 decode（生成完 tokens）後：

# scheduler.py:1349-1370
def _free_blocks(self, request: Request):
    # 先清理：如果這個 job_id 之前有 pin，先 unpin
    for req, end_time in self.pinned_requests:
        if req.job_id == request.job_id:
            self.unpin_request(req, end_time)

    # Continuum 策略 + 不是最後一步 → 考慮 pin
    if self.policy == SchedulingPolicy.CONTINUUM and not request.is_last_step:
        length_of_pin = self.tool_call_estimator.set_up_pin(request)

        if length_of_pin > 0.01:   # 浮點誤差保護
            self.pin_request(request, length_of_pin)
            del self.requests[request.request_id]
            return           # PIN — 不 free KV blocks

    # 不 pin 的情況 → 正常釋放
    self.kv_cache_manager.free(request)
    del self.requests[request.request_id]
is_last_step 很重要：如果 Agent 標記這是最後一步（job 完成了）， 就不 pin。否則會永遠佔住 VRAM，因為不會再有下一輪 request 來 re-enter。

7. ContinuumRequestQueue 排程佇列

檔案：request_queue.py:220-339

Continuum 不使用標準的 FCFS 佇列，而是用 ContinuumRequestQueue。 排程優先級：Pinned 的 job 優先 → 否則 job-level FCFS。

7.1 peek_request() — 核心排程邏輯

# request_queue.py:241-273
def peek_request(self, pinned_requests, kv_cache_manager, connector) -> Request:
    pinned_job_ids = {req.job_id for req, _ in pinned_requests}

    # === 優先級 1: Pinned request（KV cache 還在 VRAM 中）===
    earliest_request = None
    earliest_entry_time = float('inf')
    for request in self:
        if request.job_id in pinned_job_ids:
            job_entry_time = self.job_id_first_entry_time.get(request.job_id)
            if job_entry_time < earliest_entry_time:
                earliest_entry_time = job_entry_time
                earliest_request = request
    if earliest_request is not None:
        return earliest_request   # 有 pinned job → 優先排

    # === 優先級 2: Job-level FCFS ===
    # 選 job_id_first_entry_time 最早的 request
    earliest_request = None
    earliest_entry_time = float('inf')
    for request in self:
        job_entry_time = self.job_id_first_entry_time.get(request.job_id)
        if job_entry_time < earliest_entry_time:
            earliest_entry_time = job_entry_time
            earliest_request = request
    return earliest_request

為什麼要 job-level FCFS？

一個 job 會多次進出 scheduler（每輪 tool call 後 re-enter）。 如果用 request-level FCFS，新進來的 request 會排在很後面，對老 job 不公平。 Job-level FCFS 按 job 第一次進入的時間排序，確保長期公平性。

7.2 add_request()

def add_request(self, request: Request) -> None:
    # 記錄每個 job_id 第一次出現的時間
    if request.job_id not in self.job_id_first_entry_time:
        self.job_id_first_entry_time[request.job_id] = request.arrival_time
    self.append(request)

8.2 Eviction 優先級

# scheduler.py:195-234
def pop_running_request_based_on_last_step(self, request) -> (Request, bool):

    # 情況 A: 只有 1 個 running request → 只能從 pinned list 搶
    if len(self.running) <= 1:
        # 選 pin end_time 最晚的（保留最久的，最不緊急）
        return (latest_pin_end_request, True)  # True = is_unpin

    # 情況 B: 多個 running → 選非 last_step 且 entry_time 最晚的
    for req in self.running:
        if not req.is_last_step and job_entry_time > latest:
            latest_request = req
    if latest_request:
        return (latest_request, False)

    # 情況 C: 全部都是 last_step → 只能選最晚的
    return (latest_request, False)

Eviction 優先級摘要

優先級被 evict 的對象原因
1 (最先被 evict) Running 中 is_last_step=False 且 entry_time 最晚 不是最後一步 = 後面還有 tool call，可以之後重來
2 Running 中 entry_time 最晚（不管 is_last_step） 最後手段
3 (只有 1 running) Pinned list 中 end_time 最晚的 從 pin 搶回 VRAM

12.A — 深入討論：Trace 是怎麼生成和使用的？

Step 1：用 GPT-5 收集 Trace

論文 Section 3.1：「We collect and analyze 100 traces from mini-swe-agent running SWE-Bench and 100 traces from BFCL V4, both running GPT-5 as the base model.」

流程是：

mini-swe-agent (GPT-5 as backend)
  → LLM 推理 → 輸出 bash command → 真的在 Docker sandbox 裡執行 → tool output 回傳
  → LLM 推理 → ... 重複 ~10 輪
  → 記錄每一輪的 (prompt, response, tool_call, tool_output, exec_time)

這些 trace 記錄了 GPT-5 作為 coding agent 的完整對話歷史， 包括每一輪的 prompt token 數、output token 數、tool call 指令、和 tool 執行時間。

Step 2：用 Llama 在這些 Trace 上做 Evaluation？

論文 Section 6.1：「For results other than the real SWE-bench experiments in Figure 11, we evaluate on two collected workloads running GPT-5 and using poisson distribution for the arrival pattern.」

但 Repo 中沒有 trace replay 機制。 查看 mini-swe-agent 的程式碼，agent 的執行流程是： 

# agents/default.py — 真實的 agent loop，不是 replay
def run(self, task):
    self.add_message("system", system_prompt)
    self.add_message("user", task)
    while True:
        self.step()   # → query() → execute_action()

def step(self):
    response = self.query()              # 1. 送 prompt 給 LLM → 拿到 response
    action = self.parse_action(response) # 2. 從 response 解析 bash command
    output = self.env.execute(action)    # 3. 真的在 Docker 裡執行
    self.add_message("user", output)      # 4. tool output 加回 context

self.env.execute(action) 是真的在 Docker/bubblewrap sandbox 裡跑 bash 指令， 不是模擬。沒有任何 code path 可以「載入 GPT-5 trace 然後跳過 LLM 推理」。

兩種可能的實驗方式（論文沒有明確說明）

可能性做法問題
A. Llama 跑真的 Agent 用 Llama 作為 LLM backend 真正跑 SWE-bench。
GPT-5 trace 只用於分析工作負載特性（Table 1, Fig 3, 5）。
Llama 自己生成 response、自己的 output token 數。 		Llama-8B 的 coding 能力遠不如 GPT-5，會產出完全不同的 tool call sequence。
論文 Footnote 2 也承認「Base small models often fail to accomplish the task」。
但對 scheduling 實驗而言可能不影響： scheduler 只看 token 數和 tool exec time，不看程式碼正確性。
B. 用 GPT-5 trace 驅動 replay 把 GPT-5 的對話歷史餵給 Llama，控制 prompt 長度和 turn 數。
可能用某種方式固定 Llama 的 output token 數（如 max_tokens）。 		Repo 中沒有這個機制。沒有 trace loader、沒有 replay mode。
如果存在，可能在未開源的內部工具中。

output token 數的問題

如果是方式 A（Llama 跑真的 agent），output token 數完全由 Llama 決定。 同一個 SWE-bench instance：

  • GPT-5 可能一次輸出 200 tokens（精準的 patch + 解釋）
  • Llama-8B 可能輸出 50 tokens（錯誤的嘗試）或 500 tokens（冗長的 debug 過程）

這意味著論文 Table 1 的統計（Token Per Program: 70,126 ± 19,732）是 GPT-5 的數據， 不一定代表 Llama 實驗時的真實 token 數。

如果是方式 B（replay），則需要解決：

  • Prompt 長度：可以控制 — 直接餵 GPT-5 trace 中的對話歷史給 Llama
  • Output token 數：難以精確控制 — max_tokens 只能設上限，不能保證 Llama 剛好生成跟 GPT-5 一樣多的 tokens
  • Tool call 內容：如果 Llama 生成不同的 bash command → tool exec time 不同 → 整個後續 flow 發散

結論

論文沒有明確說明 Fig 8-10 的實驗中 Llama 到底是不是在 replay GPT-5 trace。 Repo 中只有真實 agent loop（方式 A），沒有 replay 機制（方式 B）。 最可能的情況是：

  • Fig 8-10（emulated）：用 Llama 跑真實 SWE-bench agent，GPT-5 trace 只用於 workload 特性分析
  • Fig 11（real）：在 Tensormesh 上跑真實 SWE-agent（可能用更強的模型）
  • Llama 跑出來的 token 數、tool call 序列跟 GPT-5 不同，但論文用 Table 1 的 GPT-5 統計來描述 workload 特性

這是一個方法論上值得質疑的點：如果 workload 統計來自 GPT-5 但實驗跑的是 Llama， 那 Table 1 的數據不代表實際實驗的工作負載。

14.1 演算法落差

論文描述Repo 實作落差
CDF：用 empirical CDF P(τ,f) = Σ𝟙[t≤τ] / |S[f]| 估計 tool call 完成機率 只有簡單平均 avg(samples) CDF 沒有實作
Cost Model：τ* = argmax P(τ,f) × Benefit(r) - Cost(τ,r) 固定閾值 if avg > 2.0s → free, else pin 2.0s Cost model 沒有實作
Benefit = CacheMissCost + OutofOrderCost 沒有 benefit 計算 沒有實作
Cost = MemUsage(r)/M × τ 沒有 cost 計算 沒有實作
Memoryfulness factor η = -Corr(k, N-k) 沒有 沒有實作
動態 TTL：每個 request 根據 tool call + cost model 算出不同的 τ* 全部 pin 固定 2.0 秒 嚴重簡化
Cold-start：|S[f]| ≤ K 時用全局 CDF，再退到 T_default 第一次見到的 tool → exec_time=0 → 直接 pin 2s（樂觀策略） 不同的 cold-start 策略
Offline profile：prefill vs context length 擬合二次曲線，測 GPU-CPU bandwidth 沒有 offline profiling 程式碼 缺失

15. 分析方法

檔案：continuum_exp/analyze.py

15.1 JCT 計算

# 每個 job 的 JCT:
JCT = max(departure_times) - min(arrival_times)  # 含所有輪次

# 全局 wall time:
wall_time = max(所有 job 的 departure) - min(所有 job 的 arrival)

15.2 統計指標

指標計算方式
num_jobs有完整 arrival + departure 的 job 數量
average_duration所有 JCT 的算術平均
median_durationP50
percentile_95P95（線性內插）
percentile_99P99（線性內插）
min_duration / max_duration最快 / 最慢的 job
total_duration全局 wall time

15.3 可衍生的 Cache 指標

從 scheduler_timestamps 的 waiting_to_running 事件中可算出：

指標計算意義
Cache Hit Ratio hit_length / prompt_length KV cache 命中率（pin 成功 → hit_length ≈ prompt_length）
Prefill Savings 1 - (prompt_length - hit_length) / prompt_length 節省了多少 prefill 計算
Queue Wait Time waiting_to_running - Request_arrival_time 在佇列中等了多久
Pin Duration unpinned_time - pinned_time KV cache 被 pin 了多久
