AI Metropolis: Out-of-Order Execution for Multi-Agent LLM Simulations

arXiv:2411.03519 MLSys 2025 Multi-Agent Out-of-Order Execution Simulation 1.3-4.15x Speedup

Zhiqiang Xie, Hao Kang, Ying Sheng, Tugrul Konuk, Yilong Zhao, Lianmin Zheng, Kurt Keutzer, Ion Stoica, Zhuohan Li

UC Berkeley & NVIDIA

Overview

The Problem: Sequential Agent Bottleneck

Multi-agent LLM simulations (e.g. Generative Agents, Stanford Town) process agents strictly in order each simulation step. Since each agent issues one or more LLM calls, the total latency grows linearly with the number of agents. In a 25-agent "Stanford Town" scenario, sequential execution means every step must wait for all 25 LLM calls to finish serially—even when most agents' outputs are independent.

Key Insight: Instruction-Level Parallelism for Agents

AI Metropolis borrows the concept of out-of-order (OoO) execution from CPU microarchitecture. Just as a CPU identifies independent instructions and executes them in parallel despite a sequential program order, AI Metropolis identifies independent agent actions and executes them concurrently. A lightweight dependency check ensures correctness: if agent B's action depends on agent A's prior result, B waits for A; otherwise B proceeds in parallel.

Core Contribution

A general framework that enables speculative and out-of-order execution of multi-agent LLM simulations. The system provides a correctness guarantee: the final simulation trace is identical to what a fully sequential execution would produce, while achieving 1.3-4.15x wall-clock speedup.

Key Mechanisms

1. Dependency Analysis

Before each simulation step, the system analyzes which agents may read or write shared state (e.g. a location, an object). Agents that operate on disjoint state are placed into independent groups that can execute concurrently. This static analysis can be augmented with a speculative mode where all agents are launched optimistically.

2. Speculative Execution

In aggressive mode, all agents are issued in parallel. Each agent takes a snapshot of the world state. After all agents return, a conflict detector checks if any agent read state that another agent concurrently modified. Conflicting agents are rolled back and re-executed with the updated state.

3. Conflict Detection

Uses a read-write set tracking mechanism similar to software transactional memory (STM). Each agent's execution records which memory locations (state variables) it reads and writes. After a batch, the system checks for WAR, WAW, and RAW conflicts between concurrent agents.

4. Rollback & Replay

When conflicts are detected, the affected agents' state changes are undone using the pre-execution snapshots. These agents are then re-executed sequentially with the correct, updated world state. The rollback cost is bounded: at worst, the system degrades to sequential execution.

5. LLM Batching Integration

By launching multiple agents simultaneously, the system naturally forms larger batches for the LLM backend. This improves GPU utilization—sequential execution underutilizes the GPU by sending single requests at a time. Batched execution achieves better throughput even without parallelism benefits from reduced total time.

// AI Metropolis execution loop pseudocode for each simulation_step: agents = get_all_agents() groups = analyze_dependencies(agents, world_state) for each independent_group in groups: snapshots = save_state(world_state) results = execute_parallel(independent_group, world_state) // batch LLM calls conflicts = detect_conflicts(results) if conflicts: rollback(world_state, snapshots, conflicts) replay_sequential(conflicting_agents, world_state) commit(results, world_state)

Throughput vs. Latency

The speedup comes from two orthogonal sources: (1) parallelism—independent agents run concurrently, and (2) batching—concurrent LLM requests form larger batches with better GPU utilization. On an 8-GPU setup, the batching benefit alone accounts for ~1.5x of the total speedup, with the remaining speedup from true parallelism.

Evaluation Benchmarks

The paper evaluates AI Metropolis on two distinct simulation frameworks that represent different agent coupling patterns and world-state structures.

Benchmark 1 — Generative Agents (Smallville)

A social simulation with 25 agents living in a small virtual town, as introduced by Park et al. (2023). Each agent has a biography, daily schedule, and memory stream. Agents move around locations, interact with objects, and occasionally talk to each other. Most agents operate in separate locations per step, yielding a naturally low conflict rate (~5–15%). AI Metropolis achieves 3–4x speedup on this benchmark.

Benchmark 2 — GameOfLife (Economic Agents)

A cellular-automata-inspired economic simulation where agents represent market participants that read and update shared price/inventory cells. Unlike Smallville, agents in GameOfLife frequently access overlapping state, making conflict rates higher (20–40%). The OOO approach still delivers 1.5–2.5x speedup because even with moderate conflicts, batching alone provides a substantial GPU utilization gain.

Scaling Experiment: 8 → 64 Agents

A key evaluation varies agent count from 8 to 64 in the Smallville setting (low-conflict). The results show near-linear speedup growth up to 32 agents, after which LLM backend saturation begins to limit further gains. At 64 agents the maximum observed speedup is 4.15x.

Ablation: Parallelism vs. Batching Contribution

To disentangle the two speedup sources, the authors run an ablation where agents execute sequentially but their LLM calls are still batched into groups (batch-only baseline). This baseline achieves ~1.5x speedup over true sequential execution. The remaining speedup (up to 4.15x ÷ 1.5x ≈ 2.8x) comes from true concurrency — agents running in parallel. This confirms that both batching and concurrency are essential contributors, with concurrency dominating at higher agent counts.

Correctness Verification The authors verify correctness by running each simulation in both sequential and OOO modes with deterministic (temperature=0) LLM inference and comparing the full simulation traces. Across all evaluated scenarios, OOO execution produces byte-identical traces to sequential execution, confirming the serializability guarantee holds in practice.

Significance & Impact

For Multi-Agent Research

Enables scaling simulations to hundreds of agents without proportional latency increase. Researchers studying emergent social behaviors, economic simulations, or multi-agent planning can iterate 2-4x faster, dramatically accelerating experimental cycles.

Correctness Guarantee

Unlike ad-hoc parallelization, AI Metropolis provides a formal serializability guarantee: the output trace is identical to a valid sequential execution. This makes it a drop-in replacement for existing sequential simulators with zero accuracy risk.

Architectural Insight

Demonstrates a powerful cross-pollination from computer architecture to AI systems. The Tomasulo/out-of-order execution paradigm, which has driven CPU performance for decades, is shown to be equally applicable to agent-based LLM workloads.

Practical Deployment

The framework is designed to integrate with existing LLM serving systems (e.g., vLLM, TGI). The conflict detection and rollback overhead is negligible compared to LLM inference time, making the approach practical for real-world multi-agent applications.

Scope Limitation The speedup is bounded by the inherent conflict rate of the simulation. Tightly coupled multi-agent scenarios (e.g., agents all interacting in the same room) see diminished returns due to frequent rollbacks and sequential replays. The paper positions AI Metropolis as a complement to, not a replacement for, careful simulation design: developers should still structure simulations to minimize unnecessary coupling between agents wherever the domain permits.

OOO Execution Pipeline (Step-by-Step)

The following walk-through traces one full simulation step under AI Metropolis, from speculative dispatch through conflict resolution.

Step 1 — World-State Snapshot
Checkpoint current state

Before any agent executes, the Simulation Step Manager takes an immutable snapshot of every shared state variable (location maps, object inventories, NPC memories). These snapshots serve as rollback targets if a conflict is later detected.

Step 2 — Dependency Oracle Query
Build read/write sets; group independent agents

The Dependency Oracle inspects each agent's planned action (parsed from its system prompt or prior context) and predicts the set of state keys the agent will read (RS) and write (WS). Agents whose RS∪WS are disjoint can be placed in the same speculative group. In practice, two agents share a conflict if WS_A ∩ RS_B ≠ ∅ (RAW), RS_A ∩ WS_B ≠ ∅ (WAR), or WS_A ∩ WS_B ≠ ∅ (WAW).

Step 3 — Speculative Parallel Dispatch
Fire all agents; batch LLM calls

Agents in the same speculative group are dispatched concurrently. Their LLM prompts are collected into a single batched inference request sent to the LLM backend (e.g., vLLM). Each agent is given the snapshotted world state as context; they do not see each other's in-flight writes. Execution proceeds speculatively — agents act on potentially stale reads.

— results return; conflict check —
No Conflict → Commit

All actual RS/WS sets are disjoint. Writes are committed atomically to world state. Step completes — equivalent to a perfectly serialized execution.

Conflict → Rollback + Replay

Conflicting agent's writes are discarded. Its state is restored from the pre-step snapshot. The agent is re-run sequentially with the now-correct world state (including all committed writes from earlier agents in the ordering).

Step 5 — Commit & Advance Clock
All agents settled; advance simulation clock

Once every agent has either committed or replayed successfully, the simulation clock advances by one tick. The resulting world state is provably identical to the state that would have been produced by sequential execution in the canonical agent ordering.

Serializability Proof Sketch The rollback-and-replay mechanism is the key to correctness. Because any agent that observed a stale value is replayed with the fully committed state, the final outcome is equivalent to processing agents in strict sequential order. The system effectively runs agents speculatively and falls back to sequential semantics on demand — a classic optimistic concurrency control strategy.

The paper's central intellectual contribution is mapping a 40-year-old CPU design technique onto LLM-based multi-agent systems. Understanding this mapping is key to appreciating both the elegance and the limits of the approach.

Key Difference from CPU OOO

In a CPU, instruction latency is nanoseconds and misprediction penalties are tens of cycles — both negligible in absolute terms. In AI Metropolis, each "instruction" (LLM call) takes hundreds of milliseconds, and a rollback re-runs the full LLM inference. This makes the penalty-to-benefit ratio much more sensitive to conflict rate. A CPU can tolerate 10–15% misprediction rates because the penalty is tiny; AI Metropolis becomes inefficient at conflict rates above ~40% because each rollback is expensive.

Implementation & System Design

Simulation Engine Architecture

AI Metropolis implements a three-layer architecture. The top layer is the Simulation Manager, responsible for global clock progression and world-state bookkeeping. The middle layer is the Scheduler, which runs the dependency oracle, forms speculative groups, and coordinates rollback. The bottom layer is the Agent Executor, which wraps each agent's LLM prompt construction, inference dispatch, and action parsing.

# Simplified Python-level structure of AI Metropolis engine class SimulationManager: world_state: WorldState # shared mutable environment agents: List[Agent] scheduler: OOOScheduler def run_step(self): snapshot = self.world_state.checkpoint() groups = self.scheduler.form_groups(self.agents, self.world_state) for group in groups: results = self._dispatch_parallel(group) # batched LLM calls conflicts = self.scheduler.detect_conflicts(results) for agent in conflicts: self.world_state.restore(snapshot, agent.write_keys) self._dispatch_sequential(agent) # replay self.world_state.commit(results) class OOOScheduler: oracle: DependencyOracle # predicts RS/WS per agent def detect_conflicts(self, results) -> List[Agent]: # Check actual (post-hoc) RS/WS sets for RAW / WAR / WAW for a, b in pairs(results): if a.ws & b.rs or a.rs & b.ws or a.ws & b.ws: yield later_in_canonical_order(a, b)

LLM Serving Integration

The Agent Executor interfaces with the LLM backend via a batching adapter. Because speculative groups can contain many agents, requests are naturally coalesced into a single batch. This is implemented as a simple async queue: agents submit their prompts, and the queue flushes to the serving layer once the whole group is ready. With vLLM as the backend, this directly exploits continuous batching for higher GPU utilization.

State Checkpointing Overhead

Snapshots use copy-on-write semantics for the world-state dictionary. Only keys that are written during a step need a deep copy; read-only keys share a reference. In practice, fewer than 5% of state keys are written per step in typical social simulations, making checkpoint overhead sub-millisecond compared to LLM inference times of hundreds of milliseconds.

Oracle Accuracy Trade-off A more conservative oracle (predicts more potential conflicts) reduces rollbacks at the cost of smaller speculative groups and less parallelism. A more optimistic oracle (predicts fewer conflicts) maximises group size and parallelism but may trigger more rollbacks. AI Metropolis defaults to an optimistic oracle with post-hoc conflict validation, which empirically maximises throughput across the evaluated benchmarks.

Limitations & Future Work

Tight-Coupling Ceiling

When agents are densely coupled (e.g., many agents interacting in a single location), conflict rates rise above ~50%, forcing most agents through rollback-and-replay. The effective speedup collapses toward 1x. For such scenarios the OOO overhead (snapshotting, conflict detection) can marginally exceed the sequential baseline.

Non-Deterministic LLM Outputs

Rollback replays the agent with a different (corrected) world-state context. If the LLM uses temperature > 0, the replayed response may differ from what the agent would have produced under the original speculative state, even after correcting for the conflict. This means the serializability guarantee holds only under deterministic (temperature=0) inference, or must be interpreted as "equivalent to some valid sequential ordering" rather than a fixed one.

Oracle Quality Dependency

The dependency oracle must accurately predict which state keys an agent will access. For open-ended LLM agents where actions are determined by free-form natural language outputs, predicting the write set before execution is fundamentally imprecise. Mis-predictions either cause unnecessary rollbacks (false positives) or missed conflict detection (false negatives, violating correctness).

Memory Scaling for Snapshots

Copy-on-write snapshots are efficient for small shared state, but simulations with very large world states (e.g., city-scale open worlds with millions of objects) may face memory pressure. Each concurrent group requires a separate snapshot; with many groups running in a pipelined fashion, snapshot memory can multiply quickly.

Multi-Step Lookahead

The current system executes agents speculatively within a single simulation step but does not pipeline across steps. Future work could explore multi-step speculative lookahead — executing step T+1 agents while step T is still resolving conflicts — analogous to a superscalar out-of-order CPU pipeline that spans multiple instruction windows.

Heterogeneous Agent Costs

All agents in the current design are treated with equal priority and expected to have similar LLM call costs. In practice, agents may vary in prompt length, required reasoning depth, and the number of tool calls they issue. A future scheduling policy could weight agents by estimated cost to form better-balanced speculative groups and reduce the tail latency within each batch.

Research Directions

  • Learned dependency oracle: train a small model to predict RS/WS sets from agent context, reducing both false positives and false negatives.
  • Hierarchical OOO: apply out-of-order execution at multiple granularities — individual LLM calls within an agent, agents within a group, groups within a step.
  • Distributed simulation: partition world state across multiple nodes, extending OOO execution to a distributed setting where network communication replaces in-process snapshotting.
  • Adaptive mode switching: automatically detect when conflict rate is too high and switch to sequential execution for that step, avoiding OOO overhead in pathological cases.
  • Integration with long-horizon planning: extend speculative execution to cover multi-step agent plans, enabling agents to speculatively commit to action sequences rather than single-step actions.
Scope Note AI Metropolis targets simulation workloads where correctness is defined by serializability with respect to a canonical sequential ordering. It is not designed for adversarial or competitive multi-agent settings where the order of execution is itself part of the game mechanics, nor for real-time interactive applications where rollback latency would be user-visible.
