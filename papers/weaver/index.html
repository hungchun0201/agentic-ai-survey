Overview

What is Weaver?

Weaver is a serving system designed for multi-LLM workloads that uses attention offloading to improve resource utilization when serving multiple large language models concurrently on shared GPU infrastructure.

In production environments, organizations often deploy multiple LLMs of different sizes and specializations (e.g., a large general model, a small fast model, a domain-specific model). Weaver addresses the inefficiency of running these models independently by intelligently offloading attention computation between co-located models to better utilize GPU compute and memory resources.

Key Insight When multiple LLMs are served on the same GPU cluster, their attention computation phases create opportunities for resource sharing. One model's idle compute during memory-bound decode can be used to accelerate another model's compute-bound prefill or attention layers.

Problem: Multi-LLM Serving Inefficiency

Challenges in Multi-Model Deployment

Serving multiple LLMs simultaneously introduces several resource management challenges:

  • GPU memory fragmentation: Each model requires its own weight memory and KV cache space, leading to underutilized memory across models.
  • Compute imbalance: Different models at different phases (prefill vs. decode) have asymmetric compute demands that waste GPU cycles.
  • Attention bottleneck: Attention computation is the dominant cost in transformer inference, yet opportunities for sharing are unexploited.
  • Isolation overhead: Running each model in isolation on dedicated GPUs leads to low average utilization across the fleet.

Core Technique: Attention Offloading

What is Attention Offloading?

Attention offloading is the process of delegating attention computation from one model's serving instance to another when the target instance has spare compute capacity. This is feasible because:

  • Attention is a well-defined, self-contained operation (Q, K, V matrices in, output out)
  • The attention computation interface is standardized across transformer models
  • During decode, GPUs are often memory-bandwidth-bound with idle compute units
  • The overhead of transferring attention tensors can be amortized over the compute savings

Why Attention Specifically? In transformer-based LLMs, attention computation is the most memory-intensive operation (due to KV cache) and becomes the bottleneck as sequence lengths grow. By offloading attention to GPUs with spare capacity, Weaver can "weave" computation across model boundaries, hence the name.

Design Considerations

Offloading Decision Logic

Weaver must carefully decide when and where to offload attention:

Factor Consideration Impact
Transfer cost Bandwidth to move Q/K/V tensors between GPUs Must be less than compute savings
Donor load Available compute cycles on receiving GPU Offload only when donor has genuine spare capacity
Memory pressure KV cache storage on donor GPU Cannot evict donor's own KV cache
Latency sensitivity SLO requirements of both models Offloading must not degrade donor's latency
Model compatibility Head dimensions and attention patterns Standardized attention kernel interface needed

Each scheduling cycle, Weaver evaluates the load across all model instances and determines optimal offloading decisions. The scheduler is designed to be non-disruptive: a model's own requests always take priority over offloaded work from other models.

Advantages of Weaver's Approach

  • Higher GPU utilization: Spare compute on lightly-loaded model instances is reclaimed for productive attention work
  • Cost efficiency: Fewer GPUs needed for the same multi-model throughput compared to isolated deployment
  • Transparent to models: Each model receives correct attention results regardless of where computation occurred
  • Dynamic adaptation: Offloading decisions adjust in real time as load patterns shift across models
  • No model modification: Works with standard transformer architectures via a unified attention computation interface

Unique Position Weaver is distinguished from other multi-model serving systems by its focus on attention-level offloading rather than coarser-grained model placement or request routing. This fine-grained approach enables sharing at the operator level, which can extract more utilization from shared GPU resources.
