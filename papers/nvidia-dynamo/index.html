Smart Router & KV-Aware Routing

The router tracks two metrics per worker -- cached block overlap and decode load -- to make optimal placement decisions.

How Routing Decisions Work

For every incoming request, the router tokenizes the input, hashes it into blocks, and queries the global KVIndexer (a prefix-tree / radix-tree) to find overlap with each worker's cached blocks.

The cost function combines prefill cost (tokens to recompute) with decode load (active sequences on each worker):

Cost FunctionFORMULA
cost = overlap_score_weight * prefill_blocks + decode_blocks

// Where:
// prefill_blocks = (input_tokens - overlap * block_size) / block_size
// decode_blocks  = active sequences on the worker
// Lower cost = better routing choice

// Example with overlap_score_weight = 1.0:
Worker 1: cost = 1.0 * 8 + 10 = 18
Worker 2: cost = 1.0 * 5 + 5  = 10  // ← selected
Worker 3: cost = 1.0 * 2 + 9  = 11

Block Tracking Mechanisms

●

Active Decoding Blocks

Tracked locally by the router throughout the request lifecycle. Incremented on new request, updated during token generation, decremented on completion.

●

Cached Blocks (KVIndexer)

Maintained globally via KVPublisher/KVIndexer. Workers emit KV cache events; the indexer builds a radix prefix tree providing accurate overlap information for every routing decision.

  • Block hashing includes LoRA adapter name for multi-adapter correctness
  • Supports SGLang RadixAttention and vLLM PagedAttention
  • Softmax temperature sampling for load distribution tuning

KV Block Manager (KVBM)

Hierarchical KV cache management across GPU, CPU, SSD, and remote storage -- extending effective context length far beyond GPU memory.

KVBM Hierarchical Memory Tiers
■

KvBlockManager Orchestration

Central coordinator across all memory tiers. Generic over host and device backends, tracking block locations, deduplicating by sequence hash, and managing the full lifecycle.

  • DevicePool, HostPool, DiskPool, RemoteStorage
  • Scheduler gates transfers relative to model iteration progress
  • TransferManager with per-path async queues
■

Block Layout & Memory Mapping

Each block is a 2D array [num_layers][page_size x inner_dim]. FullyContiguous layout stores all layers in one region with alignment-aware stride computation.

  • Typed block handles: mutable (being computed) and immutable (complete)
  • Metadata tracks priority, sequence_hash for dedup
  • Layer-separated or contiguous layout options
■

Transfer Mechanisms

Async TransferManager orchestrates D2H, H2D, H2Disk, Disk2D with per-path queues. NIXL abstracts transport: NVLink, IB/UCX, POSIX, GDS.

  • Device offload: CUDA D2H or custom kernel copy
  • Disk offload: NIXL Write via POSIX, GDS when available
  • Remote: opaque blob store via NIXL abstraction
Planner: SLA-Driven Autoscaler

Profiles workloads, predicts traffic, and right-sizes prefill and decode GPU pools to meet latency SLAs at minimum TCO.

Scaling Algorithm Pipeline

Step 1

Metric Collection

Every adjustment_interval seconds, queries Prometheus for average TTFT, ITL, request count, ISL, and OSL.

Step 2

Correction Factors

Computes prefill_correction = actual_ttft / expected_ttft and decode_correction = actual_itl / expected_itl to adapt profiles to real traffic.

Step 3

Load Prediction

Forecasts next_num_req, next_isl, next_osl using one of four predictors: Constant, ARIMA, Kalman, or Prophet.

Step 4

Replica Calculation

Computes prefill_replicas from predicted load / interpolated throughput; decode_replicas from corrected ITL target at predicted context length.

Step 5

Non-Blocking Execution

Calls connector.set_component_replicas(). Scaling is non-blocking -- the planner continues monitoring while replicas adjust.

PredictorAlgorithmBest For

📈
Correction factors adapt to reality. Prefix cache hits lower actual TTFT vs. profiled; bursty traffic raises it. The planner learns these gaps and adjusts replica targets accordingly. Request queueing, chunked prefill in decode, and metric variance are all automatically compensated.
Replica CalculationPython
# Prefill replicas
predicted_load = next_requests * next_isl / interval \
                 * min(1, prefill_correction)
prefill_replicas = ceil(
    predicted_load / interpolated_throughput / gpus_per_engine
)

# Decode replicas
corrected_itl = target_itl / decode_correction_factor
throughput_per_gpu = decode_interpolator.find_best_throughput_per_gpu(
    itl=corrected_itl,
    context_length=next_isl + next_osl / 2
)
decode_replicas = ceil(
    next_num_req * next_osl / interval
    / throughput_per_gpu / gpus_per_engine
)
Disaggregated Prefill / Decode Serving

Separates prefill and decode into independently scalable GPU pools. Each phase runs on hardware tuned for its workload.

Disaggregated Serving Request Flow

NIXL Direct GPU-to-GPU Transfer

KV cache transfers use NIXL for direct VRAM-to-VRAM copy over the optimal available transport: NVLink, InfiniBand/UCX, or RoCE. Transfers are non-blocking, allowing GPU forward passes to continue serving other requests.

Runtime-Reconfigurable xPyD

Workers can be added and removed at runtime. New workers register with the discovery service and publish RuntimeConfig. The router automatically discovers and incorporates them into routing decisions.

Backend-Specific Metadata

SGLang: Uses bootstrap_info for RDMA bootstrap. Prefill runs as background task -- decode begins while KV transfer proceeds. vLLM/TRT-LLM: Synchronous prefill with block IDs and opaque state.

Agentic Inference Features

Per-request hints for latency priority, expected output length, and cache pinning TTL. Built for multi-turn agent workflows.

🤖
Why this matters for agents: Agentic workflows involve multi-turn conversations where the same context prefix is reused across many calls. Cache pinning TTL via nvext.cache_control prevents eviction between turns. LangChain and NeMo Agent Toolkit integrations provide drop-in support for agent frameworks.
Cache Pinning via nvext.cache_controlPython
import openai

client = openai.OpenAI(base_url="http://dynamo-frontend:8000/v1")

response = client.chat.completions.create(
    model="Qwen/Qwen3-32B",
    messages=[
        {"role": "system", "content": long_system_prompt},
        {"role": "user",   "content": "Analyze this document..."},
    ],
    extra_body={
        "nvext": {
            # Pin KV cache for 300s -- survives between agent turns
            "cache_control": {"ttl": 300},
            # Hint for expected output length (improves decode scheduling)
            "expected_output_len": 512,
            # Latency priority: lower = more urgent
            "priority": 1,
        }
    },
    stream=True,
)

Quick Start

Three ways to get started -- container, PyPI, or Kubernetes.

Option A: Container

Fastest pathBash
docker run --gpus all --network host --rm -it \
  nvcr.io/nvidia/ai-dynamo/sglang-runtime:1.0.1

# Inside the container
python3 -m dynamo.frontend --http-port 8000 \
  --discovery-backend file > /dev/null 2>&1 &
python3 -m dynamo.sglang \
  --model-path Qwen/Qwen3-0.6B \
  --discovery-backend file &

curl -s localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen3-0.6B",
       "messages":[{"role":"user",
       "content":"Hello!"}]}'

Option B: PyPI Install

pip installBash
pip install "ai-dynamo[sglang]"
# or [vllm] or [trtllm]

Then start the frontend and a worker as shown in the container example. See the full installation guide for system dependencies.
