Sarathi-Serve: Chunked Prefills

Piggybacking decode tokens onto chunked prefill batches to convert memory-bound decoding into compute-bound operation, eliminating pipeline bubbles and improving LLM serving throughput by up to 1.91x.

OSDI 2024 Microsoft Research arXiv: 2308.16369 Amey Agrawal, Ashish Panwar, Jayashree Mohan et al.

Two Key Techniques

Chunked-prefills: Split a long prefill request into equal-sized compute-saturating chunks instead of processing the entire prompt at once. This creates uniform work units.
Decode-maximal batching: Use a single prefill chunk to construct a hybrid batch, filling remaining slots with decode tokens that "piggyback" on the prefill's compute. This converts memory-bound decode into compute-bound operation.

Core Problem Addressed

Prefill saturates GPU compute at batch size 1. Decode costs 200x more per token at batch size 1. The fundamental mismatch between prefill and decode phases creates pipeline bubbles and GPU underutilization.

Key Insight

Since prefill and decode share the same weight matrices for linear operations, they can be fused into a single matrix multiplication. Decode tokens computed alongside prefill add near-zero marginal cost -- weights are already loaded from memory.

2. The Problem: Prefill-Decode Mismatch

2.1 Per-Token Cost Asymmetry

The decode cost per token is 200x, 100x, and 16.7x that of prefill at batch sizes 1, 2, and 18 respectively (LLaMA-13B, A6000). This extreme asymmetry means:

  • Prefill: matrix-matrix multiplications (compute-bound, high arithmetic intensity)
  • Decode: vector-matrix multiplications (memory-bound, low arithmetic intensity)
  • Single prefill request saturates GPU, but decode needs batch size 256+ to saturate

3. Chunked Prefills

3.1 Mechanism

A prefill request with prompt length P is split into ceil(P / C) chunks of size C. Each chunk is processed in a separate iteration. The key challenge is correctly setting the attention mask so that each chunk can only attend to tokens from the current and previous chunks -- not future chunks.

3.3 Overhead Analysis

Reduced Arithmetic Intensity

Smaller chunks have lower arithmetic intensity for linear operations. Chunk size 256 loses ~12.5% prefill throughput vs full prefill. Chunk size 512 loses ~10%.

Repeated KV-Cache Reads

Subsequent chunks must re-read KV vectors from prior chunks during attention. For N chunks, chunk i reads KV from chunks 0..i-1. This overhead is small because attention is a minor fraction of total compute.

4.2 Why It Works: Fused Linear Operations

For linear operations (preproj, postproj, ffn_ln1, ffn_ln2), prefill and decode share the same weight tensors W. By concatenating their activations, both are computed in a single GEMM call:

// Without decode-maximal batching:
prefill_out = matmul(prefill_activations, W)    // shape [B,C,H] x [H,H]
decode_out  = matmul(decode_activations, W)     // shape [D,1,H] x [H,H] -- weight reloaded!

// With decode-maximal batching (SARATHI):
fused_input = concat(prefill_chunk, decode_tokens)  // shape [1,C+D,H]
fused_out   = matmul(fused_input, W)                 // shape [1,C+D,H] x [H,H]
//   Weight loaded ONCE. Decode tokens computed at marginal cost ~ 0.

5. Choosing the Ideal Chunk Size

5.1 P:D Ratio Analysis

The P:D ratio (prefill-to-decode token ratio) determines the optimal chunk size. A prefill of size P yields P/C chunks, each piggybacking B-1 decodes. Peak Sarathi performance occurs when the number of chunks equals the decode iterations:

// Peak performance condition
P:D = C / (B - 1)

// Example: chunk_size=256, batch_size=18
Optimal P:D = 256 / (18 - 1) = 15.06
// At this ratio, ALL decodes can piggyback with prefill chunks

5.2 Chunk Size Tradeoffs

Larger Chunks (e.g. 512)

5.3 Tile Quantization Effect

Critical detail: GPUs compute matmuls by partitioning into tiles (typically 128 tokens). When the total batch dimension (chunk_size + decode_count) is not a multiple of the tile size, wasted compute occurs. For example, going from 128 to 129 tokens increases iteration time by 32%. Sarathi ensures chunk + decode count aligns to tile boundaries.

7.3 Comparison with Orca Scheduling

Orca's iteration-level scheduling processes full prefills in one shot, incidentally overlapping with some ongoing decodes. Sarathi consistently outperforms Orca's best-case with throughput gains of 1.27x (1K), 1.25x (2K), and 1.23x (3K) across all sequence lengths, because chunked prefills create more piggybacking opportunities than Orca's opportunistic overlap.

8. Tradeoffs and Relation to PD Disaggregation

8.1 Sarathi vs. Prefill-Decode Disaggregation

Sarathi Approach: Co-located PD

  • Prefill and decode on the same GPU
  • Fused matrix multiplications eliminate decode overhead
  • No KV-cache transfer between machines
  • Uniform batches reduce pipeline bubbles
  • Better for moderate P:D ratios (10-50)

Disaggregation (DistServe/Splitwise)

  • Separate GPU pools for prefill and decode
  • Each pool independently optimized
  • Requires KV-cache transfer (network overhead)
  • Better for extreme P:D ratios or strict SLOs
  • Higher hardware cost (separate pools)
Key tradeoff: Sarathi's chunked prefills can delay ongoing decode requests during the prefill chunk processing. Disaggregation avoids this by isolating the two phases. The Sarathi-Serve follow-up paper (arXiv:2403.02310) explores this tradeoff in detail, showing that for most production workloads, co-located serving with chunked prefills remains more cost-effective.

8.2 Adoption and Legacy

Chunked prefills have become a standard technique in production LLM serving systems. The technique has been adopted by:

  • vLLM: enable_chunked_prefill flag, default in many configurations
  • NanoFlow (OSDI'25): uses Sarathi's chunked prefill policy for request scheduling
  • SGLang: implements chunked prefills as a core scheduling primitive
  • TensorRT-LLM: supports chunked context mode

8.3 Limitations

Assumed Uniform Requests

The analysis assumes each request has the same number of prefill and decode tokens. Real-world requests have highly variable lengths, which complicates optimal chunk size selection.

Sequence Length Scaling

Evaluated on sequences up to 3K and P:D ratios 1-200. Very long sequences (10K-100K+) introduce new challenges as attention cost grows quadratically.

9. Mathematical Model of Decode-Maximal Batching

9.1 Batch Size Constraints

The maximum permissible batch size B is constrained by GPU memory:

B = floor( (M_G - M_S) / (L * m_kv) )

// M_G  = available GPU memory (e.g., 48GB for A6000)
// M_S  = model parameter memory footprint
// L    = maximum sequence length the model supports
// m_kv = memory per KV pair per token

// Example: LLaMA-13B on A6000 (48GB)
// M_G = 48GB, M_S = 26GB, L = 2048, m_kv = ~1.2MB
// B = floor(22GB / (2048 * 1.2MB)) = 18

9.2 Piggybacking Coverage

With chunk size C and batch size B, a single prefill request generates P/C chunks. Each chunk can piggyback B-1 decode tokens. The total number of piggybacked decodes per prefill is:

total_piggybacked = (P / C) * (B - 1)

// For ALL decodes to piggyback (ideal case):
// D = P/C * (B-1), where D is total decode tokens
// Therefore: P:D = C / (B-1)

// Example: C=256, B=18
// Optimal P:D = 256/17 = 15.06
// At this ratio, SARATHI achieves peak 1.27x throughput

9.3 Decode Efficiency Formula

The per-token decode cost in decode-maximal batching can be approximated as:

// Marginal decode cost = additional attention cost only
decode_cost_sarathi = T_attn(C + D) - T_attn(C)
                     + epsilon  // negligible linear op overhead

// vs baseline where all linear ops must be re-executed:
decode_cost_baseline = T_linear(D) + T_attn(D)

// Since T_linear dominates (>80% of total time):
// decode_cost_sarathi / decode_cost_baseline ≈ 1/10 to 1/200

10. Pipeline Parallelism Integration Details

10.1 Three Types of Pipeline Bubbles

PB1: Variable Prefill Length

Different requests have different prompt sizes. In a pipeline, stage completion times vary, creating bubbles between consecutive micro-batches. Chunked prefills eliminate this by making all chunks the same size C.

PB2: Prefill-Decode Mismatch

A prefill stage followed by a decode stage has vastly different compute times. The decode stage finishes much faster, leaving the GPU idle waiting for the next micro-batch. Hybrid batches equalize compute.

PB3: Variable Decode Context

Decode attention cost grows with context length (accumulated KV-cache). Different requests at different stages of generation have different decode times. Uniform batching mitigates this variance.

11. Implementation Details and Reproducibility

11.1 Dynamic Chunking Schedule

Beyond fixed chunk sizes, Sarathi supports dynamic chunking that adapts chunk size over a request's lifetime. The scheduler computes equally spaced chunk sizes between low_chunk_size and high_chunk_size, aligned to multiples of 32. Early chunks are larger (higher prefill efficiency) while later chunks are smaller (more piggybacking for accumulated decodes):

# Dynamic chunk schedule computation
chunk_sizes = np.linspace(
    low_chunk_size,     # e.g., 128
    high_chunk_size,    # e.g., 512
    num_stages,         # e.g., 4 stages
)[::-1]               # Reverse: start large, end small
# Result: [512, 384, 256, 128]
# Stage determined by: num_processed_tokens / tokens_per_stage

11.3 Arithmetic Intensity Analysis

The fundamental asymmetry between prefill and decode is captured by arithmetic intensity (FLOPs per byte of memory accessed):

  • Prefill (batch=1, seq=512): All operations have high arithmetic intensity (1000-3000). GPU fully utilized.
  • Decode (batch=1): Arithmetic intensity drops by 100-200x. Operations become memory-bound vector-matrix multiplications.
  • Decode (batch=256): Only at this extreme batch size does decode begin to saturate compute. But KV-cache memory makes this infeasible (e.g., max batch = 18 for LLaMA-13B at seq_len=1024).

11.4 Comparison with Subsequent Work

Sarathi-Serve (2024, follow-up)

The authors' subsequent paper (arXiv:2403.02310) extends Sarathi with a taming of the latency-throughput tradeoff. It demonstrates that with proper stall-free scheduling, chunked prefills can serve as an alternative to full prefill-decode disaggregation at lower cost.

Influence on vLLM

vLLM (v0.3+) adopted chunked prefills as enable_chunked_prefill=True. This became the default configuration for many production deployments, validating Sarathi's core contribution. NanoFlow (OSDI'25) also builds on Sarathi's chunked prefill policy for request scheduling.

Broader impact: Chunked prefills are now considered a standard primitive in LLM serving systems, alongside continuous batching (Orca) and paged attention (vLLM). Sarathi's insight that mixing prefill and decode in the same batch can transform memory-bound decode into compute-bound operation has become foundational knowledge in the systems-for-ML community.

Serving Capacity Translation

A 1.33x throughput improvement directly translates to serving 33% more concurrent users on the same hardware, or equivalently, a 25% reduction in GPU costs for the same QPS target.

Pipeline Parallel Impact

The 1.91x improvement with pipeline parallelism means Sarathi makes PP a viable strategy for LLM inference, whereas without it, PP barely improves over TP-only deployments due to bubbles.

Decode Efficiency

The up to 10x decode speedup means that in decode-dominated workloads (like long-form generation), Sarathi provides the most dramatic improvements -- precisely where traditional systems are most wasteful.
