Overview

Concur identifies middle-phase thrashing as a dominant, previously uncharacterized performance pathology in offline agentic batch inference. Unlike classical memory thrashing from capacity exhaustion, middle-phase thrashing is a prolonged inefficiency period where cache efficiency collapses as long-lived agents accumulate state over time.

Key Insight: KV Cache as a Shared Resource

Concur draws a rigorous analogy between GPU KV cache and network bandwidth: both are shared, finite resources where overcommitment leads to congestion (cache thrashing / packet loss). Applying the Additive Increase Multiplicative Decrease (AIMD) algorithm from TCP congestion control, Concur dynamically regulates agent concurrency to prevent overcommitment while maximizing utilization.

Three Contributions

1. Middle-Phase Thrashing Discovery

Identifies a three-phase execution pattern (Warmup / Middle-Phase Thrashing / Cooldown) in agentic batch inference, where the middle phase dominates >90% of execution time with near-zero cache hit rates.

2. Agent-Level Admission Control

A lightweight middleware layer between agent execution frameworks and LLM serving engines. Regulates admission at agent granularity rather than request granularity, preserving execution continuity.

3. Cache-Aware AIMD Algorithm

Adapts TCP's AIMD congestion control for KV cache management. Uses KV cache usage and hit rate as feedback signals to dynamically adjust the admission window size.

During middle-phase thrashing, KV cache usage stays near saturation (80-100%) while hit rate drops precipitously, trapping the system in eviction-recomputation cycles.

Phase 1: Warmup

Agents operate at shallow reasoning depths with small, similar KV footprints. High prefix sharing yields ~90% cache hit rates. Increasing concurrency improves throughput monotonically.

Phase 2: Middle-Phase Thrashing

As agents advance to mid-horizon steps, KV cache saturates. The system enters a vicious cycle of eviction and recomputation. Adding more agents paradoxically degrades throughput. Extra recomputation accounts for 49.1% of end-to-end latency.

Phase 3: Cooldown

Agents complete workflows, releasing memory. Eviction frequency decreases and cache hit rates partially recover. The system exits the thrashing regime.

Why This Differs from Classical Thrashing

Classical memory thrashing arises from simple capacity exhaustion. Middle-phase thrashing is more insidious: it manifests as a prolonged period where the system looks busy (high GPU utilization) but most work is wasted on recomputation. The root cause is that agents progress asynchronously -- while some generate tokens, others wait for tools, and their idle KV cache entries lose LRU recency and get evicted.

Why AIMD Works Here

Network recomputation cost is O(1) per retransmission. But KV cache recomputation is O(L^2) due to the quadratic cost of re-prefilling long contexts. This makes the penalty for congestion far worse than in networking, making multiplicative decrease even more critical -- the system must exit the high-cost regime in O(log N) steps to minimize cumulative wasted compute.

Execution Workflow (Four Stages)

① Agent Admission

Agents submit generation requests to the controller. Based on system state, the controller admits or pauses the agent, preserving execution state.

② Batched Generation

Admitted agents proceed to the serving engine for batched token generation, consuming GPU-resident KV cache proportional to context length.

③ Tool Execution

Agents invoking tools become temporarily inactive but retain logical KV cache associations. The controller restricts new admissions during high-pressure regimes.

④ Feedback & Update

The controller monitors KV cache usage (U_t) and hit rate (H_t), updating the admission window W_t via the AIMD-inspired algorithm.

Cache-Aware Admission Control Algorithm

The AIMD Control Law

The admission window W_t is governed by a state-dependent transition function based on two real-time feedback signals:

Window Update Rule
W_{t+1} = W_t + alpha if U_t < U_low   (Linear Exploration)
W_{t+1} = W_t * beta if U_t > U_high AND H_t < H_thresh   (Multiplicative Decrease)
W_{t+1} = W_t otherwise   (Stabilization)

Linear Exploration (alpha = 2)

When underutilized (U_t < U_low = 0.2), the system probes for additional capacity with an additive increase. This avoids exponential overshoots while gradually maximizing GPU utilization.

Multiplicative Decrease (beta = 0.5)

When thrashing is detected (high usage + low hit rate), the window is halved. This exits the O(L^2) recomputation regime in O(log N) steps, rapidly reducing wasted compute. The quadratic cost penalty makes multiplicative cuts essential.

Stabilization & Saturation Buffer

The gap between U_low (0.2) and U_high (0.5) acts as a buffer zone. Long-context agents cause discrete memory spikes, and this buffer prevents instant penalty triggering. Healthy high-utilization runs continue unthrottled.

Agent-Level Control Primitives

// Three minimal primitives operating on agent lifecycles

def admit(agent):
    """Authorize agent's next generation step."""
    agent.state = "active"
    send_to_engine(agent.generation_request)

def pause(agent):
    """Temporarily suspend agent between generation steps.
    Applied only at step boundaries -- never interrupts in-flight computation.
    Execution state is preserved for later resumption."""
    agent.state = "pending"

def resume(agent):
    """Reactivate a paused agent once resources are available.
    Coordinated by admission control to prevent renewed congestion."""
    agent.state = "active"
    send_to_engine(agent.generation_request)

AIMD Controller Implementation

class ConcurController:
    """Cache-aware admission control using AIMD."""

    def __init__(self):
        self.W       = 1         # Initial window size
        self.alpha   = 2         # Additive increase factor
        self.beta    = 0.5       # Multiplicative decrease factor
        self.U_low   = 0.2       # Low utilization threshold
        self.U_high  = 0.5       # High utilization threshold
        self.H_thresh = 0.2      # Hit rate threshold for congestion

    def update_window(self, U_t: float, H_t: float):
        """Update admission window based on runtime signals."""
        if U_t < self.U_low:
            self.W += self.alpha             # Probe for capacity
        elif U_t > self.U_high and H_t < self.H_thresh:
            self.W = int(self.W * self.beta)  # Cut window -- congestion!
        # else: maintain current window (stabilization zone)

Key Finding: Why HiCache Fails Despite High Hit Rates

SGLang with HiCache achieves very high hit rates (96-97%) by aggressively offloading KV cache to CPU. However, under high concurrency, the PCIe bandwidth becomes a bottleneck: simultaneous offload and reload operations contend for PCIe bandwidth, causing per-step latency to increase sharply. Concur achieves competitive hit rates without CPU offloading by simply controlling how many agents are active at once.

The Quadratic Cost Argument

Unlike network packet loss where retransmission has O(1) cost, KV cache recomputation incurs O(L^2) cost because the prefill operation must process all L tokens sequentially, with each token attending to all previous tokens. This makes the analogy asymmetric in an important way:

Cost of Cache Miss (Recomputation)

When an agent with context length L has its KV cache evicted, restoration requires re-processing all L tokens. The STP (space-time product) cost is proportional to L^2, making long-context evictions catastrophically expensive.

Why Multiplicative Decrease is Essential

With O(L^2) penalty per eviction event, the system must exit the congestion regime in O(log N) steps via multiplicative cuts (beta=0.5). Additive decrease would take O(N) steps, accumulating vastly more wasted compute during the transition.

Design Principles: Beyond Request-Level Scheduling

Concur argues that the system design must shift along two dimensions simultaneously:

Request-Level to Agent-Level Granularity

Existing serving engines treat each generation step as an isolated, stateless unit of work. This fails to capture long-term state continuity inherent in agentic workflows. Agent-level scheduling preserves execution context across steps, preventing premature cache eviction.

Reactive Eviction to Proactive Admission

Rather than reacting to memory pressure after it occurs (evict-then-recompute), Concur proactively regulates how many agents enter the system. This prevents cache overcommitment from ever happening, eliminating the eviction-recomputation cycle entirely.
