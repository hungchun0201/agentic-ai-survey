InferCept: Efficient Intercept Support for Augmented LLM Inference

The first LLM inference framework targeting augmented LLMs with tool-call interceptions. Introduces the Preserve operation to pin KV cache during external calls, combined with min-waste scheduling to deliver 1.6-2x higher throughput over vLLM.

Authors: Reyna Abhyankar*, Zijian He*, Vikranth Srivatsa, Hao Zhang, Yiying Zhang arXiv:2402.01869 Affiliation: UC San Diego Hardware: NVIDIA A100 GPUs

◈ Overview

Modern LLM applications increasingly augment language models with external tools -- calculators, search engines, APIs, image generators, and chatbot interfaces. When an LLM invokes such a tool, the inference process is intercepted: decoding pauses while waiting for the external entity to respond. Existing inference systems (vLLM, Orca, DeepSpeed) treat each interception as a termination signal, discarding the KV cache and resubmitting the full context as a new request when the tool returns. This Discard approach wastes 37-40% of total model forwarding time on redundant recomputation.

InferCept introduces three primitives for handling interceptions -- Discard Preserve Swap -- and a min-waste scheduler InferCept that dynamically selects the optimal strategy per request per interception. The key insight: no single strategy dominates across all workloads because interception durations vary from sub-millisecond (calculator APIs) to tens of seconds (chatbot, image generation).

⚙ Architecture & Preserve/Resume Flow

InferCept extends vLLM's iteration-level scheduler with a multi-queue architecture that tracks requests across five distinct lifecycle states. The central innovation is that intercepted requests are not terminated -- they transition through specialized queues that determine the optimal memory strategy.

Sequence State Transitions

▶ The Preserve Operation

The Preserve operation is InferCept's key contribution: when a tool call intercepts an LLM request, the KV cache blocks remain pinned in GPU memory. This eliminates all recomputation cost upon resumption. The trade-off is that pinned blocks cannot serve other requests during the interception window.

Discard (Baseline vLLM)

On interception: free all KV cache blocks. On resume: resubmit as new request, recompute entire context from scratch. Waste = recomputation time + iteration slowdown from extended prefills.

Wdiscard = Tfwd(C) × C × M + Tfwd(C) × Cother × M
Equation 1: Discard waste from recomputation + blocking other requests
Preserve (InferCept)

On interception: pin KV cache blocks in GPU. On resume: immediately continue decoding. Waste = occupied memory × interception duration. Best for short tool calls (math, QA, VE).

Wpreserve = TINT × C × M
Equation 2: Preserve waste = interception time × context size × per-token memory
Swap (InferCept)

On interception: move KV cache to CPU via PCIe. On resume: swap back in. Avoids both recomputation and memory occupancy, but swapping itself consumes bandwidth and blocks compute.

Wswap = 2 × Tswap(C) × Cbatch × M
Equation 3: Swap waste doubled for bidirectional transfer

# From scheduler.py - pause_seq_group()
def pause_seq_group(self, seq_group, blocks_to_swapout, first_n_blocks=-1):
    seq_group.sampling_params.api_call_time = time.monotonic()

    if self.scheduler_config.api_policy == 'P':    # Preserve mode
        mode = PreemptionMode.PRESERVE
    elif self.scheduler_config.api_policy == 'H':  # Heuristic mode
        if seq_group.sampling_params.api_exec_time >= self.scheduler_config.heuristic_coef:
            mode = PreemptionMode.SWAP     # Long tool call -> swap out
        else:
            mode = PreemptionMode.PRESERVE # Short tool call -> preserve

    # Preserve: simply record in paused dict, KV cache stays in GPU
    if mode is PreemptionMode.PRESERVE:
        self.paused[seq_group.request_id] = (seq_group, PreemptionMode.PRESERVE)
        # No block freeing! KV cache remains pinned in GPU memory
# From scheduler.py - resume_seq_group()
def resume_seq_group(self, request_id):
    seq_group, mode = self.get_paused_seq_group(request_id)
    del self.paused[request_id]

    if mode is PreemptionMode.PRESERVE:
        for seq in seq_group.get_seqs(status=SequenceStatus.RESUMED_API):
            seq.status = SequenceStatus.RUNNING  # Instant resume!
        self.running.append(seq_group)  # Back to running queue immediately
    elif mode is PreemptionMode.RECOMPUTE:
        # Must recompute: goes back to waiting queue
        for seq in seq_group.get_seqs(status=SequenceStatus.RESUMED_API):
            seq.status = SequenceStatus.WAITING
        self.waiting.insert(0, seq_group)  # Front of queue (ImprovedDiscard)

☃ Min-Waste Scheduling

InferCept's scheduler operates at iteration granularity, making per-request decisions that jointly minimize GPU memory waste. The system supports five policy modes, selectable via the --api-policy flag:

Policy FlagNameDescriptionImplementation
D Discard Free all KV blocks, resubmit as new request scheduler.py -- free(seq), append to waiting
P Preserve Pin KV blocks in GPU, instant resume scheduler.py -- store in paused dict
S Swap Move KV to CPU memory, swap back on resume block_manager.py -- swap_out/swap_in
H Heuristic Preserve if short, Swap if long (threshold-based) scheduler.py -- heuristic_coef comparison
V InferCept (Full) Min-waste with chunk-fill, greedy eviction, ILP solver scheduler_v2.py + solver.py

ILP Solver for Optimal Waste Minimization

The Solver class uses Gurobi to solve a non-convex integer program that jointly determines three decision variables per intercepted request: 

# From solver.py - Gurobi ILP for min-waste interception decisions
class Solver:
    def _solve(self, C, S_gpu, t_call, t_start, t_resumed, ...):
        m = gp.Model("model")
        m.setParam("NonConvex", 2)     # Allow non-convex quadratic
        m.setParam("TimeLimit", 0.025)  # 25ms budget per solve

        # Decision variables
        c_s = m.addVar(vtype=GRB.INTEGER, name="c_s")  # Blocks to swap per epoch
        c_d = m.addVar(vtype=GRB.INTEGER, name="c_d")  # Blocks to discard per epoch
        n_e = m.addVar(vtype=GRB.INTEGER, name="n_e")  # Number of resumption epochs

        # Objective: minimize total waste
        # w_p = preserve waste (pinned blocks x time)
        # w_d = discard waste (recomputation overhead)
        # w_s = swap waste (transfer time blocking)
        # w_o = overhead waste (iteration time increase)
        w_p = s_p * (t_resumed - t_call)
        w_d = (n_e + 1) / 2 * c_d * (t_resumed - t_start)
        w_s = (n_e + 1) / 2 * c_s * (t_resumed - t_start) + f_swap * S_gpu
        w_o = S_gpu * (swap_overhead + resume_overhead - f_normal) * n_e

        m.setObjective(w_p + w_d + w_s + w_o, GRB.MINIMIZE)
        m.optimize()

        return (c_s_val, c_d_val, n_e_val)  # Optimal split

⚠ Limitations Identified by Subsequent Work

While InferCept represents a significant advance, subsequent systems -- including Continuum, KVFlow, and ThunderAgent -- have identified fundamental limitations in InferCept's static preserve model. These limitations become particularly acute in real-world agentic workloads where tool-call behavior is unpredictable.

Central Problem: InferCept's Preserve operation makes a binary, irrevocable decision at interception time. Once KV cache is pinned, it remains pinned for the entire interception duration. There is no mechanism to dynamically reclaim preserved memory if the tool call takes unexpectedly long or fails entirely.
Limitation 1: Static Pin Fails with High-Variance Tool Calls

InferCept's heuristic (policy H) uses a fixed threshold (heuristic_coef) to decide between Preserve and Swap. This works when interception times are predictable, but real agentic workloads exhibit extreme variance:

AugmentationMean TimeVarianceProblem for Preserve
Math (Calculator)9e-5 sec6e-5Low risk -- preserve is optimal
QA (Wikipedia)0.69 sec0.17Network variance can spike
Virtual Environment0.09 sec0.014Stable, preserve works
Chatbot28.6 sec15.6Extremely high variance -- preserve pins GPU indefinitely
Image Generation20.0 sec7.8Long + variable -- GPU memory stranded
TTS17.2 sec7.6Unpredictable model execution time

When a tool call that was expected to take 2 seconds takes 30 seconds (e.g., a chatbot user goes idle), the preserved KV cache blocks thousands of tokens worth of GPU memory for the entire duration. Subsequent work proposes adaptive eviction and progressive downgrade (Preserve -> Swap -> Discard) based on elapsed time.

Limitation 2: Failed/Stalled Tool Calls Block GPU Memory Indefinitely

InferCept has no timeout or watchdog mechanism for preserved requests. If an external tool call hangs, crashes, or enters an infinite loop, the GPU blocks remain pinned in the self.paused dictionary indefinitely. The code does include passive_discard which can evict preserved blocks under memory pressure, but this is a reactive, last-resort mechanism rather than a proactive policy:

Limitation 3: Ignores Cumulative Queueing Delays

InferCept's waste equations treat each interception independently. However, in multi-step agentic workflows (ReAct, Chain-of-Thought with tools), a single request may trigger 5-15 sequential interceptions. Each Preserve operation reduces available GPU capacity, which delays other requests in the waiting queue. This creates a cascading queueing effect that InferCept's waste model does not account for:

  • Request A preserves 200 blocks -> Request B's prefill is delayed
  • Request B starts late, triggers its own tool call -> preserves 150 more blocks
  • Now 350 blocks are pinned -> Request C through F cannot be scheduled
  • The total system waste far exceeds the sum of individual W_preserve calculations

Subsequent systems like Continuum introduce a queueing-theoretic model that captures these cascading effects, and KVFlow proposes dynamic migration of KV cache between GPU and CPU based on real-time queue depth rather than per-request waste estimation.

Limitation 4: Duration Estimation is Inherently Fragile

InferCept's min-waste scheduler requires knowing (or estimating) T_INT -- the interception duration. The paper proposes a dynamic estimator: T_INT = t_now - t_call, which grows linearly with elapsed time. This achieves 93% of oracle performance on the paper's benchmark workloads but has fundamental weaknesses:

  • No distribution modeling -- assumes interception time is deterministic
  • Cannot distinguish "half done" from "just started and stalled"
  • The Gurobi solver has a 25ms time budget, limiting solution quality under load
  • Profiled constants (free_recompute_tokens=128, free_swap_tokens=976) are hardcoded, not dynamically adapted to changing workload characteristics

Notable Code Quality Observations

Hardcoded Constants

The codebase contains numerous # FIXME get this from profile comments alongside hardcoded values. The batch polynomial (1.3E-05, 0.328, 24.1), swap latency 4E-05 sec/token, and saturation points are all specific to A100 GPUs and would need re-profiling for any other hardware.

Debug Code in Production

solver.py contains active print() statements in the hot path. The SequenceGroup class has commented-out comparison operators. Several FIXME markers indicate incomplete features (e.g., self.resume_queue.correction_offset is initialized but never properly calibrated).

Dual Scheduler Architecture

The codebase maintains two parallel schedulers: Scheduler (scheduler.py) handles policies D/P/S/H, while SchedulerV2 (scheduler_v2.py) handles V/G policies with chunk-fill. The engine selects between them based on the api_policy flag. This dual architecture adds complexity but allows comparing approaches.

Gurobi Dependency

The ILP solver requires a commercial Gurobi license (or academic license). The 25ms time limit per solve call is tight for large workloads. The solver uses NonConvex=2 which enables heuristic solutions for the non-convex quadratic objective, meaning results are not guaranteed optimal.

What the Codebase Does Well

Strengths of the implementation:
  • Clean separation of concerns: each policy mode is independently testable via the --api-policy flag
  • The token layout in SequenceData is elegant -- the 6-field format supports arbitrary combinations of discard, swap, and preserve within a single sequence
  • The swap_out_from_back() method in BlockSpaceManager enables incremental eviction without full sequence swap
  • Comprehensive benchmark scripts in exps/ that sweep all policies across multiple QPS rates for reproducible comparisons
  • The Simulator layer enables deterministic benchmarking without actual model inference costs, isolating scheduling effects
