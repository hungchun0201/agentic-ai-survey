1. Background & Motivation

LLM-based multi-agent systems — math-solver panels, code reviewers, MMLU debate — typically route a shared task prompt through N agents with heavily overlapping context. Under a standard prefill pipeline each agent re-encodes that shared content, so total TTFT grows roughly linearly with the number of agents. This is wasteful because the underlying KV-cache is near-identical across agents.

KVCOMM (Yan et al., arXiv 2510.12872, codebase FastMAS/KVCOMM) addresses this with an online anchor-based offset approximation: each layer maintains a small pool of anchor KV vectors keyed by placeholder-id. When an incoming segment matches an anchor, KVCOMM reuses its KV and applies a learned offset instead of re-prefilling. The paper reports:

Paper claims we want to verify

  • Up to 7.82× TTFT speedup on H100 (5-agent · 1K input · 512 prefix · 512 output; 430 ms → 55 ms).
  • ≥70% KV-reuse rate across RAG, math, and code benchmarks.
  • No accuracy degradation vs the dense-prefill baseline.
  • Works online — the anchor pool adapts at serving time, no offline calibration.

Why this matters for our work

  • KVCOMM is a strong published baseline for our own KV-cache management method in Agentic_KVCache_management.
  • We need absolute TTFT numbers on our hardware (A100-40GB, H100, H200, L40S) — paper only reports H100.
  • Reuse rate and accuracy floor are pre-requisites before claiming our own method beats KVCOMM.
  • Identifying regimes where KVCOMM under-performs directly motivates the design of our approach.

4.1 Execution modes

KVCOMM exposes two CLI flags: --execution_mode default is the standard dense-prefill baseline (re-prefill every agent); --execution_mode allow_kv_reuse is KVCOMM with the anchor pool enabled. The paper's headline 7.82× speedup is computed across these two modes. Within a single allow_kv_reuse run, each agent request is logged as mode: dense_prefill (anchor miss → fell back to full prefill) or mode: kv_reuse (anchor hit → offset-reuse path).

7. Gaps & Data-Quality Issues

G1 · No dense baseline yet

All 19 jobs used allow_kv_reuse. Without matched default runs we cannot compute the paper's headline metric (end-to-end TTFT speedup). Highest-priority gap.

G2 · MMLU H200 anomaly

dense p50 428 ms — 3.6× slower than the H100 dense p50 (120 ms) on the same workload; p95 1.9 s. Likely node contention or missing warmup — must re-run to confirm.

G3 · A100 / L40S sparse

A100 HumanEval n = 76, A100 MMLU n = 27, L40S MMLU n = 59. All produced mid-timeout or post-error; insufficient for any claim.

G4 · GSM8K H200 cancelled

Job 6459779 has 1624 records but exited with TIME LIMIT. Numbers track H100's (consistent), but the run is not clean.

G5 · HumanEval H200 hit latency high

hit p95 143 ms vs H100's 48 ms — possibly co-located with the MMLU H200 anomaly root-cause.

G6 · Reuse rate below paper target

MMLU at ~51% is materially below the paper's ≥ 70% claim. Either our anchor-pool hyperparameters differ or MMLU's short prompts limit reuse opportunity — worth confirming.
