KVCOMM

Online Cross-context KV-cache Communication for Efficient LLM-based Multi-agent Systems

NeurIPS 2025 arXiv:2510.12872 Training-Free Anchor-Based Multi-Agent LLM Llama-3.1 / Qwen2.5

Hancheng Ye1, Zhengqi Gao2, Mingyuan Ma1, Qinsi Wang1, Yuzhe Fu1, Ming-Yu Chung1, Yueqian Lin1, Zhijian Liu3, Jianyi Zhang1, Danyang Zhuo1, Yiran Chen1

1. Problem: Multi-context Redundancy

In multi-agent LLM systems вҖ” AutoGPT, GPTSwarm, AgentPrune вҖ” several specialized agents exchange messages to solve complex tasks collaboratively (retrieval-augmented QA, math reasoning, tool-using code synthesis). Every time an agent receives a message from a predecessor, its prefill stage must re-encode the full conversation history, including passages or peer outputs the sender has already encoded.

In a fully-connected M-agent graph, each message is reprocessed by every downstream agent вҖ” so prefill compute scales as O(MВІ) even though the text overlap across agents is almost complete. Existing cache-reuse techniques either (a) reuse only shared prefixes (cannot handle context changes), (b) selectively recompute sensitive tokens (fixed "top-k" policy that fails when prefix drift shifts the sensitive set), or (c) compress the cache via bit-stream coding (leaves token-level compute untouched).

Each agent in Figure 2 of the paper starts with a role-specific system prompt (its prefix context). When the same shared text (say, a retrieval passage) is read by Agent 1 and Agent 2, the resulting KV-caches are different вҖ” not because the text differs, but because the rotary position encoding (RoPE) uses different absolute positions, and the attention soaked in different system prompts produces different Key/Value features.

Why naive reuse breaks: The KV offset cache_agent_2 - cache_agent_1 is sharp: different prefix contexts produce offset magnitudes orders of magnitude larger than the contextual deviation the cache should actually represent. Blindly copying cache entries between agents causes incoherent outputs; naГҜve position-shifting (the Prompt Cache / DroidSpeak family) still models sensitivity only at a token subset that drifts with prefix.

2. Core Insight: Offset Variance Across Agents

KVCOMM observes that although the raw KV-cache for a shared token differs across agents, the offset вҖ” the per-layer difference between the cache under a new prefix and the same cache under a reference prefix вҖ” is predictable from a small pool of examples. The paper calls this the offset-variance problem.

KV Proximity

Proposition 1 (в„“вӮӮ-Lipschitz bound): two tokens that are close in embedding space have close KV vectors across layers. Formally, ||k[n] вҲ’ kМғ[n]|| вүӨ C_RВ·C_KВ·вҲҸбөў ОІ(1+ОұПғВІ) В· ||h[n] вҲ’ hМғ[n]||.

Offset Proximity

Proposition 2: under the same prefix change, similar tokens have similar KV offsets. ||О”[n_a] вҲ’ О”Мғ[n_b]|| вүӨ C_RВ·C_KВ·вҲҸбөў ОІ(1+ОұПғВІ) В· Оҙ[n_a] вҖ” offsets inherit the Lipschitz property.

Practical Consequence

Practical consequence: given a few stored example offsets (anchors) plus their embedding-based similarity weights, the offset for a new shared segment can be interpolated without re-running prefill вҖ” a "translation in cache space".

Empirical validation: Figure 4a/b plots the KV distances across 32 layers of Llama-3.1-8B for tokens grouped by "near", "mid", and "far" in embedding space вҖ” the near group is consistently closer in KV space as well. Figure 4c/d confirms the same pattern for offsets. Spearman correlation between embedding distance and KV-offset distance stays above 0.8 across most layers, validating the anchor weighting by embedding proximity.

KVCOMM is a training-free, prompt-adaptive framework. It treats shared text in every incoming message as an approximate translation problem: the text already has a canonical KV-cache (under a base/no-prefix rendering); the job is to translate it to the current agent's prefix context by adding a predicted offset vector.

4. Theoretical Foundation

4.1 Cache-space Translation

Let k[n]_ПҶ be the Key cache for token n under prefix context ПҶ. KVCOMM decomposes the cache into a base rendering plus a context-dependent offset:

k[n]_ПҶ  =  k[n]_base  +  О”[n]_ПҶ          # Eq. 6 / 7 of paper
v[n]_ПҶ  =  v[n]_base  +  О”[n]_ПҶ

# Approximation via anchor interpolation
(kМӮ / vМӮ)_(m,i)^ПҶ  вүҲ  (k/v)_(m,i)^base
                    + ОЈ_ПҲ вҲҲ A_ПҶ  w_(ПҶвҶ’ПҲ)  В·  О”(k/v)_(m,i)^ПҲ

# Anchor weights: embedding-based softmax
w_(ПҶвҶ’ПҲ)  =  softmax( -||h_ПҶ - h_ПҲ|| )_(ПҲ вҲҲ A_ПҶ)

All operations are linear over the cache tensor, so the cost of computing О” is a tiny matrix multiply instead of a full transformer forward pass вҖ” the savings in prefill compute are the source of KVCOMM's speedup.

4.2 Positional Alignment is Indispensable

Before measuring the correlation between any two tokens' caches, KVCOMM solves the RoPE positional mismatch. If a token sits at position n in one prefix but at n+О” in another, the raw keys differ by an orthogonal rotation R_О” whose difference can be orders of magnitude larger than the contextual deviation we actually care about.

KVCOMM therefore de-rotates stored keys by R_(-О”) before every similarity measurement. This "key alignment via key rotation" is ablated in Table 5 of the paper вҖ” removing it drops MMLU accuracy from 68.0% to 43.1%.

5. Architecture

KVCOMM treats a multi-agent application as a directed graph G = (M, E). Nodes are agents, each holding an identical RoPE-based LLM checkpoint plus a role-specific system prompt template. Edges are one-way message passes. Every agent maintains a prompt dictionary with placeholder segments that get filled at runtime with user queries, tool results, or upstream outputs.

Step 0 вҖ” Initialization (one-time)

Each agent precomputes and stores KV-caches for every prefix template it owns. These serve as "anchors-of-anchors" throughout the session.

вҶ“

Step 1 вҖ” Placeholder Readiness

Agent receives a message. For each placeholder slot, it checks whether the base KV-cache (under its own system prompt but without surrounding prefix) is available. Missing bases are precomputed in parallel.

вҶ“

Step 2 вҖ” Reuse or Fallback Decision

Once all base caches are ready, the agent asks: is there a reusable deviation (i.e., a matching anchor set) for this placeholder? If yes вҶ’ go to Step 3. If no вҶ’ do standard dense prefill and store the resulting deviation as a new anchor for future messages.

вҶ“

Step 3 вҖ” Offset Approximation (parallel)

The agent fetches matched anchors, computes per-anchor softmax weights from embedding proximity, and updates the placeholder's and surrounding prefix's caches via Eq. 6 / Eq. 7. All of this happens in parallel across placeholders.

вҶ“

Step 4 вҖ” Decode

The concatenated (placeholder + prefix) cache is handed to the model runner, which begins decoding without a dense prefill pass.

вҶ“

Step 5 / 6 вҖ” Anchor Update or Fallback Storage

After decoding, the response KV-cache is passed to the anchor prediction module. If the prediction module finds a similar anchor, it stores the cache in shared memory for future retrieval. Otherwise (fallback path), the response is marked "unshareable" and stored per-agent so downstream agents that re-encode it can fill in deviations via their respective contexts.

All inter-agent interaction is mediated by the KV-cache Communication Module. When multiple agents share the same user text but use different agent-specific prefixes, we avoid re-running prefilling by treating the KV-cache of the shared text under a new prefix as a "context-dependent offset" from its base KV-cache. The offset itself is predicted by interpolating from a small anchor set, aligning Key positions via RoPE de-rotation + re-rotation, adding estimated Key/Value offsets, and concatenating the adjusted segments.

6. Anchor Pool & Matching

6.1 What an Anchor Stores

Each anchor is a dict keyed by (ph_name, agent_id_ph, agent_id_pf) carrying: (1) the base KV-cache computed under no external context, (2) Key/Value offsets from the base to the actual cache under each agent's context, and (3) offsets of overlapping neighboring prefix segments. Neighboring prefix offsets are crucial because the position-dependent KV-cache shifts introduced by the attention sink mechanism emphasize local contextual dependencies.

# KVCOMM/llm/kvcomm_engine.py:1115 вҖ” what a single anchor entry stores
entry = {
    "ph_key_embedding": ph_key_base,          # base Key cache for placeholder
    "ph_value_embedding": ph_val_base,        # base Value cache
    f"{self.llm.node_id}_ph_key_delta":     ph_key_real  - ph_key_base,
    f"{self.llm.node_id}_ph_value_delta":   ph_val_real  - ph_val_base,
    f"{self.llm.node_id}_pf_key_delta":     pf_key_real  - pf_key_base,
    f"{self.llm.node_id}_pf_value_delta":   pf_val_real  - pf_val_base,
}

6.2 Anchor Selection Eligibility

When looking up anchors for a placeholder of length k, KVCOMM only considers anchors whose stored placeholder span is at least k tokens long. Shorter anchors are skipped вҖ” they cannot cover the full placeholder. This check is implemented in _select_anchor_indices:

# KVCOMM/llm/kvcomm_engine.py:806
@staticmethod
def _select_anchor_indices(anchor_list: List[Dict[str, Any]],
                           placeholder_len: int) -> List[int]:
    """Pick anchors whose placeholder span covers the current placeholder length."""
    return [
        idx
        for idx, anchor in enumerate(anchor_list)
        if anchor["ph_key_embedding"].shape[-2] >= placeholder_len
    ]

6.3 Weight Computation вҖ” The Per-Anchor Softmax

The heart of the matching logic is _compute_anchor_weight_entry. For every selected anchor, it computes two softmax-weighted similarity scores вҖ” one for prefix-facing offsets and one for placeholder-facing offsets вҖ” with temperature scaling (default 1.0):

# KVCOMM/llm/kvcomm_engine.py:814
def _compute_anchor_weight_entry(
    self,
    anchor_list, anchor_indices,
    real_key_embedding, real_value_embedding,
    placeholder_len, temperature,
):
    used_anchors = [anchor_list[idx] for idx in anchor_indices]

    # Build stacks of (anchors, ..., placeholder_len, head_dim)
    anchor_key_ph = torch.stack([a["ph_key_embedding"][..., -placeholder_len:, :]
                                  for a in used_anchors])
    real_key_ph  = real_key_embedding[..., -placeholder_len:, :]

    # в„“вӮӮ distance along the sequence dim вҶ’ softmax(-dist / T)
    sims_key_prefix = (real_key_ph.unsqueeze(0) - anchor_key_ph).norm(2, dim=-2)
    weights_key_prefix = torch.softmax(-sims_key_prefix.float() / temperature, dim=0).unsqueeze(-2)

    # Same with Value tensors вҖ” then again for placeholder offsets
    ...

There are actually four sets of weights вҖ” {Key, Value} Г— {prefix-facing, placeholder-facing} вҖ” each computed with в„“вӮӮ distance and softmax. The weights are reused across all layers because the cache-space translation is layer-wise linear.

7. Offset Approximation вҖ” offset_kv_cache_pair

The function offset_kv_cache_pair is where the actual KV-cache gets blended. Given the base placeholder cache, the base prefix cache, and a list of matched anchors, it returns a new DynamicCache that looks as if the model had been prefilled with the current agent's context вҖ” but without running the model.

# KVCOMM/llm/kvcomm_engine.py:868 (condensed)
def offset_kv_cache_pair(self, ph_id, message, request_uid,
                         base_placeholder_cache, base_prefix_cache,
                         anchor_list, temperature=1):
    placeholder_len = int(base_placeholder_cache._seen_tokens)
    real_key, real_val = self._stack_cache_tensors(base_placeholder_cache)

    # 1. Cache-hit on the (request, placeholder) key
    signature = self.anchor_signature(anchor_list)
    cache_entry = self._get_cached_anchor_weights(request_uid, ph_id, message, signature)

    if cache_entry is None:
        anchor_index = self._select_anchor_indices(anchor_list, placeholder_len)
        if not anchor_index:
            return base_placeholder_cache.copy(), base_prefix_cache.copy()
        cache_entry = self._compute_anchor_weight_entry(
            anchor_list, anchor_index, real_key, real_val,
            placeholder_len, float(temperature)
        )
        self._set_cached_anchor_weights(request_uid, ph_id, message, cache_entry)
    else:
        anchor_index = cache_entry["anchor_index"]

    # 2. Stack the four delta tensors from selected anchors
    prefix_key_delta = torch.stack([anchor_list[i][f"{self.llm.node_id}_pf_key_delta"]
                                     for i in anchor_index])
    prefix_val_delta = torch.stack([anchor_list[i][f"{self.llm.node_id}_pf_value_delta"]
                                     for i in anchor_index])
    ph_key_delta = torch.stack([anchor_list[i][f"{self.llm.node_id}_ph_key_delta"][..., :placeholder_len, :]
                                 for i in anchor_index])
    ph_val_delta = torch.stack([anchor_list[i][f"{self.llm.node_id}_ph_value_delta"][..., :placeholder_len, :]
                                 for i in anchor_index])

    # 3. Softmax-weighted interpolation (Eq. 6/7)
    layer_total_key_prefix = (cache_entry["weights_key_for_prefix"] * prefix_key_delta).sum(0)
    layer_total_val_prefix = (cache_entry["weights_value_for_prefix"] * prefix_val_delta).sum(0)
    layer_total_key_ph     = (cache_entry["weights_key_for_placeholder"] * ph_key_delta).sum(0)
    layer_total_val_ph     = (cache_entry["weights_value_for_placeholder"] * ph_val_delta).sum(0)

    # 4. Apply offsets to real cache; preserve layer 0 unchanged
    updated_ph_key = real_key + layer_total_key_ph.to(real_key.dtype)
    updated_ph_key[0] = real_key[0]          # embedding layer untouched
    updated_ph_val = real_val + layer_total_val_ph.to(real_val.dtype)
    updated_ph_val[0] = real_val[0]

    # (Similarly for prefix cache ...)
    return new_placeholder_cache, new_prefix_cache
Layer-0 quirk: The embedding layer (layer 0) is explicitly left untouched. This is because layer 0 is the raw input embedding вҖ” applying an offset there would corrupt the base representation for subsequent layers. Offsets are only applied from layer 1 onwards.

8. Anchor Prediction & Adaptive Update

Not every response can or should become an anchor. If a new cache is too dissimilar from anything in the pool, forcing it to share would inject noise. If it is too similar to everything, it is redundant. KVCOMM decides via an entropy threshold on the softmax distribution over existing anchors.

# KVCOMM/llm/kvcomm_engine.py:991 (simplified)
def predict_as_anchor(self, candidate_kv_cache, anchor_kv_cache_list,
                       anchor_len_list, anchor_activated_list,
                       top_p=0.9, entropy_eps=1e-40, test_time=False):
    if len(anchor_kv_cache_list) in [0, 1]:
        return True, anchor_activated_list        # trivial вҖ” treat as anchor

    k = candidate_kv_cache.value_cache[0].shape[-2]
    anchor_available = [i for i, (j, _) in enumerate(anchor_len_list) if j >= k]

    if len(anchor_available) > 1:
        cand_val = torch.stack(candidate_kv_cache.value_cache)[..., :k, :]
        anc_val  = torch.stack([anchor_kv_cache_list[i]["ph_value_embedding"][..., :k, :]
                                 for i in anchor_available])

        diff = (cand_val.unsqueeze(0) - anc_val).norm(2, dim=(1, 2, 3, 4, 5))
        sim = torch.softmax(-diff.float(), dim=0)

        threshold = self.llm.config.threshold                          # Оі, default 0.3
        entropy = -(sim * (sim + entropy_eps).log2()).sum()

        # If entropy too high вҶ’ candidate is equally similar to many anchors
        # вҶ’ it is not a good new anchor. Return True (mark as new anchor).
        if entropy > threshold * torch.log2(torch.tensor(sim.shape[0])):
            return True, anchor_activated_list

        # Otherwise pick top-p anchors and activate them
        sorted_sim, sorted_idx = torch.sort(sim, descending=True)
        cumulative = torch.cumsum(sorted_sim, dim=0)
        cutoff = (cumulative < top_p).nonzero(as_tuple=True)[0]
        cutoff_idx = cutoff[-1] if len(cutoff) > 0 else len(sorted_sim) - 1
        for i in sorted_idx[:cutoff_idx + 1]:
            anchor_activated_list[anchor_available[i]] += 1
        return False, anchor_activated_list

    return True, anchor_activated_list

8.1 The Entropy Criterion Explained

Intuitively: after computing softmax similarities between the candidate cache and every existing anchor, KVCOMM asks "how concentrated is the distribution?" A concentrated distribution (low entropy) means the candidate looks like one or two specific anchors вҖ” perfect for reuse, no need to add a new anchor. A diffuse distribution (high entropy, above Оі В· logвӮӮ(N)) means the candidate is equally close to many anchors вҖ” the anchor pool does not yet cover this part of cache space, so store it.

8.2 Anchor Pool Window & Eviction

Due to GPU memory limits, each placeholder's anchor pool is capped at V = 20 entries. When a new anchor arrives and the pool is full, update_anchor walks the oldest window_length anchors (default 5) and evicts the least frequently used one. This keeps the relevant-and-efficient anchors while discarding stale or never-used entries.

9.3 Why DynamicCache and Not PagedAttention?

KVCOMM directly manipulates transformers.cache_utils.DynamicCache вҖ” the layer-indexed K/V tensor container shipped with HuggingFace вҖ” rather than touching vLLM's block-based PagedAttention cache. The reason is pragmatic: DynamicCache exposes raw tensors that can be sliced, stacked, offset-added, and concatenated without any block-table plumbing, which is exactly what Eq. 6/7 require. The authors explicitly list "integration with LMCache / modern LLM-based serving systems" as the Next Step, acknowledging that DynamicCache is not production-ready for high-throughput serving.

Strengths & Limitations

Strengths

  • Training-free, drop-in: only requires a pretrained RoPE-based LLM and a multi-agent graph
  • First method to handle prefix drift across agents with theoretical guarantees (Lipschitz bounds)
  • Clean ablation: three alignment steps each contribute measurable accuracy
  • Accuracy stays within 2.5% of baseline even at 5 agents
  • Code is small (~1300 lines), readable, and fully open-source
  • Robust to request order and anchor pool size within a wide range

Limitations

  • Requires all agents to share the same base model checkpoint (RoPE identity) вҖ” cannot mix Llama and Qwen in the same graph
  • DynamicCache only вҖ” no integration with vLLM PagedAttention yet
  • Anchor pool grows with placeholder count; real-world agents with many roles may hit memory limits
  • Speedup on Agent 1 is minimal вҖ” the very first agent always pays full prefill
  • Per-anchor softmax computes on the full sequence dim each call вҖ” O(LВ·V) per token, not batched across requests
  • Benchmark only tested on single-H100, batch-1; batching behavior is unknown

13. Related Work & Open Questions

KVCOMM sits at the intersection of two research lines: multi-agent LLM orchestration (AutoGPT, GPTSwarm, AgentPrune, Camel) and KV-cache acceleration (PromptCache, CacheBlend, DroidSpeak, CacheGen, LoongServe, LMCache). Open Questions

  1. How does KVCOMM compose with CacheGen compression? KVCOMM operates on full-precision tensors; storing offsets in a compressed form could dramatically reduce pool memory.
  2. Can the anchor prediction entropy threshold Оі be learned per-task rather than fixed at 0.3?
  3. The DynamicCache layout limits batching. Does a PagedAttention-native port (the Next Step listed in README) change the speedup story at batch > 1?
  4. What happens in heterogeneous agent graphs where some agents use the model and others are tool-callers? Can KVCOMM skip tool-call turns or does the offset pool stay valid across them?
  5. The paper reports Оі sensitivity on GSM8K only. Does the same 0.3 knee hold for MMLU, HumanEval, or long-context retrieval tasks?
