Ada-KV: Optimizing KV Cache Eviction
by Adaptive Budget Allocation

NeurIPS 2025 (39th Conference on Neural Information Processing Systems) Paper Code

Overview

Ada-KV is the first adaptive head-wise budget allocation strategy for KV cache eviction. While existing Top-k eviction methods (SnapKV, Pyramid) uniformly distribute cache budgets across all attention heads, Ada-KV recognizes that heads exhibit vastly different attention concentration patterns -- sparse heads need smaller budgets, while dispersed heads need larger ones.

Theoretical Foundation

L1 Eviction Loss

Ada-KV quantifies generation quality degradation after cache eviction as the L1 distance between pre- and post-eviction attention outputs:

L1 Eviction Loss = ||y - ŷ||1

Theorem 3.1: Upper Bound

ε = 2hC - 2C ∑i∈[1,h] ∑j∈[1,n] Iij Aij

Where C = Max{||ViWiO||∞} is a constant, Iij is the eviction indicator, and Aij is the attention weight.

Theorem 3.2: Top-k Optimality

The Top-k eviction decision minimizes ε under any given budget allocation {Bi}:

{Ii*} = argmin ε ⇒ ε* = 2hC - 2C ∑i ∑Aij∈Top-k(Ai,Bi) Aij

Theorem 3.3: Adaptive Allocation Minimizes the Bound

The adaptive budget allocation {Bi*} from Algorithm 1 achieves the minimum upper bound ε** compared to any allocation strategy, including uniform allocation:

ε** = min{Bi} ε*

Algorithm

Figure 2 (from paper): Overview of Ada-KV. A global Top-k selection over concatenated head-wise attention weights yields per-head frequencies, which are then used as each head's adaptive cache budget.

Algorithm 1: Ada-KV Budget Allocation

Pseudocode
Input:  Total budget B; Attention weights per head i: {A_i}
Output: Allocated budgets {B_i*}

1: A = Cat({A_i})                # Concatenate across heads
2: top_weights = Top-k(A, k=B)   # Select B largest weights globally
3: f_i = count(top per head i)    # Count selections per head
4: B_i* = f_i                     # Assign budgets proportionally
Return {B_i*}

Algorithm 2: Ada-SnapKV / Ada-Pyramid Integration

Pseudocode
Input:  Budget B, observation window tokens X^win, KV caches
Output: Retained KV cache

1: for i = 1 to h do
2:     Q_i^win = X^win * W_i^Q
3:     A_bar_i = softmax(Q_i^win * K_i^T)
4:     A_bar_i = maxpooling(A_bar_i)      # Importance scoring
5: end for
6: B' = B - winsize * h                  # Non-window budget
7: {B_i*} = AdaKV(B', {A_bar_i})         # <-- ADAPTIVE ALLOCATION
8: {B_i*} = alpha * {B_i*} + (1-alpha) * (B'/h)  # Safeguard (alpha=0.2)
9: Top-k select per head using {B_i*}
10: Concatenate with observation window KV
Return retained cache
Safeguard Parameter α = 0.2: A small floor prevents excessively small budgets for highly sparse heads. The final budget blends the adaptive allocation with a uniform baseline: Bi* = α × Bi* + (1-α) × (B'/h). This prevents degenerate cases where a head receives zero budget.

Variable-Length FlashAttention

Adaptive allocation creates variable-sized caches per head. Ada-KV implements a flattened cache storage layout that concatenates all heads within a layer into a single tensor, enabling efficient computation via variable-length FlashAttention with custom CUDA kernels.

Python -- DynamicCacheSplitHeadFlatten
class DynamicCacheSplitHeadFlatten(Cache):
    """
    Flattened cache layout for variable-length head caches.
    Concatenates all heads into a single tensor per layer.
    """
    def __init__(self):
        super().__init__()
        self.key_cache: List[List[torch.Tensor]] = []
        self.value_cache: List[List[torch.Tensor]] = []

    def update(self, key_states, value_states, layer_idx, cache_kwargs):
        if len(self.key_cache) <= layer_idx:
            # Phase 1 (prefill): store initial cache
            self.key_cache.append(key_states)
            self.value_cache.append(value_states)
        else:
            # Phase 2 (decode): update via CUDA kernel
            head_lens = cache_kwargs["head_lens"]
            cu_klen = cache_kwargs["cu_klen"]
            from tiny_api_cuda import update_flatten_view
            self.key_cache[layer_idx] = update_flatten_view(
                self.key_cache[layer_idx], key_states, head_lens, cu_klen
            )
        return self.key_cache[layer_idx], self.value_cache[layer_idx]

GQA Compatibility

Existing methods like SnapKV and Pyramid redundantly replicate grouped KV caches. Ada-KV implements GQA-compatible selection using mean attention weight within each group as the selection criterion, eliminating redundancy and achieving up to 4x cache reduction in models like Llama-3.1-8B.

Detailed Implementation Flow

The monkey-patching approach allows Ada-KV to be integrated without modifying model source code. Here is how the attention forward pass is modified:

Python -- Adaptive Attention Forward
def adaptive_llama_attention_forward(self, hidden_states, attention_mask,
                                      position_ids, past_key_value, ...):
    # Standard Q, K, V projections
    query_states = self.q_proj(hidden_states)
    key_states = self.k_proj(hidden_states)
    value_states = self.v_proj(hidden_states)

    # Apply rotary position embeddings
    query_states, key_states = apply_rotary_pos_emb(
        query_states, key_states, cos, sin
    )

    # Phase 1: During prefill with eviction
    if past_key_value is None and should_compress:
        # Compute full attention for observation window
        win_q = query_states[:, :, -window_size:, :]
        attn_weights = torch.matmul(win_q, key_states.transpose(2, 3))
        attn_weights = F.softmax(attn_weights, dim=-1)

        # Max-pool importance scores
        importance = F.max_pool1d(
            attn_weights.mean(dim=2), kernel_size=kernel_size, padding=padding
        )

        # === ADA-KV: Adaptive budget allocation ===
        all_importance = torch.cat([importance[h] for h in range(num_heads)])
        _, global_top_idx = all_importance.topk(total_budget)
        head_budgets = count_per_head(global_top_idx, num_heads)
        head_budgets = alpha * head_budgets + (1 - alpha) * uniform_budget

        # Per-head Top-k selection with variable budgets
        for h in range(num_heads):
            top_k_idx = importance[h].topk(head_budgets[h]).indices
            selected_keys[h] = key_states[:, h, top_k_idx]
            selected_values[h] = value_states[:, h, top_k_idx]

        # Store in flattened cache layout
        past_key_value.update(flatten(selected_keys), flatten(selected_values))

    # Phase 2: Standard decoding with variable-length FlashAttention
    else:
        # Use flash_attn_varlen_func for efficient variable-length attention
        attn_output = flash_attn_varlen_func(
            query_states, key_cache, value_cache,
            cu_seqlens_q, cu_seqlens_k, max_seqlen_q, max_seqlen_k
        )
    return attn_output

Deeper Analysis

Why Uniform Allocation Fails

The core insight of Ada-KV is empirically grounded. An analysis of Llama-3.1-8B-Instruct using aggregated attention weights reveals that attention concentration varies dramatically across heads:

Sparse Heads (~30% of all heads)

Nearly all attention weight concentrated on a small number of tokens (top 5% captures >90% of attention). These heads need very small budgets -- retaining just a few critical KV pairs captures almost all the information.

Dispersed Heads (~20% of all heads)

Attention spread broadly across many tokens. The top 5% captures only ~30-40% of attention. These heads need much larger budgets to avoid significant information loss from eviction.

Mixed Heads (~50% of all heads)

Moderate concentration. These benefit most from fine-grained adaptive allocation rather than a coarse uniform assignment.

Question-Aware vs Question-Agnostic Compression

Ada-KV demonstrates an important distinction often overlooked in prior work:

ScenarioDescriptionBaseline GapAda-KV Improvement
Question-AwareQuestion is known before compression; attention weights computed with question queriesSmall gap from full KVConsistent but modest
Question-AgnosticCompression occurs before questions are known (e.g., prefix caching, multi-turn dialogue)Large gap from full KVSubstantially larger gains
Practical Importance: Question-agnostic compression is far more realistic in production deployments. When a shared prefix is cached for multiple future queries, the system cannot know which queries will arrive. Ada-KV's adaptive allocation provides particularly large benefits in this harder scenario, improving SnapKV scores from 44.02 to 53.29 at 80% budget on Ruler.

Computational Efficiency

A common concern with adaptive allocation is computational overhead from variable-sized caches. Ada-KV addresses this with its flattened cache layout and custom CUDA kernels:

OperationStandard (uniform)Ada-KV (adaptive)Overhead
Peak Memory (1024 budget)BaselineComparable~0%
Decoding LatencyBaselineComparable<2%
Prefill (with allocation)Baseline+ Cat + Top-k + Count~5%

The allocation step (Algorithm 1) involves a single concatenation, one top-k selection, and counting -- all highly parallelizable operations. The variable-length FlashAttention during decoding matches the efficiency of standard FlashAttention.

Safeguard Parameter Sensitivity

The safeguard parameter α (default 0.2) blends adaptive with uniform allocation to prevent degenerate budgets:

Bi* = α × Bi*(adaptive) + (1 - α) × (B / h)

At α = 0 (fully adaptive), a few very sparse heads may receive near-zero budgets, causing instability. At α = 1 (fully uniform), no benefit from adaptation. The default α = 0.2 provides a strong floor while preserving most of the adaptive benefit.

Theoretical Details

Proof Sketch: Why Adaptive Allocation Minimizes ε**

The proof of Theorem 3.3 proceeds by contradiction. Suppose there exists an allocation {B'i} with ε' < ε**. The adaptive allocation selects the globally top-B attention weights; any alternative allocation must include at least one weight not in this global top-B set, replacing a weight that is in it. Since the replaced weight is larger (it was in the global top-B), the sum decreases, contradicting the optimality assumption.

ε** = 2hC - 2C ∑i∈[1,h] ∑Aij∈Top-k(Ai,Bi*) Aij ≤ ε'   for all {B'i}

Relationship to Information Theory

The eviction loss bound can be interpreted through an information-theoretic lens. The attention weights Aij represent how much information from token j contributes to the output through head i. Maximizing the sum of retained attention weights is equivalent to maximizing the retained information flow through the multi-head attention mechanism.

Connection to Optimal Transport

Ada-KV's budget reallocation can be viewed as an optimal transport problem: given a fixed total budget B, find the assignment of budgets to heads that minimizes the total "transport cost" (eviction loss). The adaptive allocation from Algorithm 1 provides the closed-form solution to this transport problem, avoiding the need for iterative optimization.

Formal Algorithm Walkthrough

To make the adaptive allocation concrete, consider a layer with h=4 heads and total budget B=8. Suppose the concatenated attention weights (sorted) and their head assignments are:

RankWeight ValueHeadAction
10.35Head 2Retain (budget B2 += 1)
20.30Head 2Retain (budget B2 += 1)
30.28Head 1Retain (budget B1 += 1)
40.25Head 2Retain (budget B2 += 1)
50.22Head 4Retain (budget B4 += 1)
60.20Head 2Retain (budget B2 += 1)
70.18Head 3Retain (budget B3 += 1)
80.15Head 1Retain (budget B1 += 1)

Result: B1=2, B2=4, B3=1, B4=1. Head 2 (dispersed) receives 4x the budget of heads 3 and 4 (sparse). Under uniform allocation, each head would receive B/h = 2, wasting budget on sparse heads while starving the dispersed head.

After safeguard blending (α=0.2): B2* = 0.2 × 4 + 0.8 × 2 = 2.4. This ensures even sparse heads retain a minimum viable budget.

Scalability Considerations

The adaptive allocation step has O(B log B) complexity for the Top-k selection, where B is the total budget. Since B is typically small relative to the sequence length (e.g., B=128-2048 vs sequences of 16K-128K tokens), this overhead is negligible. The dominant cost remains the attention computation itself.

Memory Layout Innovation: Ada-KV's flattened cache layout is a significant engineering contribution. By concatenating all per-head caches into a single contiguous tensor and maintaining a cumulative length array (cu_klen), the implementation avoids Python-level iteration over heads during decoding. The custom CUDA kernel update_flatten_view performs O(1) per-head cache updates in parallel.

Position in the KV Eviction Landscape

Ada-KV is orthogonal to most other innovations in KV cache eviction. While methods like NACL focus on which tokens to evict and Attention-Gate focuses on learning when to evict, Ada-KV focuses on how to distribute the eviction budget. Limitations and Future Directions

Budget Floor Sensitivity

The safeguard parameter α = 0.2 works well across tested scenarios, but extremely sparse or extremely dispersed head distributions may require tuning. An adaptive α that adjusts based on the variance of attention concentration could further improve robustness.

Cross-Layer Allocation

Ada-KV currently allocates budgets independently within each layer. Future work could explore cross-layer budget allocation, where some layers receive more total budget than others (similar to PyramidKV but within the Ada-KV framework).

Dynamic Reallocation

During generation, the importance of KV pairs may shift as context grows. Dynamic budget reallocation at decode time, combined with efficient cache reshaping, could further improve long-generation quality.

Theoretical Extensions

The current theoretical framework uses L1 distance. Extensions to Lp norms or divergence measures could provide tighter bounds and potentially different allocation strategies.

Key Takeaways

Summary of Contributions:
1. Theoretical: Establishes L1 eviction loss upper bound and proves Top-k eviction minimizes it under any budget allocation.
2. Algorithmic: Proposes the first adaptive head-wise budget allocation (Algorithm 1) and proves it minimizes the upper bound across all possible allocations.
3. Practical: Demonstrates plug-and-play integration with two SOTA methods (Ada-SnapKV, Ada-Pyramid) via efficient CUDA kernels and flattened cache layout.
4. Empirical: Evaluates on 29 datasets across both question-aware and question-agnostic scenarios, showing consistent improvements with minimal overhead.

CUDA Kernel & Storage Details

Why a Custom CUDA Kernel Is Needed

Standard PyTorch operations assume a uniform number of KV pairs per head, making them incompatible with Ada-KV's variable-length caches. The custom kernel update_flatten_view operates directly on the flattened contiguous buffer, updating each head's slice in a single GPU dispatch rather than H sequential Python calls.

Flattened Storage Layout

Ada-KV stores all per-head KV slices contiguously in a single tensor per layer. Two auxiliary arrays track the variable boundaries:

ArrayShapeDescription
key_cache[l][sum(Bi), d_head]Flattened key vectors for all heads in layer l
value_cache[l][sum(Bi), d_head]Flattened value vectors for all heads in layer l
head_lens[h]Per-head cache sizes: head_lens[i] = Bi*
cu_klen[h+1]Cumulative sum of head_lens (0, B1*, B1*+B2*, ...)

Decode Kernel Dispatch

During auto-regressive decoding each new token generates one new KV pair per head. The update kernel appends the new KV pair to the correct slice in the flattened buffer using cu_klen as an offset table:

CUDA (conceptual) -- update_flatten_view
// Each thread block handles one head
__global__ void update_flatten_view(
    float* flat_cache, // [sum(Bi), d_head]
    float* new_kv,     // [h, d_head]
    int*   cu_klen,    // [h+1]
    int    d_head
) {
    int head_id = blockIdx.x;
    int slot    = cu_klen[head_id + 1] - 1; // last position of this head
    for (int d = threadIdx.x; d < d_head; d += blockDim.x)
        flat_cache[slot * d_head + d] = new_kv[head_id * d_head + d];
}

Because the write offset for each head is independent, all heads execute in parallel. This single kernel replaces H Python-level index operations and avoids CUDA stream synchronization overhead between heads.

Variable-Length FlashAttention During Decode

Ada-KV leverages flash_attn_varlen_func from the FlashAttention-2 library. This function accepts cumulative sequence-length arrays (cu_seqlens_k) that describe the variable-length batches, avoiding padding and allowing the attention kernel to skip unnecessary computations:

Python -- Variable-Length Decode Attention
from flash_attn import flash_attn_varlen_func

attn_output = flash_attn_varlen_func(
    q=query_states.view(-1, num_heads, head_dim),
    k=key_cache[layer_idx],    # shape [sum(Bi), head_dim]
    v=value_cache[layer_idx],  # shape [sum(Bi), head_dim]
    cu_seqlens_q=cu_seqlens_q, # [batch*h+1] -- one query token per head
    cu_seqlens_k=cu_klen,      # [h+1] -- variable KV lengths
    max_seqlen_q=1,            # decoding: always 1 query token
    max_seqlen_k=max_budget,
    causal=False,              # KV already evicted; no mask needed
)

Recommended Configurations by Use Case

High-Throughput Serving

base_capacity=256, window_size=16, floor_alpha=0.2, pyram_mode=False. Prioritises memory savings and decode speed. Suitable for question-agnostic prefix caching where many concurrent requests share the same context.

Quality-First RAG

base_capacity=1024, window_size=64, floor_alpha=0.1, pyram_mode=False. Maximises retrieval accuracy for document QA. The larger window and lower floor let the adaptive allocation work more freely on diverse document heads.
