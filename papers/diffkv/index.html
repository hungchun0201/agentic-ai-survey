01 Overview: Three Levels of Differentiation

DiffKV advances KV cache compression by recognizing that keys, values, and individual tokens should be treated differently rather than with a one-size-fits-all strategy. Prior methods like H2O, SnapKV, and uniform quantization (KIVI, Atom) either prune uniformly or quantize all KV entries at the same precision. DiffKV introduces three levels of differentiation within the KV cache:

1. Keys vs. Values

Keys have a broader impact on attention via the shared softmax denominator. Values only affect their own token's contribution. DiffKV assigns higher precision to keys (e.g., K8V4 instead of uniform K4V4).

Keys: INT8 Values: INT4

2. Token Importance

Tokens contribute differently based on their attention scores. DiffKV uses a hierarchical strategy: critical tokens get high precision (K8V4), moderate tokens get lower (K4V2), and least important tokens are pruned entirely.

High-Prec Low-Prec Pruned

3. Per-Head Dynamic Sparsity

The number of critical tokens varies dynamically across attention heads and requests. DiffKV manages memory per-head and per-request rather than allocating uniformly, adapting to each head's sparsity pattern.

Key Insight: Attention scores span seven orders of magnitude, while value vector norms cover only two. This stark disparity proves that keys play a far more critical role in determining attention output, justifying asymmetric precision assignment.

Key Implementation Details

BlockAllocator (Circular Free List)

GPU-resident circular buffer managing page IDs. Uses start_block_pos and end_block_pos as pointers. Allocation advances the start pointer; freeing advances the end pointer. Both wrap around modulo num_blocks.

class BlockAllocator:
  def __init__(self, device, num_blocks, ...):
    self.free_blocks = torch.arange(
      num_blocks, dtype=torch.int32, device=device)
    self.start_block_pos = 0
    self.end_block_pos = num_blocks - 1
    self.block_refs = torch.zeros(
      (num_blocks,), dtype=torch.int32, device=device)

Bidirectional Page Table

4D tensor [max_seqs, layers, kv_heads, max_blocks] where high-precision pages fill left-to-right and low-precision pages fill right-to-left within each head's entry. The last dimension size of kv_len_tables is 2 to track both precision levels.

self.block_tables = torch.full(
  (max_num_seqs, num_layers, num_kv_heads,
   max_block_table_len),
  -1, dtype=torch.int32, device='cuda')
self.kv_len_tables = torch.full(
  (max_num_seqs, num_layers, num_kv_heads, 2),
  0, dtype=torch.int32, device='cuda')

CacheEngine: Unified Pages

GPU cache is a flat 2D tensor of [num_blocks, block_size] in INT16. Each block is a unified page whose block_size is computed by CacheConfig.compute_cache_block() to fit keys, values, quantization metadata, scores, and indices at the target precision.

class CacheEngine:
  def allocate_gpu_cache(self) -> KVCache:
    return torch.empty(
      size=(self.num_gpu_blocks, self.block_size),
      dtype=torch.int16, device='cuda')

Quantization Configurations

Supported configs as (kbits, vbits, n_kgroups, n_vgroups) tuples. The heterogeneous pair uses different bit-widths for keys and values, the core innovation of DiffKV.

04 Parallel KV Compaction Mechanism

The core scalability challenge is that DiffKV's compression creates O(#requests × #heads) heterogeneous memory regions per inference step. Parallel KV compaction handles this entirely on-GPU using two phases: Planning and Coordination.

KV Compaction Workflow (Prompt Phase)
Step 1: Allocate pages conservatively (all high-precision)
Step 2: Planning — each head determines memory needs
Head A: 1 high, 1 low
Head B: 2 high, 1 low
Step 3: Coordination — parallel prefix sum
Step 4: Recycle unused pages to free list

CUDA Kernel Implementation

The memory management kernels in mem_mgt_kernels.cu launch with a 3D grid of (num_seqs, num_layers, num_kv_heads), enabling full parallelism across all sequences, layers, and attention heads simultaneously.

Prompt Phase: allocate_seqs_kernel

Conservatively allocates pages for all tokens at high precision. Each CUDA block handles one (seq, layer, head) triple. Uses block-level reduction to compute per-sequence offsets into the free list.

// Grid: (num_seqs, num_layers, num_kv_heads)
// Each block allocates pages for one head of one sequence
__global__ void allocate_seqs_kernel(...) {
  const int seq_idx = blockIdx.x;
  const int layer_idx = blockIdx.y;
  const int head_idx = blockIdx.z;

  // Parallel reduction: compute offset for this seq
  int used_blocks = 0;
  for(int b = threadIdx.x; b < seq_idx; b += NUM_THREADS)
    used_blocks += num_blocks_ptr[b];
  used_blocks = blockReduceSum<int>(used_blocks);

  // Fill block_tables from circular free list
  for(int i = threadIdx.x; i < num_blocks; i += NUM_THREADS) {
    int pos = (start_pos + i) % num_gpu_blocks;
    block_tables[offset + i] = free_blocks[pos];
    block_refs[free_blocks[pos]] += 1;
  }
}

Bidirectional Swap in free_prompt_kernel

After compression, low-precision pages must be moved to the right side of the bidirectional page table. A symmetric swap avoids data corruption when source and destination ranges overlap.

// Symmetric swap to avoid overlap corruption
const int num_swaps = MIN(num_blocks_low, num_slots_to_mid);
for(int off = threadIdx.x; off < num_swaps; off += NUM_THREADS) {
  int src = base + num_blocks_high + off;   // left side
  int dst = base + max_blocks - 1 - off;     // right side
  // Swap, not copy, to handle overlapping regions
  int tmp = block_tables[src];
  block_tables[src] = block_tables[dst];
  block_tables[dst] = tmp;
}

Decode Phase Memory Ops

  • prepare_append_seqs — check if current page is full; compute new page ID via prefix sum
  • append_seqs — allocate at most 1 new page per head, update block tables and kv_len
  • free_seqs — return all pages when request finishes

Memory Management Overhead

On-GPU parallel KV compaction contributes <0.2% of total prompt latency and <0.9% of generation latency. This is up to 1000× faster than the CPU-based multi-threaded alternative.

05 Custom Sparse Attention Kernel

DiffKV implements a custom CUDA attention kernel (72K lines in sparse_attention_kernels.cu) that processes mixed-precision KV caches with fused on-the-fly dequantization.

Design Decisions

  • High-precision pages processed first, then low-precision — mitigates load imbalance from mixed quantization levels
  • Different vectorization for keys vs. values: Keys use feature-dimension vectorization (dot product pattern); values use token-dimension vectorization (reduction pattern)
  • Ultra-long sequence support: Sequences are split along the token dimension into partitions (PARTITION_SIZE = 4800), processed in parallel blocks, then merged
  • Triton kernel for prompt phase: triton_flash_attention.py computes attention scores for the last 64 tokens to drive the compression policy
Bandwidth Optimization: DiffKV's data layout ensures coalesced memory access for both key and value reads. With K8V8 (halving vs FP16), the theoretical speedup is 2×; DiffKV achieves 1.7× in practice. The gap comes from quantization metadata access and dequantization overhead. Speedup grows with sequence length as bandwidth optimization becomes more dominant.

08 KV Compression Policy (Algorithm 1)

DiffKV's compression policy operates identically in both prompt and generation phases, applying hierarchical classification per token based on attention score significance.

(K8V4)

Threshold Calibration

Parameters alpha_h and alpha_l are calibrated on the MATH training split. The thresholds are model-specific:

Llama3-8Balpha_h=1, alpha_l=0.02
Llama3-70Balpha_h=1, alpha_l=0.0
Qwen2.5-7Balpha_h=3, alpha_l=0.04
Qwen2.5-32Balpha_h=3, alpha_l=0.0
QwQ-32Balpha_h=3, alpha_l=0.0

Buffer Window

The most recent W tokens (default W=64, called kv_buffer_size in the code) are always kept at high precision to prevent premature compression. This buffer is defined by NUM_TOKENS_SCORE = 64 in both Python and CUDA.

09 How to Reproduce

Prerequisites

  • Python >= 3.10
  • NVIDIA GPUs with Ada, Hopper, or newer architectures (L40, A100, H100, etc.)
  • CUDA toolkit compatible with PyTorch 2.x

Step 1: Install from Source

# Clone the repository
git clone https://github.com/zyqCSL/DiffKV.git
cd DiffKV

# Install (compiles CUDA extensions)
pip install -e .

# Set PYTHONPATH
export PYTHONPATH=$PWD:$PYTHONPATH

Step 2: Quick Functional Verification

# Create logs directory
mkdir -p logs

# Run small model verification
cd param_tuning
./try_small_models.sh

# Parse and summarize results
python read_small_models_results.py

# Example: Llama3-8B on single GPU
python benchmarks/benchmark_throughput.py \

# QwQ-32B thinking model (2 GPUs, up to 5.4x throughput)

Add tuple to quantized_kv_bits list and update compute_cache_block()

2. cache_kernels.cu

3. long_prompt_cache_kernels.cu

Same pattern as cache_kernels.cu for long-prompt variant

5. param_tuning/util/util.py

Update BITS_TO_QUANT_GROUPS mapping
