Overview

RLCache decomposes cache management into three separate RL-driven decisions -- caching (what to store), TTL estimation (how long), and eviction (what to remove) -- each handled by a dedicated agent. This is the first work to model cache management as a multi-task RL control problem, compared to static heuristics (LRU, LFU, FIFO) that cannot adapt to changing workload patterns.

System Architecture

The Three RL Agents

1. Caching Strategy Agent

Decision: Given a cache miss on key k with values v, should we store this in the cache?

State: Encoded key, cache utility (hit-rate), operation type (new/update/miss).

Action: Binary -- {cache, do not cache}.

Reward: +1 for hit on a cached item, -1 for miss on an item we chose not to cache. Penalty for caching items that get evicted without being hit.

Architecture: DQN with experience replay.

2. TTL Estimation Agent

Decision: How long (in seconds) should a cached entry remain valid?

State: Encoded key, hit count, step code, cache utility, operation type.

Action: Continuous float value in [0, max_ttl].

Reward: Minimizes difference between estimated TTL and "optimal" TTL (time until next access or invalidation). Rewards accurate predictions; penalizes both over-caching (stale data) and under-caching (premature expiry).

Architecture: PPO with continuous action space (FloatBox).

3. Eviction Strategy Agent

Decision: When cache is full, which entry to evict?

State: Per-entry features -- access frequency, recency, size, TTL remaining.

Action: Binary per entry -- {evict, keep}. Scans all entries in cache.

Reward: +1 if evicted entry is never accessed again before it would have expired, -1 if evicted entry is requested soon after eviction (causing a miss).

Architecture: DQN with IntBox(0, 2) action space.

RLObserver Framework

A key challenge in applying RL to cache management is delayed rewards. When the caching agent decides to cache an item, the reward (hit or miss) arrives much later. RLObserver solves this by tracking incomplete experiences and associating future events with past decisions.

Python -- Observer Event Types
class ObservationType(Enum):
    Hit            = "hit"              # Cache hit (positive signal)
    Miss           = "miss"             # Cache miss (negative signal)
    Write          = "write"            # Entry written to cache
    Expiration     = "expiration"       # TTL expired (potential waste)
    EvictionPolicy = "eviction_policy"  # Evicted by agent
    Invalidate     = "invalidate"       # Data changed, cache invalid
    SetNotInCache  = "set_not_in_cache" # SET for uncached item

Multi-Agent and Multi-Task Configurations

Multi-Agent Configuration

Three separate RL agents, each with their own neural network, replay buffer, and training loop. The RLObserver routes events to the appropriate agent based on which decision triggered the event.

Advantage: Specialization -- each agent learns a focused policy.

Challenge: Coordination -- agents may conflict (e.g., caching agent stores, eviction agent immediately removes).

Multi-Task Configuration

A single PPO agent handles all three decisions with a shared representation. The action space is a tuple: (should_cache, ttl_value, evict_decision).

Advantage: Implicit coordination through shared parameters.

Challenge: Harder to train; may not specialize as effectively.

Key Finding: Both configurations outperform static heuristics (LRU, LFU, FIFO). The multi-agent configuration achieves slightly higher hit-rates, while the multi-task agent adapts faster to workload changes due to shared representation learning.

Multi-Task Agent Architecture

The multi-task agent uses a single neural network with shared feature extraction layers and three task-specific output heads:

Python -- Multi-Task Agent (Conceptual)
class MultiTaskCacheAgent(nn.Module):
    def __init__(self, state_dim):
        super().__init__()
        # Shared feature extractor
        self.shared = nn.Sequential(
            nn.Linear(state_dim, 128),
            nn.ReLU(),
            nn.Linear(128, 64),
            nn.ReLU(),
        )
        # Task-specific heads
        self.cache_head = nn.Linear(64, 2)    # {cache, skip}
        self.ttl_head = nn.Linear(64, 1)      # continuous TTL
        self.evict_head = nn.Linear(64, 2)    # {evict, keep}

    def forward(self, state, task):
        features = self.shared(state)
        if task == "cache":
            return self.cache_head(features)
        elif task == "ttl":
            return torch.sigmoid(self.ttl_head(features)) * max_ttl
        elif task == "evict":
            return self.evict_head(features)

The shared layers learn a common representation of cache state that benefits all three decisions. Training alternates between tasks based on which events occur in the cache simulation.

TTL Estimation Results

The TTL agent learns to dynamically adjust entry lifetimes. Entries that are accessed frequently receive longer TTLs; rarely accessed entries receive short TTLs, saving space for hotter keys.

Dynamic TTL: The TTL agent reduces the gap between assigned TTL and optimal TTL from ~45% (fixed TTL) to ~15% after training. This directly translates to better cache utilization -- hot keys stay cached longer, cold keys expire faster.

Workload Adaptation

When the YCSB workload changes mid-experiment (e.g., from read-heavy to write-heavy), RL agents adapt within ~2000 steps while heuristic methods remain sub-optimal. The multi-task agent adapts fastest due to its unified state representation.

TTL Learning Curve

The TTL agent shows a distinctive learning curve. Initially, it assigns conservative (long) TTLs, then progressively learns to shorten TTLs for infrequently accessed items while maintaining long TTLs for hot keys. After approximately 5000 training steps, the TTL difference between the agent's estimate and the optimal value stabilizes at ~15%, compared to ~45% for fixed TTL strategies.

Eviction Strategy Comparison

The RL eviction agent learns a policy that implicitly combines aspects of multiple heuristics. Analysis of the learned policy reveals it considers:

Recency Signal

Like LRU, the agent preferentially evicts items that have not been accessed recently. However, it does not strictly follow LRU ordering.

Frequency Signal

Like LFU, frequently accessed items are protected. The agent learns frequency thresholds that adapt to the workload.

TTL Awareness

Unlike any heuristic, the agent considers remaining TTL. Items about to expire naturally are deprioritized for eviction since they will leave soon anyway.

Relevance to Agentic Semantic Caching

TTL for LLM Response Caching

In agentic AI systems, LLM responses can be cached to avoid redundant API calls. But responses have time-varying validity -- a weather query cached for 1 hour vs. a math fact cached indefinitely. RLCache's TTL agent provides a template for learning semantic TTL policies.

Selective Caching Agent

Not all LLM responses should be cached. The caching decision agent learns when caching is beneficial -- analogous to deciding which agent tool-call results are worth storing vs. recomputing.

Multi-Agent Coordination

RLCache's framework for coordinating multiple agents with delayed rewards directly maps to multi-agent LLM systems where different agents manage different aspects of a shared knowledge cache.

class CacheManager:
    def __init__(self, config, cache, backend, result_dir):
        self.cache = cache                 # TTLCache instance
        self.backend = backend             # Persistent storage

        # Three RL-driven strategies
        self.caching_strategy, \
        self.eviction_strategy, \
        self.ttl_strategy = strategies_from_config(config, ...)

        # Observer orchestrates delayed rewards
        self.observer = ObserversOrchestrator(
            [self.caching_strategy, self.eviction_strategy,
             self.ttl_strategy], result_dir, self.cache_stats
        )

    def _set(self, key, values, operation_type):
        # 1. TTL agent estimates how long to cache
        ttl = self.ttl_strategy.estimate_ttl(key, values, operation_type)

        # 2. Caching agent decides whether to cache at all
        should_cache = self.caching_strategy.should_cache(
            key, values, ttl, operation_type
        )

        if should_cache:
            # 3. If cache full, eviction agent picks what to remove
            while self.cache.is_full():
                evicted = self.eviction_strategy.trim_cache(self.cache)
                for k in evicted:
                    self.observer.observe(k, ObservationType.EvictionPolicy, {})

            self.cache.set(key, values, ttl)
            self.observer.observe(key, ObservationType.Write, {'ttl': ttl})

RL Agent Design Deep-Dive

Caching Agent: DQN Architecture

The caching agent uses a Deep Q-Network (DQN) with experience replay to learn the should_cache policy. The key challenge is the delayed nature of rewards -- when the agent decides to cache an item, the reward (a cache hit) may not arrive for minutes or hours.

Python -- Caching Strategy
class RLCachingStrategy(CachingStrategy):
    def should_cache(self, key, values, ttl, operation_type):
        # Build state representation
        encoded_key = self.key_vocab.add_or_get_id(key)
        state = CachingAgentSystemState(
            encoded_key=encoded_key,
            cache_utility=self.cache_stats.cache_utility,
            operation_type=operation_type.value,
        )

        # Query DQN agent for action
        action = self.agent.get_action(state.as_array())
        # action = 0: do not cache, action = 1: cache

        # Store incomplete experience for later reward
        self._incomplete_experiences.set(
            key, {'state': state, 'action': action},
            ttl=self.experience_ttl
        )
        return action == 1

TTL Agent: Continuous Action Space

The TTL agent is the most innovative component. It uses PPO (Proximal Policy Optimization) with a continuous action space (FloatBox) to output a TTL value in [0, max_ttl]. The reward function incentivizes accurate TTL predictions:

Reward Function

Positive reward: When an item is accessed (hit) before its TTL expires -- the TTL was long enough.
Negative reward: Two cases: (a) item expires before being accessed (TTL too short, caused a miss) or (b) item sits in cache for much longer than needed (TTL too long, wasting space).
Optimal: TTL matches the inter-access time distribution of the key.

Experimental Reward Variant

An experimental reward function weighs the difference between estimated TTL and optimal TTL (time until next access or invalidation). This directly measures TTL accuracy rather than using proxy signals like hit/miss.

Eviction Agent: Cache-Wide Scanning

When the cache is full, the eviction agent scans all entries and decides for each whether to evict or keep. This is expensive (proportional to cache size) but occurs infrequently. The agent learns a policy that combines aspects of LRU (recency), LFU (frequency), and domain-specific patterns.

Hyperparameter Initialization Impact

The paper investigates the sensitivity of RL agents to hyperparameter initialization and finds that while initial performance varies, all agents converge to similar final policies after sufficient training. This suggests the RL approach is robust to initialization, a critical property for practical deployment.

Limitations & Future Directions

Simulation Gap

Experiments use a simulated cache environment (YCSB traces). Deployment in production systems would require handling additional complexities: network latency, concurrent access, distributed caching.

Training Overhead

RL agents require a warm-up period before outperforming heuristics. During this period, the cache operates sub-optimally. Future work could explore pre-training agents on historical traces.

Distributed Cache Manager

The current implementation is single-node. Extending to distributed caches (e.g., Redis cluster) would require coordinating agents across nodes, an open research problem.

Model-Based RL

Current agents are model-free (DQN, PPO). Model-based RL could learn a transition model of cache dynamics, potentially improving sample efficiency and convergence speed.

Framework Design Details

Observer Event Flow

The RLObserver orchestrates event routing to all three agents. When an event occurs, the observer identifies which agents have pending (incomplete) experiences related to the affected key and delivers the reward signal:

Python -- Observer Orchestration
class ObserversOrchestrator:
    def __init__(self, strategies, result_dir, cache_stats):
        self.strategies = strategies   # [caching, eviction, ttl]
        self.cache_stats = cache_stats

    def observe(self, key, observation_type, metadata):
        """Route events to all registered strategy observers."""
        for strategy in self.strategies:
            # Each strategy decides if this event completes
            # one of its pending experiences
            strategy.observe(key, observation_type, metadata)

        # Log event for analysis
        self._log_event(key, observation_type)

Position in the Caching Research Landscape

RLCache pioneered the application of RL to cache management in 2019. Since then, the ideas have been extended in multiple directions:

Traditional Caching (2019)

RLCache operates on web/database caches with explicit keys, TTLs, and backend storage. The three-agent decomposition provides a template for any cache management system.

KV-Cache in LLMs (2023-2026)

Methods like H2O, NACL, Attention-Gate, Ada-KV apply similar eviction concepts to transformer attention KV caches. The eviction decision maps directly to RLCache's eviction agent, but operates on attention weights rather than access patterns.

Semantic Caching (2024+)

In agentic AI systems, entire LLM responses are cached based on semantic similarity. RLCache's TTL agent concept maps to predicting response validity duration, and the caching agent maps to deciding whether a response is worth caching at all.
