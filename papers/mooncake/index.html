1. Overview

Mooncake is the production serving platform for Kimi, a leading LLM chatbot by Moonshot AI. The paper, awarded Best Paper at FAST'25 (23rd USENIX Conference on File and Storage Technologies), presents a KVCache-centric disaggregated architecture that fundamentally rethinks how LLM inference clusters manage their most precious intermediate state: the KV cache.

Key Insight: More Storage for Less Computation

Modern GPU clusters have significant underutilized CPU, DRAM, SSD, and NIC resources. Mooncake pools these into a distributed KV cache that enables aggressive prefix cache reuse, cutting redundant prefill computation. The global cache achieves up to 136% higher hit rates than node-local caching and reduces prefill GPU time by up to 48%. Mathematically, the paper proves that when B/G > 2ds / (gqa * (apd + bd^2)), cache reuse is beneficial for TTFT -- a condition easily met with modern RDMA networks.

Disaggregated Architecture

Unlike coupled systems (e.g., vanilla vLLM), Mooncake separates prefill and decoding into independent clusters, each with distinct optimization objectives. A central Conductor scheduler orchestrates request routing with KV-cache-aware prefill scheduling and load-balanced decoding assignment. This separation avoids the prefill-decoding interference that plagues long-context serving.

Request Lifecycle (4 Steps)
1) KVCache Reuse: Prefill instance loads reusable prefix cache from Mooncake Store into GPU memory.
2) Incremental Prefill: Prefill completes on uncached tokens; new KVCache is stored back to CPU/DRAM.
3) KVCache Transfer: Transfer Engine streams KVCache layer-by-layer to the decoding node's CPU memory, overlapped with prefill.
4) Decoding: Decoding instance loads full KVCache and joins the continuous batching loop.

Memory Allocator Subsystem

The store's memory management is built on two allocator strategies. The CachelibBufferAllocator wraps Facebook's CacheLib slab allocator, providing efficient fixed-size allocation with 4MB-aligned pools. The OffsetBufferAllocator offers bin-based allocation with precise free-region tracking for variable-sized objects. Both implement a common BufferAllocatorBase interface with allocate, deallocate, and getLargestFreeRegion methods. 

// Buffer allocator hierarchy (allocator.h)
class BufferAllocatorBase {
    virtual std::unique_ptr<AllocatedBuffer> allocate(size_t size) = 0;
    virtual void deallocate(AllocatedBuffer* handle) = 0;
    virtual size_t capacity() const = 0;
    virtual size_t getLargestFreeRegion() const = 0;
};

Each transport implements memory registration, transfer submission, and status polling. The Slice struct is the fundamental transfer unit, containing source/destination addresses, RDMA keys, and completion tracking with lock-free atomic counters.

4. Core Mechanisms

Mathematical Foundation (Section 2.2 of Paper)
The paper provides a formal condition under which KVCache reuse is beneficial. Given GPU computation throughput G, NIC bandwidth B, model dimension d, GQA heads gqa, element size s, and matched prefix length p, reuse reduces TTFT when: B/G > 2ds / (gqa * (apd + bd2)) For LLaMA3-70B on 8xA800 (G = 8x312 TFLOPS), this yields a minimum required bandwidth of only 6 GB/s -- easily met by a single 100 Gbps NIC. On 8xH800, the requirement rises to 19 GB/s, still well within a single NIC's capacity. This proves that distributed KV cache is not just feasible but highly advantageous for production serving.

4.1 KV Cache Pool Management

Paged Block Storage

KV caches are stored as paged blocks in the distributed cache pool. Block sizes are determined by model dimensions and network granularity, typically 16-512 tokens per block (256 KB at the default block size of 256 per the paper's testbed). Each block has a hash key derived from its prefix content for deduplication. The paper notes that for LLaMA3-70B, a single token requires 320 KB of KVCache storage (2 bytes x 2 x d/gqa x layers).

The implementation uses Facebook's CacheLib slab allocator for memory management (CachelibBufferAllocator) alongside an OffsetBufferAllocator for finer-grained control. LRU eviction reclaims blocks when the pool is full, unless the block has an active lease.

Multi-Replica Support

Hot KVCache blocks are automatically replicated across multiple nodes by the Conductor's cache load-balancing strategy. The Replica system supports three types: MEMORY (DRAM/VRAM), DISK (distributed file system), and LOCAL_DISK (node-local SSD). The paper's experiments show that the top-100 hottest cache keys in conversation workloads end up with replicas on nearly every prefill instance, while the synthetic workload with dispersed hotspots shows fewer replicas per key.

Failure Handling and Resilience

In multi-NIC environments, temporary NIC failures are common. The Transfer Engine automatically detects unavailable connections, identifies alternative paths from the topology matrix, and resubmits the 16 KB slice to a different RDMA NIC device. It can also detect problems with RDMA contexts and completion queues, temporarily blacklisting faulty resources until recovery. The implementation includes a WorkerPool for managing RDMA worker threads, an EndpointStore with on-demand connection establishment, and retry logic with configurable max_retry_cnt per slice.

4.3 Conductor Scheduling

KVCache-centric Scheduling Algorithm

The Conductor runs Algorithm 1 from the paper for every incoming request. For each prefill instance, it compares block keys to find the best prefix match length, then estimates TTFT as the sum of:

  • T_queue -- estimated queue wait on that instance
  • T_transfer -- KVCache transfer time (size and network congestion)
  • T_prefill -- GPU computation for uncached tokens (polynomial regression model)

If a remote cache hit is long but the local instance is overloaded, the algorithm may prefer a shorter local prefix plus local computation over a remote transfer. The kvcache_balancing_threshold controls when hotspot migration is triggered: if a remote prefix match is sufficiently longer than the local match (by this multiplicative threshold), the cache is proactively replicated. 

// Algorithm 1: KVCache-centric Scheduling (pseudocode from paper)
for instance in prefill_pool:
    best_len, best_instance = FindBestPrefixMatch(P, block_keys)
    if instance.prefix_len / best_len > kvcache_balancing_threshold:
        prefix_len = best_len                 // use remote cache
        transfer_len = best_len - instance.prefix_len
        T_transfer = EstimateKVCacheTransferTime(transfer_len)
    else:
        prefix_len = instance.prefix_len      // use local cache
        T_transfer = 0
    T_queue = EstimatePrefillQueueTime(instance)
    T_prefill = EstimatePrefillExecutionTime(R, prefix_len)
    TTFT = T_transfer + T_queue + T_prefill
    if TTFT < best_TTFT: update(p, TTFT)
if TTFT > TTFT_SLO: reject(R)  // early rejection

Chunked Pipeline Prefill (CPP)

For long-context requests, Mooncake groups every X nodes in the prefill cluster into a pipelined prefill node group. Input tokens are partitioned into chunks (no larger than prefill_chunk), processed simultaneously by different nodes. CPP offers two key advantages over sequence parallelism (SP): 1) it requires cross-node communication only at pipeline stage boundaries (not all-reduce per layer), leading to better MFU; 2) it naturally handles both short and long contexts without dynamic partitioning overhead. The paper notes this is the first application of pipeline-based acceleration for inference (previously explored only in training).

P/D Ratio Optimization

The paper explores the optimal ratio of prefill (P) to decoding (D) nodes. Increasing prefill nodes reduces TTFT but increases TBT (fewer decoders). The experiments show that a ~1:1 P/D ratio maximizes effective request capacity (around 87-89% effective ratio), as the workloads of prefill and decoding clusters become balanced. In production, Mooncake fixes the P/D ratio and only switches node roles when significant load fluctuations are detected, since online traffic statistics remain generally stable.

6. Integration Ecosystem

Mooncake has become a foundational infrastructure component in the LLM serving ecosystem, with official integrations into all major inference frameworks.

vLLM Integration

Official KV Connector since Dec 2024 (MooncakeStoreConnector and MooncakeTransferEngineConnector). Replaces gloo/nccl for inter-node KVCache transfer in P/D disaggregated setups. Mean TTFT is up to 25% lower than TCP-based transports. Also integrated with vLLM-Ascend for Huawei NPUs and vLLM-Omni for multi-modal pipelines.

SGLang Integration

Serves as HiCache storage backend for SGLang's hierarchical KV caching (Sept 2025), extending RadixAttention across device/host/remote tiers. Also supports EPD disaggregation (Encode-Prefill-Decode) for decoupling multimodal encoders. Integrated expert parallelism kernels enable fault-tolerant MoE inference. Ant Group reports 84% TTFT reduction via cache hits.

TensorRT-LLM Integration

Integrated since Dec 2025 as mooncake_utils in TensorRT-LLM's cache transmission layer for KVCache transfer in P/D disaggregated inference. Also supported as a plugin by NVIDIA's NIXL communication library.

LMCache & Broader Ecosystem

LMCache supports Mooncake Store as a remote connector (Apr 2025). PyTorch Ecosystem member since Feb 2026. TorchSpec uses Mooncake to decouple inference and training for speculative decoding. FlexKV (Tencent/NVIDIA) and xLLM (JD) build hybrid KV cache management on Mooncake. LightX2V uses it for disaggregated video model deployment. checkpoint-engine (Moonshot AI) updated Kimi-K2 (1T params) across thousands of GPUs in ~20s.

7.4 End-to-End Latency Breakdown

At 128k input tokens with 95% prefix cache ratio, TTFT is dominated by Transfer and Load Cache phases rather than Prefill computation. Prefix caching reduces prefill time by 92%. The Schedule, Transfer, and Load Cache steps proceed asynchronously with inference, so their overhead does not block Mooncake throughput. Even accounting for all overhead, prefix caching in Mooncake reduces TTFT by 86% for 128k-token inputs.

7.6 Bandwidth Requirements

The paper studies how system performance degrades under constrained bandwidth. When total communication bandwidth exceeds 100 Gbps, average TTFT remains below 2s (well under the recomputation baseline). Below 100 Gbps, TTFT increases sharply and transfer times diverge from theoretical predictions due to network congestion. Mooncake recommends a minimum of 100 Gbps per node for optimal performance, a threshold easily met by a single modern RDMA NIC.

Key Gaps
The primary gap between the paper and the open-source implementation is the Conductor scheduler -- the global KVCache-centric scheduling component that is central to Mooncake's production deployment. This is proprietary to Moonshot AI. However, the core data-plane components (Transfer Engine and Mooncake Store) are fully open-sourced and have been adopted by all major inference frameworks. The integration with vLLM, SGLang, and TensorRT-LLM effectively allows third parties to build their own scheduling layers on top of these components.
Beyond the Paper
The open-source repository has evolved significantly beyond the FAST'25 paper scope: Elastic Expert Parallelism for MoE models (mooncake-ep), high availability with leader election and snapshotting, local hot cache with Count-Min Sketch frequency admission, SSD/file-backed storage with offloading, disk replica support for persistent caching, TENT (next-gen Transfer Engine rewrite), and the P2P Store for distributed checkpoint management (used to update Kimi-K2 at 1T parameters). The project has also grown from RDMA-only to supporting 11+ transport protocols.

Summary

Mooncake represents a significant contribution to LLM systems research and production practice. Its core insight -- that KV cache management is the central bottleneck in LLM serving -- is validated by both rigorous mathematical analysis and production deployment at massive scale. The open-source release of the Transfer Engine and Mooncake Store has catalyzed adoption across the entire LLM serving ecosystem, with integrations into vLLM, SGLang, TensorRT-LLM, LMCache, and the PyTorch Ecosystem. While the Conductor scheduler remains proprietary, the data-plane components are production-grade, well-engineered, and represent the state of the art in distributed KV cache transfer. The FAST'25 Best Paper Award is well-deserved recognition of work that bridges the gap between systems research and real-world deployment at 100B+ tokens per day.
