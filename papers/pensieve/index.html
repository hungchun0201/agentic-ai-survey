Pensieve: Stateful Large Language Model Serving

arXiv:2312.05516
Stateful Serving Multi-Turn Conversations GPU-CPU Cache 1.14-3.0x Throughput

Overview

The Redundant Recomputation Problem

Existing stateless serving systems (vLLM, TensorRT-LLM) prepend the entire conversation history to each new request and re-process it from scratch. As conversations grow, prefill cost dominates: for 32 requests with 2048-token history, prefill takes 4x longer than 200 steps of generation. Most of this computation is redundant—the same history tokens are processed repeatedly.

Pensieve: Make Serving Stateful

Pensieve saves a conversation's processed context (KV-tokens) in a multi-tier GPU-CPU cache and reuses it across follow-up requests. Only new user prompt tokens need processing. Key challenges addressed: limited GPU memory, token-level eviction between tiers, and handling non-contiguous KV cache scattered across GPU memory.

This analysis reveals that for long conversations, prefill is not a productive operation — it is overhead. The only productive prefill work is on the new user message tokens (typically 10–50 tokens). By caching the KV state of previous turns, Pensieve eliminates up to 97% of prefill FLOPs in a 10-turn conversation.

Three Research Challenges Pensieve Solves

C1
GPU memory is limited: An A100 80 GB holds the model weights plus only a limited number of KV token entries. Solved by the CPU tier — extend capacity 10–20× with pinned DRAM.
C2
KV cache becomes non-contiguous: Eviction and recovery scatter token chunks across GPU VRAM. Standard FlashAttention requires contiguous memory. Solved by the multi-token attention kernel with indirect tensor indexing.
C3
PCIe transfers add latency: Moving KV data between CPU and GPU takes time proportional to data size. Solved by pipelining transfers layer-by-layer with model computation, and by proactive ahead-of-time eviction before GPU pressure builds.

Core Mechanisms

Multi-Tier Cache Hierarchy

Pensieve organises KV cache storage as a two-level hierarchy. GPU VRAM is fast but scarce; CPU DRAM is slower but far larger. The system keeps hot tokens on GPU and offloads cold tokens to CPU, always maintaining enough free GPU slots for the next prefill batch.

Tier 0 — GPU VRAM (fast, limited)
evict cold tokens (swap-out) / recover hot tokens (swap-in)
Tier 1 — CPU Pinned Memory (slower, large)
full eviction when CPU also full
Evicted — must recompute on next hit

Granularity is 32-token chunks rather than pages. This is finer than vLLM's page-level management and allows the eviction policy to reason about individual token groups, resulting in higher effective cache utilisation.

Cost-Aware Eviction Policy

Naive LRU evicts the least recently used tokens. Pensieve improves on this by accounting for the recomputation cost of each evicted chunk. Under causal self-attention, recomputing token at position s in a sequence of length l costs O(s) — earlier tokens are cheaper to recompute than later ones.

# Retention value for a chunk at position s in seq length l # T = time since last access (idle turns) V(s, l, T) = Cost(s, l) / T # Cost(s,l): proportional to number of KV pairs that # depend on position s under causal attention Cost(s, l) = (l - s) * head_dim # early tokens: low cost → evict first

The eviction loop runs during decode pause points (the gap between iteration batches). It is blocked when a PCIe swap-in transfer is already in flight to prevent duplex-bus contention, which was observed to degrade throughput by 18–20%.

Empirical result: Pensieve's cost-aware eviction achieves 4.4% higher CPU cache hit rate and 14.6% fewer recomputed tokens compared to plain LRU on ShareGPT.

Multi-Token Attention Kernel

Standard decoding attention (e.g., FlashAttention, PagedAttention) processes one query token per request — it is a matrix-vector operation. Pensieve's prefill step must process multiple new query tokens against a cached KV context that may be non-contiguous in GPU memory. This requires a custom kernel.

Standard Decoding Kernel
Q: [1 × d] per request
K, V: contiguous pages
MatVec: O(seq_len × d)
Cannot handle multi-query prefill over non-contiguous cache
Pensieve Multi-Token Kernel
Q: [m × d] per request
K, V: non-contiguous chunks via index table
MatMul: O(m × seq_len × d)
Causal masking for ragged batch; built on NVIDIA Cutlass

The kernel uses an index table that maps logical KV positions to physical GPU memory blocks. Single-token generation is treated as the m=1 special case, so prefill and decode share one unified kernel path without an extra dispatch overhead.

Non-Contiguous KV Cache Recovery

When a conversation's KV cache has been partially or fully swapped out to CPU, Pensieve must recover it to GPU before the next prefill. The complication is GPU memory fragmentation: free slots may be scattered, so the recovering data cannot be placed contiguously. Pensieve handles this in two steps.

1
CPU-side reorganisation: The CPU copies chunks from their CPU buffer locations into a contiguous staging region in pinned memory. This is cheap because CPU bandwidth to pinned DRAM is high and runs in parallel with GPU computation.
2
Layer-pipelined PCIe transfer: KV data for layer L+1 is DMA-transferred over PCIe while the GPU executes the attention computation for layer L. GPU stream events enforce ordering so that the attention kernel for layer L+1 only starts after its KV data has arrived.
If a small fraction of tokens was dropped from both GPU and CPU (fully evicted), Pensieve splits the request into two sub-requests: one that recomputes the dropped tokens, and one that processes the new prompt tokens using the recovered cache. The two sub-requests are joined before output is returned to the user.

Ahead-of-Time Swap-Out

Proactive eviction begins when GPU free slots fall below a 25% watermark. Transfers overlap with decode compute, hiding the PCIe latency entirely under normal load. The scheduler explicitly separates the swap-out and swap-in phases: it will not issue a new swap-out while a swap-in transfer is in progress to avoid saturating the PCIe bus in both directions simultaneously.

Unified Batch Scheduling

Pensieve's scheduler batches prefill and decode requests together in every iteration. FCFS admission prevents head-of-line blocking. Requests that share cached KV history are colocated to maximise hit rates. Single-token decode is a degenerate m=1 case of multi-token attention, so no separate kernel launch is needed — GPU utilisation is improved compared to alternating prefill / decode passes.

Why Larger Models Benefit More

Llama2-70B uses Grouped Query Attention (GQA) with group size 8, which reduces the KV cache footprint by 8× compared to full MHA. This means Pensieve can keep proportionally more conversation history in GPU VRAM, boosting the GPU hit rate. For OPT-13B (no GQA), the GPU is more memory-pressured and more tokens must be swapped to CPU or fully evicted.

Robustness to User Think Time

Pensieve's gains are not sensitive to user think time (the gap between conversation turns). Testing across think times of 30s, 60s, 120s, 300s, and 600s on ShareGPT shows consistent 1.6–3.0x improvement. Even with 600-second gaps, the cached KV state is retained and reused — demonstrating that practical deployment will sustain the benefit across diverse user interaction patterns.

On the UltraChat dataset (longer conversations, more turns), the absolute gains are larger because the cached history grows over time, making each incremental prefill proportionally cheaper to skip.

Evaluation Metrics

Primary: Throughput (tokens/s)

Total output tokens generated per second across all concurrent requests. Measured at steady state under a fixed arrival rate. Higher is better.

Secondary: Cache hit rates, recomputed tokens

GPU hit rate (fraction of history tokens found in GPU VRAM), CPU hit rate (fraction found in CPU after GPU miss), and the number of tokens that must be recomputed due to full eviction.

Latency (TTFT, TBT) is not the primary focus of Pensieve's evaluation. The paper argues that in multi-turn serving, system throughput (conversations served per unit time) is the more meaningful metric because latency is already bounded by think time between turns.

Significance

Stateful Serving Paradigm

Establishes that LLM serving should be stateful for multi-turn workloads. Eliminates the dominant cost: redundant history recomputation across conversation turns. This insight applies beyond chat — any workflow that reuses a long prompt prefix (RAG, tool descriptions, few-shot examples) stands to benefit.

Novel Attention Kernel

Multi-token attention over non-contiguous KV cache is a reusable building block, enabling efficient prefill with scattered cached context. The kernel is not Pensieve-specific — it can serve as the foundation for any system that stores KV cache in a paged or fragmented layout and needs to prefill new tokens against it.

Cost-Aware Eviction

Eviction accounts for recomputation cost (causal attention makes earlier tokens cheaper). 4.4% higher CPU cache hit rate and 14.6% fewer recomputed tokens vs LRU. The principle generalises: any tiered cache whose eviction cost is non-uniform should use a cost-proportional retention value rather than pure recency.

Real-World Validation

Evaluated on ShareGPT (real user conversations) and UltraChat. Benefits persist across user think times from 30 to 600 seconds, demonstrating robust practical value. This range covers both fast-paced user interaction and slower, agent-driven conversation loops.

Why Stateful Serving Matters for Agentic AI

Agentic AI systems differ from single-turn query-response patterns in a critical way: they maintain persistent context across many steps. A ReAct agent executing a 20-step tool-use loop is re-presenting the same accumulated scratchpad at every step. A RAG pipeline with fixed tool descriptions prepends identical text with every query. An LLM-driven workflow with system prompts carries the same prefix across all calls. In every case, the KV cache for the shared prefix is recomputed from scratch in a stateless system.

ReAct / Tool-use agents
Growing scratchpad reused at every step. Pensieve saves the full history KV, so each step only processes the latest action token.
Multi-session user assistants
Users return after hours. Pensieve's CPU tier keeps conversation history alive without GPU occupancy between sessions.
RAG + shared system prompts
Identical tool/document prefixes across many users. Prefix caching (a special case of Pensieve's cache) amortises this across all requests sharing the prefix.

Limitations

●
Single-node only: Pensieve assumes all requests for a conversation are served by the same node. Distributed KV cache sharing across nodes (as explored by systems like DistServe or Mooncake) is not addressed and would require a network-tier extension.
●
No prefix sharing across conversations: The cache is per-conversation. Multiple users sharing the same system prompt or RAG context cannot benefit from cross-session prefix deduplication. SGLang's RadixAttention or systems using a shared prefix pool would be complementary.
●
PCIe bandwidth ceiling: At very high request rates, PCIe becomes the bottleneck when many conversations need simultaneous CPU-to-GPU recovery. The paper mitigates this by separating swap-in and swap-out phases, but the fundamental bandwidth limit remains.
●
Speculative decoding not explored: The interaction between Pensieve's non-contiguous KV cache and speculative decoding (which requires rolling back the KV state on token rejection) is a non-trivial engineering challenge that the paper leaves as future work.

Future Directions

Disk-tier extension

Adding NVMe SSD as a third tier below CPU DRAM could extend conversation retention from hours to days. This would be particularly valuable for personal AI assistants where users expect context to persist across multi-day sessions.

Cross-conversation prefix sharing

Combining Pensieve's tiered cache with a trie-based prefix deduplication structure (as in SGLang RadixAttention) could eliminate redundant KV storage for shared system prompts across many concurrent users.

Distributed stateful serving

Extending stateful caching to multi-node clusters via KV cache migration or replicated CPU buffers would remove the single-node affinity constraint, enabling load-balanced deployment at cloud scale.

Learned eviction policy

The current cost-aware formula is hand-designed. A learned predictor that estimates per-token reuse probability from conversation metadata (turn length, user idle pattern) could further improve cache efficiency beyond the 14.6% gain over LRU.

Key Takeaways for Practitioners

1.
If your workload involves multi-turn conversations with histories longer than ~500 tokens, Pensieve-style stateful serving will reduce prefill latency and increase system throughput significantly. The benefit grows with history length.
2.
Models with GQA (e.g., Llama2-70B, Llama3) have dramatically smaller KV footprints and benefit most from tiered caching. Prefer GQA models when memory pressure is a concern.
3.
Session affinity (routing all requests from one conversation to the same GPU node) is a prerequisite for stateful serving. This adds a constraint to load balancing that must be handled by the routing layer (e.g., consistent hashing on session ID).
4.
CPU pinned memory is the cheapest capacity extension. A server with 512 GB RAM can hold 10–20× more conversation history than the GPU alone. This is available on commodity cloud instances today with no hardware changes.

System Implementation

Implementation Scope

Pensieve is a production-grade research prototype written in C++ and CUDA, totalling approximately 7,000 lines of new code on top of a baseline inference engine. It is not a fork of vLLM but shares the conceptual goal of paged KV storage, replacing the page table with Pensieve's tiered index structure.

Three Core Subsystems

1. Cache Manager

Maintains per-conversation KV metadata: for each token chunk, its tier (GPU / CPU / evicted), physical address, and last-access timestamp. Implements the retention-value priority queue for eviction decisions. Exposes an API to the scheduler for querying availability and requesting transfers.

2. Transfer Engine

Manages all asynchronous DMA transfers between GPU and CPU via CUDA streams. Tracks in-flight transfers with GPU events. Implements the pipeline schedule that interleaves layer-N attention with layer-(N+1) KV transfer. Enforces the PCIe duplex rule: blocks eviction stream if recovery stream is active.

3. Multi-Token Attention Kernel (CUDA)

A single CUDA kernel built on NVIDIA Cutlass GEMM primitives. Given a ragged batch of requests (each with a different number of new query tokens and a different length of cached KV), it computes scaled dot-product attention with causal masking. The KV input is an indirect tensor whose physical locations are given by a per-request index table, enabling non-contiguous access without gather preprocessing.

Scheduler Iteration Loop

Each scheduling cycle consists of the following steps, executed at every decode step boundary:

Step 1  Admit new requests (FCFS) up to GPU memory budget
Step 2  Query Cache Manager: which admitted requests need CPU→GPU recovery?
Step 3  Issue recovery transfers (if PCIe not busy with eviction)
Step 4  Build unified batch: (prefill requests) + (active decode requests)
Step 5  Launch multi-token attention kernel over full batch
Step 6  Return generated tokens; update KV cache metadata
Step 7  If GPU free slots < 25%: issue ahead-of-time eviction (GPU→CPU)
Step 8  Repeat

Related Work & Context

KV Cache Management Landscape

KV cache management has become a central concern in LLM serving research. Pensieve addresses the multi-turn stateful case, but several contemporaneous systems address adjacent problems.

System Key idea Relation to Pensieve

Historical Context: Why Now?

Stateful caching was not practical in earlier serving systems because: (1) models were smaller and history recomputation was fast; (2) GPU memory was precious and sharing it between the model weights and a KV cache store seemed wasteful; (3) multi-turn interactions were less common. The combination of very large models, long multi-turn agentic sessions, and tight GPU supply in 2023–2024 made Pensieve's trade-offs favorable.

Industry Adoption Signal

Several Pensieve-adjacent ideas have appeared in production systems since the paper's publication. vLLM v0.4+ introduced prefix caching (GPU-only). OpenAI and Anthropic have announced KV cache persistence for paid tiers. The SGLang runtime added RadixAttention for prefix sharing. These independent convergences validate the core thesis: multi-turn serving must be stateful.

Core insight, restated: In a stateless serving system, the marginal cost of each additional conversation turn grows linearly with history length. In Pensieve, the marginal cost is bounded by the cost of processing only the new tokens — O(new_tokens) instead of O(total_history). For agentic workloads where history grows rapidly, this is the difference between a system that degrades under long sessions and one that remains efficient.
