Autellix: An Efficient Serving Engine for LLM Agents as General Programs

The first LLM serving system that treats agentic programs as first-class citizens, achieving 4-15x throughput improvement over vLLM through program-aware scheduling.

01 Overview

Large language model (LLM) applications are evolving beyond simple chatbots into dynamic, general-purpose agentic programs. These programs scale LLM calls and output tokens to help AI agents reason, explore, and solve complex tasks. However, existing LLM serving systems like vLLM ignore dependencies between programs and calls, missing significant optimization opportunities. Autellix is the first LLM serving system designed to run programs, not individual LLM calls, inspired by how traditional OS schedulers manage entire processes across CPU cores.

Key Insight: Existing LLM serving systems treat each LLM call independently, but agentic programs submit tens to hundreds of related calls forming dynamic DAGs. By tracking cumulative program-level statistics (service time, waiting time, thread metadata), Autellix can prioritize calls from shorter programs to complete first, eliminating both call-level and program-level head-of-line blocking.

Problem: Call-Level HoL

Long-running LLM calls with many decode steps block shorter calls from executing, causing cascading delays. FCFS scheduling in vLLM amplifies this effect under heavy load.

Problem: Program-Level HoL

Even with preemptive scheduling (MLFQ), newer programs are repeatedly delayed by subsequent calls from older, longer programs. Without program context, schedulers cannot distinguish these cases.

Solution: Program-Aware Scheduling

Autellix prioritizes LLM calls by their program's cumulative service time. Short programs finish first, and critical-path awareness further optimizes multi-threaded DAG execution.

02 System Architecture

Autellix is a higher-level serving engine built as a layer on top of existing LLM engines such as vLLM v0.6.1. It operates as a stateful system where programs execute on users' local machines, establish sessions with the Autellix backend, and submit LLM calls over time. A global process table tracks program metadata, enabling both the scheduler and load balancer to make program-aware decisions.

03 PLAS: Program-Level Attained Service

PLAS extends the Least-Attained-Service (LAS) algorithm from operating systems to the program level. For single-threaded programs, PLAS prioritizes LLM calls based on their source program's total cumulative service time. Programs that have consumed less compute get higher priority, allowing shorter programs to complete first and reducing overall waiting time.

p(cj) = Σck.id = cj.id, k < j tk
PLAS Priority: The priority of call cj equals the sum of execution times of all prior completed calls from the same program. Lower values = higher priority.
Non-Clairvoyant Design: Unlike Shortest-Job-First (SJF) or Shortest-Remaining-Processing-Time (SRPT), PLAS requires zero prior knowledge of program runtimes. It only uses the cumulative service time of calls that have already completed, making it fully non-clairvoyant.

Multi-Level Queue-Based Discretization

To avoid degenerate round-robin behavior from continuous priorities, Autellix discretizes program priorities into K queues (Q1 to QK), inspired by MLFQ in operating systems. Each queue covers a priority range, and calls are demoted to lower-priority queues as their program accumulates more service time.

Anti-Starvation Mechanism

Discrete prioritization risks starving long, low-priority programs. Autellix promotes a call c from program p back to Q1 when the ratio of total waiting time to total service time exceeds a threshold:

Wtotal / Ttotal ≥ β
Where Wtotal = Wp + Wc (program + call wait) and Ttotal = Tp + Tc (program + call service time). After promotion, wait and model_time counters are reset.

04 ATLAS: Adaptive Thread-Level Attained Service

While PLAS handles single-threaded programs effectively, multi-threaded programs (like MCTS, Map-Reduce) form dynamic DAGs where a program's completion time is dictated by the critical path -- the longest sequence of dependent calls from start to finish. ATLAS is a pragmatic generalization of PLAS that incorporates critical-path awareness.

p(cj) = { 0  if cj is root  |  maxck ∈ P(cj) { p(ck) + tk }  otherwise }
ATLAS Priority: p(cj) recursively estimates the longest chain of accumulated service time leading to cj, providing a non-clairvoyant critical path estimation.

    Best-case: 11 vs worst-case: 14 units

Key Simplification: Rather than tracking full DAG dependency structures, ATLAS maintains a single scalar per program: the longest observed critical path. Each active LLM call inherits this value as its initial priority and updates the program's critical path estimate upon completion if its own path is longer. This simple mechanism continuously refines the estimate without tracking individual dependencies.

PLAS vs ATLAS

  • PLAS: Uses total cumulative service time across all threads as priority. Optimal for single-threaded programs.
  • ATLAS: Uses maximum critical-path service time. Naturally groups parallel calls from the same program, preventing straggler delays.
  • Both are non-clairvoyant and share the same multi-level queue infrastructure.

Thread Grouping Benefit

Because all calls of a given program derive priorities from the same process table entry, ATLAS naturally groups a program's parallel calls at similar priority levels. This prevents the scenario where MLFQ inadvertently disrupts threads in the same program by aggressively prioritizing shorter individual requests from other programs.

Core Scheduling Algorithm

procedure SCHEDULER(Queues Q1,...,QK, Table pt) for c in C_arrived do // Arriving LLM calls c.service = pt[c.pid].service // Fetch priority from process table c.q_idx = i, s.t. Q_ilo ≤ c.service ≤ Q_ihi // Assign to priority queue Qc.q_idx.append(c), c.quanta = Qc.q_idx.quanta end for for c in {Q1, Q2, ..., QK} do if c.finished() then // Finished jobs update table UPDATE_PROCESS_TABLE(c, pt) Qc.q_idx.remove(c) end if if c.quanta ≤ 0 then // Call demotion Qc.q_idx.remove(c), Qc.q_idx+1.append(c) c.q_idx += 1, c.quanta = Qc.q_idx.quanta end if if wait/service ≥ β then // Anti-Starvation Qc.q_idx.remove(c), Q1.append(c) c.wait = 0, c.model_time = 0 // Reset counters end if end for Bout = [] // Schedule next batch for c in {Q1, Q2, ..., QK} do if engine.can_fit(c) then Bout.append(c) else break end for end procedure
procedure UPDATE_PROCESS_TABLE(Call c, Table pt) pd = pt[c.pid] // PLAS: total service time // ATLAS: max critical path = max(current, this call's chain) pd.service = max(pd.service, c.model_time) // Update other metrics (wait, thread data, timestamps)... end procedure

05 Locality-Aware Load Balancer

As agentic workloads scale, deploying multiple engine replicas is necessary. Autellix's load balancer exploits a critical observation: LLM calls within the same program achieve 90%+ KV-cache hit rates due to shared conversation history, while cross-program cache hit rates decay exponentially with input length. The load balancer distinguishes between short and long requests to optimize for both data locality and load balance.

KV-Cache Hit Rate Analysis

  • Intra-program: Cache hit rates remain above 90% across all input lengths due to shared conversation context and history.
  • Inter-program: Cache hit rates decay exponentially with input length; programs only share system prompts, so longer inputs see diminishing returns.
  • Threshold of 2048 tokens separates requests where locality provides negligible vs substantial benefit.

Load Balancer Algorithm

procedure LOAD_BALANCER(Call c, Table pt, Engines) if LEN(c.tokens) ≤ 2048 then assigned = LEAST_USED(Engines) // Short: balance load else if c.pid in pt then assigned = pt[c.pid] // Long: use existing engine else assigned = LEAST_USED(Engines) pt[c.pid] = assigned // Record for future calls end if end if return assigned end procedure
Why not prefix trees? Prior work (e.g., SGLang) uses complex prefix trees to quantify data locality. Autellix's simple threshold-based approach avoids this overhead while achieving comparable or better results, because agentic workloads have a clear bimodal structure: calls within a program share extensive context, while cross-program sharing is minimal beyond system prompts.

Tail Latency Performance (P95 / P99)

Fairness Under Preemption: Preemptive scheduling can reduce average latency but risks increasing tail latency by starving long-running programs. Autellix's anti-starvation mechanism maintains consistently lower P95/P99 latencies in 7 of 8 test scenarios, improving throughput by up to 1.7x at P95/P99 targets compared to MLFQ and vLLM-opt.

Memory Management Optimizations

Multi-Step Scheduling

Instead of running the scheduler at every decoding step, Autellix runs it once every N steps. This reduces total KV-cache swaps while overprovision ensures new requests fill in as others complete early. Reduces swapping frequency without impacting responsiveness.

Batched Swap Kernel

vLLM transfers each KV block individually via cudaMemcpyAsync, creating fragmented transfers. Autellix gathers all KV blocks into a contiguous buffer for a single bulk transfer, reducing swap times by 3-7x and achieving 1.3x higher throughput from the kernel alone.

In offline scenarios where all programs are submitted at the start, Autellix consistently outperforms baselines:

Comparison to Optimal Policy (SRPT)

Clairvoyant Upper Bound: In simulation with full knowledge of program runtimes, Shortest Remaining Processing Time (SRPT) represents the optimal policy. Autellix outperforms all other non-clairvoyant policies (FCFS, Round Robin, MLFQ) but a noticeable gap remains relative to SRPT, showing that prior knowledge of program structure could significantly boost performance. This motivates future work on compiler optimizations like branch prediction and speculative execution for agentic programs.

08 Significance: Why Program-Level Scheduling Matters

Autellix represents a fundamental shift in how LLM serving systems are designed: from optimizing individual request latencies to optimizing end-to-end program completion times. This paradigm shift has broad implications for the emerging agentic AI ecosystem.

Paradigm Shift: Request-Level vs Program-Level Optimization

Traditional: Request-Level (vLLM)

Each LLM call scheduled independently
No awareness of program structure
Optimize TTFT / TPOT per request
Stateless API (no session tracking)
⇒

Autellix: Program-Level

Calls scheduled by program context
Dynamic DAG tracking via process table
Optimize program end-to-end latency
Stateful sessions with KV-cache locality

Key Contributions

  • Formalization: First to formalize agentic programs as dynamic, non-deterministic DAGs of LLM calls and external interrupts.
  • PLAS/ATLAS: Two non-clairvoyant scheduling algorithms that extend OS-inspired attained-service concepts to LLM serving, handling both single- and multi-threaded programs.
  • Locality-aware LB: Simple threshold-based load balancer that outperforms complex prefix-tree approaches for agentic workloads.
  • Practical system: ~5K lines atop vLLM, easily deployable with existing programming and agent frameworks via stateful API.

Future Directions

  • Compiler optimizations: Branch prediction and speculative execution for agent DAGs -- anticipating next LLM calls before current ones finish.
  • Post-training integration: Accelerating distributed RL training (e.g., Deepseek-R1, OpenAI o1) by reducing batch makespan during on-policy sampling.
  • Closing the SRPT gap: Exploiting partial knowledge of program structure to approach clairvoyant optimal performance.
  • Security: Current prototype lacks safeguards against user modification of the underlying package -- an area for hardening.
The OS Analogy: Just as operating systems evolved from running individual programs sequentially (batch processing) to managing multiple processes with sophisticated schedulers (time-sharing), LLM serving systems must evolve from optimizing individual requests to managing entire agentic programs. Autellix represents this transition: the process table is analogous to the OS process control block, PLAS/ATLAS to CPU scheduling policies (like Completely Fair Scheduler), and the locality-aware load balancer to NUMA-aware memory allocation.
