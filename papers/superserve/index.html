Overview

The Three Requirements

ML inference serving must simultaneously satisfy: R1 Latency (meet SLO deadlines), R2 Accuracy (maximize prediction quality), and R3 Resource Efficiency (minimize GPU usage). Under unpredictable, bursty request rates (peak 50x average in Microsoft Azure Functions), static model selection forces a painful tradeoff: provision for peak (waste resources) or optimize for average (miss SLOs during bursts).

SuperServe: Dynamic Subnet Actuation

SuperServe introduces SubNetAct, a mechanism that dynamically routes requests through different subnets within a single weight-shared SuperNetwork. Instead of loading/switching between separate models (100ms+ latency), SubNetAct actuates subnets in-place in under 1ms. Combined with SlackFit, a reactive scheduling policy, SuperServe automatically navigates the latency-accuracy tradeoff in real-time.

Key Mechanisms

SubNetAct: Instant Model Actuation

Three novel operators inserted into SuperNets: LayerSelect (skip/include layers by depth D), SubnetNorm (per-subnet BatchNorm statistics, only 500x smaller than shared layers), WeightSlice (select channel subsets by expand ratio E and width multiplier W). Together they enable near-instantaneous (<1ms) actuation of any subnet from ~10^19 architecture space.

SlackFit: Slack-Based Scheduling

Uses remaining slack of the most urgent query as a proxy for system load. High slack → high latency bucket → higher accuracy models. Low slack (traffic burst) → low latency bucket → lower accuracy but faster models. Operates on Pareto-optimal subnets (~10^3) with monotonic latency/accuracy properties for O(log n) decision-making.

SuperNetwork Architecture

Single OFAResNet-based supernet trained once via NAS. Extracted subnets share weights, vastly superior to individually trained ResNets at equivalent FLOPs. Memory footprint 2.6x lower than loading 6 separate models while spanning 73-80% accuracy range at 0.9-7.5 GFLOPs.

Latency Bucket Design

Pareto-optimal subnets organized into evenly-sized latency buckets. Each bucket contains (subnet, batch_size) pairs with similar latency but different accuracy. SlackFit selects the bucket matching available slack, then picks maximum batch size for throughput. Three properties exploited: P1 monotonic latency with batch size, P2 monotonic latency with accuracy, P3 monotonic latency difference across batch sizes.

Significance

Reactive Over Predictive

Challenges the conventional wisdom that model switching is too slow for reactive policies. SubNetAct enables orders-of-magnitude faster actuation than model loading, making reactive scheduling practical for sub-second bursty traffic.

Continuous Accuracy Dial

Instead of discrete model choices, SuperServe provides a continuous accuracy-latency knob. The system automatically discovers the best operating point in real-time, eliminating manual model selection.

Fault Tolerance

When workers fail, SubNetAct automatically shifts to lower-accuracy but faster subnets on remaining workers, maintaining 0.999 SLO attainment even with 50% capacity loss. No reconfiguration needed.

Scalability

Linear scaling to 33,000 QPS across 32 workers while maintaining 0.999 SLO attainment. The architecture is designed for production deployment with 17,500 lines of C++ and gRPC-based communication.
