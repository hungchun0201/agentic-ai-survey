Overview

ThunderAgent introduces a program-aware abstraction for agentic inference that treats entire multi-turn workflows as first-class LLM Programs, rather than isolated request-response pairs. This end-to-end view enables unified management of heterogeneous resources including KV caches, system states, disk memory, and network ports.

Core Insight

Existing inference systems (vLLM, SGLang + Kubernetes) treat each LLM call as a stateless request, losing context about the overall agentic workflow. This causes KV cache thrashing (up to 7.14x latency increase), cross-node memory imbalance (51% divergence), and resource leakage from orphaned tool environments.

What is an Agentic Program?

ThunderAgent abstracts each agentic workflow as a tuple that persists across multiple model invocations and tool executions:

P = ⟨ ID , c , T , L , tau , s ⟩

ID -- Global Identifier

Unique identifier linking all requests from the same workflow, enabling cross-turn KV cache tracking.

c -- Context Token Count

Number of tokens in the KV cache, tracking the workflow's GPU memory footprint in real time.

T -- Tool Environments

Set of tool environments (Docker sandboxes, DB connections) tied to the program for lifecycle management.

L -- Backend Placement

GPU node assignment for spatial locality, enabling dynamic migration across data-parallel replicas.

tau -- Execution Phase

Current phase: Reasoning (actively generating tokens) or Acting (waiting for tool execution).

s -- Scheduling Status

One of {Active, Paused, Terminated}, controlling resource allocation and garbage collection.

1. KV Cache Thrashing

Request-level scheduling evicts KV caches during tool execution intervals. When tools complete, the system must re-prefill the entire interaction history. This increases end-to-end latency by up to 7.14x and worsens as parallel workflow count increases.

2. Cross-Node Memory Imbalance

KV-aware routers pin workflows to fixed nodes to maximize cache locality. But workflows have heterogeneous memory footprints, causing up to 51% memory divergence between data-parallel nodes over 37+ minutes of RL rollout.

3. Tool Lifecycle Obliviousness

Tool orchestrators cannot synchronize with LLM engines. Unused Docker sandboxes and API servers accumulate, causing disk usage to exceed system capacity. Environment preparation time grows linearly with batch size.

Scheduling Policy: Pause/Restore with Hysteresis

Periodic Thrashing Detection

Unlike baseline schedulers that only check admission conditions, ThunderAgent implements a periodic monitor that evaluates memory at fixed intervals Delta-t, detecting thrashing mid-execution as context lengths grow during decoding.

Scheduling Scores

Both pause and restore use a shortest-first policy based on Lemma 4.1 (recomputation cost scales quadratically with context length), with phase-aware prioritization:

Restore Score

S_restore(P) = 1 / c_P + I(tau = Reasoning)

Prioritizes restoring Reasoning programs (actively decoding) with smallest context to maximize decode + prefill throughput.

Pause Score

S_pause(P) = 1 / c_P + I(tau = Acting)

Prioritizes pausing Acting programs (waiting for tools) with smallest context to minimize recomputation cost and free cached memory.

Time-Decay for Acting Programs

To balance caching cost against recomputation, acting programs' token weights decay exponentially with tool execution time:

// Thrashing condition with time-decay for acting programs
C_total < SUM(c_p, p in L, tau=R)    // All reasoning programs
       + SUM(c_q * f(t_q), q in L, tau=A) // Acting programs with decay

// Optimal decay function (memoryless property, proven in Appendix E.1)
f(t) = 2^(-t)  // Exponential decay

Cost Model: Space-Time Product

ThunderAgent formalizes GPU resource consumption using the Space-Time Product (STP), isolating effective costs from non-productive usage:

Cost_total = Cost_decode + Cost_prefill + Cost_recompute + Cost_unused + Cost_caching

Effective Work (minimize)

Cost_decode + Cost_prefill -- the only components contributing to actual inference throughput.

KV Thrashing Waste

Cost_recompute -- re-prefilling evicted context. Scales quadratically with context length (Lemma 4.1): Cost ~ c_i^2.

Imbalance Waste

Cost_unused -- idle GPU memory from cross-node imbalance. Addressed by global program-aware waiting queue.

Idle Caching Waste

Cost_caching -- holding KV cache for acting programs during tool execution. Managed by time-decay eviction.

Quadratic Recomputation Cost (Lemma 4.1)

Given a program P_i with context length c_i, the recomputation cost of re-prefilling its KV cache is proportional to c_i^2. This is because re-prefill processes c_i tokens, and during this time the memory footprint ramps from 0 to c_i. The STP integral yields Cost_recompute proportional to c_i^2. This motivates the shortest-first eviction policy.

Tool Resource Management

1

Hook-Based Garbage Collection

Lifecycle hooks couple tool resource persistence to program scheduling status. When a program reaches Terminated, an immediate teardown reclaims Docker sandboxes, network sockets, and compute slots -- preventing the linear resource leakage seen in baseline systems.

2

Asynchronous Environment Preparation

ThunderAgent monitors the global waiting queue: when a high-priority program approaches the restore threshold, the system asynchronously restores its execution environment before GPU memory is allocated. This hides initialization overhead for tool-heavy workloads like coding and science agents.

3

Global Program-Aware Waiting Queue

Unifies waiting queues across all backend replicas. Paused programs are node-agnostic (KV cache released), enabling dispatch to any replica with available capacity. This improves cross-node memory balance without sacrificing cache locality.

Key Robustness Finding

ThunderAgent maintains maximum throughput even when parallel workflow count scales beyond GPU memory limits. Baseline systems suffer severe throughput collapse in this regime, while ThunderAgent automatically adapts without manual tuning, a capability critical for real-world deployments.

Repository Analysis

ThunderAgent ships as a ~3,600-line Python package built on FastAPI + httpx. The runtime is a stateful reverse proxy that sits in front of one or more vLLM / SGLang / SkyRL engines, tracks each agentic workflow as a "Program", and periodically rebalances KV-cache pressure across backends. Below we walk through the actual source — every file referenced has been verified in the cloned repo.

Claims vs Implementation

Each row pairs a concrete claim in the arXiv paper with the file in the repo that realises (or fails to realise) it. Legend: ✓ faithfully implemented · ~ partially implemented or diverges · ✗ not in code.

Paper Claim Status Where in the Code
Program as the first-class scheduling unit carrying (ID, c, T, L, τ, s) ✓ Program dataclass in ThunderAgent/program/state.py — every field present (context_len, backend_url, status, state). T (tool set) is carried indirectly via docker_ids in the request's extra_body.
Program-aware scheduler: reasoning > acting priority; shortest-first eviction ✓ MultiBackendRouter._greedy_resume in scheduler/router.py groups candidates as reasoning_group + new_program_group + acting_group and within each group sorts ascending by total_tokens (shortest-first). _pause_until_safe pauses ACTING programs first, then REASONING, smallest first.
Periodic monitor running every Δt on each DP backend ✓ _scheduler_loop() sleeps self._scheduler_interval (default 5.0s, CLI --scheduler-interval) and calls _scheduled_check() which iterates every registered backend.
Exponential time-decay f(t) = 2^(-t) on idle acting programs ✓ BackendState.remaining_capacity_with_decay() in backend/state.py: acting_decayed += p.total_tokens * (2.0 ** -t) where t = now − p.acting_since. Opt-in via --use-acting-token-decay.
AIMD-style load control with λ_max / λ_min thresholds ~ Paper shows two thresholds; released code uses a simpler zero-based check — pause when remaining_capacity() < 0, resume whenever headroom exceeds BUFFER_PER_PROGRAM (= 100 tokens). No λ_max / λ_min knobs in config.py or __main__.py.
Best-Fit-Decreasing bin-packing for resume placement ✓ _greedy_resume() sorts candidates descending by total_tokens, sorts backends descending by remaining_capacity, places the largest program on the backend with the most headroom, then re-sorts. Explicitly documented as "BFD" in the docstring.
KV-cache locality preserved on reassignment ✓ _pause_program() records state.origin_backend; active (unpaused) programs are never migrated. Migration happens only after the KV cache has already been evicted by vLLM, so there is nothing to lose.
Global program-aware waiting queue ✓ MultiBackendRouter.global_waiting_queue: Dict[str, PausedInfo] guarded by pause_resume_lock. Atomic claim via _claim_specific_paused().
Hook-based tool-resource GC on program termination ~ Client-facing only: release_program() in router.py clears router state and unregisters from the backend. There is no Docker/sandbox lifecycle hook in the released repo — the client harness is expected to teardown containers after POSTing /programs/release. The 4.2× disk-saving number therefore depends on the agent harness.
Asynchronous environment preparation (prebuild tools while waiting for GPU) ✗ Described on thunderagent.ai and in the paper's §3, but no corresponding code path exists in this repo — no prebuild hook, no async tool-init callback on PAUSED programs.
vLLM + SGLang + Kubernetes integration ~ vLLM (backend/vllm_metrics.py) and SGLang (backend/sglang_metrics.py) adapters ship; SkyRL is a bonus third backend. No Kubernetes manifests / Operators / HPA logic exist in the repo — the K8s claim is not backed by released code.
3.58× peak serving throughput, 3.92× RL rollout throughput ~ Benchmark harnesses sit under examples/inference and examples/rl_training, but no committed run configs pin concurrency to the paper's numbers — reproducing the exact 3.58× / 3.92× requires setting parallelism manually per Fig. 6 / Fig. 8.
~100% KV-cache hit rate at steady state ✓ ProfileState.on_request_end records kv_hit_rate = cached_tokens / prompt_tokens; the CSV output (step_profiles.csv) is the exact data series used in the paper's Fig. 10.
OpenAI-compatible drop-in: only add program_id ✓ app.get_program_id() reads payload["program_id"] or payload["extra_body"]["program_id"], defaulting to "default". All other OpenAI fields pass through verbatim.

How to Reproduce

Hardware requirements
  • Single-node serving: 1× 8×H100 (80 GB) for Qwen3-32B; smaller models (Llama-8B, Qwen3-14B) on 1× RTX 6000 Ada.
  • RL-rollout: 2× 8×H100 nodes (as tested on thunderagent.ai).
  • ThunderAgent itself has no GPU dependency — pure Python + FastAPI + httpx, runs on the login/head node.

1. Install

# Python 3.10+
git clone https://github.com/HaoKang-Timmy/ThunderAgent.git
cd ThunderAgent
pip install -e .

2. Start an inference backend (example: vLLM)

uv pip install vllm --torch-backend=auto
vllm serve Qwen/Qwen3-32B --port 8000

3. Launch ThunderAgent

thunderagent \
  --backend-type vllm \
  --backends http://localhost:8000 \
  --port 9000 \
  --router tr \
  --scheduler-interval 5 \
  --acting-token-weight 1.0 \
  --use-acting-token-decay \
  --metrics --profile

For multi-backend runs pass a comma-separated list, e.g. --backends http://node-0:8000,http://node-1:8000.

4. Send requests with program_id

from openai import OpenAI
client = OpenAI(base_url="http://localhost:9000/v1", api_key="x")
client.chat.completions.create(
    model="Qwen/Qwen3-32B",
    messages=[...],
    extra_body={"program_id": "task-42",
                "docker_ids": ["sandbox-42"]},
)

Status & Roadmap (from thunderagent.ai)

The project site lists three headline numbers beyond the paper: 1.5–3.6× serving throughput, 1.8–3.9× RL rollout on 2×8×H100, and 4.2× disk savings from lifecycle-aware GC. The released repo fully backs the first two; the lifecycle-aware GC is still a client-side contract (expect it as a first-class hook in a future release). Kubernetes integration is also advertised but not yet in the repo.
