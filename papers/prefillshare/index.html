PrefillShare: A Shared Prefill Module for KV Reuse in Multi-LLM Disaggregated Serving

Eliminating redundant prefill computation across multi-agent pipelines by decoupling the shared base model from task-specific decode workers

arXiv 2602.12029 Preprint 2026 Multi-LLM Serving

Shared prefill pools · Prefix cache reuse across heterogeneous fine-tuned models · Disaggregated inference

1 Overview

Modern agentic AI pipelines orchestrate multiple specialized LLMs in a chain: a Planner proposes solutions, a Coder generates code, and a Reviewer validates it. Despite each agent being a separate fine-tuned model, they typically share a common backbone (e.g., LLaMA-3.1-8B) and receive an ever-growing session context that is mostly identical across agents.

PrefillShare exploits this observation by introducing a single frozen prefill worker that serves the shared base model for all agents. Because the prefill is common, KV caches produced during earlier agent calls can be directly reused by subsequent agents through prefix cache hits, dramatically reducing redundant computation.

Key Insight

In multi-LLM agentic workflows, up to 90%+ of prefix tokens overlap across consecutive model calls. PrefillShare captures this by decoupling the frozen prefill module from model-specific decoding parameters.

2 The Multi-LLM Serving Problem

Agentic Patterns Create Redundant Prefill

In frameworks like ReAct and Reflexion, each agent turn appends a new response to the session context and forwards the entire history to the next model. Under standard disaggregated serving, each model has its own prefill worker, causing the same prefix to be recomputed independently for every model invocation.

Three Concrete Challenges

1. Duplicated KV Cache Computation

With N models in the chain, the same prefix of length L is prefilled N times. For the Reflexion pattern with 4 models and 10 turns, this can mean 40 redundant prefill operations for the shared prefix.

2. GPU Memory Waste

Each model's prefill worker independently stores KV caches for the same prefix, consuming N times the necessary GPU memory and limiting the number of concurrent sessions.

3. Cross-Model State Isolation

Standard disaggregated serving treats each model as an isolated endpoint with its own prefill pool. There is no mechanism to share computed KV states across different model endpoints.

3 System Architecture

PrefillShare introduces a shared prefill pool running the frozen base model alongside multiple model-specific decode pools, coordinated by a lightweight proxy.

Routing Proxy

Maintains a routing table mapping each User ID to a specific prefill worker. Ensures prefix cache locality: returning users hit the same prefill worker where their session's KV cache resides.

Shared Prefill Pool

Hosts the frozen base model. All agent transitions route through this single pool. The prefix KV cache from Model A's invocation is directly reused when Model B is called for the same session.

Decode Pools (per model)

Each fine-tuned model has a dedicated decode pool with its task-specific weights (LoRA adapters or full fine-tune). KV caches are transferred from the prefill pool for autoregressive generation.

4 Core Mechanism: Decoupled Prefill

How Fine-Tuning Preserves Prefix Shareability

The key theoretical insight is that fine-tuning (whether full or LoRA) modifies the model's behavior through decoding parameters (output projection, LM head, adapter weights). The base transformer layers that compute KV caches remain unchanged or nearly unchanged. Therefore, a shared prefill module using the frozen base model produces KV caches that are directly usable by all fine-tuned variants during decoding.

// PrefillShare Workflow for Multi-Agent Session
Session session = proxy.getOrCreate(user_id);

// Turn 1: Planner (Model A)
prompt_A = build_prompt(session, "planner");
// Prefill: base model computes KV cache (cold start)
kv_cache = prefill_pool.compute(prompt_A);  // ~2000 tokens
response_A = decode_pool_A.generate(kv_cache);
session.append(response_A);

// Turn 2: Coder (Model B) - same session
prompt_B = build_prompt(session, "coder");
// Prefill: PREFIX CACHE HIT! Only compute new tokens
kv_cache = prefill_pool.compute(prompt_B);  // Only ~200 new tokens!
response_B = decode_pool_B.generate(kv_cache);
session.append(response_B);

// Turn 3: Reviewer (Model C) - even longer prefix hit
prompt_C = build_prompt(session, "reviewer");
// Prefill: 95%+ cache hit from accumulated history
kv_cache = prefill_pool.compute(prompt_C);  // Only ~150 new tokens!
response_C = decode_pool_C.generate(kv_cache);

Progressive Cache Accumulation

As the session progresses through multiple agent turns, the prefix cache monotonically grows. Each subsequent model invocation finds an increasingly longer prefix already cached, approaching near-zero marginal prefill cost.

7 Analysis

vLLM KV Transfer and Staging at High Concurrency

At high concurrency, the aggregate GPU-resident KV footprint on decode workers can temporarily exceed available memory. vLLM handles this by staging KV caches in CPU memory and reloading them on demand. This is an implementation-level behavior for handling transient memory pressure, not an inherent limitation of shared-prefill execution.

Overflow-Induced Staging

Alternative designs could mitigate staging via stricter admission control, decode-to-prefill backpressure, or per-session reservation of GPU-resident KV buffers. These are orthogonal to PrefillShare.

Why Shared Prefill Works for Fine-Tuned Models

The shared prefill succeeds because fine-tuning primarily modifies the model's generation behavior, not its representation of input context. The KV cache computed by the base model during prefill captures the contextual representation, which remains valid when decoded with fine-tuned parameters. This is analogous to how LoRA adapters modify output projections while preserving the attention mechanism.

Impact of Context Growth Rate

The benefit of PrefillShare scales with the ratio of shared prefix to total context. In multi-agent settings, this ratio is naturally high because each agent appends only a short response (~100-500 tokens) relative to the accumulated history (~2000-10000 tokens).

8 Key Takeaways

Architectural Innovation

PrefillShare is the first system to recognize that multi-LLM agent pipelines share a common prefill computation and to exploit this with a unified prefill pool. This architectural insight converts O(N) prefill cost to O(1) for N models sharing a backbone.

Production Viability

Built atop vLLM's standard disaggregated serving with no custom kernel modifications. The proxy, routing table, and prefix cache management are implemented as a lightweight orchestration layer.

Generality

Validated on two different backbone models (LLaMA-3.1-8B and Qwen3-14B) and two representative agentic patterns (ReAct and Reflexion), showing consistent gains across architectures and workload types.

Implications for Agentic Serving

As agentic AI systems increasingly use multi-model orchestration, PrefillShare demonstrates that the serving infrastructure must evolve from isolated model endpoints to shared prefill architectures that exploit cross-model redundancy.

Connection to Broader Survey

PrefillShare addresses a key gap identified in agentic AI serving: while systems like DistServe and Splitwise optimize single-model disaggregation, they do not consider the cross-model KV reuse opportunity that arises in multi-LLM agent pipelines. PrefillShare is complementary to PPD (per-request routing), HCache (state restoration), and Pensieve (multi-tier caching).

9 Proxy-Based Orchestration Design

Routing Table Architecture

The proxy maintains a persistent routing table mapping each User ID to a specific prefill worker. This ensures prefix cache locality: when a returning user issues a new request (for any model in the pipeline), the proxy routes it to the same prefill worker that holds the session's accumulated KV cache.

// Proxy routing logic
class PrefillShareProxy:
    routing_table: Dict[UserID, PrefillWorkerID]
    session_cache: Dict[UserID, SessionState]

    def route_request(self, user_id, model_name, prompt):
        // Step 1: Route to assigned prefill worker
        worker_id = self.routing_table.get(user_id)
        if worker_id is None:
            worker_id = self.assign_least_loaded_worker()
            self.routing_table[user_id] = worker_id

        // Step 2: Prefill on shared base model
        // Worker holds prefix KV cache from prior model calls
        kv_cache = self.prefill_pool[worker_id].compute(prompt)
        // PREFIX HIT: only new tokens computed!

        // Step 3: Transfer KV to model-specific decode worker
        decode_pool = self.get_decode_pool(model_name)
        response = decode_pool.generate(kv_cache)

        // Step 4: Update session and route next model
        self.session_cache[user_id].append(response)
        return response

Session Management Across Agent Transitions

Across agent transitions (Planner to Coder to Reviewer), the proxy applies the same prompt-construction rule -- incorporating each model's generated text into the session context in a consistent format. This ensures that both PrefillShare and the baseline experience identical context growth and token-length profiles.

Inference Workflow Detail

  1. Session initiation: User creates a session, proxy assigns a prefill worker via round-robin or least-loaded policy
  2. Model A request: Proxy routes to assigned prefill worker. Worker computes KV cache using base model (cold start). KV transferred to Model A's decode pool for generation.
  3. Model B request (same session): Proxy routes to the same prefill worker. Worker finds prefix cache hit (84-98% of tokens already cached). Only new tokens are prefilled. KV transferred to Model B's decode pool.
  4. Subsequent models: Each subsequent call achieves progressively higher cache hit rates as the session grows.
Implementation Note

The proxy, routing policy, and prefix-cache management are implemented as a lightweight orchestration layer atop vLLM's disaggregated serving pipeline. All implementation details align with the standard vLLM API, requiring no custom kernel modifications.

Limitations

  • Backbone dependency: All models must share the same base architecture. Cross-architecture sharing (e.g., LLaMA and Mistral) is not supported.
  • KV transfer overhead: Each model-specific decode still requires KV transfer from the prefill pool, which can become a bottleneck at very high concurrency.
  • Fine-tuning divergence: If fine-tuning significantly modifies early transformer layers (rare but possible with long training), KV cache quality may degrade for downstream decoding.
  • vLLM staging behavior: At high concurrency, GPU memory overflow triggers CPU staging of KV caches on decode workers, increasing latency. This is an implementation-level concern, not an inherent limitation of the shared-prefill concept.

Future Directions

Heterogeneous Model Families

Extending shared prefill to models with partial architectural overlap (e.g., different numbers of layers but same attention mechanism).

Adaptive Cache Eviction

Developing prefix cache eviction policies that are aware of multi-model session patterns, prioritizing sessions with more remaining agent transitions.

Disaggregated Decode Optimization

Combining PrefillShare with PPD-style append-prefill routing to further reduce Turn 2+ latency within individual model decode pools.
