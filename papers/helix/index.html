Overview

Modern cloud platforms deploy a heterogeneous mix of GPU types (H100, A100, L4, T4) that vary significantly in compute capacity, memory, and cost. Serving LLMs across such heterogeneous and geo-distributed clusters poses fundamental challenges that existing homogeneous serving systems (vLLM, Orca) cannot address.

Helix is the first system to jointly optimize model placement and request scheduling for heterogeneous GPU clusters. Its key insight is to formulate LLM serving as a max-flow problem on a directed, weighted graph where nodes represent GPU instances and edges capture both GPU and network heterogeneity through their capacities.

Key Opportunity
Eight NVIDIA L4 GPUs can match the FP16 compute of a single H100, while offering greater total memory capacity (192 GB vs. 80 GB), lower power consumption, lower cost, and easier availability. Heterogeneous serving unlocks these cost-effective alternatives.

Challenge 1: Model Placement

How to partition model layers across GPUs with different compute capacities and memory sizes? Uniform partitioning leads to under-utilization of powerful GPUs and bottlenecks on weak ones.

Challenge 2: Request Scheduling

How to route requests through a pipeline of heterogeneous GPUs? Fixed pipelines with round-robin scheduling cannot accommodate varying compute and network capacities.

Challenge 3: Network Awareness

Geo-distributed clusters have vastly different inter-region bandwidths (60-200 Mbps). Model placement and scheduling must be network-aware to avoid congestion.

Why Joint Optimization of Placement + Scheduling Matters

Max-Flow Formulation

Helix transforms a cluster of compute nodes with assigned model layers into a directed graph with edge capacities. The edge capacity denotes the number of tokens compute nodes and network connections can process/transmit per second. The max flow from source to sink equals the maximum serving throughput of the cluster.

Flow Graph Construction for a 3-Node Cluster

Graph Construction Rules

For each compute node c_i, two connected vertices (c_in, c_out) are created. The capacity of the edge (c_in, c_out) represents the max tokens/second the node can process, computed as the minimum of the node's compute and network throughput. Valid network connections must satisfy:

  1. Connection from coordinator to c_i iff c_i holds the first layer of the model
  2. Connection from c_j to coordinator iff c_j holds the last layer
  3. Connection from c_i to c_j iff c_j holds layers immediately needed after inference on c_i
# From simulator/scheduler/global_maxflow/network_flow.py
class FlowGraph:
    def add_compute_node(self, node_uid, inference_throughput,
                          inbound_nic_tp, outbound_nic_tp):
        #      inbound TP          inference TP         outbound TP
        # _in -----------> _start --------------> _end -------------> _out
        self.flow_graph.add_edge(f"{node_uid}_in", f"{node_uid}_start",
                                 capacity=inbound_nic_tp)
        self.flow_graph.add_edge(f"{node_uid}_start", f"{node_uid}_end",
                                 capacity=inference_throughput)
        self.flow_graph.add_edge(f"{node_uid}_end", f"{node_uid}_out",
                                 capacity=outbound_nic_tp)

MILP-Based Model Placement

To find the model placement that maximizes max-flow (and thus serving throughput), Helix uses a Mixed-Integer Linear Program (MILP). The key advantage: the MILP has a linear number of variables and constraints relative to the number of compute nodes, making it tractable for real clusters.

MILP Speedup Techniques

(1) Prune slow network connections to reduce problem size. (2) Hint the MILP solver with heuristic solutions from Petals/Swarm. (3) Use early stopping when the solution is close to the upper bound (sum of compute throughputs / number of layers).

Runtime Scheduling: Per-Request Pipelines

Unlike existing systems that use a fixed set of pipelines, Helix introduces per-request pipelines where each request gets its own independent pipeline through the cluster. This enables fine-grained utilization of heterogeneous resources.

Interleaved Weighted Round-Robin (IWRR)

Each node in the topology graph has an IWRR scheduler bound to it. The IWRR selects the next node proportional to the max-flow allocation on each outbound edge. This follows the optimal flow distribution without creating bursts.

KV-Cache Estimation

The scheduler maintains expected KV-cache usage per node using average output length estimates. Nodes exceeding the high-water mark are masked out during scheduling to prevent OOM.

Dynamic Batching

Each compute node uses best-effort dynamic batching: all requests that arrived during the previous batch's processing are included in the next batch, without additional waiting.

Scheduling Flow

# From simulator/scheduler/global_maxflow/scheduler_core.py
class SchedulerCore:
    def schedule(self, request: InferenceRequest) -> bool:
        if request.phase == RequestPhase.Initialization:
            # Build a per-request pipeline from source to sink
            pipeline = []
            current_node_uid = self.cluster.source_node.node_uid
            while current_node_uid != -1:
                scheduler_node = self.scheduler_nodes[current_node_uid]
                # IWRR scheduling: choose next node proportional to flow
                next_stage = scheduler_node.schedule_initialization(request)
                if next_stage.link_uid == -1:
                    # Reject in offline mode, retry later
                    return False
                pipeline.append(next_stage)
                current_node_uid = next_stage.node_uid
            request.set_pipeline(pipeline)
            # Register in KV-cache expectation tracker
            self.kv_expectation.add_request(...)

        elif request.phase == RequestPhase.Increment:
            # Follow the same pipeline (KV cache is pinned)
            for stage in request.mini_pipeline:
                self.scheduler_nodes[current_uid].schedule_increment(
                    request, stage.link_uid, stage.node_uid
                )

Implementation

Prototype System

Multi-replica pipeline parallel system: 1.5k LoC Python + 1.7k LoC C++. Uses vLLM 0.4.0 as execution engine, ZeroMQ for inter-node communication, and Gurobi as MILP solver.

Discrete-Event Simulator

14k LoC Python high-fidelity simulator supporting heterogeneous GPUs and networks. Simulation errors are less than 5% for all metrics when compared with the prototype system.

Unified Page Pool

Implemented a unified page pool atop vLLM to support partial inference (where a node runs only a subset of model layers), enabling the per-request pipeline execution.

Why Does Helix Win?

Swarm's heuristic model placement often creates bottlenecks by under-utilizing powerful GPUs. Separate Pipelines (SP) achieve balanced compute but suffer from redundant model loading. Helix's MILP finds globally optimal placements that balance compute, memory, and network constraints simultaneously, while per-request pipelines flexibly route requests to avoid congestion.

Relevance to Agentic AI Systems

Helix addresses a critical infrastructure challenge for deploying agentic AI at scale. As organizations rush to deploy autonomous agents, they face GPU scarcity and heterogeneous hardware availability. Helix provides the system foundation to serve LLM-powered agents across whatever GPUs are available.

GPU Scarcity Solution

Agent deployments often cannot secure homogeneous GPU clusters. Helix enables organizations to aggregate whatever GPUs they have (H100s, A100s, L4s, T4s) into a unified serving cluster, maximizing resource utilization for agent inference.

Geo-Distributed Agent Serving

Agents may need to be deployed close to users across multiple regions. Helix's network-aware scheduling handles the bandwidth constraints of cross-region communication, enabling globally distributed agent serving.

Cost Optimization

By using cheaper commodity GPUs (L4, T4) alongside expensive ones (H100), Helix reduces the cost per agent interaction. The MILP-based optimization ensures maximum throughput for any budget constraint.

Dynamic Request Handling

Agent workloads are inherently bursty -- tool calls, multi-step reasoning, and parallel sub-tasks create variable request patterns. Helix's per-request pipelines adapt to these dynamics without fixed pipeline rigidity.

Scaling to Large Agent Deployments

For clusters with hundreds of nodes, Helix recommends partitioning into smaller sub-clusters using heuristics, then applying the MILP independently to each. This hierarchical approach makes Helix practical for enterprise-scale agent serving infrastructure where thousands of concurrent agent sessions must be supported across diverse hardware.
