Cortex

Workflow-Aware Resource Pooling and Scheduling for Agentic Serving

arXiv:2510.14126 October 2025 Columbia University & Google Scheduling Resource Pooling Agentic AI

Overview

Cortex is a prototype workflow-aware serving platform designed for agentic workloads. Its core principle is stage isolation: provisioning dedicated resource pools for each distinct stage of an agentic workflow. This simple yet powerful strategy mitigates inter-stage interference in compute and memory, leading to better KV-cache utilization, higher throughput, and more predictable performance.

Core Insight

A single shared pool of "generic" LLM engines is a poor fit for agentic workflows whose call graphs contain heterogeneous stages. Each stage (SQL generation, execution, error fixing) has distinct latency profiles, memory requirements, and caching opportunities.

Key Innovation

Cortex provisions a dedicated engine pool for every workflow stage. An engine pool is a homogeneous set of workers (GPUs for LLM decoding or CPU executors for SQL) managed by a stage-local scheduler with its own queue, cache, and scaling policy.

Target Workload

Natural-language-to-SQL (NL2SQL) agentic workflows that turn plain-English questions like "What were Europe's sales last quarter?" into successfully executed SQL queries through iterative multi-stage LLM reasoning loops.

The Problem: Workflow-Agnostic Serving

Today's LLM platforms remain workflow-agnostic. Popular frameworks treat every stage as an isolated LLM call and schedule them in FCFS order, or use basic prioritization without awareness of the internal workflow structure.

Current Approach (Shared Pool)

  • Treat each LLM call as isolated, blind to workflow stages
  • Miss caching opportunities (e.g., 5 identical schema prompts = 5 cold misses)
  • Cannot accelerate stragglers (e.g., slow SQL back-end)
  • Over-provision expensive GPU instances or suffer SLO violations
  • Batching couples runtimes of different stages, delaying token emission

Cortex Approach (Stage-Isolated Pools)

  • Dedicated pool per workflow stage with custom configuration
  • Stage-local caches keep only relevant context hot
  • Orchestrator tracks per-request SLO slack across the DAG
  • Independent scaling per-stage based on actual bottleneck
  • Stable, stage-local latency models enable predictable SLOs
Performance Impact of Shared Serving

When heterogeneous calls (SQL generation and error fixing) share an engine, batching couples their runtimes, delays token emission, and makes an LLM call's latency depend on its batch-mates -- undermining remaining-time/SLO-aware prioritization and wrecking predictability.

System Architecture

Client requests enter an orchestrator that consumes a compiled call graph, computes per-request SLO slack, and dispatches work to stage pools that are managed by an engine-allocation layer.

Benefits of Stage Isolation

No Cross-Stage Interference

When heterogeneous calls share an engine, batching couples their runtimes and delays token emission. Isolating stages restores stable, stage-local latency models.

Independent Scaling

A fast monitor scales out or in only the pool that threatens the SLO. Run-once stages like SQL generation can be lightly provisioned while critical-path pools for the error fixer get more resources.

Reduced KV Cache Footprint

Each engine holds only its stage-specific context. A shared engine must keep both stages' contexts hot on each replica, effectively duplicating KV-cache memory usage.

Per-Stage SLO Control

Per-stage variance is far smaller than end-to-end variance (dominated by variable retry depth). This per-stage control can meet the same SLOs with much higher efficiency.

Workflow-Aware Orchestrator

The orchestrator is the brain of Cortex. It is workflow-aware: it tracks each request's position in the call graph, predicts the next set of eligible operators, and attaches a priority key derived from SLO slack, stage selectivity, and expected service time.

Engine Allocation Layer

The engine-allocation layer routes each sub-call to a concrete pool instance that maximizes locality:

  • KV-cache/prompt affinity for generation stages
  • Warm database connections for SQL execution
  • Load balancing across replicas within each pool
  • Priority-based reordering based on SLO slack
  • Admission control when a stage becomes the bottleneck
SLO Slack Computation

The orchestrator computes per-request SLO slack as the difference between the deadline and the estimated remaining execution time across all downstream stages. Requests with low slack receive higher priority, and the system can opportunistically let compatible stages borrow idling engines to reduce fragmentation and raise utilization.

KV Cache Utilization Analysis

Total KV footprint of the NL2SQL workflow's LLM stages is significantly lower when each stage runs in its own Cortex pool. Each engine holds only its stage-specific context.
Why Co-location Wastes Memory

A shared engine must keep both stages' contexts hot on each replica, effectively duplicating KV-cache memory usage. The reclaimed GPU memory raises effective batch size and/or beam width, translating directly into higher throughput and tighter tail latency.

Toward Agent-Native Serving

Stage isolation is only a starting point. The Cortex paper outlines broader still-unrealized mechanisms needed for native agentic serving, laying the groundwork for a new generation of workflow-aware platforms.

Malleable Workflows

Agentic workflows are malleable in both what they compute and how they run. The computation can shrink (swap a heavyweight model for a lighter variant, prune retries) or stretch (boost straggling tails with beefier engines, increase SQL concurrency). A central open challenge is designing an interface that lets workflows declare their malleability.

Speculative Execution

LLM agents are inherently fertile ground for speculation. Cortex can speculate on the most probable branches in a workflow and pre-warm associated engines or pre-execute next steps. In NL2SQL, the platform can generate many candidate queries and evaluate them in parallel instead of serially.

Agentic State

Cortex envisions treating intermediate data -- from KV-cache entries to fetched tool results -- as a first-class, multi-tier "agentic state." A workflow-wide shared tier could act as a publish/subscribe fabric: agents advertise artifacts (SQL results, vector-search hits) and downstream calls check this tier before recomputing.

Malleability Knobs

Knob Type Example Shrink Action Stretch Action
Model size GPT-4 vs GPT-3.5 Switch to lighter model when SLO is tight Use larger model when slack allows
Parallelism Number of candidate SQL queries Reduce fan-out to conserve resources Increase parallel explorations
Retry depth Max error-fix iterations Cap retries under pressure Allow deeper refinement with slack
Reasoning length Chain-of-thought tokens Shorten reasoning to reduce latency Full reasoning for higher accuracy
Key Architectural Implication

The two malleability strategies (shrink and stretch) interplay: if a small-model engine is saturated yet a large-model engine sits idle waiting for refinement, the system can opportunistically let the larger model generate candidates itself, provided it can access or reconstruct the necessary context.

Conceptual API

While Cortex is a prototype, its design suggests a declarative workflow definition API where stage isolation and SLO tracking are first-class constructs:

# Define an NL2SQL workflow with Cortex stage isolation
from cortex import Workflow, StagePool, SLOPolicy

# Each pool has its own scheduler, cache, and scaling config
sql_gen_pool = StagePool(
    name="sql_generator",
    engine_type="gpu",
    model="codellama-34b",
    cache_policy="prefix_sharing",   # schema prefix reuse
    max_replicas=4,
    scaling="auto"
)

sql_exec_pool = StagePool(
    name="sql_executor",
    engine_type="cpu",
    db_connections=16,          # warm connection pool
    admission_control=True
)

err_fix_pool = StagePool(
    name="error_fixer",
    engine_type="gpu",
    model="codellama-34b",
    cache_policy="error_template",  # error pattern caching
    priority="slo_slack"          # critical path priority
)

# Workflow definition with SLO
workflow = Workflow(
    name="nl2sql",
    stages=[sql_gen_pool, sql_exec_pool, err_fix_pool],
    slo=SLOPolicy(latency_p99=5.0, retry_budget=3)
)

Key Takeaways

1. Stage Isolation Works

Simply separating workflow stages into dedicated pools yields measurable improvements in KV-cache utilization (~50% reduction), latency predictability, and scaling efficiency.

2. Workflows Need Awareness

Treating each LLM call as an isolated request leaves enormous optimization potential on the table: caching, priority, and scaling all benefit from workflow-level knowledge.

3. Foundation for Advanced Serving

Stage isolation is a stepping stone to malleable workflows, speculative execution, and shared agentic state -- the building blocks of truly agent-native serving platforms.
