Overview

SideQuest is a novel approach that shifts KV cache management from fixed heuristics to an intelligent, model-driven process. Instead of relying on static attention scores (H2O, SnapKV) or fixed token budgets, SideQuest empowers the Large Reasoning Model (LRM) itself to decide which KV cache entries to evict by analyzing the semantic relevance of past tool outputs to the current task state.

Key Insight: Token Utility is Non-Monotonic

In multi-step agentic reasoning, a token's importance is highly dynamic and non-monotonic. A search result that appears irrelevant at step t (low attention score) may become the critical pivot at step t+n when the agent needs to cite sources in its final answer. Static heuristics based on past attention scores fundamentally cannot capture this behavior. SideQuest uses the model's own reasoning capability to make semantically-aware eviction decisions.

Three Contributions

1. Heuristic Failure Analysis

Demonstrates that static importance metrics (attention scores, frequency counts) fail to capture the dynamic, non-monotonic utility of tokens in multi-step agentic reasoning. Heuristic pruning causes high rates of unparsable responses and model collapse.

2. Model-Driven Memory Management

Introduces SideQuest, a framework that uses the LRM itself to perform KV cache compression via an auxiliary "side quest" thread running in parallel with the main reasoning task. This achieves self-referential garbage collection.

3. Scalable Data Synthesis Pipeline

Develops a hindsight-based annotation pipeline that generates high-quality training data from only 215 correct traces. The model is fine-tuned with LoRA using a joint distillation + cross-entropy objective in just 3 epochs.

Fixed Token Budgets: A Fundamental Mismatch

Task token counts range from a few thousand to over 120k. A 16k token budget fails on complex queries, while a 24k budget wastes memory on simple ones. SideQuest adaptively adjusts context size based on the problem's instantaneous difficulty, discovering the optimal token budget per query without a priori tuning.

Four-Step Workflow

1. Parallel Execution

At regular intervals (every K reasoning turns), SideQuest forks the generation process. The auxiliary thread receives a snapshot of the shared context with the trigger phrase ** Memory management mode ** appended.

2. Staleness Reasoning

The model produces a dedicated reasoning trace analyzing which tool outputs (cursors) are redundant. It reflects on the open cursors and determines which are no longer needed for the current task state.

3. Eviction

The auxiliary thread outputs a structured command like {del_cursors: [0]}, flagging specific KV cache entries for removal. Only tool-response cursors are targeted.

4. Synchronization

The system waits for the main thread's current turn to complete before executing the eviction. All auxiliary thread tokens are then discarded, introducing zero permanent token overhead to the main context.

Overhead Characteristics

Storage: Zero additional KV cache -- the auxiliary thread operates on the shared context. Compute: Marginal cost, as long-context agentic workloads are memory-bandwidth bound, not compute bound. Shared-prefix attention kernels (Cascade Inference, FastTree) eliminate redundant memory loads for the shared context portion.

Heuristics see Cursor 1's low attention scores at step 3 and would evict it. But the LRM understands the semantic relationship: Cursor 1 contains source URLs that will be needed for citation in the final answer. Algorithm Details

Algorithm 1: SideQuest Operation

def sidequest(query, model, trigger_interval=4, trigger_phrase="** Memory management mode **"):
    """Main SideQuest execution loop integrated with ReAct."""
    context = [query]
    turn = 0
    aux_thread = None

    while True:
        # Phase 1: Check if auxiliary thread has results
        if aux_thread is not None and aux_thread.is_finished():
            evict_ids = aux_thread.get_output()
            context.clear_kv(evict_ids)  # Prune stale tool outputs
            aux_thread = None

        # Phase 2: Spawn auxiliary thread at intervals
        if turn % trigger_interval == 0 and aux_thread is None:
            aux_context = context + trigger_phrase
            aux_thread = async_generate(model, aux_context)

        # Phase 3: Main ReAct execution
        response = model(context)
        context.append(response)

        if response.contains_final_answer():
            return response

        if response.contains_tool_call(tool):
            observation = execute_tool(tool)
            context.append(observation)
        else:
            break  # End of turn if no tool called

        turn += 1

Algorithm 2: Training Data Generation

def generate_training_data(dataset, base_policy, annotator, interval=4):
    """Hindsight-based data synthesis pipeline."""
    train_data = []

    for task in dataset:
        trace = run_inference(base_policy, task)
        if not is_correct(trace):
            continue

        # Annotate each cursor with its last-use turn index
        for cursor in trace.cursors:
            cursor.last_use = last_use_index(trace, cursor)

        # Add main trace with logit distillation targets
        logits = base_policy.forward(trace)
        train_data.append((trace, "main", logits))

        # Generate auxiliary traces at each interval
        for t in range(0, len(trace), interval):
            expired = [c for c in trace.cursors if c.last_use < t]
            open_set, closed_set = random_partition(expired)
            reasoning = annotator.generate(trace[:t], open_set)
            train_data.append((reasoning, "aux"))

    return train_data

Limitations & Future Directions

Data Scale Limitation

The current model was fine-tuned on only 215 correct traces from FRAMES. Performance on out-of-distribution BrowseComp shows a 5% accuracy degradation, suggesting that scaling the training data would close this gap.

Tool-Response Scope Only

Current eviction targets only tool-response cursors. Extending to compress the agent's own intermediate reasoning steps (thought pruning) is a promising future direction that could yield additional memory savings.

Beyond Memory Management

The SideQuest architecture of parallel auxiliary threads on shared context generalizes to safety checks, prompt-injection detection, and governance tasks -- all running as "side quests" without polluting the main reasoning context.

Complementarity with Hierarchical Methods

SideQuest can be deployed inside sub-tasks of Recursive Language Models or Context Fold systems. It acts as a garbage collector for the active linear stream within each branch, ensuring local context remains lean.
