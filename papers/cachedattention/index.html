CachedAttention: Cost-Efficient LLM Serving for Multi-turn Conversations

A hierarchical KV cache persistence system that reuses cached attention states across multi-turn conversations, eliminating redundant recomputation through DRAM and SSD storage with overlapped access, position-encoding decoupling, and scheduler-aware placement.

Overview

CachedAttention is a new attention mechanism that enables the reuse of KV caches across multi-turn conversations, significantly reducing the repetitive computation overhead that plagues current LLM serving engines. When a conversation session becomes inactive, CachedAttention saves the KV cache to a hierarchical storage system (AttentionStore) spanning DRAM and SSDs. Upon resumption, it loads and reuses the saved KV cache, avoiding redundant recomputation of historical tokens.

Core Insight

In multi-turn conversations, up to 99% of the prefilling cost comes from redundant recomputation of historical tokens. CachedAttention eliminates this by persisting KV caches to external storage and reusing them when conversations resume, requiring only the prefilling of new input tokens per turn.

The Multi-Turn Conversation Problem

Prevalence of Multi-Turn Conversations

Analysis of 90K real ChatGPT conversations (ShareGPT dataset) reveals that 73% of conversations are multi-turn, and 30% of sessions exceed 4K tokens. Each new turn requires recomputing the KV cache of all previous turns, creating a linearly growing recomputation burden.

Four Key Challenges

1. High KV Cache Access Overhead

Loading KV cache from host memory to HBM takes ~192ms for 2K tokens (5 GB on LLaMA-65B). GPUs are blocked during transfer, wasting computation cycles.

2. Massive Storage Requirements

KV cache occupies 2.5 MB/token for LLaMA-65B. HBM fills in 14 seconds with 4 A100 GPUs. Host memory overflows in <1 minute. Disk storage is essential.

3. Placement Across Tiers

Disks have ~5 GB/s bandwidth vs. tens of GB/s for DRAM. Random conversation arrivals mean most KV caches end up on disk. Intelligent placement is critical.

4. Context Window Overflow

When context exceeds model limits (e.g., 4K for LLaMA-2), truncation changes token positions, invalidating embedded positional encodings in saved KV caches.

Overlapped KV Cache Access

CachedAttention uses two key techniques to overlap KV cache transfer with computation, hiding the access latency.

Layer-wise Pre-loading (DRAM to HBM)

Instead of loading the entire KV cache before computation begins, CachedAttention loads each layer's KV cache just before that layer needs it, overlapping loading with computation of other layers.

Buffer Size Formula

When Tload * Lhist > Tpref * Lnew (loading time exceeds partial prefill time), a buffer of size Sbuf = B(Tload * Lhist - Tpref * Lnew) fills the time gap, where B is the PCIe bandwidth. With a 15-layer buffer (PF-B15), prefilling time is reduced by 61% compared to no pre-loading.

Why Not LRU or FIFO?

Traditional Policies Fail for KV Caches

LRU and FIFO rely only on historical access patterns. CachedAttention's scheduler-aware eviction leverages future access information from the job queue. With 128G DRAM / 10T SSD: CachedAttention achieves 86% hit rate vs. LRU's 58% and FIFO's 48%. The key advantage: CachedAttention knows which KV caches will be needed soon, pre-fetching them from disk, while evicting those farthest from the queue head.

Position-Encoding Decoupled KV Cache Truncation

When historical tokens exceed the context window limit, LLM engines truncate the oldest tokens. This changes every remaining token's position, invalidating stored KV caches. CachedAttention solves this through positional encoding decoupling.

How It Works

# Saving KV cache to AttentionStore:
def save_kv_cache(kv_cache, position_ids):
    # Remove positional encoding before storing
    kv_no_pos = decouple_position(kv_cache, position_ids)
    attention_store.save(session_id, kv_no_pos)

# Loading KV cache from AttentionStore:
def load_kv_cache(session_id, new_position_ids):
    kv_no_pos = attention_store.load(session_id)

    # If context overflow, truncate directly on KV cache
    if len(kv_no_pos) + new_tokens > context_window:
        kv_no_pos = kv_no_pos[truncate_start:]  # Direct truncation!

    # Re-embed new positional encoding
    kv_cache = embed_position(kv_no_pos, new_position_ids)
    return kv_cache

Cache Capacity Requirements

Storage Ratio Insight

When the storage capacity / DSpUT ratio reaches 0.1, hit rate reaches 51%. At ratio 0.25, hit rate reaches 98% and throughput peaks. This means CachedAttention does not require unbounded storage -- a modest ratio of cache capacity to distinct sessions per unit time achieves near-perfect hit rates.

Context Overflow Impact

With positional encoding embedded in KV caches (baseline "OF"), context overflow causes hit rate drops of 17.6-18.4% across models. CachedAttention's decoupled encoding maintains full hit rates despite overflow, as truncation is applied directly to the saved KV cache without invalidation.

Robust to Arrival Rate Changes

Even as session arrival rate increases 4x (from 0.5 to 2.0), the KV cache hit rate drops only from 82% to 77%, and TTFT increases modestly from 0.122s to 0.154s. CachedAttention maintains substantial performance improvements across varying load conditions.

Key Takeaways

Hierarchical Persistence

KV caches are persisted across HBM, DRAM, and SSDs with scheduler-aware placement. This three-tier hierarchy provides both the capacity (TB-scale on SSDs) and performance (DRAM for upcoming jobs) needed for real-world multi-turn serving.

Overlapped Access

Layer-wise pre-loading and async saving hide KV cache transfer time behind computation. With a 15-layer buffer, prefilling time is reduced by 61%. Async saving reduces execution time by 13-15%.

Encoding Decoupling

By stripping positional encoding before storage and re-embedding on load, CachedAttention supports direct KV cache truncation. This maintains quality (PPL within 0.02 of full recomputation) while avoiding invalidation on context overflow.
