Not All Prefills Are Equal: PPD Disaggregation for Multi-turn LLM Serving

Dynamic routing of append-prefill operations to decode nodes with cached KV states, reducing Turn 2+ TTFT by up to 68% while maintaining competitive TPOT

arXiv 2603.13358 PD Disaggregation

Zongze Li, Jingyu Liu, Zach Xu, Yineng Zhang, Tahseen Rabbani, Ce Zhang

University of Chicago · Together AI

1 Overview

Figure 1 (from paper): P99 TTFT vs. Tokens-per-Second (TPS) Pareto frontiers under a long-context workload (10k input, 100 output tokens, 5 turns) at three load levels. D-local capable configurations (blue) achieve lower TTFT while maintaining competitive TPS. The 'Best' annotation marks the configuration selected by PPD's dynamic routing system.

Standard Prefill-Decode (PD) disaggregation sends every prefill to dedicated P nodes and transfers KV caches to D nodes. While effective for single-turn requests, this creates critical inefficiencies for multi-turn conversations: every follow-up turn recomputes KV for the entire history and saturates P-to-D bandwidth.

PPD distinguishes between full prefill (Turn 1, no cached context) and append prefill (Turn 2+, extending cached KV). The key finding: append-prefill causes an order-of-magnitude less interference with decode than full prefill, making it safe to run locally on D nodes.

2 Key Insight: Full vs. Append Prefill

The Core Discovery

Not all prefill operations are equally disruptive to decoding. Full prefill (processing a new prompt from scratch) causes ~48% decode slowdown at batch size 200, while append prefill (processing only new tokens while reusing cached KV) causes only ~2% slowdown.

Figure 2 (from paper): Prefill-decode interference. Decode TPOT degradation when co-locating with one full-prefill vs. one append-prefill operation (both processing 1,024 tokens). Full prefill causes significant slowdown; append prefill remains close to baseline.

Why the Gap Exists

Full prefill has O(n^2) attention complexity for n input tokens, consuming massive compute and memory. Append prefill computes attention only for the m new tokens (attending over n+m keys), with O(m(n+m)) complexity where m is much less than n in typical follow-up turns.

5 PPD Dynamic Routing Algorithm

PPD operates in two phases: an offline benchmarking phase and an online per-request decision phase.

// Phase 1: Offline Table Construction
for each discretized workload psi_i in benchmark_grid:
    // Measure T2 TTFT and TPOT under x=0 (route to P) and x=1 (route to D)
    delta_ttft = (TTFT_x0 - TTFT_x1) / TTFT_x0
    delta_tpot = (TPOT_x1 - TPOT_x0) / TPOT_x0
    score = w_ttft * delta_ttft - w_tpot * delta_tpot
    x_star(psi_i) = 1 if score > 0 else 0

// Phase 2: Online Per-Request Decision
def route(request):
    if request.turn == 1:
        return x = 0  // Always do KV transfer for Turn 1
    psi_hat = discretize(request.n_in, request.n_out, request.n_ctx, qps)
    return x_star(psi_hat)  // Lookup: <1ms overhead

Configurable TTFT-TPOT Trade-off

Operators tune the balance via weights w_ttft and w_tpot. Higher w_tpot penalizes TPOT degradation more, routing more requests via traditional PD path.

8 Key Takeaways

Architectural Insight

The order-of-magnitude interference gap between full and append prefill opens a new design dimension: decode nodes can safely handle Turn 2+ requests locally.

No Universal Best

Static routing cannot universally optimize TTFT, TPOT, and throughput. PPD's dynamic per-request routing achieves Pareto-optimal trade-offs with <1ms decision overhead.

Complementary to KV Cache Systems

PPD addresses how requests are scheduled (scheduling isolation), while distributed KV cache systems address where KV states are stored (caching). The two are fully compatible.

Production Ready

Built on vLLM's disaggregated serving, using ZeroMQ for service discovery and routing proxy. Decision via precomputed lookup tables with negligible overhead.

9 Implementation Details

System Architecture on vLLM

PPD is implemented as a routing layer atop vLLM's disaggregated serving infrastructure:

// KV Transfer Protocol
P servers: kv_role = kv_producer
    // Generate KV caches and send via ZeroMQ
D servers: kv_role = kv_consumer
    // Receive KV caches, store in local prefix cache
    // For Turn 2+ processing when x>0

// Session Management
session_table[conv_hash] = {
    "turn_count": int,         // Current conversation turn
    "assigned_pd": str,        // P/D node assignment
    "last_access": timestamp   // For TTL-based eviction (60 min)
}
// Conversation IDs: MD5 hash of first user message

Service Discovery

Backend servers register with the proxy via ZeroMQ heartbeats every 10 seconds. The proxy maintains instance lists for each server type (P, D, R) and removes servers after 30 seconds without heartbeat, enabling dynamic scaling and automatic failure recovery.

Routing Proxy Implementation

Implemented using Quart (async Flask). Handles:

  • Request parsing: Extract conversation context and determine turn number
  • Session lookup/creation: Manage conversation-to-server mappings
  • Request forwarding: Stream responses from backend to client
  • Statistics collection: Track routing decisions and latency metrics

11 Relationship with Other Systems

vs. AMPD (Concurrent Work)

AMPD also identifies multi-turn PD inefficiencies and proposes routing to decode nodes. AMPD uses online queue-delay estimation and offline hardware planning. PPD takes an analysis-driven approach, rigorously quantifying the micro-architectural interference gap to demonstrate why routing is viable, and formalizes multi-turn serving as a mathematical optimization over routing ratio x.

vs. Distributed KV Cache (Mooncake)

Mooncake provides shared KV storage via RDMA, addressing where prefix states are stored. PPD addresses how requests are scheduled based on cache characteristics. The two are fully compatible and complementary: distributed cache maximizes prefix reuse, PPD ensures cache-hitting requests aren't interrupted by cache-missing ones.

vs. DuetServe (Intra-GPU)

DuetServe achieves PD isolation within a single GPU via SM partitioning. PPD operates at the cluster level, routing between dedicated P and D machines. They address complementary granularities.

vs. PrefillShare (Multi-LLM)

PrefillShare shares prefill across multiple models; PPD shares D-node compute for multi-turn continuation within a single model. Both exploit the insight that not all prefill work needs dedicated P nodes.

Scaling Validation

The x=1 advantage generalizes across:

  • Turn scaling (2-16 turns): The TTFT gap between x=0 and x=1 widens with more turns, as accumulated context amplifies KV transfer overhead
  • Model scaling (8B-30B): The relative improvement remains stable at ~70%, indicating the benefit stems from architectural properties rather than model-specific characteristics

12 Mathematical Framework

Formal Problem Definition

PPD formalizes multi-turn serving performance as an optimization over the routing parameter x (fraction of append-prefill operations routed to decode nodes):

// Performance of disaggregated serving for multi-turn conversations
J(x; pi, phi) := E_{psi, SLO ~ phi}[ SLO( P_pi(1-x), D_pi(x), psi ) ]

// Where:
//   x in [0, 1] -- fraction of AP operations routed to D
//   pi -- node assignment (how many P vs D nodes)
//   psi -- workload (QPS, input/output lengths, turns)
//   phi -- SLO distribution (TTFT, TPOT targets)

// Standard PD is a special case:
x = 0  // All AP goes to P (traditional PD)
// PPD optimizes x per-request:
x* = argmax_x J(x; pi, phi)

Attention Complexity Analysis

The interference gap is rooted in attention complexity differences:

OperationComplexityContext
Full Prefill (Turn 1)O(n^2) attention for n tokensNo cached context, all tokens new
Append Prefill (Turn 2+)O(m * (n + m)) where m << nn cached tokens, m new tokens
Typical ratio (m/n)m ~ 5-10% of nShort new prompt, long history

For a typical follow-up turn with m = 500 new tokens and n = 10,000 cached tokens, append prefill attention is ~20x cheaper than full prefill attention. This explains why append prefill causes negligible decode interference.

KV Transfer Bandwidth Analysis

For Llama-3-8B with a 2K-token context, each KV transfer is approximately 512 MB. At 3.1 average turns per conversation under ShareGPT, the baseline PD system generates 3.1 KV transfers per conversation. PPD with x=1 generates only 1 transfer (Turn 1), reducing KV transfer volume by ~75%. This difference explains why baseline PD configurations experience service degradation (KV transfer saturation) at high QPS while PPD remains stable.

14 Core Mechanisms

PPD's design rests on three tightly coupled mechanisms: (1) conversation-turn classification, (2) offline decision-table construction, and (3) an online routing proxy that steers each request in under 1 ms. Together they implement the routing objective max J(x; π, φ) without modifying vLLM's inference engine.

Mechanism 1 — Turn Classification

Every incoming request is first classified as either a Turn-1 (full-prefill) or Turn-2+ (append-prefill) request. Classification is based on the presence of a matching conversation-ID in the session table, which maps an MD5 hash of the initial user message to the assigned P/D pair. If no session entry exists, the request is treated as Turn 1 and unconditionally routed through the standard PD path — KV is computed on a P node and transferred to a D node. If a session entry is found, the request is an append-prefill candidate and proceeds to the decision lookup.

Mechanism 2 — Offline Decision-Table Construction

Before deployment, a benchmarking script sweeps the discretized workload space and records TTFT and TPOT for x=0 and x=1 at each operating point. The result is a lookup table keyed by (n_in_bin, n_out_bin, n_ctx_bin, qps_bin) that maps each cell to the Pareto-optimal routing choice for the operator's TTFT/TPOT weight vector.

Discretization Strategy

Token counts and QPS values are bucketed into logarithmically-spaced bins. This keeps the table compact (a few thousand cells) while preserving enough resolution to capture workload-sensitive routing decisions. Interpolation between bins is not used — the nearest bin is selected to keep online latency under 1 ms.

Weight Vector Semantics

The operator-supplied weights (w_ttft, w_tpot) are normalized to sum to 1. When w_ttft = 1 the table aggressively favors D-local routing for every cell; when w_tpot = 1 it reverts to pure PD. Intermediate values produce a mixed table that exploits the low-load regime where append-prefill interference is genuinely negligible.

Mechanism 3 — Online Routing Proxy

The routing proxy is an async HTTP gateway (Quart) sitting in front of the vLLM cluster. It performs five steps per request: parse → classify → lookup → forward → record. The critical path — steps 2 through 4 — adds under 1 ms to end-to-end latency because the decision table is held in-process memory and accessed with a single dictionary lookup.

KV Cache Locality and Session Affinity

Session affinity is a prerequisite for D-local append-prefill to work correctly. The session table pins each conversation to a specific (P, D) pair from Turn 1 onward. This guarantees that when a Turn-2+ request arrives and is routed to the D node (x=1), the full KV history of the conversation is already resident in that D node's prefix cache — no remote fetch is needed. Sessions expire after 60 minutes of inactivity, freeing D-node memory for new conversations.

Throughput and Stability

Key Reliability Result

PPD (x=1) achieves 100% request success rate at all evaluated QPS levels on both ShareGPT and WildChat. The baseline (x=0) degrades at 9 out of 12 QPS levels for 2P_2D and at 7 out of 9 for 3P_1D — failures caused entirely by NVLink KV-transfer saturation.

Throughput (requests completed per second before SLO violation) follows a similar pattern: x=1 extends the stable operating range to 38.3% more workload–QPS cells than any other single configuration, giving it the highest average win rate across the full configuration space.

Scaling Behaviour

The per-turn improvement is monotonically increasing because each additional conversation turn adds to the accumulated KV context, which in turn increases the cost of the KV transfer that x=1 avoids. By Turn 16 the improvement stabilizes near 70%, consistent across 8B and 30B models — confirming the gain is architectural rather than model-specific.

Network Bandwidth Impact

The most operationally significant result from the evaluation is the elimination of KV-transfer-induced service outages. In a 4-node cluster with NVLink bandwidth shared among all active KV transfers, the baseline (x=0) saturates the P-to-D channel at moderate-to-high QPS. Routing Turn 2+ requests locally (x=1) reduces total KV transfer volume by a factor proportional to the mean conversation length, which is 3.1 on ShareGPT. The resulting bandwidth headroom allows PPD to absorb higher QPS bursts without degradation, as confirmed by the zero failure rate across all tested QPS levels.

16 Limitations & Future Work

PPD represents a significant step toward multi-turn-aware LLM serving, but several constraints bound the current design. Understanding them is important both for practitioners deploying PPD today and for researchers extending it.

Binary Routing (x ∈ {0, 1})

Although the framework defines x as a continuous fraction, the production implementation uses a binary lookup (route to P or route to D). Fractional routing — splitting a workload stream with probability x — was explored in the configuration study but found to add coordination overhead without consistent benefit over x=1 in the evaluated settings. Future work could explore per-batch fractional routing where the proxy controls the mixing ratio adaptively at runtime.

D-Node Memory Pressure

Routing append-prefill to D nodes increases the KV footprint on those nodes. Each active conversation occupies KV memory proportional to its accumulated context length. At high conversation counts or very long contexts (e.g., 64K tokens), D nodes may exhaust their HBM capacity before P nodes. The 60-minute TTL partially mitigates this, but a memory-aware routing variant that monitors D-node HBM utilization and falls back to x=0 when pressure exceeds a threshold remains an open design point.

Offline Table vs. Online Load

The decision table is constructed under controlled benchmark conditions and may not perfectly reflect production traffic distributions. Sudden traffic spikes or workload shifts (e.g., a change in average output length due to a model update) can make the precomputed table suboptimal. An online adaptation mechanism that periodically re-evaluates a sample of routing decisions using shadow traffic and updates affected table cells could close this gap without the latency cost of full re-benchmarking.

Single-Proxy Single Point of Failure

The current implementation uses a single routing proxy process. In production environments requiring five-nines availability this is a critical gap: proxy failure halts all request routing. A replicated proxy with a distributed session store (e.g., Redis) and leader-election would address this. The stateless decision-table lookup makes horizontal scaling straightforward — the session state is the only shared mutable component.

Scope: Multi-GPU, Not Multi-Datacenter

All experiments run on a 4-GPU node with NVLink interconnect (bandwidth ~600 GB/s). Real deployments often span multiple nodes connected by InfiniBand or Ethernet where KV transfer latency is far higher. The relative advantage of x=1 should be even larger in such environments because KV transfer cost is amplified, but the specific thresholds for D-node memory pressure and TPOT degradation may differ and would require re-profiling.

Two-Turn Measurement Scope

The configuration-space study measures TTFT and TPOT for Turn 1 and Turn 2 only. Turn-3+ behavior is inferred from the interference analysis and the scaling study but not exhaustively measured across all 17 node configurations and 18 workloads. A full multi-turn evaluation at scale would strengthen the empirical foundation, particularly for long-tail conversations with 10+ turns.

Future Work Directions

Adaptive Table Updates

Integrate a lightweight online learner that continuously refines the routing table from live traffic observations, enabling the system to self-tune as workload distributions shift over time without requiring a full offline benchmark re-run.

Memory-Aware Routing

Extend the routing objective to include D-node HBM utilization as a constraint. When a D node's KV occupancy exceeds a configurable threshold, the proxy automatically increases the fraction of append-prefill routed to P nodes, trading some TTFT for memory safety.

Integration with Speculative Decoding

Speculative decoding introduces draft-model prefill during each decode step, a third type of prefill distinct from both full and append. Characterizing its interference profile and extending the routing framework to include a speculative-prefill path could yield further TPOT improvements on D nodes that run speculative models.

Multi-Node KV Disaggregation

Combining PPD's scheduling insight with a distributed KV cache layer (e.g., Mooncake over RDMA) could allow the session-affinity constraint to be relaxed: append-prefill could be routed to any D node that can fetch the required KV prefix from a shared store within the time budget, increasing load-balancing flexibility.

Bottom Line

PPD's core insight — that append-prefill interference is an order of magnitude smaller than full-prefill interference — is robust across GPU generations and model scales. The specific implementation choices (binary routing, single proxy, 60-minute TTL) are engineering decisions that can be relaxed as the system matures. The mathematical framework and the interference characterization provide a solid foundation for the future extensions described above.
