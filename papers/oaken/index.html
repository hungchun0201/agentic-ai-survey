Oaken: Fast and Efficient LLM Serving with Online-Offline Hybrid KV Cache Quantization

A hardware-algorithm co-designed acceleration solution that achieves both high bandwidth and capacity for batched LLM inference through hybrid KV cache quantization

Authors: Minsu Kim, Seongmin Hong, RyeoWook Ko, Soongyu Choi, Hunjong Lee, Junsoo Kim, Joo-Young Kim, Jongse Park
Affiliations: KAIST & HyperAccel Venue: ISCA 2025, Tokyo

Overview

Modern LLM serving systems batch multiple requests to achieve high throughput, but attention operations are un-batchable, making memory bandwidth the critical bottleneck. Existing solutions trade off bandwidth for capacity (HBM vs LPDDR) or rely on quantization with high runtime overhead. Oaken jointly co-designs an algorithmic technique (online-offline hybrid KV cache quantization) with custom hardware modules to achieve unprecedented levels of effective bandwidth and capacity simultaneously.

Core Insight

Existing hybrid quantization methods (e.g., KIVI, KVQuant) achieve high accuracy by using different precisions for outliers vs inliers, but the cost of online outlier detection (sorting, topK) is prohibitively high, negating the performance gains. Oaken solves this by determining thresholds offline and applying them online with near-zero overhead.

Motivation: The Bandwidth-Capacity Dilemma

Why KV Cache is the Bottleneck

In batched LLM inference, the generation phase produces one token per request per iteration. The key-value cache for attention is per-request and cannot be shared across batches, forcing enormous memory reads that are bounded by memory bandwidth. As batch sizes grow, the KV cache dominates memory usage -- exceeding model weights at batch size 16 for Llama2-13B.

Figure: Oaken occupies the sweet spot in the bandwidth-capacity trade-off by combining KV cache quantization with custom hardware modules on LPDDR memory.

Three Key Observations on KV Distribution

◆

Observation 1: Layer-Specific Ranges

The min-max range of KV cache values varies significantly across models and decoder layers, driven by differences in model weights. Quantization factors should be determined per model and per layer.

◆

Observation 2: Dataset Insensitivity

KV cache value ranges remain consistent across different datasets (Wikitext2, PIQA, Hellaswag). A global quantization factor per layer is sufficient regardless of input prompts.

◆

Observation 3: Channel-Concentrated Outliers

High-magnitude values concentrate in specific channels. However, exceptions exist (discontinuous patterns), motivating multiple quantization groups segmented by value magnitude within each vector.

Component 1: Threshold-Based Online-Offline Hybrid Quantization

Oaken separates each per-token KV vector into three groups: outer (large-magnitude outliers), middle (inliers, the majority), and inner (small-magnitude outliers). Thresholds are determined offline via profiling and applied online during inference.

Figure: Three-group separation of KV values using offline-determined thresholds. Middle group (4-bit), Inner/Outer groups (5-bit with group-shift).
Offline Profiling, Online Application

Oaken runs ~100 sample inferences offline to collect topK values per decoder layer, computing four threshold boundaries. These thresholds are then applied online with simple comparisons -- no sorting or topK at runtime. The profiling takes ~10 minutes even for Llama2-70B and only needs to be done once per model.

Component 2: Group-Shift Quantization

Directly quantizing outer groups with wide value ranges causes high quantization loss. Oaken shifts each group using the offline thresholds, narrowing the range and enabling low-bit quantization:

# Group-shift: narrow the range before quantization
for values in outer_group:
    if value > 0:
        shifted = value - T_hi_o   # Subtract upper threshold
    else:
        shifted = value - T_lo_o   # Subtract lower threshold
    quantized = uniform_quantize(shifted, bits=5)
    restored  = quantized + threshold  # Restore at dequantization

Component 3: Fused Dense-and-Sparse Encoding

Inliers (middle group) are stored as a dense 4-bit matrix. Outliers use sparse COO format. Oaken fuses both by embedding 4 bits of each 5-bit outlier into the zero positions of the dense matrix (where outliers were zeroed out), reducing each outlier entry from 23 bits to just 8 bits while maintaining memory alignment.

Figure: Fused dense-and-sparse encoding reduces outlier storage from 23 bits to 8 bits by embedding outlier data into zero positions of the dense matrix.

Hardware Architecture

Oaken Accelerator Overview

The Oaken accelerator comprises compute cores, memory controllers, a host interface, and an interconnect. Each compute core contains a Matrix Processing Unit (MPU), Vector Processing Unit (VPU), and a DMA unit with custom quantization/dequantization engines and a Memory Management Unit (MMU).

Figure: Oaken accelerator architecture. Custom quantization/dequantization engines and MMU are embedded in each core's DMA unit.

Quantization Engine Pipeline

1. Threshold Decomposer

Partitions incoming activations into outer, middle, and inner groups using offline thresholds. Performs group-shift by subtracting streamed thresholds.

2. Inlier Quantizer

Handles 4-bit uniform quantization for the middle group. Computes per-token min/max scaling factors dynamically at runtime.

3. Outlier Quantizer

Quantizes inner/outer groups to 5-bit. Generates COO index + group flag for sparse storage. Uses zero-remove shifter for fused encoding.

Memory Management Unit (MMU)

The MMU manages page-based reading/writing of quantized KV cache with two management tables:

  • Dense Management Table: virtual-to-physical address mappings for dense inlier matrices (predictable, well-aligned sizes)
  • Sparse Management Table: separate mappings for variable-size sparse outlier data

This dual-table design maximizes burst-mode memory bandwidth utilization while avoiding fragmentation.

Implementation Code

Core Quantizer: Multi-Group Threshold Decomposition

class OakenQuantizer:
    QUANTIZE_BITS = 4
    OUTLIER_BITS = 5

    @classmethod
    def get_multigroup_threshold(cls, input_tensor, threshold_lowers, threshold_uppers):
        """Split KV vector into outer/middle/inner groups using offline thresholds"""
        group_masks = []
        group_tensors = []
        prev_thr_low, prev_thr_up = None, None

        for idx, (thr_low, thr_up) in enumerate(zip(threshold_lowers, threshold_uppers)):
            if idx == len(threshold_lowers) - 1:
                # Inner-most group: values between previous thresholds
                mask = torch.logical_and(input_tensor > prev_thr_low,
                                         input_tensor < prev_thr_up)
            elif prev_thr_low is not None:
                # Middle groups: between current and previous thresholds
                mask = torch.logical_or(
                    torch.logical_and(prev_thr_low < input_tensor, input_tensor <= thr_low),
                    torch.logical_and(thr_up <= input_tensor, input_tensor < prev_thr_up))
            else:
                # Outer-most group: beyond the outermost thresholds
                mask = torch.logical_or(input_tensor <= thr_low, thr_up <= input_tensor)

            group_masks.append(mask)
            group_tensors.append(input_tensor * mask)

        return group_tensors, group_masks

Comparison with Existing Methods

Why Oaken Wins

Unlike KVQuant and KIVI that achieve good accuracy but cannot translate reductions into throughput (due to expensive online sorting), and unlike Tender/QServe that achieve low bitwidth but sacrifice accuracy (due to coarse channel reordering), Oaken finds the optimal trade-off: offline threshold profiling eliminates runtime overhead, group-shift quantization preserves accuracy, and fused encoding minimizes memory footprint. Combined with custom hardware, Oaken converts algorithmic gains directly into wall-clock speedup.

Real-World Benchmark Performance

Using production traces from Conversation (short generation) and BurstGPT (long generation), Oaken-LPDDR demonstrates 1.3-1.5x throughput improvement over GPU baselines. The advantage grows with longer generation sequences where KV cache bandwidth becomes increasingly dominant.

Sensitivity to Sequence Length

When sweeping total sequence length from 1K to 32K on Llama2-13B with batch size 16:

  • Below 8K tokens: GPU baselines (QServe, vLLM) outperform Oaken due to higher parallelizable compute resources available on GPUs
  • 8K-16K tokens: Oaken-HBM surpasses all baselines as the KV cache bandwidth bottleneck becomes dominant
  • Above 16K tokens: HBM-based systems (QServe, Oaken-HBM) run out of memory. Oaken-LPDDR alone can serve sequences up to 32K, demonstrating the combined power of KV quantization and large-capacity LPDDR memory

Design Insights and Trade-offs

Group Ratio Selection

Oaken uses a default group ratio of 4% outer, 90% middle, 6% inner. This was selected from a Pareto-optimal analysis sweeping group compositions while fixing total outlier ratio at 10%:

Group Ratio (o/m/i)Outlier BitsEffective BitwidthWikitext2 PPL
4 / 90 / 654.85.526
90 / 1054.85.804

4 / 90 / 3 / 355.65.516

Using three groups provides the optimal balance: it achieves the lowest perplexity among 4.8 effective-bitwidth configurations while maintaining 8-bit memory alignment for the fused dense-and-sparse encoding. Four or five groups would improve accuracy marginally but disrupt memory alignment (9-bit COO entries) or require additional padding, negating the benefits.

Token-Level Batch Scheduling

Oaken employs a sophisticated scheduling strategy that overlaps KV cache quantization/dequantization with DMA reads and attention computations from other requests. During the generation phase, each compute core processes a single output token from one request. The scheduler exploits the fact that KV cache operations cannot be shared across requests to interleave them with shareable operations, effectively hiding the quantization latency behind useful computation.
