ServerlessLLM: Fast Cold-Start Inference

Reducing LLM serverless inference cold-start latency by 10-200x through multi-tier checkpoint loading, efficient live migration of LLM inference, and startup-time-optimized model scheduling.

Three Core Contributions

  1. Fast multi-tier checkpoint loading -- Loading-optimized checkpoint format with a multi-tier storage pipeline that saturates bandwidth across GPU memory, DRAM, SSD, and remote storage.
  2. Efficient live migration of LLM inference -- Migrating only intermediate tokens (10-100s KB) instead of KV-cache (1-10s GB), then recomputing KV-cache on the destination.
  3. Startup-time-optimized model scheduling -- Dynamic programming-based scheduler that considers checkpoint locality across the storage hierarchy to minimize startup time.

2. The Cold-Start Problem

2.1 Why Cold Starts Are Critical

Azure Trace Data

Over 40% of serverless functions experience a cold-start rate exceeding 25%. Approximately 25% of functions experience a cold-start rate greater than 60%, within a 5-minute keep-alive interval.

LLM Checkpoint Sizes

Grok-1: 600+ GB. DBRX: 250 GB. Mixtral-8x22B: 280 GB. LLaMA-2-70B: 130 GB. Downloading 130GB from S3 at 5GB/s takes a minimum of 26 seconds.

2.2 Cold-Start Breakdown

Loading the OPT-30B model into 4 GPUs using PyTorch takes 34 seconds. Loading LLaMA-2-70B into 8 GPUs takes 84 seconds. A single inference token generation takes <100ms. The loading latency exceeds inference time by 340-840x.

2.3 Existing Solutions and Their Gaps

Over-subscribing GPUs

AWS keeps warm instances. Requires costly GPU reservations for LLMs with their large memory footprint.

Host Memory Caching

Cache checkpoints in DRAM. Works for small models (<few GB) but LLMs exceed host memory capacity. Cache misses frequent.

Storage Servers

Additional network storage. Even with 100Gbps NICs, model downloads can exceed 20 seconds through optimized storage servers.

3.2 Loading-Optimized Checkpoint Format

Sequential Chunk-Based Reading

Tensors grouped into per-GPU partitions. Files contain only binary data (no metadata interleaved). Enables large sequential reads (16MB chunks) instead of small scattered reads.

Direct Tensor Addressing

Tensor index file maps names to (GPU_id, offset, size) tuples. Tensors aligned to memory word sizes. GPU memory address computed directly: base + offset. No serialization/deserialization needed.

4.2 Multi-Round Live Migration Process

Live Migration Protocol (7 Steps)
1
Scheduler sends model load request to dest server
2
Scheduler sends migration request (dest address) to src server
3
src sends resume request with intermediate tokens to dest
4
dest recomputes KV-cache from tokens (10x faster than generating)
5
src stops inference, sends all remaining tokens to dest
6
Scheduler unloads model A from src, starts loading model B
7
Request router replaces src with dest in routing table
Why tokens not KV-cache? Intermediate tokens are 10-100s KB (compact). KV-cache is 1-10s GB (expensive to transfer). Recomputing KV-cache from 1000 tokens is ~10x faster than generating 1000 new tokens, making recomputation on the destination extremely efficient.

5.2 Startup Time Estimation

// Model loading time estimation
T_load = q + n / b
  // q = queuing time (models ahead in server's loading queue)
  // n = model size in bytes (or partition size for multi-GPU)
  // b = available bandwidth (slowest tier in pipeline)

// Model migration (resume) time estimation
T_resume = a * (t_in + t_out) + b
  // t_in  = number of input tokens
  // t_out = number of output tokens generated so far
  // a, b  = model-specific constants (profiled)

// Server selection: minimize total startup time
T_startup = min(T_load(server) + T_migrate(server)) for server in cluster

5.3 Fault Tolerance

ServerlessLLM stores server status (GPU, DRAM/SSD states) in a reliable key-value store (etcd/ZooKeeper). The scheduler handles three failure scenarios:

  • Source failure during loading (before step 2): Abort migration, unload from destination
  • Failure during migration (steps 2-3): Clear resumed KV-cache on destination
  • Destination failure: Cancel migration, source continues serving

6. Implementation Deep Dive

6.1 Controller (sllm/controller.py)

class SllmController:
    def __init__(self, config):
        self.request_routers = {}       # per-model routing
        self.registered_models = {}     # model registry

    async def start(self):
        # Initialize store manager (tracks checkpoint locality)
        self.store_manager = ray.remote(StoreManager).remote()
        await self.store_manager.initialize_cluster.remote()

        # Choose scheduler based on config
        if enable_storage_aware:
            scheduler = StorageAwareScheduler(config)
        else:
            scheduler = FcfsScheduler(config)

        # Choose router (round-robin or migration-aware)
        if enable_migration:
            router = MigrationRouter()
        else:
            router = RoundRobinRouter()

6.2 Storage-Aware Scheduler

class StorageAwareScheduler(FcfsScheduler):
    """Scheduler that considers checkpoint locality for server selection."""

    def __init__(self, config):
        self.enable_migration = config.get("enable_migration", False)

    # MigrationPlan: target_node_id + source_instance to migrate
    # AllocationPlan: node_id + estimated_latency + migration_plans

    async def _find_best_server(self, model_name, num_gpus):
        """Evaluate all servers, select one minimizing startup time."""
        best_plan = None
        for node in self.cluster_nodes:
            # Check if model checkpoint is on this node's storage
            locality = await self.store_manager.get_locality(
                model_name, node.id)

            # Estimate load time based on storage tier
            load_time = self._estimate_load_time(
                model_name, node, locality)

            # Consider migration if GPUs occupied
            if self.enable_migration:
                migration_plan = self._plan_migration(node, num_gpus)
                total_time = load_time + migration_plan.total_latency
            else:
                total_time = load_time

            if total_time < best_plan.latency:
                best_plan = AllocationPlan(node.id, total_time, ...)

8. Relevance to Multi-Agent Architectures

8.1 The Multi-Agent Cold-Start Challenge

Multi-agent AI systems deploy many specialized LLMs -- code agents, reasoning agents, retrieval agents -- with highly bursty per-model demand. When an agent orchestrator routes to a model that is not currently loaded, the cold-start penalty directly impacts end-to-end agent latency.

Multi-Agent Cold-Start Scenario
Agent Orchestrator
Routes tasks to specialists
➤
Code Agent (Loaded) -- 2ms response
Math Agent (Cold) -- 30s cold start!
RAG Agent (Loaded) -- 5ms response
Safety Agent (Cold) -- 15s cold start!

8.2 How ServerlessLLM Helps

Fast Model Swapping

With 6-8.2x faster checkpoint loading, cold starts for a 7B model drop from ~8s to ~0.9s. For 70B models, from 84s to ~10s. This makes agent-level model switching practical.

Locality-Aware Scheduling

The scheduler learns which models are cached where, routing requests to servers with local copies. For frequently-used agents, this often eliminates cold starts entirely.

Live Migration for Load Balancing

When one agent model gets a burst of requests, live migration moves existing inferences away from congested servers without interruption, freeing GPUs for the bursty model.

Resource Efficiency

Instead of over-provisioning GPUs for each possible agent model, ServerlessLLM enables a shared pool where models are loaded on demand. With just 1 GPU per node, it matches the latency of systems requiring 4 GPUs.

8.3 Limitations and Future Directions

Sequential Model Loading

Currently loads models sequentially on the I/O path. Concurrent loading on multiple servers with fairness guarantees is planned for future work.

Checkpoint Placement

Smart checkpoint placement across the cluster (which models to cache where) is treated as orthogonal and not addressed. This is an important optimization for multi-agent workloads with known access patterns.

Critical for agentic AI: As multi-agent architectures grow with 10-100s of specialized models, the cold-start problem becomes a first-order bottleneck. ServerlessLLM's 10-200x latency reduction makes the difference between a responsive agent system (<1s model switches) and one that breaks user experience (30+ second delays).

9.2 Key Optimizations in the Data Path

OptimizationTechniqueBenefitSpeedup
Bulk Reading Group tensors sequentially, read large contiguous blocks Reduces per-tensor overhead for small tensors (<1MB avg) 1.2x
Direct I/O O_DIRECT flag bypasses OS page cache Avoids double-copy through kernel page cache 2.1x
Multi-threading Multiple I/O threads per storage tier Exploits SSD/NVMe parallelism (multiple channels) 2.3x
Pinned Memory Pre-allocated pinned DRAM for GPU DMA transfers Zero-copy from DRAM to GPU via DMA engine 1.4x
Pipeline Overlap I/O across storage tiers All tiers active simultaneously 1.5x
Multiplicative gains: The optimizations compound: 1.2 * 2.1 * 2.3 * 1.4 * 1.5 = 12.2x theoretical maximum. Practical gains are 6-8.2x due to bottleneck effects at different storage tiers.

9.3 Decoupled Loading and Initialization

ServerlessLLM decouples the binary data loading from model initialization. A Model Manager allocates GPU memory and loads binary data, while the inference process initializes the model object by setting data pointers via CUDA IPC handles. This decoupling enables:

  • Pre-scheduled checkpoint loading before inference requests arrive
  • Overlapping loading with initialization of previously loaded layers
  • Multiple model partitions loaded in parallel across GPUs via parallel PCIe links

10.2 LoRA Adapter Loading

ServerlessLLM also supports loading LoRA adapters in PEFT format. For a rank-32, 1GB adapter on LLaMA-70B:

  • ServerlessLLM: 83.5ms loading latency
  • Safetensors: 370ms loading latency
  • Speedup: 4.4x faster for adapter loading

This is particularly relevant for multi-agent systems where different agents may use the same base model with different fine-tuned adapters.
